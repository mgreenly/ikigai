# Decision 17 — Structured MCP adoption: `send` returns `structuredContent`, typed error codes, declared `outputSchema`

**Problem.** notify's one domain tool, `send`, was written for the pre-adoption
appkit MCP transport: its success path calls `appkitmcp.JSONResult(v)` (a typed
value marshalled to a text-only block) and its two failure paths hand-build a
`{"error":{code,message,field}}` envelope and pass the marshalled JSON *string*
to the old `appkitmcp.ErrorResult(string)`. The suite has moved (see
`docs/structured-mcp-design.md`): `appkit/mcp` now speaks revision `2025-06-18`,
`JSONResult` is deleted, `ErrorResult` is re-signed to
`ErrorResult(code ErrorCode, msg string)` over a **closed** error vocabulary,
and every domain verb is expected to declare an `outputSchema` and return
`structuredContent` beside its mirrored text so a deterministic caller (the
scripts runtime) parses a machine object, never prose. notify does not compile
against the new appkit until it adopts the plane. This Decision is notify's
adoption of that shared contract; it changes no domain behavior — the priority
mapping, tags join, click validation, ntfy publish hop, and the fact that
`send` reports the real ntfy outcome synchronously are all unchanged.

**Decision.** Reshape the three result sites in `internal/mcp/tools.go` to the
new appkit surface, formalizing the JSON `send` already emits without reshaping
it.

1. **Success → `StructuredResult`.** `toolSend`'s success return becomes
   `return appkitmcp.StructuredResult(map[string]any{"ok": true})`. The new
   helper returns `(map[string]any, error)`; because `toolSend` already returns
   `(map[string]any, error)`, the error is propagated verbatim (never
   swallowed, never discarded into a bare result). The emitted object is
   **identical** to today's — `{"ok":true}` — now carried as
   `structuredContent` with the same JSON mirrored into the text block.

2. **Typed error codes over the closed vocabulary.** Both failure paths adopt
   `appkitmcp.ErrorResult(code, msg)`. The mapping from notify's failure modes
   to the shared vocabulary is:

   | failure (site) | pre-adoption string code | new `ErrorCode` |
   |---|---|---|
   | empty/missing `message` | `validation` | `appkitmcp.ErrValidation` |
   | `priority` outside `min\|low\|default\|high\|urgent` | `validation` | `appkitmcp.ErrValidation` |
   | `click` not a well-formed absolute URL | `validation` | `appkitmcp.ErrValidation` |
   | ntfy rejected the publish (non-2xx) **or** was unreachable | `upstream` | `appkitmcp.ErrSourceUnavailable` |

   The remap that matters: notify's old `upstream` code is **not** in the
   closed vocabulary and is retired. An ntfy rejection or an unreachable ntfy
   server is a failure of the downstream push source, so it maps to
   `source_unavailable` — the vocabulary's slot for "a dependency this verb
   needs did not answer." The local `validationErr`/`upstreamErr` helpers and
   the hand-marshalled `{"error":{…}}` envelope are deleted; `ErrorResult`
   builds the envelope (`structuredContent: {code, message}` + the human
   message as the text block).

   - **The `field` sub-key is dropped.** The old `validationErr` attached a
     separate `field` key ("message"/"priority"/"click"). The shared
     `structuredContent` shape is `{code, message}` only. This loses no
     information: every validation message already names its field verbatim
     ("message is required and must be non-empty", "priority must be one of
     min, low, default, high, urgent", "click must be a well-formed absolute
     URL"), so the human message remains self-identifying and the machine code
     is the branch key.
   - **No secret ever appears in an error.** The `source_unavailable` message
     stays the fixed, topic-free, token-free string
     ("the notification service rejected the request or was unreachable") — the
     existing non-leak guarantee is preserved through the new helper.

3. **Declared `outputSchema` for `send`.** `send` is a machine verb, so its
   descriptor gains an `outputSchema` mirroring the JSON it emits verbatim — an
   object with one required boolean `ok`. Authored as a literal through
   notify's existing schema helpers (`obj`/`typ`, which already produce the
   appkit `objectSchema` shape), e.g.:

   ```go
   // sendOutputSchema mirrors the {"ok": true} success object send returns.
   var sendOutputSchema = obj(map[string]any{
       "ok": typ("boolean"),
   }, "ok")
   ```

   attached as the `send` Tool's `OutputSchema`. The chassis surfaces it in
   `tools/list` (appkit omits the key when nil; a declared schema is emitted).

4. **Prose exceptions: none among notify's domain tools.** notify declares
   exactly one domain tool, `send`, and it is structured. It has **no**
   documentation tool (`guide`/`describe`) and no raw-content read, so there is
   no domain tool that keeps `TextResult` and omits an `outputSchema`. The
   chassis `health` and `reflection` already declare schemas and return
   structured results centrally (appkit); notify does nothing for them.

5. **Guard swap: not applicable.** notify holds **no** hand-copied loopback
   guard predicate — it serves no `/feed` of its own (it is a consumer; the
   chassis mounts and guards the consumer machinery) and no content endpoints.
   A source scan confirms the only `X-Owner-Email` mention in notify's Go is a
   comment in `cmd/notify/main.go`. There is nothing to migrate to
   `server.LoopbackOnly`/`Router.HandleLoopback`.

The change is confined to `internal/mcp/tools.go` and its tests
(`tools_test.go`, `send_test.go`, which swap their success/error assertions
from the old text-only/`{"error":{…}}` shapes to `structuredContent` +
`{code,message}`). No product change, no migration, no schema change, no other
package touched.

**Rejected.**
- **Keeping a notify-local `upstream` code beside the shared vocabulary** —
  reintroduces the drift the closed vocabulary exists to prevent; a machine
  caller branching on `source_unavailable` across services would miss notify.
- **Mapping ntfy failure to `internal`** — `internal` is for a bug in notify
  itself; a rejecting or unreachable *external* push source is precisely
  `source_unavailable`, and conflating the two would mislead a caller's retry
  decision.
- **Preserving the `field` key by widening the shared error shape** — the
  vocabulary is `{code, message}` by contract (`docs/structured-mcp-design.md`);
  the field is already in the message, so widening buys nothing and forks the
  shape.
- **Swallowing `StructuredResult`'s error into a bare result** — hides a
  marshal failure behind a false success; the handler already returns an error
  channel, so honest propagation is free.

**Verification** (through the composition-root seam — the appkit/mcp handler
built by `NewHandler` over a mock-ntfy `push.Client`, driven with
`tools/list`/`tools/call` over the real `ServeHTTP`).
- R-A918-YY6H — a successful `send` (mock ntfy accepts) returns a result
  carrying `structuredContent` equal to `{"ok": true}` **and** a text block
  that is the same JSON object; a text-only result with no `structuredContent`
  key (the retired `JSONResult` shape) fails.
- R-AA95-CPX6 — `tools/list` advertises `send` with an `outputSchema` that is
  an object schema requiring a boolean `ok`
  (`type == "object"`, `properties.ok.type == "boolean"`, `required` contains
  `"ok"`); a `send` descriptor missing the `outputSchema` key fails. (notify's
  only domain tool declares a schema; `health`/`reflection` are chassis-owned;
  there are no prose-exception domain tools to exclude.)
- R-ACOY-49EK — an invalid `send` (missing/empty `message`, `priority` outside
  the enum, or a non-absolute `click`) returns an error result whose
  `structuredContent.code` is exactly `validation` and fires **zero** POSTs to
  the mock ntfy server; a bare-string code or a fired push fails.
- R-ADWU-I159 — when ntfy rejects the publish (non-2xx) or is unreachable,
  `send` returns an error result whose `structuredContent.code` is exactly
  `source_unavailable` (the retired `upstream` code must be absent) and whose
  human message contains neither the configured topic nor the API token; a
  message carrying either secret, or the `upstream` code, fails.
