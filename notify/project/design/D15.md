# Decision 15 — the session-gated locations opt into the apex `@login_bounce`: a logged-out human navigation goes to sign-in, not a bare 401

**Decision.** notify's nginx fragment (`notify/etc/nginx.conf`) opts its **two
browser-facing, session-gated locations** into the dashboard-owned apex
`@login_bounce` primitive by adding **one line** — `error_page 401 =
@login_bounce;` — to each. Today a logged-out browser navigating to either of
them gets the bare nginx **401** the `auth_request /_session-authn` deny
produces; with the opt-in, the apex `@login_bounce` (a dashboard-owned external
contract, defined in the apex `server` block exactly where `/_session-authn` is —
D20 in the dashboard's `project/`) turns a logged-out **navigation** into a `302`
to `/login?return_to=…` and returns the person to the page after they sign in,
while a scripted `fetch`/XHR still gets the clean `401` it can handle.
`@login_bounce` makes that navigate-vs-fetch distinction itself (off
`Sec-Fetch-Mode`); notify only has to reference it. sites made this exact move
first (its D24); this Decision applies the same one-line-per-location opt-in to
notify's fragment.

The two locations that gain the line — every session-gated (`auth_request
/_session-authn`) location in the fragment:

```
location = /srv/notify/ {                 # landing root (proxied to the process)
    auth_request /_session-authn;
    error_page 401 = @login_bounce;      # ← added
    …
}
location /srv/notify/static/ {            # landing assets
    auth_request /_session-authn;
    error_page 401 = @login_bounce;      # ← added
    …
}
```

**What deliberately does NOT get the line.** The change is confined to the
browser/session tiers:

- `location /srv/notify/ {` — the **bearer** tier (`auth_request /_authn`) fronting `/srv/notify/mcp` and every other sub-path: it must keep returning `401` with its `WWW-Authenticate` challenge so MCP clients can discover the AS; redirecting an MCP call to an HTML login page would break the OAuth bootstrap.
- `location = /srv/notify/.well-known/oauth-protected-resource` and the 429 re-emit named location — unauthenticated PRM bootstrap and the bearer rate-limit re-emit; neither is a browser session gate.

The change is **purely additive** — one line per session-gated location, every
existing directive (`auth_request /_session-authn`, the `proxy_pass`, the
headers) retained and nothing rewritten. It adds **no new suite dependency**
beyond the one notify already leans on: it referenced apex-owned
`/_session-authn` before, and now also references apex-owned `@login_bounce`
(both provided by the dashboard's apex block).

**Rejected.**
- **Adding `error_page 401 = @login_bounce;` to the bearer tier** — the bearer
  401 is a protocol signal (the `WWW-Authenticate` challenge MCP clients use to
  discover the AS); bouncing it to an HTML login page breaks MCP client
  discovery. The opt-in is browser-only.
- **Having notify define its own bounce (a local named location / `if`)** — the
  bounce is a suite-wide contract owned once by the apex; duplicating it in
  every service fragment is exactly the copy-paste the shared `@login_bounce`
  exists to avoid.
- **Bouncing all requests regardless of `Sec-Fetch-Mode`** — a logged-out
  browser-side `fetch` would then receive login HTML with a `200` instead of a
  `401` it can handle; the navigate-vs-fetch split (in `@login_bounce`) is
  deliberate and this fragment inherits it for free.

**Verification.** (Proven by extending the existing nginx content-assertion test
(`cmd/notify/main_test.go`) that reads `notify/etc/nginx.conf` from disk and asserts over its
content; nginx is not run by the suite.)
- R-3IZH-DPMO — in `notify/etc/nginx.conf`, each of the two session-gated locations —
  the exact-match landing root `= /srv/notify/` and the asset tier
  `/srv/notify/static/` — contains **both** `auth_request /_session-authn;` and
  `error_page 401 = @login_bounce;`.
- R-3K7D-RHDD — the bearer prefix `location /srv/notify/` (the `auth_request /_authn` tier) does **not** contain `error_page 401 = @login_bounce;` (the opt-in is confined to the session tiers; the
  bearer 401 is left intact).
- R-3LFA-5942 — the change is additive: every pre-existing location still appears
  and each session-gated location retains its `auth_request /_session-authn;`
  and its `proxy_pass` — no existing directive is removed or rewritten by the
  opt-in.
