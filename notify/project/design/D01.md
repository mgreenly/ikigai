# Decision 1 — The landing handler and its v1 content (service name + version)

**Decision.** notify owns its landing page. A new package `notify/internal/web/`
exposes a constructor — `web.LandingHandler(service, version string) http.Handler`
— that returns an `http.Handler` rendering the v1 landing page: a Carbon-styled,
centered card showing the **service name** (display type) and the **running
version** (mono label) and nothing else. There is **no shared landing handler**
across apps; this code lives under `notify/`. The two content values are passed in
as plain strings — the composition root supplies them from the chassis
(`rt.Service()` → `"notify"`, `rt.Version()` → the build version, D2) — so the
handler is pure over its inputs and needs no chassis, DB, clock, or network at
test time.

**Rendering.** The page is an embedded template rendered once per request (or
once at construction and written verbatim — either is fine; the observable
contract is the output):

- The HTML is embedded via `//go:embed` from a `*.html` file in
  `notify/internal/web/` (e.g. `landing.html`). The service name and version are
  the only dynamic substitutions; everything else is static markup.
- The template is rendered with the standard library (`html/template` preferred —
  the two values are trusted but `html/template` is the correct default for HTML;
  `text/template` is acceptable since the inputs are not attacker-controlled).
- The response sets `Content-Type: text/html; charset=utf-8` explicitly and
  returns `200 OK` for the page.
- The markup links the app's **own** embedded `tokens.css` and applies the Carbon
  type scale (Space Grotesk display for the name, IBM Plex Mono for the version
  label) and neutral palette. The Carbon asset wiring is D3; this Decision owns
  that the page is *rendered with name + version* and is a single centered card.

**Why a constructor over strings, not over the chassis.** Passing `service` and
`version` as strings (rather than handing the package the `*appkit.Router`) keeps
`internal/web` free of any chassis dependency and makes the handler trivially
table-testable with fixed values. The composition root is the only place that
reaches into `rt` for the two strings.

**Rejected.**
- **A shared cross-app landing handler** (e.g. in appkit) — the brief's locked
  design: each app's page will diverge to serve that app's purpose, so each app
  owns its page. A shared handler would have to be un-shared at the first
  divergence. v1's uniformity is a content coincidence, not a shared component.
- **Rendering anything beyond name + version** in v1 (owner email, push history,
  consumer-cursor state, links) — out of product scope; v1 is exactly name +
  version.
- **Pulling the values from env or a new config knob** — the chassis already
  exposes the authoritative service name and build version; a second source could
  drift from what `version`/`/health` report.
- **Reading the identity header (`X-Owner-Email`) to personalize the page** — the
  gate is coarse (any logged-in user); v1 shows no per-user content, so the
  handler ignores identity entirely.

**Verification.**
- R-LAND-3C8K — `web.LandingHandler("notify", v)` serving a `GET /` returns
  HTTP `200`.
- R-LAND-5D1M — the rendered body contains the service name passed in (the page
  shows `"notify"`).
- R-LAND-7E4N — the rendered body contains the version string passed in (e.g. a
  test version like `"9.9.9-test"` appears verbatim), proving the page reflects
  the running version rather than a hardcoded constant.
- R-LAND-9F6P — the landing response sets `Content-Type: text/html; charset=utf-8`.
