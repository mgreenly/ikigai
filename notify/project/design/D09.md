# Decision 9 — Adopt `registry`: resolve notify's own and its peers' loopback ports by name at the composition root

**Decision.** notify stops writing loopback port literals in its Go source and
instead references itself and its peers **by name** through the shared `registry`
library (`module registry`, the repo-root leaf module). Resolution happens
**once, at the composition root** (`cmd/notify/main.go`) — never per request —
because the table is compile-time data: if a port ever changes we rebuild the
binaries anyway. This is **behavior-preserving**: `registry`'s table already
carries notify's current values (`notify=3201`, `crm=3100`, `prompts=3002`), so
every resolved value is byte-identical to the literal it replaces.

**Why.** Today "which port a service answers on" is duplicated across the suite
and drifts silently — a renumber only surfaces at deploy, uncatchable by the
compiler. notify hardcodes three such literals: its own `Port` default and two
peer-feed fallback URLs. Making `registry` the single source of truth means
notify asks the table for the address and never repeats the number.

**The three call sites (all in `cmd/notify/main.go`).**

- **Own port** — the appkit `Spec.Port` literal becomes a name lookup, the strict
  composition-root form that panics on an unknown name (a programming error):

  ```go
  appkit.Main(appkit.Spec{
      App:   "notify",
      Mount: "/srv/notify/",
      Port:  registry.MustPort("notify"), // was: 3201
      ...
  })
  ```

- **crm feed default** — the `CRM_FEED_URL` fallback literal becomes the registry
  base URL with the route appended by the caller (`registry` owns the address,
  not the path):

  ```go
  feedURL: envOr(getenv, "CRM_FEED_URL", registry.BaseURL("crm")+"/feed"),
  // was: "http://127.0.0.1:3100/feed"
  ```

- **prompts feed default** — identical shape for the second consumer loop:

  ```go
  promptsFeedURL: envOr(getenv, "PROMPTS_FEED_URL", registry.BaseURL("prompts")+"/feed"),
  // was: "http://127.0.0.1:3002/feed"
  ```

**The env-override layer is unchanged.** `NOTIFY_PORT`, `CRM_FEED_URL`, and
`PROMPTS_FEED_URL` remain the override layer exactly as before — a set value
still wins. `registry` supplies only the **default** that the override falls back
to, in place of the literal. `resolveConsumerCfg`'s `envOr(getenv, KEY, default)`
structure is untouched; only the `default` argument changes from a literal to a
`registry` call. The two ntfy secrets (`NTFY_TOPIC`, `NTFY_API_KEY`) and the
`NOTIFY_FROM` / `NOTIFY_NTFY_BASE_URL` plain config are entirely unaffected.

**Module wiring (inside `notify/` only).** `notify/go.mod` gains a
`require registry v0.0.0` and a committed `replace registry => ../registry`,
mirroring the existing `appkit` and `eventplane` in-repo replace-siblings. This
is the only build-graph change and it lives wholly under `notify/`.

> **External precondition (OUT OF SCOPE — not built by any notify phase).** The
> repo-root `go.work` must carry `use ./registry`, and the `registry` module
> itself must exist and be green. Both are owned at the repo root / by the
> `registry` project and are assumed satisfied. No notify Decision or phase edits
> `../go.work`, `../registry/`, `../bin/`, nginx, or any sibling module — the
> executor runs from `notify/` and cannot reach outside it.

**Resolve-at-startup, not per-request.** The lookups above run when the process
composes its config: `Spec.Port` is read once by appkit at boot, and the feed
URLs are resolved once in `resolveConsumerCfg` and stored on the `consumerCfg`
struct that the consumer workers read. No handler or request path calls
`registry`. This matches D3's contract (the table is compile-time data) and adds
no runtime cost.

**Scope guard.** This Decision changes **only** how three existing values are
sourced. It adds no new config key, no new package, no schema change (**no
migration**), and no behavior a user or peer can observe — the feed URLs and the
listen port are identical to today. The consumer loops, the ntfy push domain, the
MCP surface, and the landing page are untouched.

**Rejected.**
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; looking it up per request buys nothing and violates D3's
  resolve-once intent. Resolve at the composition root, store the value, reuse it.
- **Dropping the `*_FEED_URL` env overrides now that a default exists.** The
  overrides are the documented seam tests and dev use to point a loopback loop at
  a mock feed server; `registry` replaces the literal *default*, not the override
  mechanism. Removing them would break the e2e tests that inject a mock feed URL.
- **`registry.Port("notify")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form here (D3).
- **Appending the path inside `registry` (a `FeedURL(name)` helper).** Rejected in
  D3: routes are an appkit/eventplane convention, not addressing. The caller
  writes `registry.BaseURL("crm") + "/feed"`.

**Verification.**
- R-RGSP-4A1K — **notify's own listen port is `registry.MustPort("notify")`**, not
  a `3201` integer literal, at the composition root. Falsifiable: the served/emitted
  port must equal `registry.MustPort("notify")`; a hardcoded `Port: 3201` in the
  Spec fails the intent (proven concretely by the manifest drift guard R-RGDR,
  which emits with `registry.MustPort("notify")` and matches the committed
  `PORT=3201`).
- R-RGCF-4B2L — **the crm feed URL default is `registry.BaseURL("crm") + "/feed"`**
  when `CRM_FEED_URL` is unset. Falsifiable: with `CRM_FEED_URL` empty,
  `resolveConsumerCfg(...).feedURL` must equal `registry.BaseURL("crm")+"/feed"`
  (== `"http://127.0.0.1:3100/feed"`); any other value, or a surviving literal,
  fails it.
- R-RGPF-4C3M — **the prompts feed URL default is
  `registry.BaseURL("prompts") + "/feed"`** when `PROMPTS_FEED_URL` is unset.
  Falsifiable: with `PROMPTS_FEED_URL` empty,
  `resolveConsumerCfg(...).promptsFeedURL` must equal
  `registry.BaseURL("prompts")+"/feed"` (== `"http://127.0.0.1:3002/feed"`).
- R-RGEO-4D4N — **an explicit env override still wins over the registry default.**
  Falsifiable: with `CRM_FEED_URL` (resp. `PROMPTS_FEED_URL`) set to a sentinel
  URL, `resolveConsumerCfg` returns that sentinel, not the `registry`-derived
  default — proving `registry` supplies only the fallback, not a hardcode.
