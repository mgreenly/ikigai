# Decision 11 — Consumer loops through `Spec.Consumers` (chassis-owned)

**Problem.** notify hand-writes its two event-plane consumer loops at the
composition root: `runConsumer` and `runPromptsConsumer` are structurally
identical ~30-line blocks (resolve config, assemble `consumer.Config`, log,
`consumer.Run`, wrap the error), launched through `Spec.Workers` via a
`var rt *appkit.Router` capture that smuggles the Router out of `Handlers`.
The consumer role is declared in three Spec places that must manually agree
(`Consumes`, `Subscriptions`, the `Workers` loops), and notify's env names
predate the suite convention: it reads `CRM_FEED_URL` / `PROMPTS_FEED_URL` and
a single app-wide `NOTIFY_FROM`, where scripts and prompts read
`<APP>_<SRC>_FEED_URL` / `<APP>_<SRC>_FROM` per upstream. The chassis now owns
all of this (appkit design D10, consumed here as a fixed external contract);
notify is the pilot conversion.

**Decision.** notify declares its consumer role as the `Spec.Consumers` table
and deletes its hand-rolled wiring:

```go
// cmd/notify — the whole consumer declaration.
Consumers: []appkit.Consumer{
    {
        Source:        "crm",
        Subscriptions: []consumer.Subscription{push.Subscription()},
        Handler: func(rt *appkit.Router) consumer.Handler {
            cfg := mustNtfyCfg(os.Getenv) // same fail-loud resolution as the send verb
            client := push.NewClient(cfg.ntfyBase, cfg.ntfyTopic, cfg.ntfyToken, rt.Logger())
            return push.Handler(client, rt.Logger())
        },
    },
    {
        Source:        "prompts",
        Subscriptions: push.PromptsSubscriptions(),
        Handler: func(rt *appkit.Router) consumer.Handler {
            cfg := mustNtfyCfg(os.Getenv)
            client := push.NewClient(cfg.ntfyBase, cfg.ntfyTopic, cfg.ntfyToken, rt.Logger())
            return push.PromptsHandler(client, rt.Logger())
        },
    },
},
```

- **Deleted:** `runConsumer`, `runPromptsConsumer`, the `Workers` field, the
  `var rt *appkit.Router` capture, the legacy `Consumes` and `Subscriptions`
  Spec fields (both derived from the table by the chassis — setting them
  alongside `Consumers` is a chassis startup error), and the
  `feedURL`/`promptsFeedURL`/`from` members of the config struct with their
  `CRM_FEED_URL`/`PROMPTS_FEED_URL`/`NOTIFY_FROM` reads. The chassis resolves
  `NOTIFY_CRM_FEED_URL`/`NOTIFY_PROMPTS_FEED_URL` (defaults from `registry`)
  and `NOTIFY_CRM_FROM`/`NOTIFY_PROMPTS_FROM` (default `tail`).
- **Kept:** the ntfy config resolution (base/topic/token, fail-loud on a
  missing secret — the exact `resolveConsumerCfg` discipline, shrunk to its
  ntfy remnant), `push.Handler`/`push.PromptsHandler` and their subscription
  lists, and the per-loop client construction (each handler factory builds its
  own `push.Client`, matching the pre-conversion behavior where each worker
  built its own).
- **Behavior is unchanged:** same two upstreams, same subscriptions (whose
  Filter strings and handler-side event matching are the canonical-key forms
  owned by D16), same best-effort push semantics, same `feed_offset` cursors
  (`consumer_id`
  `"notify"` == `Spec.App`, sources `"crm"`/`"prompts"` — the chassis derives
  the same values notify hardcoded), and the manifest still says
  `CONSUMES=crm,prompts` byte-for-byte (R-RGDR-4F6Q keeps pinning it).
- **Dev wiring (boundary-crossing).** `notify/.envrc` drops its
  `CRM_FEED_URL`/`NOTIFY_FROM` exports (inside `notify/`), and `bin/start`'s
  `launch_notify` swaps its legacy `CRM_FEED_URL`/`PROMPTS_FEED_URL` exports
  for the generic `<APP>_<SRC>_FEED_URL` form it already uses for scripts and
  prompts. The `bin/start` line crosses the `notify/` boundary and is verified
  by the live `bin/start` smoke, not the Go suite (the workspace's standing
  convention for repo-root collaborators). Prod sets none of these variables,
  so no box change.

**Rejected.**
- **Keeping the `Workers` wiring with an eventplane helper** — folds some
  lines but keeps the capture dance and the three-field agreement; the pilot
  exists to prove the declared form.
- **Sharing one `push.Client` across the two factories via a captured
  variable** — reintroduces the capture dance the seam removes; two clients is
  the status quo and the client is stateless config + an `http.Client`.
- **Keeping `NOTIFY_FROM` as an app-wide alias** — a second resolution path in
  parallel with the chassis convention is drift by construction; nothing sets
  it in prod and dev moves with the same commit.

**Verification** (the Spec value directly, and the handler behavior through a
mock ntfy server — the same substrate the existing push tests use).
- R-4DG9-3Q97 — notify's Spec declares **exactly two** `Consumers` entries, in
  order `crm` then `prompts`, whose subscription lists equal
  `push.Subscription()` and `push.PromptsSubscriptions()` respectively — a
  dropped upstream, a swapped order, or a diverged subscription list fails
  this.
- R-4EO5-HHZW — each entry's handler factory, invoked with a Router, returns a
  working handler: fed its subscribed event it issues the same ntfy POST the
  pre-conversion worker issued (asserted against a mock ntfy server), and fed
  a non-subscribed event it issues none.
