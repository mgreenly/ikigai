# Decision 15 — Event-routing conformance: kind `recorded`, empty subject

> **External ordering (operator-sequenced, not built by any ledger phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before ledger's conformance phase runs**. It equally assumes appkit (which
> holds `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, ledger's conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and the
old type names drop their redundant noun prefix (`source` already says ledger
deals in transactions). It is a hard cutover — no compatibility period —
inside the no-live-data migration window.

**What ledger publishes today (ground truth, not vacuous).** ledger is a real,
wired producer: `Spec.Feed: "/feed"`, `Spec.Events: ledger.Events` (an
`outbox.Registry` of one `EventType` in `internal/ledger/events.go`), the
`Producer` hook wiring `NewOutboxProducer` onto `Service.Outbox`, and one
emission path — `Service.persist` (shared by `Record` and `Reverse`, so every
committed transaction emits exactly one event, reversal mirrors included) →
`AppendRecorded` → `transactionRecordedEvent` → `outbox.Append(tx,
Event{Type: "transaction.recorded", Payload})`, atomic with the journal write,
`Ring()` after commit. The single published type converts.

**Decision.** ledger adopts the revised envelope exactly over what it
publishes, per the routing doc's producer key map:

| today (`type`) | revised key | kind | subject |
|---|---|---|---|
| `transaction.recorded` | `ledger:recorded` | `recorded` | `""` |

The subject is **empty** — the routing doc's producer key map records
`ledger:recorded` subjectless, and no routing dimension has been identified
(an account path is a *set* per transaction, not a name; consumers that select
by account filter in their handler on the payload, as today). A future subject
is a fresh design change, not scoped here. The payload shape is **unchanged**
apart from D14's `external_ref` addition — the routing key addresses; the
payload remains the detail record.

1. **Constant and event builder** (`internal/ledger/events.go`):
   `eventTransactionRecorded = "transaction.recorded"` becomes
   `kindRecorded = "recorded"` (still declared once, referenced at both the
   emit site and the registry so the two cannot drift).
   `transactionRecordedEvent` returns `outbox.Event{Kind: kindRecorded,
   Subject: "", Payload: raw}`. Nothing else in the `Service`/`EventSink` seam
   changes (same tx via `persist`, `Ring()` after commit; the
   one-event-per-committed-transaction rule, mirrors included, is untouched).

2. **The registry becomes a family** (`internal/ledger/events.go`):
   `ledger.Events` becomes an `outbox.Registry` of one `outbox.Family` —
   `{Kind: "recorded", Subject: "", Description: …, Sample:
   sampleTransactionRecorded}`. The description keeps its current meaning
   (a balanced transaction committed to the immutable journal, reversal
   mirrors included with non-null `reverses_id`) with the type name updated;
   the Sample stays the filled-in `transactionRecordedPayload` (now carrying
   `external_ref`, D14). `Spec.Events` wiring is untouched; Append-time kind
   gating and the chassis reflection surface hang off the same registry, now
   family-shaped.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. ledger appends **one
   new timestamped migration** minted with `bin/create-migration ledger
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across.
   The frozen `003_outbox.sql` is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest**
   outbox migration, asserting it contains the revised `outbox.SchemaSQL`
   verbatim. (This exactly mirrors gmail's D18/phase-16 and dropbox's
   D22/phase-23 handling.)

4. **Reflection/consumer surface.** The chassis reflection tool (appkit-owned)
   reflects the family registry; ledger's observable is that its one kind —
   and only it — is listed, with schema/example still derived from the same
   Sample. ledger consumes nothing (producer only), so there is no
   filter/`consumer.Config` side to conform.

**Rejected.**
- **Keeping the `transaction.` prefix as the kind (`transaction.recorded`)** —
  the revision's explicit rename rule: `source` carries the noun; a
  `ledger:transaction.recorded` key says "ledger's domain entity" twice.
- **A subject now (e.g. `/<txn-id>` or an account path)** — the txn id is
  useless for routing (never known in advance) and an account path is one
  subject per *posting*, not per event; minting one speculatively bakes in a
  choice the first real routing consumer should make.
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are retention data with no live
  consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: the emission claims run over a
real SQLite database migrated through ledger's full embedded migration set,
outbox rows read back by SQL; the feed claim runs the real eventplane
`FeedHandler` over `httptest`).

- R-FXKF-JD3L — after a `Record` and a `Reverse` commit, the two outbox rows
  read back by SQL from the real SQLite database each carry `kind` exactly
  `recorded` (no `transaction.` prefix) and `subject = ''`, each appended on
  the same tx as the journal write (a rolled-back write appends no row —
  baseline atomicity preserved), with the reversal row's payload carrying
  non-null `reverses_id`.
- R-FYSB-X4UA — `ledger.Events` declares exactly one family with kind
  `recorded` and empty subject; through the assembled MCP handler,
  reflection's index lists exactly that one publish, its detail returns a
  schema and example that agree (every example field in the schema,
  `transactionRecordedPayload`-shaped, `external_ref` present), and an
  undeclared kind returns the typed unknown-kind error naming `recorded` as
  the sole declared kind.
- R-G184-OOBO — applying ledger's full embedded migration set (`db.FS`,
  through the appkit runner) to a fresh real SQLite database yields an
  `outbox` table with `kind` and `subject` columns and **no** `type` column;
  the DDL drift guard asserts the newest outbox migration contains the
  revised `outbox.SchemaSQL` verbatim while `003_outbox.sql` remains
  byte-identical to its committed (frozen) body.
- R-G2G1-2G2D — end-to-end over ledger's own substrate: an event appended
  through ledger's producer path (`persist` → `AppendRecorded` → `Append` on
  the migration-created table, source `ledger` — the `Spec.App` value) and
  served by the real eventplane `FeedHandler` over an `httptest` connection
  frames `event: ledger:recorded` (the canonical subjectless key) with a
  `data:` JSON envelope carrying `kind`/`subject` keys and **no** `type` key.
