# Decision 1 — The landing handler and its v1 content (service name + version)

**Decision.** ledger owns its landing page *content*. The page is a Carbon-styled,
centered card showing the **service name** (display type) and the **running
version** (mono label) and nothing else. The template lives on disk at
`ledger/share/www/landing.html` (shipped in the release `share` tier, per D10 and
appkit's chassis design); the handler is a small `http.Handler` at the composition
root that renders it through the chassis-loaded site:

```go
// cmd/ledger — inside Spec.Handlers; rt.WWW() is the appkit/web Site loaded
// from the www root (Spec.WWW).
func landingHandler(site *web.Site, service, version string) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        _ = site.Render(w, "landing.html",
            struct{ Service, Version string }{service, version})
    })
}
```

There is **no shared landing page** across apps: each service ships its own
`share/www/landing.html`; only the rendering *mechanism* (template loading, static
serving) is chassis-owned. The two content values are passed as plain strings — the
composition root supplies them from the chassis (`rt.Service()` → `"ledger"`,
`rt.Version()` → the build version, D2) — so the handler is pure over its inputs
and the template file and needs no chassis, DB, clock, or network at test time.

**Rendering.**

- The template is `html/template`, parsed by the chassis at startup
  (`appkit/web.Load` over the www root); the service name and version are the only
  dynamic substitutions; everything else is static markup.
- The response sets `Content-Type: text/html; charset=utf-8` (the chassis `Render`
  sets it) and returns `200 OK` for the page.
- The markup links the app's **own** `static/tokens.css` (served from the same www
  root by the chassis static mount) and applies the Carbon type scale (Space
  Grotesk display for the name, IBM Plex Mono for the version label) and neutral
  palette. The asset wiring is D3; this Decision owns that the page is *rendered
  with name + version* and stays a single centered card.

**Why strings, not the chassis.** Passing `service` and `version` as strings
(rather than handing the handler the `*appkit.Router`) keeps the handler trivially
table-testable with fixed values. The composition root is the only place that
reaches into `rt` for the two strings and the loaded site.

**Rejected.**
- **A shared cross-app landing page** — each app's page will diverge to serve that
  app's purpose, so each app owns its page *content*; only the mechanism is shared,
  in appkit.
- **Rendering anything beyond name + version** in v1 (owner email, account
  balances, links) — out of product scope; v1 is exactly name + version.
- **Pulling the values from env or a new config knob** — the chassis already
  exposes the authoritative service name and build version; a second source could
  drift from what `version`/`/health` report.
- **Reading the identity header (`X-Owner-Email`) to personalize the page** — the
  gate is coarse (any logged-in user); v1 shows no per-user content, so the
  handler ignores identity entirely.

**Verification** (over an `appkit/web` Site loaded from the repo-real
`ledger/share/www` tree, driven with httptest — the landing handler of D1/D2).
- R-LAND-3C9D — the landing handler serving a `GET /` returns HTTP `200`.
- R-LAND-5E1F — the rendered body contains the service name passed in (the page
  shows `"ledger"`).
- R-LAND-7G2H — the rendered body contains the version string passed in (e.g. a
  test version like `"9.9.9-test"` appears verbatim), proving the page reflects
  the running version rather than a hardcoded constant.
- R-LAND-9J4K — the landing response sets `Content-Type: text/html; charset=utf-8`.
