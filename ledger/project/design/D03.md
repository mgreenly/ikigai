# Decision 3 — ledger's own Carbon design assets (shipped in `share/www/static`)

**Decision.** ledger ships its **own** copy of the Carbon design assets and serves
them from its own mount; the landing page references those assets, never a
cross-service URL. Concretely:

- A `ledger/share/www/static/` directory holds a **vendored copy** of
  `design/tokens.css` (the Carbon CSS custom properties) and the woff2 fonts the
  Carbon type pairing needs: `space-grotesk.woff2` (display), `ibm-plex-sans.woff2`
  (body), `ibm-plex-mono-400.woff2` + `ibm-plex-mono-500.woff2` (mono labels) —
  the same set every service carries.
- The directory ships in the release `share` tier (`bin/ship` bundles
  `ledger/share/`; the box reads it at `share/current/www`) and is served under the
  stable in-service path — `GET /static/…` — by the **chassis static mount**
  (appkit's `Spec.WWW` integration), with correct content types
  (`text/css; charset=utf-8` for `tokens.css`, `font/woff2` for the fonts). No
  ledger Go code serves assets.
- `tokens.css` declares its `@font-face` rules pointing at the service-relative
  font paths (D8), and the landing template links `tokens.css` from the same
  `/static/` path. No asset is fetched from the dashboard or any other service at
  runtime.

**Why each app ships its own copy.** Each service is a self-contained release: its
assets deploy, swap, and roll back with its binary, and nothing at runtime depends
on another app's asset URLs. `module ledger` cannot reach the sibling `design/`
tree anyway (a module boundary), so vendoring the tokens + fonts into
`ledger/share/www/static/` keeps that independence — the same reasoning that
originally justified per-service embedded copies, now realized as shipped files
(the suite reverted embedding: assets are release artifacts on disk, visible and
diffable in the release tree, not bytes hidden in the binary).

**Carbon fidelity.** The page consumes the **semantic** Carbon tokens
(`--color-*`, `--space-*`, type tokens), not raw ramps, per `design/carbon.md`:
monochrome neutrals; blue `#2563EB` reserved as the only signal color (used here
only for a focus ring / link if present, never decoratively); 4px spacing grid;
tight 4px radius; hairline borders over shadow. The landing card is minimal and
on-system: service name in **Display** (Space Grotesk, 56/1.04/700), version in
the **Mono · Label** token (IBM Plex Mono, 12/1.4/500), centered within the
1120px max content width.

**Staleness note (vendored copy).** Because `tokens.css` and the fonts are a
vendored copy, they can drift from `design/` over time. That is an accepted cost
of the module boundary (the dashboard lives with the same), and is out of scope
to automate here; if the design system changes, the copy is re-vendored as a new
plan phase. v1 copies the current `design/tokens.css` and the four woff2 fonts
verbatim.

**Rejected.**
- **Linking the dashboard's `/static/tokens.css` at runtime** — couples ledger to
  the dashboard's asset URLs and breaks the self-contained one-binary model; a
  dashboard asset rename would silently unstyle ledger.
- **A shared design module imported by every app** — the suite deliberately keeps
  each app a standalone module and a standalone release; vendoring is the
  established pattern.
- **Embedding the assets in the binary (`//go:embed`)** — the suite's original
  approach, deliberately reverted (D10): embedded assets are invisible on the box,
  cost a rebuild for any dev tweak, and re-implement per service what the shipped
  `share` tier plus the chassis static mount do uniformly.
- **Inlining all CSS into the HTML and shipping no fonts** — loses the woff2 type
  pairing (the page would fall back to system fonts and look off-system).
- **Serving the fonts from a CDN** — the box is single-tenant and may run with no
  external egress guarantees; assets must be local.

**Verification** (drive the chassis static mount over the repo-real
`ledger/share/www` tree with httptest).
- R-ASST-3T7V — a `GET /static/tokens.css` is served with status `200` and a CSS
  content type (`text/css`).
- R-ASST-5W9X — the landing template's stylesheet link and the page's font
  references resolve to ledger's **own** `/static/` path (the rendered HTML
  contains the app-local asset path and contains **no** cross-service/dashboard
  asset URL).
- R-ASST-7Y2Z — the woff2 font assets are shipped and served: a `GET` for a font
  path (e.g. `/static/fonts/space-grotesk.woff2`) returns `200` with a `font/woff2`
  content type.
