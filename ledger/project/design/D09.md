# Decision 9 — Adopt `registry`: resolve ledger's own loopback port by name, and guard the port literals

**Problem.** ledger hardcodes its loopback port as the bare integer literal
`3101` in three places: the `appkit.Spec.Port` field in `cmd/ledger/main.go`, the
manifest byte-equality test in `cmd/ledger/main_test.go` (`Port: 3101`), and the
`proxy_pass http://127.0.0.1:3101/…` assertions in `internal/web/nginx_test.go`.
"Which port ledger answers on" is therefore duplicated across the tree and drifts
silently — a renumber only surfaces at deploy, uncatchable by the compiler.
Unlike a consumer (notify), ledger is an event-plane **producer**: it resolves
**no** peer feed URLs, so the only address it owns is its own port. This makes the
adoption smaller than notify's — no peer-feed default resolution — and lets one
Decision carry both the adoption and its guards.

**Decision.** ledger writes no loopback port literal in its Go source: it
references itself **by name** through the shared `registry` library
(`module registry`, the repo-root leaf module). Resolution happens **once, at the
composition root** (`cmd/ledger/main.go`) — never per request — because the table
is compile-time data (if a port ever changes we rebuild the binaries anyway).
This is **behavior-preserving**: `registry`'s table carries ledger's current
value (`ledger=3101`), so every resolved value is byte-identical to the literal it
replaced. Two guards then enforce that the literal stays gone.

1. **Own-port resolution (`cmd/ledger/main.go`).** The `appkit.Spec.Port` value
   `3101` becomes a name lookup — the strict composition-root form that panics on
   an unknown name (a programming error):

   ```go
   appkit.Main(appkit.Spec{
       App:   "ledger",
       Mount: "/srv/ledger/",
       Port:  registry.MustPort("ledger"),
       ...
   })
   ```

   Nothing else in `main.go` changes: the Producer / Handlers hooks, the outbox
   wiring, and the `ManifestExtras` are untouched. ledger resolves, defaults, or
   overrides **no** peer address — it has none.

2. **Module wiring (inside `ledger/` only).** `ledger/go.mod` gains
   `require registry v0.0.0` and the committed `replace registry => ../registry`,
   mirroring the existing `appkit` and `eventplane` in-repo replace-siblings.

3. **Source-scan guard.** A test walks every `*.go` file under the `ledger/`
   module and fails if any file's source contains a bare loopback-port address
   literal (`127.0.0.1:30` followed by two digits). The needle is **assembled at
   runtime** (e.g. `"127.0.0.1:" + "30"`) so the guard's own source can never trip
   it, and it **skips its own filename**. After this Decision there are zero such
   literals in ledger's Go source, so the scan is green; reintroducing a hardcoded
   loopback URL turns it red.

4. **Deploy-artifact drift guards.** The two static deploy artifacts that *must*
   still carry the literal port — `etc/manifest.env` and `etc/nginx.conf` — are
   left with their literal `3101` (nginx cannot call Go; the manifest is generated
   metadata), but the ledger tests that police them derive their expected port
   from `registry`:
   - the manifest byte-equality test (`cmd/ledger/main_test.go`) emits with
     `Port: registry.MustPort("ledger")` instead of `Port: 3101`; the emitted
     `PORT=3101` still byte-matches the committed `etc/manifest.env` today, and a
     future renumber makes `manifest.Emit` produce a different `PORT` line so the
     assertion fails;
   - the nginx fragment tests assert the `proxy_pass` upstream origin against
     `registry.BaseURL("ledger")` (== `"http://127.0.0.1:3101"`) — the exact
     landing root and the `/static/` variant — so a renumber makes the
     still-literal `etc/nginx.conf` disagree with `registry` and the test goes red.
   Both re-pointed tests keep **every** existing behavioral assertion (the session
   gate, the exact-match vs prefix distinction, the manifest's non-port fields);
   only the **port value** they compare against stops being a hardcoded number and
   becomes a `registry` call.

**External precondition (OUT OF SCOPE — not built by any ledger phase).** The
repo-root `go.work` must carry `use ./registry`, and the `registry` module itself
must exist and be green. Both are owned at the repo root / by the `registry`
project and are assumed satisfied. No ledger Decision or phase edits `../go.work`,
`../registry/`, `../bin/`, nginx, or any sibling module — the executor runs from
`ledger/` and cannot reach outside it.

**Rejected.**
- **Splitting adoption and guards into two Decisions (notify's D9/D10 shape).**
  notify split because its D9 *also* moved peer-feed default resolution
  (`CRM_FEED_URL`/`PROMPTS_FEED_URL`) into the chassis and needed a separate
  guard Decision. ledger is a producer with **no** peer-feed literals, so the
  adoption is one `Spec.Port` change plus its two guards — one build-turn, one
  Decision.
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; resolve once at the composition root and reuse.
- **`registry.Port("ledger")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form.
- **An AST/int-literal scan for the bare `3101`.** Too noisy to be falsifiable —
  `3101` carries no context marking it a port. The own-port integer literal is
  eliminated by the `registry.MustPort("ledger")` change and proven gone by the
  manifest drift guard (which emits from `registry` and matches the committed
  file).
- **Regenerating `etc/nginx.conf` / `etc/manifest.env` from `registry`.** Out of
  scope: nginx-config generation and the manifest artifact are owned elsewhere; a
  drift-catching test delivers the safety without a codegen step.
- **Leaving the nginx/manifest tests as literal `3101` assertions.** They would
  keep two bare `30xx` literals in ledger's Go source and, worse, keep *passing*
  after a `registry` renumber — the exact silent drift this adoption removes.

**Verification.**
- R-4VDW-DRQH — **ledger's own listen port is `registry.MustPort("ledger")`**, not
  a `3101` integer literal, at the composition root. The Spec is passed inline to
  `appkit.Main` and is not directly inspectable, so this is proven **concretely by
  the manifest drift guard R-4WLS-RJH6**, which emits with
  `registry.MustPort("ledger")` and byte-matches the committed `PORT=3101`; a
  hardcoded `Port: 3101` in the Spec would leave the emitted port and the resolved
  port able to diverge undetected, which the guard forbids.
- R-4WLS-RJH6 — **the committed `etc/manifest.env` port agrees with `registry`.**
  The manifest byte-equality test emits `manifest.Emit(manifest.Fields{… Port:
  registry.MustPort("ledger") …})` and byte-matches the committed `etc/manifest.env`.
  Falsifiable: a `registry` value differing from the committed `3101` produces a
  different `PORT` line and fails the equality (the intended drift alarm).
- R-4XTP-5B7V — **the committed `etc/nginx.conf` proxy targets agree with
  `registry`.** The nginx fragment tests assert the exact landing location contains
  `"proxy_pass " + registry.BaseURL("ledger") + "/;"` and the static location
  contains `"proxy_pass " + registry.BaseURL("ledger") + "/static/;"`. Falsifiable:
  a `registry` value differing from the committed `3101` makes the still-literal
  fragment disagree and fails the assertions.
- R-4Z1L-J2YK — **no bare `127.0.0.1:30xx` loopback-address string literal exists
  in ledger's Go source.** A guard test walks the module's `*.go` files (skipping
  itself, needle assembled at runtime) and fails on any such literal; it is green
  after this Decision and goes red if a hardcoded loopback URL is reintroduced.
