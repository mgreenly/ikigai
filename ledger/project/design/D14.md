# Decision 14 — `external_ref`: opt-in idempotency for derived transactions

**Problem.** Agents derive journal entries from source documents (a dropbox
bill, a bank feed line, a gmail invoice) and event-driven pipelines re-run:
the same document can reach `record` twice. Today ledger cannot tell a re-run
from a genuinely repeated manual entry — unlimited identical `record` calls
are legal by design (two identical coffee purchases are two facts). The
journal needs a way for a *derived* transaction to name its source document so
a second attempt is rejected, without constraining manual entry at all.

**Decision.** `transactions` gains a nullable `external_ref` TEXT column that
is **unique among non-null values**; the `record` verb (and `reverse`, see
below) gains an optional `external_ref` string parameter. Present → stored on
the transaction and enforced unique forever; absent → exactly today's
behavior (NULL, unlimited identical entries remain legal).

*Schema.* One **new timestamped migration**, minted with
`bin/create-migration ledger external_ref` (never a hand-picked number;
committed migrations `001`–`003` are immutable and untouched):

```sql
ALTER TABLE transactions ADD COLUMN external_ref TEXT NULL;
-- Unique among NON-NULL values only: the partial index is the intent —
-- NULL (no ref) rows are unconstrained; every non-null ref is claimed forever.
CREATE UNIQUE INDEX idx_transactions_external_ref
    ON transactions(external_ref) WHERE external_ref IS NOT NULL;
```

The partial unique index is the *real* enforcement: uniqueness holds even for
a hypothetical writer that bypasses the service. The service's pre-check (next
paragraph) exists only to shape the error.

*Domain seam.* `RecordInput` gains `ExternalRef *string`; `Transaction` gains
`ExternalRef *string`; `Store.InsertTransaction` writes the column and the
scan helpers read it. `Service.Record` (and `Reverse`, when a ref is supplied)
looks the ref up inside the same write tx before inserting
(`SELECT id FROM transactions WHERE external_ref = ?`); a hit aborts with a
new sentinel carrying the existing transaction:

```go
// ErrDuplicateRef: the supplied external_ref is already claimed by an
// existing transaction (wrapped errors name that transaction's id).
ErrDuplicateRef = errors.New("ledger: duplicate ref")
// e.g. fmt.Errorf("%w: external_ref %q already recorded by transaction %s",
//      ErrDuplicateRef, ref, existingID)
```

Because the journal insert, the postings, and the outbox append all ride one
tx, a rejected call leaves **no partial state**: no transaction row, no
posting rows, no event. A constraint violation from the index (the race
backstop under SQLite's single writer) maps to the same sentinel.

*Wire error.* `translateLedgerError` gains one case, in ledger's existing
sentinel→wire vocabulary (`unbalanced`, `bad_root`, `already_reversed`,
`not_found`, `validation`):

```
{"error":{"code":"duplicate_ref","message":"external_ref \"…\" already recorded by transaction <id>"}}
```

The message **names the existing transaction id** so an agent can `get` it and
confirm the earlier recording instead of retrying.

*Validation.* The ref is an **opaque, verbatim string** — stored exactly as
supplied, no trimming, no format validation. The one rejected value is the
**empty string**: `external_ref: ""` fails at the MCP boundary with
`ErrValidation` (`validation` on the wire), consistent with
`CanonicalizeAccount`'s empty-input rejection — an empty ref can only be a
caller bug, and silently treating it as "absent" would hide it. Omitting the
parameter (or JSON `null`) means "no ref".

*Convention (documented, not validated).* Refs are `<source>:<identifier>` —
e.g. `dropbox:/bills/aws/2026-06.pdf@<content_hash>`,
`gmail:<message-id>` — so refs from different connectors cannot collide. This
is a naming convention stated on the tool surface, **never** enforced: any
non-empty string is legal. It is documented in three places: the `record`
tool's `external_ref` schema description, the `reverse` tool's, and a new
static `external_refs` string field on the `describe` payload (the discovery
surface agents read first), beside `unit`/`roots`/`recipes`.

*Read surfaces.* `transactionJSON` (the shared record/reverse/reconcile/`get`
return shape) includes `"external_ref"` when the transaction carries one and
**omits** the key when NULL — the same present-when-set style as
`reverses_id`/`reversed_by_id`. `register` does **not** expose it: its rows
are posting-grained and already omit all transaction-linkage fields
(`reverses_id` included); a consumer holding a register line follows `txn_id`
→ `get`. `balance` aggregates and carries no transaction fields.

*Event payload.* `transactionRecordedPayload` gains
`ExternalRef *string \`json:"external_ref"\`` — always present, `null` when
absent, matching the payload's existing `reverses_id` style — so feed
consumers can dedup/correlate against the source document. The registry
`Sample` gains a filled-in value.

*`reverse` interaction — pinned explicitly (the silent-wrong-implementation
edge).*

- A reversal is a **new** transaction and may carry its **own** optional
  `external_ref` (the `reverse` tool gains the same optional parameter, same
  empty-string rejection, same uniqueness check) — e.g. the credit note that
  triggered the reversal.
- Reversing a transaction does **NOT** free the original's ref. Uniqueness is
  forever: the ref names the source document's transaction, reversed or not.
  After reversing txn A (ref `dropbox:/bills/x.pdf@h`), a new `record` with
  that same ref is still rejected with `duplicate_ref` naming **A**. The
  partial index guarantees this structurally (the reversed row keeps its
  ref); no code path may clear `external_ref`.
- The mirror does **not** inherit the original's ref (it couldn't — the index
  forbids it): `reverses_id` is the correction linkage; `external_ref` is
  source-document identity.

**Rejected.**

- *Treating `""` as absent* — hides a caller bug; an explicit empty ref is
  malformed input, and ledger's style (`CanonicalizeAccount`) rejects empty
  strings rather than coercing them.
- *Idempotent success (returning the existing transaction as if newly
  recorded)* — tempting for pipelines, but it makes a real conflict (two
  *different* documents mapping to one ref) invisible; a typed error naming
  the existing transaction lets the agent decide, and `get` is one call away.
- *Validating the `<source>:<identifier>` format* — the ref is an opaque
  claim check; enforcing a format buys nothing and breaks the first connector
  with an unanticipated identifier shape.
- *Freeing the ref on reversal* — a re-run of the same source document after
  a reversal would silently re-book it; the reversal already corrected the
  books, and re-recording the document is a *new decision* an agent must take
  deliberately (with a suffixed ref if truly intended).
- *A separate `external_refs` mapping table* — a second write target for what
  is one fact about one transaction; the nullable column + partial unique
  index says it in one place with SQLite enforcing it.
- *UNIQUE constraint on the column itself* — SQLite UNIQUE treats NULLs as
  distinct so it would *work*, but the partial index states the intent
  (`WHERE external_ref IS NOT NULL`) explicitly rather than leaning on NULL
  semantics trivia.

**Verification** (real substrates throughout: every claim runs over a real
SQLite database migrated through ledger's full embedded migration set —
`db.FS` via the appkit runner — never a mock; row counts and refs are read
back by SQL).

- R-FP14-UYWQ — `record` with `external_ref` stores the string verbatim on
  the transactions row (read back by SQL) and both the `record` response and
  a subsequent `get` include `"external_ref"` with that value; a transaction
  recorded **without** a ref omits the `"external_ref"` key from both, and
  two consecutive no-ref `record` calls with identical date/description/
  postings **both insert** (transactions row count increases by two —
  baseline unlimited manual entry preserved).
- R-FQ91-8QNF — a second `record` with the **same** `external_ref` (and any
  postings) fails with `ErrDuplicateRef` whose message names the **first**
  transaction's id, and leaves the journal unchanged: transactions row count,
  postings row count, and outbox row count are all identical before and after
  the rejected call (no partial insert, no event).
- R-FRGX-MIE4 — applying the full migration set to a fresh real SQLite
  database yields `transactions.external_ref` nullable TEXT plus the partial
  unique index: by **direct SQL** (bypassing the service), two rows with NULL
  `external_ref` both insert, and a second row with an already-present
  non-null ref fails with a uniqueness constraint error — proving the schema
  itself enforces the claim, not the service pre-check.
- R-FSOU-0A4T — `record` (and `reverse`) with `external_ref: ""` is rejected
  at the MCP boundary with the `validation` wire code and inserts nothing
  (transactions row count unchanged); omitting the parameter is not an error.
- R-FTWQ-E1VI — `reverse` accepts its own optional `external_ref` (stored on
  the mirror, returned in the mirror's shape, uniqueness-checked); and after
  reversing a transaction that carries ref X, a new `record` with ref X is
  **still** rejected with `duplicate_ref` naming the original transaction —
  reversal does not free the ref.
- R-FV4M-RTM7 — the recorded-transaction event payload
  (`transactionRecordedPayload`; its routing key is D15's decision) read back by SQL from the
  real outbox row carries `"external_ref"`: the value when the transaction
  has one, JSON `null` when it does not; a `reverse` mirror's event carries
  the **mirror's own** ref (or null), never the original's.
- R-FWCJ-5LCW — through the assembled MCP handler: the `record` and `reverse`
  input schemas advertise optional `external_ref` whose description states
  the `<source>:<identifier>` convention; the `describe` payload carries the
  static `external_refs` convention field; and a duplicate-ref `record`
  driven through `tools/call` returns the `duplicate_ref` error envelope
  naming the existing transaction id.
