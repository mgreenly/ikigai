# Decision 16 — Structured MCP adoption: `structuredContent`, output schemas, closed error vocabulary

*Cites the suite contract `docs/structured-mcp-design.md` (the single verb
surface for agents and machines). This Decision states ledger's conformance to
it; it does not restate the contract.*

**Problem.** ledger's domain tools were written against the pre-revision
`appkit/mcp` API: every success result is built by `appkitmcp.JSONResult(v)` (a
Go value marshalled to a **text-only** content block, no `structuredContent`),
and every error is built by `appkitmcp.ErrorResult(translateLedgerError(err))`
where `translateLedgerError` returns a **hand-built JSON string** carrying a
ledger-private error vocabulary (`unbalanced`, `bad_root`, `already_reversed`,
`duplicate_ref`, plus `not_found`/`validation`/`internal`) escaped through a
local `jsonEscape` helper. The chassis has since cut over (appkit phases 12–14):
`JSONResult` is **deleted**; `StructuredResult(v any) (map[string]any, error)`
emits `structuredContent` **plus** a mirrored text block from one typed value;
`ErrorResult(code ErrorCode, msg string)` takes a **typed code from a closed
vocabulary** (`ErrValidation`, `ErrNotFound`, `ErrConflict`, `ErrTooLarge`,
`ErrSourceUnavailable`, `ErrInternal`); and `Tool.OutputSchema map[string]any`
is surfaced by `tools/list` when non-nil. ledger does not compile against the
new appkit, and its results carry no machine channel. This Decision conforms
ledger's `internal/mcp` surface, **preserving the JSON each tool already emits
verbatim** — the machine rendering is formalized, never reshaped.

**Decision.**

*1. Success results swap `JSONResult` → `StructuredResult`.* Each of the six
domain **result** verbs returns `appkitmcp.StructuredResult(v)` over the exact
value it marshalled before, so its `structuredContent` is that JSON object and
the mirrored text block carries the identical JSON. `StructuredResult` now
returns `(map[string]any, error)`; the domain handlers already return
`(map[string]any, error)`, so each site becomes a direct
`return appkitmcp.StructuredResult(v)` — the marshal error is **propagated, not
swallowed** (it surfaces as the chassis JSON-RPC error, the same class the
transport uses for a handler-returned error). The emitted objects are unchanged:

- `record`, `reverse`, `get` → `transactionJSON(t)` (the shared transaction
  shape).
- `reconcile` → `{"transactions": [transactionJSON(t), …]}`.
- `balance` → `{"lines": [{"account","amount_cents"}], "total", "unit"}`.
- `register` → `{"lines": [{"txn_id","date","description","posting_id",
  "account","amount_cents","status","running_total"}], "unit"}`.

*2. Each domain result verb declares an `outputSchema`.* A hand-authored
`map[string]any` JSON-schema literal, in the house style of `appkit/mcp`'s
`objectSchema` helper, is attached to each of the six verbs' `mcp.Tool`,
mirroring the JSON that verb already emits (`type`/`properties`/`required`).
`amount_cents`, `total`, and `running_total` are `{"type":"integer"}` (signed
minor USD cents). `record`/`reverse`/`get` reuse one shared
`transactionSchema()` literal (a small private helper), `reconcile` wraps it in
`{"transactions": array(transactionSchema())}`. The transaction schema's
`required` names the always-present keys (`id`, `date`, `description`,
`created_at`, `postings`); the present-when-set keys (`reverses_id`,
`reversed_by_id`, `external_ref`) are declared as properties but **not**
required, matching `transactionJSON`'s omit-when-absent behavior. appkit does
not validate results against the schema at runtime (a suite non-goal); the
schema is the declared machine contract and is proven by the test suite.

*3. Errors carry a typed code from the closed vocabulary.* `translateLedgerError`
is re-signed from `func(error) string` to `func(error) map[string]any`: it maps
each ledger domain sentinel to `appkitmcp.ErrorResult(code, msg)` and returns
the finished result map, so each call site drops the outer
`appkitmcp.ErrorResult(…)` wrapper (becomes `return translateLedgerError(err),
nil`). `jsonEscape` and the hand-built `{"error":{"code":…}}` string envelopes
are **deleted** — `ErrorResult` builds the structured `{code, message}` itself.
The sentinel → closed-code mapping, stated explicitly (the ledger-private code
words `unbalanced`/`bad_root`/`already_reversed`/`duplicate_ref` are **retired**;
their distinguishing detail moves into the human `message`):

| ledger sentinel | closed code | message |
|---|---|---|
| `ErrUnbalanced` | `validation` | the sentinel's wrapped text (postings do not sum to zero) |
| `ErrBadRoot` | `validation` | fixed: names the five roots and points at `describe` |
| `ErrValidation` | `validation` | the sentinel's wrapped text (bad date, unknown status, too few postings, multiple elisions, empty `external_ref`, malformed period) |
| `ErrNotFound` | `not_found` | the sentinel's wrapped text (names the missing id) |
| `ErrAlreadyReversed` | `conflict` | fixed: directs the agent to reverse the mirror instead |
| `ErrDuplicateRef` | `conflict` | the sentinel's wrapped text (names the existing transaction's id) |
| *(unmapped)* | `internal` | fixed: `internal error` |

Both `ErrUnbalanced` and `ErrBadRoot` collapse to `validation` (both are
malformed-input failures); both `ErrAlreadyReversed` and `ErrDuplicateRef`
collapse to `conflict` (both are "the requested write conflicts with the
journal's immutable state"). Machine callers branch on the code; the specific
domain condition remains legible in the message. This mapping is the current
statement of ledger's wire error shape — D14 (`ErrDuplicateRef`) and D11 (which
described the pre-adoption hand-built envelope) defer to it.

*4. `describe` is the sole prose exception.* `describe` is a **discovery /
documentation** tool — the first call an agent makes, teaching the account
model, roots, reconciliation states, recipes, and the live chart; the suite
contract names `describe` a documentation tool that "remain[s] plain text …
declare[s] no output schema." Machines never parse `describe` (they read tool
schemas from `tools/list`). It therefore keeps its content **verbatim** but does
**not** go through `StructuredResult`: it returns
`appkitmcp.TextResult(string(json.Marshal(payload)))` — the same marshalled
discovery payload in the text block, with **no** `structuredContent` and **no**
declared `outputSchema`. This is byte-preserving (identical to what `JSONResult`
put in the text block) and is the classification, not a reshape. The chassis
`health` and `reflection` tools are chassis-owned and already structured
(appkit) — ledger declares nothing for them.

The seven-verb domain contract, the immutable journal, account canonicalization,
period expansion, and every emitted JSON object are otherwise untouched; this
Decision changes only the transport-envelope construction (`StructuredResult` vs
`JSONResult`, typed vs hand-built errors) and adds the declared schemas.

*Guard:* ledger owns **no** hand-copied loopback guard predicate — its `/feed`
is mounted by the chassis (`Spec.Feed`), already wrapped by appkit's shared
loopback guard. The suite contract's guard swap is a no-op here; this Decision
introduces no guard change.

**Rejected.**

- **Keeping the hand-built `{"error":{"code":"unbalanced",…}}` string
  envelopes** — a ledger-private error vocabulary that machine callers cannot
  branch on portably; the suite mandates the shared closed vocabulary so a
  script sees the same six codes from every service.
- **A distinct closed code per ledger sentinel** — the closed vocabulary is
  deliberately small; `unbalanced` and `bad_root` are both `validation`,
  `already_reversed` and `duplicate_ref` are both `conflict`. Forcing a wider
  vocabulary to preserve the old code words would fork the suite contract.
- **Making `describe` structured (StructuredResult + `outputSchema`)** — it is a
  documentation surface for readers, explicitly a prose exception in the suite
  contract; declaring a schema for a discovery blob that machines never parse
  adds a contract to maintain for no consumer.
- **Reshaping any success payload while adding its schema** — the schema
  formalizes the existing JSON; changing a field to "improve" it would break
  the very agents and (future) machine callers this formalizes.
- **Swallowing `StructuredResult`'s marshal error** (e.g. `res, _ :=`) — hides a
  serialization bug behind an empty/partial result; the error is propagated.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a migrated in-memory `ledger.Service`, driven via its
real `ServeHTTP` `tools/list`/`tools/call` seam; the same harness D11/D14 use).

- R-9FRN-SGDT — through the assembled handler, `tools/list` declares a **non-nil
  `outputSchema`** for each of the six domain result verbs (`record`, `reverse`,
  `reconcile`, `balance`, `register`, `get`) and **no** `outputSchema` for
  `describe` (the prose exception, named and excluded) — table-driven over the
  returned descriptors. A verb missing its schema, or `describe` carrying one,
  fails.
- R-9GZK-684I — a successful `record`, `reverse`, and `get` each return a result
  whose `structuredContent` is the full transaction object (`id`, `date`,
  `description`, `created_at`, and `postings[]` each with `account`,
  `amount_cents` as a JSON **number**, `status`, `ord`, `id`) **and** whose
  single text block parses to JSON identical to that `structuredContent`
  (mirror). Falsifiable: the pre-adoption `JSONResult` results carried **no**
  `structuredContent` key.
- R-9I7G-JZV7 — a successful `balance` result's `structuredContent` equals
  `{"lines":[{"account","amount_cents"}],"total","unit"}` with `amount_cents`
  and `total` as JSON **numbers** (not strings), mirrored verbatim in the text
  block.
- R-9JFC-XRLW — a successful `register` result's `structuredContent` equals
  `{"lines":[{"txn_id","date","description","posting_id","account",
  "amount_cents","status","running_total"}],"unit"}` with `amount_cents` and
  `running_total` as JSON **numbers**, mirrored verbatim in the text block.
- R-9KN9-BJCL — a successful `reconcile` result's `structuredContent` equals
  `{"transactions":[…]}` where each element is the full transaction object,
  mirrored verbatim in the text block.
- R-9LV5-PB3A — a `describe` `tools/call` result carries the marshalled
  discovery payload in its text block and **no** `structuredContent` key —
  proving it did not go through `StructuredResult` (the prose-exception
  behavior). Presence of `structuredContent` on `describe` fails.
- R-9N32-32TZ — a `record` whose explicit postings do not sum to zero returns a
  tool error with `isError:true` and `structuredContent.code == "validation"`
  (the retired `unbalanced` code appears nowhere in the result).
- R-9OAY-GUKO — a `record` posting to an account whose root is not one of the
  five types returns `isError:true`, `structuredContent.code == "validation"`,
  and a human message that names the five roots / points at `describe` (the
  retired `bad_root` code appears nowhere).
- R-9PIU-UMBD — a `get` for an unknown transaction id returns `isError:true`
  and `structuredContent.code == "not_found"`.
- R-9QQR-8E22 — a `reverse` of an already-reversed transaction returns
  `isError:true` and `structuredContent.code == "conflict"` (the retired
  `already_reversed` code appears nowhere), the message directing the agent to
  the mirror.
- R-9RYN-M5SR — a `record` (or `reverse`) reusing an `external_ref` already
  claimed returns `isError:true`, `structuredContent.code == "conflict"` (the
  retired `duplicate_ref` code appears nowhere), and a human message naming the
  **existing** transaction's id.
- R-9T6J-ZXJG — each MCP-boundary input rejection — fewer than two postings,
  more than one elided amount, a malformed `YYYY-MM-DD` date, an unknown
  reconciliation status, an empty `external_ref`, a malformed `period` — returns
  `isError:true` with `structuredContent.code == "validation"` (table-driven
  over the rejections).
- R-9UEG-DPA5 — closed-vocabulary invariant: across every domain tool error
  produced by the surface, `structuredContent.code` is always one of the closed
  set (`validation`, `not_found`, `conflict`, `too_large`,
  `source_unavailable`, `internal`), `isError` is `true`, and the human message
  is non-empty; no result contains a hand-built `{"error":{"code":…}}` envelope
  or any retired ledger-private code word (`unbalanced`, `bad_root`,
  `already_reversed`, `duplicate_ref`).
