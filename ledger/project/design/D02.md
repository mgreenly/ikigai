# Decision 2 — Route wiring: `GET /{$}` mounted ungated through `Spec.Handlers`

**Decision.** The landing handler is mounted at the service's **exact root path**
`/` through ledger's existing `appkit.Spec.Handlers` hook, beside the `POST /mcp`
mount, using the Go 1.22+ exact-match pattern `GET /{$}`:

```go
Handlers: func(rt *appkit.Router) error {
    conn := rt.DB()
    if conn == nil {
        return fmt.Errorf("ledger: no DB handle on router")
    }
    svc = ledger.NewService(conn) // outbox injected by the Producer hook after Handlers

    // Landing page — human web UI. Ungated IN-PROCESS: nginx is the gate
    // (auth_request /_session-authn, see D4). `{$}` matches ONLY the exact root
    // path, so it never shadows /mcp, /health, /feed, or the PRM well-known. The
    // page renders through the chassis-loaded site (D1/D10); the static route is
    // chassis-mounted (Spec.WWW), not wired here.
    rt.Handle("GET /{$}", landingHandler(rt.WWW(), rt.Service(), rt.Version()))

    h, err := mcp.NewHandler(svc, rt) // D11 — appkit/mcp handler over the domain Service
    if err != nil {
        return err
    }
    rt.Handle("POST /mcp", rt.RequireIdentity(h))
    return nil
},
```

**Why ungated in-process.** ledger runs no token logic; nginx is the sole trust
boundary (Conventions). The landing route is mounted **without** an in-process
identity wrapper — exactly as `POST /mcp` delegates its bearer gate to nginx's
`auth_request /_authn`, the landing route delegates its cookie gate to nginx's
`auth_request /_session-authn` (D4). Wrapping it in `rt.RequireIdentity` would be
wrong: that expects the bearer-injected `X-Owner-Email`/`X-Client-Id` headers,
which a browser session request does not necessarily carry the same way, and the
page is meant to be coarse, not per-resource.

**Why `{$}` and not `/`.** In the Go 1.22+ `http.ServeMux` pattern language, a
trailing `/` is a **subtree** pattern that matches every path under it, while
`{$}` matches **only** the exact path. ledger's internal routes are `/`,
`/static/` (chassis-mounted, D10), `/mcp`, `/health`, `/feed`, and
`/.well-known/oauth-protected-resource` (nginx has already stripped the
`/srv/ledger/` mount prefix). Mounting the landing page at `GET /{$}` guarantees it
answers the bare root **and nothing else** — it cannot shadow `/mcp` or any other
route. A subtree `/` would swallow unmatched paths and is explicitly rejected.

**Composition-root growth, not a rewrite.** `cmd/ledger/main.go` already carries
the `Handlers` hook (it builds the Service and mounts `POST /mcp`); this Decision
owns the landing line of that hook. The landing page renders through the
chassis-loaded site (`rt.WWW()`, D10) and the MCP handler is the `appkit/mcp`
handler (D11); no service-side `GET /static/` mount is wired here — the chassis
owns it. That is the sanctioned incremental growth of the one shared wiring file,
not a rewrite of ledger's domain.

**Rejected.**
- **A subtree `GET /`** — swallows every unmatched path (would intercept a typo'd
  `/mcpx`, future routes, etc.) and muddies the "exact root only" contract. `{$}`
  is precise.
- **Wrapping the route in `rt.RequireIdentity`** — double-gates with the wrong
  (bearer) mechanism and breaks the coarse cookie-gated human door; the gate is
  nginx's job (D4).
- **A second `appkit.Spec` field or a new hook for web pages** — unnecessary; the
  existing `Handlers` hook mounts arbitrary routes, and the landing page is just
  another route on the same mux.
- **Mounting the page on a sub-path** (e.g. `/landing`) — the product constant is
  that the human reaches it at the mount **root** (`/srv/ledger/`), which is the
  service's `/`.

**Verification.**
- R-ROUT-2M6N — with the landing handler registered on an `http.ServeMux` at
  `GET /{$}` (as the composition root wires it), a `GET /` is dispatched to the
  landing handler and returns the landing page (`200`, name+version body).
- R-ROUT-4P8Q — the `{$}` exact-root registration does **not** shadow a sibling
  `POST /mcp` registration on the same mux: a `POST /mcp` still dispatches to the
  mcp handler, not the landing handler.
- R-ROUT-6R1S — a non-root path that is not otherwise registered (e.g. `GET /nope`)
  is **not** captured by the `{$}` landing pattern (it does not return the landing
  page) — proving `{$}` is exact-match, not a subtree.
