# Decision 10 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** ledger's landing page and static assets are embedded in the binary
(`//go:embed` in `ledger/internal/web/`: `landing.html`, `static/tokens.css`, the
four woff2 fonts), and a single `web.LandingHandler(service, version)` both
renders the page and file-serves `/static/…` from the embedded `staticFS`. That
is the suite's original approach, now being reverted: web assets are release
artifacts that belong on disk — visible and diffable in the versioned release
tree, editable in dev without a rebuild, served by one chassis mechanism instead
of twelve drifting copies. crm proved the conversion; notify is the fuller
reference. This Decision is the service-side half of appkit's chassis design (its
D5 www-root resolution, D6 `appkit/web`, D7 `Spec.WWW`/`Router.WWW()` — consumed
here as fixed external contracts).

**Decision.** ledger serves its web surface from the on-disk www root.

1. **The files move (same bytes).** `ledger/internal/web/landing.html` →
   `ledger/share/www/landing.html`; `ledger/internal/web/static/` →
   `ledger/share/www/static/` (`tokens.css` and the four woff2 fonts,
   byte-for-byte). `bin/ship` already bundles `ledger/share/` into the release
   `share/<version>` tier; on the box the chassis resolves the root to
   `share/current/www`, in dev to `./share/www` or the `LEDGER_WWW_PATH` override.
2. **The Spec opts in.** `cmd/ledger/main.go` sets `WWW: true`. The chassis loads
   the site at serve (failing startup loudly if the root is missing), auto-mounts
   `GET /static/`, and exposes the loaded site as `rt.WWW()`. ledger's other Spec
   fields (the `Feed: "/feed"` producer role, `Migrations`, `Events`,
   `ManifestExtras`, the Producer hook) are untouched.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` becomes a few-line handler over `rt.WWW()` (D1/D2 carry the exact
   shape), rendering `landing.html` with the injected service and version. The
   two service-side static mounts — `rt.Handle("GET /static/{file...}", landing)`
   **and** the `/static/` branch inside `LandingHandler` — are **deleted**; the
   chassis owns `GET /static/`.
4. **`ledger/internal/web/` is deleted** — the `LandingHandler`, the `embed.go`
   shim, `landing.html`, the `static/` tree, and the `web`-package tests
   (`web_test.go`, `nginx_test.go`). The landing/asset tests move to the
   composition-root package (`cmd/ledger`), loading the repo-real `share/www` tree
   via `appkit/web` (a relative path from the package dir), so they exercise the
   exact files that ship. The nginx fragment tests (`nginx_test.go`) relocate to
   `cmd/ledger` with their assertions intact, carrying the registry-derived
   proxy-target expectations D9 already introduced.
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_ledger` exports
   `LEDGER_WWW_PATH="$repo/ledger/share/www"` beside its existing exports, so a
   repo-root-launched dev binary finds the source tree. This one line crosses the
   `ledger/` boundary and is verified by the live `bin/start` smoke, **not** the Go
   suite.
6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6/D7/D8), the MCP surface, the outbox producer, and the migrations are
   untouched. The embedded migrations (`internal/db`) stay embedded — they are not
   web assets (their de-shim is D12).

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work completes; embedded
  assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's nginx-served user
  content, backed up with `state/`; ledger's UI assets are release artifacts and
  live in the shipped `share` tier (appkit D5 records the full argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition** — two sources of
  truth; the conversion is atomic within one service.

**Verification** (over the repo-real `ledger/share/www` tree; D1's R-LAND-\*, D2's
R-ROUT-\*, D3's R-ASST-\*, D7's R-HOME-4M6R, and D8's R-7\*-\* ids continue to pin
the page/asset behaviors through the new substrate, relocated to `cmd/ledger`).
- R-509H-WUP9 — an `appkit/web` Site loaded from `ledger/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html` with
  an injected service and version visible in the output — proving the shipped
  template, not an embedded copy, is the page source.
- R-51HE-AMFY — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  ledger-side static handler registered (`ledger/internal/web/` is gone and no
  `GET /static/` mount remains in `cmd/ledger`) — proving the assets ship on disk
  and the chassis serves them.
