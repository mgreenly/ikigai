# Decision 12 — Delete the chassis shims (`internal/db` wrappers) and true up the doctrine doc

**Decision.** Two cleanups that land the reference shape whole, mirroring crm's
and notify's shim deletion:

1. **`internal/db` shrinks to its legitimate content.** The `Open` and `Migrate`
   wrapper functions in `ledger/internal/db/db.go` are deleted — appkit owns DB
   opening (`appkit/db.Open`, the WAL/FK/single-writer pragmas) and the migration
   run (`appkit/db.LoadMigrations` + `Migrate`, driven at serve/migrate through
   `Spec.Migrations: db.FS`). What remains is what genuinely must live in a ledger
   package:
   - the embedded migration set (`//go:embed migrations/*.sql`; `go:embed` cannot
     cross modules) and its `FS` export handed to `Spec.Migrations`;
   - the app-side guard tests: `migrations_load_test.go` (ledger's real set loads
     through `appkit/db.LoadMigrations`) and `migrations_outbox_test.go` (the
     outbox-DDL drift guard asserting the **newest** outbox migration contains
     `outbox.SchemaSQL` verbatim — the event-plane invariant that every
     producer's outbox DDL is identical; since D15 that newest migration is the
     timestamped `outbox_routing` one, and `003_outbox.sql` is the frozen
     pre-revision bootstrap the guard no longer targets). Both are genuinely
     ledger's, not appkit's.
   - **Test harnesses that called the wrappers call appkit's db package directly.**
     Three files build a schema through `db.Migrate` today and swap only that
     plumbing — **no test assertion changes**:
     - `internal/db/db_test.go` (`TestOpenAndMigrate`, `TestMigrate_IsIdempotent`)
       — `Open` → `appkitdb.Open`, `Migrate` → a local two-line helper over
       `appkitdb.LoadMigrations(FS, "migrations")` + `appkitdb.Migrate` (the notify
       `migrateNotify` pattern). Their assertions (a migration applied; re-apply is
       a no-op — the app-side guard that ledger's set declares each version once)
       are unchanged.
     - `internal/ledger/ledger_test.go` (the `openDB` helper) and
       `internal/mcp/tools_test.go` (its handler-setup helper) — the
       `db.Migrate(ctx, conn)` call becomes the same `appkitdb.LoadMigrations` +
       `appkitdb.Migrate` two lines over `db.FS`. Every domain and tool assertion
       is untouched.

2. **`ledger/AGENTS.md` (and its `CLAUDE.md` symlink — one file, edit
   `AGENTS.md`) states the converted truth**, per the suite doctrine (docs state
   current truth; purge what the change falsified; no archaeology). The falsified
   claims to purge and replace:
   - **`internal/mcp` as "JSON-RPC 2.0 MCP transport … `mcp.go` is the transport,
     unchanged"** → `internal/mcp` is the **seven-domain-tool declaration**
     (`Instructions` + `Tools(svc)` + `NewHandler`) over the `appkit/mcp` chassis
     transport; the transport and the `health`/`reflection` tools are chassis-owned.
   - **`internal/db` as "SQLite open (WAL, FK, single-writer) + embedded migration
     runner"** → `internal/db` holds only the embedded migration set (`FS`) and the
     load + outbox-DDL drift guards; SQLite open and the migration runner
     are `appkit/db` (`Spec.Migrations`).
   - **The stale layout entries `internal/server` and `internal/logging`** — these
     packages do **not** exist in the ledger tree (routing, the PRM document, the
     identity gate, `/feed`, security headers, graceful shutdown, and structured
     slog are all appkit's); purge both entries. (`internal/ids` stays — it exists.)
   - **The web surface** — the landing page is served from `share/www` through
     `Spec.WWW` and `rt.WWW()` (the chassis static mount serves `/static/`), not an
     embedded `internal/web` handler; state this and add a `share/www` layout entry.
   - **The "eight fixed verbs" framing** (which counts `health` as a ledger verb
     and omits `reflection`) → the **domain** surface is **seven** verbs (`record`,
     `reverse`, `reconcile`, `balance`, `register`, `get`, `describe`); `health`
     and `reflection` are the chassis-supplied standard tools, for **nine** tools
     on the wire.
   - **The manifest/port story** — note that ledger resolves its own port via
     `registry.MustPort("ledger")` (the manifest still emits the literal
     `PORT=3101`, guarded against drift by the registry-derived test).

**Rejected.**
- **Keeping the wrappers for import brevity** — a per-service shim that can drift,
  for two saved lines per call site; crm's and notify's shim deletions already
  made this call.
- **Deleting `internal/db` entirely** — the migrations embed must live in a ledger
  package and the outbox-DDL drift guard (the newest outbox migration contains
  `outbox.SchemaSQL` verbatim; D15)
  is genuinely app-side; the package shrinks to its legitimate content instead.
- **Leaving `AGENTS.md` for a later pass** — the file would assert wiring (a local
  JSON-RPC transport, a DB open/runner, non-existent `internal/server`/
  `internal/logging` packages, an embedded landing page) that no longer exists in
  the same tree that changed it; docs move with the code.

**Verification.** Structural — no `R-XXXX-XXXX` ids (shim deletion + doc truth; the
behaviors are already pinned by the existing domain/MCP/migration suites and the
D9–D11 ids). Realized by a plan phase whose done bar is:

- `grep -n "func Open\|func Migrate" ledger/internal/db/db.go` returns no matches,
  while `grep -n "go:embed migrations" ledger/internal/db/db.go` still matches and
  both `ledger/internal/db/migrations_load_test.go` and
  `ledger/internal/db/migrations_outbox_test.go` still exist.
- `grep -n "internal/server\|internal/logging\|JSON-RPC" ledger/AGENTS.md` returns
  no matches, and `grep -c "share/www\|Spec.WWW\|appkit/mcp" ledger/AGENTS.md` is
  ≥ 1.
- The suite stays green (design Conventions commands) with no assertion changes.
