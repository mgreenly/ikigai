# Decision 10 — init-box installs the box-baseline PDF tooling (poppler-utils)

## Decision

The suite's content-plane design (`docs/content-plane-design.md`) settled that a
sandboxed prompts run gets its PDF tooling **from the box**: the sandbox can rely
on `pdftotext`, `pdftoppm`, and `pdfinfo` being present without any per-run or
per-service install. On AL2023 those binaries ship in one package,
**`poppler-utils`**, installed with `dnf`.

That makes the tooling **box-global substrate** — exactly the class of thing
`opsctl init-box` already owns. init-box today installs its substrate packages as
one call through the existing `System` seam
(`internal/opsctl/initbox.go`, step 1):

```go
if err := o.System.InstallPackages(ctx, "nginx", "certbot"); err != nil { … }
```

This Decision extends that same call by one package — no new seam, no new verb,
no new option:

```go
// 1. Packages: nginx + certbot (front door) and poppler-utils (the box-baseline
//    PDF tooling — pdftotext/pdftoppm/pdfinfo — that sandboxed prompts runs
//    rely on per the content-plane design).
if err := o.System.InstallPackages(ctx, "nginx", "certbot", "poppler-utils"); err != nil {
    return fmt.Errorf("init-box: install packages: %w", err)
}
```

The install remains step 1 of `InitBox`, before the cert/nginx branch, so it runs
on the `--skip-cert` path too. **Idempotency is unchanged**: the `System` seam's
contract (`seam.go`) is `dnf install -y <pkgs…>`, which is a no-op for
already-present packages, and init-box is documented and tested as safe to
re-run — adding a package to the list preserves that property.

## Rejected

- **Per-service install via `opsctl setup prompts --packages poppler-utils`.**
  The mechanism exists (`SetupOptions.Packages`, an operator flag), but it makes
  the tooling's presence depend on the operator remembering a flag on one
  service's setup invocation — a convention, not a provisioning guarantee. The
  content-plane design assigns the tooling to the **box**, not to the prompts
  service; the box-global one-time verb is init-box, exactly as with
  nginx/certbot. (`--packages` remains the right home for deps a single service
  privately owns, like scripts' python3.11.)
- **A named package-list constant or a new "system packages" seam.** init-box
  passes its package literals inline at the single call site; introducing a
  constant or a dedicated step for a one-package addition restructures more than
  it adds. If the box-baseline list grows further, factoring it out is a future
  Decision.
- **Installing inside the prompts sandbox at run time.** Slow (per-run dnf),
  requires network and root inside a sandbox that deliberately has neither, and
  contradicts the content-plane decision that the box provides the tooling.

## Verification

- R-WHC0-I9HL — `InitBox` requests the package install **including
  `poppler-utils`**: through the fake `System`, the recorded ops contain
  `install-packages:nginx,certbot,poppler-utils` (including on the `--skip-cert`
  path). Falsifiable: today's `initbox.go` records
  `install-packages:nginx,certbot`, so a wrong (unchanged) implementation fails
  this.
- R-WIJW-W18A — **Real-substrate (live box `int.ikigenba.com`).** After
  `opsctl init-box` on the box, `pdftotext`, `pdftoppm`, and `pdfinfo` resolve
  and run (e.g. `command -v pdftotext pdftoppm pdfinfo` succeeds and
  `pdftotext -v` exits cleanly), and re-running init-box succeeds with the
  package already present. That AL2023's dnf actually provides these binaries
  under the name `poppler-utils` cannot be falsified by the fake `System` (which
  accepts any package name); it requires the real package manager. The
  observable proof is the runnable binaries, not merely that an install was
  requested.
