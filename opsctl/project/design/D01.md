# Decision 1 — Restore reconstructs `cache/` owned by the service user

## Decision

`opsctl restore <app>` destroys and rebuilds the on-box app tree. Its filesystem
contract is: wipe `state/`, wipe `cache/`, remove `.generation` sidecars, recreate
`AppDir`, untar the archived `state/` back into place. The archived `state/`
returns with its `<app>:<app>` ownership intact (the snapshot tar is created and
extracted as root, preserving owners).

This Decision adds one clause to that contract: **after `state/` is replaced,
restore recreates `cache/` as an empty directory owned by `<app>:<app>` (mode
0755).** Without it, `cache/` is left absent; on restart the unprivileged service
user cannot `mkdir cache/` under the root-owned `AppDir` and the unit crash-loops
(the failure that took down `ledger`). `cache/` is regenerable and is **not**
restored from a backup — snapshots archive only `state/` — so restore recreates it
*empty* and lets the service rebuild its contents on first run.

Cache reconstruction is **privileged**: setting `<app>:<app>` ownership requires
root, via the existing `System` seam already used to chown `state/` at setup
(`internal/opsctl/setup.go` `System.ChownTree(ctx, app, app, …)`). The current
filesystem step `replaceStateFromArchive(ctx, l Layout, archive string)` is a free
function with only a `Layout` and no access to `System`; satisfying this Decision
therefore moves the cache recreate+chown step onto the `Restore` method (which
holds the `Opsctl` receiver and its `System`) or threads `System` into the step.
Illustrative shape:

```go
// after replaceStateFromArchive succeeds, within (*Opsctl).Restore:
l := o.layout(app)
if err := os.MkdirAll(l.CacheDir(), 0o755); err != nil {
    return fmt.Errorf("restore: recreate cache: %w", err)
}
if err := o.System.ChownTree(ctx, app, app, l.CacheDir()); err != nil {
    return fmt.Errorf("restore: chown cache: %w", err)
}
```

The recreate+chown happens after `state/` is in place and before the deferred unit
restart, so the service user owns a writable `cache/` the moment it starts.

## Rejected

- **Fix only in `setup.go` (chown `cache/` at provision time).** Insufficient for
  the restore path: restore wipes `cache/` *after* setup ran, so a setup-time
  chown is already gone by the time the unit restarts. Setup-side `cache/`
  ownership is a separate, additive concern for the non-restore (fresh-provision)
  path and is out of scope for this Decision.
- **Leave `cache/` absent and let the service create it.** This is the current
  behavior and the defect: the service runs unprivileged under a root-owned
  `AppDir` and cannot create the directory — it crash-loops.
- **Restore `cache/` from the backup.** Snapshots archive only `state/`; `cache/`
  is regenerable derived data. Archiving and restoring it would bloat snapshots
  and risk shipping stale cache state. Recreating it empty is correct.

## Verification

- R-WP3M-PO1V — After `opsctl restore <app>` completes, `cache/` **exists** as an
  empty directory (it is not absent). Falsifiable: the current implementation
  leaves `cache/` deleted, so a wrong implementation fails this.
- R-WQBJ-3FSK — The recreated `cache/` is owned by **`<app>:<app>`, not `root`**
  (verified through the fake `System` seam: restore issues a `ChownTree` for
  `CacheDir()` to owner/group `<app>`). Pins the discriminating property — a
  root-owned `cache/` must fail this; "directory exists" alone does not satisfy
  it.
- R-WRJF-H7J9 — **Real-substrate (live box `int.ikigenba.com`).** After
  `opsctl restore <app>` on the box, the unit returns to `active` and the loopback
  `/health` endpoint responds 200, demonstrating the real service user can write
  `cache/` on restart. This claim — that `<app>:<app>` ownership actually lets the
  live service write — cannot be falsified by the fake `System` (which accepts any
  chown); it requires a real restore + unit restart on the box. The observable
  proof is the running, health-200 service, not merely that a chown was issued.
