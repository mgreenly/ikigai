# Decision 7 — setup provisions the served `www` tree owned by the service user

## Decision

A service that serves files off disk (today only `sites`, selected by
`WWWDirsFor` returning a non-empty dir list) gets a `state/www` tree with exactly
two tier subdirs — `public/` and `private/`. A site's files live **directly**
under its tier (`public/<name>/…`, `private/<name>/…`): there is **no** `working/`
staging tree and no publish-symlink indirection.

**The tree is owned by the service user (`<app>:<app>`), mode `0750`, and nothing
else.** No `web` group, no setgid, no world-readable bits. This is the whole
invariant, and it is the *ordinary* `state/` ownership — `state/www` needs no
special treatment beyond being owned by the app that reads and writes it.

The reason there is nothing special to do: **nginx never reads `state/www` off
disk.** The sites service serves each tier through its own loopback process, and
the front door **proxies** to it (`proxy_pass http://127.0.0.1:3004/public/` and
`…/private/`) rather than `alias`ing the directory. Only the `sites` process
(running as `sites:sites`) touches the tree, so app-user ownership at `0750` is
exactly right: the process reads and writes it, and — because the bytes are not
`other`-readable — the private tier stays off every other local user, with the
front door's session gate the sole public path to it. There is no nginx worker
that must traverse the tree, hence no shared group and no setgid to keep such a
group sticky.

Consequently there is **no dedicated `ensureWWWPerms` helper and no `Chmod`
seam**: the served tree's correctness is just its owner, which each path already
establishes with the ordinary recursive `ChownTree(app, app, …)`:

- **setup** (`setup.go`, served-tree branch, `len(opts.WWWDirs) > 0`) creates the
  `public/` and `private/` dirs `0750`, then hands the state tree to the app user
  with `ChownTree(app, app, l.StateDir())` — the same call the worker branch
  already makes, now applied to the served-tree branch too, covering `state/www`.
- **deploy** re-establishes it via its existing state-ownership sweep (D8).
- **restore** re-establishes it after replacing state (D9).

`WWWDirsFor(root, app)` still returns the dirs setup creates for `sites`
(`{WWWDir, WWWPublicDir, WWWPrivateDir}`, parent first), and `layout.go` exposes
`WWWDir`/`WWWPublicDir`/`WWWPrivateDir` — and **no** `WWWWorkingDir`. Both
enumerate the same two-tier shape; neither mentions `working`.

**Orphan removal.** The previous `web`-group model is gone, and so is every symbol
that existed only to serve it: the `internal/opsctl/www.go` helper
(`ensureWWWPerms`), the `System.Chmod` seam method (its only caller was the setgid
loop), and, now that init-box no longer creates the group, the
`System.EnsureSystemGroup` and `System.AddUserToGroup` seam methods (their only
callers were the init-box `web`-group lines). None of these has any other consumer;
they are removed rather than left dangling.

**Boundary.** D7 defines the served-tree shape (`public`/`private`, no `working`)
and its ownership (service user, `0750`) at `setup`. It does not change the
DEFAULT-app branch (D5), the unit/user/nginx-fragment steps, or `deploy`/`restore`
(D8/D9 own those paths). Apps with no served tree (`WWWDirsFor` empty) are
unaffected.

## Rejected

- **Keep the `web` group + setgid so nginx can read the tier off disk (the prior
  web-group model).** That model existed only because nginx served the tiers by
  `alias`ing the directory and therefore had to traverse `state/`. The sites
  service now serves the bytes itself and the front door proxies to it, so nginx
  never reads the tree — the group, the setgid, the `ensureWWWPerms` helper, the
  `Chmod` seam, and the init-box group/membership are all dead. Retaining dead
  machinery "in case" is exactly the orphaned code this decision removes. Rejected.
- **World-readable `0755 app:app`.** Exposes the private tier on disk to every
  local user. `0750 app:app` keeps private bytes off `other` and needs no group.
  Rejected.
- **Keep a trimmed `ensureWWWPerms` that only chowns `app:app` (no setgid, no
  group).** It would just wrap the `ChownTree(app, app, …)` the state sweep
  already performs — a helper with a single trivial line and one caller per path.
  Folding it into the ordinary state chown removes the indirection. Rejected.
- **Keep a `working/` staging tier.** Removed with the publish step it existed to
  stage; files are authored and served in place. Rejected.

## Verification

- R-QFXB-VARQ — **The served-tree model carries no `working/` segment.**
  `WWWDirsFor("/opt", "sites")` returns exactly the three parent dirs
  `{WWWDir(), WWWPublicDir(), WWWPrivateDir()}` — three entries, none whose path
  ends in `/working` — and a `project/`-excluded grep over `internal/` for
  `WWWWorkingDir` and `www/working` returns no matches. *(unit: direct `WWWDirsFor`
  call + a scoped grep)*
- R-3K9X-IPJZ — **Setup of a served-tree app owns `state/www` as the service user,
  with no `web`/setgid step.** After `Setup` of an app whose `WWWDirsFor` is
  non-empty (`sites`), the fake `System` has recorded a
  `ChownTree(app, app, StateDir())` (which covers `state/www`) and has recorded
  **no** `ChownTree(_, "web", _)` over any path and **no** `Chmod` of a www path.
  The `public/` and `private/` tier dirs exist at mode `0750`. Falsifiable: the
  prior helper recorded `ChownTree(app, "web", WWWDir())` and `Chmod(d, 02750)`.
  *(unit: temp `OPSCTL_ROOT` + fake `System`)*
- R-AUAI-EX87 — **Setup of a no-served-tree app touches no www state.** After
  `Setup` of an app whose `WWWDirsFor` is empty (a fragment or worker app other
  than sites), the fake `System` records no `Chmod` of a www path and creates no
  `state/www` tree; the worker symlink / fragment-file behavior (D5/D01) is
  unchanged. *(unit: temp `OPSCTL_ROOT` + fake `System`)*
- R-3LHT-WHAO — **No orphaned served-tree code remains.** A `project/`-excluded
  grep over `opsctl/internal/` finds **none** of: `ensureWWWPerms`, `stateWWWFragment`,
  `EnsureSystemGroup`, `AddUserToGroup`, a `Chmod(` seam method on `System`, or the
  `"web"` group literal — and `internal/opsctl/www.go` no longer exists. The suite
  stays green with those symbols gone (no dangling caller, no unused-import/method
  breakage). Falsifiable today: each of those symbols is present. *(unit: scoped
  grep + green build/test)*
