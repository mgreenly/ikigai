# Decision 4 — `opsctl deploy` renders and installs the apex block for the DEFAULT app

## Decision

`opsctl deploy` today swaps three symlinks (`bin/run`, `etc/current`,
`share/current`) and reloads nginx, then restarts the unit (`deploy.go`). For a
normal path-routed service that is enough to apply an nginx change: `opsctl setup`
installed `conf.d/locations/<svc>.conf` as a **symlink → `/opt/<svc>/etc/current/
nginx.conf`**, and every service fragment carries a **literal** loopback port
(commit dc3dfd6), so swapping `etc/current` re-points the symlink and the reload
picks up the new fragment verbatim. No deploy-time render is needed.

The **apex/`DEFAULT` app** (dashboard) cannot use that path. Its nginx config is
not a location fragment but the whole apex vhost, and it lives at a **regular
file** `l.ApexBlockPath()` (`/etc/nginx/conf.d/<app>.conf`), written **once** by
`opsctl init-box` via `renderApexBlock` (`templates.go`, `initbox.go`). It carries
`__DOMAIN__` and `__PORT__` placeholders that are **host-specific** — `__DOMAIN__`
feeds `server_name` and the `ssl_certificate …/live/__DOMAIN__/…` paths — so it
**cannot** be hardcoded in the committed source the way service ports were, and it
**cannot** be a verbatim symlink to `etc/current`. It must be **rendered per box
at install time**. Because `deploy` never re-installs it, a dashboard deploy today
swaps the binary but leaves the apex routing frozen at its `init-box` state.

This Decision adds an **apex-aware branch to `Deploy`**: when the app being
deployed is the `DEFAULT` app, deploy renders its now-current apex source and
installs it, validated, before the nginx reload. Non-DEFAULT apps are unchanged.

The branch runs **after** the `etc/current` atomic swap and **before** the
existing `NginxReload` (`deploy.go` step 4→5), so it renders from the
just-swapped version:

```go
// After the etc/current swap, before o.System.NginxReload(ctx):
mf, err := parseManifest(l.ActiveManifest())      // etc/current/manifest.env
if err != nil { return fmt.Errorf("deploy: read manifest: %w", err) }
if mf.Default {                                    // DEFAULT=true — the apex app
    domain := strings.TrimSpace(os.Getenv("IKIGENBA_DOMAIN"))
    if domain == "" {
        return fmt.Errorf("deploy: %s is the DEFAULT app but IKIGENBA_DOMAIN is not set "+
            "(the apex block needs it for server_name and the cert path)", app)
    }
    src, err := os.ReadFile(l.ActiveNginxConf())   // etc/current/nginx.conf (apex source w/ placeholders)
    if err != nil { return fmt.Errorf("deploy: read apex source: %w", err) }
    block := renderApexBlock(string(src), domain, mf.Port)
    if err := writeFileAtomic(l.ApexBlockPath(), []byte(block), 0o644); err != nil {
        return fmt.Errorf("deploy: write apex block: %w", err)
    }
    if err := o.System.NginxTest(ctx); err != nil { // validate BEFORE reload; abort on failure
        return fmt.Errorf("deploy: apex block failed nginx -t, not reloading: %w", err)
    }
    o.logf("rendered apex block %s (domain=%s port=%d)", l.ApexBlockPath(), domain, mf.Port)
}
// ... existing o.System.NginxReload(ctx) follows
```

**Inputs, all already on the box, none new:**
- **Is-it-the-apex** — the app's own **`manifest.env`** carries `DEFAULT=true`
  (dashboard) vs `DEFAULT=false` (every service). Deploy reads the freshly-swapped
  `etc/current/manifest.env`; the discriminator is a manifest field, **not** a
  hardcoded `"dashboard"` name, so any future `DEFAULT=true` app is handled.
- **`__DOMAIN__`** — `os.Getenv("IKIGENBA_DOMAIN")`, already loaded into opsctl's
  process from `/etc/ikigenba/env` by D3's `LoadEnvFile`. Absent → hard, named
  failure (see below), never an empty substitution.
- **`__PORT__`** — the manifest's `PORT` (dashboard `PORT=3000`), the same value
  `init-box` took as `--port`.
- **The apex source** — `etc/current/nginx.conf`, i.e. the committed
  `dashboard/etc/nginx.conf` (placeholders intact) shipped in the bundle and
  staged like any other app's `nginx.conf`. So the apex block is now
  **version-bound**: it renders from whatever the deployed release carries.

**`renderApexBlock` is reused unchanged** — the identical substitution `init-box`
performs, so an `init-box`-rendered block and a `deploy`-rendered block from the
same source are byte-identical. `writeFileAtomic`, `System.NginxTest`,
`System.NginxReload` already exist and are seam-isolated. The privilege model is
intact: the binary only *carries* the source, **opsctl (root)** renders and
installs.

**Validate before reload, abort on failure (Dokku discipline).** `nginx -t` runs
on the freshly-written block *before* `NginxReload`. On failure deploy returns an
error and does **not** reload or restart; because nginx reload is graceful and the
running config is untouched until a successful reload, the previous apex routing
stays live. A malformed apex block is a **security** defect (the apex carries
`/_authn` and the trusted-header injection), so failing closed here is
load-bearing, not cosmetic.

**Boundary.** D4 changes only `Deploy`'s handling of the `DEFAULT` app's nginx
block. It does **not** touch the service-fragment path (symlink + literal port,
unchanged), the three-symlink swap, migrate/backup/chown/restart/prune, or
`init-box` (which still bootstraps the *first* apex block + the TLS cert
chicken-and-egg on a greenfield box — deploy assumes the cert already exists).
The broader "every service emits its own fragment" restructure (research option
a-ii) is out of scope: services already re-apply via their `etc/current` symlink.

## Rejected

- **Leave the apex block to `init-box` only; deploy just swaps the binary.** The
  status quo — the reason dashboard's routing silently lags its binary. A routing
  change (e.g. a new `location`) would ship un-applied until an operator re-ran
  `init-box` by hand. Rejected: routing must move atomically with the binary that
  needs it.
- **Hardcode `__DOMAIN__`/`__PORT__` in the committed apex source and symlink it
  to `etc/current`, like the service fragments.** Impossible for `__DOMAIN__`: the
  committed source is shared across accounts, and the domain (`server_name`, cert
  path) is per-box. Rejected — the apex genuinely must be rendered per box.
- **Detect the apex app by hardcoding the name `"dashboard"` in deploy.** Couples
  deploy to one app name and to init-box's `--default-app` default drifting.
  Reading `DEFAULT` from the manifest is the same source of truth the rest of the
  suite uses (`MOUNT=/`, `DEFAULT=true`) and needs no second place to keep in
  sync. Rejected.
- **Render with an empty domain when `IKIGENBA_DOMAIN` is unset.** Would write an
  apex block with an empty `server_name`/cert path and reload it — a broken,
  possibly insecure front door. Failing loudly with the value named (the product's
  "failures are loud and specific" promise) is required. Rejected.
- **Reload first, then `nginx -t`; or restart on failure.** Reloading an unvalidated
  block risks cutting the whole box over to broken routing. Validate-then-reload,
  abort-on-fail (leaving the running config) is the safe order. Rejected.

## Verification

- R-MSOP-5MDA — **Deploy renders and installs the apex block for a `DEFAULT`
  app.** Given a staged release whose `etc/current/manifest.env` has `DEFAULT=true`
  and `PORT=<p>` and whose `etc/current/nginx.conf` contains `__DOMAIN__` and
  `__PORT__`, with `IKIGENBA_DOMAIN=<d>` set, after `Deploy` the file at
  `l.ApexBlockPath()` contains `<d>` (as `server_name`/cert path) and the literal
  `<p>`, and contains **no** `__DOMAIN__` or `__PORT__` occurrence. Falsifiable: a
  deploy that skips the render leaves the old block (or none) with placeholders
  intact. *(unit: temp OPSCTL_ROOT + fake `System`)*
- R-MTWL-JE3Z — **Deploy of a `DEFAULT` app with `IKIGENBA_DOMAIN` unset fails
  loudly and installs nothing.** With `DEFAULT=true` but no `IKIGENBA_DOMAIN` in
  the environment, `Deploy` returns a non-nil error whose message names
  `IKIGENBA_DOMAIN`, `l.ApexBlockPath()` is left unchanged (the prior block, or
  absent), and `System.NginxReload` is **not** called. Pins fail-closed on the
  missing host value — an implementation that renders an empty `server_name` and
  reloads fails this. *(unit: fake `System` records calls)*
- R-MV4H-X5UO — **Deploy validates the rendered apex with `nginx -t` before
  reload and aborts on failure.** With a fake `System.NginxTest` returning an
  error for a `DEFAULT` app, `Deploy` returns an error, and `System.NginxReload`
  and the unit restart are **not** called (call order recorded: `NginxTest`
  precedes any `NginxReload`, and no `NginxReload`/`Restart` occurs after a failed
  test). Falsifiable: a deploy that reloads regardless, or tests after reloading,
  fails this. *(unit: fake `System` records call order + outcomes)*
- R-MXKA-OPC2 — **Deploy of a non-`DEFAULT` app installs no apex block.** Given a
  staged release with `DEFAULT=false` (a normal service manifest), after `Deploy`
  no file is written at `l.ApexBlockPath()` and the apex-render branch is not
  entered (the existing `etc/current` symlink + reload path is the only nginx
  effect). Pins the discriminator — an implementation that renders for every app
  (or keys off something other than `DEFAULT`) fails this. *(unit: temp
  OPSCTL_ROOT + fake `System`)*
- R-CNPY-3Z4Y — **The apex domain is sourced from the process environment, never
  the manifest.** Given a `DEFAULT=true` release whose `etc/current/manifest.env`
  contains **no** `IKIGENBA_DOMAIN` key (the manifest never carries the per-box
  domain — it holds only `DEFAULT`/`MOUNT`/`PORT`), with `IKIGENBA_DOMAIN=<d>` set
  in the environment, `Deploy` renders the apex block using `<d>` and succeeds.
  Conversely, a domain placed **only** in the manifest (with the environment
  unset) does **not** satisfy the requirement — `Deploy` fails per R-MTWL-JE3Z.
  This is the sharp regression guard for the bug where the render read
  `manifest["IKIGENBA_DOMAIN"]` instead of `os.Getenv("IKIGENBA_DOMAIN")`: that
  implementation passes an env-in-manifest test but fails this one (env set,
  manifest domain absent → it errors). *(unit: temp OPSCTL_ROOT + fake `System`;
  the test seeds the domain via the process env, e.g. `t.Setenv`, and never into
  the manifest)*
- R-MYS7-2H2R — **Real-substrate (live box `int.ikigenba.com`).** Deploying the
  dashboard on the box renders the apex block with the box's **real**
  `IKIGENBA_DOMAIN` and `PORT`, `nginx -t` passes against the real nginx + the
  real apex TLS cert, the reload succeeds, and afterward `https://int.ikigenba.com/`
  serves the dashboard **and** the path-routed `/srv/<svc>/` mounts still resolve
  through the freshly-installed apex `include …/locations/*.conf`. This cannot be
  falsified by a unit test with a fake `System` (which never runs real `nginx -t`
  against the real cert, nor proves the include still resolves), so it is
  **operator-verified out-of-loop** and tracked in a verification doc; the loop
  phase realizing D4 covers R-MSOP-5MDA / R-MTWL-JE3Z / R-MV4H-X5UO / R-MXKA-OPC2
  (partial-Decision split). The observable proof is the apex + all service routes
  serving after a real dashboard deploy.
