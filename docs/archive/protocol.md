# Client-Server Protocol

## Transport

WebSocket with JSON messages. Minimal overhead (2-6 bytes/message), enables future web clients.

## Connection Lifecycle

1. Client connects to server via WebSocket (`ws://localhost:1984/ws` during development)
2. **Handshake**: Client sends `hello`, server responds with `welcome` and session_id
3. Client begins sending queries with session_id
4. Server generates correlation_id for each exchange
5. On disconnect: conversation ends

## Handshake

The handshake occurs before the standard message envelope format.

**Client â†’ Server (hello)**
```json
{
  "type": "hello",
  "identity": "hostname@username"
}
```

**Server â†’ Client (welcome)**
```json
{
  "type": "welcome",
  "session_id": "uuid"
}
```

After handshake completes, all messages use the standard envelope format with session_id.

## Message Envelope

```json
{
  "session_id": "uuid",
  "correlation_id": "uuid",
  "type": "message_type",
  "payload": {}
}
```

- `session_id`: WebSocket session (entire conversation), generated by server during handshake
- `correlation_id`: Exchange ID (query-response pair), generated by server for each user_query
- `type`: Message type
- `payload`: Type-specific data

**Correlation ID Flow:**
- Client sends `user_query` with `correlation_id: null`
- Server generates new correlation_id for this exchange
- Server includes correlation_id in all `assistant_response` chunks
- Future: Client echoes correlation_id when sending `tool_result` (Phase 6)

## Message Types

### Client â†’ Server

**`user_query`**

Phase 1 payload is the OpenAI Chat Completions API request body (without API key):

```json
{
  "type": "user_query",
  "payload": {
    "model": "gpt-4o",
    "messages": [
      {
        "role": "user",
        "content": "string"
      }
    ],
    "stream": true,
    "temperature": 0.7,
    "max_tokens": 4096
  }
}
```

Server injects API credentials before forwarding to OpenAI.

**`tool_result`** (Phase 6)
```json
{
  "type": "tool_result",
  "payload": {
    "tool_id": "uuid",
    "result": "string or object",
    "error": "string or null"
  }
}
```

### Server â†’ Client

**`assistant_response`**

Phase 1 forwards OpenAI's streaming response chunks directly:

```json
{
  "type": "assistant_response",
  "payload": {
    "id": "chatcmpl-...",
    "object": "chat.completion.chunk",
    "created": 1234567890,
    "model": "gpt-4o",
    "choices": [
      {
        "index": 0,
        "delta": {
          "content": "string"
        },
        "finish_reason": null
      }
    ]
  }
}
```

Final chunk has `finish_reason: "stop"` and empty `delta`.

**`tool_delegation`** (Phase 6)
```json
{
  "type": "tool_delegation",
  "payload": {
    "tool_id": "uuid",
    "tool_name": "file_read",
    "parameters": {}
  }
}
```

**`error`**

Sent by server when errors occur. Server closes WebSocket after sending.

```json
{
  "type": "error",
  "payload": {
    "source": "openai|server|protocol",
    "error": {
      // OpenAI: full error object from API
      // Server/protocol: {"message": "description"}
    }
  }
}
```

## Error Handling

- Server sends `error` message, then closes WebSocket
- Client reconnects for new session
- No automatic retriesâ€”fail fast

## Tool Execution

All tools execute on client. Some tools (web search, RAG queries) proxy through server internally. Client uses uniform tool interface regardless of implementation.

## Notes

- Trust-based identity, no authentication
