# Decision 9 — No site footer: remove the `.site-footer` from every page and delete its CSS

**Decision.** The dashboard renders **no site footer on any page.** The
`<footer class="site-footer">…</footer>` element is removed from every template
that carries it, and its `.site-footer` CSS rule (with its section comment) is
deleted from `ui/static/app.css`. The footer previously showed the account host
(`{{.Host}}`) on the full pages and the literal `ikigenba` on the PAT-created
partial; none of that survives.

The footer lives in exactly four template locations today, all of which lose it:

- `ui/html/index.html` — **both** branches: the logged-out `.signin-wall` footer
  and the logged-in landing/home footer.
- `ui/html/profile.html` — the profile-page footer.
- `ui/html/partials/pat_created.tmpl` — the PAT-created fragment footer
  (`<footer class="site-footer">ikigenba</footer>`).

And the style is deleted from `ui/static/app.css`:

- the `.site-footer { … }` rule and its `/* ---- Footer --- */` section comment.

This is a pure template + CSS deletion. No route, handler, view-model, migration,
or schema changes; no other markup moves. Removing the element (rather than hiding
it) is deliberate — no dead markup and no orphaned CSS is left behind.

**Rejected.**
- **Hiding the footer with `display: none`** — leaves dead markup and a live CSS
  rule for an element that is never shown. Remove the element outright.
- **Keeping the `.site-footer` CSS "in case it comes back"** — dead CSS; delete it.
  A future footer is a fresh decision, not a commented-out relic.
- **Removing it from only some pages** — the intent is site-wide; a footer on the
  profile page but not the landing would be an inconsistent surface. All four
  locations lose it together.

**Verification.**
- R-EFJZ-FRQ1 — No page or fragment the dashboard serves renders a footer: the
  response bodies of the logged-out `GET /`, the logged-in `GET /` (live session),
  and `GET /profile` (live session) each contain **no** `<footer` substring and no
  `site-footer` substring; the PAT-created fragment returned by the create-PAT POST
  likewise contains no `<footer`/`site-footer`; and the served stylesheet
  (`GET /static/app.css`, or the embedded `ui/static/app.css`) contains **no**
  `.site-footer` selector. A footer left on any one of these surfaces, or the CSS
  rule left in place, fails this.
