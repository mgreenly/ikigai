# Decision 10 ‚Äî The banner is shared app-shell chrome: wordmark‚Üíhome, monogram avatar, sign-out

**Decision.** The `<header class="banner">` is the same **app-shell chrome** on every
signed-in page (the landing/home and the profile page). It carries exactly three
things: the **wordmark** (now a link **home**), the **sign-out** control, and the
**monogram avatar** (the D5 identity affordance, linking to `/profile`), in that
left-to-right order. The banner **never shows the owner's email as text** and carries
**no separate "Home" button** ‚Äî the wordmark absorbs the home affordance.

This resolves the ugly profile banner (a bordered "Home" button + raw email text +
sign-out, with the email duplicated by the page body's "Signed in as ‚Ä¶") and makes the
two signed-in pages share one coherent header.

- **Wordmark ‚Üí home.** The `ikigenba` wordmark becomes a link to `/`
  (`<a href="/" class="wordmark">ikigenba</a>`) on **both** the landing and profile
  banners. Clicking the logo goes home ‚Äî the universal convention ‚Äî so no separate
  navigation button is needed. (On the landing, the link target is the current page;
  that is harmless and standard.)
- **Profile banner adopts the shared chrome.** The profile banner's right side becomes
  identical to the landing's: **sign-out then the monogram avatar** (the D5 avatar,
  `<a href="/profile" class="avatar" aria-label="Profile ‚Äî {{.Owner}}"
  title="{{.Owner}}">{{.OwnerInitial}}</a>`). The old **bordered "Home" button** and
  the **`<span class="owner">{{.Owner}}</span>` email text are removed** ‚Äî identity is
  already stated by the profile body's "Signed in as ‚Ä¶" line and carried by the
  avatar's accessible name. The profile view model gains the same `OwnerInitial` field
  the landing uses (computed by the shared `ownerInitial` helper), so the avatar
  renders on both pages.
- **The avatar has a hover affordance.** Because the avatar is an icon-only link, it
  needs a visible interactive state to read as clickable. On hover the avatar's fill
  **darkens from `--color-accent` to `--accent-700`** ‚Äî the exact darkening
  `.btn-primary:hover` already uses ‚Äî via `.identity .avatar:hover`, with the same
  transition as the other controls. This applies to the avatar on **both** pages
  (the one shared `.avatar` rule).

This decision governs only the **banner chrome** shared by the signed-in pages. It does
not change page bodies, routes, the service inventory, or the D5 landing composition
beyond making the wordmark a home link.

**Rejected.**
- **Keeping the profile banner's "Home" button** ‚Äî a dedicated button is redundant once
  the wordmark links home; the logo-as-home pattern is universally understood.
- **Keeping the email as banner text on profile** ‚Äî it duplicates the body's "Signed in
  as ‚Ä¶" and is already the avatar's accessible name; two identity displays in one
  viewport is clutter.
- **Cursor-only affordance for the avatar** (no hover state) ‚Äî a pointer cursor alone
  doesn't signal clickability at a glance; a fill change on hover matches the button
  idiom and makes the control obviously interactive.
- **A distinct home affordance per page** (e.g. a "Back to home" link only on profile) ‚Äî
  the whole point is one shared banner; asymmetric chrome is what made it ugly.

- R-VUQA-WM5Z ‚Äî The profile banner (`GET /profile`, live session) renders the monogram
