# Decision 23 — Purge legacy auth state and enforce the `owner_id` invariant (`NOT NULL`)

## Decision

D18 introduced `owner_id` as a **nullable** column added by `ALTER TABLE ADD
COLUMN`, on the assumption that no rows predate it ("the box DB is reset at
cutover"). That assumption did not hold on the live box: real auth artifacts
predated the migration, took `owner_id = NULL`, and every `owner_id` read scans
the column into a **non-nullable Go `string`**. `database/sql` cannot scan
`NULL` into `string`, so each such read fails with
`converting NULL to string is unsupported`:

- `pat.scanPAT` (shared by `ValidatePAT`, `ListByOwner`, `GetByPublicID`) — a
  single legacy PAT breaks token validation *and* makes `ListByOwner` return an
  error, which the profile UI renders as an empty token list ("no PATs").
- the `oauth_chains` row scan behind `TokenStore.ValidateAccess` — a legacy
  chain breaks bearer-token introspection.

The identity model treats `owner_id` as **always present** on an auth artifact
(D18 stamps it on every creation path). The defect is that the schema permitted
a state the code forbids. This decision makes the invariant real in the schema
rather than defending against it in code.

### One self-contained, forward-only migration

A single new migration (`bin/create-migration dashboard
purge_auth_and_enforce_owner_id`) does two things, in order, atomically:

**Part A — purge all auth/identity state.** Delete every row from the auth
graph, so no legacy `NULL`-`owner_id` row survives to violate the new
constraint. Referential integrity forces the whole graph, not just the four
`owner_id` carriers: `oauth_tokens` hangs off `oauth_chains`, `oauth_authcodes`
carries a `chain_id`, and the identity itself is shared. Cleared tables:

```
oauth_tokens
oauth_authcodes
oauth_chains
personal_tokens
web_sessions
oauth_state
dcr_clients
identities
```

`audit_log` and `schema_migrations` are **preserved** — the audit trail
(including this incident) is history, not auth state. The result is a genuine
clean slate: every client re-registers (DCR), every user re-logs-in (new
identity + session), every PAT is re-minted — all going through the stamping
paths, so every future row carries a real `owner_id`.

**Part B — enforce `owner_id TEXT NOT NULL`.** Rebuild the four carrier tables
(`web_sessions`, `oauth_authcodes`, `oauth_chains`, `personal_tokens`) so
`owner_id` is `NOT NULL`. SQLite has no `ALTER COLUMN`, so this is the standard
table rebuild — `CREATE` the new-shape table, (no row copy: Part A emptied it),
`DROP` the old, `ALTER … RENAME` the new into place — executed with
`foreign_keys` handling per the runner's migration contract. Because Part A ran
first the rebuild is trivial and cannot fail on a stray `NULL`.

Ordering is guaranteed because both parts live in **one** migration applied by
the appkit runner: on the live box it purges then enforces; on a fresh DB the
deletes are no-ops and the rebuild simply yields `NOT NULL` tables. There is no
manual sequencing to get wrong.

### No Go changes

Nothing in the read or write paths changes:

- **Writes** already stamp `owner_id` on all four carriers (D18), and always
  with a **non-empty** handle — `callback.go` resolves the identity via
  `ResolveOrCreate` and **fails loud** (logs `callback.resolve_identity`,
  returns 500) on error, so no completion path ever writes an empty or absent
  handle. `NOT NULL` is satisfied by every existing writer.
- **Reads** scan `owner_id` into a plain `string`; once the column is
  `NOT NULL` with no legacy rows, that scan is correct **by construction** —
  the failure mode it had (a `NULL` in the column) can no longer exist.

The earlier one-off production backfill (setting `owner_id` from `identities` on
`owner_email`) was a throwaway hotfix; this migration supersedes it by clearing
those rows entirely.

## Rejected

- **NULL-tolerant read (`sql.NullString` for `owner_id`).** A defensive
  fallback that papers over a state the identity model says cannot exist; it
  keeps the column nullable, so the bug class stays open and the "always
  present" invariant remains a convention rather than a fact. Against the
  project's principles (*fail loudly; enforce with systems, not willpower; no
  defensive fallbacks*).
- **Backfill `owner_id` from `identities` on `owner_email` (preserve tokens).**
  Keeps every existing token working, but leaves the column nullable (bug class
  open) and re-derives the durable handle through the **mutable** email — the
  exact anti-pattern D18 rejected. Also contrary to the goal of a clean slate.
- **Separate operational DB reset + a plain `NOT NULL` migration.** Relies on an
  operator remembering to reset the DB before deploy; on a DB that was not
  reset, the migration fails at boot. Not self-contained and not testable as one
  unit.
- **`CHECK (owner_id <> '')` in addition to `NOT NULL`.** Redundant: the sole
  writers resolve a real handle and fail loud on error, so an empty string can
  never be written. Extra constraint for a state the code already precludes —
  over-engineering.
- **Purge only the four `owner_id` carriers.** Leaves `oauth_tokens`,
  `oauth_state`, `dcr_clients`, and `identities` behind — orphaning
  `oauth_tokens` from their deleted chains and keeping stale identities. A clean
  slate requires cascading across the whole auth graph.

## Verification

- R-6QJD-1MUY — applying the migration to a **real SQLite DB** pre-seeded with
  rows in `identities`, `web_sessions`, `oauth_authcodes`, `oauth_chains`,
  `oauth_tokens`, `oauth_state`, `personal_tokens`, and `dcr_clients` (a legacy
  `personal_tokens`/`oauth_chains` row with `owner_id = NULL` among them)
  completes without error and leaves **every one of those eight tables with zero
  rows**, while a pre-seeded `audit_log` row remains present. Proves the purge is
  complete, cascades the whole graph, spares the audit trail, and does not choke
  on the legacy `NULL` rows.
- R-6RR9-FELN — after the migration, each of `web_sessions`, `oauth_authcodes`,
  `oauth_chains`, `personal_tokens` **rejects an `INSERT` whose `owner_id` is
  `NULL`**, the real SQLite engine returning a `NOT NULL` constraint error
  (verified against real SQLite, not a mock). Proves the column is actually
  `NOT NULL` on all four carriers — a claim only the real storage engine can
  falsify.
- R-6SZ5-T6CC — against a DB migrated to the new schema, minting a personal
  access token (`pat.Store.Create`, which stamps `owner_id`) and then calling
  `ListByOwner` returns that token and `ValidatePAT` accepts it — with **no
  `converting NULL to string` scan error**. Proves the shared `scanPAT` read
  path (list + validate) is healthy end-to-end on the enforced schema, i.e. the
  "no PATs" / "can't connect" regression is closed.
- R-6U72-6Y31 — against a DB migrated to the new schema, issuing an OAuth chain
  (`TokenStore.IssueChainAndTokens`, which stamps `owner_id`) and then calling
  `ValidateAccess` with that access token returns the valid token bound to a
  chain whose `owner_id` equals the stamped handle — no `NULL`-scan error.
  Proves the `oauth_chains` bearer-introspection read path is healthy on the
  enforced schema.
