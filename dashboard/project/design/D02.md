# Decision 2 ‚Äî The profile route is session-gated in-process (redirect when signed out)

**Decision.** `GET /profile` is gated to a **signed-in owner**, decided
**in-process** ‚Äî the dashboard owns its own session and runs no nginx
`auth_request` over its own routes (design spine). The gate reuses the existing
session seam, not a new mechanism.

`handleProfile` resolves the owner exactly as the index does:

- Read the `dashboard_session` cookie and look it up via `a.sessions.Lookup`
  (through the existing `(*app).sessionOwner` helper).
- **Signed in** (live session) ‚Üí render the profile page for `sess.OwnerEmail`.
- **Signed out** (cookie missing, or `session.ErrInvalid`, or any lookup miss) ‚Üí
  **do not render profile content**; redirect to `/` (`http.StatusFound`, 302),
  which is the login page for a signed-out visitor. A present-but-dead cookie is
  cleared on the way out (`clearSessionCookie`), mirroring `handleIndex`'s cleanup,
  so the browser stops resending a value that can never redeem.

A page that redirects (rather than the bare `401` `requireSession` returns for the
fetch/XHR endpoints) is the right human-page behavior: a browser landing on
`/profile` without a session is bounced to sign-in, not shown a JSON-shaped error.
The state-changing token/grant POSTs the profile page submits keep their existing
`requireSession` (401) + same-origin guards ‚Äî this decision governs only the
**page GET**.

The gate is deliberately **coarse**: any signed-in owner sees the profile of the
session they hold (their own email). There is no per-resource check beyond "a live
session" ‚Äî consistent with the single-tenant box model.

**Rejected.**
- **Bare `401` for `GET /profile` when signed out** ‚Äî correct for the XHR/POST
  endpoints, hostile for a top-level page navigation; a human should be sent to
  sign-in.
- **An nginx `= /profile` `auth_request /_session-authn` location** ‚Äî that is the
  path-routed services' pattern; the apex is behind appkit's Apex bypass and gates
  in-process. Adding a fragment here would be a second, divergent gate for a route
  the dashboard already authenticates itself.
- **Rendering profile content and hiding controls with CSS when signed out** ‚Äî
  leaks the page shell and is not a gate; the signed-out visitor must not receive
  profile content at all.

**Verification.**
- R-DB04-GATE ‚Äî `GET /profile` with no or an invalid session does **not** render
  profile content; it responds with a redirect (302) whose `Location` is `/`. A
  present-but-invalid session cookie is cleared on that response.
- R-DB05-SESS ‚Äî `GET /profile` with a valid session renders the profile page and
  shows the signed-in owner's email.
