# Decision 18 — Capture identity at login: decode the claims and stamp the handle onto every auth artifact

## Decision

`(iss, sub)`/`name`/`picture` are available only at **login** (in the verified
id_token); the `auth_request` introspection endpoints never re-contact Google
(D19). So the identity handle must be **resolved at login and carried on the
auth artifact** each introspection path later reads. Two parts.

### Part A — `googleidp` decodes the identity claims

`internal/googleidp` today decodes only `iss/aud/exp/sub/email/hd/
email_verified` and its `Identity` carries only `Sub/Email/HostedDomain/
EmailVerified`. Extend both to carry the full identity:

```go
type idTokenClaims struct {
    // …existing…
    Iss     string `json:"iss"`     // already read for validation; now surfaced
    Name    string `json:"name"`    // optional
    Picture string `json:"picture"` // optional
}

type Identity struct {
    Sub, Email, HostedDomain string
    EmailVerified            bool
    Iss, Name, Picture       string // NEW; Name/Picture empty when Google omits them
}
```

`verifyIDToken` populates the new fields after the existing signature/iss/aud/
exp checks. Absent `name`/`picture` decode to empty strings; the identity stays
valid (they are attributes, not gates).

### Part B — resolve at callback and stamp the handle onto every artifact

`(*app)` gains an `identity *identity.Store` field. The Google callback resolves
the identity once and threads its handle into every auth-artifact creation path.

`internal/server/callback.go`'s `handleCallback` already holds the verified
`identity` (googleidp) before dispatching to `callbackWeb` / `callbackMCP`. It
calls `a.identity.ResolveOrCreate(ctx, identity.Claims{Iss,Sub,Email,Name,
Picture})` to get the handle, and passes the handle (alongside the email it
already passes) into both completion paths.

The handle is then persisted on each artifact the introspection paths read. A
second forward-only migration (`bin/create-migration dashboard
add_owner_id_to_auth_artifacts`) adds a nullable `owner_id TEXT` column to the
four carrier tables, and each creation path writes it:

| artifact | table | creation seam | how it gets the handle |
|---|---|---|---|
| web session | `web_sessions` | `session.SessionStore.Create` | `callbackWeb` passes the resolved handle |
| OAuth auth code | `oauth_authcodes` | `oauth` authcode issue | `callbackMCP` passes the resolved handle |
| OAuth chain | `oauth_chains` | `oauth.TokenStore.IssueChainAndTokens` | copied from the consumed authcode row |
| personal token | `personal_tokens` | `pat.Store.Create` | `handlePATCreate` passes the caller session's handle |

Signatures gain an `ownerID string` parameter next to the existing
`ownerEmail` (both are written; `owner_email` stays exactly as today):

```go
func (s *SessionStore) Create(ctx, ownerEmail, ownerID string) (Issued, error)
func (s *TokenStore) IssueChainAndTokens(ctx, tx, clientID, ownerEmail, ownerID, resource string) (TokenPair, error)
func (s *Store) Create(ctx, ownerEmail, ownerID, label string) (plaintext string, p PAT, err error) // pat
```

The authcode carries `owner_id` so it survives the authcode→chain exchange (the
token endpoint has no id_token at exchange time): authcode issue writes
`owner_id`, authcode consume returns it, and `IssueChainAndTokens` copies it
onto the chain. The PAT path reads the handle off the creating owner's session
(which now carries it), so it never re-resolves.

Every creation path populates `owner_id`. Its **storage invariant** — that the
column is `NOT NULL` and holds no legacy `NULL` rows — is owned by D23 (Purge
legacy auth state and enforce the `owner_id` invariant); this decision covers
only capturing the handle at login and writing it onto each artifact.

This decision changes **no** header output and **no** introspection behavior —
that is D19. Here the handle only gets captured and stored.

## Rejected

- **Re-resolve the identity at introspection instead of stamping it.**
  Impossible: introspection has only the opaque token/cookie and never sees
  `(iss, sub)`. The handle must be carried on the artifact.
- **Look the identity up by `owner_email` at introspection (email join).**
  Re-derives the "stable" handle through the mutable email — a live token with a
  now-stale email would miss the updated identity row, defeating the point.
  Stamping the handle at creation is a direct, email-independent read.
- **A `userinfo` fetch to get name/picture.** Unnecessary network round-trip;
  the id_token already carries them under the `profile` scope. Held in reserve
  only if they prove unreliable (see research).
- **Add `owner_id` only to the tables strictly needed today.** All four carriers
  are on a real introspection path (`web_sessions`→session, `oauth_chains`→bearer,
  `personal_tokens`→PAT) or the carrier that feeds one (`oauth_authcodes`→chain),
  so all four need it.

## Verification

- R-VPQ6-37CQ — `verifyIDToken` (or its claims-decoding seam) applied to an
  id_token payload carrying `iss`/`name`/`picture` produces an `Identity` whose
  `Iss`/`Name`/`Picture` equal those claims; applied to a payload **lacking**
  `name`/`picture`, the returned `Identity` has empty `Name`/`Picture` and is
  still returned as valid (no error, `Sub`/`Email` intact).
- R-VQY2-GZ3F — completing a **web** login writes a `web_sessions` row whose
  `owner_id` equals the handle `ResolveOrCreate` returns for that login's
  `(iss, sub)` (asserted against a real temp DB by reading the row back), while
  `owner_email` is unchanged from today.
- R-VS5Y-UQU4 — completing the **OAuth** authorization-code path writes the
  identity handle onto `oauth_authcodes.owner_id`, and exchanging that authcode
  issues an `oauth_chains` row whose `owner_id` equals the same handle (the
  handle survives authcode→chain).
- R-VTDV-8IKT — creating a **personal access token** writes a `personal_tokens`
  row whose `owner_id` equals the handle carried on the creating owner's session
  (no re-resolution), while `owner_email` is unchanged from today.
