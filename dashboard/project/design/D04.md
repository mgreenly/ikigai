# Decision 4 — OAuth grant management moves to the profile page

**Decision.** The **OAuth grant** surface — the live list of the owner's MCP token
grants (OAuth chains), its Server-Sent-Events live refresh, and per-grant revoke —
**moves from the landing/home page to the profile page**. The grant *capability* is
unchanged; only the page that embeds the block and the post-revoke redirect target
change.

- **View.** The `grants_block` partial and its live-stream container (the
  `<div id="grants-block" data-stream="/grants/stream" data-fragment="/grants/fragment">`
  wrapper that `app.js` upgrades) render on the **profile** page. `handleProfile`
  populates `Grants` for the signed-in owner via `a.oauthTokens.ListChainsByOwner`
  for the first paint, exactly as `handleIndex` does today. They are **removed**
  from the logged-in `/` (landing) template and `handleIndex` stops populating
  `Grants`.
- **Live refresh.** `GET /grants/stream` (SSE) and `GET /grants/fragment` (the HTML
  partial the stream client swaps in) are **unchanged** — same `requireSession`
  gate, same payloads. Only the page that *hosts* the `#grants-block` container
  moves; the endpoints it polls are the same.
- **Revoke.** `POST /grants/{public_id}/revoke` keeps its same-origin +
  `requireSession` guards, its not-found-on-mismatch behavior, and its
  grant-change `Publish`, but on success it **redirects to `/profile`**
  (`http.StatusSeeOther`) instead of `/`.

**Rejected.**
- **Moving the SSE/fragment endpoints under `/profile/...`** — pointless churn; they
  are session-authenticated fetch endpoints, not pages, and the JS references them
  by their existing paths. Only the container's host page moves.
- **Dropping the live-stream when the block moves** (re-render on navigation
  instead) — would silently downgrade a working feature; the live grants block is
  moved intact, stream wiring and all.
- **Leaving grants on the landing too** — same conflation D3 removes for PATs; the
  landing must carry no grant administration.

**Verification.**
- R-DB09-GRNT — `GET /profile` with a valid session renders the live-grants block:
  the `#grants-block` container with its `data-stream="/grants/stream"` /
  `data-fragment="/grants/fragment"` hooks and the `grants_block` partial for the
  owner's grants.
- R-DB10-GRVK — On success, `POST /grants/{public_id}/revoke` redirects to
  `/profile` (not `/`).
- R-DB11-GRNX — `GET /` with a valid session (the landing/home page) contains **no**
  grants block.
