# Decision 1 — Three-page topology and the route map

**Decision.** The dashboard's human web surface is **three pages**, served by two
routes. `GET /` stays a single handler that branches on the session — **login**
when logged out, **landing/home** when logged in (the audience split that exists
today). A **new** `GET /profile` route serves the **profile** page. No other human
page is added by this change.

- **`GET /` (handler `handleIndex`)** keeps its `{{if .Owner}}` branch:
  - **logged out → the login page**: the sign-in wall (wordmark, the control-plane
    tagline, "Sign in with Google", and a diminished name-origin colophon). No PAT
    form, no grants block, no email/profile controls. Its composition is **D7**.
  - **logged in → the landing/home page**: the owner's email at the top, the
    connect-your-agent install instructions, and the service list. Token and grant
    administration is **not** on this page (it moves to profile — D3, D4).
- **`GET /profile` (new handler `handleProfile`)** serves the profile page (D2 gates
  it; D3/D4 fill it). It is registered in `(*app).register` alongside the existing
  routes in `routes.go`.
- The profile page needs its own template. It is added to the `template.ParseFS`
  set in `server.go` (e.g. `html/profile.html`) so it shares the template set with
  the partials it reuses (`pat_block`, `grants_block`, `pat_created`) — a broken
  template fails startup, not a request.

This decision fixes the **topology and routing only**; what each page contains and
how the profile page is gated are D2–D5.

**Rejected.**
- **A third route for the landing** (e.g. `GET /home`) — the logged-in home is the
  *same URL* a returning owner already has bookmarked (`/`); splitting it to a new
  path would break that and contradict "the landing is the logged-in `/` view".
- **A name+version landing card like the path-routed services get** — the
  dashboard is the apex; its logged-in landing **is** its web home, not a generic
  card. Explicitly out of scope (product Scope).
- **Folding profile into `/` as a fourth `{{if}}` branch / query param** — the
  whole point is two pages with two framings; one template juggling home +
  settings is the hybrid page we are removing.

**Verification.**
- R-DB01-PG3A — `GET /` with no or an invalid session renders the **login** page
  (the sign-in control is present) and contains none of the logged-in controls: no
  PAT-create form, no grants block, no email/profile link.
- R-DB02-LND7 — `GET /` with a valid session renders the **landing/home** page: the
  owner's email is shown at the top, the connect-your-agent install instructions
  are present, and the service list is present.
- R-DB03-PRF9 — `GET /profile` is a registered route distinct from `GET /` (it is
  reachable and does not fall through to the index handler).
