# Decision 5 — Landing composition: service-name links, profile-avatar nav, sign-out

**Decision.** With token/grant administration moved away (D3, D4), the
landing/home page is composed of three things and gains its navigation:
**install instructions** (kept), the **service list** (each name now a **link** to
that service's page), a **monogram profile avatar** (linking to the profile page),
and **sign-out** (stays here).

- **Service-name links.** Each row in the service list links the service **name**
  to that service's own human landing page at its mount root `/srv/<svc>/` — not to
  the raw MCP resource URL. The mount is already known: `inventory.Read` yields each
  service's `Mount` (e.g. `/srv/crm/`). `serviceRows` (in `inventory.go`/the index
  view builder) gains a mount-derived **href** field (the `Mount` value itself,
  which already carries its trailing slash) alongside the existing display name and
  raw MCP `URL`, and the landing template renders the name as
  `<a href="{{.Href}}">…</a>`. The raw MCP `URL` is retained on each row; only the
  **name** becomes a link.
- **Service list uses the shared list chrome, not a table.** The service list is
  rendered with the dashboard's **shared `.list`/`.row` idiom** — the same
  hairline-bordered single surface, one `<li class="row">` per item, that the
  profile page's PAT and grant lists use — **not** a bespoke `<table>`. This is the
  one place in the app that had diverged into raw table markup; adopting the shared
  chrome makes the whole app read as one list language. The row is the standard
  `1fr auto auto` grid: the linked **name** (`.name`) sits left, the raw MCP **URL**
  (`.meta`, mono/subtle) and its copy control sit right, matching the name/meta/action
  shape of the PAT and grant rows. The `NAME`/`URL` column headers are dropped — the
  shared list has no header row and the rows are self-evident without intro copy.
- **Each row's MCP URL is copyable.** The raw MCP `URL` on each row carries a
  **copy-to-clipboard control** (the existing `.copy-btn` affordance, reused from the
  connect-agent snippets), so the URL a person needs for hand-wiring a non-suite MCP
  client is one click to grab rather than a manual select. The URL sits in a `<code>`
  the shared copy JS reads; the button carries an accessible name naming the service.
  This is the direct affordance that replaces the removed explanatory intro line.
- **No section intro copy.** Neither the connect-agent section nor the service-list
  section renders an intro paragraph (`.section-intro`). The section heading plus the
  self-explanatory controls beneath it carry the meaning; the two prose lines ("Paste
  the line for your agent…" and "The raw resource URLs, for wiring…") are removed as
  redundant chrome.
- **Profile → monogram avatar.** The profile affordance in the banner is a **solid
  accent-filled circle bearing the uppercased first letter of the owner's email** — a
  monogram avatar — that links to the profile route (`<a href="/profile"
  class="avatar" …>{{.OwnerInitial}}</a>`). The full email is **not** shown as text;
  it is carried as the link's accessible name (`aria-label`) and `title` (hover
  tooltip) so it stays reachable to assistive tech and on hover. The initial is
  computed in the view builder (a new `OwnerInitial` field on `indexData`), not in
  the template. This avatar is the only path the product offers to reach the profile
  page.
- **Avatar sits furthest right; sign-out to its left.** In the banner's `.identity`
  nav the reading order is **sign-out then avatar**, so the monogram avatar is the
  right-corner identity anchor (the conventional avatar position) and the destructive
  sign-out control does not occupy that habitual corner.
- **Sign-out stays on the landing.** The existing `POST /logout` form/button remains
  on the landing/home page (it does **not** move to profile). Signing out is an
  action you take from where you land.
- **Install instructions stay.** The connect-your-agent section (the Claude/Codex
  `curl … | bash` snippets) is unchanged and remains on the landing.

This decision governs only the **landing** page's composition. It does not change
the `/services` JSON inventory endpoint, the install scripts, or `mcpResourceURL`.

**Rejected.**
- **Linking the service name to the raw MCP resource URL** (`…/srv/<svc>/mcp`) —
  that endpoint is a bearer-gated JSON-RPC surface, not a human page; the click
  target is the service's human landing at the mount root (product Contractual
  constant).
- **Moving sign-out to the profile page** — sign-out belongs where the owner lands;
  burying it on a settings page hides the most common exit (product promise).
- **A hamburger / overflow menu holding profile + sign-out** — a disclosure menu is
  for crowded or space-constrained nav (many items, small screens); the banner holds
  exactly two controls with ample room, so hiding them behind a click hurts
  discoverability. The design system favors visible controls at this density.
- **Showing the owner's email as text** (the prior form) — the monogram avatar is the
  conventional identity anchor; the email is retained as the avatar's accessible name
  and tooltip, so nothing is lost while the banner reads cleaner.
- **Hashing the email into a per-owner avatar color** — one owner per box (bounded
  scope); a single accent fill is on-palette in light and dark and sufficient.
- **Dropping the raw MCP URL** — it is still useful for hand-wiring a non-suite MCP
  client; it is retained on every row (now with a copy control), only its
  presentation changes from a table column to a copyable `.meta` value.
- **Keeping the `<table>` / adding Carbon structured-list markup** — the dashboard
  already has one list idiom (`.list`/`.row`); a second table language for one list
  is the inconsistency being removed, and literal Carbon component markup is more
  than this bounded surface needs. Reuse the chrome that already exists.
- **Keeping the section intro paragraphs** — with the name linked and the URL made
  copyable, the prose ("Paste the line…", "The raw resource URLs, for wiring…") only
  restates what the controls already show; it is removed rather than maintained.
