# Decision 5 ‚Äî Landing composition: service-name links, email‚Üíprofile nav, sign-out

**Decision.** With token/grant administration moved away (D3, D4), the
landing/home page is composed of three things and gains its navigation:
**install instructions** (kept), the **service list** (each name now a **link** to
that service's page), the owner's **email** (now a **link** to the profile page),
and **sign-out** (stays here).

- **Service-name links.** Each row in the service list links the service **name**
  to that service's own human landing page at its mount root `/srv/<svc>/` ‚Äî not to
  the raw MCP resource URL. The mount is already known: `inventory.Read` yields each
  service's `Mount` (e.g. `/srv/crm/`). `serviceRows` (in `inventory.go`/the index
  view builder) gains a mount-derived **href** field (the `Mount` value itself,
  which already carries its trailing slash) alongside the existing display name and
  raw MCP `URL`, and the landing template renders the name as
  `<a href="{{.Href}}">‚Ä¶</a>`. The raw MCP `URL` column (for hand-wiring any client)
  is retained as reference text; only the **name** becomes a link.
- **Email ‚Üí profile.** The owner's email shown at the top of the landing is a link
  to the profile route (`<a href="/profile">{{.Owner}}</a>`). This is the only path
  the product offers to reach the profile page.
- **Sign-out stays on the landing.** The existing `POST /logout` form/button remains
  on the landing/home page (it does **not** move to profile). Signing out is an
  action you take from where you land.
- **Install instructions stay.** The connect-your-agent section (the Claude/Codex
  `curl ‚Ä¶ | bash` snippets) is unchanged and remains on the landing.

This decision governs only the **landing** page's composition. It does not change
the `/services` JSON inventory endpoint, the install scripts, or `mcpResourceURL`.

**Rejected.**
- **Linking the service name to the raw MCP resource URL** (`‚Ä¶/srv/<svc>/mcp`) ‚Äî
  that endpoint is a bearer-gated JSON-RPC surface, not a human page; the click
  target is the service's human landing at the mount root (product Contractual
  constant).
- **Moving sign-out to the profile page** ‚Äî sign-out belongs where the owner lands;
  burying it on a settings page hides the most common exit (product promise).
- **A separate top-nav bar with a "Profile" button** ‚Äî the email *is* the affordance
  (a familiar "click your identity to manage your account" pattern); a second
  control is breadth this change does not buy.
- **Dropping the raw MCP URL reference column** ‚Äî it is still useful for hand-wiring
  a non-suite MCP client; only the name gains a link.

**Verification.**
- R-DB12-LINK ‚Äî On the landing/home page, each service row links its **name** to
  that service's mount root `/srv/<svc>/` (the `Mount` href), not to the MCP
  resource URL.
- R-DB13-MAIL ‚Äî On the landing/home page, the owner's email is a link whose target
  is `/profile`.
- R-DB14-SOUT ‚Äî The sign-out control (the `POST /logout` form) is present on the
  landing/home page.
- R-DB15-INST ‚Äî The landing/home page retains the connect-your-agent install
  instructions (the Claude and Codex install snippets).
