# Decision 4 — Retire the stable sibling path and its hand-placed artifacts

**Decision.** Once every reader goes through `current` (D1–D3), the stable sibling
path `<app>/etc/manifest.env` (on the box: `/opt/<app>/etc/manifest.env`) is dead
and is removed everywhere, with **no fallback** left behind:

- No source reader or tool references the sibling path. `appkit/inventory`,
  `bin/registry`, and `bin/start` read only `etc/current/manifest.env`. The stale
  layout comment on `Layout.ManifestPath` ("regenerated by `<app> manifest` on
  every swap") is corrected or the accessor removed, since nothing maintains it.
- On the live box, the hand-placed bridge symlink `/opt/crm/etc/manifest.env`
  (added mid-investigation) and any leftover stale sibling files (e.g. ledger's
  hand-copied `/opt/ledger/etc/manifest.env`) are deleted. They are no longer read.
- **`opsctl` is unchanged.** It already publishes `etc/current/manifest.env`; with
  readers going through `current`, prod is correct with no deploy-tool edit.

**Operational sequencing (box).** The reader change (D1) must be **deployed** to
the box before the bridge is removed — otherwise crm drops off the list again in
the window between. And because the dashboard's OAuth-AS resource list is derived
**only at startup** (`deriveResources`, unlike the per-request `/services` list),
the dashboard must be **restarted** after crm's manifest becomes readable for
`/srv/crm/mcp` tokens to actually mint and introspect — until then the service
appears in the list but its MCP still `401`s at nginx. This restart requirement is
recorded here as the design consequence; the plan carries it as an explicit step.

**Rejected.**
- *Keep the sibling as a compatibility shim.* A dead-but-present path regresses on
  the next `opsctl convert` (which moves the old sibling into the version dir) and
  re-hides the failure mode. Retirement must be complete.

**Verification.**
- R-YSVR-SL00 — a service directory that has **only** `etc/manifest.env` and no
  `etc/current/` is **not** listed by `inventory.Read` (and not resolved by
  `bin/registry`). Proves the sibling path is fully retired with no silent fallback
  keeping it alive.
- R-YU3O-6CQP — on the box's real prod layout, with the hand-placed bridge symlink
  `/opt/crm/etc/manifest.env` **removed**, the deployed dashboard's `GET /services`
  still lists `crm` (resolved through `/opt/crm/etc/current/manifest.env`). Proves
  the end state stands on the `current` path alone, not on the bridge — exercised
  against the live box after deploying the D1 reader, the real substrate whose
  whole point is that no sibling file exists.
