# Decision 11 — Event-routing conformance: the chassis compiles and plumbs the family/kind revision

> **External ordering (operator-sequenced, not built by any appkit phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry` (replacing `EventType`),
> `Registry.Detail(kind)`/`UnknownKindError`, `routing.Key`, the kind/subject
> envelope and canonical-key SSE framing, and `consumer.Event{Kind, Subject}`
> (no `Type`) — specified in `eventplane/project/design/` (its D1–D4) and
> realized by **eventplane plan phases 01–04, which must be BUILT before
> appkit's conformance phase runs**. appkit is the **hinge**: every service
> compiles against appkit, so in build order appkit's conformance comes
> **after eventplane's phases 01–04 and before every service's own
> conformance phase** — until appkit lands, no service conformance can build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and
selection is one glob over the canonical `source:kind<subject>` key. It is a
hard cutover — no compatibility period — inside the no-live-data migration
window. appkit imports eventplane in `appkit.go`, `verbs.go`, `feed/feed.go`,
`mcp/mcp.go`, `server/server.go`, and their tests; after eventplane's cutover,
appkit does not compile until it conforms.

**What appkit couples to today (ground truth, not vacuous).**
- `Spec.Events outbox.Registry` (`appkit.go`) — threaded through `verbs.go`
  into `feed.Options.Registry` (Append-time gating) and
  `server.Options.Events` / `mcp.Options.Events` (reflection), with the
  `Publishes func() outbox.Registry` live-provider preference mirrored at each
  hop and exposed as `Router.Events()`/`Router.Publishes()`.
- The chassis `reflection` tool (`mcp/mcp.go`) — `event_type` detail parameter,
  `Registry.Detail(type)`, `*outbox.UnknownEventTypeError`. Its revision is
  **D9's** (rewritten in place); this Decision carries the rest of the module.
- `Spec.Consumers`/`Spec.Subscriptions` (`[]consumer.Subscription`) — appkit
  declares, concatenates, and reflection-renders subscriptions but never calls
  `Subscription.Match` and never interprets `Filter`; the struct's fields
  (`Source`, `Filter`, `Description`, `Handler`) are unchanged by the
  revision, so this seam is compile-stable (filters become key globs, opaque
  to appkit).
- Test fixtures: `mcp/mcp_test.go` and `server/server_test.go` build
  `outbox.Registry`/`outbox.EventType` literals with `Type:`;
  `consumers_test.go` hand-frames SSE envelopes carrying a `"type"` key —
  under the revised consumer contract an envelope with empty `kind` is
  engine-skipped, so those fixtures would hang the consumer tests, not merely
  fail them.

**Decision.** appkit conforms as pure plumbing — it adopts the revised types
at every seam and adds **no** routing policy of its own:

1. **`Spec.Events` keeps its type, which changes meaning.** The field stays
   `outbox.Registry`; the registry is now family-based. All pass-through hops
   (`verbs.go` → `feed.Options.Registry`, `server.Options.Events`,
   `mcp.Options.Events`, the `Publishes` providers, `Router.Events()`) are
   type-identical and survive as-is; their doc comments (e.g. `appkit.go`'s
   "rejects any unregistered `ev.Type`", `feed/feed.go`'s "event-type
   registry") are updated to speak kinds/families.
2. **The `reflection` tool speaks kinds** — specified in D9 (rewritten in
   place); realized together with this Decision in one phase.
3. **No outbox migration in appkit.** appkit is a library: it owns no service
   database and no outbox table (its only embedded SQL is the
   `internal/testmigrations` chassis fixture, which has no outbox). Each
   producing service re-applies the revised `outbox.SchemaSQL` through its own
   migration runner in its own spec — verified here as a structural fact, not
   an id.
4. **Test fixtures convert, proving the plumbing on real substrates.** Registry
   literals become `Family{Kind, Subject, Description, Sample}` values;
   hand-framed consumer-test envelopes carry `kind`/`subject` (and a
   canonical-key `event:` line) instead of `type`; the producer path is proven
   end-to-end through appkit's own seams (below). No occurrence of
   `outbox.EventType`, `UnknownEventTypeError`, or the `event_type` tool
   parameter survives anywhere in `appkit/` outside `project/`.

**Rejected.**
- **An appkit-side compatibility shim (accepting `event_type` or re-exporting
  `EventType`)** — a dual-format period, explicitly forbidden by the suite
  decision; it would also make appkit the only module speaking both dialects.
- **appkit-owned outbox migration or DDL guard** — services own their DBs and
  their DDL drift guards (gmail D18's pattern); a library migration has no
  database to run against.
- **Conforming appkit and eventplane in one combined phase** — crosses the
  workspace scope boundary (each `project/` governs only its own tree); the
  ordering is expressed as the external-ordering banner instead.
- **Interpreting `Subscription.Filter` in appkit (validating filters against
  upstream families at startup)** — validation-by-family is
  `Registry.CouldMatch`, a producer-side/service-side concern (prompts' spec);
  the chassis renders filters opaquely today and keeps doing so.

**Verification** (real substrates throughout: a real `t.TempDir()` SQLite
database via `modernc.org/sqlite` under the real eventplane outbox; the wire
claim through the handler appkit actually mounts, over `httptest` — the
existing feed/consumer harness style).

- R-7JFR-R31S — through `feed.Start` with a family registry declaring only
  kind `create`, `Append` on the returned `Producer.Outbox` of an event with
  kind `delete` fails with an error naming the declared kinds, and an event
  with kind `create` and a `/`-rooted subject succeeds (read back by SQL from
  the real database); with an empty registry, an undeclared kind is accepted
  (gating stays opt-in through appkit's pass-through, exactly as before).
- R-7LVK-IMJ6 — an event appended through the `feed.Start` producer (kind
  `create`, subject `/bills/a.pdf`, source per `Options.Source`) and served by
  the returned `Producer.Handler` over an `httptest` connection frames
  `event: <source>:create/bills/a.pdf` (the canonical key) with a `data:`
  JSON envelope carrying `kind` and `subject` keys and **no** `type` key —
  appkit's producer mount carries the revision end to end.
