# Decision 2 — `bin/registry` resolves through `current`

**Decision.** The repo-root shell reader `bin/registry` — which resolves a service
name to its loopback addr / feed URL / MCP resource by reading the manifest — moves
onto the same `current`-relative path as `appkit/inventory` (D1). Both the
single-service lookup and the list loop change:

```sh
# manifest_path <name>
echo "${REGISTRY_ROOT}/$1/etc/current/manifest.env"   # was ${REGISTRY_ROOT}/$1/etc/manifest.env

# list loop
for path in "$REGISTRY_ROOT"/*/etc/current/manifest.env; do ...
```

`REGISTRY_ROOT` still defaults to `/opt`. This keeps the two manifest readers
(Go inventory, shell registry) reading the identical on-disk path so they never
diverge on which services exist.

**Rejected.**
- *Leave `bin/registry` on the sibling path.* Then the two readers disagree: the
  dashboard (through `current`) and `bin/registry` (sibling) would list different
  service sets, and registry would break the same way inventory did on the next
  fresh setup. The whole point of D1 is one path for everyone.

**Verification.**
- R-YQFZ-11IM — against a fixture `REGISTRY_ROOT`, `bin/registry` resolves a
  service whose manifest is at `<root>/<name>/etc/current/manifest.env` (addr /
  feed / mcp lookups and the list subcommand all succeed), and does **not** resolve
  one whose manifest is only at the sibling `<root>/<name>/etc/manifest.env`.
  Exercised by `bin/registry.test.sh` with real fixture directories and a real
  `current` symlink.
