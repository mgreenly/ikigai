# Decision 9 — Chassis-owned standard tools: `health` and `reflection`

**Problem.** Every service's MCP surface carries the same two non-domain tools
— `health` (the health envelope, same bytes as the HTTP `/health` route) and
`reflection` (the event-plane self-description over the registry and
subscriptions) — re-implemented per service from the same Router accessors. They
are pure chassis: a change to the envelope or the reflection schema is a
twelve-edit change today.

**Decision.** `appkit/mcp.New` (D8) registers `health` and `reflection`
automatically from its `Options`; services declare only domain tools. The
implementations replicate the per-service semantics exactly:

- **`health`** — no input; returns `server.Envelope(version, service, details)`
  as a JSON text result, where `details` comes from `Options.Health` (nil →
  `{}`), so the MCP tool and the HTTP `/health` route render from the single
  envelope builder they already share.
- **`reflection`** — speaks the **family-based** registry of the event-routing
  revision (`eventplane` D3: `outbox.Registry` is `[]outbox.Family`, each
  `{Kind, Subject, Description, Sample}`). Optional `kind` string input — the
  detail key is the family's **kind**, matching `Registry.Detail(kind)`; the
  old `event_type` parameter is gone (hard cutover, no alias).
  - Without `kind`: returns the event-graph index — `publishes` rendered from
    `Options.Publishes()` when non-nil (the dynamic-producer preference, e.g.
    cron) else `Options.Events`, each entry the family's
    `{kind, subject, description}` (`Registry.Index()`, declaration order,
    subject the human description of the subject shape) — and `subscribes`
    rendered from `Options.Subscriptions` (nil → empty), unchanged:
    `{source, filter, description}` per subscription, the filter an opaque
    string appkit renders and never interprets.
  - With `kind`: returns that family's detail
    (`Registry.Detail(kind)` — `{kind, subject, description, schema, example}`,
    schema reflected from the Sample's type and example marshaled from its
    value, so the two cannot drift); an undeclared kind returns an `isError`
    result whose message names the declared kinds (built from eventplane's
    typed `*outbox.UnknownKindError`, which replaces `UnknownEventTypeError`).
- Both appear in `tools/list`; the `reflection` descriptor's `inputSchema`
  declares the optional `kind` property (and no `event_type`), its description
  updated to match.
- The names are **reserved**: D8's `New` rejects a declared Tool named `health`
  or `reflection`, so chassis and service can never silently disagree about who
  answers.

**Rejected.**
- **Leaving health/reflection service-side (plumbing-only hoist)** — keeps two
  copy-pasted tools in every service and re-opens drift on the reflection
  schema; the inputs (Envelope, Registry, Subscriptions) are all chassis-held
  already.
- **A services-may-override seam** — no service has ever legitimately varied
  either tool; an override seam is a hypothetical-future flag, rejected on
  principle (complexity only when forced).
- **Folding reflection into health** — different axes (liveness vs event
  graph), different consumers; the suite's tool vocabulary already treats them
  as distinct.
- **Keeping `event_type` as the detail parameter (or accepting both)** — the
  routing revision deletes the flat type; a kind is not a type string, and an
  alias is a dual-format period the suite decision explicitly forbids. Detail
  is keyed exactly as `Registry.Detail` is: by kind.
- **Keying detail by full canonical key or `kind+subject`** — subjects are
  dynamic (cron schedule names, dropbox paths) and not enumerable; the family
  — hence its detail — is identified by kind alone (eventplane D3).

**Verification** (through the D8 `ServeHTTP` seam, with real family-shaped
`outbox.Registry` and `consumer.Subscription` values).
- R-ML2U-CBQM — with zero declared tools, `tools/list` still lists `health` and
  `reflection`; `tools/call health` returns a result embedding the
  `server.Envelope` shape — `status`, `service`, `version` matching Options,
  and `details` from the injected reporter (a reporter returning
  `{"queue": 3}` is visible in the result).
- R-7EK6-8030 — `tools/call reflection` (no arguments) over a family registry
  returns `publishes` as one `{kind, subject, description}` entry per family
  in declared order (a two-family registry with kinds `create`/`delete` and a
  subject description `/<mirror path>` reports exactly those, verbatim) and
  `subscribes` listing the injected subscriptions' `{source, filter,
  description}`; with `Options.Publishes` set, its returned registry is used
  instead of `Options.Events` (a kind present only in the live provider
  appears; one present only in Events does not).
- R-7FS2-LRTP — `tools/call reflection` with `kind` naming a declared family
  returns that family's detail: `kind`, `subject`, `description`, plus a
  `schema` and `example` that agree (every example field appears in the
  schema — the Sample-derived no-drift property preserved).
- R-7GZY-ZJKE — `tools/call reflection` with an undeclared `kind` returns an
  `isError` result whose message contains the declared kind names — not a
  crash, not an empty success; the error path goes through eventplane's typed
  `*outbox.UnknownKindError` (an `errors.As` match), not string matching.
- R-7I7V-DBB3 — the `reflection` entry in `tools/list` declares an
  `inputSchema` with a `kind` property and **no** `event_type` property — a
  build that renames the handler parameter but leaves the advertised schema
  stale fails this.
