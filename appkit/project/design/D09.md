# Decision 9 — Chassis-owned standard tools: `health` and `reflection`, structured

**Problem.** Every service's MCP surface carries the same two non-domain tools
— `health` (the health envelope, same bytes as the HTTP `/health` route) and
`reflection` (the event-plane self-description over the registry and
subscriptions) — pure chassis, re-implemented per service before D8/D9 hoisted
them. This revision: under the structured-results contract (D8) the standard
tools must themselves conform — declared `outputSchema`, `structuredContent`
results, coded errors — or the chassis would exempt itself from the contract
it imposes.

**Decision.** `appkit/mcp.New` (D8) registers `health` and `reflection`
automatically from its `Options`; services declare only domain tools. Both are
full citizens of the structured contract:

- **`health`** — no input; returns `StructuredResult` of
  `server.Envelope(version, service, details)` plus the caller identity
  fields (`owner_email`, `client_id`), where `details` comes from
  `Options.Health` (nil → `{}`). Its `tools/list` descriptor declares an
  `outputSchema`: an object with `status`, `service`, `version`,
  `owner_email`, `client_id` string properties and an open-object `details`
  (service-specific keys, deliberately unconstrained).
- **`reflection`** — speaks the **family-based** registry of the
  event-routing revision (`eventplane` D3), keyed by **kind**; the semantics
  of both forms are unchanged from the built revision:
  - Without `kind`: the event-graph index — `publishes` rendered from
    `Options.Publishes()` when non-nil else `Options.Events`
    (`Registry.Index()`, declaration order, each family
    `{kind, subject, description}`), and `subscribes` from
    `Options.Subscriptions` (nil → empty), `{source, filter, description}`
    per subscription, the filter an opaque string appkit never interprets.
  - With `kind`: that family's detail (`Registry.Detail(kind)` —
    `{kind, subject, description, schema, example}`, schema reflected from
    the Sample's type and example marshaled from its value, so the two cannot
    drift).
  - Both forms return via `StructuredResult`. The declared `outputSchema` is
    a `oneOf` of the two result shapes (index object | detail object) — one
    tool, two forms, both machine-consumable.
  - An undeclared kind returns `ErrorResult(ErrValidation, msg)` where `msg`
    names the declared kinds (built from eventplane's typed
    `*outbox.UnknownKindError`, an `errors.As` match, not string matching) —
    a machine caller branches on `structuredContent.code == "validation"`.
- Both appear in `tools/list` with their `outputSchema`s; the `reflection`
  descriptor's `inputSchema` declares the optional `kind` property.
- The names are **reserved**: D8's `New` rejects a declared Tool named
  `health` or `reflection`, so chassis and service can never silently disagree
  about who answers.

**Rejected.**
- **Exempting the standard tools from the structured contract** — the chassis
  would be the one non-conforming tool provider in the suite, and machine
  callers (`health` is the natural liveness probe for a scripts run) would
  need a prose parser for exactly two tools.
- **Two separate reflection tools (index + detail) to avoid the `oneOf`
  schema** — doubles the reserved-name surface and breaks every existing
  caller for a schema nicety; `oneOf` is standard JSON Schema and the two
  forms are already discriminated by the presence of the `kind` argument.
- **Leaving health/reflection service-side; a services-may-override seam;
  folding reflection into health; `event_type` aliases; keying detail by full
  canonical key** — all rejected in the built revisions, unchanged.

**Verification** (through the D8 `ServeHTTP` seam, with real family-shaped
`outbox.Registry` and `consumer.Subscription` values).
- R-ML2U-CBQM — with zero declared tools, `tools/list` still lists `health` and
  `reflection`; `tools/call health` returns a result embedding the
  `server.Envelope` shape — `status`, `service`, `version` matching Options,
  and `details` from the injected reporter (a reporter returning
  `{"queue": 3}` is visible in the result).
- R-WWZ1-HCPK — `tools/call health` carries `structuredContent` whose
  `status`/`service`/`version`/`details` equal the parsed text block's (the
  mirrored renderings agree), and the `health` descriptor in `tools/list`
  declares an `outputSchema` naming those properties — a build that leaves
  the standard tools text-only fails this.
- R-7EK6-8030 — `tools/call reflection` (no arguments) over a family registry
  returns `publishes` as one `{kind, subject, description}` entry per family
  in declared order (a two-family registry with kinds `create`/`delete` and a
  subject description `/<mirror path>` reports exactly those, verbatim) and
  `subscribes` listing the injected subscriptions' `{source, filter,
  description}`; with `Options.Publishes` set, its returned registry is used
  instead of `Options.Events` (a kind present only in the live provider
  appears; one present only in Events does not).
- R-7FS2-LRTP — `tools/call reflection` with `kind` naming a declared family
  returns that family's detail: `kind`, `subject`, `description`, plus a
  `schema` and `example` that agree (every example field appears in the
  schema — the Sample-derived no-drift property preserved).
- R-WY6X-V4G9 — both reflection forms return `structuredContent` (index:
  `publishes`+`subscribes`; detail: the family object) equal to their parsed
  text blocks, and the `reflection` descriptor declares an `outputSchema`
  covering both forms — asserted on the descriptor carrying a schema and on
  both calls' `structuredContent`, not on schema-validation machinery.
- R-7GZY-ZJKE — `tools/call reflection` with an undeclared `kind` returns an
  `isError` result whose message contains the declared kind names — not a
  crash, not an empty success; the error path goes through eventplane's typed
  `*outbox.UnknownKindError` (an `errors.As` match), not string matching.
- R-WZEU-8W6Y — that undeclared-kind error additionally carries
  `structuredContent` of `{code: "validation", message: <msg>}` — a machine
  caller branches on the code without reading the message.
- R-7I7V-DBB3 — the `reflection` entry in `tools/list` declares an
  `inputSchema` with a `kind` property and **no** `event_type` property — a
  build that renames the handler parameter but leaves the advertised schema
  stale fails this.
