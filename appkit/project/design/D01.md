# Decision 1 — Manifest readers resolve *through* the per-app `current` symlink

**Decision.** `appkit/inventory.Read` reads each service's manifest at the path
that goes through the deploy `current` directory symlink, not the sibling file
beside it. The glob changes from the sibling path to the current-relative path;
everything else (parse via `appkit/manifest.Parse`, `MCP=true` filter, sort by
`Name`, per-manifest skip-on-error) is unchanged.

```go
// before: matches <root>/<svc>/etc/manifest.env  (a sibling of `current`)
// after:  matches <root>/<svc>/etc/current/manifest.env  (through `current`)
matches, err := filepath.Glob(filepath.Join(root, "*", "etc", "current", "manifest.env"))
```

`os.Open` transparently follows the `current` symlink (and the second hop
`current → <version>/`), so a repointed `current` is observed on the next read
with no reader change. The service `Name` still comes from the manifest body
(`env["APP"]`), so introducing the `current` path element does not affect name
derivation.

This is the whole fix's center of gravity: `appkit/inventory` is the single reader
behind the signed-in landing list, the public `GET /services` endpoint, **and**
the dashboard's OAuth-AS resource derivation (`deriveResources`) — all three move
together because they share this one function. The result is exactly one
indirection per app (`etc/current`), read the same way nginx already reads
`etc/current/nginx.conf`.

**Rejected.**
- *Keep the sibling `etc/manifest.env` and have a tool maintain it on every
  deploy.* This is the two-indirection design that caused the bug: the maintainer
  step was removed and nothing noticed. Reinstating a second maintained path is
  more moving parts to prove a file that `current` already points at.
- *Reader tries `etc/current/manifest.env`, falls back to `etc/manifest.env`.* A
  fallback keeps the dead path alive forever and hides exactly the "sibling went
  missing" failure we are fixing. No fallback.
- *Derive the service name from the path (`<svc>` element) instead of `APP`.*
  Unnecessary and now more fragile, since the immediate parent dir is `current`,
  not `<svc>`.

**Verification.**
- R-YO06-9I18 — `inventory.Read(root)` returns a service whose manifest lives at
  `<root>/<svc>/etc/current/manifest.env`, and does **not** return a service whose
  manifest exists only at the sibling `<root>/<svc>/etc/manifest.env` (with no
  `etc/current/`). Proves the read path goes through `current`, not the sibling.
- R-YP82-N9RX — `inventory.Read` reflects whatever version `current` targets, via
  the symlink: with `etc/current → verA/` where verA's manifest has `MCP=true` the
  service is listed; repoint `etc/current → verB/` where verB's manifest has
  `MCP=false` and the same `Read` no longer lists it. Proves it follows the live
  `current` symlink rather than a pinned or cached directory. (Real temp-dir +
  real symlinks as the substrate — the behavior is symlink resolution, so it is
  exercised against actual symlinks, not a fake filesystem.)
