# Decision 3 — Local dev runtime layout mirrors the box (`bin/start` stages a prod-shaped manifest root)

**Decision.** With D1/D2 reading through `etc/current`, the local dev layout must
present the same shape, so one code path serves both. Today `bin/start` points
`DASHBOARD_MANIFEST_ROOT` at the **repo root**, where manifests are the authored
source files `<svc>/etc/manifest.env` with no `current` symlink — which would force
exactly the sibling-path/fallback we are deleting. Instead, `bin/start` stages a
prod-shaped **runtime** manifest root under `tmp/`:

```
tmp/opt/<svc>/etc/<version>/manifest.env      # copied from <svc>/etc/manifest.env
tmp/opt/<svc>/etc/current  ->  <version>       # relative symlink, exactly like the box
```

and exports the readers at that root:

```
DASHBOARD_MANIFEST_ROOT=$RUN_DIR/opt
REGISTRY_ROOT=$RUN_DIR/opt
```

`<version>` is a stable local literal (the committed `<svc>/VERSION`, or `dev`) —
its exact value is irrelevant; only the `etc/<version>/manifest.env` + `current`
shape matters. The **source tree is untouched**: `<svc>/etc/manifest.env` remains
the authored file, and the prod-shaped tree is a disposable runtime artifact under
`tmp/`, wiped by `bin/stop --clean` with the rest of dev state.

**Rejected.**
- *Point the readers at the repo source tree.* No `current` there, so it only
  works with the fallback D1 rejects — local and prod would run different code
  paths, and "works locally" would stop proving "works on the box."
- *Commit a `current` symlink into each source `etc/`.* Drags box-deploy machinery
  into the source tree (a symlink pointing at `.`), coupling source layout to
  deploy layout. The runtime-staging approach keeps source clean.

**Verification.**
- R-YRNV-ET9B — after `bin/start` brings up the suite, the staged runtime root has
  `<svc>/etc/current/manifest.env` (a `current` symlink into a version dir) for
  every launched service, and the dashboard is pointed at that root; `GET /services`
  on the running local suite returns the full service set **including `crm`**.
  Exercised as a live smoke against the actually-running suite (the real substrate
  is `bin/start`'s staging + the real dashboard reading it), not a unit stub — the
  claim is that the launcher and reader agree end-to-end.
