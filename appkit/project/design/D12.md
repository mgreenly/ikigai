# Decision 12 — The loopback-only route class: `LoopbackOnly` + `Router.HandleLoopback`

**Problem.** The suite has a third class of route besides gated (identity
required) and ungated (public liveness/PRM): **loopback-only** — reachable by
on-box callers, never through nginx (`/feed`, content endpoints, dropbox's
filesystem API; `docs/content-plane-design.md`). Today that boundary is a
hand-copied handler guard living in the *consumers* of the chassis (eventplane
`outbox/feed.go:53`, dropbox `internal/dropbox/write.go`, prompts
`internal/prompt/content.go`), and it keys on `X-Owner-Email != "" ||
X-Forwarded-Proto != ""`. That predicate conflates "came from outside" with
"carries identity" — and the suite's identity contract
(`docs/structured-mcp-design.md`) now has loopback machine callers *asserting*
`X-Owner-Email`/`X-Client-Id` themselves, so the old guard would 404 exactly
the legitimate callers the contract admits.

**Decision.** The guard becomes a chassis-owned route class with the narrowed
predicate — "did this request cross nginx?" — answered by `X-Forwarded-Proto`
alone, which nginx stamps on every proxied request:

```go
package server // appkit/server

// LoopbackOnly wraps a handler that must never be reachable through the
// front door. A request carrying X-Forwarded-Proto (any value) transited
// nginx and is answered with a bare 404 — not 403, revealing nothing about
// what lives at the path. Identity headers are deliberately NOT part of the
// predicate: loopback callers assert X-Owner-Email/X-Client-Id themselves
// (docs/structured-mcp-design.md) and must pass.
func LoopbackOnly(next http.Handler) http.Handler
```

- **`Router.HandleLoopback(pattern string, h http.Handler)`** — the third
  registration class beside `Handle`/`RequireIdentity`: registers `h` at
  `pattern` wrapped in `LoopbackOnly`, so services declare content endpoints
  and filesystem APIs declaratively instead of hand-copying guard lines.
- **The chassis wraps its own `/feed` mount** in `LoopbackOnly` (the
  `opts.Feed` mount in `server.New`), making the mount, not the feed handler,
  own the boundary. eventplane's inline guard at `outbox/feed.go:53` becomes
  redundant and is deleted **in eventplane's own change** — outside this
  design's scope boundary, as are the service-side guard copies (each
  service's adoption phase replaces its hand guard with `HandleLoopback`).

**Rejected.**
- **A bare shared predicate services call inside handlers** — keeps the
  boundary a per-handler discipline instead of a mount-time property, which
  is exactly how the current drift-prone copies came to exist.
- **Keeping `X-Owner-Email` in the predicate** — 404s the caller-asserted
  identity contract's legitimate loopback callers; the header means "who this
  acts for", not "where this came from".
- **nginx `location … { return 404; }` as the mechanism** — nginx fragments
  are optional defence-in-depth per the content-plane design, never the
  mechanism; the handler guard must stand alone.
- **403 instead of 404** — a 403 confirms the path exists; the existing
  guards' 404 reveals nothing and is preserved.

**Verification** (via `httptest` through the real handler/mux — the substrate
that can falsify routing).
- R-X0MQ-MNXN — a request to a `LoopbackOnly`-wrapped handler carrying
  `X-Forwarded-Proto` (both `https` and any other non-empty value) is answered
  `404` and the inner handler is **not** invoked (a recording handler stays
  uncalled).
- R-X1UN-0FOC — a request **without** `X-Forwarded-Proto` reaches the inner
  handler even when it carries `X-Owner-Email` and `X-Client-Id` — the
  discriminating property against the old guard; a build that keys on
  identity headers fails this.
- R-X32J-E7F1 — a route registered via `Router.HandleLoopback` on a real
  `server.New` server behaves per the two behaviors above at its pattern
  (guarded through the real route table, not a unit-tested wrapper alone).
- R-X4AF-RZ5Q — with `Options.Feed` set, `GET /feed` with `X-Forwarded-Proto`
  returns `404` and without it reaches the feed handler — the chassis mount
  itself is wrapped (a build that leaves the mount bare and relies on the
  handler's own guard fails once eventplane's inline guard is gone).
