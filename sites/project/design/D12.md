# Decision 12 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** sites's landing page and its Carbon design assets were embedded in
the binary (`//go:embed` in `sites/internal/web/`), like every service's. The
suite is reverting that: web assets are release artifacts that belong on disk —
visible and diffable in the versioned release tree, editable in dev without a
rebuild, served by one chassis mechanism instead of twelve drifting copies. crm
proved the conversion; notify followed; sites converts to the same shape. This
Decision is the service-side half of appkit's chassis design (its D5 www-root
resolution, D6 `appkit/web`, D7 `Spec.WWW`/`Router.WWW()` — consumed here as a
fixed external contract). It **rewrites the mechanism** D1 (landing render), D2
(landing route), D3 (Carbon assets), and D8 (FOUT self-serve) describe: the page
*content*, the route pattern, the Carbon fidelity, and the swap-free font
loading are all unchanged — only the substrate moves from embed to `share/www`
and from a sites-side handler to the chassis.

**Decision.** sites serves its landing/static web surface from the on-disk www
root.

1. **The files move (same bytes).** `sites/internal/web/landing.html` →
   `sites/share/www/landing.html`; `sites/internal/web/static/` →
   `sites/share/www/static/` — `tokens.css` and the four woff2 fonts
   (`space-grotesk.woff2`, `ibm-plex-sans.woff2`, `ibm-plex-mono-400.woff2`,
   `ibm-plex-mono-500.woff2`) verbatim. `bin/ship sites` bundles `sites/share/`
   into the release `share/<version>` tier; on the box the chassis resolves the
   root to `share/current/www`, in dev to `./share/www` or the `SITES_WWW_PATH`
   override.

2. **The Spec opts in.** `sitesSpec()` in `cmd/sites/main.go` sets `WWW: true`.
   The chassis loads the site at serve (failing startup loudly if the root is
   missing), auto-mounts `GET /static/`, and exposes the loaded site as
   `rt.WWW()`.

3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` (D2) becomes a few-line handler over
   `rt.WWW().Render(w, "landing.html", …)` (D1/D2 carry the exact shape). The
   route pattern (`GET /{$}`, ungated in-process — nginx is the session gate,
   D4) and its exact-root semantics are unchanged. The service-side
   `GET /static/` mount (the old `web.StaticHandler`) is **deleted** — the
   chassis owns it.

4. **`sites/internal/web/` is deleted** — `embed.go`, `web.go`
   (`LandingHandler`/`StaticHandler`), `landing.html`, `static/`, and
   `web_test.go`. The landing/asset tests move to the composition-root package
   (`cmd/sites`), loading the repo-real `share/www` tree via `appkit/web` (a
   relative path from the package dir) so they exercise the exact files that
   ship. The nginx fragment tests (`nginx_test.go`) relocate to `cmd/sites`
   unchanged in their assertions — the fragment itself does not change (its
   `/srv/sites/static/` location still proxies to upstream `/static/`, now
   served by the chassis static mount rather than `web.StaticHandler`).

5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_sites` exports
   `SITES_WWW_PATH="$repo/sites/share/www"` beside its existing `SITES_DB_PATH`
   / `SITES_ROOT` exports, so a repo-root-launched dev binary finds the source
   tree. This one line crosses the `sites/` boundary and is verified by the live
   `bin/start` smoke, **not** the Go suite.

6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6/D7/D8), the MCP surface (D13), the domain store/layout/mirror
   wiring, and the migrations are untouched. The embedded migrations
   (`internal/db`) stay embedded — they are not web assets (D14 governs that
   package).

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work exists to complete;
  embedded assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is sites's own nginx-served user
  content (the `public/`/`private/` disk tiers, backed up with `state/`); the
  landing/Carbon UI assets are release artifacts and live in the shipped `share`
  tier (appkit D5 records the full argument). The two are deliberately distinct:
  `share/www/static` is the service's **own** design assets (chassis-served),
  `state/www/{public,private}` is the **user's** hosted websites (nginx off
  disk).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition** — two sources
  of truth; the conversion is atomic within the service.

**Verification** (over the repo-real `sites/share/www` tree; D1's R-LAND-\*,
D2's R-ROUT-\*, D3's R-ASST-\*, D7's R-HOME-9S3W, and D8's R-6\*-\*/R-675A ids
continue to pin the page/asset behaviors through the new substrate).
- R-0SF5-VPQF — an `appkit/web` Site loaded from `sites/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html`
  with an injected service and version visible in the output — proving the
  shipped template, not an embedded copy, is the page source.
- R-0TN2-9HH4 — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  sites-side static handler registered — proving the assets ship on disk and the
  chassis serves them.
