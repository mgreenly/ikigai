# Decision 3 — sites's own Carbon design assets (shipped in `share/www/static`)

**Decision.** sites ships its **own** copy of the Carbon design assets and serves
them from its own mount; the landing page references those assets, never a
cross-service URL. Concretely:

- A `sites/share/www/static/` directory holds a **vendored copy** of
  `design/tokens.css` (the Carbon CSS custom properties) and the woff2 fonts the
  Carbon type pairing needs: `space-grotesk.woff2` (display), `ibm-plex-sans.woff2`
  (body), `ibm-plex-mono-400.woff2` + `ibm-plex-mono-500.woff2` (mono labels) —
  the same set every service carries.
- The directory ships in the release `share` tier (`bin/ship sites` bundles
  `sites/share/`; the box reads it at `share/current/www`, D12) and is served
  under the stable in-service path — `GET /static/…` — by the **chassis static
  mount** (appkit's `Spec.WWW` integration), with correct content types
  (`text/css; charset=utf-8` for `tokens.css`, `font/woff2` for the fonts). No
  sites Go code serves assets.
- `tokens.css` declares its `@font-face` rules pointing at the **service-relative**
  font paths (D8; `/srv/sites/static/fonts/…` as seen by the browser once nginx
  re-adds the mount prefix), and the landing template links `tokens.css` from the
  same `/static/` path. No asset is fetched from the dashboard or any other
  service at runtime.

**Why these design assets do not collide with the static tiers.** The landing
assets are served by the sites **process** (the chassis static mount) under the
service's `/static/…` path — reached via the session-gated `/srv/sites/static/`
proxy (D8), entirely separate from the disk-backed `public/`/`private/` tiers
(`state/www`) nginx serves directly. The `/static/` prefix is a path the sites
binary owns; it is not one of the `state/www` static-tier folders.

**Why each app ships its own copy.** The dashboard precedent
(`dashboard/ui/static/tokens.css` is a *vendored* copy precisely because a shared
design tree cannot be reached across the module boundary) makes this
non-negotiable: `module sites` owns its own assets. Shipping the tokens + fonts in
`sites/share/www/static/` keeps sites a self-contained single release with no
runtime dependency on another app's assets — the whole point of the
one-binary-per-service model. Its assets deploy, swap, and roll back with its
binary.

**Carbon fidelity.** The page consumes the **semantic** Carbon tokens
(`--color-*`, `--space-*`, type tokens), not raw ramps, per `design/carbon.md`:
monochrome neutrals; blue `#2563EB` reserved as the only signal color (used here
only for a focus ring / link if present, never decoratively); 4px spacing grid;
tight 4px radius; hairline borders over shadow. The landing card is minimal and
on-system: service name in **Display** (Space Grotesk, 56/1.04/700), version in
the **Mono · Label** token (IBM Plex Mono, 12/1.4/500), centered within the
1120px max content width.

**Staleness note (vendored copy).** Because `tokens.css` and the fonts are a
vendored copy, they can drift from `design/` over time. That is an accepted cost
of per-service independence (the dashboard lives with the same), and is out of
scope to automate here; if the design system changes, the copy is re-vendored as
a new plan phase. v1 copies the current `design/tokens.css` and the four woff2
fonts verbatim.

**Rejected.**
- **Linking the dashboard's `/static/tokens.css` at runtime** — couples sites to
  the dashboard's asset URLs and breaks the self-contained one-release model; a
  dashboard asset rename would silently unstyle sites.
- **A shared design module imported by every app** — the suite deliberately keeps
  each app a standalone module and a standalone release; vendoring is the
  established pattern (dashboard).
- **Embedding the assets in the binary (`//go:embed`)** — the suite's original
  approach, deliberately reverted (D12): embedded assets are invisible on the
  box, cost a rebuild for any dev tweak, and re-implement per service what the
  shipped `share` tier plus the chassis static mount do uniformly.
- **Inlining all CSS into the HTML and shipping no fonts** — loses the woff2 type
  pairing (the page would fall back to system fonts and look off-system).
- **Serving the fonts from a CDN** — the box is single-tenant and may run with no
  external egress guarantees; assets must be local.

**Verification** (drive the chassis static mount over the repo-real
`sites/share/www` tree with httptest — D12).
- R-ASST-3H7N — a `GET /static/tokens.css` is served with status `200` and a CSS
  content type (`text/css`).
- R-ASST-5K9Q — the landing template's stylesheet link and the page's font
  references resolve to sites's **own** `/static/` path (the rendered HTML
  contains the app-local asset path and contains **no** cross-service/dashboard
  asset URL).
- R-ASST-7M2S — the woff2 font assets are shipped and served: a `GET` for a font
  path (e.g. `/static/fonts/space-grotesk.woff2`) returns `200` with a
  `font/woff2` content type.
