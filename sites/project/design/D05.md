# Decision 5 — Docs state current truth: state the standardized landing card (no "no UI" claim to purge)

**Decision.** The suite-wide doctrine for this change is *docs state current
truth; purge what the change falsified, rewrite in place, no archaeology, no
footnotes.* For most services that means deleting a stale "**no UI**" line. **sites
is the exception:** it has **no `AGENTS.md`/`CLAUDE.md` and no "no UI" claim to
purge** — it already documents itself as a web-page host. Its self-description
lives in the `cmd/sites/main.go` package doc comment, whose opening line already
reads *"Command sites is the loopback-only static-website host behind nginx"* and
which already enumerates the PRM doc, the bearer-gated MCP endpoint, and the
public/private static tiers.

What that self-description does **not** yet mention is the **new standardized,
dynamic landing card** at the bare mount root (D1–D4). So this Decision's doc work
is **additive, not a purge**: amend the `cmd/sites/main.go` package doc comment to
state that, alongside its existing surfaces, sites now serves a Carbon-styled
**human web landing page** at the bare mount root `/srv/sites/`, rendered
in-process and gated by the dashboard browser **session** (the same
`/_session-authn` gate the private static tier already uses).

**What changes (doc/comment edits only — no behavior):**

- In `cmd/sites/main.go`'s **package doc comment** (the `// …` block above
  `package main`), add the dynamic landing card to the enumerated surfaces: sites
  serves the PRM doc, the bearer-gated MCP endpoint, the public/private static
  tiers from disk, **and** a session-cookie-gated dynamic landing card at the
  mount root. Keep every still-true statement (loopback-only, header-trust, "no
  token logic," not an event-plane producer) exactly as is.
- Do **not** add a changelog, a "previously…", or a footnote. State the present
  as if it were always so.
- There is **no `AGENTS.md`/`CLAUDE.md`** in sites and **no "no UI" string** to
  remove — so, unlike the other services' Decision 5, this phase removes nothing;
  it only adds the one truthful sentence.

**Why this is its own phase, performed by the build loop — not by the doc
author.** The doc author (this design+plan workspace) deliberately does **not**
edit `cmd/sites/main.go` now: code and docs must evolve **together**, so the
self-description update is sequenced as a plan phase that the build loop performs
**after** the landing page exists (D1–D4 shipped). Doing it earlier would make the
comment claim a card the code does not yet render. (This is the one Decision-5
edit that lands in a Go file's *comment* rather than a markdown doc, because
sites's doctrine lives in that package comment — it changes no code, only the
documentation prose.)

**Scope guard.** Only the missing landing-card sentence is added. sites's other
self-description — the static-website domain, the header-trust model, "no token
logic," the loopback bind, "not an event-plane producer," the MCP surface — is
**unchanged and still true**; this Decision does not touch it, and it changes no
executable code.

**Verification.** This is a **documentation** Decision with no Go behavior change,
so it mints no `R-XXXX-XXXX` ids; it is realized by a **structural** plan phase
whose done bar is a content check rather than an id-tagged test:

- `cmd/sites/main.go`'s package doc comment mentions that sites serves a human web
  **landing page** at the bare mount root `/srv/sites/`, gated by the dashboard
  **session**, alongside its existing surfaces.
- No "no UI" claim is introduced anywhere (there was none to purge), and no
  changelog/footnote is added.
- The suite stays green: the edit is comment-only, so `cd sites && go build ./...`,
  `cd sites && go vet ./...`, `cd sites && gofmt -l .` (no output),
  `cd sites && go test ./...`, and `bin/check-migrations sites` are all
  unaffected.
