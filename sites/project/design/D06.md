# Decision 6 — Landing page layout (sites-specific; supersedes the cron-canonical conformance)

**Decision.** sites's landing page **no longer conforms to the cron canonical
template.** The prior version of this Decision pinned
`sites/share/www/landing.html` byte-for-byte to `cron/internal/web/landing.html`
(eyebrow, a big display `<h1>` name, a lead paragraph, and a bordered three-cell
**Service / Version / API** `<dl>`). That conformance is **retired by operator
directive**: sites adopts its own tighter layout. This is an intentional
divergence from the suite standard — sites' page will visibly differ from the
other services' pages, which stay on the canonical until they diverge in their
own projects. It stays within the product promises (a Carbon-styled,
session-gated page showing the running version and the list of sites); product
requires Carbon styling, not the cron shape.

**The layout.** The centered `main` is a single top-to-bottom stack:

1. **Home link** — `.home`, absolute top-left, linking to `/` (unchanged, D7).
2. **Eyebrow** — `.eyebrow`, the short descriptor `Static website host` (unchanged text).
3. **Name + version, one heading line** — the service name `{{.Service}}` with the
   running version `{{.Version}}` inline immediately after it. The name uses the
   display font at the **28px** scale the removed "Sites" `<h2>` used (not the big
   `<h1>` clamp); the version keeps the existing `.version` treatment (mono,
   **uppercase**, muted, label size) and sits beside the name with left spacing.
4. **Lead paragraph** — `<p>`, reworded to `Hosts file-backed static websites and
   serves them through the suite gateway.` (the leading word "Sites" is dropped so
   the name is not duplicated).
5. **The filter bar — above the table.** Between the lead and the table: a
   **search input** (fuzzy filter on slug) and a **Clear** control. These are the
   filter affordances for D22, rendered **hidden-until-JS** (see below).
6. **The site listing table** — one row per site (D19), following the filter bar
   directly, with **no** "Sites" `<h2>` section heading above it. Its three
   data-bearing headers — **Slug**, **Creator**, **Created** — are **sortable**:
   each carries a sort hook (a `data-sort-key` of `name` / `createdBy` /
   `createdAt`) that D22's controller binds; the **Visibility** header is not
   sortable. Alongside the table markup sits a **hidden no-match message**
   element (e.g. `<p class="no-match" hidden>No sites match.</p>`) the D22
   controller reveals when a filter excludes every row.
7. **The pager — below the table.** Directly **after** the listing table: a
   **Prev / Next pager** with a `Page X of Y` readout, rendered hidden-until-JS
   and only ever revealed by the controller when the listing exceeds one page
   (D22 owns that behavior).

**Sort affordances (CSS over a semantic attribute).** The active sort column is
communicated by the **`aria-sort` attribute** (`ascending` / `descending`) that
D22's controller stamps on the active `<th>` (and removes from the others) —
the semantically correct HTML mechanism, machine-checkable and screen-reader
native. The visual is **pure CSS over that attribute**: a direction caret
appended via `::after` (`th[aria-sort="ascending"]::after { content: " ▲" }`,
`descending` → `" ▼"`), so no extra DOM stamping and no caret in the server
markup. Sortable headers (`th[data-sort-key]`) get `cursor: pointer` and a
hover affordance so they read as clickable. The server render carries **no**
`aria-sort` (sort state is client state; the default is D22's).

**Hidden-until-JS — and hidden must actually hide.** The interactive controls
(search input, Clear, the pager region, the no-match message) carry the
`hidden` attribute (plus the `js-only` class, and a root `no-js`→`js` class the
controller flips on init) in the server-rendered markup. Because the user-agent
default `[hidden] { display: none }` **loses to any authored `display`
declaration on a class selector** (e.g. `.controls { display: flex }`), the
stylesheet must make hidden win explicitly:

```css
[hidden] { display: none !important; }
```

With JS off the page is exactly the plain full table of D19 with no visible
search / sort / pager / clear affordance; with JS on the controller reveals
them. This is a D6 layout concern; the behavior behind the controls is D22.

**Removed wholesale** from the former canonical markup:

- the big display `<h1>` (its scale drops to 28px and it now carries name **and**
  version together);
- the bordered three-cell **Service / Version / API** `<dl>`, **including the
  `POST /mcp` API value** — the API endpoint is gone from the page entirely;
- the separate `<h2>Sites</h2>` section heading.

The service name therefore appears **exactly once** — in the heading line — rather
than being repeated as an `<h1>`, a `Service` `<dl>` cell, and an `<h2>`.

**Styling is unchanged for the surviving elements.** `.home`, `.eyebrow`,
`.version`, the lead `<p>`, and the `.site-table` all keep their existing
token-based CSS, and `static/tokens.css` is retained as-is (the Carbon token set)
— it is simply no longer diff-pinned to cron's copy. Only two things change: the
heading font-size becomes `28px` / line-height `1.15` (the values the removed
`.sites h2` used), and `.version` gains an inline left margin so it sits on the
name's baseline. The removed markup takes its now-dead CSS with it (the `dl` /
`dl > div` / `dd` rules and their responsive counterparts, the `.sites h2` rule,
and the `dt` member of the shared label selector). The name and version stay
dynamic (`{{.Service}}` / `{{.Version}}`), supplied by the composition root as
before (D1) — no handler-signature change for this part.

Illustrative markup (data substitutions only; the listing table is D19):

```html
<a class="home" href="/">Home</a>
<section aria-labelledby="page-title">
  <div class="eyebrow">Static website host</div>
  <h1 id="page-title">{{.Service}} <span class="version">{{.Version}}</span></h1>
  <p>Hosts file-backed static websites and serves them through the suite gateway.</p>
</section>
<div class="controls js-only" hidden> <!-- search input + Clear -->
</div>
<!-- site listing table — D19 -->
<!-- hidden no-match message -->
<div class="pager js-only" hidden>    <!-- Prev · Page X of Y · Next -->
</div>
```

**Rejected.**
- **Retaining the cron-canonical Service / Version / API card.** The explicit
  operator directive is to override it. Keeping the `<dl>` re-duplicates the name
  and surfaces a `POST /mcp` value with no on-page use.
- **Proposing the new layout to the canonical so the whole suite moves together.**
  Out of scope: the canonical lives in cron's codebase and each other service's
  page is that service's own file; `sites/project` can only change sites. Suite-wide
  convergence, if wanted, is separate work in those projects.
- **Keeping the big `<h1>` scale with the version on its own line.** The operator
  chose the 28px combined line; the oversized title clashes with the small inline
  mono version.
- **Lowercasing the version.** The operator chose to keep the existing uppercase
  `.version` treatment.
- **Re-styling from scratch / new token values.** Surviving elements keep their
  existing Carbon token CSS; only the heading scale and the version's placement
  change.
- **Pager above the table (with the filter bar).** The prior version of this
  Decision grouped all controls between the lead and the table; the operator
  directed the split — filter above where the eye starts, pager below where
  paging is decided after scanning the visible rows.
- **Server-rendered sort carets in the markup.** A caret in the template is
  static content the controller must then manage; `aria-sort` + CSS `::after`
  keeps the visual derived from the one semantic attribute the controller
  already stamps.

**Verification.** (Rendered over the repo-real `share/www` tree via `appkit/web`
with fixed inputs — the same substrate D1/D19 use.)
- R-WKGI-FVFJ — rendering `landing.html` places the service name and the version
  on a **single heading line** (the version string appears **inside the same
  `<h1>` element** as the service name), and the page contains **no** Service /
  Version / API card: the substring `POST /mcp` is absent, and there is no
  `Version`/`API` field label from the old `<dl>`. A wrong implementation that
  keeps the old three-cell card, or renders the version under a separate labeled
  cell instead of inline with the name, fails.
- R-WLOE-TN68 — the eyebrow text is exactly `Static website host` and the lead
  paragraph is exactly `Hosts file-backed static websites and serves them through
  the suite gateway.`; the former lead beginning with `Sites hosts file-backed`
  (leading "Sites") is **absent**. A wrong implementation that leaves the old
  duplicated-name wording fails.
- R-83NK-DUW1 — rendering `landing.html` (non-empty `Sites` slice, repo-real
  `share/www`): the **search input and Clear control appear between the lead
  paragraph and the listing table**, and the **Prev/Next pager region (with its
  page-readout element) appears after the listing table** — asserted by document
  order (lead < controls < table < pager). A **hidden no-match message** element
  is present adjacent to the table. The table's Slug, Creator, and Created
  headers carry `data-sort-key` of `name`, `createdBy`, and `createdAt`
  respectively; the Visibility header carries none. A layout with the pager
  above the table, the search below it, a missing no-match node, or wrong/missing
  sort hooks fails.
- R-84VG-RMMQ — hidden actually hides: the interactive controls (search, Clear,
  pager, no-match message) carry the `hidden` attribute in the server render,
  **and** the page's stylesheet contains a `[hidden]` rule forcing
  `display: none` with precedence over the controls' own `display` declarations
  (i.e. `[hidden] { display: none !important; }` — the authored
  `.controls`/`.pager` `display: flex` must not defeat the attribute). An
  implementation whose controls render visibly without JS (the shipped v0.13.0
  defect) fails.
- R-863D-5EDF — sort affordance CSS is present in the rendered page's styles:
  caret rules keyed on the semantic attribute
  (`th[aria-sort="ascending"]` → an appended `▲`, `th[aria-sort="descending"]`
  → an appended `▼`) and a `cursor: pointer` rule targeting the sortable headers
  (`th[data-sort-key]`); the server markup itself carries **no** `aria-sort`
  attribute. A page with no caret rules, no pointer affordance, or a
  server-stamped `aria-sort` fails.
