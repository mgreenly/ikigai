# Decision 9 — Resolve sites's own port and the dropbox mirror address by name through the shared `registry`

**Problem.** sites writes loopback port literals into its own Go source at the
composition root (`cmd/sites/main.go`):

1. Its **own** port — `Port: 3004` in `sitesSpec()`.
2. The **dropbox** peer address — `"http://127.0.0.1:3200"`, the fallback default
   for the `DROPBOX_BASE_URL` override that wires the mirror client
   (`sites.NewMirrorClient(base)`).

These same numbers are written down in ~five other places across the suite (each
service's `main.go` default, its `.envrc`, `manifest.env`, the peer maps of the
services that consume it, the local nginx front door, `bin/start`). A renumber
has to be made in lockstep everywhere, and a Go literal that drifts out of step
is invisible to the compiler — it only surfaces at deploy. sites has no single
place it can ask "what port is `sites`?" or "where is `dropbox`?".

The new leaf library `registry` (module `registry`, repo root) holds the one
authoritative `name → port` table for the suite and turns a service **name** into
its loopback **address**. It does no I/O and no runtime discovery: the table is
compile-time data and the host is the constant `127.0.0.1`. Its table pins
`sites → 3004` and `dropbox → 3200`, i.e. exactly the values sites hardcodes
today — so adopting it is **behavior-preserving**, no service renumbers.

**Decision.** Make the `registry` table the single source of truth for the two
loopback addresses sites references, resolving both **by name at the composition
root** (`main`), which runs once at startup — never per request. Three
coordinated changes, all confined to `sites/`; no new Go logic beyond two
call-site swaps and one import.

1. **Own port — `sitesSpec()`.**
   `Port: 3004` → `Port: registry.MustPort("sites")`. `MustPort` is the strict
   composition-root form: an unknown name is a programming error and panics.
   `registry.MustPort("sites")` returns `3004`, so the Spec is unchanged in value.
   appkit's existing `SITES_PORT` env layer still overrides this default — the
   registry supplies the **literal's replacement**, not a new override tier.

2. **Dropbox mirror default — the `Handlers` hook.**
   ```go
   base := config.EnvOr(os.Getenv, "DROPBOX_BASE_URL", "http://127.0.0.1:3200")
   ```
   →
   ```go
   base := config.EnvOr(os.Getenv, "DROPBOX_BASE_URL", registry.BaseURL("dropbox"))
   ```
   `registry.BaseURL("dropbox")` composes exactly `"http://127.0.0.1:3200"` (the
   loopback host is quarantined inside `registry`; the caller appends no path —
   `NewMirrorClient` derives `<base>/list` and `<base>/content` as before). The
   `DROPBOX_BASE_URL` env override still takes precedence when set: this is
   appkit's `config.EnvOr` layering, unchanged — the registry replaces only the
   hardcoded **default** the override falls back to.

3. **Module wiring — `sites/go.mod`.** Add `require registry v0.0.0` and a
   committed `replace registry => ../registry`, mirroring the existing
   `appkit`/`agentkit`/`eventplane` replace-siblings, so the production build
   (`GOWORK=off`) resolves the leaf library deterministically with or without the
   root workspace.

After this, `cmd/sites/main.go` imports `registry`, and **no bare loopback port
literal remains in sites's production Go source** — the `registry` table is the
only place a port sites cares about is written.

**External preconditions (out of scope — owned at the repo root / by the
`registry` project, assumed satisfied).** These are NOT sites's work and no phase
here touches them: the repo-root `go.work use ./registry` entry, and the
`registry` module itself (its table, its API, its guardrail tests). sites only
imports the library and wires its own `go.mod`.

**Boundary — the nginx fragment is deliberately NOT converted.** sites's
`etc/nginx.conf` carries literal `127.0.0.1:3004` in five `proxy_pass` lines (PRM,
`/mcp`, landing root, `/static/`), and the relocated `cmd/sites/nginx_test.go`
(moved there from `internal/web/` by D12) asserts them.
This is a **config artifact nginx reads directly** — nginx cannot call a Go
library, so `registry` cannot supply its address. The fragment's port literal
(and the test that mirrors it) is the one intentional exception to "no `30xx`
literal remains": the guardrail (R-7L9F-XW3H) is scoped to **production Go source
only** and explicitly excludes `*_test.go` and the non-Go fragment. Converting the
nginx front door to a generated/registry-fed form is separate, out-of-scope work
owned outside sites's Go module.

**Rejected.**
- **`registry.Port("sites")` (checked form) at the composition root.** The checked
  `(int, ok)` form is for callers that handle a miss; at `main`, an unknown own
  name is a bug that must fail loudly. `MustPort` is the right strict form here.
- **Keeping `"http://127.0.0.1:3200"` and only adopting the own port.** Leaves a
  peer loopback literal in production Go — exactly the drift-prone duplication
  this adoption removes. Both literals are in scope.
- **Re-resolving via `registry` on every request.** The table is compile-time
  data; if it ever changes the binaries are rebuilt anyway. Resolving once at the
  composition root and storing the values is both sufficient and what the registry
  contract intends. (sites already resolves the dropbox base once in `Handlers`;
  this preserves that.)
- **A versioned `require registry vX.Y.Z` without a `replace`.** In-repo libraries
  are never a versioned require in this suite (they are replace-siblings); a
  committed `replace` is what makes the `GOWORK=off` prod build deterministic.

**Verification** (in-package tests in `cmd/sites`, plus content assertions over
`../../go.mod` and a source-tree scan — no network, no running suite).

- R-7K2P-QN4D — **`sitesSpec().Port` equals `registry.MustPort("sites")` (which is
  `3004`).** The Spec's port is resolved from the registry by name, not written as
  a literal. Falsifiable: a `Port:` set to a bare int, or resolving to any value
  other than `registry.MustPort("sites")`, fails it. Asserted in-package with no
  network.
- R-7L9F-XW3H — **No production Go file in the module carries a hardcoded loopback
  port literal.** A walk of every `*.go` under `sites/` **excluding `*_test.go`**
  finds no `127.0.0.1:30` substring and no standalone `3004` token. Falsifiable:
  reintroducing `"http://127.0.0.1:3200"` or a literal `Port: 3004` in production
  Go fails it. `etc/nginx.conf` (not Go) and the nginx fragment test
  (`cmd/sites/nginx_test.go` after D12; a test asserting that config artifact) are
  deliberately out of scope — see **Boundary**.
- R-7M4C-BV8J — **`registry.BaseURL("dropbox")` composes exactly
  `"http://127.0.0.1:3200"`.** The default sites now feeds the `DROPBOX_BASE_URL`
  fallback resolves, via the registry, to the exact address the old literal held —
  proving the swap is behavior-preserving. Falsifiable: a different host, port, or
  a trailing path fails it. (The `DROPBOX_BASE_URL` env override still wins when
  set — unchanged appkit `config.EnvOr` layering, not re-implemented here.)
- R-7N6R-TZ2Q — **`sites/go.mod` declares `require registry` and a committed
  `replace registry => ../registry`.** A content assertion over `../../go.mod`
  finds both, so the `GOWORK=off` prod build resolves the leaf library
  deterministically. Falsifiable: a missing `require` or `replace` fails it.
