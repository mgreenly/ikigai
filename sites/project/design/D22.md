# Decision 22 — Client-side filter, sort, and pagination of the landing listing

**Decision.** The landing page (D19) carries an in-browser control layer over the
site listing: a fuzzy **search filter** on the slug, **sortable columns**
(name / created-at / created-by), **pagination** past ten rows, and a one-click
**clear**. All of it runs **client-side in JavaScript** — the `GET /{$}` handler,
the store, the DB, and nginx are **unchanged**. This is purely additive to D19's
server render. The filter bar sits above the table and the pager below it (D6
owns that placement).

**Where the logic runs and why.** sites is a single box hosting "a handful of
small static websites" (product scope), so the whole site list is tiny: the
server already ships every row, and shipping it once more as data for the browser
is trivial. Client-side gives instant per-keystroke filtering with no round-trip,
and keeps the feature a self-contained layer that touches no Go request path. A
no-JS browser still gets the complete list (D19's server-rendered table), just
un-enhanced.

**The data path — a JSON island (D19).** The controller does not scrape the
DOM. The server emits the site rows a second time as a JSON array in an inline
`<script type="application/json" id="sites-data">` (D19 owns that render and its
`url`/sortable-timestamp contract). The controller parses that array once on load
and drives everything from the data, so the tested logic operates on plain row
objects, never DOM nodes. The controller makes **zero network requests** — no
fetch, no XHR, no cookie reads; everything it can ever know arrives baked into
the server render. (That is also why its browser test needs no auth: the session
cookie is spent at nginx, a layer the page itself never sees.)

**The module — pure functions plus a thin controller.** `share/www/static/landing.js`
is structured so **almost all logic is pure functions over data**, with the DOM
work reduced to dumb glue:

```js
// Pure — no DOM, no globals, deterministic. Exposed for testing.
function filterSites(rows, query)        // subsequence-fuzzy predicate on slug
function sortRows(rows, key, dir)        // 'name'|'createdAt'|'createdBy', 'asc'|'desc'
function paginate(rows, page, size)      // rows.slice((page-1)*size, page*size)
function nextSort(state, key)            // header-click toggle rule
function defaultState()                  // {query:'', sortKey:'createdAt', dir:'desc', page:1}
function reduce(state, action)           // setQuery|setSort|setPage|clear -> new state
function computeView(rows, state)        // orchestrates the above -> a view model

// The view model computeView returns (everything the DOM needs, all derived):
// { rows, showControls, empty, noMatch, showPager, page, pageCount, rangeFrom, rangeTo, sortKey, dir }

globalThis.SitesLanding = {
  filterSites, sortRows, paginate, nextSort, defaultState, reduce, computeView,
};

// DOM bootstrap — skipped entirely under a non-browser runtime (goja).
if (typeof document !== 'undefined') {
  document.addEventListener('DOMContentLoaded', () => initController(/* reads #sites-data */));
}
```

**`initController` — a real, branchless transcription.** The controller is the
only impure code, and it is **implemented, not stubbed** (the shipped v0.13.0
left it an empty function — the page rendered dead controls; this Decision
closes that). On `DOMContentLoaded` it:

1. **parses `#sites-data` once** into the `rows` array and initializes
   `state = defaultState()`;
2. **reveals the enhancement**: flips the root `no-js` class to `js` and removes
   the `hidden` attribute from the controls the view model says to show
   (`showControls` gates the filter bar; `showPager` gates the pager — with zero
   sites everything stays hidden and D19's server empty state stands);
3. **wires the listeners** — `input` on the search box (`setQuery`), `Escape` in
   the search box (`setQuery('')`), `click` on each `th[data-sort-key]`
   (`setSort`), `click` on Prev/Next (`setPage`), `click` on Clear (`clear`);
4. **on every action** dispatches through `reduce`, calls
   `computeView(rows, state)`, and **stamps the returned view model into the
   DOM**: rebuilds `<tbody>` from `view.rows` (slug anchor using each row's
   `url`, visibility label, creator, created-at display string), sets the
   `Page X of Y` readout from `view.page`/`view.pageCount`, toggles the pager
   per `view.showPager`, toggles the no-match message per `view.noMatch`, and
   stamps `aria-sort` (`ascending`/`descending` per `view.dir`) on the active
   `th[data-sort-key]` while removing it from the others (D6's CSS renders the
   caret and pointer affordances off those attributes).

It contains **no branching logic of its own** — every decision (which rows,
whether the pager shows, the page label, the empty-state kind) is computed by
the pure `computeView`; the controller only transcribes the view model onto the
DOM. The `globalThis.SitesLanding` handle exists so a non-browser runtime can
call the pure functions directly.

**The behaviors (pure):**

- **Filter — subsequence fuzzy, case-insensitive, predicate.** `filterSites`
  keeps a row when the lowercased query is a **subsequence** of the lowercased
  slug (its characters appear in order, not necessarily adjacent): `dsb` matches
  `dashboard`. It is a **predicate** — it drops non-matches and **preserves the
  input order** of the survivors; it never re-ranks by match quality (the sort
  columns own order). An empty query returns all rows unchanged.
- **Sort — two-state toggle, real-timestamp chronology, stable.** `sortRows`
  orders by `name` (slug), `createdAt`, or `createdBy`. `createdAt` sorts by the
  row's **machine-sortable timestamp** field (D19), not the human display string.
  Ties break by slug ascending, so order is deterministic. `nextSort(state, key)`
  is the click rule: clicking the **active** column flips `dir` (`asc`↔`desc`);
  clicking a **different** column switches `sortKey` to it at `asc`. The default
  is `createdAt` / `desc` (newest first).
- **Paginate — fixed page size 10.** `paginate(rows, page, 10)` returns exactly
  `rows.slice((page-1)*10, page*10)`. `computeView` shows the pager **only when
  the filtered-and-sorted set exceeds 10 rows**, derives `pageCount`, a
  `Page X of Y` label and an `N–M of T` range, and clamps `page` into
  `[1, pageCount]`.
- **State & clear.** `defaultState()` is `{query:'', sortKey:'createdAt',
  dir:'desc', page:1}`. `reduce`: `setQuery` and `setSort` both **reset `page`
  to 1** (a filter/sort change can never strand you on a now-gone page);
  `setPage` changes only the page; `clear` returns exactly `defaultState()`.
  The Clear control fires `clear`; Escape in the search box fires `setQuery('')`.
- **Empty states (derived by `computeView`).** With **zero sites** the island is
  empty and `computeView` reports `showControls:false` (no dead search box over
  an empty box — D19's server empty state stands). With sites present but the
  filter matching **none**, it reports `noMatch:true` (the controller reveals
  D6's no-match message) while keeping the controls visible so the query can
  be edited or cleared; no pager renders.

**Progressive enhancement.** The interactive controls (search input, sort
affordances, pager, clear) are rendered **hidden-until-JS**, and the stylesheet
guarantees `hidden` actually hides (D6). JS off → the plain full table with no
inert affordances; JS on → the controller reveals the controls and the table
becomes live.

**Testing — two tiers, each covering the other's blind spot.**

- **goja owns the logic (broad, cheap, pure Go).** A Go test loads the repo-real
  `share/www/static/landing.js` into `github.com/dop251/goja` (pure-Go ES engine,
  test scope only, no cgo, inside `go test ./...`) and calls the exposed
  `SitesLanding.*` pure functions against fixed inputs — fuzzy semantics, sort
  order, pagination arithmetic at the exact boundaries, the toggle rule, the
  reducer, and the view-model derivations, all proven against the code that
  actually ships. goja has no `document`, so loading the module runs only the
  pure definitions; the controller bootstrap is inert. Logic is tested **here,
  never in the browser** — the browser tier never re-proves boundaries.
- **The browser proves the wiring (narrow, minimal — D23).** `initController`'s
  runtime claim — that typing, clicking, and paging actually change the DOM — is
  a real-browser fact goja structurally cannot see. D23 owns that proof: one
  headless-Chrome session over a seeded, auth-free `httptest` page that touches
  each control exactly once. This Decision's ids stay pure/structural; the
  wiring ids are D23's.
- **Accepted residual risk: the Escape binding.** Escape-in-the-search-box shares
  the search input (a listener target the D23 filter step already proves live),
  and its semantics are goja-covered (`setQuery('')` via the reducer). Its
  specific key binding is deliberately not browser-driven — the minimal-chromedp
  principle (D23) caps the browser tier at one proof per control.

**Rejected.**
- **Server-side query params (`?q=&sort=&dir=&page=`).** A round-trip per
  keystroke, a Go fuzzy matcher, and sort/paginate logic in the handler — far
  more machinery than a tiny always-shipped list needs, and it would rebuild the
  request path this Decision deliberately leaves untouched.
- **DOM manipulation of server-rendered `<tr>` rows (no JSON island).** Then the
  logic is DOM-bound and cannot be tested in goja; it would need a browser for
  everything. The island keeps the logic pure and falsifiable in `go test`.
- **A node/vitest test runner for the JS.** Drags a second-language toolchain
  into a pure-Go repo; `go test ./...` would no longer be the whole green bar.
  goja tests the same pure functions without it.
- **Leaving `initController` verification structural-only (the prior version of
  this Decision).** That shipped a green suite over a dead UI: the stub passed
  every structural assert while filter and sort did nothing. Retired by operator
  directive; the wiring is now proven for real in D23.
- **Ranking filtered results by match score.** Fights the user-controlled sort;
  the filter is a predicate and the sort columns own order.
- **Numbered page buttons; URL/hash-persisted state.** Prev/Next with a
  `Page X of Y` readout suffices for a handful of sites, and in-memory state
  (reload = clean slate) reinforces "easy to clear"; both rejected as
  unnecessary machinery.

**Verification.** (Ids R-HU67-LJBW … R-7V8B-GA0T drive the real shipping
`landing.js` in goja with fixed inputs; R-ICGP-C3GB is a structural assertion
over the shipped `share/www` tree. The runtime wiring ids live in D23.)

- R-HU67-LJBW — `filterSites` keeps a row when the query is a **subsequence** of
  the slug: given slugs `["docs","dashboard","blog"]`, query `dsb` returns
  exactly `["dashboard"]`. A **substring** implementation (which needs `dsb`
  contiguous) returns `[]` and fails.
- R-HVE3-ZB2L — `filterSites` is **case-insensitive**: query `DOCS` matches slug
  `docs`, and query `DsB` matches `dashboard`. A case-sensitive implementation
  returns `[]` and fails.
- R-HWM0-D2TA — `filterSites(rows, "")` (empty query) returns **all** rows in the
  original input order. An implementation that drops or reorders rows on the
  empty query fails.
- R-HXTW-QUJZ — `filterSites` **preserves input order** among survivors and does
  not re-rank: with slugs ordered `["zebra-app","able-app"]` and query `app`,
  the result is `["zebra-app","able-app"]` (input order), not resorted or
  score-ranked. An implementation that reorders survivors fails.
- R-HZ1T-4MAO — `sortRows(rows,'name','asc')` orders slugs ascending and
  `sortRows(rows,'name','desc')` is its **exact reverse** for a set of distinct
  slugs. An implementation that ignores `dir` (or does not reverse) fails.
- R-I1HL-W5S2 — `sortRows(rows,'createdAt','desc')` orders by the row's
  **machine-sortable timestamp**, newest first, even when the human
  `createdAt` display strings do **not** sort lexically in the same order: given
  rows whose sortable timestamps ascend while their display strings do not, the
  result is true reverse-chronological. An implementation that sorts on the
  display string fails.
- R-I2PI-9XIR — `sortRows(rows,'createdBy','asc')` orders rows by creator
  ascending. An implementation that ignores the `createdBy` key fails.
- R-I3XE-NP9G — `sortRows` breaks ties on **slug ascending**: two rows sharing a
  `createdAt` timestamp come back in slug order regardless of input order,
  deterministically. An implementation with an unstable/no tie-break can return
  the other order and fails.
- R-I55B-1H05 — `nextSort` toggle rule: `nextSort({sortKey:'createdAt',dir:'desc'},
  'createdAt')` yields `dir:'asc'` (same key flips direction);
  `nextSort({sortKey:'createdAt',dir:'desc'}, 'name')` yields
  `{sortKey:'name',dir:'asc'}` (a new key switches and starts ascending). An
  implementation that does not flip on repeat, or does not reset to `asc` on a
  new key, fails.
- R-I6D7-F8QU — `paginate(rows, page, 10)` returns exactly
  `rows[(page-1)*10 .. page*10)`: for 34 rows, page 1 has 10 rows (the first 10),
  page 4 has 4 rows (the last 4), and page 5 returns `[]`. An off-by-one
  (inclusive upper bound, 1-based slice error, or wrong page origin) fails at one
  of these boundaries.
- R-I7L3-T0HJ — `computeView` reports `showPager:true` **iff** the
  filtered-and-sorted row count is **> 10**: exactly 10 rows → `showPager:false`;
  11 rows → `showPager:true`. An implementation that shows the pager at ≤10 (or
  hides it above 10) fails.
- R-I8T0-6S88 — `computeView` derives a correct `pageCount`, a `Page X of Y`
  label, and an `N–M of T` range for the current page, and **clamps** an
  out-of-range `page` into `[1, pageCount]`: 34 rows at `page:4` → `pageCount:4`,
  range `31–34 of 34`; `page:9` clamps to page 4 with the same range. A wrong
  page-count/range computation, or no clamp, fails.
- R-IA0W-KJYX — `computeView` empty-state kinds: with **zero** input rows it
  returns `{showControls:false, empty:true}`; with input rows present but the
  query matching **none** it returns `{showControls:true, noMatch:true,
  showPager:false}`. An implementation that conflates the two (hides controls on
  a no-match, or shows the no-match message when the box is simply empty) fails.
- R-IB8S-YBPM — `defaultState()` deep-equals `{query:'', sortKey:'createdAt',
  dir:'desc', page:1}`, and `reduce(anyState, {type:'clear'})` returns exactly
  that same default regardless of the prior state. An implementation whose
  default is not created-at-desc/page-1, or whose clear is a partial reset (e.g.
  clears the query but leaves the sort or page), fails.
- R-7V8B-GA0T — `reduce` page-reset invariant: `reduce({...,page:3},
  {type:'setQuery',query:'x'})` and `reduce({...,page:3},
  {type:'setSort',key:'name'})` both return `page:1`, while `reduce({...,page:3},
  {type:'setPage',page:2})` returns `page:2` (a page change does **not** reset).
  An implementation that leaves `page` unchanged on a query/sort change fails.
- R-ICGP-C3GB — the shipped tree contains `share/www/static/landing.js`, and the
  rendered `landing.html` references it with a `<script src="static/landing.js">`
  (deferred). A tree missing the file, or an HTML that never loads it, fails.
  (Structural assertion over the repo-real `share/www`; the served-asset path is
  the chassis `/static/` mount that already serves `tokens.css`.)
