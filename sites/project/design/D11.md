# Decision 11 — Rewire the MCP file tools onto `internal/files` and drop `agentkit`

**Problem.** With the confined filesystem logic now native (**D10**),
`internal/mcp` still imports `agentkit/tools` + `agentkit/wire` and builds
agent-shaped tool-use blocks in `files.go`. The MCP surface must be moved onto the
native package, the agentkit bridge deleted, and the dependency removed from
`sites/go.mod` — sites was the local `agentkit` module's **only** consumer, so
this severs the tie completely. The move must **not** change the tool surface a
connected agent sees: the same 15 tools, same names.

**Decision.** Make `internal/mcp` a pure wire layer over `files`: each file
tool's handler decodes its MCP `arguments` into a small typed struct (the existing
`unmarshalArgs` device), validates the site, resolves the sandbox root
(`layout.WorkingDir(site)`), calls the matching `files.*` function, and formats
the return with sites's existing result helpers. No `agentkit`, no `wire` blocks,
no `Dispatch`, no schema reflection.

1. **Delete the bridge.** Remove the `agentkit/tools` + `agentkit/wire` imports
   and the `agentkitSchemas`/`toolFile`/`renderToolResultBlock`/`contentText`
   machinery from `internal/mcp/files.go`. The four bridged tools become ordinary
   handlers calling `files.Read`/`files.Edit`/`files.Glob`/`files.Grep`. The
   native `file_write`/`file_list`/`mkdir` handlers call `files.Write`/`files.List`/
   `files.Mkdir` and use `files.ConfinePath`; sites's old in-package `confinePath`
   is deleted.

2. **Hand-write the four schemas.** The `inputSchema` for `file_read`/`edit`/
   `glob`/`grep` is written with the existing `obj(...)`/`descTyp(...)` helpers —
   exactly how every other sites tool (`file_write`, `file_list`, `create`, …)
   already declares its schema — instead of being reflected from agentkit. Each
   keeps its required `site` property. No reflection library is introduced.

3. **Cleaner result shapes.** Because `files.*` returns native Go, the handlers
   serialize the ordinary way with `toolResultJSON`/`toolResultText`:
   `file_read` → a text block of the file content; `file_edit` →
   `{edited, site, replaced}`; `file_glob` → `{site, matches:[…]}`; `file_grep` →
   `{site, matches:[{path,line,text},…]}`. This drops the old
   stringified-JSON-inside-a-text-block form (an artifact of feeding an LLM);
   `file_write`/`file_list` already returned structured JSON and are unchanged.

4. **Typed confinement envelope.** A confinement rejection from any file tool is
   detected with `errors.Is(err, files.ErrEscapes)` and mapped to the existing
   `errResultMsg("path_escapes_working_dir", …)` envelope; other errors map to
   their existing per-tool codes. The escape rejection is thus one consistent,
   distinguishable outcome across all seven filesystem tools.

5. **Drop the dependency.** Remove `require agentkit v0.0.0` and
   `replace agentkit => ../agentkit` from `sites/go.mod` and `go mod tidy`. No new
   dependency is added (no remote agentkit, no reflection library). The orphaned
   local `agentkit/` source tree at the repo root is **left on disk** — deleting a
   repo-root module is suite-level work outside this `project/`'s scope.

The exposed surface is unchanged: `tools/list` still advertises exactly the same
15 tools (`health`, `describe`, `create`, `list`, `delete`, `mkdir`, `publish`,
`unpublish`, `sync`, `file_write`, `file_read`, `file_edit`, `file_glob`,
`file_grep`, `file_list`); only the four file tools' schemas shift from reflected
to hand-written and their result bodies get cleaner. Nothing is added or removed.

**Rejected.**
- **Keep the `agentkit` import for the `wire`/result types only.** Pointless: once
  the logic is native, nothing needs agentkit's block types; the goal is zero
  agentkit in sites.
- **Preserve the exact old stringified-JSON-in-text result shape.** Bug-for-bug
  fidelity to an LLM-feeding artifact, for which there is no client (migration
  window: no real customers) and no internal caller. Take the clean shape.
- **Let confinement errors go generic.** Loses the `path_escapes_working_dir`
  distinction that the security boundary deserves and that an existing test
  asserts; the sentinel makes it first-class and uniform.

**Verification** (driven through the MCP handler with `httptest`-style
`tools/call` as the existing `internal/mcp` tests do, plus a source/`go.mod`
scan; no network, no agent).

- R-0FMU-J775 — **No Go file under `sites/` imports an `agentkit` package.** A
  scan of every `*.go` in the module finds no import path equal to or prefixed by
  `agentkit` (neither the local `agentkit/...` nor `github.com/ikigenba/agentkit`).
  Falsifiable: reintroducing any agentkit import fails it. *(source-tree scan
  test)*
- R-0GUQ-WYXU — **`sites/go.mod` no longer references `agentkit`.** A content
  assertion over `go.mod` finds no `require agentkit` line and no
  `replace agentkit =>` directive. Falsifiable: either directive remaining fails
  it. *(go.mod content-assertion test)*
- R-0I2N-AQOJ — **The file tools return the cleaner structured shapes.** Through
  `tools/call`: `file_glob` returns a result whose JSON carries a `matches` array
  (not a stringified array in text), `file_grep` returns `matches` of
  `{path,line,text}` objects, `file_edit` returns `{edited,site,replaced}`, and
  `file_read` returns the file content as text. Falsifiable: a glob/grep result
  that is a JSON string stuffed in a text block, or a missing `matches`/`replaced`
  field, fails it.
- R-0JAJ-OIF8 — **A confinement rejection yields the `path_escapes_working_dir`
  envelope.** A `tools/call` to any file tool with a `..`-escaping or
  outside-root path returns `isError` with the error envelope whose `code` is
  `path_escapes_working_dir`. Falsifiable: a generic code, or a non-error result,
  fails it. *(driven for at least `file_read` and `file_list`)*
- R-0KIG-2A5X — **The 15-tool surface is unchanged.** `tools/list` returns exactly
  the set `{health, describe, create, list, delete, mkdir, publish, unpublish,
  sync, file_write, file_read, file_edit, file_glob, file_grep, file_list}` — no
  tool added or removed by the migration, and every `inputSchema` is an object
  schema. Falsifiable: a missing, added, or renamed tool fails it.
