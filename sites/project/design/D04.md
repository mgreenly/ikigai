# Decision 4 — nginx fragment: the exact-match session-gated `= /srv/sites/` landing root beside the existing static tiers

**Decision.** sites's nginx fragment (`sites/etc/nginx.conf`) gains **one new
location** — an **exact-match** location for the bare mount root that
session-gates the human landing page and **proxies** it to the service's root.
sites is the special case in this suite-wide change: it **already** serves web
pages and **already** uses the `/_session-authn` gate, so this addition reuses an
already-present gate and slots the standardized landing card at the bare root,
beside (not on top of) the existing tiers:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (the SAME
# gate the private static tier below already uses). Exact-match `=` so it carves
# out ONLY the bare mount root: it never shadows the `/srv/sites/public/` or
# `/srv/sites/private/` tier PREFIXES (those match paths UNDER those folders) nor
# the exact `= /srv/sites/mcp`. A failed session check yields 401. Unlike the
# disk-backed tiers, the landing root is PROXIED to the sites process, which
# renders the dynamic name+version page.
location = /srv/sites/ {
    auth_request /_session-authn;
    auth_request_set $sites_session_owner $upstream_http_x_owner_email;
    proxy_set_header  X-Owner-Email $sites_session_owner;
    proxy_pass http://127.0.0.1:__PORT__/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**How it coexists with the existing fragment.** sites's fragment already has
**five** locations (it has **no** `location /srv/sites/` catch-all prefix — every
route is named explicitly):

1. `location = /srv/sites/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match, no `auth_request`).
2. `location = /srv/sites/mcp` — the bearer-gated MCP endpoint (exact-match,
   `auth_request /_authn`), carrying the `$authn_status` / `error_page 500 =
   @sites_authn_500` rate-limit re-emit machinery.
3. `location /srv/sites/public/` — the **public** static tier (**prefix**, NO
   `auth_request`), `alias`-ed from disk (`/opt/sites/www/served/public/`).
4. `location /srv/sites/private/` — the **private** static tier (**prefix**,
   `auth_request /_session-authn`), `alias`-ed from disk
   (`/opt/sites/www/served/private/`).
5. `location @sites_authn_500` — the named location that re-emits a faithful `429`
   (with `Retry-After`) when the bearer introspection was rate-limited.

nginx matching rules make an **exact-match `=` location win over any prefix
location** for that one path regardless of source order. The bare root
`/srv/sites/` is **not a path under** `/srv/sites/public/` or
`/srv/sites/private/` (those prefixes only match `…/public/<x>` and
`…/private/<x>`), and it is a **different exact path** from `= /srv/sites/mcp` and
the PRM well-known. So the new `= /srv/sites/` exact match cleanly answers the
**bare root only**, while:

- `/srv/sites/public/<file>` and `/srv/sites/private/<file>` still hit their
  prefix tiers and are served from disk;
- `/srv/sites/mcp` still hits the bearer-gated exact match;
- the PRM well-known and `@sites_authn_500` are untouched (different paths /
  named location).

Nothing in the existing fragment is removed or rewritten — the change is **purely
additive** (one new `location` block).

**Why `/_session-authn`, not `/_authn` — and why this is a no-new-dependency
case for sites.** A browser carries the `dashboard_session` cookie, not a bearer
token, so the bearer gate (`/_authn`) would refuse it. `/_session-authn` is the
dashboard-owned, loopback-only internal location that validates the cookie
against the web-session store and returns `200` + `X-Owner-Email` on success or
`401` on failure. sites's **private static tier already uses this exact gate**
(`location /srv/sites/private/ { auth_request /_session-authn; … }`), so unlike
the other services adopting this change, sites adds **no new suite dependency** —
the landing page simply reuses a gate already present in its own fragment. The
gate is deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer `= /srv/sites/mcp` block
carries the `$authn_status` / `error_page 500 = @sites_authn_500` machinery
because token introspection can return `429` (rate-limited) which `auth_request`
collapses to a synthetic `500`. The session check is a cookie lookup, not a
rate-budgeted token introspection, so the landing location does **not** need the
429 re-emit dance — it maps the session subrequest's allow/deny directly
(`200`→serve, `401`→401), exactly as the existing `/srv/sites/private/`
session-gated tier does (it likewise omits the `@sites_authn_500` machinery).

**Why proxy (not `alias` from disk) for the landing root.** The public/private
tiers are static files served straight off disk with `alias`. The landing root is
**proxied** to `http://127.0.0.1:__PORT__/` because the page is rendered
**in-process** by the sites binary (so it reflects the live `rt.Version()`, D1) —
it is not a disk file. This is the one route in sites's fragment that proxies the
bare root to the upstream.

**Templating.** `__PORT__` stays templated (substituted from `etc/manifest.env`
`PORT=3010` at install time by `opsctl setup sites` on the box / `nginx/run` in
dev), exactly like the existing locations. The fragment is **not** a vhost — no
`server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/sites.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — a logged-out
  browser must get `401`. The page is gated, just coarsely. (The *public static
  tier* is intentionally public; the *landing card* is not.)
- **Serving the landing card via an `alias` disk tier** — a disk file cannot
  reflect the running version; the landing root proxies to the in-process render
  (D1).
- **A prefix `location /srv/sites/` for the page** — would swallow the bearer
  `= /srv/sites/mcp` and the static tiers under one gate; the exact-match `=` is
  what cleanly carves the bare root out without touching the rest.
- **Adding the `@sites_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  private tier omits it for the same reason).
- **Editing any of the five existing locations** — unnecessary and risky; the
  change is purely an added exact-match location.

**Verification.** (Proven by a test that reads `sites/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-3P6T — `sites/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/sites/ {` block (distinct from the tier
  prefixes `location /srv/sites/public/ {` and `location /srv/sites/private/ {`).
- R-NGNX-5R8V — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-7T1X — that block **proxies** to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:__PORT__/;` — trailing slash), so the upstream
  sees `/` (it is not an `alias` disk tier).
- R-NGNX-9W4Z — the **five** pre-existing locations are retained alongside it: the
  `= /srv/sites/.well-known/oauth-protected-resource` PRM bootstrap, the
  bearer-gated `= /srv/sites/mcp` with `auth_request /_authn`, the public
  `location /srv/sites/public/` tier (no `auth_request`), the private
  `location /srv/sites/private/` tier with `auth_request /_session-authn`, and the
  `@sites_authn_500` named re-emit all still appear in the file.
