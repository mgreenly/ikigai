# Decision 1 — The landing handler and its v1 content (service name + version)

**Decision.** sites owns its standardized landing page. A new package
`sites/internal/web/` exposes a constructor —
`web.LandingHandler(service, version string) http.Handler` — that returns an
`http.Handler` rendering the v1 landing page: a Carbon-styled, centered card
showing the **service name** (display type) and the **running version** (mono
label) and nothing else. There is **no shared landing handler** across apps;
this code lives under `sites/`. The two content values are passed in as plain
strings — the composition root supplies them from the chassis (`rt.Service()` →
`"sites"`, `rt.Version()` → the build version, D2) — so the handler is pure over
its inputs and needs no chassis, DB, clock, or network at test time.

**Why dynamic, in-process (unlike sites' static tiers).** sites already serves
HTML off disk through nginx (the public/private static tiers), but the landing
card is **rendered in-process by the sites binary** so it can interpolate the
live `rt.Version()` — a static file on disk could not stay truthful across a
deploy/rollback without being rewritten. The landing handler is therefore Go
code mounted on the service's mux, not another disk alias.

**Rendering.** The page is an embedded template rendered once per request (or
once at construction and written verbatim — either is fine; the observable
contract is the output):

- The HTML is embedded via `//go:embed` from a `*.html` file in
  `sites/internal/web/` (e.g. `landing.html`). The service name and version are
  the only dynamic substitutions; everything else is static markup.
- The template is rendered with the standard library (`html/template` preferred —
  the two values are trusted but `html/template` is the correct default for HTML;
  `text/template` is acceptable since the inputs are not attacker-controlled).
- The response sets `Content-Type: text/html; charset=utf-8` explicitly and
  returns `200 OK` for the page.
- The markup links the app's **own** embedded `tokens.css` and applies the Carbon
  type scale (Space Grotesk display for the name, IBM Plex Mono for the version
  label) and neutral palette. The Carbon asset wiring is D3; this Decision owns
  that the page is *rendered with name + version* and is a single centered card.

**Why a constructor over strings, not over the chassis.** Passing `service` and
`version` as strings (rather than handing the package the `*appkit.Router`) keeps
`internal/web` free of any chassis dependency and makes the handler trivially
table-testable with fixed values. The composition root is the only place that
reaches into `rt` for the two strings.

**Rejected.**
- **A shared cross-app landing handler** (e.g. in appkit) — the locked design:
  each app's page will diverge to serve that app's purpose, so each app owns its
  page. A shared handler would have to be un-shared at the first divergence. v1's
  uniformity is a content coincidence, not a shared component.
- **Serving the landing card as another static disk tier** (like public/private)
  — a disk file cannot reflect the running version across a deploy without being
  regenerated; the in-process render keeps name+version truthful for free.
- **Rendering anything beyond name + version** in v1 (owner email, published-site
  counts, links) — out of product scope; v1 is exactly name + version.
- **Pulling the values from env or a new config knob** — the chassis already
  exposes the authoritative service name and build version; a second source could
  drift from what `version`/`/health` report.
- **Reading the identity header (`X-Owner-Email`) to personalize the page** — the
  gate is coarse (any logged-in user); v1 shows no per-user content, so the
  handler ignores identity entirely.

**Verification.**
- R-LAND-3C9K — `web.LandingHandler("sites", v)` serving a `GET /` returns
  HTTP `200`.
- R-LAND-5E2M — the rendered body contains the service name passed in (the page
  shows `"sites"`).
- R-LAND-7G4P — the rendered body contains the version string passed in (e.g. a
  test version like `"9.9.9-test"` appears verbatim), proving the page reflects
  the running version rather than a hardcoded constant.
- R-LAND-9J6R — the landing response sets `Content-Type: text/html; charset=utf-8`.
