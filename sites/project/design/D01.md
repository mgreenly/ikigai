# Decision 1 — The landing handler and its v1 content (service name + version)

**Decision.** sites owns its landing page *content*. The page is a
Carbon-styled, centered card showing the **service name** (display type) and the
**running version** (mono label) and nothing else. The template lives on disk at
`sites/share/www/landing.html` (shipped in the release `share` tier, per D12 and
appkit's chassis design); the landing route is a small handler at the
composition root that renders it through the chassis-loaded site:

```go
// cmd/sites — inside Spec.Handlers; rt.WWW() is the appkit/web Site loaded from
// the www root (Spec.WWW).
rt.Handle("GET /{$}", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    if r.URL.Path != "/" {
        http.NotFound(w, r)
        return
    }
    _ = rt.WWW().Render(w, "landing.html",
        struct{ Service, Version string }{rt.Service(), rt.Version()})
}))
```

There is **no shared landing page** across apps; each service ships its own
`share/www/landing.html` and only the rendering *mechanism* (template loading,
static serving) is chassis-owned. The two content values are supplied from the
chassis (`rt.Service()` → `"sites"`, `rt.Version()` → the build version, D2) — so
the page is pure over those inputs and the template file.

**Why dynamic, in-process (unlike sites' static tiers).** sites already serves
HTML off disk through nginx (the public/private static tiers, `state/www`), but
the landing card is **rendered in-process by the sites binary** so it can
interpolate the live `rt.Version()` — a static file on disk could not stay
truthful across a deploy/rollback without being rewritten. The landing route is
therefore Go code mounted on the service's mux, not another disk alias. (Its
*template and assets* still ship on disk in `share/www`, D12 — but they are
release artifacts rendered by the process, not one of the nginx-served
`state/www` user tiers.)

**Rendering.**

- The template is `html/template`, parsed by the chassis at startup
  (`appkit/web.Load` over the www root); the service name and version are the
  only dynamic substitutions; everything else is static markup.
- The response sets `Content-Type: text/html; charset=utf-8` (the chassis
  `Render` sets it) and returns `200 OK` for the page.
- The markup links the app's **own** `static/tokens.css` (served from the same
  www root by the chassis static mount) and applies the Carbon type scale (Space
  Grotesk display for the name, IBM Plex Mono for the version label) and neutral
  palette. The Carbon asset wiring is D3; this Decision owns that the page is
  *rendered with name + version* and is a single centered card.

**Why strings, not the chassis object.** The composition root reaches into `rt`
for the two strings and the loaded site (`rt.WWW()`); the template itself sees
only `{Service, Version}`, so the rendered output is fully determined by those
two values and the shipped template file — no DB, clock, or network at test time.

**Rejected.**
- **A shared cross-app landing page** (e.g. in appkit) — the locked design: each
  app's page will diverge to serve that app's purpose, so each app owns its page
  *content*; only the rendering mechanism is shared, in appkit. v1's uniformity
  is a content coincidence, not a shared component.
- **Serving the landing card as another static disk tier** (like public/private)
  — a disk file cannot reflect the running version across a deploy without being
  regenerated; the in-process render keeps name+version truthful for free.
- **Embedding the template in the binary (`//go:embed`)** — the suite's original
  approach, deliberately reverted (D12): the template and assets are release
  artifacts on disk in `share/www`, visible and diffable in the release tree, not
  bytes hidden in the binary; the chassis (`appkit/web`) loads and renders them.
- **Rendering anything beyond name + version** in v1 (owner email, published-site
  counts, links) — out of product scope; v1 is exactly name + version.
- **Pulling the values from env or a new config knob** — the chassis already
  exposes the authoritative service name and build version; a second source could
  drift from what `version`/`/health` report.
- **Reading the identity header (`X-Owner-Email`) to personalize the page** — the
  gate is coarse (any logged-in user); v1 shows no per-user content, so the
  handler ignores identity entirely.

**Verification** (over an `appkit/web` Site loaded from the repo-real
`sites/share/www` tree, driven with httptest — D12).
- R-LAND-3C9K — the landing handler serving a `GET /` returns HTTP `200`.
- R-LAND-5E2M — the rendered body contains the service name passed in (the page
  shows `"sites"`).
- R-LAND-7G4P — the rendered body contains the version string passed in (e.g. a
  test version like `"9.9.9-test"` appears verbatim), proving the page reflects
  the running version rather than a hardcoded constant.
- R-LAND-9J6R — the landing response sets `Content-Type: text/html; charset=utf-8`.
