# Decision 9 — Runs live under the service-owned `cache/` dir, not the root-owned AppDir

**Problem.** On the D01 on-box layout, `scripts` **crash-loops to `failed` on
boot**: `unlinkat /opt/scripts/runs: permission denied`, exit 1, systemd "start
request repeated too quickly". It is down on the live box.

The cause is `scriptsRuntimeRoot` picking the wrong parent for the rebuildable
`runs/` tree. `cmd/scripts/main.go` boots by resolving:

```go
rootDir := scriptsRuntimeRoot(cfg.DBPath, cfg.GenerationPath, os.Getenv)
runsDir := filepath.Join(rootDir, "runs")
if err := recreateRunsDir(runsDir); err != nil { return err }   // RemoveAll + MkdirAll
```

`scriptsRuntimeRoot` today has two branches:

```go
if strings.TrimSpace(getenv("IKIGENBA_ROOT")) != "" || strings.TrimSpace(getenv("SCRIPTS_DB_PATH")) != "" {
    return filepath.Dir(filepath.Dir(dbPath))   // on-box branch → the AppDir root
}
return filepath.Dir(generationPath)             // local-dev branch → the cache dir
```

On the box `IKIGENBA_ROOT=/opt`, so appkit composes
`DBPath = /opt/scripts/state/scripts.db` and
`GenerationPath = /opt/scripts/cache/scripts.db.generation`
(`appkit/config/config.go` `composeDataPaths`). The on-box branch returns
`filepath.Dir(filepath.Dir(DBPath))` = `/opt/scripts` — the **AppDir**, which
opsctl (D01) creates `root:root 0755`, deliberately **not** service-writable.
`runsDir` is then `/opt/scripts/runs`, and `recreateRunsDir` calls
`os.RemoveAll("/opt/scripts/runs")` + `os.MkdirAll(...)` on every boot.
Unlinking and recreating the `runs` **directory entry itself** requires **write
on its parent** `/opt/scripts` — which the `scripts` service user does not have.
Denied → the boot returns the error → exit 1 → crash-loop.

Local dev never hit this because there the *other* branch returns the cache dir
(`filepath.Dir(GenerationPath)`), and the dev cache dir is writable by the dev
user. The bug is specific to the on-box branch pointing one level too high, at a
directory that is intentionally root-owned.

**Decision.** Resolve the runs parent to the **cache dir in every layout** —
`filepath.Dir(cfg.GenerationPath)`. The two-branch `getenv` switch is removed;
`scriptsRuntimeRoot` narrows to a single argument:

```go
// scriptsRuntimeRoot returns the parent of the rebuildable runs/ tree. Runs are
// scratch, not durable state, so they live under the service-owned cache dir
// (the generation sidecar's parent) in every layout. On the D01 on-box layout
// that is <root>/scripts/cache (owned by the scripts user), never the root:root
// AppDir <root>/scripts — so recreateRunsDir can unlink and recreate runs/ on
// boot without write on the AppDir.
func scriptsRuntimeRoot(generationPath string) string {
    cacheDir := filepath.Dir(generationPath)
    if cacheDir == "" {
        return "."
    }
    return cacheDir
}
```

and the call site becomes `rootDir := scriptsRuntimeRoot(cfg.GenerationPath)`.
On the box that yields `rootDir = /opt/scripts/cache` (owned `scripts:scripts`)
and `runsDir = /opt/scripts/cache/runs`, so `recreateRunsDir` operates entirely
within a service-owned directory and boot succeeds. Nothing else in the boot
wiring changes: `runsDir := filepath.Join(rootDir, "runs")`,
`runner.New(store, rootDir, runTTL)`, and
`svc := script.NewService(store, runsDir, run)` are untouched — only the value
`scriptsRuntimeRoot` returns moves from the AppDir to the cache dir.

**Why `cache/` is the right tier (not `state/`, not a new dir).** The runs tree
is **rebuildable scratch that we do not promise to preserve** — `recreateRunsDir`
already wipes and recreates it wholesale on every boot, and the crash-recovery
sweep (§5.2) marks orphaned runs failed rather than resuming them. `cache/` is
exactly the wipeable tier of the D01 layout: opsctl backup snapshots `state/`
and **never** `cache/` (backup/restore is box-level, `state/`-only). Putting
runs under `cache/` therefore matches their durability contract — a lost or
wiped `cache/runs` costs nothing the suite promised to keep. `state/` is wrong
(it is the durable, backed-up tier; a boot-time `RemoveAll` there would be a
foot-gun), and a bespoke service-writable dir would be redundant with `cache/`.

**Supersession.** D9 corrects the runs location fixed by the older, pre-DNN
"scripts layout" work, whose tests carry the id **R-4LKF-FB23** (defined only in
the test files — `cmd/scripts/main_test.go`, `internal/web/web_test.go`,
`internal/runner/runner_test.go` — not owned by any `project/design` Decision).
Those tests currently pin runs at the AppDir root (the buggy location):

- `cmd/scripts/main_test.go` `TestScriptsBootsFromOpsctlLayoutAndServesHealth`
  computes `runsDir := filepath.Join(appRoot, "runs")` and asserts it is
  recreated empty. The build moves this to
  `runsDir := filepath.Join(cacheDir, "runs")` and additionally asserts the
  AppDir-root `filepath.Join(appRoot, "runs")` is **not** created. The id
  `R-4LKF-FB23` is retained — the test's intent ("boots from the opsctl layout
  and serves health") is preserved; only the runs path is corrected.
- `internal/web/web_test.go`
  `TestFreshSetupBootsFromNewScriptsLayoutAndPassesHealth` likewise computes
  `runsDir := filepath.Join(root, "runs")`; the build moves it to
  `filepath.Join(cacheDir, "runs")` and asserts the AppDir-root `runs` is absent.
  Id retained.
- `internal/web/web_test.go` `TestCompositionRootAdoptsNewScriptsLayout` pins the
  exact source lines of the composition root. Update its expected substrings:
  `scriptsRuntimeRoot(cfg.DBPath, cfg.GenerationPath, os.Getenv)` →
  `scriptsRuntimeRoot(cfg.GenerationPath)`. The other pinned lines
  (`runsDir := filepath.Join(rootDir, "runs")`,
  `runner.New(store, rootDir, runTTL)`,
  `svc := script.NewService(store, runsDir, run)`) are unchanged. Id retained.
- `internal/runner/runner_test.go`
  `TestSpawnUsesRebuildableRunsDirOutsideState` passes a `dataDir` directly into
  `runner.New` and asserts `<dataDir>/runs/<run_id>/`; it does **not** exercise
  `scriptsRuntimeRoot`, so it stays **unchanged**. Id retained.

**Boundary.** D9 owns only where the rebuildable `runs/` tree is rooted (the
value `scriptsRuntimeRoot` returns) and the composition-root call site. It does
**not** change: the DB/state paths (`cfg.DBPath`, still under `state/`), the
generation sidecar path (`cfg.GenerationPath`, still under `cache/`), the
runner's per-run `<dataDir>/runs/<run_id>/` tree logic, the crash-recovery sweep,
the service's MCP/landing/feed surfaces, or any nginx fragment. It is orthogonal
to D1–D8 (the landing-page series) and touches no `internal/web` landing code.

**Rejected.**
- **Put runs under `state/`.** `state/` is the durable, opsctl-backed-up tier; a
  boot-time `RemoveAll` of a subtree there is dangerous and misrepresents runs as
  preserved. Rejected — runs are explicitly not preserved.
- **`chown`/`chmod` the AppDir, or `mkdir /opt/scripts/runs` service-writable in
  opsctl setup.** Would keep runs at the root and make the AppDir (or a child of
  it) service-writable, weakening D01's deliberate `root:root` AppDir. A code fix
  that targets the already-service-owned `cache/` is simpler and changes no
  box-provisioning. Rejected.
- **Only clear the *contents* of `runs/` on boot instead of removing the dir.**
  Would also avoid the parent-write requirement (removing children needs write on
  `runs/`, which the service has), and could stay at the AppDir root. But it
  keeps runs in a root-owned parent for no benefit, complicates
  `recreateRunsDir` (walk-and-unlink children vs one `RemoveAll`), and still
  misplaces scratch outside the wipeable tier. Rejected in favor of relocating to
  `cache/` and keeping the simple `RemoveAll` + `MkdirAll`.
- **Keep the two-branch switch, just fix the on-box branch to return the cache
  dir.** Both branches would then return `filepath.Dir(generationPath)`, so the
  `getenv` switch and the `dbPath` parameter are dead weight. Collapsing to the
  single expression is the honest form. Rejected in favor of the collapse.

**Verification** (unit test on `scriptsRuntimeRoot`, plus the updated
opsctl-layout boot tests).

- R-RUNS-CDIR — **`scriptsRuntimeRoot` returns the cache dir (the generation
  sidecar's parent) for the on-box layout, not the AppDir.** A direct unit test:
  for `generationPath = <root>/scripts/cache/scripts.db.generation`,
  `scriptsRuntimeRoot(generationPath)` returns `<root>/scripts/cache` — and
  **not** `<root>/scripts`. An implementation that returns
  `filepath.Dir(filepath.Dir(dbPath))` (the AppDir root) fails this.
- R-RUNS-BOOT — **Booting from the opsctl layout recreates the runs scratch dir
  under `cache/`, and never at the AppDir root.** The updated
  `TestScriptsBootsFromOpsctlLayoutAndServesHealth` (and its `web_test.go` fresh-
  setup sibling) serve `/health` `ok` and assert `<root>/scripts/cache/runs`
  exists and is empty, while `<root>/scripts/runs` (the root-owned AppDir root)
  does **not** exist. A boot that recreates runs at the AppDir root fails this.
