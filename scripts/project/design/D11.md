# Decision 11 — Consumer loops through `Spec.Consumers` (chassis-owned) + composition-root normalization

**Problem.** scripts hand-writes its five event-plane consumer loops at the
composition root and declares its consumer role in two Spec fields that must
manually agree with those loops:

- `cmd/scripts/main.go` builds a `workers []func(context.Context) error` slice,
  one closure per upstream in `sources = ["cron","crm","ledger","dropbox","prompts"]`,
  each calling a single ~25-line `runConsumer(ctx, rt, src)` that resolves
  `SCRIPTS_<SRC>_FEED_URL` / `SCRIPTS_<SRC>_FROM`, assembles `consumer.Config`
  (with `ConsumerID: "scripts"`), logs the start line, calls `consumer.Run`, and
  wraps the error — the exact boilerplate appkit D10 folds into the chassis.
- The role is declared three times that must agree: `Spec.Consumes = sources`
  (manifest `CONSUMES=`), `Spec.Subscriptions = func(){ return consume.Subscriptions(sources) }`
  (reflection), and the `workers` loops (the actual runtime).
- `main()` smuggles the Router out of `Handlers` with a `var rt *appkit.Router`
  capture (`spec.Handlers` sets `rt = r`; the worker closures read it) and then
  **mutates the Spec after construction**: `spec := scriptsSpec(); spec.Handlers = …;
  spec.Workers = workers`. This is the post-construction mutation the reference
  shape (notify/crm) forbids — the whole Spec should be one fully-formed literal.

The chassis now owns all of this (appkit design D10, `Spec.Consumers`, consumed
here as a fixed external contract). This Decision converts scripts to the declared
table **and** normalizes the composition root to the reference inline shape. Note
scripts already reads the chassis env convention `SCRIPTS_<SRC>_FEED_URL` /
`SCRIPTS_<SRC>_FROM` (verified in the current `feedURLEnv`/`fromEnv`), so — unlike
notify — there is **no env-name migration**; only the wiring shape changes.

**Decision.** scripts declares its consumer role as the `Spec.Consumers` table,
one entry per upstream in the day-one order, and deletes the hand-rolled wiring:

```go
// cmd/scripts — the whole consumer declaration, inside the scriptsSpec() literal.
Consumers: []appkit.Consumer{
    {Source: "cron",    Subscriptions: consume.Subscriptions([]string{"cron"}),    Handler: scriptsConsumer("cron")},
    {Source: "crm",     Subscriptions: consume.Subscriptions([]string{"crm"}),     Handler: scriptsConsumer("crm")},
    {Source: "ledger",  Subscriptions: consume.Subscriptions([]string{"ledger"}),  Handler: scriptsConsumer("ledger")},
    {Source: "dropbox", Subscriptions: consume.Subscriptions([]string{"dropbox"}), Handler: scriptsConsumer("dropbox")},
    {Source: "prompts", Subscriptions: consume.Subscriptions([]string{"prompts"}), Handler: scriptsConsumer("prompts")},
},

// scriptsConsumer is the per-upstream handler factory. The chassis calls it with
// the finished Router AFTER Handlers (and Producer) have run, so svcRef — the
// domain Service that registerRoutes built — is populated. Each loop fans that
// upstream's events to matching scripts (consume.Handler is unchanged).
func scriptsConsumer(source string) func(*appkit.Router) consumer.Handler {
    return func(rt *appkit.Router) consumer.Handler {
        return consume.Handler(svcRef.RunForEvent, svcRef.ScriptsForEvent, source, rt.Logger())
    }
}
```

- **Deleted:** `runConsumer`, the `workers` slice construction and the `var rt
  *appkit.Router` capture in `main()`, the per-source env helpers `feedURLEnv` /
  `fromEnv`, the `feedDefault` function, the `consumerID` const, the `sources`
  package var, and the legacy `Spec.Consumes` / `Spec.Subscriptions` fields (both
  derived from the table by the chassis — setting either alongside `Consumers` is
  a chassis startup error). The chassis resolves `SCRIPTS_<SRC>_FEED_URL`
  (default `registry.BaseURL(src) + "/feed"`) and `SCRIPTS_<SRC>_FROM` (default
  `"tail"`) — the identical resolution `runConsumer` performed by hand.
- **Kept:** `internal/consume` untouched — `consume.Subscriptions` and
  `consume.Handler` (the fan-out semantics: fire-and-run per matching script,
  `ErrSkip` on a malformed envelope, cursor advances for every event) are the
  event semantics the chassis is agnostic to. The `svcRef` **and** `storeRef`
  package vars stay: `svcRef` is the Handlers→consumer hand-off (the consumer
  needs the *same* built Service — its runner, `runsDir`, and store — that
  `registerRoutes` constructs, not a re-derived one), and `storeRef` is the
  existing Handlers→Producer hand-off. This is a real deviation from notify, whose
  consumer factories rebuild a stateless push client from env; scripts' consumer
  needs the live domain Service, so the hand-off var is load-bearing, not
  boilerplate.
- **Composition-root normalization (menu item f).** `scriptsSpec()` returns one
  fully-formed `appkit.Spec` literal — `Consumers` and `Handlers: registerRoutes`
  declared **inside** it — and `main()` collapses to `appkit.Main(scriptsSpec())`
  with **no** post-construction `.Handlers` / `.Workers` assignment and no `var rt`
  capture. `Spec.Workers` is dropped entirely: scripts' only Workers were the
  consumer loops (the `python3`-exec runner spawns per-run goroutines from the
  Service and the crash-recovery sweep runs inline in `registerRoutes` — **neither
  is a `Spec.Worker`**), so once the loops move to `Consumers` there is no Worker
  left to keep. (The task inventory's "runner/sweep in Workers" was inaccurate:
  grep confirms `Spec.Workers` held only `runConsumer` closures.)
- **Behavior is unchanged:** same five upstreams in the same order, same broad
  one-subscription-per-source in-edges (`Filter: "**"` under the event-routing
  revision — D17 owns the filter string and the fan-out's key matching), same
  `feed_offset` cursors (`consumer_id` `"scripts"`
  == `Spec.App`, one row per source — the chassis derives exactly the values
  `runConsumer` hardcoded), and the manifest still says
  `CONSUMES=cron,crm,ledger,dropbox,prompts` byte-for-byte.
- **Self-chaining stays OUT.** The `TODO(self-chaining)` comment (adding
  `"scripts"` pointed at the local `/feed`) is **not** realized here — it is a
  deliberate future item; this conversion preserves the day-one five-upstream
  behavior exactly. The comment is dropped from `main()` (there is no longer a
  `sources` slice to annotate); the intent lives on in ARCHITECTURE/PLAN and this
  note.
- **Dev wiring:** none. `bin/start`'s `launch_scripts` already exports the generic
  `export_feed_urls SCRIPTS cron crm ledger dropbox prompts` form the chassis
  reads; `.envrc` sets no feed vars. No boundary-crossing change (contrast the
  `SCRIPTS_WWW_PATH` line D12 needs).

**Retired ids.** `R-RGST-PEER` (D10, `TestFeedDefaultUsesRegistryBaseURLs`) pinned
scripts' local `feedDefault(src) == registry.BaseURL(src)+"/feed"`. With
`feedDefault` deleted, peer feed-URL default resolution becomes **chassis-owned**
and is pinned by appkit's own D10 ids (`R-464U-T3T1` default = registry loopback,
`R-47CR-6VJQ` env override). `R-RGST-PEER`'s test is deleted with the function it
covered — the behavior did not vanish, it moved behind the chassis contract. The
other D10 ids stay: `R-RGST-SELF` (own port via `MustPort`), `R-RGST-DBOX` (the
import fetcher's `DROPBOX_BASE_URL` default — **not** a consumer feed; it stays in
`registerRoutes`), `R-RGST-NLIT` (no `30xx` literal), `R-RGST-GMOD` (go.mod
require/replace) are all unaffected.

**Rejected.**
- **Rebuilding the Service inside each consumer factory from `rt.DB()`** — would
  stand up a *second* runner, `runsDir`, and store parallel to the one Handlers
  built, double-running the crash-recovery sweep and diverging run state; the
  `svcRef` hand-off shares the single live Service, which is the correct and
  existing behavior.
- **Keeping a one-line `sources` slice and looping to build `Consumers`** — the
  manifest verb must read the sources statically without a Router (appkit D10
  rejected a whole-table provider for exactly this), and an explicit five-entry
  literal is the reference shape; a loop hides the declaration order the manifest
  and reflection depend on.
- **Keeping `Spec.Workers` for the runner/sweep** — they are not Workers today and
  making them so would change their lifecycle (the sweep must complete *before*
  serving; a Worker runs *alongside* serve); left exactly as-is in
  `registerRoutes`.
- **Realizing self-chaining in the same change** — scope creep; a new upstream is
  a behavior change with its own verification, not part of a shape conversion.

**Verification** (the Spec table directly, and the handler factory against a real
SSE feed — the substrate appkit's own D10 consumer-engine verification uses).
- R-8WN1-0VQI — `scriptsSpec().Consumers` has **exactly five** entries in the
  order `cron, crm, ledger, dropbox, prompts`, each entry's `Subscriptions` equal
  to `consume.Subscriptions([]string{entry.Source})` (a single broad `"**"`
  subscription for that source, per D17), and `scriptsSpec()` sets **neither** legacy
  `Consumes` **nor** `Subscriptions` — a dropped upstream, a reordering, a diverged
  subscription, or a lingering legacy field fails this. The manifest byte-equality
  test (`R-8IAN-FB87`) derives its expected `CONSUMES=` from these five sources, so
  the same drift fails the manifest too.
- R-8XUX-ENH7 — an entry's handler factory, invoked with a Router after `svcRef`
  is set, returns a working `consumer.Handler`: fed a well-formed event whose
  canonical key matches a script's trigger filter (D17) it fires a run for that
  script (via the injected `RunForEvent`/`ScriptsForEvent`), and fed a malformed
  envelope (empty kind/id) it returns `consumer.ErrSkip` — the same
  `consume.Handler` contract the pre-conversion worker drove, now reached
  through the factory.
