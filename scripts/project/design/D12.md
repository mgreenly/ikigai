# Decision 12 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** scripts' landing page and static assets are embedded in the binary
(`//go:embed` in `scripts/internal/web/`): `embed.go` embeds `landing.html` and
the `static/` tree (`tokens.css` + four woff2 fonts), `web.go` renders the page
(`web.LandingHandler`) and serves the assets (`web.StaticHandler`), and
`cmd/scripts/main.go` mounts both (`GET /{$}`, `GET /static/`). The suite is
reverting embedding suite-wide: web assets are release artifacts that belong on
disk — visible and diffable in the versioned release tree, editable in dev without
a rebuild, served by one chassis mechanism instead of twelve drifting copies. crm
and notify are the pilot conversions; scripts follows the same shape. This
Decision is the service-side half of appkit's chassis design (its D5 www-root
resolution, D6 `appkit/web`, D7 `Spec.WWW`/`Router.WWW()` — consumed here as a
fixed external contract), and it **supersedes the embedded-substrate mechanism**
described in D1 (the `web.LandingHandler` constructor), D2 (the service-side
`GET /static/` mount), D3 (the `//go:embed static` asset serving), and D8 (the
`StaticHandler` wiring). Those Decisions' *behaviors* — the page shows service
name + version, the assets carry the right content types, the fonts are
self-served swap-free, the nginx static gate — are retained and re-proven on the
new substrate; only the loading mechanism changes.

**Decision.** scripts serves its web surface from the on-disk www root.

1. **The files move (same bytes).** `scripts/internal/web/landing.html` →
   `scripts/share/www/landing.html`; `scripts/internal/web/static/` →
   `scripts/share/www/static/` — `tokens.css` and all four fonts
   (`space-grotesk.woff2`, `ibm-plex-sans.woff2`, `ibm-plex-mono-400.woff2`,
   `ibm-plex-mono-500.woff2`). `bin/ship` already bundles `<svc>/share/` into the
   release `share/<version>` tier; on the box the chassis resolves the root to
   `share/current/www`, in dev to `./share/www` or the `SCRIPTS_WWW_PATH` override.
2. **The Spec opts in.** `scriptsSpec()` sets `WWW: true`. The chassis loads the
   site at serve (failing startup loudly if the root is missing), auto-mounts
   `GET /static/`, and exposes the loaded site as `rt.WWW()`.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `registerRoutes` stays ungated in-process (nginx is the session gate, D4/D8),
   but its handler renders through `rt.WWW().Render(w, "landing.html", data)` with
   the `{Service, Version}` struct — the crm/notify landing shape — instead of
   `web.LandingHandler(rt.Service(), rt.Version())`. The service-side
   `rt.Handle("GET /static/", web.StaticHandler())` line is **deleted** — the
   chassis owns the static mount.
4. **`scripts/internal/web/` is deleted** — `embed.go`, `web.go`, `landing.html`,
   the `static/` tree, and all of its `*_test.go`. The landing/asset/font tests and
   the composition-root/boot tests currently in that package (`web_test.go`) and
   the nginx-fragment test (`nginx_test.go`) relocate to `cmd/scripts` (package
   `main`), loading the repo-real `scripts/share/www` tree via `appkit/web`
   (`web.Load` on a relative path from the package dir) so they exercise the exact
   files that ship. The nginx-fragment assertions are unchanged (they read
   `../../etc/nginx.conf`, the same relative depth from `cmd/scripts` as from
   `internal/web`, and keep their registry-derived port expectations).
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_scripts` exports
   `SCRIPTS_WWW_PATH="$repo/scripts/share/www"` beside its existing exports, so a
   repo-root-launched dev binary finds the source tree. **This one line crosses the
   `scripts/` boundary** — it is the single sanctioned exception, verified by the
   live `bin/start` smoke, not the Go suite. Prod sets no such variable (the
   chassis resolves `share/current/www`), so no box change.
6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6), the MCP surface, the consumer/producer wiring, and the migrations
   are untouched. The embedded migrations (`internal/db`) stay embedded — they are
   not web assets (D14).

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work completes; embedded
  assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's user content,
  backed up with `state/`; scripts' UI assets are release artifacts and live in the
  shipped `share` tier (appkit D5 records the argument).
- **Keeping a thin `internal/web` for the landing closure** — with the mechanism in
  `appkit/web` there is nothing left but a render closure; the composition root is
  the right home for it (crm/notify precedent). Unlike `internal/mcp` (D13), no
  domain logic, schema, or test harness justifies the package.
- **Serving assets from both embed and disk during a transition** — two sources of
  truth; the conversion is atomic within scripts.

**Retained ids (re-proven on the `scripts/share/www` substrate).** D1's
R-LAND-7Q3D / R-LAND-9R5F / R-LAND-1S7G / R-LAND-3T9H (page renders name+version,
`text/html` content type), D2's R-ROUT-8U2J / R-ROUT-1V4K / R-ROUT-3W6L (`GET /{$}`
exact-root dispatch, no shadowing), D3's R-ASST-5X8M / R-ASST-7Y1N / R-ASST-9Z3P
(assets served with correct content types, app-local paths, no cross-service URL),
D7's R-HOME-8R2V (Home link to the apex), and D8's R-M59W-5CAW / R-M6HS-J41L /
R-M8XL-ANIZ / R-MA5H-OF9O / R-MBDE-270D (font-display `optional`, service-relative
font `src`, service-relative stylesheet link, crossorigin preloads, nginx static
gate) keep their assertions; their tests move to `cmd/scripts` and load the shipped
tree through `appkit/web` / the chassis static mount rather than `internal/web`'s
embed. The D4 nginx ids (R-NGNX-2A5Q / R-NGNX-4B7R / R-NGNX-6C9S / R-NGNX-8D1T) and
D6's structural tokens/markup check move unchanged.

**Verification** (over the repo-real `scripts/share/www` tree; the retained ids
above continue to pin the page/asset behaviors through the new substrate).
- R-8Z2T-SF7W — an `appkit/web` Site loaded from `scripts/share/www` (the actual
  shipped tree, resolved relative to the `cmd/scripts` test package) renders
  `landing.html` with an injected service and version visible in the output —
  proving the shipped template, not an embedded copy, is the page source.
- R-90AQ-66YL — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  scripts-side static handler registered — proving the assets ship on disk and the
  chassis serves them.
