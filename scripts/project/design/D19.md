# Decision 19 — Structured MCP adoption: `structuredContent`, output schemas, typed error codes

**Problem.** scripts' domain tools were written before the suite settled its
machine-facing verb contract. Today `internal/mcp/tools.go` builds every
success result with the local `toolResultJSON(v)` wrapper over the chassis
`JSONResult(v)` — a struct marshalled to a JSON string and shipped inside a
single text block — and every failure with `toolResultErr(msg)` over
`ErrorResult(msg)`, a bare human string with no machine-branchable code. No tool
declares an `outputSchema`. That surface is agent-shaped: an agent tolerates
JSON-in-a-string and prose errors, but the scripts runtime (the suite's first
deterministic MCP caller) must parse results and branch on error kind without
sniffing message text.

The suite-level contract that fixes this is settled in
`docs/structured-mcp-design.md`: a domain verb is registered once and answers
every caller, carrying `structuredContent` for machines and a mirrored text
block for agents in one response; each verb declares an `outputSchema` as its
machine contract; errors carry a code from a closed vocabulary; and
documentation tools plus raw-content reads stay plain text. appkit has already
landed the transport side of that contract (protocol `2025-06-18`;
`Tool.OutputSchema`; `StructuredResult(v) (map[string]any, error)`;
`ErrorResult(code ErrorCode, msg string)` over the closed vocabulary
`validation`/`not_found`/`conflict`/`too_large`/`source_unavailable`/`internal`;
`JSONResult` deleted). scripts currently does **not** compile against that appkit
— this Decision is the service-side adoption that makes it green again. This is a
**formalization of the JSON scripts already emits, never a reshaping of any
result**: the same fields, in the same casing, now also travel as
`structuredContent` under a declared schema.

This Decision owns **no guard change**: scripts hand-rolls no loopback guard
(its only loopback endpoints are the chassis-owned `/feed`, already wrapped by
appkit's shared guard; `internal/script/dropbox.go` is a loopback *caller*, not a
guarded endpoint). The `X-Forwarded-Proto`-only guard is entirely upstream.

**Out of scope.** The scripts *runtime* feature that this machine surface enables
— the runner-injected `suite` Python module, the `GET /run-content` holder
endpoint, and the `content_url` on `run_fs_list` entries described in
`docs/structured-mcp-design.md` — is a **separate later spec** and is not
designed or planned here. This Decision changes only the shape of the existing
sixteen tools' results.

## Tool classification

The sixteen domain tools split into **thirteen structured** verbs and **three
prose exceptions**. The split is fixed here so a build turn cannot guess it:

**Structured (return `StructuredResult`, declare an `outputSchema`):**

| tool | emitted JSON (verbatim, unchanged) |
|---|---|
| `create` | `{script_id}` |
| `import` | `{script_id, name}` |
| `list` | `{scripts: [ScriptDetail…]}` |
| `get` | one `ScriptDetail` object |
| `update` | one `Script` object |
| `delete` | `{deleted}` |
| `set_trigger` | one `Trigger` object |
| `clear_trigger` | `{cleared}` |
| `run` | `{run_id, status, started_at}` |
| `run_list` | `{runs: [Run…]}` |
| `run_get` | one `Run` object |
| `run_cancel` | `{cancelled}` |
| `run_fs_list` | `{entries: [FileEntry…]}` |

**Prose exceptions (keep `TextResult`, declare NO `outputSchema`):**

- `describe` — a documentation tool: it returns the human `describeText`
  overview, exactly the `guide`/`describe` case the suite contract exempts.
- `run_output` — a **raw-content read**: it returns a run's captured
  stdout/stderr text verbatim (the byte stream, not a fact object).
- `run_fs_read` — a **raw-content read**: it returns a file's text from the run
  dir verbatim.

The two raw-content reads are prose because their payload *is* opaque text a
machine consumes as bytes, not a structured fact; forcing them into a JSON
object would reshape a byte stream into a wrapper, which the content plane
(future `run-content` holder, out of scope) exists to avoid.

## Output-schema fidelity — the casing trap

Each structured tool declares a hand-authored `map[string]any` JSON-schema
literal (`type`/`properties`/`required`, `additionalProperties: true` where the
object is open), following the `objectSchema` helper house style in
`appkit/mcp/mcp.go`. scripts writes its own small private helpers
(`objSchema`/`strp`-style literals) in `internal/mcp`, mirroring the chassis
pattern; it does not import chassis-private helpers.

The schemas must mirror the JSON the tools **already emit, byte-for-byte**, and
the domain structs are **not uniformly cased** — a schema that "helpfully"
snake_cases everything would diverge from the wire and is wrong:

- `Script` and `ScriptDetail` (and `Trigger`) carry **no `json` tags**, so they
  marshal with **Go field names verbatim**: `ID`, `OwnerEmail`, `Name`, `Body`,
  `Config`, `SourcePath`, `CreatedAt`, `UpdatedAt`; `ScriptDetail` inlines those
  and adds `RunningCount` and `LastRun`. `Config` is the one nested tagged
  object (`interpreter`, `timeout_secs`, both `omitempty`).
- `Run` and `FileEntry` **do** carry `json` tags, so they marshal snake_case:
  `id`, `script_id`, `status`, `exit_code`, `started_at`, `ended_at`, `error`,
  `trigger_source`, `trigger_kind`, `trigger_subject`, `trigger_event_id`,
  `stdout_path`, `stderr_path`, `elapsed_secs`; and `path`, `is_dir`, `size`.

Where a nested object's exact key set is incidental to the machine contract
(`Config`, the embedded `Script` inside `ScriptDetail`), the schema may declare
it `type: object, additionalProperties: true` rather than re-enumerating every
field — but the **top-level** keys of each result are enumerated and required as
the table above emits them, in the casing above. The declared schema is proven
against the emitted `structuredContent`, not hand-eyeballed (see R-C43Q-0BYO).

## Structured errors — the code mapping

`toolResultErr(msg)` becomes a mapping helper `structuredError(err) map[string]any`
that returns `ErrorResult(code, err.Error())`, choosing `code` by
`errors.Is` against the domain's sentinel errors — the only classification the
domain actually raises:

- `errors.Is(err, script.ErrNotFound)` → `mcp.ErrNotFound` (`not_found`)
- `errors.Is(err, script.ErrValidation)` → `mcp.ErrValidation` (`validation`)
- otherwise → `mcp.ErrInternal` (`internal`)

The result keeps `isError: true` and the human message (unchanged text) and adds
`structuredContent: {code, message}`, so a machine branches on `code` and never
on message text. The `paramError` → JSON-RPC `-32602` path is untouched (those
are transport-level malformed-argument errors, not tool results).

**On the three unused codes** (`conflict`, `too_large`, `source_unavailable`) —
this is a deliberate mapping, not an omission: scripts' domain raises none of
them today, and this adoption formalizes existing classification rather than
manufacturing finer codes the domain does not distinguish.

- `too_large`: the 1 MiB import cap is modelled by the domain as an
  `ErrValidation` (`internal/script/service.go`, `maxImportBytes`), so it maps to
  `validation` like every other bad input. Reclassifying it would require a new
  domain sentinel, out of scope here.
- `conflict`: `import` upserts on `(owner, source_path)` — re-importing a path is
  an update, never a collision — and no other verb has a conflict state.
- `source_unavailable`: a dropbox-mirror fetch failure inside `import` surfaces
  today as an unclassified error and therefore maps to `internal`; giving it a
  distinct code is a future domain-sentinel refinement, noted for the reviewer,
  not smuggled in by string-sniffing a wrapped message here.

## `StructuredResult` returns an error — handle it honestly

`JSONResult(v) (map[string]any, error)` and `StructuredResult(v) (map[string]any,
error)` share a signature, so each `return toolResultJSON(x)` site becomes a
`return mcp.StructuredResult(x)` (or the local `toolResultJSON` wrapper is
repointed at `StructuredResult`). The marshal error is **not** swallowed: a
`StructuredResult` error is a genuine internal marshalling failure and propagates
as the handler's `error` return (rendered by the chassis as a JSON-RPC
`-32603`), exactly as `JSONResult`'s error did. No site discards it or coerces it
into a fake success.

**Decision.** `internal/mcp/tools.go` is edited in place: `toolResultJSON` is
repointed at `StructuredResult` (or replaced at each call site); `toolResultErr`
becomes `structuredError(err)` keyed on the sentinels above; the thirteen
structured tools each gain a hand-authored `OutputSchema` in their `Tool`
descriptor; `describe`, `run_output`, and `run_fs_read` keep `TextResult` and
declare no schema. `internal/mcp/mcp.go`, `describe.go`, and the composition root
are unchanged. Wire tool count stays **18** (16 domain + chassis
`health`/`reflection`); D13's partition is preserved.

**Rejected.**
- **A second loopback domain API** mirroring the MCP verbs — the road not taken
  in `docs/structured-mcp-design.md`: a permanent second surface to guard and
  keep in sync, to save a machine caller one JSON-RPC envelope. MCP is the single
  verb surface.
- **String-sniffing wrapped error messages to synthesize `too_large` /
  `source_unavailable`** — fragile and a reshaping of classification the domain
  does not make; the honest mapping keys on the domain's real sentinels, and
  finer codes wait for real domain sentinels.
- **Snake-casing every output schema for tidiness** — would diverge the declared
  contract from the bytes `Script`/`ScriptDetail`/`Trigger` actually emit
  (PascalCase, untagged). The schema mirrors the wire, not a preference.
- **Making `run_output` / `run_fs_read` structured** — they return opaque byte
  streams; wrapping them in a fact object is exactly what the content plane
  exists to prevent.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a real `script.Service` on an `appkitdb` temp DB,
driven by `httptest` against the real handler; the substrate that can falsify
each claim is scripts' own assembled MCP layer).

- R-C0G0-V0QL — for **every** structured domain tool (`create`, `import`,
  `list`, `get`, `update`, `delete`, `set_trigger`, `clear_trigger`, `run`,
  `run_list`, `run_get`, `run_cancel`, `run_fs_list`), a successful `tools/call`
  result carries a `structuredContent` **object** (not a JSON string) that
  deep-equals the JSON parsed from the result's mirrored text content block —
  proving `StructuredResult` replaced `JSONResult` on every structured site and
  the two renderings agree. A table-driven test over the thirteen tools; a tool
  still returning a bare text block (no `structuredContent`) fails it.
- R-C1NX-8SHA — the three prose tools `describe`, `run_output`, and
  `run_fs_read` return a text content block and **no** `structuredContent` key in
  their `tools/call` result — proving they stayed on `TextResult`. A tool that
  gained a `structuredContent` fails it.
- R-C2VT-MK7Z — in `tools/list`, each of the thirteen structured domain tools
  carries a non-empty `outputSchema`, and each of the three prose tools
  (`describe`, `run_output`, `run_fs_read`) carries **no** `outputSchema` key —
  the exact 13-present / 3-absent partition. A prose tool that declares a schema,
  or a structured tool missing one, fails it.
- R-C43Q-0BYO — schema fidelity against the wire: for `update` (a `Script`) and
  `run_get` (a `Run`), every top-level key present in the tool's emitted
  `structuredContent` is a declared property of that tool's `outputSchema`, and
  the `update` schema declares the **PascalCase** `Script` keys (`ID`, `Name`,
  `Body`, `Config`, …) while the `run_get` schema declares the **snake_case**
  `Run` keys (`id`, `script_id`, `exit_code`, …) — proving the schema mirrors the
  actual casing on the wire. A schema that snake_cases the `Script` keys diverges
  from the emitted object and fails it.
- R-C5BM-E3PD — a validation failure surfaces the `validation` code: a `create`
  call with an empty `name` returns `isError: true` and
  `structuredContent.code == "validation"` (mapped from `script.ErrValidation`).
  A result carrying `internal` or no code fails it.
- R-C6JI-RVG2 — a not-found failure surfaces the `not_found` code: a `get`
  (or `run_get`) call with an unknown id returns `isError: true` and
  `structuredContent.code == "not_found"` (mapped from `script.ErrNotFound`), not
  `validation` and not `internal`.
- R-C7RF-5N6R — a non-sentinel failure defaults to the `internal` code: an
  `import` whose injected `Fetcher` returns a plain (non-`ErrNotFound`,
  non-`ErrValidation`) error returns `isError: true` and
  `structuredContent.code == "internal"` — proving the default arm of the mapping,
  distinct from the two sentinel arms.
- R-CA77-X6O5 — structural: no `JSONResult` reference remains in scripts'
  non-test source —
  `grep -rn 'JSONResult' internal cmd --include='*.go' | grep -v _test.go`
  returns empty.
