# Decision 3 — scripts's own Carbon design assets (shipped in `share/www/static`)

> **Substrate note (D12).** These assets originally lived in
> `scripts/internal/web/static/`, embedded via `//go:embed`. **D12 de-embedded
> them:** they now ship on disk under `scripts/share/www/static/` and are served by
> the chassis static mount (`Spec.WWW`). This Decision is rewritten to that
> substrate; the self-contained-copy argument and the `R-ASST-*` ids are unchanged.

**Decision.** scripts ships its **own** copy of the Carbon design assets and serves
them from its own mount; the landing page references those assets, never a
cross-service URL. Concretely:

- A `scripts/share/www/static/` directory holds a **vendored copy** of
  `design/tokens.css` (the Carbon CSS custom properties) and the woff2 fonts the
  Carbon type pairing needs: `space-grotesk.woff2` (display), `ibm-plex-sans.woff2`
  (body), `ibm-plex-mono-400.woff2` + `ibm-plex-mono-500.woff2` (mono labels) —
  the same set the dashboard already ships.
- The chassis serves this directory (via `Spec.WWW` / the auto-mounted
  `GET /static/`) under a stable in-service path — `GET /static/…` — with correct
  content types (`text/css; charset=utf-8` for `tokens.css`, `font/woff2` for the
  fonts).
- `tokens.css` declares its `@font-face` rules pointing at the **service-relative**
  font paths (`/srv/scripts/static/fonts/…` as seen by the browser, i.e. the
  service's `/static/fonts/…` once nginx re-adds the mount prefix), and the landing
  template links `tokens.css` from the same `/static/` path. No asset is fetched
  from the dashboard or any other service at runtime.

**Why each app ships its own copy.** The brief and the dashboard precedent make
this non-negotiable: the suite deliberately keeps each app self-contained rather
than sharing a design module. Shipping the tokens + fonts under
`scripts/share/www/static/` keeps scripts independent with no runtime dependency on
another app's assets — the whole point of the one-binary-per-service model. (The
files ship in the release `share` tier rather than embedded in the binary — D12's
suite-wide de-embed — but the independence property is identical.)

**Carbon fidelity.** The page consumes the **semantic** Carbon tokens
(`--color-*`, `--space-*`, type tokens), not raw ramps, per `design/carbon.md`:
monochrome neutrals; blue `#2563EB` reserved as the only signal color (used here
only for a focus ring / link if present, never decoratively); 4px spacing grid;
tight 4px radius; hairline borders over shadow. The landing card is minimal and
on-system: service name in **Display** (Space Grotesk, 56/1.04/700), version in
the **Mono · Label** token (IBM Plex Mono, 12/1.4/500), centered within the
1120px max content width.

**Staleness note (vendored copy).** Because `tokens.css` and the fonts are a
vendored copy, they can drift from `design/` over time. That is an accepted cost
of the module boundary (the dashboard lives with the same), and is out of scope
to automate here; if the design system changes, the copy is re-vendored as a new
plan phase. v1 copies the current `design/tokens.css` and the four woff2 fonts
verbatim.

**Rejected.**
- **Linking the dashboard's `/static/tokens.css` at runtime** — couples scripts to
  the dashboard's asset URLs and breaks the self-contained one-binary model; a
  dashboard asset rename would silently unstyle scripts.
- **A shared design module imported by every app** — `go:embed` cannot cross the
  module boundary, and the suite deliberately keeps each app a standalone module;
  vendoring is the established pattern (dashboard).
- **Inlining all CSS into the HTML and shipping no fonts** — loses the woff2 type
  pairing (the page would fall back to system fonts and look off-system) and
  diverges from the dashboard's embedded-assets approach the brief points at.
- **Serving the fonts from a CDN** — the box is single-tenant and may run with no
  external egress guarantees; assets must be local.

**Verification** (over the shipped `scripts/share/www` tree through the chassis
static mount).
- R-ASST-5X8M — a `GET /static/tokens.css` is served by the chassis static mount
  with status `200` and a CSS content type (`text/css`).
- R-ASST-7Y1N — the landing template's stylesheet link and the page's font
  references resolve to scripts's **own** `/static/` path (the rendered HTML
  contains the app-local asset path and contains **no** cross-service/dashboard
  asset URL).
- R-ASST-9Z3P — the woff2 font assets ship and are served: a `GET` for a font path
  (e.g. `/static/fonts/space-grotesk.woff2`) returns `200` with a `font/woff2`
  content type.
