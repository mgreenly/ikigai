# Decision 18 — Event-routing conformance: producer kinds `succeeded`/`failed`, subject = /<script name>

> **External ordering (operator-sequenced, not built by any scripts phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before this Decision's phase runs**. It equally assumes appkit (which holds
> `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, this Decision's phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject` and producer registries become **families**. Hard
cutover, no compatibility period, inside the no-live-data migration window.

**What scripts publishes today (ground truth, not vacuous).** scripts is a
real, wired producer: `Spec.Feed: "/feed"`, `Spec.Events: script.Events` (an
`outbox.Registry` of two `EventType`s in `internal/script/outcome.go`), the
`Producer` hook injecting the outbox onto the `Store`, and one emission path —
`Store.FinishRun` → `completionEvent` → `outbox.Append(tx, Event{Type,
Payload})` on the **same transaction** as the run's terminal-state write
(at-most-once per run; a cancelled run emits nothing; `Ring()` after commit).
Both published types — `EventSucceeded = "scripts.succeeded"`, `EventFailed =
"scripts.failed"` — convert.

**Decision.** scripts adopts the revised envelope exactly over what it
publishes, per the routing doc's producer key map — **the script's name as the
subject**:

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `scripts.succeeded` | `scripts:succeeded/nightly-export` | `succeeded` | `/` + the script's name |
| `scripts.failed` | `scripts:failed/nightly-export` | `failed` | `/` + the script's name |

The kinds **drop the `scripts.` prefix** — grounded in the actual type
strings: unlike prompts, whose `run.succeeded` prefix is the domain *entity*
(the run), scripts' prefix is literally the service name, the exact redundant
restatement the routing doc's rename rule deletes (`source` already says
scripts; a `scripts:scripts.succeeded/...` key would say it twice). The
routing doc's key map records `scripts:succeeded|failed/<script name>`
verbatim. The **subject** is `/` + the script's name — the routing dimension a
chaining consumer wants (`scripts:succeeded/nightly-export` fires exactly one
downstream prompt or script). Names are user text, so the derivation is
pinned, mirroring prompts D25: subject is `""` when the script name is empty
(reachable — the runner finishes a run of a tombstoned/deleted script with
`ScriptName == ""`); otherwise `"/" + name` with any newline replaced by a
space (`routing.ValidSubject` requires a single-line `/`-rooted path; a name
containing `/` simply yields extra routing segments, which is fine — the
subject is a name, and the payload still carries the exact `script_name`).

1. **Constants, payload, and event builder** (`internal/script/outcome.go`):
   `EventSucceeded`/`EventFailed` become **kind** constants with values
   `"succeeded"`/`"failed"`; `completionEvent` derives the subject per the
   rule above and returns `outbox.Event{Kind, Subject, Payload}` (same
   `(event, shouldEmit, err)` shape; cancelled still emits nothing). The
   `completionPayload` is otherwise **unchanged** — `script_id`,
   `script_name`, `run_id`, `status`, `exit_code`, `stdout`/`stderr` tails +
   truncation flags, `error` (failed only) — except its nested `trigger`
   object follows D17's run-row reshape: `triggerPayload` becomes
   `{source, kind, subject, event_id}` (JSON keys `source`/`kind`/`subject`/
   `event_id`; no `type` key), populated from `FinishRunInput.TriggerKind`/
   `TriggerSubject`. The same-tx atomicity, the at-most-once rule,
   post-commit `Ring()`, and the cancelled-emits-nothing rule are untouched.

2. **The registry becomes families** (`internal/script/outcome.go`):
   `script.Events` becomes an `outbox.Registry` of two `outbox.Family`
   entries — `{Kind: "succeeded", Subject: "/<script name>",
   Description: …, Sample: sampleSuccess}` and likewise `failed` (Samples
   updated to the reshaped trigger object with revised-model values, e.g.
   trigger source `cron`, kind `tick`, subject `/nightly`). `Spec.Events`
   wiring is untouched; Append-time kind gating and the chassis reflection
   surface hang off the same registry, now family-shaped.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`;
   each producer re-applies the DDL through its own runner. scripts appends
   **one new timestamped migration** minted with `bin/create-migration
   scripts outbox_routing` (never a hand-picked number) that drops and
   recreates the outbox table per the revised `outbox.SchemaSQL` verbatim (a
   `DROP TABLE`/`DROP INDEX` preamble + the constant's DDL). Outbox rows are
   disposable retention data inside the migration window; nothing is
   migrated across. The frozen `004_outbox.sql` is **not touched**.
   *Divergence from the siblings:* gmail/dropbox/ledger/prompts each have an
   existing `internal/db/migrations_outbox_test.go` DDL drift guard to
   re-point; scripts has **none** (grounded: no test in scripts asserts
   `outbox.SchemaSQL` today). This Decision **adds** that guard —
   `internal/db/migrations_outbox_test.go`, sibling-shaped — pointed at the
   **newest** outbox migration, asserting it contains the revised
   `outbox.SchemaSQL` verbatim, closing a gap the hand-rolled setup left.
   (Otherwise this exactly mirrors gmail's D18/phase-16, dropbox's
   D22/phase-23, ledger's D15/phase-13, and prompts' D25/phase-29 handling.)

4. **Reflection/consumer surface.** The chassis reflection tool reflects the
   family registry: scripts' two kinds — and only they — are listed with the
   `/<script name>` subject description, beside the D17 `"**"` subscribes.
   The consumer half is D17's; nothing here touches it.

**Rejected.**
- **Keeping the `scripts.` prefix as kinds (`scripts.succeeded`)** — the
  revision's explicit rename rule: `source` carries the noun, and here the
  prefix *is* the source name, the clearest-cut case in the suite; contrast
  prompts, which keeps `run.` because run is its entity, not its name.
- **Renaming to entity-style kinds (`run.succeeded`, prompts' shape)** — the
  routing doc's key map records bare `succeeded|failed` for scripts, and the
  current type strings carry no entity to preserve; inventing one now is a
  gratuitous rename beyond conformance.
- **Subject = `/<script_id>`** — nobody routes on a ULID; the name is the
  human routing handle, the id stays in the payload for machines.
- **An empty subject (gmail/ledger's choice)** — scripts has an identified
  routing dimension *today*: chaining consumers want "when *that* script
  succeeds", which the name gives and emptiness forecloses.
- **Rejecting/renaming scripts whose names make awkward subjects** — the
  create surface is out of scope and a name is a name; the pinned derivation
  (newline → space, empty → `""`) keeps `Append` valid without constraining
  users.
- **Editing `004_outbox.sql` in place / an `ALTER TABLE` preserving rows** —
  committed migrations are immutable, and preserving ≤7-day retention rows
  inside the migration window is pointless ceremony that would fabricate
  kind/subject for old rows.
- **Skipping the drift guard because scripts never had one** — the guard is
  the suite's mechanism for keeping a library-owned DDL and a service-held
  migration in lockstep; leaving scripts as the one producer without it
  preserves an accident, not a decision.

**Verification** (real substrates throughout: emission claims run over a real
SQLite database migrated through scripts' full embedded migration set
(`db.FS`), outbox rows read back by SQL; the feed claim runs the real
eventplane `FeedHandler` over `httptest`; the reflection claim drives the
assembled `appkit/mcp` handler).

- R-82AG-F74Z — after `FinishRun` lands a succeeded run of a script named
  `nightly-export`, a failed run of the same script, and a cancelled run: the
  real SQLite outbox holds exactly two rows, with `kind` exactly `succeeded`
  and `failed` (no `scripts.` prefix), each `subject` exactly
  `/nightly-export`, each appended on the same tx as the terminal-state
  write; the failed row's payload carries `error` and both payloads' nested
  `trigger` object carries `kind` and `subject` keys and **no** `type` key; a
  run finished with an empty `ScriptName` yields `subject = ''`.
- R-83IC-SYVO — `script.Events` declares exactly two families with kinds
  `succeeded`/`failed` and subject description `/<script name>`; through the
  assembled MCP handler, reflection's index lists exactly those two
  publishes, the detail of `succeeded` returns a schema and example that
  agree (every example field in the schema, `completionPayload`-shaped, the
  trigger example carrying `kind`/`subject` and no `type`), and an undeclared
  kind returns the typed unknown-kind error naming the two declared kinds.
- R-84Q9-6QMD — applying scripts' full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table
  with `kind` and `subject` columns and **no** `type` column; the newly added
  DDL drift guard (`internal/db/migrations_outbox_test.go`) asserts the
  newest outbox migration contains the revised `outbox.SchemaSQL` verbatim
  while `004_outbox.sql` remains byte-identical to its committed (frozen)
  body.
- R-85Y5-KID2 — end-to-end over scripts' own substrate: an event appended
  through scripts' producer path (`FinishRun` → `completionEvent` → `Append`
  on the migration-created table, source `scripts` — the `Spec.App` value)
  for a succeeded run of script `nightly-export` and served by the real
  eventplane `FeedHandler` over an `httptest` connection frames
  `event: scripts:succeeded/nightly-export` (the canonical key) with a
  `data:` JSON envelope carrying `kind` and `subject` keys and **no** `type`
  key.
