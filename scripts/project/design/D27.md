# Decision 27 — `suite.files` share paths: client-side leading-slash normalization, absolute-canonical teaching

**Problem.** The file share's paths are absolute (`/`-rooted mirror display
paths — D24 already states this), but nothing on the client side repairs a
missing leading slash, and `describe`'s worked example
(`suite.files.put("reports/summary.pdf", …)`) actively teaches the relative
spelling. Against the share server as it stood, that spelling half-worked
(`mkdir`) or failed as an opaque `ToolError(source_unavailable)` (`put`) —
the 2026-07-15 live e2e hit exactly this. The share service (dropbox D24, its
project) now canonicalizes server-side; this Decision is the scripts-side
half: the `suite` module sends canonical paths regardless of user spelling,
and the taught contract says so. One canonicalizer (the server), one
convenience shim (this client); the client deliberately does **not**
duplicate lexical cleaning.

**Decision.**

**1. Leading-slash normalization in `suite.py`.** A module-level helper in
`internal/runner/suite.py`:

```python
def _share_path(p):
    """Return the share path rooted: prefix '/' when missing."""
    if p and not p.startswith("/"):
        return "/" + p
    return p
```

applied to every share-path argument before it is URL-escaped into the query:
`stat(path)`, `get(share_path, …)`, `put(…, share_path)`, `delete(path)`,
`mkdir(path)`, and **both** of `move(src, dest)`'s arguments. `list(path=…)`
applies it to a *given* prefix only; an absent/`None` path still sends no
`path` param (list-everything is unchanged). Local filesystem arguments
(`put`'s `source`, `get`'s `dest`) are untouched — they are run-dir paths,
not share paths. `fetch(content_url, …)` is untouched — content URLs arrive
complete from payloads and tool results, never assembled from parts.

**2. The taught contract says absolute.** `describe`'s runtime-contract
section states share paths plainly — absolute, `/`-rooted, with a relative
spelling treated as rooted — and its worked example carries the leading slash
(`suite.files.put("summary.pdf", "/reports/summary.pdf")`). The `suite.py`
`_Files` docstrings say the same in one line.

**Rejected.**

- **Full client-side canonicalization** (`posixpath.normpath` mirroring the
  server's `path.Clean`). Two canonicalizers drift; the server owns cleaning
  (dropbox D24), the client only guarantees rootedness so its own documented
  examples are true against any server version.
- **No client change — rely on the server fix alone.** Leaves the runtime's
  behavior version-skewed: a script written from `describe`'s examples would
  work or 500 depending on which share build the box runs. The shim makes the
  module self-consistent with its own documentation.
- **Raising `ValueError` on relative spellings.** A local error for input
  with exactly one meaning; the module's convention is native exceptions for
  programming errors, and a relative share path is not one — it is an
  accepted spelling of the canonical path.

**Verification.**

- R-ZECX-40UZ — every share-path argument is sent rooted: driven through the
  probe harness against a recording stand-in server, `stat("a/b")`,
  `get("a/b", …)`, `put(…, "a/b")`, `delete("a/b")`, `mkdir("a/b")`, and
  `move("a/b", "c/d")` (both arguments) each arrive with query `path` (or
  `from`/`to`) equal to `/a/b` (`/c/d`); already-absolute spellings pass
  through byte-identical.
- R-ZFKT-HSLO — `list` is the deliberate exception: called with no path it
  sends **no** `path` param (list-everything unchanged); called with
  `"a"` it sends `path=/a`.
- R-ZGSP-VKCD — the taught contract is absolute: `describe`'s output contains
  a `suite.files.put` example whose share-path argument begins with `/`, and
  a statement that share paths are absolute with relative spellings treated
  as rooted; no `describe` example passes a relative share path.
