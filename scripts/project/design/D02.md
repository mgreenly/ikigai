# Decision 2 — Route wiring: `GET /{$}` mounted ungated through `Spec.Handlers`

**Decision.** The landing handler is mounted at the service's **exact root path**
`/` through scripts's existing `appkit.Spec.Handlers` hook — concretely inside
`registerRoutes`, beside the `POST /mcp` mount — using the Go 1.22+ exact-match
pattern `GET /{$}`:

> **Substrate note (D12/D13).** This Decision owns the **route shape** — `GET /{$}`
> mounted ungated through `Handlers`, exact-match, no shadowing. The *handler behind
> it* changed with the chassis conversions: D12 renders the landing through
> `rt.WWW()` (and deletes the service-side `GET /static/` mount — the chassis
> auto-mounts it), and D13 assembles the `/mcp` handler via `appkit/mcp`. The
> `{$}` semantics and the `R-ROUT-*` ids below are unchanged.

```go
// registerRoutes wires scripts' domain on appkit's server. appkit has already
// resolved config, opened, and migrated the shared single-writer DB.
func registerRoutes(rt *appkit.Router) error {
    // ... existing Store/runner/Service construction, crash-recovery sweep ...

    // Landing page — human web UI. Ungated IN-PROCESS: nginx is the gate
    // (auth_request /_session-authn, see D4). `{$}` matches ONLY the exact root
    // path, so it never shadows /mcp, /health, /feed, or the PRM well-known.
    // The chassis site (Spec.WWW, D12) renders the shipped share/www template.
    rt.Handle("GET /{$}", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        _ = rt.WWW().Render(w, "landing.html", struct{ Service, Version string }{rt.Service(), rt.Version()})
    }))

    // No service-side GET /static/ mount — the chassis auto-mounts it (Spec.WWW, D12).
    handler, err := mcp.NewHandler(svc, rt) // appkit/mcp assembly (D13)
    if err != nil {
        return err
    }
    rt.Handle("POST /mcp", rt.RequireIdentity(handler))
    return nil
}
```

**Why ungated in-process.** scripts runs no token logic; nginx is the sole trust
boundary (Conventions). The landing route is mounted **without** an in-process
identity wrapper — exactly as `POST /mcp` delegates its bearer gate to nginx's
`auth_request /_authn`, the landing route delegates its cookie gate to nginx's
`auth_request /_session-authn` (D4). Wrapping it in `rt.RequireIdentity` would be
wrong: that expects the bearer-injected `X-Owner-Email`/`X-Client-Id` headers,
which a browser session request does not necessarily carry the same way, and the
page is meant to be coarse, not per-resource.

**Why `{$}` and not `/`.** In the Go 1.22+ `http.ServeMux` pattern language, a
trailing `/` is a **subtree** pattern that matches every path under it, while
`{$}` matches **only** the exact path. scripts's internal routes are `/`, `/mcp`,
`/health`, `/feed`, and `/.well-known/oauth-protected-resource` (nginx has
already stripped the `/srv/scripts/` mount prefix). Mounting the landing page at
`GET /{$}` guarantees it answers the bare root **and nothing else** — it cannot
shadow `/mcp`, the loopback `/feed`, or any other route. A subtree `/` would
swallow unmatched paths and is explicitly rejected.

**Composition-root growth, not a rewrite.** `cmd/scripts/main.go`'s
`registerRoutes` already builds the script Service (the store, the `python3`-exec
runner, the crash-recovery sweep) and mounts `POST /mcp`; this Decision adds the
landing route to that existing function. That is the sanctioned incremental growth
of the one shared wiring file, not a rewrite of scripts's domain — the Service, the
event-plane consumers (the `Spec.Consumers` table, D11), and the `Producer` outbox
hook are untouched.

**Rejected.**
- **A subtree `GET /`** — swallows every unmatched path (would intercept a typo'd
  `/mcpx`, future routes, etc.) and muddies the "exact root only" contract. `{$}`
  is precise.
- **Wrapping the route in `rt.RequireIdentity`** — double-gates with the wrong
  (bearer) mechanism and breaks the coarse cookie-gated human door; the gate is
  nginx's job (D4).
- **A second `appkit.Spec` field or a new hook for web pages** — unnecessary; the
  existing `Handlers` hook (via `registerRoutes`) mounts arbitrary routes, and the
  landing page is just another route on the same mux.
- **Mounting the page on a sub-path** (e.g. `/landing`) — the product constant is
  that the human reaches it at the mount **root** (`/srv/scripts/`), which is the
  service's `/`.

**Verification.**
- R-ROUT-8U2J — with the landing handler registered on an `http.ServeMux` at
  `GET /{$}` (as the composition root wires it), a `GET /` is dispatched to the
  landing handler and returns the landing page (`200`, name+version body).
- R-ROUT-1V4K — the `{$}` exact-root registration does **not** shadow a sibling
  `POST /mcp` registration on the same mux: a `POST /mcp` still dispatches to the
  mcp handler, not the landing handler.
- R-ROUT-3W6L — a non-root path that is not otherwise registered (e.g. `GET /nope`)
  is **not** captured by the `{$}` landing pattern (it does not return the landing
  page) — proving `{$}` is exact-match, not a subtree.
