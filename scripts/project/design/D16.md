# Decision 16 ‚Äî the bearer tier's identity plumbing is named for scripts, not prompts

**Decision.** `scripts/etc/nginx.conf`'s bearer tier carries copy-paste heritage
from the prompts fragment: its identity variables and its 429 re-emit named
location are named after **prompts** ‚Äî `auth_request_set $prompts_owner` /
`$prompts_client`, `proxy_set_header X-Owner-Email $prompts_owner` /
`X-Client-Id $prompts_client`, `error_page 500 = @prompts_authn_500;`, and
`location @prompts_authn_500`. This Decision renames every one of them to the
scripts-owned equivalents ‚Äî `$scripts_owner`, `$scripts_client`, and
`@scripts_authn_500` ‚Äî leaving behavior identical.

This is **hygiene, not an outage fix**, and the Decision says so precisely:
nginx tolerates a duplicate named location (verified empirically ‚Äî a config with
two `location @prompts_authn_500` blocks passes `nginx -t`), and
`auth_request_set` variables are per-request, set only in the location that
matched, so nothing malfunctions today even with both fragments installed in the
one apex server block. But the fragment claims another service's names inside
that shared block: two services define `@prompts_authn_500`, making the
`error_page 500` jump target ambiguous to a reader (and fragile to nginx
behavior changes), and the `$prompts_*` variables misattribute scripts's
identity plumbing on sight. The rename removes the ambiguity at zero behavioral
cost. No existing scripts test pins the wrong names, so the rename breaks
nothing.

**Rejected.**
- **Leaving it as-is** ‚Äî it works, but a shared server block where two fragments
  define the same named location and pass identity through another service's
  variables is exactly the kind of silent coupling that turns a future prompts
  fragment change into a scripts incident.
- **Folding the rename into the D15 bounce opt-in** ‚Äî D15 is purely additive
  (one line per session-gated location, nothing rewritten, pinned by
  R-49T9-SNXY); mixing a rename into it muddies both proofs. Two
  Decisions, two phases.

**Verification.** (Proven by extending the existing nginx content-assertion test
(`cmd/scripts/main_test.go`) that reads `scripts/etc/nginx.conf` from disk;
nginx is not run by the suite.)
- R-4EOV-BQWQ ‚Äî in `scripts/etc/nginx.conf`, the bearer prefix `location
  /srv/scripts/` sets and forwards identity through scripts-named variables
  (`auth_request_set $scripts_owner` / `$scripts_client`, `proxy_set_header
  X-Owner-Email $scripts_owner` / `X-Client-Id $scripts_client`), its 429
  re-emit is `error_page 500 = @scripts_authn_500;`, and `location
  @scripts_authn_500` is defined.
- R-4FWR-PINF ‚Äî the string `prompts` does not occur anywhere in
  `scripts/etc/nginx.conf` (no prompts-named variable, named location, or
  comment reference survives the rename).
