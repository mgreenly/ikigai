# Decision 10 — Adopt `registry`: resolve scripts' own port and peer addresses by name at startup

**Problem.** scripts hardcodes loopback port literals in three places in its
composition root, plus illustrative literals in a peer client's doc comments:

- Its **own** port, twice — `Port: 3003` in `scriptsSpec()`
  (`cmd/scripts/main.go`) and the `3003` argument to
  `config.Resolve("scripts", "/srv/scripts/", 3003, os.Getenv)` in
  `registerRoutes`.
- Its **peer** feed URLs — the package-level `feedDefaults` map:
  `cron → http://127.0.0.1:3005/feed`, `crm → :3100`, `ledger → :3101`,
  `dropbox → :3200`, `prompts → :3002`.
- Its **dropbox content base** — the `DROPBOX_BASE_URL` fallback
  `"http://127.0.0.1:3200"` in `registerRoutes`, mirrored in
  `internal/script/dropbox.go` doc comments.

Every one of these numbers is a copy of a value the suite writes down in ~five
places per service (`main.go`, `.envrc`, `manifest.env`, each consumer's peer
map, nginx, `bin/start`). A renumber has to be found and edited in lockstep; a
miss is invisible to the compiler and only surfaces at deploy. There is no single
place all of Go can ask "what port is `cron`?" — so scripts' peer map silently
duplicates cron's, ledger's, dropbox's, and prompts' own numbers.

The suite now has that single place: `registry`, a standalone zero-dependency Go
module at the repo root (`module registry`), whose slice-of-structs table is the
one authoritative `name → port` map (registry D2) and whose API (registry D3)
turns a name into a port or a loopback base URL:

- `registry.MustPort(name) int` — port, **panics** on an unknown name (strict
  composition-root form).
- `registry.Port(name) (int, bool)` — checked form.
- `registry.BaseURL(name) string` — `"http://127.0.0.1:<port>"`, **no path**;
  the caller appends `/feed`, `/content`, etc.

The registry table already carries scripts' and its peers' current numbers
verbatim (`scripts 3003`, `cron 3005`, `crm 3100`, `ledger 3101`, `dropbox 3200`,
`prompts 3002`), so adopting it is **behavior-preserving**: no service renumbers
and `etc/manifest.env` stays byte-identical (`PORT=3003`).

**Decision.** scripts references itself and its peers **by name** and asks
`registry` for the address, resolving **once at the composition root** (startup),
never per request. The registry supplies the **default**; every existing env-var
override is kept as the override layer in front of it.

1. **Own port via `MustPort`.** In `scriptsSpec()`:
   ```go
   Port: registry.MustPort("scripts"),
   ```
   and in `registerRoutes`:
   ```go
   cfg, err := config.Resolve("scripts", "/srv/scripts/", registry.MustPort("scripts"), os.Getenv)
   ```
   `MustPort` (panics on an unknown name) is correct here: `"scripts"` is a
   compile-time constant that is in the table, so a miss would be a programming
   error, surfaced loudly at boot — never a silent zero port. The resolved value
   is `3003`, so `manifest.Emit` still produces the identical `PORT=3003` line and
   the committed-manifest byte-equality test (`R-8IAN-FB87`) still passes.

2. **Peer feed defaults via `BaseURL`.** *(Superseded by D11.)* D10 replaced the
   `feedDefaults` map with a pure `feedDefault(source)` helper used by `runConsumer`.
   **D11 deletes both** `feedDefault` and `runConsumer`: with the consumer loops
   moved to the chassis `Spec.Consumers` table, peer feed-URL default resolution
   (`SCRIPTS_<SRC>_FEED_URL`, default `registry.BaseURL(src)+"/feed"`) becomes
   **chassis-owned** — the identical resolution, now pinned by appkit's own D10 ids
   (`R-464U-T3T1` default, `R-47CR-6VJQ` override) rather than scripts' retired
   `R-RGST-PEER`. The original D10 wording is retained below for history; the helper
   it describes no longer exists after D11.
   ```go
   // feedDefault is a source's loopback dev fallback: the registry-owned address
   // plus the /feed route. The event plane bypasses nginx, so this is a direct
   // 127.0.0.1 address; production overrides it via SCRIPTS_<SRC>_FEED_URL.
   func feedDefault(source string) string {
       return registry.BaseURL(source) + "/feed"
   }
   ```
   ```go
   feedURL := config.EnvOr(os.Getenv, feedURLEnv(source), feedDefault(source))
   ```
   `runConsumer` runs **once per upstream at startup** (one `Workers` entry per
   source), so `feedDefault` is evaluated at boot and its result is captured in
   the `consumer.Config` the loop reuses — not re-looked-up per event. For all
   five sources the result equals the prior literal (`BaseURL("cron")+"/feed"` ==
   `http://127.0.0.1:3005/feed`, etc.). The `SCRIPTS_<SRC>_FEED_URL` override is
   unchanged.

3. **Dropbox base via `BaseURL`.** In `registerRoutes`:
   ```go
   dropboxBase := config.EnvOr(os.Getenv, "DROPBOX_BASE_URL", registry.BaseURL("dropbox"))
   svc.Fetcher = script.NewHTTPFetcher(dropboxBase)
   ```
   Resolved once at the composition root and field-injected onto the service, as
   today. `registry.BaseURL("dropbox")` == `http://127.0.0.1:3200`, so the default
   is unchanged; the `DROPBOX_BASE_URL` override still wins when set. The
   `dropbox.go` doc comments that quote the literal (`e.g. http://127.0.0.1:3200`,
   `default http://127.0.0.1:3200`) are reworded to name `registry.BaseURL("dropbox")`
   so no bare loopback literal survives anywhere in non-test source.

4. **Module wiring (inside `scripts/`).** Add `require registry v0.0.0` and a
   committed `replace registry => ../registry` to `scripts/go.mod`, exactly like
   the existing `appkit`/`eventplane` replace-siblings. This is the only file
   outside `cmd/scripts` and the tests that this Decision touches.

**Startup-resolve seam (why once, not per-request).** The registry table is
compile-time data and the host is a constant, so a resolved address cannot change
while the binary runs; if the table ever changes we rebuild the binary anyway.
Every resolution therefore happens at the composition root — `scriptsSpec()` for
the port, `registerRoutes` for the dropbox base, and each `runConsumer` (a
startup `Workers` call) for a peer feed — and the concrete value is stored (in the
`Spec`, on `svc.Fetcher`, in the `consumer.Config`) and reused. No lookup runs on
a request or per event.

**Override layer preserved.** registry supplies the **default** that sits *behind*
each existing env override, occupying exactly the slot the literal held:
`SCRIPTS_<SRC>_FEED_URL` still overrides a peer feed, `DROPBOX_BASE_URL` still
overrides the dropbox base, and appkit's own `SCRIPTS_PORT` still overrides the
port. Adoption changes only what the override falls back to — a named registry
lookup instead of a hardcoded number.

**No bare loopback literal remains.** After this Decision the `registry` table is
the **only** place a scripts service port is written. A guardrail test scans
scripts' non-test Go source and fails on any surviving loopback port literal (a
`127.0.0.1:3xxx` URL or a bare `3003`), so a future hand-typed port cannot creep
back in uncaught. This is the mechanical enforcement of the adoption, in the
suite's "enforce with systems, not willpower" spirit.

**Boundary.** D10 owns only the composition-root address resolution in
`cmd/scripts/main.go`, the two `dropbox.go` doc comments, the `scripts/go.mod`
require/replace, and the tests that pin these. It does **not** change any
observable behavior (ports, URLs, and `manifest.env` are identical), any schema,
the landing-page series (D1–D8), the runs-dir Decision (D9), the MCP/feed/health
surfaces, or the runtime override semantics. It is orthogonal to D1–D9 and touches
no `internal/web` landing code except the one `nginx_test.go` alignment noted
below.

**External preconditions (owned at the repo root — NOT built by this Decision).**
Two pieces of registry wiring live outside `scripts/` and are assumed satisfied:

- The `registry` module itself (`registry/go.mod` + the package) already exists at
  the repo root.
- The repo-root `go.work` needs a `use ./registry` entry so local dev resolves the
  module (production forces `GOWORK=off` and relies on the committed `replace`).
  `go.work` is repo-root-owned; a plan phase must not edit it. If a local `go
  build` cannot find `registry`, that entry is missing and is an operator/root
  task, not a scripts-scoped change.

**`etc/nginx.conf` stays as-is (deploy artifact, not Go).** The static nginx
fragment still contains `proxy_pass http://127.0.0.1:3003/;`. `registry` is a
Go-only compile-time table and cannot be embedded in a raw nginx config without a
generation step, which is nginx/deploy-layer work out of this scope. What D10
*does* do is align the `nginx_test.go` content assertions to the registry: the
test derives its expected proxy_pass port from `registry.MustPort("scripts")`
instead of a literal `3003`, so if scripts is ever renumbered in the table the
nginx fragment's drift is caught by scripts' own test suite. Regenerating or
hand-editing the shipped `nginx.conf` on a renumber remains repo-root/deploy work.

**Rejected.**

- **`registry.Port("scripts")` (checked form) for the own port.** The name is a
  compile-time constant guaranteed to be in the table; a miss is a bug, not a
  runtime condition. `MustPort` fails loudly at boot, matching the suite's "fail
  loudly / unknown state is unacceptable" principle. The checked form would force
  dead `ok`-handling for a case that cannot legitimately occur. Rejected.
- **Keep the `feedDefaults` map, just build it from `registry.BaseURL`.** A
  package-level map built in an `init` re-introduces a second structure to keep in
  sync with `sources` and is not directly unit-testable per source. A pure
  `feedDefault(source)` helper is smaller, has no init-order surface, and is
  trivially testable against the five known values. Rejected in favor of the
  helper.
- **Resolve addresses lazily / per request.** Pointless work on a hot path for
  data that is fixed at compile time, and it scatters registry calls across the
  request surface instead of quarantining them at the composition root. Rejected —
  resolve once at startup, store, reuse.
- **Also rewrite `etc/nginx.conf` to erase its `3003`.** The nginx fragment is a
  shipped deploy artifact owned by the nginx/root layer, not Go source registry
  can reach; rewriting it (or adding a generation step) is out of scope. Aligning
  the *test* to registry gives the anti-drift guardrail without crossing the
  boundary. Rejected for this scope.
- **Drop the env overrides now that registry supplies defaults.** The overrides
  are the production seam (the deploy wrapper composes real `SCRIPTS_<SRC>_FEED_URL`
  / `DROPBOX_BASE_URL` values); registry only replaces the *literal fallback*, not
  the override. Removing them would change deploy behavior. Rejected.

**Verification** (behavioral where the value is observable; source/content
assertions where the seam is composition-root wiring, matching the existing
`TestCompositionRootAdoptsNewScriptsLayout` pattern).

- R-RGST-SELF — **scripts resolves its own loopback port by name via
  `registry.MustPort("scripts")`, and the value is `3003`.** A behavioral unit
  test: `spec := scriptsSpec()` asserts `spec.Port == registry.MustPort("scripts")`
  **and** `spec.Port == 3003`; plus a source assertion that
  `cmd/scripts/main.go` passes `registry.MustPort("scripts")` (not a literal
  `3003`) as the port argument to `config.Resolve`. Falsifiable: a `Port: 3003`
  literal, or a `config.Resolve(..., 3003, ...)` literal, fails it. The
  byte-identical `manifest.env` (`R-8IAN-FB87`) independently confirms the value
  did not move.
- R-RGST-PEER — *retired by D11.* The peer feed defaults it pinned (via scripts'
  local `feedDefault`) are chassis-owned once the consumer loops move to
  `Spec.Consumers`; the behavior (`registry.BaseURL(src)+"/feed"` default per
  upstream) is pinned by appkit's `R-464U-T3T1`/`R-47CR-6VJQ`. The
  `TestFeedDefaultUsesRegistryBaseURLs` test is deleted with the `feedDefault`
  function it covered.
- R-RGST-DBOX — **the dropbox content base default resolves via
  `registry.BaseURL("dropbox")` == `http://127.0.0.1:3200`, behind the
  `DROPBOX_BASE_URL` override.** A behavioral assertion `registry.BaseURL("dropbox")
  == "http://127.0.0.1:3200"` plus a source assertion that `cmd/scripts/main.go`
  uses `registry.BaseURL("dropbox")` as the `DROPBOX_BASE_URL` `EnvOr` fallback
  (and no `"http://127.0.0.1:3200"` literal remains in `main.go`/`dropbox.go`).
  Falsifiable: a literal fallback, or a different resolved value, fails it.
- R-RGST-NLIT — **no bare loopback port literal remains in scripts' non-test Go
  source.** A guardrail test walks every non-`_test.go` `.go` file under the
  `scripts` module and asserts none matches `127\.0\.0\.1:3\d\d\d` (any loopback
  service URL) and none contains a standalone `3003` token (scripts' own port) —
  the registry table is the only place a port is written. Falsifiable:
  re-introducing `Port: 3003` or a `http://127.0.0.1:3005/feed` literal in source
  fails it. *(The test itself and other `_test.go` files are excluded; the shipped
  `etc/nginx.conf` is not Go source and is out of this walk.)*
- R-RGST-GMOD — **`scripts/go.mod` requires `registry` with a committed
  `replace registry => ../registry`, and the module builds green with `registry`
  linked.** A content assertion that `scripts/go.mod` contains both a `registry`
  require and `replace registry => ../registry`; the suite building/vetting green
  (an unresolved `registry` import cannot compile) is the behavioral half.
  Falsifiable: a missing require or replace, or a build that fails to resolve
  `registry`, fails it.
