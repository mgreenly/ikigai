# Decision 13 — MCP surface over `appkit/mcp`: `internal/mcp` becomes the tool table

**Problem.** scripts' `internal/mcp` carries the copy-pasted JSON-RPC transport
(`ServeHTTP`, the `initialize` / `notifications/initialized` / `tools/list` /
`tools/call` dispatch, `writeJSONRPCError`/`writeJSONRPCResult`, the result
envelopes, the local `Identity` type) plus its own `health` tool implementation —
the same near-identical plumbing every service duplicated. The chassis now owns
all of it (appkit D8 transport, D9 standard tools, consumed here as a fixed
external contract). What is genuinely scripts' is its **sixteen domain tools**:
`describe`, `create`, `import`, `list`, `get`, `update`, `delete`, `set_trigger`,
`clear_trigger`, `run`, `run_list`, `run_get`, `run_output`, `run_cancel`,
`run_fs_list`, `run_fs_read` — their descriptors, schemas, validation, and the
`script.Service` dispatch.

Two facts make scripts' conversion diverge from notify's, and this Decision states
both honestly rather than pretending the surface is byte-identical:

- **scripts today exposes no `reflection` tool.** Its hand-rolled surface is 17
  tools — 16 domain tools **plus a local `health`**, and no `reflection`
  (`TestToolsListReturns17` pins exactly that). The chassis registers **both**
  `health` and `reflection` unconditionally (appkit D9 reserves the names and
  rejected an override seam). So the conversion **adds `reflection`**: the wire
  surface goes 17 → **18** tools. This is not a regression — scripts is a producer
  *and* a five-upstream consumer with a populated `Events`/`Subscriptions`, exactly
  the service `reflection` is meant to self-describe (crm and notify already expose
  it); the omission was the hand-rolled surface's gap, and adopting the chassis
  closes it. It is called out here because it changes the tool count, which a naive
  "byte-for-byte" reading would miss.
- **scripts' `health` details live in the wrong place.** The static runtime
  contract (`python_version`, `bash_version`, `network`, `packages`) is currently
  hardcoded in `internal/mcp`'s `runtimeContract()` and injected only into the
  **MCP** `health` tool; scripts sets no `Spec.Health`, so the **HTTP `/health`**
  route returns empty `details`. The chassis `health` tool (D9) sources `details`
  solely from `Options.Health` == `rt.Health()` == `Spec.Health`, feeding **both**
  the MCP tool and the HTTP route from one reporter. The fix is to move the runtime
  contract into `Spec.Health`.

**Decision.** `internal/mcp` is reshaped from a transport into a **declaration**,
and the runtime contract moves to the composition-root health reporter:

```go
// internal/mcp — what remains.
const Instructions = /* scripts' initialize instructions string, byte-identical */

// Tools declares scripts' sixteen DOMAIN tools over the script.Service (descriptors,
// schemas, and dispatch bodies unchanged in wire content). health and reflection are
// NOT declared — the chassis registers them (declaring either fails New's reserved-
// name check). Each Tool.Handler wraps the existing dispatch body; server.Identity's
// OwnerEmail is the owner the domain calls already thread.
func Tools(svc *script.Service) []mcp.Tool

// NewHandler assembles the /mcp http.Handler at the composition root: appkit/mcp.New
// over Instructions + Tools(svc) + the Router-threaded Service/Version/Health/Events/
// Publishes/Subscriptions.
func NewHandler(svc *script.Service, rt *appkit.Router) (http.Handler, error)
```

```go
// cmd/scripts — the runtime contract becomes the Spec health reporter, feeding BOTH
// the HTTP /health route and the chassis MCP health tool (appkit D9).
Health: func(ctx context.Context) (map[string]any, error) {
    return map[string]any{
        "python_version": ">=3.11",
        "bash_version":   ">=5.0",
        "network":        true,
        "packages":       "stdlib",
    }, nil
},
```

- **Deleted from `internal/mcp`:** the JSON-RPC plumbing (`jsonRPCRequest`,
  `toolCallParams`, `ServeHTTP`, `handleToolCall`, `writeJSONRPCResult` /
  `writeJSONRPCError` / `idOrNull`), the local `Identity` type (replaced by
  `appkit/server.Identity`), the local `toolResultText` / `toolResultErr` /
  `toolResultJSON` envelope helpers (replaced by the chassis
  `mcp.TextResult`/`JSONResult`/`ErrorResult`), the `toolDescriptors()` /
  `desc`/`obj`/`typ` map-descriptor form (folded into the `[]mcp.Tool` literal —
  the same names/descriptions/schemas), the local `toolHealth` and its
  `runtimeContract()` (health is chassis-owned; the contract moves to `Spec.Health`),
  and the `handleToolCall` → `dispatchTool` switch's transport wrapper (each case's
  body becomes a `Tool.Handler`; the `paramError` → `-32602` mapping is preserved via
  the handler returning that error, which the chassis renders as the JSON-RPC error).
- **Kept:** the sixteen domain tool descriptors and schemas byte-for-byte (names,
  descriptions, `inputSchema` including `configSchema`), the `Instructions` string
  (byte-identical to the current `initialize` instructions), the `describe` tool and
  its `describeText`, and every dispatch body (`svc.Create`/`Import`/`List`/`Get`/
  `Update`/`Delete`/`SetTrigger`/`ClearTrigger`/`Run`/`RunList`/`RunGet`/`RunOutput`/
  `RunCancel`/`RunFsList`/`RunFsRead` with the same argument decoding, `parseArgs`
  behavior, and error → `isError` mapping). `describe` stays a **domain** tool (it
  is scripts', not a chassis standard tool) — the surface keeps it.
- **Wire surface after conversion:** 18 tools — the 16 domain tools (unchanged
  bytes) **plus** chassis `health` and `reflection`. Versus 17 before: `reflection`
  is **added** (chassis-owned) and `health`'s **descriptor description** changes to
  the chassis generic text ("…for this service…"; the current scripts-specific
  "…for the scripts service…details is the static runtime contract…" wording is
  retired — the chassis owns the descriptor). The `health` **result** is unchanged:
  `details` still carries the runtime contract (now via `Spec.Health`), and
  `owner_email`/`client_id` are still present (the chassis adds them). The HTTP
  `/health` route gains those `details` (was `{}`) — a benign superset that removes
  the pre-existing MCP-vs-HTTP divergence.
- **Tests:** the domain behavioral tests (`tools_test.go`) keep their assertions and
  drive the **assembled** appkit/mcp handler, built the crm/notify way — `appkitdb`
  for the DB (D14), `server.New` to capture a Router (with `Health` set to the
  runtime-contract reporter and `Events: script.Events`), then `mcp.NewHandler(svc,
  rt)`. `TestToolsListReturns17` is **replaced** by the R-91IM-JYPA 18-tool
  partition test. `TestHealthDetailsContract` and `TestDescribeNonEmpty` /
  `TestCreateGetList` / `TestRunSpawns` / `TestSetTriggerValidAndInvalid` /
  `TestRunOutputAndFs` / `TestImportDispatch` / `TestErrorMapping` keep their
  assertions and only swap the constructor.

**Rejected.**
- **Keeping the local transport** — the status quo; a protocol fix would still be a
  many-service edit.
- **Suppressing `reflection` to preserve the 17-count** — impossible by design
  (appkit D9 reserves the names and rejected an opt-out) and wrong: a producer +
  consumer should self-describe its event graph. The count change is documented, not
  hidden.
- **Declaring `health`/`reflection` in scripts' table** — fails `New`'s reserved-name
  check; their inputs (Envelope, Registry, Subscriptions) are chassis-held already.
- **Leaving the runtime contract in `internal/mcp`** — the chassis health tool reads
  `details` only from `Spec.Health`; a copy in the mcp package would be dead for the
  MCP tool and still absent from HTTP `/health`. One reporter, both surfaces.
- **Dissolving `internal/mcp` into `cmd/scripts`** — unlike the post-D12 web remnant,
  the sixteen tools' schemas, argument decoding, and error mapping with their test
  harness are a real package; it stays, reshaped to the declaration.

**Verification** (through the composition-root seam — the appkit/mcp handler built
by `NewHandler` over a real `script.Service` on an `appkitdb` temp DB).
- R-91IM-JYPA — `tools/list` through the assembled handler returns **exactly
  eighteen** tools: the sixteen domain tools (`describe`, `create`, `import`, `list`,
  `get`, `update`, `delete`, `set_trigger`, `clear_trigger`, `run`, `run_list`,
  `run_get`, `run_output`, `run_cancel`, `run_fs_list`, `run_fs_read`) declared by
  scripts **plus** `health` and `reflection` supplied by the chassis — proving the
  table/chassis partition (a table that still declares its own `health` fails `New`'s
  reserved-name check; a missing standard tool fails the count). This replaces
  `TestToolsListReturns17`.
- R-92QI-XQFZ — the runtime contract is served from the single `Spec.Health`
  reporter through **both** surfaces: the MCP `health` tool result's `details`
  carries `python_version`/`bash_version`/`network`/`packages` (plus `owner_email`/
  `client_id` on the envelope), **and** the HTTP `GET /health` route's `details`
  carries the same four keys — proving the contract moved to `Spec.Health` and no
  longer lives only in the MCP package.
