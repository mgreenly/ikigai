# Decision 17 — Event-routing conformance: triggers become canonical filter strings (trigger surface + consumer)

> **External ordering (operator-sequenced, not built by any scripts phase).**
> This Decision consumes the **revised eventplane API** — the kind/subject
> envelope, `routing.Key`/`routing.Match`/`routing.ValidKind`/
> `routing.ValidSubject`, `outbox.Family`/`Registry.CouldMatch`, and
> `consumer.Event{Kind, Subject}` + `Event.Key()` — specified in
> `eventplane/project/design/` (its D1–D4) and realized by **eventplane plan
> phases 01–04, which must be BUILT before this Decision's phase runs**. It
> equally assumes appkit (the chassis consumer engine, `Spec.Events
> outbox.Registry`, the reflection tool) compiles against that revision —
> appkit's own conformance is appkit's spec, out of scope here. Until both
> land, this Decision's phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): events are addressed by a canonical routing
key `source:kind<subject>` matched with one shared glob matcher. scripts has
the same trigger machinery prompts has, and everything it does today is built
on the old model (ground truth): a trigger is a
`(script_id, source, event_filter)` row (`script_triggers`, migration
`002_scripts.sql`, PK `(script_id, source, event_filter)`); `validateTrigger`
(`internal/script/trigger.go`) checks the pair against `knownProducers` — a
static per-source list of concrete `type` strings over five upstreams — with a
hand-carved **cron special case** (any filter matching `cron.*` accepted
blind); `Store.ScriptsForEvent(source, eventType)` matches stored filters
against the flat type with a local `globMatch`/`path.Match`;
`consume.Handler` reads `ev.Type` and skips on `ev.Type == "" || ev.ID == ""`;
`consume.Subscriptions` declares one broad `Filter: "*"` in-edge per source;
the run row persists `trigger_source`/`trigger_type`/`trigger_event_id`. A
script that cares about `/bills/**` must today fire on **every** dropbox file
event and burn a run discovering it wasn't for it — the failure the revision
exists to fix. Hard cutover, no compatibility period, inside the no-live-data
migration window (trigger rows are explicitly wipeable per the routing doc).

**Decision.** A trigger becomes **`(script_id, filter)`**, where `filter` is a
single canonical-key glob matched by `eventplane/routing.Match` against an
event's `routing.Key(source, kind, subject)` — e.g.
`dropbox:create/bills/**/*.pdf`, `cron:tick/bill-sweep`,
`prompts:run.succeeded/collect-bills`. The `(source, event_filter)` pair, the
static per-source **type** vocabulary, and the cron special case are all
replaced. This mirrors prompts' D23 exactly except where scripts' code
genuinely differs; each difference is named below.

1. **Filter well-formedness: the source segment is literal.** A valid filter
   contains a `:`, and the segment before the first `:` must be a **non-empty
   literal source name** — no glob metacharacters (`*`, `?`, `[`) in the
   source segment. Globs in the source segment are **rejected**: a trigger's
   binding is consumed per-upstream (one `Spec.Consumers` entry per source,
   one feed, one cursor), so a multi-source filter could never be honored by
   the delivery machinery and would silently under-fire. The literal source
   must be one of the **five** consumed sources
   (`cron`, `crm`, `ledger`, `dropbox`, `prompts`); anything else is rejected
   naming the known set (today's unknown-source behavior, preserved).
   *Divergence from prompts:* prompts validates six sources including its own
   feed (self-chaining); scripts consumes five and deliberately excludes
   itself — self-chaining stays out, exactly as D11 kept it out.

2. **Family validation: a static kinds table + `Registry.CouldMatch`
   semantics.** scripts holds per-source family knowledge the way it holds
   `knownProducers` today — a **static table of kinds**, subjects open:

   ```go
   // internal/script/trigger.go — replaces knownProducers. Kinds follow the
   // routing doc's producer key map; each producer's own spec loop owns its
   // final choice, and this table tracks it exactly as the old type table did.
   var knownFamilies = map[string][]string{
       "cron":    {"tick"},
       "crm":     {"contact.created", "contact.updated", "contact.tagged", "contact.untagged"},
       "ledger":  {"recorded"},
       "dropbox": {"create", "modify", "delete"},
       "prompts": {"run.succeeded", "run.failed"},
       // NOTE deliberately no "scripts" entry: self-chaining (a sixth consumer
       // loop on the local /feed) remains a future item, per D11.
   }
   ```

   `validateTrigger(filter)` (after well-formedness) builds an
   `outbox.Registry` of `Family{Kind: k}` values from the source's kinds and
   asks `Registry.CouldMatch(source, filter)` — accepted iff the filter could
   match some family's kind with **some** subject (subject left open, the
   eventplane D3 contract). This deletes the cron special case: cron is an
   ordinary producer whose kind is `tick` and whose subjects are dynamic —
   `cron:tick/anything` validates, exactly like `dropbox:create/anything`.
   A malformed glob surfaces `routing.Match`'s error as `ErrValidation`.
   **Live reflection queries are rejected** (the alternative): validating a
   trigger by calling each peer's `reflection` at `set_trigger` time makes a
   write verb depend on every upstream being up, adds an HTTP client scripts'
   domain layer doesn't have, and buys only earlier detection of the same
   drift the static table carries today — the architecture keeps the static
   copy and its known, accepted drift cost.

3. **Storage: one new timestamped migration** minted with
   `bin/create-migration scripts trigger_filters` (never a hand-picked
   number). Inside the migration window it **drops and recreates**
   `script_triggers` — columns `(script_id, source, filter, created_at)`,
   `PRIMARY KEY (script_id, filter)`, index on `source`, keeping the
   `REFERENCES scripts(id) ON DELETE CASCADE` FK (the tombstone-delete model
   is untouched) — carrying no rows across (old `(source, event_filter)`
   pairs are not mechanically convertible; trigger rows are wipeable per the
   routing doc). `source` is **derived** from the filter's literal source
   segment at `SetTrigger` time and stored so `ScriptsForEvent` keeps its
   per-source indexed lookup. The same migration reshapes the **runs**
   trigger context: `trigger_type` becomes `trigger_kind` + `trigger_subject`
   (rebuild or ALTER — builder's choice; run rows are migration-window
   state). The frozen `002_scripts.sql` is **not touched**.

4. **Store + service surface** (`trigger.go`, `store.go`, `service.go`,
   `model.go`): `Trigger{ScriptID, Source, Filter, CreatedAt}` (Source
   derived, Filter the canonical glob); `SetTrigger(ctx, owner, scriptID,
   filter)` validates then upserts on `(script_id, filter)`;
   `ClearTrigger(ctx, owner, scriptID, filter)` deletes that binding.
   Owner-scope semantics are **preserved exactly**: a missing/foreign script
   is `ErrNotFound` (the `ownsScript` gate), and — *divergence from prompts,
   whose ClearTrigger returns ErrNotFound for an unheld filter* — clearing a
   filter the script never held remains a silent no-op success, scripts'
   current `DELETE`-without-`requireOne` behavior. `ScriptsForEvent(ctx,
   source, key string)` selects the source's rows and matches each stored
   filter against the delivered event's **canonical key** with
   `routing.Match` (a malformed stored filter matches nothing — defense in
   depth; writes are validated). The local `globMatch`/`path.Match` helper is
   deleted — the shared matcher is the only matcher.
   `Run{TriggerKind, TriggerSubject}` replace `TriggerType` (JSON
   `trigger_kind`/`trigger_subject` — the `run_get`/`run_list` wire shape
   follows); `RunForEvent(ctx, scriptID, source, kind, subject, eventID
   string, payload []byte)` and `newRun` thread both; `FinishRunInput`
   gains `TriggerKind`/`TriggerSubject` (no `TriggerType`) and the runner's
   `finish` closure threads them (the payload shape is D18's).
   `TriggerSources()` keeps returning the static source set, now the
   `knownFamilies` keys (five).

5. **Consumer side** (`consume.go`): `consumer.Event` now carries
   `Kind`/`Subject` and `Key()` (eventplane D4). The poison check becomes
   `ev.Kind == "" || ev.ID == ""` → `ErrSkip`; the lookup becomes
   `lookup(ctx, source, ev.Key())` (`LookupFunc` takes the key); the fire
   becomes `fire(ctx, scriptID, source, ev.Kind, ev.Subject, ev.ID,
   ev.Payload)` (`FireFunc` follows). `Subscriptions` stays one broad
   in-edge per source, but the filter string becomes **`"**"`** — under the
   doublestar dialect `*` never crosses `/`, so the honest "everything from
   this upstream" glob is `**` (the old `"*"` would misdescribe the in-edge
   for any subject-bearing key). Fire-and-forget semantics (never stall,
   lookup failure logs + advances, matched-zero is success, unbounded
   goroutine fan-out detached on `context.Background()`) are unchanged.

6. **The run input + MCP surface.** *Divergence from prompts:* scripts pins
   **no envelope file** — a run receives only the event's raw `payload`
   bytes on stdin and `$EVENT_JSON` (`runner.execute`; `"{}"` for a manual
   run), and that contract is **unchanged**: the envelope's kind/subject are
   routing fields the script never sees, so there is no `event.json`
   truth-up here. `set_trigger` becomes `(script_id, filter)`,
   `clear_trigger` `(script_id, filter)`; tool descriptions and the
   `describe` text (`internal/mcp/describe.go` TRIGGERS section) state the
   canonical-key contract (`source:kind<subject>` examples, the doublestar
   dialect) in place of the `(source, event_filter)` prose. The error
   mapping is preserved: an invalid filter or unknown source is a domain
   error → `isError` content envelope (D13's mapping), never a JSON-RPC
   error. scripts' `create` has no inline triggers array (unlike prompts),
   so no other tool schema changes.

**Rejected.**
- **Keeping `(source, filter)` as two arguments with the filter matching
  `kind<subject>`** — the canonical key already carries the source, and the
  routing doc's principle is one string, one matcher; two fields re-create the
  seam the revision deletes.
- **Glob in the source segment (`*:create/**`)** — subscriptions are
  per-upstream; a multi-source trigger would need delivery machinery that
  doesn't exist and would silently miss sources added later. A user who wants
  two sources sets two triggers.
- **Adding a `"scripts"` family for self-chaining while we're in here** — a
  new upstream is a behavior change with its own consumer loop and
  verification (D11 rejected the same scope creep); conformance changes shape,
  not the consumed-source set.
- **Live reflection validation** — rejected in §2 above (availability
  coupling, an HTTP dependency in the domain layer, same drift exposure).
- **No validation beyond well-formedness** — an unsatisfiable trigger
  (`dropbox:nosuchkind/**`) would sit silent forever; today's
  reject-unsatisfiable posture is kept, now via the eventplane-owned
  existential check instead of a local type-list scan.
- **Migrating old trigger rows into filters** — `(source, event_filter)` with
  old type names cannot be mechanically rewritten into kind/subject globs
  (the old filters matched types, not subjects); the migration window
  sanctions the wipe, and hand-set triggers are re-set in seconds.
- **Adopting prompts' ErrNotFound-on-unheld-filter ClearTrigger** — a
  semantics change smuggled into a shape conversion; scripts' silent-no-op
  clear is today's contract and stays.
- **Keeping `path.Match` for stored filters** — the one-matcher principle;
  `path.Match` cannot express `**` and its dialect would diverge from every
  other selector in the suite.

**Verification** (real substrates throughout: migration claims apply scripts'
full embedded migration set (`db.FS`) through the appkit runner to a fresh real
SQLite database; matching claims run the real `eventplane/routing` matcher;
store claims run real SQLite; the consume claim drives `consume.Handler` with
recording fire/lookup functions the way the existing `consume_test.go`
substrate does; the MCP claim drives the assembled `appkit/mcp` handler over a
real `script.Service`, the D13 harness).

- R-7TR5-QSY4 — applying the full embedded migration set to a fresh real
  SQLite database yields a `script_triggers` table with columns exactly
  `script_id`, `source`, `filter`, `created_at` (no `event_filter` column),
  PK `(script_id, filter)`, and a `runs` table with `trigger_kind` and
  `trigger_subject` columns and **no** `trigger_type` column, while
  `002_scripts.sql` remains byte-identical to its committed (frozen) body.
- R-7UZ2-4KOT — well-formedness discriminates: `create/bills/**` (no `:`),
  `:create/**` (empty source), `*:create/**` and `drop*:create/**` (glob in
  the source segment), and `github:push/**` (unknown source) are each rejected
  with `ErrValidation` (the unknown-source error naming the five known
  sources — and **not** naming `scripts`); `dropbox:create/bills/**` passes
  well-formedness.
- R-7W6Y-ICFI — family validation via the real eventplane check:
  `dropbox:create/bills/**/*.pdf` and `dropbox:*` are accepted;
  `dropbox:nosuchkind/**` is rejected; `cron:tick/some-schedule-nobody-declared`
  is accepted (subject open — the cron special case is gone, no `cron.*`
  carve-out remains in source); a malformed glob (`dropbox:create/[`) is
  rejected with `ErrValidation`, never silently accepted.
- R-7XEU-W467 — the reshaped store round-trips: `SetTrigger` of
  `dropbox:create/bills/**` on the owner's script stores derived
  `source = "dropbox"` (read back by SQL); a repeat upserts (no duplicate
  row); `ClearTrigger(owner, scriptID, filter)` removes exactly that binding;
  clearing a filter the script never held is a **no-op success** (scripts'
  preserved semantics); a missing/foreign script yields `ErrNotFound` on both
  verbs.
- R-7YMR-9VWW — `ScriptsForEvent(ctx, "dropbox", key)` over real SQLite with
  the real matcher: with triggers `dropbox:create/bills/**/*.pdf` (script A)
  and `dropbox:**` (script B), the key `dropbox:create/bills/aws/1.pdf`
  returns exactly {A, B} and `dropbox:create/notes.txt` returns exactly {B};
  a trigger `dropbox:create/bills/*.pdf` does **not** match
  `dropbox:create/bills/aws/1.pdf` (`*` never crosses `/` — the shared
  dialect, not `path.Match`).
- R-7ZUN-NNNL — `consume.Handler` fed a `consumer.Event` with `Kind`/`Subject`
  set looks up by the event's canonical `Key()` and fires with that kind and
  subject (observed through recording fire/lookup functions); an event with
  empty `Kind` returns `consumer.ErrSkip`; and `consume.Subscriptions` yields
  one subscription per source with `Filter` exactly `"**"`.
- R-812K-1FEA — through the assembled MCP handler over a real
  `script.Service`: `set_trigger` accepts `(script_id, filter)` and returns
  the binding (derived source included), `clear_trigger` accepts
  `(script_id, filter)`, an invalid filter or unknown source renders as an
  `isError` content envelope (not a JSON-RPC error), the two tools' schemas
  advertise `filter` and no `source`/`event_filter` argument survives on the
  wire, and the `describe` text states the canonical-key contract (contains a
  `source:kind` example and no `event_filter`).
