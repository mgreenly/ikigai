# Decision 6 — Conform the landing page to the cron canonical template

**Decision.** scripts's landing page is brought to **visual and structural parity
with the cron landing page**, which is the suite's canonical reference — the one
page that holds up the Carbon style guidelines. cron's page is a centered `main`
with an eyebrow label, a display `<h1>` showing the service name, a one-line lead
paragraph, and a bordered three-cell **Service / Version / API** definition list,
over its own embedded `tokens.css`. The v1 page (D1–D3) diverged — its own card
markup and its own token values — so this Decision **replaces that markup and
those tokens wholesale with cron's**, keeping only the data that must reflect
**this** service.

**Canonical source of truth.** Two files, as committed on `main`:

- `cron/internal/web/landing.html` — the exact page markup + inline `<style>`.
- `cron/internal/web/static/tokens.css` — the Carbon design tokens.

The conform target is **defined by those two files**. There is no frozen second
copy to keep in sync: if cron's canonical page ever changes, this page re-derives
from it by the same diff.

> **Substrate note (D12).** The landing file moved from
> `scripts/internal/web/landing.html` to `scripts/share/www/landing.html` (D12's
> de-embed). This Decision's conform target and its structural done-bar are
> rewritten to the `share/www` path; the markup contract (identical to cron's page
> bar three substitutions) is unchanged. The canonical source is still cron's
> committed page on `main`.

**What changes (markup + tokens only — no Go logic, no handler signature, no
nginx):**

- **`scripts/share/www/landing.html`** becomes **identical to
  `cron/internal/web/landing.html` except exactly three substitutions.** Every
  other line — the inline `<style>`, the `<h1>{{.Service}}</h1>`, the
  `Service / Version / API` `<dl>` whose API value is `POST /mcp`, and the
  `static/tokens.css` link — is cron's, byte-for-byte:

  | line | cron (canonical) | scripts |
  |---|---|---|
  | `<title>` | `{{.Service}} · cron` | `{{.Service}} · scripts` |
  | `.eyebrow` text | `Scheduled event emitter` | `Script runner` |
  | lead `<p>` | `Cron keeps named schedules in SQLite and emits typed event-plane messages at minute boundaries.` | `Scripts runs deterministic Python scripts wired to suite events and publishes completion events back to the event plane.` |

- **`scripts/share/www/static/tokens.css`** becomes **byte-identical to
  `cron/internal/web/static/tokens.css`**. The `static/fonts/` woff2 set is already
  the shared set and is unchanged.

- The chassis landing render (`rt.WWW().Render`, D12), the `GET /{$}` route wiring
  in `cmd/scripts/main.go`, and the nginx fragment are **untouched** — this Decision
  changes only rendered markup and the token values. The service name and version
  stay dynamic (`{{.Service}}` / `{{.Version}}`), supplied by the composition root
  as before.

**Why convergence by diff against cron, not a fresh re-style.** The v1 pages
diverged independently per service; pinning every page to cron's exact bytes is
the only deterministic way to converge them and keep them converged. The done bar
is a byte comparison, not a visual judgement call.

**Rejected.**
- A shared cross-app landing handler — each app still owns its page (D1); this is
  content/style convergence, not code sharing.
- Editing only the eyebrow/description on the existing divergent markup — leaves
  the structural and token differences that are the actual style violation.
- Embedding a frozen copy of cron's HTML in this Decision — would fork the
  canonical page into nine drifting copies; pointing at the one cron file keeps a
  single source of truth.

**Verification.** Structural — no `R-XXXX-XXXX` ids (markup/asset content, like
D5). Realized by a plan phase whose done bar is a content check:

- `diff scripts/share/www/static/tokens.css cron/internal/web/static/tokens.css`
  (run from the repo root) prints nothing.
- `scripts/share/www/landing.html` differs from `cron/internal/web/landing.html`
  on **only** the three lines in the table above — carrying `scripts`, `Script runner`,
  and `Scripts runs deterministic Python scripts wired to suite events and publishes completion events back to the event plane.` respectively; all other lines are identical.
- The suite stays green: `cd scripts && go build ./...`, `go vet ./...`,
  `gofmt -l .` (prints nothing), and `go test ./...`.
