# Decision 16 — Prove no loopback-port literal survives, and guard the deploy artifacts against registry drift

**Problem.** The adoption in D15 is only complete if it is **enforced by a test**,
not by good intentions (the suite's "enforce with systems" principle). After D15
there are still `127.0.0.1:3100` literals in crm's tree, of two kinds:

- **Go-source loopback-address strings** — for crm these live *only in test
  assertions*: `cmd/crm/main_test.go` asserts the nginx fragment's `proxy_pass`
  upstreams as the literal strings `http://127.0.0.1:3100/` (landing),
  `http://127.0.0.1:3100/.well-known/oauth-protected-resource` (PRM bootstrap),
  and `http://127.0.0.1:3100/static/` (static), and the manifest byte-equality
  test emits `Port: 3100`. crm never had peer-feed literals (it is a producer, so
  D15 removed the single non-test literal), so these test assertions are the
  entire remaining Go-source surface.
- **Non-Go deploy artifacts** — `etc/nginx.conf` is an nginx fragment (nginx
  cannot call Go) and `etc/manifest.env` is generated deploy metadata; both
  legitimately contain the literal `3100`. `registry` cannot *own* their contents
  without generating them, which is out of scope. What it *can* do is make crm's
  own tests assert those artifacts match `registry`, converting a silent drift
  into a red test.

**Decision.** Two guards, both living entirely under `crm/` and running in the
ordinary `cd crm && go test ./...`:

**Guard 1 — source-scan (`R-X1CA-0373`).** A test in `cmd/crm` (or a small
dedicated `internal/` guard package) walks every `*.go` file under the `crm/`
module root and fails if any file's source contains a bare loopback-address
literal of the form `127.0.0.1:30` followed by two digits. Two constraints make
it robust:

- **It must not match itself.** The test assembles the forbidden needle from
  fragments at runtime (e.g. `"127.0.0.1:" + "30"`) rather than embedding the full
  literal, and skips its own filename, so the guard's own source can never trip
  it.
- **It targets the loopback *address* form only.** The needle is `127.0.0.1:30` +
  two digits, so the legitimate `127.0.0.1:0` (`net.Listen` for a free port) and
  `127.0.0.1:%d` (the `waitForHealth` URL format string) test literals do **not**
  match — those bind ephemeral/formatted ports, not the fixed `30xx` upstream that
  drifts. After D15 + the guard-2 test re-pointing there are zero `127.0.0.1:30xx`
  literals, so the scan is green; reintroducing a hardcoded upstream like
  `"http://127.0.0.1:3100/"` turns it red.

**Guard 2 — deploy-artifact drift (`R-X2K6-DUXS`).** The existing manifest and
nginx tests are re-pointed at `registry` so their expected port is *derived*, not
hardcoded, keeping every one of their current behavioral assertions:

- **`cmd/crm/main_test.go` manifest test** (`TestManifestLibraryByteEqualsCommittedFile`
  — its id is minted and covered by the repo-root `project/` workspace, so it is
  cited here by name only) currently emits `manifest.Emit(manifest.Fields{... Port: 3100
  ...})` and compares to the committed `etc/manifest.env`. Change the literal to
  `Port: registry.MustPort("crm")`. Every other `Fields` value (App, Mount,
  Default, MCP, Feed, the `OUTBOX_RETENTION_*` extras) and the byte-equality
  assertion stay as they are; the emitted `PORT=3100` still matches the committed
  file today. If `registry` renumbers crm, `manifest.Emit` produces a different
  `PORT` line and the byte-equality assertion fails — surfacing that the committed
  `manifest.env` is stale. This is also where R-X04D-MBGE (D15's own-port id) is
  concretely proven, since the emitted port derives from `registry.MustPort("crm")`.
- **`cmd/crm/main_test.go` nginx tests** — the landing (`R-NGNX-6M9N`), PRM-survival
  (`R-NGNX-8P1Q`), and static (`R-SWNU-U5QA`) location tests assert the fragment's
  `proxy_pass` upstreams. Build the expected origin from `registry.BaseURL("crm")`
  (== `"http://127.0.0.1:3100"`) and assert the fragment contains
  `registry.BaseURL("crm") + "/"`, `registry.BaseURL("crm") + "/.well-known/oauth-protected-resource"`,
  and `registry.BaseURL("crm") + "/static/"` respectively. Keep the exact-match vs
  prefix distinction, the session/bearer `auth_request` gate assertions, the feed
  `return 404` survival, and the PRM ungated check exactly as they are — only the
  port value they compare against becomes a `registry` call. The assertions hold
  today; a renumber makes the still-literal `etc/nginx.conf` disagree with
  `registry.BaseURL("crm")` and the tests go red.

**Scope guard.** These guards add / re-point tests only — no production code
beyond D15, no schema change (**no migration**), and **no edit** to
`etc/nginx.conf` or `etc/manifest.env` themselves (their literal `3100` stays;
the tests now police it). Nothing outside `crm/` is touched.

**Rejected.**
- **Regenerating `etc/nginx.conf` / `etc/manifest.env` from `registry`.** Out of
  scope and unnecessary: nginx-fragment authoring and the manifest artifact are
  owned elsewhere; a drift-catching test delivers the safety without a codegen
  step or a cross-module dependency. (crm's own nginx.conf header comment states
  the fragment ships verbatim — it is deliberately literal.)
- **An AST / int-literal scan for the bare `3100`.** Too noisy to be falsifiable —
  `3100` carries no context marking it as a port. Deriving the manifest and nginx
  expectations from `registry` (guard 2) proves the own-port literal is gone far
  more precisely.
- **Leaving the nginx / manifest tests as literal `3100` assertions.** They would
  keep four bare `127.0.0.1:3100` literals in crm's Go source (violating D15's
  end-state) and, worse, would keep *passing* after a `registry` renumber — the
  exact silent drift this whole adoption removes.

**Verification.**
- R-X1CA-0373 — **no bare `127.0.0.1:30xx` loopback-address string literal exists
  in crm's Go source.** A guard test walks the module's `*.go` files (skipping
  itself, needle assembled at runtime) and fails on any such literal; the
  ephemeral `127.0.0.1:0` and formatted `127.0.0.1:%d` test literals are not the
  `30xx` form and do not match. Falsifiable: reintroducing a hardcoded upstream
  like `"http://127.0.0.1:3100/"` makes it red; with D15 + guard-2 re-pointing it
  is green.
- R-X2K6-DUXS — **the committed `etc/manifest.env` and `etc/nginx.conf` ports agree
  with `registry`.** The manifest test emits with `registry.MustPort("crm")` and
  byte-matches the committed `manifest.env`; the nginx tests assert the fragment's
  `proxy_pass` upstreams against `registry.BaseURL("crm")`. Falsifiable: a
  `registry` value differing from the committed artifacts' `3100` fails these tests
  (the intended drift alarm); matching values pass.
