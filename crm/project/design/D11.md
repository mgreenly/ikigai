# Decision 11 — The `guide` tool and its embedded document

**Problem.** D10 moves the per-type field catalog out of `save`, and D9 points an
agent at "guide" for field catalogs and worked examples. That reference has to
exist and has to be reachable **in-band**, from any MCP client, without the host
app doing anything special. MCP resources are the semantically "correct" home for
reference text, but they are the weak channel — most clients do not auto-surface
resources into an agent's context, so an agent told to "call guide" might have no
way to. A **tool** is the channel with the hard delivery guarantee: it appears in
`tools/list` in every client and the agent can call it like any other.

**Decision.** Add a `guide` tool to crm's MCP surface — a read-only,
self-description tool that returns crm's usage guide as text. It joins `health`
and `reflection` as a **non-domain** tool: it is not a per-entity verb, so it does
**not** violate the "tool count is a function of verbs, not entities" principle
(`crm/CLAUDE.md`) — adding entities still adds no tools, and `guide` adds one
cross-cutting discovery tool, once.

Three coordinated changes, all in the existing `internal/mcp/` package; no new
package, no schema:

1. **Embed the document.** Author `internal/mcp/guide.md` and embed it:

   ```go
   //go:embed guide.md
   var guideDoc string
   ```

   Its content is the reference relocated from `save` (D10) plus worked examples:
   the **entity model**, the **per-type field catalog**
   (`organization`/`contact`/`deal`/`task`, the fields D10 removed from `save`),
   a **Basics** section (single-entity create / find / log / complete), and an
   **Advanced** section (the dedup/`force` override, the declarative
   set-replacement gotcha, the derived read-only deal `status`, filtered
   `search`, correcting an interaction). Interactions are logged, not saved.

2. **List it.** Add a `desc(tool("guide"), …)` entry to `toolDescriptors()`. It is
   **flat** — no `topic` parameter; the whole guide is returned in one call — so
   its `inputSchema` is `obj(map[string]any{})` (an object with no properties and
   no required fields). Description:

   > `Return the CRM usage guide — the entity model, per-type field catalogs,`
   > `and worked basic and advanced examples. Read once before your first save.`

3. **Dispatch it.** Add `case tool("guide"): return h.toolGuide()` to
   `dispatchTool()`, where `toolGuide()` returns `toolResultText(guideDoc)` — the
   embedded document as the MCP `tools/call` text result. It reads no DB, takes no
   identity, and never errors.

This brings crm's tool surface to **eight** (`search`, `get`, `save`, `delete`,
`log`, `health`, `reflection`, `guide`); `internal/mcp/tools_test.go`'s
exact-count assertion (currently seven) is updated to eight as part of realizing
this Decision.

**Rejected.**
- **An MCP `resource` (`crm://guide`) instead of a tool.** Weak delivery: many
  clients never surface resources to the agent, so "call guide" could be
  unreachable — defeating the product promise of use-from-connection-alone. A
  tool is guaranteed to appear in `tools/list`. (A resource could be added
  **additionally** later, same bytes, for clients that render resources; it is
  not a substitute and is out of scope here.) Rejected.
- **A `topic` parameter that carves the guide into sections.** For crm's size the
  whole guide is roughly one screen; a `topic` param adds a schema, a dispatch
  branch, and a decision the agent must make, for no real token saving over
  returning the flat document. Start flat; add `topic` only if the guide grows.
  Rejected.
- **Fold the guide into `reflection`.** `reflection` self-describes crm's
  **event-graph** edges — a different axis (what crm publishes/subscribes), with
  its own `event_type` parameter and shape. Overloading it with usage prose would
  muddy both. Keep them distinct. Rejected.
- **Return the guide from a non-MCP HTTP route (e.g. `GET /guide`).** Not in the
  agent's tool channel; the agent would need an out-of-band URL and a fetch it may
  not have. The point is in-band discovery. Rejected.

**Verification** (drive `tools/list` and `tools/call` via the `tools_test.go`
`rpc` harness).

- R-PK2P-EO04 — **The `guide` tool is listed.** `tools/list` returns a tool named
  `guide` (bare name; `toolPrefix` is empty) carrying a non-empty `description`,
  and the total tool count is **eight**. A surface still at seven tools, or one
  missing a `guide` entry, fails. *(tools/list)*
- R-PLAL-SFQT — **Calling `guide` returns the usage document — the catalog D10
  removed plus worked examples.** `tools/call` with `name:"guide"` returns a
  non-error text result whose content includes the per-type field catalog (it
  contains `given_name` and `amount_cents` — the very fields removed from `save`)
  **and** worked examples (it contains at least one example `save`/`log` call and
  the word `stage`). An empty or catalog-less guide fails. *(tools/call)*
- R-PMII-67HI — **`guide` is flat and input-free.** The `guide` tool's
  `inputSchema` declares **no** `required` fields, and a `tools/call` with empty
  arguments (`{}`, or omitted) returns the document without an `isError` result —
  so an agent needs no parameter knowledge to read it. A schema demanding a
  `topic` (or a call that errors on empty args) fails. *(tools/list + tools/call)*
