# Decision 10 — Lean tool descriptions: relocate the `save` field catalog

**Problem.** `save`'s description is the one heavy string on crm's surface: the
`saveDescription` const (`internal/mcp/tools.go`) is ~230 words / ~1.2 KB and
carries a full **per-type field catalog** (`organization`/`contact`/`deal`/`task`
field-by-field). Because `tools/list` returns every description on **every**
listing, that catalog is paid for on every connection whether or not the agent is
about to `save`. The catalog is reference material — needed when constructing a
`save`, not when deciding whether to use it — so it belongs behind an on-demand
fetch (`guide`, D11), not in the always-loaded listing.

**Decision.** Replace the `saveDescription` const with a **lean** description
that keeps only the cross-cutting `save` semantics an agent needs to decide *when
and how* to call it, and moves the per-type field catalog out to `guide` (D11).
No per-tool breadcrumb to `guide` — the single pointer lives in `instructions`
(D9). The other tool descriptions (`search`, `get`, `delete`, `log`, `health`,
`reflection`) are already lean and are **unchanged** by this Decision. The new
`saveDescription`:

> `Create (omit id) or update (provide id) an organization, contact, deal, or`
> `task. Upsert. Only the fields you send change; set-valued fields (emails,`
> `phones, tags, deal contacts) replace the whole set — omit to leave`
> `untouched, send [] to clear. On create a dedup probe runs; a match returns a`
> `duplicate error with existing_id unless force:true. Interactions are not`
> `saved here — use log.`

The `inputSchema` for `save` is **unchanged** (its `fields` property already reads
"entity-specific; see this tool's description" — still accurate at the level of
"there are per-type fields"; the enumerated catalog is what moves). No behavior
change: `save` validates and dispatches exactly as before.

**Rejected.**
- **Keep the full catalog in `save`.** The status quo; pays the 1.2 KB on every
  listing. The whole point is to stop that. Rejected.
- **Add a `guide` breadcrumb to `save` (and `search`).** Deliberately not: the
  agreed design references `guide` in exactly two places — `instructions` (D9)
  and the `guide` tool itself (D11). A per-tool breadcrumb re-bloats descriptions
  and duplicates the pointer. The accepted cost is that an agent which ignores the
  `instructions` pointer and calls `save` cold may need one corrective round; the
  server's per-type validation messages already guide that self-correction.
  Rejected.
- **Drop the set-valued-field and dedup sentences too, for maximum leanness.**
  Those are not per-type catalog — they are cross-cutting *semantics* (declarative
  replace, the `duplicate`/`force` contract) an agent must know to call `save`
  correctly at all. Keep them. Rejected.

**Verification** (drive `tools/list` via the `tools_test.go` `rpc` harness; assert
on the `save` tool's `description`).

- R-PGF0-9CS1 — **The `save` description no longer carries the per-type field
  catalog.** The `save` tool's `description` contains **none** of the catalog
  markers `Fields by type`, `given_name`, or `amount_cents` — the enumerated
  per-entity fields have moved to `guide`. The prior description (which contained
  all three) fails this. *(tools/list)*
- R-PIUT-0W9F — **The `save` description retains the cross-cutting semantics.**
  The `save` tool's `description` still states the upsert/dedup/declarative-set
  contract — it contains `force` (the duplicate override), the set-replacement
  rule (mentions `tags`/`emails` as a set that replaces, with `[]` to clear), and
  directs interactions to `log`. A description stripped of these (a bare "creates
  or updates") fails. *(tools/list)*
