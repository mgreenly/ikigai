# Decision 9 — Self-routing service `instructions`

**Problem.** An agent connecting to crm reads the MCP `initialize` response
first; its `instructions` string is the one always-loaded orienting text, seen
before any tool schema. Today it reads:

> "A sales CRM over organizations, contacts, deals, tasks, and interactions.
> Start with search to find things, then get, save, and log."

Two gaps. (1) It names the entities only in crm's internal nouns
(`organizations`, `contacts`, `deals`), so an agent whose user says "companies",
"people", or "pipeline" has nothing to match against — routing to crm depends on
an **external** vocabulary skill. (2) It gives no pointer to a fuller reference,
so an agent that needs field shapes has nowhere to go but the (now-lean, D10)
tool descriptions or a guess.

**Decision.** Rewrite the `initialize` `instructions` string (in
`internal/mcp/mcp.go`) so the always-loaded text carries the routing vocabulary
and the one pointer to `guide`. It stays two-to-three sentences — the surface's
orienting blurb, not its reference. The exact string:

> `Sales CRM over organizations (companies), contacts (people), deals`
> `(pipeline/opportunities), tasks (to-dos), and interactions (notes, calls,`
> `emails, meetings). Typical flow: search to find things, get for a full`
> `record, save to create or update, log to record an interaction. Call guide`
> `once for field catalogs and worked examples before your first save.`

The parenthetical synonyms (`companies`, `people`, `pipeline/opportunities`,
`to-dos`, `notes, calls, emails, meetings`) are the routing hooks: they are the
words a user actually types, so a client's tool/instructions search matches crm
without an external alias skill. The final sentence is the **only** in-surface
mention of `guide` besides the `guide` tool's own description (D11) — no per-tool
breadcrumbs (D10 keeps the other descriptions clean).

This is a pure string change at one site; no schema, no new field, no behavior
change to any tool.

**Rejected.**
- **Leave the instructions as-is and rely on the external vocabulary skill.**
  Defeats the product promise that crm is usable from its own connection with no
  external skill. Rejected.
- **Push the synonyms into every tool description instead.** Repeats the
  vocabulary N times and bloats the always-loaded `tools/list`; the `instructions`
  string is loaded once and is the natural home for orientation. Rejected.
- **Put the field catalog into `instructions` too.** That is exactly the bulk
  D10 removes from `save`; the always-loaded blurb must stay short. The catalog
  belongs in `guide` (D11), pulled on demand. Rejected.

**Verification** (drive `mcp.Handler.ServeHTTP` with an `initialize` request via
the `tools_test.go` `rpc` harness; assert on `result.instructions`).

- R-PDZ7-HTAN — **`instructions` carries the routing vocabulary, not just crm's
  internal nouns.** The `initialize` `instructions` string contains the
  user-facing synonyms `companies`, `people`, and `pipeline` (or `opportunities`)
  — so an agent whose user says those words can match crm. The prior string,
  which named only `organizations`/`contacts`/`deals`, fails this: absence of at
  least these three synonyms is the failure. *(initialize)*
- R-PF73-VL1C — **`instructions` points to `guide`.** The `initialize`
  `instructions` string contains the word `guide`, directing an agent to the
  read-only usage guide before its first `save`. A string with no `guide`
  reference fails. *(initialize)*
