# Decision 3 — Embedded Carbon design assets (crm's own copy)

**Decision.** crm embeds its **own** copy of the Carbon design assets and serves
them from its own mount; the landing page references those embedded assets, never
a cross-service URL. Concretely:

- A `crm/internal/web/static/` directory holds a **vendored copy** of
  `design/tokens.css` (the Carbon CSS custom properties) and the woff2 fonts the
  Carbon type pairing needs: `space-grotesk.woff2` (display), `ibm-plex-sans.woff2`
  (body), `ibm-plex-mono-400.woff2` + `ibm-plex-mono-500.woff2` (mono labels) —
  the same set the dashboard already embeds at `dashboard/ui/static/fonts/`.
- The package embeds the directory via `//go:embed static` into an `embed.FS`,
  and the handler/mux serves it under a stable in-service path — `GET /static/…`
  — with correct content types (`text/css; charset=utf-8` for `tokens.css`,
  `font/woff2` for the fonts).
- `tokens.css` declares its `@font-face` rules pointing at the **embedded** font
  paths (`/srv/crm/static/fonts/…` as seen by the browser, i.e. the service's
  `/static/fonts/…` once nginx re-adds the mount prefix), and the landing template
  links `tokens.css` from the same embedded `/static/` path. No asset is fetched
  from the dashboard or any other service at runtime.

**Why each app embeds its own copy.** The brief and the dashboard precedent
(`dashboard/ui/static/tokens.css` is a *vendored* copy precisely because
`go:embed` cannot reach across the module boundary into the sibling `design/`
tree) make this non-negotiable: `module crm` cannot `//go:embed` files outside
`crm/`. Vendoring the tokens + fonts into `crm/internal/web/static/` keeps crm a
self-contained single binary with no runtime dependency on another app's assets —
which is the whole point of the one-binary-per-service model.

**Carbon fidelity.** The page consumes the **semantic** Carbon tokens
(`--color-*`, `--space-*`, type tokens), not raw ramps, per `design/carbon.md`:
monochrome neutrals; blue `#2563EB` reserved as the only signal color (used here
only for a focus ring / link if present, never decoratively); 4px spacing grid;
tight 4px radius; hairline borders over shadow. The landing card is minimal and
on-system: service name in **Display** (Space Grotesk, 56/1.04/700), version in
the **Mono · Label** token (IBM Plex Mono, 12/1.4/500), centered within the
1120px max content width.

**Staleness note (vendored copy).** Because `tokens.css` and the fonts are a
vendored copy, they can drift from `design/` over time. That is an accepted cost
of the module boundary (the dashboard lives with the same), and is out of scope
to automate here; if the design system changes, the copy is re-vendored as a new
plan phase. v1 copies the current `design/tokens.css` and the four woff2 fonts
verbatim.

**Rejected.**
- **Linking the dashboard's `/static/tokens.css` at runtime** — couples crm to
  the dashboard's asset URLs and breaks the self-contained one-binary model; a
  dashboard asset rename would silently unstyle crm.
- **A shared design module imported by every app** — `go:embed` cannot cross the
  module boundary, and the suite deliberately keeps each app a standalone module;
  vendoring is the established pattern (dashboard).
- **Inlining all CSS into the HTML and shipping no fonts** — loses the woff2 type
  pairing (the page would fall back to system fonts and look off-system) and
  diverges from the dashboard's embedded-assets approach the brief points at.
- **Serving the fonts from a CDN** — the box is single-tenant and may run with no
  external egress guarantees; assets must be local.

**Verification.**
- R-ASST-2B8C — a `GET /static/tokens.css` (or wherever the package mounts it) is
  served by the web package with status `200` and a CSS content type
  (`text/css`).
- R-ASST-4D1E — the landing template's stylesheet link and the page's font
  references resolve to crm's **own** embedded `/static/` path (the rendered HTML
  contains the app-local asset path and contains **no** cross-service/dashboard
  asset URL).
- R-ASST-6F3G — the woff2 font assets are embedded and served: a `GET` for an
  embedded font path (e.g. `/static/fonts/space-grotesk.woff2`) returns `200`
  with a `font/woff2` content type.
