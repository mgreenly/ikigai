# Decision 18 — Event-routing conformance: kinds keep `contact.*`, subject = `/<contact id>`

> **External ordering (operator-sequenced, not built by any crm phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before crm's conformance phase runs**. It equally assumes appkit (which holds
> `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, crm's conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and events
gain a subject — the producer's name for the thing the fact happened to — so
consumers can route without reading payloads. Today a consumer that cares about
one contact's funnel moves must be woken by **every** contact event on the box,
because the contact id lives only in the payload, where filters cannot see it.
It is a hard cutover — no compatibility period — inside the no-live-data
migration window.

**What crm publishes today (ground truth, not vacuous).** crm is a real, wired
producer: `Spec.Feed: "/feed"`, `Spec.Events: crm.Events` (an `outbox.Registry`
of four entries in `internal/crm/events.go`), the `Producer` hook
(`cmd/crm/main.go`) injecting the outbox into `crm.Service`, whose `Save` path
builds events via `contactEvents` and appends each with
`Outbox.Append(tx, ev)` on the **same tx** as the domain write, `Ring()` after
commit. The four first-wave types — `contact.created`, `contact.updated`
(full contact snapshot, `contactSnapshotPayload`), `contact.tagged`,
`contact.untagged` (segment-membership delta, `contactTagPayload` =
`{contact_id, tag}`) — are declared once as unexported constants in
`internal/crm/events.go` and referenced at both the emit sites and the
registry. All four published types convert; conformance is not vacuous.

**Decision.** crm adopts the revised envelope exactly over what it publishes,
**keeping the entity-prefixed kind names** and taking the **contact id as the
subject** — the routing doc's key map direction
(`crm:contact.created/<contact-id>`):

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `contact.created` | `crm:contact.created/01J9Z2K7P3QC8M4R6T0V2X5YA` | `contact.created` | `/` + the contact's ULID |
| `contact.updated` | `crm:contact.updated/01J9Z2K7P3QC8M4R6T0V2X5YA` | `contact.updated` | `/` + the contact's ULID |
| `contact.tagged` | `crm:contact.tagged/01J9Z2K7P3QC8M4R6T0V2X5YA` | `contact.tagged` | `/` + the contact's ULID |
| `contact.untagged` | `crm:contact.untagged/01J9Z2K7P3QC8M4R6T0V2X5YA` | `contact.untagged` | `/` + the contact's ULID |

The kinds **keep their `contact.` prefix**: unlike gmail ("mail" is all gmail
deals in) or dropbox ("file" is all dropbox deals in), the source `crm` names a
multi-entity service — the documented second wave (`deal.stage_changed`,
`deal.won`, `interaction.logged`, …) needs the entity noun in the kind, so
`contact` is **domain information, not redundancy** (the prompts precedent:
`run.succeeded` kept its noun). The subject is the **id of the contact the
fact happened to** — for the snapshot events the contact the snapshot
describes, for the tag events the contact whose segment membership changed
(the tag itself is data and stays in the payload). Payload shapes are
**unchanged** — `contactSnapshotPayload` and `contactTagPayload` keep every
field, including `id`/`contact_id` (those are payload data, not a stale
discriminator; crm's payloads carry no `event`/`type` field to drop). The
routing key addresses; the payload remains the detail record.

1. **Constants and event builders** (`internal/crm/events.go`): the four
   unexported constants keep their string values (`eventContactCreated =
   "contact.created"`, etc. — they are now *kind* constants). `contactEvents`
   emits `outbox.Event{Kind: typ, Subject: "/" + s.ID, Payload: raw}` for the
   snapshot event and `contactTagEvent` emits `outbox.Event{Kind: typ,
   Subject: "/" + contactID, Payload: raw}` for each tag delta. Nothing else
   in the `Service`/outbox seam changes (same tx as the domain write, `Ring()`
   after commit, the tag diff still read off the `Summary` side-band).

2. **The registry becomes families** (`internal/crm/events.go`): `crm.Events`
   becomes an `outbox.Registry` of four `outbox.Family` entries —
   `{Kind: "contact.created", Subject: "/<contact id>", Description: …,
   Sample: sampleContactSnapshot}` etc. Descriptions keep their current
   meaning with the addressing updated. `Spec.Events` wiring is untouched;
   Append-time kind gating and the chassis reflection surface hang off the
   same registry, now family-shaped, with schema/example still derived from
   the same Sample values.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. crm appends **one new
   timestamped migration** minted with `bin/create-migration crm
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across.
   The frozen `003_outbox.sql` is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest**
   outbox migration, asserting it contains the revised `outbox.SchemaSQL`
   verbatim. (This exactly mirrors gmail D18/phase-16, dropbox D22/phase-23,
   and ledger D15/phase-13.)

4. **Reflection/consumer surface.** The chassis reflection tool (appkit-owned)
   reflects the family registry; crm's observable is that its four kinds —
   and only they — are listed with the `/<contact id>` subject description.
   crm consumes nothing (producer only), so there is no
   filter/`consumer.Config` side to conform.

**Rejected.**
- **Dropping the `contact.` prefix (kinds `created`/`tagged`)** — the
  revision's rename rule deletes *redundant* nouns only; `crm` (the source) is
  not "contacts", and the second-wave `deal.*`/`interaction.*` kinds would
  collide or blur without the entity noun. The routing doc's own key map
  records `crm:contact.created/<contact-id>`.
- **An empty subject (gmail's choice)** — the contact id is exactly the
  routing dimension a consumer needs (`crm:contact.*/01J9Z2K7P…` selects one
  contact's whole lifecycle; `crm:contact.tagged/**` selects all tag gains);
  leaving it empty preserves the fire-on-everything failure the revision
  exists to fix, and the routing doc's key map directs the contact id.
- **Subject = the tag for `contact.tagged`/`contact.untagged`** — the subject
  names the thing the fact happened *to*, which is the contact; the tag is
  the delta detail and stays in the payload. It would also split one entity's
  events across two subject conventions.
- **Stripping `id`/`contact_id` from the payloads now that the subject
  carries the id** — payloads are the self-describing detail record and their
  shapes are contract; unlike dropbox's `event` field this is data, not a
  duplicated envelope discriminator, and consumers already read it.
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are retention data with no live
  consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: the emission claims run over the
existing `internal/crm` harness — a real SQLite database through the full
embedded migration set, the real `Service.Save` path — reading outbox rows
back by SQL; the feed claim runs the real eventplane `FeedHandler` over
`httptest`).

- R-8HHB-24SG — after a `Service.Save` creates a contact with one tag and a
  second `Save` updates that contact adding a tag and removing another, the
  outbox rows read back by SQL from the real SQLite database carry `kind`
  values exactly `contact.created`, `contact.updated`, `contact.tagged`, and
  `contact.untagged`, **every** row's `subject` equal to `/` + that contact's
  ULID, each appended on the same tx as the domain write (baseline atomicity
  preserved) — and the payload JSON shapes are unchanged
  (`contactSnapshotPayload` fields on the snapshot rows; exactly
  `contact_id` + `tag` on the tag rows).
- R-8IP7-FWJ5 — `crm.Events` declares exactly four families with kinds
  `contact.created`/`contact.updated`/`contact.tagged`/`contact.untagged` and
  subject description `/<contact id>` on each; through the assembled MCP
  handler, reflection's index lists exactly those four publishes, the detail
  of `contact.created` returns a schema and example that agree (every example
  field in the schema, `contactSnapshotPayload`-shaped), and an undeclared
  kind returns the typed unknown-kind error naming the four declared kinds.
- R-8JX3-TO9U — applying crm's full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table
  with `kind` and `subject` columns and **no** `type` column; the DDL drift
  guard asserts the newest outbox migration contains the revised
  `outbox.SchemaSQL` verbatim while `003_outbox.sql` remains byte-identical
  to its committed (frozen) body.
- R-8L50-7G0J — end-to-end over crm's own substrate: an event appended
  through crm's producer path (`contactEvents` → `Append` on the
  migration-created table, source `crm` — the `Spec.App` value) for a created
  contact and served by the real eventplane `FeedHandler` over an `httptest`
  connection frames `event: crm:contact.created/<that contact's ULID>` (the
  canonical key) with a `data:` JSON envelope carrying `kind` and `subject`
  keys and **no** `type` key.
