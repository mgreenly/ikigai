# Decision 19 — Structured MCP adoption: `structuredContent`, declared `outputSchema`, typed error codes

**Problem.** The suite settled a single verb surface for agents and machines: a
domain verb is registered once and answers every caller, its result carrying
`structuredContent` for deterministic callers and mirrored text for agents, its
tool declaring an `outputSchema`, and its errors carrying a code from a shared
closed vocabulary (`docs/structured-mcp-design.md`). appkit already shipped the
transport half (protocol `2025-06-18`; `Tool.OutputSchema`;
`StructuredResult(v) (map[string]any, error)`; `ErrorResult(code ErrorCode,
msg string)` over the closed vocabulary `validation`/`not_found`/`conflict`/
`too_large`/`source_unavailable`/`internal`; `JSONResult` **deleted**). crm's
`internal/mcp/tools.go` has not yet adopted it: its five domain result sites
still call the deleted `appkitmcp.JSONResult`, its `toolErr` JSON-encodes a
nested `{"error":{code,message,field?,existing_id?}}` string into the **old**
single-argument `ErrorResult`, and no domain tool declares an `outputSchema`. crm
does not compile against the current appkit until this lands.

crm has **no hand-copied loopback guard site** — it mounts no loopback-private
route of its own, and the chassis already wraps its `/feed` mount with
`server.LoopbackOnly` (appkit). So the suite's guard-swap item does **not**
apply here; this Decision is result-shape and error-code only.

**Decision.** Adopt the structured contract in `internal/mcp/tools.go`, changing
result *encoding* only — the JSON each verb emits is unchanged, now formalized as
its schema:

- **Success results via `StructuredResult`.** Each of the five domain verb
  handlers (`toolSearch`, `toolGet`, `toolSave`, `toolDelete`, `toolLog`)
  replaces its `appkitmcp.JSONResult(v)` tail with `appkitmcp.StructuredResult(v)`
  over the **same** value. Because the chassis helper now returns
  `(map[string]any, error)`, the marshal error is propagated up the handler's
  existing `(map[string]any, error)` return — not swallowed, not discarded. The
  values are unchanged: search returns `{items, next_cursor}`, get returns the
  per-type `Card` (open object), save/log return a `Summary`, delete returns
  `{ok:true}`.

- **Declared `outputSchema` per domain verb** — a hand-authored `map[string]any`
  JSON-schema literal on each `mcp.Tool`, mirroring the JSON that verb already
  emits, built with the existing `obj`/`typ` helpers plus a small private
  `summarySchema()` for the shared `Summary` shape:
  - `search` → `obj({items: array<summarySchema>, next_cursor: {type:string}},
    "items")`.
  - `get` → the per-type `Card` is a composed `map[string]any` whose keys vary by
    entity, so its schema is the **open object** `{type:object,
    additionalProperties:true}` — the honest schema for a polymorphic card, not a
    fabricated fixed shape.
  - `save` → `summarySchema()`; `log` → `summarySchema()`.
  - `delete` → `obj({ok: {type:boolean}}, "ok")`.
  - `summarySchema()` = `obj({id, type, label, updated_at: {type:string},
    fields: {type:object, additionalProperties:true}}, "id", "type", "label",
    "updated_at")` — `fields` is the open, `omitempty` per-type disambiguator.
  - `guide` declares **no** `OutputSchema` (left nil), so `tools/list` omits the
    key for it (chassis omits a nil schema).

- **Typed error codes.** `errorEnvelope`/`toolErr` are replaced by a single
  mapping from crm's domain errors onto the appkit `ErrorCode` closed vocabulary,
  returning `appkitmcp.ErrorResult(code, msg)` directly (the wire error is now
  `isError:true` + `structuredContent:{code,message}` + a mirrored text block —
  the chassis shape). The nested `{"error":{…}}` envelope and the JSON-in-a-string
  body are **retired**. The mapping:
  - `*crm.DuplicateError` → `ErrConflict`. A duplicate is a uniqueness conflict;
    `duplicate` is **not** in the closed vocabulary. The human message is the
    error's own text, which already embeds `existing_id=<id>` (service.go dedup
    sites), so the agent's recovery path (update, or retry `force:true`) is
    unchanged; a machine branches on `code == "conflict"`.
  - `*crm.ValidationError` → `ErrValidation`. The message is the carrier's
    corrective text; the previously-separate `field` is folded into that message
    (it is already named in the corrective wording), since `ErrorResult` carries
    only `{code, message}`.
  - `errors.Is(err, crm.ErrValidation)` → `ErrValidation`;
    `errors.Is(err, crm.ErrNotFound)` → `ErrNotFound`;
    `errors.Is(err, crm.ErrConflict)` → `ErrConflict`.
  - default (an unclassified error) → `ErrInternal` with a fixed
    `"internal error"` message (no leak of internal detail).
  - `too_large` and `source_unavailable` are in the vocabulary but crm has no
    failure mode that produces them; they are correctly never emitted.

- **Prose exception.** `guide` (D11) is a documentation tool: it returns the
  embedded `guide.md` via `TextResult` and declares no `outputSchema`. It is the
  service's only prose tool. The chassis `health`/`reflection` are not crm's to
  classify — appkit declares their schemas and returns them structured centrally.

The wire tool set, names, descriptions, and input schemas are unchanged; only the
result encoding gains `structuredContent`/`outputSchema` and the error `code`
moves to the closed vocabulary. This Decision supersedes D13's note that the
`{"error":{code,message,…}}` content is carried verbatim (D13 rewritten in place
to defer the result/error shape here).

**Rejected.**
- **A second loopback domain API for machine callers** — the suite-level road not
  taken (`docs/structured-mcp-design.md`): a permanent mirror surface to save one
  JSON-RPC envelope. MCP is the single verb surface.
- **Keeping the nested `{"error":{…}}` envelope alongside the code** — it would
  duplicate `code`/`message` in two shapes and force machine callers to reach
  past `structuredContent`; the chassis `{code, message}` is the one contract.
- **A fabricated fixed `outputSchema` for `get`** — the `Card` is deliberately a
  per-type composed map; pinning a fixed property set would be a schema the tool
  does not honor. `{additionalProperties:true}` is the truthful contract.
- **Runtime schema validation of every result in the handler** — a suite
  non-goal; conformance is the test suite's job, not the transport's.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a real migrated DB, driven by `tools_test.go` with
`initialize`/`tools/list`/`tools/call`, matching the D13 test strategy).
- R-5Y60-E30A — `tools/list` through the assembled handler shows a non-nil
  `outputSchema` (a `type:object` JSON schema) on **each** of the five domain
  verbs `search`, `get`, `save`, `delete`, `log`, and **no** `outputSchema` key
  on `guide`. Table-driven over the six crm tools; a domain verb missing its
  schema, or `guide` carrying one, fails.
- R-5ZDW-RUQZ — a successful `tools/call` to each domain verb returns a result
  whose `structuredContent` deep-equals the JSON parsed from that same result's
  single text block (`content[0].text`) — the machine and agent renderings are
  the same value — and whose shape matches the verb (`delete` → exactly
  `{ok:true}`; `save`/`log` → a `Summary` object carrying `id`/`type`/`label`/
  `updated_at`; `search` → `{items, next_cursor}`). A result with a text block
  but no `structuredContent` fails. Table-driven against the real handler + DB.
- R-60LT-5MHO — every domain-verb **error** result sets `isError:true` and a
  `structuredContent.code` drawn only from the appkit closed vocabulary
  (`validation`/`not_found`/`conflict`/`too_large`/`source_unavailable`/
  `internal`), with the result carrying **no** top-level `error` key and **no**
  legacy `duplicate` code. A result nesting `{"error":{…}}`, or whose
  `code == "duplicate"`, fails.
- R-61TP-JE8D — a `save` create whose primary email (or org domain/name)
  collides with an existing live row, with `force` absent/false, returns
  `isError:true` and `structuredContent.code == "conflict"` (not `"duplicate"`,
  not `"validation"`), and the human text still names the existing row
  (`existing_id=` present). Driven against the real handler over a migrated DB
  with the colliding row pre-inserted; a wrong code or a dropped `existing_id`
  fails.
- R-631L-X5Z2 — a `get` (and a `delete`) of an id that does not resolve to a
  live row returns `isError:true` and `structuredContent.code == "not_found"`.
  Any other code fails.
- R-65HE-OPGG — a `save` rejected at the decode/validation seam (a `deal` with a
  client-supplied `status`, or a create missing a required field) returns
  `isError:true` and `structuredContent.code == "validation"`. Any other code
  fails.
