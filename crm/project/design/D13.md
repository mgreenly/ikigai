# Decision 13 — MCP surface over `appkit/mcp`: `internal/mcp` becomes the tool table

**Problem.** crm's `internal/mcp` carries the copy-pasted JSON-RPC transport
(`ServeHTTP`, the `initialize`/`tools/list`/`tools/call` dispatch,
`writeJSONRPCError`, the result envelopes) plus its own `health` and
`reflection` implementations — ~300 lines shared byte-for-byte in spirit with
ten other services. The chassis now owns all of it (appkit D8 transport, D9
standard tools, consumed here as a fixed external contract); what is genuinely
crm's is the domain tool table and the instructions string.

**Decision.** `internal/mcp` is reshaped from a transport into a **declaration**:

```go
// internal/mcp — what remains.
const Instructions = /* the D9 string */

// Tools declares crm's six domain tools over the domain Service:
// search, get, save, delete, log (D9/D10 descriptors + schemas, handlers
// delegating to svc exactly as dispatchTool did), and guide (D11, the
// embedded guide.md). health and reflection are NOT declared — the chassis
// registers them (their inputs all come from Router accessors).
func Tools(svc *crm.Service) []mcp.Tool

// NewHandler assembles the /mcp http.Handler at the composition root:
// appkit/mcp.New over Instructions + Tools(svc) + the Router-threaded
// Service/Version/Health/Events/Subscriptions.
func NewHandler(svc *crm.Service, rt *appkit.Router) (http.Handler, error)
```

- **Deleted:** the JSON-RPC plumbing (`jsonRPCRequest`, `ServeHTTP` dispatch,
  `writeJSONRPCError`, the local `Identity` type), the local
  `toolHealth`/`toolReflection`/`renderSubscriptions`/
  `reflectionUnknownTypeError` implementations, and the local result-envelope
  helpers (replaced by the chassis result helpers).
- **Kept:** the descriptor/schema helpers (`desc`/`obj`/`typ`/…, now producing
  `mcp.Tool` values), the tool descriptions (D10), the verb handlers'
  decode-and-delegate bodies (now `Tool.Handler` funcs over
  `appkit.Identity`), and the embedded `guide.md` (D11). The **result and error
  encoding** is settled separately by D19 (structured MCP adoption): success
  results carry `structuredContent`, tools declare `outputSchema`, and errors
  carry a closed-vocabulary `code` — D13 only relocates the transport, not the
  result shape.
- **The tool surface is unchanged**: same eight tools (`search`, `get`, `save`,
  `delete`, `log`, `guide` declared; `health`, `reflection` chassis-registered),
  same descriptions, input schemas, and instructions. Every existing D9–D11
  Verification id is re-proven through the new seam; the `tools_test.go` harness
  swaps its handler constructor and keeps its assertions. (Result/error
  *encoding* is D19's concern.)

**Rejected.**
- **Keeping the local transport** — the status quo; a protocol fix would still
  be an eleven-service edit.
- **Declaring health/reflection in crm's table** — the chassis reserves the
  names (appkit D8) precisely so services cannot fork the standard tools; their
  inputs (Envelope, Registry, Subscriptions) are chassis-held already.
- **Dissolving `internal/mcp` into `cmd/crm`** — the tool table is ~500 lines
  of descriptors, schemas, and handlers with its own test harness; that is a
  real package, unlike the post-D12 web remnant.

**Verification** (through the composition-root seam — the appkit/mcp handler
built by `NewHandler` over a real migrated DB).
- R-MW1X-S9EV — `tools/list` through the assembled handler returns **exactly
  eight** tools: the six crm declares (`search`, `get`, `save`, `delete`,
  `log`, `guide`) **plus** `health` and `reflection` supplied by the chassis —
  proving the table/chassis partition (a table that still declares its own
  health/reflection fails `New`'s reserved-name check; a surface missing them
  fails the count).
