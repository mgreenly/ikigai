# Decision 13 — The LLM-call footprint: `llm_calls` table + recorder seam

**Decision.** Every call the wiki makes to an LLM is recorded as one durable row, so a failed or aborted ingest leaves an inspectable trail of exactly what the model was asked, what it returned, the parameters it ran under, and the tokens it burned (product: "Record and inspect every LLM call"). The recording happens at the **single seam every provider round-trip funnels through** — `sendText` in `internal/llm` — so extract's parse-retries, compile's tighten loop, and ask's calls are all captured by one mechanism, each round-trip its own record.

**The recorder seam (`internal/llm`).** A `Recorder` is injected into `Client` (nil = no-op, so unit tests and any non-recording path stay simple). `sendText` builds one `CallRecord` per provider round-trip and hands it to the recorder.

```go
// CallRecord is one provider round-trip's footprint.
type CallRecord struct {
    ID        string    // ULID
    Stage     string    // from CallSite.Stage: "extract"|"compile"|"ask-subject"|"ask-synthesis" (D19); eval adds "judge" (D21)
    JobID     string    // from ctx via WithJobID; "" for ask (no job)
    Attempt   int       // 1-based within a JSON retry / tighten loop
    Provider  string    // e.g. "anthropic"
    Model     string
    Params    string    // JSON: {"temperature":…, "reasoning":…}
    Request   string    // JSON: {"system":…, "user":…} actually sent (incl. corrective re-prompt)
    Response  string    // raw final assistant text ("" on transport error)
    Usage     string    // token/cost JSON when the provider reports it ("" otherwise)
    Err       string    // transport / stream / cancellation error; "" on success
    StartedAt time.Time
    EndedAt   time.Time
}

// Recorder persists one call record. Injected into Client; a nil Recorder is a no-op.
type Recorder interface {
    Record(ctx context.Context, rec CallRecord) error
}
```

**Where the metadata comes from.**
- **Stage** — a new `CallSite.Stage` field. Each stage already owns its `DefaultCallSite` (extract D6, compile D7, ask D9), so each labels itself there; the composition root cannot silently drop it. (`CallSite` grows one field; the seam's faithful-application guarantee, D5, now covers `Stage` too.)
- **JobID** — threaded through the Go `context`. `internal/llm` exposes `WithJobID(ctx, id) context.Context` and `JobID(ctx) string`. The **worker** wraps the integrate context once per job (`ctx = llm.WithJobID(ctx, job.ID)`); `sendText` reads it back. `ask` never sets it, so its records carry `JobID == ""`. The job association never enters extract/compile/ask signatures. (Chosen over an explicit `jobID` parameter on `JSON[T]`/`Converse`, which would pollute every stage call with an id the stage code does not otherwise use.)
- **Attempt** — the `JSON[T]` loop stamps its 1-based iteration counter onto each round-trip's record; compile's tighten loop (D7) does the same. A single-shot call records `Attempt == 1`.

**Eager, detached writes — records are NOT in the job's transaction.** Each record is persisted the moment its round-trip completes, in its own auto-commit write, *outside* the job's final atomic commit (D14). This is the point of the feature: a job that ultimately **fails or is aborted still leaves every call record behind**. Two consequences the build must honor:
- The recorder writes with a **detached context** (`context.WithoutCancel(ctx)`), so recording an *aborted* call — whose request context was just cancelled — still succeeds, landing a row with `Err: "context canceled"` that shows exactly where the abort hit.
- Under single-writer SQLite the recorder's write is sequential with the rest of the worker; because integrate holds **no** DB transaction across the LLM calls (D4 intent, realigned in D14), an eager record write never deadlocks against a held job transaction.

**Faithful capture.** `Request` is the JSON of the messages actually sent (`{system, user}`), so the corrective re-prompt on a retry is captured verbatim, not the original prompt. `Params` serializes the `CallSite`'s temperature and reasoning as resolved for the call — the exact lever that drifted in D6/D7 — so a wiring bug (provider-default temp, reasoning-on) is visible in the record, distinguishable from a prompt bug. `Provider`/`Model` come from the `Client`'s configured provider and `CallSite.Model`. `Usage` is wired from agentkit's reported usage on the final stream event when present; absent that, `""`.

**Persistence (`LLMCallStore` in `internal/wiki`).** A SQLite-backed `Recorder`. New forward migration (`bin/new-migration wiki`), matching the live schema's TEXT-timestamp (`RFC3339Nano`) convention and the suite's no-`FOREIGN KEY` style:

```sql
CREATE TABLE llm_calls (
  id          TEXT PRIMARY KEY,            -- ULID
  job_id      TEXT NOT NULL DEFAULT '',    -- → jobs.id by convention; '' for ask
  stage       TEXT NOT NULL,               -- 'extract'|'compile'|'ask-subject'|'ask-synthesis' (D19); 'judge' only in eval (D21)
  attempt     INTEGER NOT NULL DEFAULT 1,
  provider    TEXT NOT NULL DEFAULT '',
  model       TEXT NOT NULL DEFAULT '',
  params      TEXT NOT NULL DEFAULT '{}',
  request     TEXT NOT NULL DEFAULT '',
  response    TEXT NOT NULL DEFAULT '',
  usage       TEXT NOT NULL DEFAULT '',
  error       TEXT NOT NULL DEFAULT '',
  started_at  TEXT NOT NULL DEFAULT '',
  ended_at    TEXT NOT NULL DEFAULT ''
);
CREATE INDEX llm_calls_job  ON llm_calls (job_id, started_at, id);  -- per-job inspect + D15 paging
CREATE INDEX llm_calls_time ON llm_calls (started_at, id);          -- timeframe paging
```

Records are **append-only and kept forever** — no update, no delete in this release (pruning deferred, product). A re-run (D14) replaces a job's *claims* but its old call records remain, accumulating beside the new attempt's.

**How it's tested.** A **capturing `Recorder` mock** is wired into `llm.Client` beside the existing scripted provider mock (D5 testing strategy):
- A `JSON[T]` call scripted bad-then-good emits **two** records: `Attempt 1` carrying the malformed response, `Attempt 2` carrying the good one — both carrying the `CallSite`'s `Stage`, `Model`, and `Params`, and the second carrying the corrective re-prompt in `Request`.
- A scripted transport/stream error emits **one** record with non-empty `Err` and empty `Response`.
- With `WithJobID(ctx, "J")` set, records carry `JobID == "J"`; with no job on the context (the ask path), `JobID == ""`.
- The detached-write behavior is proven by calling `Record` with an **already-cancelled** context against a real temp SQLite and asserting the row still lands.
- `LLMCallStore.Record` round-trips every column against a real temp SQLite migrated by the appkit runner.

**Rejected.**
- **Recording inside the job's atomic commit** — would discard exactly the records you most want (a failed/aborted job's), since the job's write-set is rolled back. Eager out-of-band writes are the whole value. Rejected.
- **An explicit `jobID` parameter on `JSON[T]`/`Converse`** — honest but noisy: it threads a job id through extract/compile/ask, which have no other use for it, and invites a stage to forget to pass it. Context-threading keeps the association entirely out of stage code. Rejected.
- **Reusing the existing `agentkit.Conversation.Log io.Writer` JSONL sink** — a flat log file is not queryable, not filterable, and not paginable; the product promises a listable, filterable footprint, which requires a table. The `Log` sink stays as an orthogonal debug convenience. Rejected as the primary record.
- **One record per `JSON[T]` call (final attempt only)** — throws away the failed attempts, which are precisely the structured-content failures being diagnosed. One record per round-trip is required. Rejected.

- R-VOZW-FQYU — A `JSON[T]` call scripted bad-then-good emits two records — `Attempt 1` with the malformed response and `Attempt 2` with the good response — and the `Attempt 2` `Request` carries the corrective re-prompt, not the original prompt.
