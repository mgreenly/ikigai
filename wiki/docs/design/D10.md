# Decision 10 — The MCP tool surface (`internal/mcp`) + identity

**Decision.** The product surface is MCP. Transport is copied near-verbatim from crm: JSON-RPC 2.0 over `POST /mcp` (`initialize`, `notifications/initialized`, `tools/list`, `tools/call`), tools declared as hand-coded maps dispatched by a `switch`, result helpers `toolResultText/Err/JSON`. The route is mounted `rt.RequireIdentity(mcp.NewHandler(...))` — nginx is the trust boundary; wiki does no token logic.

**Fifteen-verb surface.** The original eight verbs below, plus the five control/footprint verbs (`jobs`, `jobs_count`, `abort`, `rerun`, `llm_calls`) and the cursor-paginated reshape of `subjects`/`claims` — all owned by **D16** — plus the two merge verbs (`merge`, `merges`) owned by **D27** (this section's eight-verb framing is superseded there; R-MUQ4 below now asserts the fifteen-verb set). No public `search` verb — this release ships no keyword/body search at all (D8).

A subject is named everywhere by its public `type/slug` **path** (D11), never its internal id. The `subject` input of `page`/`claims` is a path, resolved via `SubjectStore.GetByPath`; every subject reference in a result is a path.

> **`subjects`/`claims` reshaped by D16.** Both now accept `limit`/`cursor` and return a `{items, next_cursor}` cursor-paginated envelope (D15) rather than a bare list. The per-item shapes below (and their paths-out / no-internal-id guarantees) are unchanged; they are simply wrapped in `items`.

| tool | input | → result | dispatches to |
|---|---|---|---|
| `ingest` | `{text (req), title?, tags?}` | `{job_id}` | `wiki.Service.Ingest` (owner from identity) |
| `status` | `{job_id (req)}` | `{status, received_at, started_at?, finished_at?, error?, subjects?}` (`subjects` are paths) | `wiki.Service.JobStatus` |
| `ask` | `{question (req)}` | `{found, answer, citations:[{path,title}]}` | `ask.Asker.Ask` (read-only) |
| `subjects` | `{type?, q?, limit?}` | `[{path, type, name, has_page}]` | registry list, filtered by type + name-substring |
| `claims` | `{subject (req)}` — a path | `[{id, text, job}]` | resolve path → claims-by-subject |
| `page` | `{subject (req)}` — a path | `{subject, title, body}` (`subject` is the path; `body` carries the D12 link footer) or not-found | resolve path → page-with-links (D12) |
| `health` | `{}` | `appkit.Envelope(version, service, details)` + identity | chassis health |
| `reflection` | `{event_type?}` | `{publishes:[], subscribes:[]}` | empty (no event plane) |

- **`reflection` is empty for free** — the Spec's event fields are all empty (D2), so the tool returns `{publishes:[], subscribes:[]}`; still declared (copy crm/notify).
- **`health`** renders the shared envelope; the ungated chassis `/health` endpoint is separate and appkit-owned.
- **Not-found is a result, not a crash** — `status` for an unknown job, `page`/`claims` for an unmatched subject path return a clean not-found result, never a 500. Because the path token is the `UNIQUE` `norm_name` (D11), a path names at most one subject — there is no ambiguous case to surface.
- **Paths in, paths out** — `page`/`claims` resolve the `subject` input through `GetByPath` (D11); `subjects`, `status` (its produced-subjects list), the `page` result's own `subject`, and `ask` citations all carry `type/slug` paths. The internal ULID never crosses the MCP boundary.
- **The `page` body carries the link footer** — the handler returns `RenderFooter(body, mentions, mentionedBy)` from `Service.PageWithLinks` (D12); the footer's hrefs are `type/slug` paths.
- **Owner attribution** — `ingest` stamps `owner` from `X-Owner-Email` (`appkit.IdentityFrom(ctx)`); phase-1 is single-owner (attribution, not isolation).
- **Tool verbs are unprefixed** (`toolPrefix=""`, like crm); the dashboard/nginx brand them `ikigenba_wiki_*`. After deploy, restart the dashboard so it re-reads the manifest.
- **Each `inputSchema` is valid JSON Schema.** A tool with no required properties **omits** the `required` key entirely — it must never emit `"required": null`. A null `required` is invalid JSON Schema, and strict MCP clients (Claude's) reject the *entire* `tools/list` for it ("tools fetch failed"), even though every tool name and the rest of each schema are correct. `subjects` is the one verb with no required field, so it is the trigger; the shared object-schema helper must drop `required` when the slice is empty rather than serialize a nil slice to null.

**Rejected.**
- **A public `search` verb** — product scope lists only ingest / status / ask / inspect (subjects, claims, page) / health / reflection; this release ships no keyword/body search at all (D8 — deferred to a later release), so there is nothing to expose. Out of scope.
- **`ingest_url` / `ingest_text` split** (wiki.bak) — phase 1 ingests text only; one `ingest` verb. Dropped.
- **Auto-generated tool schemas** — the suite hand-codes MCP tool maps (crm pattern) for a stable, reviewable surface; kept.

**Verification.**
- R-MUQ4-K1JS — `tools/list` returns exactly the fifteen verbs `{ingest, status, ask, subjects, claims, page, health, reflection, jobs, jobs_count, abort, rerun, llm_calls, merge, merges}` with their input schemas — no more, no fewer. (The five control/footprint verbs are owned by D16 and the two merge verbs by D27; this id asserts only the exact membership of the list.)
- R-MVY0-XTAH — `reflection` returns `{publishes:[], subscribes:[]}` (the empty phase-1 reflection).
- R-MX5X-BL16 — `health` renders the envelope reporting the service up, with version and service name, behind identity.
- R-MYDT-PCRV — `status` for an unknown job, and `page` / `claims` for an unmatched subject path, return a clean not-found result, not an error or crash.
- R-MZLQ-34IK — `POST /mcp` is mounted behind `RequireIdentity`, and `ingest` attributes `owner` from `X-Owner-Email`.
- R-N4KO-2WTZ — every tool's `inputSchema` is well-formed JSON Schema: `type: "object"`, and `required` is either a non-empty array of property names or **absent** — never `null`. In particular the `subjects` schema (no required field) omits `required` entirely, so a strict client accepts the whole `tools/list`.
- R-01OQ-Y5YV — `subjects` returns each row's `type/slug` path and no internal id (every entry carries `path`, none carries a ULID).
- R-02WN-BXPK — `page` and `claims` accept a `type/slug` path as `subject`, resolve it via `GetByPath` to the right subject, and return a clean not-found result for a path that matches no subject.
- R-044J-PPG9 — `status` reports a finished job's produced subjects as `type/slug` paths, not internal ids.
- R-03GW-PX5K — the `page` **result** is exactly `{subject, title, body}` — `subject` is the requested `type/slug` path, `body` carries the D12 link footer — and it contains **no internal id** (neither the page ULID nor the subject ULID appears in the serialized result).
- R-04HB-QM7T — the `claims` **result** is a list of `{id, text, job}` (the claim id and the job id are the only identifiers; both are public handles, not subject identity) and contains **no `subject_id`/subject ULID** nor any other internal subject identifier. (The `claims` table carries no timestamp, so the result has no `created_at`.)
