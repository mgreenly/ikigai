# Decision 3 — The phase-1 data model

**Decision.** Four core tables — `jobs`, `subjects`, `claims`, `pages` — replacing wiki.bak's ten. (A fifth table, `llm_calls`, is added by **D13** as the LLM-call footprint; it is owned there, not here.) Dropped: `aliases` (returns in **D25** as the manual-merge forward-routing record), `runs` (folded into `jobs`), `dup_flags`, `stale_notes`, `page_vectors`, `asks`. (An early migration created a `pages_fts` FTS5 table; it is removed by a later drop migration now that keyword retrieval is deferred — see **D8**.) Suite style: relationships are documented in comments with **no `FOREIGN KEY`**; `UNIQUE` and `CHECK` are the only DB-enforced constraints. Subject **identity is exact normalized-name match, name-only** — `UNIQUE(norm_name)`; `type` is mutable metadata, **first-writer-wins** (set when the subject is minted, not overwritten by later ingests).

```sql
-- jobs — the ingest work-item AND its execution status (collapses inbox+runs).
-- work-item half (id..received_at) is PERMANENT; execution half (status..usage) mutates.
CREATE TABLE jobs (
  id          TEXT PRIMARY KEY,                -- ULID; the ingest handle returned to the caller
  owner       TEXT NOT NULL,                   -- X-Owner-Email (attribution)
  source_text TEXT NOT NULL,                   -- raw ingested text — PERMANENT, never cleared
  title       TEXT NOT NULL DEFAULT '',        -- caller-supplied, optional
  tags        TEXT NOT NULL DEFAULT '[]',      -- JSON array, caller-supplied
  sha256      TEXT NOT NULL,                   -- content identity of source_text
  received_at INTEGER NOT NULL,                -- epoch ms
  status      TEXT NOT NULL DEFAULT 'pending', -- 'pending'|'working'|'done'|'failed'
  started_at  INTEGER,                         -- epoch ms; null until claimed
  finished_at INTEGER,                         -- epoch ms; null until terminal
  error       TEXT NOT NULL DEFAULT '',        -- detail when status='failed'
  usage       TEXT,                            -- token/cost JSON; nullable
  CHECK (status IN ('pending','working','done','failed'))
);
CREATE INDEX jobs_status ON jobs (status);     -- pending scan

-- subjects — the registry; identity = exact normalized-name match (no aliasing in phase 1)
CREATE TABLE subjects (
  id          TEXT PRIMARY KEY,                -- ULID
  type        TEXT NOT NULL,                   -- 'entity'|'event'|'concept' (closed set); metadata, first-writer-wins
  kind        TEXT NOT NULL DEFAULT '',        -- freeform prompt-anchored subtype (metadata)
  name        TEXT NOT NULL,                   -- canonical display name as extracted
  norm_name   TEXT NOT NULL,                   -- normalize(name) — see below
  occurred_at TEXT,                            -- nullable ISO-8601 prefix; type='event' only; first-writer-wins
  created_by  TEXT NOT NULL,                   -- → jobs.id that first minted this subject
  created_at  INTEGER NOT NULL,
  UNIQUE (norm_name),                          -- identity key: one normalized name → one subject → one page
  CHECK (type IN ('entity','event','concept'))
);

-- claims — the durable, queryable extracted claims (NEW in phase 1; wiki.bak kept these only in-run)
CREATE TABLE claims (
  id         TEXT PRIMARY KEY,                 -- ULID
  subject    TEXT NOT NULL,                    -- → subjects.id
  job        TEXT NOT NULL,                    -- → jobs.id; provenance AND the citation ref
  text       TEXT NOT NULL,                    -- self-contained prose claim, no pronouns
  created_at INTEGER NOT NULL
);
CREATE INDEX claims_subject ON claims (subject);  -- fold-by-subject + inspect
CREATE INDEX claims_job     ON claims (job);

-- pages — one lossy page per subject; inline [job-id] citations; hard 12k cap
CREATE TABLE pages (
  subject    TEXT PRIMARY KEY,                 -- = subjects.id
  title      TEXT NOT NULL DEFAULT '',         -- denormalized for listing/inspection
  body       TEXT NOT NULL,                    -- markdown: frontmatter + prose + inline [job-id] cites
  version    INTEGER NOT NULL DEFAULT 0,       -- plain monotonic counter (NOT an optimistic CAS guard)
  updated_at INTEGER NOT NULL,
  CHECK (length(body) <= 12000)                -- hard backstop for the 12,000-char product cap
);
-- (no pages_fts: the legacy FTS5 table is dropped by a later migration — keyword retrieval deferred, D8)
```

**`normalize(name) string`** — a pure, deterministic function in `internal/wiki`: **NFKC → casefold (lower) → trim → collapse internal whitespace → strip diacritics** (NFD, drop combining marks, recompose). Same pipeline wiki.bak used. Example: `"  Café "` and `"cafe"` produce the same key.

**Citation model.** A claim's `job` is its provenance and its citation ref; the compiled page body carries inline `[job-id]` citations as in-page provenance. (`ask` cites whole *pages* by subject + title, not job ids — D9; it no longer surfaces a `sources` list.) "View a subject's raw claims" = `claims WHERE subject=?`; "the raw text behind a claim" = `jobs.source_text WHERE id = claim.job`.

**Rejected.**
- **`UNIQUE(type, norm_name)`** (research §3 / the handoff) — binds identity to the fuzzy, wobble-prone `type` label, so a type flip between ingests spuriously forks a subject and orphans its page (research §5's failure mode). Rejected for name-only.
- **Optimistic-commit version guard + conflict-retry** (`runs.conflicts`, CAS on `pages.version`, `dup_flags`, `stale_notes`) — pointless under single-writer SQLite (`SetMaxOpenConns(1)`) with a serialized worker; there is exactly one writer. `version` is kept only as a plain counter. (Answers research §6 Q7: simplified out.)
- **Keeping claims as ephemeral in-run state** (wiki.bak's `Manifest`) — phase-1 product promises permanent, inspectable claims, so they are a first-class durable table.
- **`page_vectors` now** — the vector lane is deferred; it lands as a later migration when retrieval returns as its own decision (D8), blocked by nothing here.
- **Per-claim offsets/confidence** (research §5 residual-trap mitigation) — noted as a known gap / fast-follow, out of scope for the thin slice; `claims` stays minimal.

**Verification.**
- R-7SNG-0G9A — Inserting a second `subjects` row whose `norm_name` equals an existing row's is rejected by the `UNIQUE(norm_name)` constraint (one normalized name maps to at most one subject).
- R-7TVC-E7ZZ — `normalize` is deterministic and applies the full pipeline (NFKC, casefold, trim, collapse internal whitespace, strip diacritics): e.g. `normalize("  Café ") == normalize("cafe")`, and distinct names normalize distinctly.
- R-7V38-RZQO — A `pages` write with `body` longer than 12,000 characters is rejected by the `CHECK (length(body) <= 12000)` constraint (the hard cap backstop).
- R-7WB5-5RHD — A `subjects` write whose `type` is outside `{entity,event,concept}` is rejected by the `CHECK` constraint.
