# Decision 18 — Output-token budget & honest truncation handling

**Decision.** The wiki never set an output-token budget on its LLM calls, so every generation ran at the agentkit Anthropic adapter's default ceiling (4096). A large extract overruns that ceiling, the provider stops mid-JSON, `ExtractJSON` (D5) carves an incomplete value, and the job dies with the opaque `unexpected end of JSON input` — after burning every `MaxParseRetries` re-prompt, each of which truncates identically. This decision (1) gives each call site an explicit, generous output budget, (2) makes the seam **detect truncation** and surface a **distinct, recorded reason**, and (3) makes truncation **terminal for the call** so no retry is wasted re-truncating. It realizes the product promise "A failed ingest tells you why, plainly." This extends the D5 seam (`CallSite` gains one field; faithful-application now covers it) and the D6/D7 call-site defaults; the recorder is D13, the atomic no-partial-write failure path is D4 (R-MB7F).

## (a) `MaxTokens` on the call site, faithfully applied

`llm.CallSite` (D5) grows one field, mirroring how D13 added `Stage`:

```go
type CallSite struct {
    // … Model, Temperature, Reasoning, System, MaxParseRetries, Stage (D5/D13) …
    MaxTokens int // output-token ceiling for this stage; > 0 in production. 0 ⇒ adapter default (the latent bug).
}
```

`JSON[T]` / `Converse` (D5) set it on the Conversation's generation settings — `GenSettings{… MaxTokens: site.MaxTokens}` — so the configured ceiling actually reaches the provider instead of the silent 4096. The D5 "faithful-application" guarantee now covers `MaxTokens` alongside Model/System/Temperature/Reasoning. A production JSON call site **never leaves `MaxTokens` at 0**; the default call sites below set it explicitly so the wiring cannot silently fall back to the adapter default.

## (b) Truncation detected at the single seam, with a recorded reason

Detection lives in `sendText` (D13's single round-trip seam), where the reported usage is in hand. After draining the stream, when the provider's reported **output-token usage meets or exceeds `site.MaxTokens`** (and `MaxTokens > 0`), the response is a **truncation**, not a normal reply:

- `sendText` returns a distinct sentinel error (`llm.ErrTruncated`, wrapping the stage/usage) instead of the (truncated) text.
- The D13 `CallRecord` for that round-trip is still written, with the **truncated `Response` retained** and `Err` set to the truncation reason (e.g. `output truncated at max tokens (N)`) — so the footprint shows exactly what happened, distinguishable from both a clean success (empty `Err`) and a transport error (empty `Response`, R-VRFP).

Anthropic stops generation exactly at `max_tokens` when it truncates, so output-usage-at-ceiling is a reliable signal; extract/compile disable reasoning (D6/D7), so output tokens are the visible tokens with no reasoning confound. When the provider reports **no** usage at all, detection cannot fire and the call falls through to the existing parse/retry path unchanged — no regression, just no upgraded message in that rare case.

## (c) Truncation is terminal — no futile re-prompt

`JSON[T]`'s retry loop treats `ErrTruncated` as **non-retriable**: it is returned immediately without consuming a `MaxParseRetries` iteration, because a re-prompt under the same ceiling truncates identically. (A normal unmarshal/validate failure still retries as before, D5 R-JCEE.) The error propagates to integrate (D14), whose failure path ends the job `failed` with this reason and writes **zero** subjects/claims/pages (D4 R-MB7F) — the "nothing half-built" half of the product promise.

## (d) The budgets

Each stage owns its ceiling in its `DefaultCallSite` (D6 extract, D7 compile), generously above realistic output and tunable via `ManifestExtras`:

- **extract** — `MaxTokens` sized well above the largest realistic per-document extraction (the truncating sources were ~4–5 KB → ~4.5–5 K output tokens; the budget sits comfortably above that, e.g. **16384**). The contract is "generously above realistic extract output," not the exact integer.
- **compile** — `MaxTokens` sized above the 12,000-char page cap (a product constant; ~3–4 K tokens) plus structural overhead, e.g. **8192**.

These are headroom, not a guarantee that any document fits: a pathologically large source can still exceed the budget, and when it does it now fails **honestly** (b/c) instead of cryptically. Chunking oversized sources to remove the ceiling entirely remains a deferred follow-on.

**Rejected.**
- **Just raising the budget without detecting truncation.** A bigger ceiling pushes the failure out but never removes it; the next-larger document still dies as `unexpected end of JSON input` with three wasted retries. Detection + a clear reason is the durable fix. Rejected (insufficient alone).
- **Inferring truncation from the parse error text** (`unexpected end of JSON input`). Brittle — couples wiki to the encoding/json error wording and confuses a genuinely malformed-but-complete reply with a truncated one. Usage-at-ceiling is a precise, provider-grounded signal. Rejected.
- **Retrying a truncation with a larger budget automatically.** Doubles cost and latency on the request path and still has a ceiling; for batch ingest, failing fast with a recorded reason the operator acts on (re-ingest smaller, or raise the manifest knob) is simpler and cheaper. Rejected (deferred with chunking).
- **Reading the finish reason off the stream.** agentkit's `Stream` exposes `Usage()`/`Err()`/`Warnings()` but not the round-trip `FinishReason` (it lives on the internal `RoundTrip`), so the seam cannot read `FinishMaxTokens` directly; output-usage-vs-ceiling is the available, equivalent signal. Noted, not a true rejection.

**Verification.**
- R-MSKH-GPX5 — `JSON`/`Converse` applies the call site's `MaxTokens` to the Conversation's generation settings (a capturing mock provider confirms the configured ceiling reaches the provider), instead of leaving it unset for the adapter default.
- R-MTSD-UHNU — When the provider reports output usage at/above the call site's `MaxTokens` with a truncated body, `JSON` returns a distinct truncation error (not a generic JSON-parse error), and the call's `CallRecord` retains the truncated `Response` and carries the truncation reason in `Err`.
- R-MV0A-89EJ — A truncation is terminal for the call: `JSON` returns immediately without re-prompting, even with `MaxParseRetries > 0` (exactly one round-trip is made), distinguishing it from a retriable unmarshal/validate failure.
- R-MW86-M158 — `extract.DefaultCallSite` and `compile.DefaultCallSite` each carry a non-zero `MaxTokens` generously above their realistic output, and the extractor/compiler the composition root builds are constructed from them — so the production stages never run at the 4096 adapter default.
