# Decision 7 — The compile stage (`internal/compile`): full recompile from claims, 12k cap enforced

**Decision.** Compile rebuilds one subject's page from its **complete claim set** — identity + claims only, **never a prior page body, never the source document**. This is the full-recompile-from-claims path that structurally immunizes against generational drift (research §5) and satisfies the D4 anti-poisoning invariant. It deletes wiki.bak's incremental-merge, superseded/citation-preservation, and stale-note machinery.

```go
package compile

const PageCharCap = 12000 // product constant (D3); characters = Unicode code points (utf8.RuneCountInString ↔ SQLite length())

type Compiler struct { c *llm.Client; site llm.CallSite; maxTighten int; log *slog.Logger }
func New(c *llm.Client, site llm.CallSite, log *slog.Logger) *Compiler

// Compile rebuilds ONE subject's page from its COMPLETE claim set — identity +
// claims only, never a prior page body, never the source document. Guarantees a
// body of ≤ PageCharCap characters.
func (c *Compiler) Compile(ctx context.Context, s wiki.Subject, claims []wiki.Claim) (title, body string, err error)
```

Each ingest that touches a subject recompiles its whole page from all its claims (this job's + every prior job's), O(claims) per affected subject — fine for a personal wiki. No prior-page-body parameter exists, so the broken-telephone loop is impossible by construction.

**Prompt** (port wiki.bak's `DefaultMergePrompt`, simplified to phase 1):
1. **Task framing** — given ONE subject's identity + its complete claim set (each claim tagged with its `[job-id]` citation); rebuild the page from these claims only; never see prior pages or source documents.
2. **Fold / prose discipline** — flowing prose, never a bullet ledger; corroborating claims attach their citation rather than repeat; on contradiction keep **both** statements with citations in a marked "Conflicting accounts" section, never silently overwrite; every factual sentence carries its inline `[job-id]` citation.
3. **Lead discipline** — the first paragraph states what the subject IS (identity card: canonical name, type).
4. **Size cap** — the page **must not exceed 12,000 characters**; when claims overflow, compress lower-salience detail rather than truncate mid-thought; keep the lead and citation coverage.
5. **Output schema** — `{title, body}` (single page; no `superseded`, no `stale_notes`, no multi-page write-set).
6. **Worked example.**

**12k enforcement — three layers (closes the wiki.bak gap):**
- **Prompt instruction** (§4 above).
- **Go enforcement in `Compile`:** the D5 `validate` hook asserts title/body non-empty; then `Compile` measures `utf8.RuneCountInString(body)`. If over cap → **recompile-tighter**: re-prompt with a corrective note ("page is N chars; hard limit 12000 — compress lower-salience claims, keep the lead and all citations") up to `maxTighten` times. If still over → **last resort: truncate at a paragraph/rune boundary ≤ 12000 and log a warning** (the page is lossy by contract; truncation is acceptable loss, never corruption of the durable claims/raw record). `Compile` thus never returns a body > 12,000 characters.
- **DB `CHECK(length(body) <= 12000)`** (D3) — the backstop that turns any escaped bug into a loud tx abort, not silent corruption.

**Call site** — the package owns its designed defaults and exposes them so the wiring cannot silently drop them: `compile.DefaultCallSite(model string) llm.CallSite` returns `{Model: model, Temperature: ptr(0), Reasoning: DisableReasoning(), MaxTokens: <budget above the 12k page cap, D18>}` (`maxTighten` is a separate `New` parameter, default 2; further configurable via `ManifestExtras`). The non-zero `MaxTokens` (D18) keeps a claim-heavy compile off the 4096 adapter default. **The composition root constructs the compiler from `DefaultCallSite`, never by hand-building a bare `llm.CallSite{Model: …}`** — the original drift was that bare construction, which ran compile at provider-default temperature with reasoning **on** rather than the designed deterministic, reasoning-off configuration. (Compile sets no `MaxParseRetries`; the robust `ExtractJSON` carve in D5 covers its parse path.)

**Rejected.**
- **Incremental merge into the prior page body** (wiki.bak) — the broken-telephone / generational-drift path; full recompile from claims is the immunizing choice (research §5). Rejected.
- **`superseded` citation-preservation + set-difference accounting** — under full recompile, citations are a pure function of the current claim set, so nothing is carried across versions to preserve. Dropped (answers research §6 Q7).
- **`stale_notes` neighbor flagging** — lint machinery, deferred.
- **Multi-page batch compile** — bought cross-page coherence, but phase 1 has no inter-page links; per-subject compile is simpler and matches the D4 stage factoring. Rejected.
- **Failing the job when the cap can't be met** — would make a claim-heavy subject permanently un-ingestable; truncating the lossy page (logged) is the better failure mode. Rejected.

**Verification.**
- R-FQLB-QWS6 — `Compile` is invoked with subject identity + claims only and has no prior-page-body parameter (full recompile from claims; anti-poisoning).
- R-FT14-IG9K — `Compile` never returns a body exceeding 12,000 characters under any input.
- R-FU90-W809 — Given an over-cap candidate followed by a within-cap candidate, `Compile` returns the within-cap candidate (recompile-tighter loop).
- R-FVGX-9ZQY — Given only over-cap candidates, `Compile` truncates the final candidate to ≤ 12,000 characters at a boundary and logs a warning (last-resort path).
- R-FWOT-NRHN — The compile call site uses `Temperature: 0` and `DisableReasoning()`.
- R-4DS4-RXYX — `compile.DefaultCallSite` carries `Temperature: 0` and `DisableReasoning()`, and the compiler the composition root builds is constructed from it — so the running compile stage is the designed deterministic, reasoning-off configuration rather than provider defaults.
