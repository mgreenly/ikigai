# Decision 8 — No retrieval lane this release: keyword search removed, hybrid deferred

**Decision.** This release ships **no retrieval lane**. `ask` (D9) finds pages by **subject extraction + exact-normalized resolution**, not by searching page bodies, so the FTS5 keyword machinery has no consumer and is **removed**. Keyword and vector/hybrid retrieval return as their own decision in the later release that actually ships search — built against whatever retrieval shape we want then, not constrained by a dormant seam carried forward.

Concretely, removed from the service:

- **`internal/retrieve`** — the `Hit`/`Retriever`/`Service`/`keywordRetriever` seam, `ftsPhrase`, the bm25 keyword lane, and the registry-first pin. Deleted; nothing imports it.
- **`PageStore.Search`** (the FTS-backed page search method on the page store) — deleted with the table it queries.
- **`pages_fts`** — the external-content FTS5 table. Because committed migrations are immutable (project rule: never edit or delete a committed migration), the table is removed by a **new forward migration** (`bin/new-migration wiki drop_pages_fts`) issuing `DROP TABLE pages_fts;`. The original `CREATE VIRTUAL TABLE pages_fts` migration stays frozen on disk; the drop migration is the forward correction.
- **The in-tx FTS sync** in the ingest commit path (D4) — the explicit `pages_fts` delete-by-OLD-value-then-insert dance. Removed: the commit tx no longer touches any FTS table, eliminating the external-content sync trap that now guards nothing.

Knock-on doc edits (kept coherent, owned by their home decisions):
- **D3** (data model) — schema is now four tables (`jobs`, `subjects`, `claims`, `pages`); `pages_fts` is dropped by the new migration.
- **D4** (ingest) — the commit step no longer performs a `pages_fts` sync.
- **D2** (composition root) — the `internal/retrieve` package and the `search default/cap` config knobs (`SearchLimits`) leave the layout and `ManifestExtras`.
- **D10** (MCP surface) — still no public `search` verb; the rationale is now "this release has no keyword search at all," not "it's internal to ask."

**Rejected.**
- **Keep the seam dormant (retrieve + pages_fts + sync in place, unconsumed)** — ships tested-but-unreachable code and keeps the external-content sync trap live in D4's tx for no current benefit. Violates "pay complexity only when forced / delete unused code rather than carry a shim." Rejected.
- **Expose a public `search` MCP verb to give keyword search a consumer** — re-adds scope the product just deferred to a later release. Out of bounds here. Rejected.
- **Delete Decision 8 entirely** — would leave the `DROP TABLE` migration and the D4 commit change unexplained. Repurposing D8 as the explicit removal record keeps a single coherent place for the deferral rationale and the drop migration.

- R-PIGW-C9EM — An ingest integrates and commits end-to-end against the FTS-free schema with **no reference to any `pages_fts` table** in the commit path (the page upsert and `done` transition succeed with no FTS sync step).
