# Decision 11 — Subject addressing: the public path == identity (`type/norm_name`)

**Decision.** Every subject has one public identifier — a `type/<norm_name>` **path** (e.g. `entity/acme-corp`) — and the opaque ULID `subjects.id` never leaves the service. The path token **is** the subject's `norm_name` (D3): the canonical `[a-z0-9-]` form is already path-safe, so the path is just the type joined to it with a `/`, with **no further transform**. The path is **derived, not stored** (a pure function of `type` + `norm_name`), so it adds no column, no migration, and no write-path change. Resolution back to a subject is a **direct, indexed** lookup.

(Terminology: elsewhere in the design this identifier is called the `type/slug` path. The "slug" is **exactly** `norm_name` — there is no separate slug transform — so `type/slug` and `type/norm_name` name the same string; this Decision owns its shape.)

This is a thin seam in `internal/wiki` consumed by D12 (the link footer) and D10 (the MCP surface).

```go
// Path is the public identifier for a subject: "<type>/<norm_name>".
// norm_name is already the canonical [a-z0-9-] path token (D3 Normalize:
// NFKC + lowercase + diacritic-strip + non-[a-z0-9]→'-' + collapse + trim),
// so Path applies NO further transform — there is no separate slug step.
func Path(s Subject) string             // s.Type + "/" + s.NormName

// GetByPath resolves a public path back to its subject by direct match:
// split into (type, norm_name), look up the row whose norm_name == token
// (norm_name is globally UNIQUE — at most one), then require its Type == type.
//   - empty token or no row → (Subject{}, ErrSubjectNotFound)
//   - row found but wrong type → (Subject{}, ErrSubjectNotFound)
//   - exactly the matching subject → (subject, nil)
// Exact only — no fuzzy/alias matching (D3 identity). Because norm_name is
// UNIQUE, the path token names at most one subject; there is no ambiguous case.
func (s *SubjectStore) GetByPath(ctx context.Context, path string) (Subject, error)

var ErrSubjectNotFound = errors.New("wiki: subject not found")
```

**Why derived, not stored.** `norm_name` is the subject's identity (D3, `UNIQUE(norm_name)`) and never changes once minted, so a path built from it is **stable** without being persisted. Storing a `slug`/path column would add a migration, a write-path step, and a second source of truth that could drift from `norm_name` — all for no gain, since the path token *is* `norm_name`. The lookup is a single indexed equality on the `UNIQUE(norm_name)` key — `O(log n)`, no scan and no per-row recomputation.

**No ambiguity is possible.** Because the path token is exactly `norm_name` and `UNIQUE(norm_name)` admits one subject per token, a path resolves to at most one subject. The earlier model derived the path through a lossy `slug()` (collapsing spaces/`/`), which could map two distinct `norm_name`s to one path and so needed an `ErrAmbiguousPath` guard; that collapse — and the error — are gone. The only miss is `ErrSubjectNotFound` (unknown token, empty token, or right token under the wrong type).

**Scope of the path surface.** The path is the subject identifier in every outward-facing place — `subjects` listing, `page`/`claims` lookup inputs, the `status` produced-subjects list (D10), `ask` citations (D9), the D12 link footer, and the `merge`/`merges` paths (D27). Job ids and claim ids are separate handles and are out of scope: a job id is the public ingest handle (`ingest` returns it, `status` consumes it), and is not a subject identifier.

**Rejected.**
- **A stored path/`slug` column with `UNIQUE(type, slug)` minted at subject creation** — adds a migration, a write-path step, and collision-suffixing machinery (`acme-corp-2`); produces a *second* identity that can drift from `norm_name`. The derived path needs none of it and is stable because `norm_name` is. Rejected.
- **A separate lossy `slug()` transform over `norm_name` (the prior shape).** Rejected: once `norm_name` is already the canonical `[a-z0-9-]` token (D3), a second transform is redundant, makes the path no longer round-trip to a single identity, forces an `O(N)` scan-and-compare in `GetByPath`, and reintroduces `ErrAmbiguousPath` from punctuation collapse.
- **Reverse-engineering the path token back to a name (hyphens → spaces, then normalize-match)** — lossy and ambiguous; the original `Name` is stored for display, and resolution is a direct `norm_name` match needing no inversion. Rejected.
- **Exposing the ULID as a fallback identifier** — the product forbids showing internal ids; a fallback would re-leak them. Rejected.

**Verification.**
- R-DRX6-PWSW — `Path(Subject{Type:"entity", NormName:"lorenzo-de-medici"})` returns exactly `"entity/lorenzo-de-medici"` — the type joined to `NormName` with a single `/`, with no further transformation of the token (a path built via any punctuation/space transform at this step fails). Pure function, table-tested directly.
- R-DT53-3OJL — against a real temp SQLite (per the Testing strategy: path resolution runs on the real engine), for a subject stored with `norm_name = "lorenzo-de-medici"` and `type = "entity"`, `GetByPath(ctx, "entity/lorenzo-de-medici")` returns that subject via a direct `norm_name` equality match (not a scan); and `GetByPath(ctx, "concept/lorenzo-de-medici")` — same token, wrong type — returns `ErrSubjectNotFound`.
- R-DUCZ-HGAA — against the same real temp SQLite, `GetByPath` for a path whose token matches no stored `norm_name` returns `ErrSubjectNotFound`, and an input with an empty token (e.g. `"entity/"`) returns `ErrSubjectNotFound` rather than matching a row.
