# Decision 21 — The judge call site + scoring semantics (`internal/eval`)

**Decision.** Scoring an extract run against a blessed case (D20) is two stages: **subjects matched deterministically**, then **claims judged by meaning** by a pinned strong LLM. The judge is just another call site over the D5 `llm.JSON` seam (`Stage:"judge"`), so it is recorded (D13), retried, and truncation-guarded like every other call — and, being eval-only, it is configured by the D22 CLI rather than a service manifest knob. Scoring produces raw counts **and** the actual items (so a human can eyeball), plus subject and claim precision/recall, per case and aggregated; it renders **no pass/fail** (product: scores only).

## Stage 1 — subjects matched deterministically (no LLM)

A predicted subject and a gold subject are **the same subject** exactly when their identity `(Type, Normalize(Name))` agrees — the *same* match `ask` and page-links use. `normalize` (currently unexported in `internal/wiki`) is **exported as `wiki.Normalize(string) string`** for this reuse; eval imports it rather than re-deriving normalization, so "found" means precisely what the running wiki means by subject identity.

```go
type SubjectScore struct {
    Found        []string // type/slug of subjects in BOTH gold and prediction
    Missed       []string // type/slug present in gold, absent from prediction
    Hallucinated []string // type/slug present in prediction, absent from gold
}
```

This partition is pure and table-testable: a prediction whose name differs from gold only by case/whitespace still **matches** (normalizer reuse); a genuinely different subject lands in `Missed`/`Hallucinated`.

## Stage 2 — claims judged by meaning, one call per matched subject

For each **matched** subject pair, one judge call compares that subject's gold claims against its predicted claims and partitions gold into **covered**/**missed** and predicted into **extra**. A claim counts covered when a predicted claim **conveys the same fact**, not when strings match — that semantic call is the judge's entire job and the reason it is an LLM, not string equality.

Unmatched subjects need **no** judge call: every gold claim of a **missed** subject counts missed, every predicted claim of a **hallucinated** subject counts extra. The judge fires only where there is something to compare.

```go
// Judge wraps the pinned judge call site over the shared llm.Client.
type Judge struct { c *llm.Client; site llm.CallSite }
func NewJudge(c *llm.Client, site llm.CallSite) *Judge

// DefaultJudgeCallSite is the pinned reference yardstick: a STRONG model held fixed across runs,
// deterministic (Temperature 0). Overridable via the D22 CLI, but the default is pinned.
func DefaultJudgeCallSite() llm.CallSite
// => { Stage:"judge", Model: anthropic.ModelOpus48, Temperature: ptr(0),
//      Reasoning: agentkit.Level("low"), MaxTokens: 16384, MaxParseRetries: 2 }

func Score(ctx context.Context, j *Judge, c Case, predicted []extract.ExtractedSubject) (CaseResult, error)
```

The judge prompt numbers each subject's gold and predicted claims and asks which gold numbers are conveyed by some predicted claim and which predicted numbers are conveyed by no gold claim. The structured reply is validated through the `llm.JSON` validate hook — indices must be in range and each gold/predicted claim classified once; a malformed or out-of-range verdict is a bounded re-prompt (D5 R-JCEE), identical to extract's retry.

```go
// judgeVerdict is the per-subject structured reply (validated; indices 1-based as numbered in the prompt).
type judgeVerdict struct {
    CoveredGold    []int // gold-claim numbers a predicted claim conveys
    ExtraPredicted []int // predicted-claim numbers no gold claim conveys
}
```

## Results and aggregation

```go
type ClaimScore struct { Covered, Missed, Extra int }

type CaseResult struct {
    Case       string
    Difficulty string         // "easy" | "medium" | "hard"
    Subjects   SubjectScore
    Claims     ClaimScore     // summed over matched subjects
    // per-subject covered/missed/extra claim TEXT retained for eyeballing (omitted here for brevity)
}

type Metrics struct { Precision, Recall float64 } // subjects: found/(found+halluc), found/(found+missed)
                                                   // claims:   covered/(covered+extra), covered/(covered+missed)
type Totals struct {
    Overall       struct{ Subjects, Claims Metrics; Cases int }
    ByDifficulty  map[string]struct{ Subjects, Claims Metrics; Cases int }
}
func Aggregate(results []CaseResult) Totals
```

Precision/recall are defined for subjects (found vs hallucinated/missed) and claims (covered vs extra/missed). `Aggregate` rolls the per-case partitions into dataset-wide and per-difficulty metrics. The model/config/judge stamp that makes two runs comparable lives on the `Scorecard` the CLI assembles (D22), not here — `Score` reports what happened on one case.

**Rejected.**
- **String/normalized equality for claims instead of a judge.** Claims are free prose ("Gygax co-created D&D" vs "D&D was co-created by Gary Gygax"); string match reports endless false misses and makes every model look broken. Semantic judging is the point. Rejected.
- **One judge call per whole case (all subjects batched).** A single giant prompt whose failure is all-or-nothing, whose `CallRecord` is hard to read, and which can't be reasoned about per subject. Per-matched-subject calls keep each judgment focused and individually recorded. Rejected.
- **A non-pinned / non-deterministic judge** (provider-default temp, or letting the model-under-test double as judge). The yardstick must not move while the thing being measured changes; a wobbling or self-grading judge makes cross-run comparison meaningless. Pinned strong + Temperature 0. Rejected.
- **Re-deriving normalization inside eval.** A second normalizer drifts from the wiki's real subject identity, so "found" would stop meaning what `ask` means. Export and reuse `wiki.Normalize`. Rejected.

**Verification.**
- R-DRME-T4FA — Subject matching partitions predicted vs gold by `(Type, wiki.Normalize(Name))`: a prediction whose name differs from gold only by case/whitespace lands in `Found`; a gold-only subject lands in `Missed`; a prediction-only subject lands in `Hallucinated`. (Pure, no LLM.)
- R-DSUB-6W5Z — For a matched subject, `Score` issues one judge call through `llm.JSON` with the judge `CallSite` (`Stage:"judge"`), and a scripted mock judge verdict marks a differently-worded predicted claim as **covering** its gold claim — proving claims are scored by the judge's meaning verdict, not string equality.
- R-DU27-KNWO — No judge call is made for unmatched subjects: a **missed** subject contributes all its gold claims to `Claims.Missed` and a **hallucinated** subject contributes all its predicted claims to `Claims.Extra`, with the mock judge recording zero invocations for those subjects.
- R-DVA3-YFND — A judge reply with an out-of-range or malformed claim index is rejected by the validate hook and bounded-re-prompted (a bad-then-good verdict succeeds; an always-bad verdict errors rather than miscounting), reusing the D5 retry contract.
- R-DWI0-C7E2 — `DefaultJudgeCallSite()` is the pinned yardstick: `Model == anthropic.ModelOpus48`, `Temperature == 0`, `MaxTokens >= 16384`, `Stage == "judge"` — a strong, deterministic judge held fixed independent of the model under test.
- R-DXPW-PZ4R — `Aggregate` computes subject and claim precision/recall over a set of `CaseResult`s both overall and grouped by difficulty, with the counts matching the summed per-case partitions (e.g. claim recall == covered/(covered+missed)).
