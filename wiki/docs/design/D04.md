# Decision 4 — The ingest pipeline and worker

**Decision.** Ingest is fire-and-return at the front door and integrated by a **single** background worker goroutine. wiki.bak's multi-integrator pool, cron selection, optimistic-commit conflict loops, and re-merge/re-resolve machinery are all dropped.

**Front door — `wiki.Service.Ingest(ctx, owner, text, title, tags) (jobID string, err error)`.** Computes `sha256(text)`, `INSERT`s one `jobs` row (`status='pending'`, `source_text=text`, `received_at=now`), nudges the worker, and returns the job id — the handle. No extraction on the request path: a single INSERT, never an LLM call. (wiki.bak returned a receipt-with-dup-flag; phase 1 returns the job id directly.)

**Status — `wiki.Service.JobStatus(ctx, jobID)`** reads `jobs` by id → `{status, received_at, started_at?, finished_at?, error?}` plus the subject ids the job produced (so a finished ingest is inspectable).

**Worker — one goroutine, concurrency 1, no knob.** A `Spec.Workers` entry running `worker.Run(ctx, svc)`. Loop: claim the oldest `pending` job → integrate → commit/fail → loop; when none pending, wait on a `sync.Cond` doorbell (`Nudge` after ingest) with a poll-timeout fallback; exit only on `ctx.Done()`. A single worker means compile always sees a stable, complete claim set for a subject — removing the contention that wiki.bak's conflict loops existed to handle. Multi-worker (with per-subject serialization) is a deferred follow-on; phase 1 is hard-1 with no concurrency knob (cannot be set into a footgun).

**Claim / commit protocol (crash-safe; the DB is the truth):**
1. **Claim** (short tx): `UPDATE jobs SET status='working', started_at=now WHERE id=<oldest pending>` — a DB-visible `working` state.
2. **Integrate** (NO db tx held; LLM calls would block the single writer):
   - `extract(ctx, job.source_text)` → `[]ExtractedSubject{type, kind, name, occurred_at, claims[]}`.
   - resolve each subject by `normalize(name)` against `subjects` (read-only) → existing id or freshly minted id.
   - for each affected subject, gather **its full persisted claim set + this job's new claims**, then `compile(ctx, subject, claims)` → `(title, body)`.
3. **Commit** (one tx, all-or-nothing): upsert subjects (first-writer-wins `type`/`occurred_at`), insert this job's claims, upsert pages (`version++`), `UPDATE jobs SET status='done', finished_at=now`. (No `pages_fts` sync — keyword retrieval is removed, D8.)
4. **Failure:** any error in step 2/3 → tx `UPDATE jobs SET status='failed', error=<msg>`, writing no subjects/claims/pages. No job-level retry loop — agentkit retries transient LLM faults internally; a post-retry failure is terminal (re-ingest to retry). Dead-letter/backoff deferred.
5. **Boot sweep** (worker startup): `UPDATE jobs SET status='pending' WHERE status='working'` — requeues jobs orphaned by a crash; safe because integration writes nothing until the atomic step-3 commit, so a re-run starts clean.

**Stage factoring (enables the deferred `rebuild`).** `extract` and `compile` are independent, DB-free units — `extract(ctx, text)` and `compile(ctx, subject, claims)`. The worker orchestrates them; a future recompile calls `compile` over a subject's existing claims with no `extract`. Stage internals (prompts, json-mode, 12k enforcement) are D5/D6.

**Anti-poisoning invariant (stated constraint).**
> `extract` reads **only** `job.source_text`. `compile` reads **only** the subject's persisted `claims` — never a prior page, never the raw document re-extracted. The integrate orchestration never passes page text into `compile`.

Structurally enforced by the stage signatures: `compile(ctx, subject, claims)` has no page-body parameter.

**Rejected.**
- **Worker pool with concurrency knob** — concurrent workers compile from stale claim sets when they touch the same subject; the contention is exactly what wiki.bak's optimistic-commit/conflict-loop code handled. A single worker deletes that whole problem. Rejected for phase 1.
- **wiki.bak's RAM-only in-flight set + commit-only stamping** — an orphan-avoidance trick for a multi-worker pool; with a single worker the simpler DB-visible `working` + boot-sweep-requeue is equivalent and gives a queryable working state. Rejected.
- **Job-level retry/backoff/dead-letter** — agentkit already retries transient LLM faults; layering job retries on top adds machinery for no phase-1 benefit. Rejected (deferred).
- **Holding a DB tx across the LLM calls** — would serialize all DB access behind a multi-second network call under single-writer SQLite. Rejected; only the short claim and the final commit hold a tx.

**Verification.**
- R-M8RN-87WV — `Ingest` returns a job id produced by a single INSERT, with no LLM/extract call on the request path (fire-and-return).
- R-M9ZJ-LZNK — A `pending` job is claimed by the worker and advances `pending → working → done`, producing its subjects, claims, and page.
- R-MB7F-ZRE9 — When integration errors mid-run, the job ends `failed` with a non-empty `error` and **zero** subjects/claims/pages are written (the commit is atomic).
- R-MCFC-DJ4Y — A second job naming an existing subject (same `normalize(name)`) appends its claims and bumps that subject's page `version`, creating no duplicate subject (knowledge compounds per subject).
- R-MDN8-RAVN — On worker startup, jobs left in `working` (crash orphans) are reset to `pending` and reprocessed.
- R-MG31-IUD1 — `compile` is invoked with the subject's claim set and never receives page-body input (the anti-poisoning invariant, asserted at the integrate orchestration seam).
