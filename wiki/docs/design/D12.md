# Decision 12 — Page links: read-time mention detection + markdown footer

**Decision.** A page's links are **computed when the page is read, not stored**. There is no `page_links` table, no migration, and no change to the ingest write path (D4). When the `page` tool returns a page, the service projects its links from the *current* subjects, the *current* merge aliases (D25), and *current* page bodies, and a pure renderer appends a markdown footer to the returned body. Detection is **exact normalized match** against a subject's identity key **or any merge-registered alias key** that points at it (D25) — still the exact-normalized match of D3/D9, **nothing fuzzy** — in **both directions**. A match via an alias resolves to the **canonical** subject: a link always renders the survivor's name and `type/slug` path, never the folded-away variant's text or a dead path.

**Why read-time.** A stored index drifts on three axes: a page rewrite changes its outbound links, minting a new subject can create inbound links into already-compiled pages that mention it, *and* a merge that mints an alias can create inbound/outbound links wherever pages used the folded short name. The feature_request's write-time scan maintained only the first axis. Projecting at read time is always correct on all three, needs no migration, and leaves D4's commit path untouched — "pay complexity only when forced." A materialized `page_links` index is a noted later optimization for when per-read scanning costs too much.

```go
// SubjectKeys is a subject plus every normalized name that resolves to it: its
// own NormName ∪ the norm_name of every D25 alias whose subject_id is this
// subject. All keys are already Normalize'd (D3), so a match is a whole-run check.
type SubjectKeys struct {
    Subject
    Keys []string
}

// Mentions returns every subject in `others` (the page's own subject excluded by
// the caller) at least one of whose Keys occurs as a whole, hyphen/edge-bounded
// run in Normalize(body). A match via an alias key yields the CANONICAL subject,
// so links always render canonical. Pure and deterministic — table-tested directly.
//   match := Normalize(body) contains key bounded by non-alphanumeric runes or
//            string ends (so "cat" does not match "category"); key ∈ Keys.
func Mentions(body string, others []SubjectKeys) []Subject

// LinkedPage is a page plus its read-time link projection.
type LinkedPage struct {
    Page
    Mentions    []Ref // outbound: subjects this page's body names (by name or alias)
    MentionedBy []Ref // inbound:  subjects whose page body names this one (by name or alias)
}
// Ref is a link target: the public path (D11) + display name for link text.
type Ref struct { Path, Name string }

// PageWithLinks reads the page for subjectID and projects its links over the
// current corpus. It loads all subjects and all aliases (AliasStore.ListAll, D25),
// groups aliases by subject_id, and builds one SubjectKeys per subject:
//   outbound = Mentions(thisBody, allOtherSubjectKeys);
//   inbound  = for each other subject with a page, this subject ∈ Mentions(thatBody, [thisKeys]).
// Because thisKeys includes this subject's aliases, a page that names it only by a
// folded short name still counts in both directions, rendered as the canonical subject.
func (s *Service) PageWithLinks(ctx context.Context, subjectID string) (LinkedPage, error)

// RenderFooter appends the markdown footer to a page body and returns the
// combined text. Pure — the D10 page handler calls it to build the result body.
func RenderFooter(body string, mentions, mentionedBy []Ref) string
```

**Footer format** — appended to the returned body, outbound before inbound; a section is omitted when empty; the whole footer (including the `---` rule) is omitted when both are empty:

```markdown

---
**Mentions:** [Acme Corp](entity/acme-corp) · [John Smith](entity/john-smith)
**Mentioned by:** [Deal 2024-Q3](event/deal-2024-q3)
```

Link text is the target subject's display `Name`; the href is its `type/slug` `Path` (D11). Targets within each section are de-duplicated and ordered deterministically (by path).

**The 12k cap is untouched.** The cap (product constant, enforced by D3/D7) governs the **compiled body** — what `compile` writes and the `pages.body` `CHECK` stores. The footer is presentation appended at read time, so the returned text may exceed 12,000 chars; the stored body still may not. Nothing in this decision writes to `pages`.

**Self-exclusion.** A page never links to its own subject even when its body contains its own name; `PageWithLinks` excludes `subjectID` from the outbound candidate set and from inbound.

**One canonical subject per key (no ambiguous double-link).** While an alias `k → W` exists, ingest can never mint a *live* subject whose `norm_name` is `k`: `Resolver` is subjects-first, but the miss falls through to the alias and routes the ingest to `W` (D25), so the duplicate is never created. Therefore each matched key maps to exactly one canonical subject, and the alias/identity key sets never collide to produce two links for one surface form.

**Rejected.**
- **A `page_links` table populated in D4's commit (the feature_request approach)** — drifts on the new-subject axis (inbound links lag until recompile), adds a migration and a write-path step. Read-time projection is always correct with none of that. Rejected (materialization deferred).
- **A structured `links` field instead of a body footer** — considered; the product calls for a human-readable markdown footer on the returned page, so the footer is rendered into the body. Rejected per product.
- **Fuzzy / substring matching on non-alias variants (link on arbitrary partial or variant names)** — would diverge from D3/D9 identity and manufacture wrong links; exact normalized whole-run match only. Still **rejected**. (Merge-registered aliases are *not* this: an alias key is an exact normalized identity key the owner deliberately created, so matching it cannot manufacture a wrong link — that is why aliases are now in, while fuzzy/partial stays out.)
- **A separate alias-matching pass bolted onto `Mentions(body, []Subject)`** — rejected for folding aliases into the candidate's `Keys` (`SubjectKeys`): one symmetric matcher covers both directions and both key kinds, with no second code path that could drift from the first.
- **Counting the footer against the 12k cap** — would let added links silently shrink how much a page can say; the cap governs compiled content, the footer is render-time. Rejected.

**Verification.**
- R-ZUDC-NJIP — a returned page's footer "Mentions" line links every *other* subject whose normalized name occurs (whole-word) in the body, each as `[Name](type/slug)`, deduped and deterministically ordered.
- R-ZVL9-1B9E — the footer "Mentioned by" line links every subject whose own page body names this subject (inbound), by `type/slug` path.
- R-ZWT5-F303 — a page does not link to itself: its own subject never appears in "Mentions" or "Mentioned by" even when its name occurs in its body.
- R-ZY11-SUQS — detection is exact normalized whole-run match only: a body containing a name that is **neither an exact subject `norm_name` nor a registered alias `norm_name`** produces no link, and a substring inside a larger token (`cat` in `category`) is not a match.
- R-ZZ8Y-6MHH — when a page has no outbound and no inbound links, no footer (and no `---` rule) is appended; the returned body equals the stored body.
- R-00GU-KE86 — "Mentions" renders before "Mentioned by"; when exactly one side is empty, that section is omitted while the other still renders.
- R-1WP9-CLM9 — *(pure, table-tested)* given `SubjectKeys` for survivor `W` (canonical name "Giorgio Vasari", path `entity/giorgio-vasari`) carrying an alias key `vasari`, `Mentions("Vasari records that…", [W])` returns `W`, and its rendered `Ref` is `[Giorgio Vasari](entity/giorgio-vasari)` — the **canonical** name and path, never the alias surface text `vasari` and never a dead `entity/vasari` path.
- R-1XX5-QDCY — *(pure, table-tested)* inbound symmetry: a page body that names `W` **only** by an alias key (`vasari`, with `W`'s own `norm_name` absent from the body) still counts as mentioning `W` — `W ∈ Mentions(thatBody, [W-with-alias-keys])` — so that page appears under `W`'s "Mentioned by".
- R-1Z52-453N — *(real temp SQLite, via `PageWithLinks`)* the reported-bug regression guard: with an alias row `vasari → W` installed in a real DB and a third subject `F` whose stored page body says the bare short name "Vasari", `PageWithLinks(F)` projects an outbound link to `W` rendered canonically (`[Giorgio Vasari](entity/giorgio-vasari)`); with **no** such alias row the same body yields no link — proving the projection consults `AliasStore.ListAll` and is alias-aware end to end. *(composes with D26 R-HUDR-AWS9, which proves a real merge creates that alias row.)*
