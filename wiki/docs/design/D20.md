# Decision 20 — The extract evaluation harness (`internal/eval`): dataset + runner

**Decision.** A new reusable library `internal/eval` holds the off-line evaluation machinery for the extract call site: loading a gold dataset, and running the **real production extract** over each case. It owns no scoring (D21) and no CLI (D22) — it is the seam both of those build on, and the seam tests drive directly. Its load-the-dataset and run-the-real-extractor behavior is fully testable with **no live LLM** (the runner takes an already-constructed `*extract.Extractor`, mock-backed in tests).

## The runner reuses the production path — it never reimplements extract

```go
package eval

// Run executes PRODUCTION extract over a case under a caller-supplied Extractor.
// The Extractor is constructed exactly as cmd/wiki constructs it — extract.New(client, site)
// (D6/D19) — so prompt rendering, validation, retry, truncation handling, and call recording
// are identical to the running service. Run adds nothing of its own to the extraction.
func Run(ctx context.Context, ex *extract.Extractor, c Case) ([]extract.ExtractedSubject, error)
```

`Run` calls `ex.Extract(ctx, c.Header, c.Text)` and returns its result unchanged. There is **no second prompt, no second parser, no reimplementation** in `eval` — that identity is the whole point ("the CLI uses the same function the code uses"). What varies between runs is the `llm.CallSite` baked into `ex` (model / reasoning / temperature / budget, supplied by the D22 CLI via the D19 resolver); the case's text and header are held fixed.

## A case is one directory; a dataset is a folder of them

```
testdata/eval/extract/<case-name>/
    document.txt     ← raw source text, fed VERBATIM to extract (human-readable, diffable)
    gold.json        ← header + difficulty + the blessed expected subjects and claims
```

`testdata/` so the Go toolchain ignores it in builds while it still ships in-repo. `document.txt` stays plain prose (often KB-sized) rather than being escaped into JSON; `gold.json` is **stdlib JSON** (no YAML dependency — claims are short strings).

```go
type Case struct {
    Name       string
    Difficulty string                 // closed set: "easy" | "medium" | "hard"
    Header     extract.DocumentHeader  // ReceivedAt comes from gold.json — deterministic, never wall-clock
    Text       string                  // document.txt, verbatim
    Gold       []GoldSubject
}

// GoldSubject is one blessed expected subject. Matching identity is (Type, normalized Name) (D21);
// Kind/OccurredAt are not part of the gold contract this release — extraction quality is judged on
// which subjects and which claims, not their subtype or date.
type GoldSubject struct {
    Type   string   // "entity" | "event" | "concept"
    Name   string
    Claims []string // the claims a perfect extraction should make about this subject
}

func LoadCase(dir string) (Case, error)        // parse + validate document.txt and gold.json
func LoadDataset(root string) ([]Case, error)  // each immediate subdirectory is one case
```

**`gold.json` shape:**

```json
{
  "difficulty": "medium",
  "header": { "source": "mcp:ingest_text", "title": "…", "tags": ["…"], "received_at": "2024-01-15" },
  "subjects": [
    { "type": "entity", "name": "Gary Gygax",
      "claims": ["Gary Gygax co-created Dungeons & Dragons.", "Gary Gygax co-founded TSR."] }
  ]
}
```

## Validation is at the load boundary, fail-loud (principles: validate at boundaries, fail loudly)

`LoadCase` rejects, with a clear error rather than a degraded case:
- a missing or malformed `gold.json`, or a missing/empty `document.txt`;
- a `difficulty` outside `{easy, medium, hard}`;
- a gold subject with a `type` outside `{entity, event, concept}`, an empty `name`, or zero `claims` — the same shape the real extract `validate` (D6) enforces on predictions, so the gold is held to the contract it is measured against;
- a malformed `received_at` (not an ISO date).

Once loaded, a `Case` is internal trusted data — scoring (D21) does not re-validate it.

**Rejected.**
- **One inline JSON (or YAML) file per case** carrying the document as a string field. Escaping KB-sized prose into JSON is unreadable and undiffable; YAML adds a dependency the suite otherwise avoids. The split keeps the document as prose and the gold as stdlib JSON. Rejected.
- **`Run` constructing the `Extractor` (and the `llm.Client`) itself** from a model string. That would put a second extractor-construction path in `eval`, the exact drift D6 warns against; taking a ready `*extract.Extractor` keeps construction in one place (the composition root / the D22 CLI, sharing the D19 resolver) and keeps `eval` unit-testable with a mock-backed extractor. Rejected.
- **A free-form difficulty string.** Un-aggregatable — D21 reports per-difficulty rollups, which needs a closed, ordered set. Rejected.
- **Putting `Kind`/`OccurredAt` in the gold and scoring them.** This release judges *which subjects* and *which claims*; subtype and date are extraction details the product does not promise to evaluate yet. Keeping them out of the gold keeps blessing a case tractable. Rejected (deferrable).

**Verification.**
- R-VXAT-MMTX — `LoadCase` on a well-formed case directory returns a `Case` whose `Text` is `document.txt` verbatim, whose `Header.ReceivedAt` is the `gold.json` `received_at` (not the wall clock), whose `Difficulty` is the parsed label, and whose `Gold` carries every expected subject with its `Type`, `Name`, and `Claims`.
- R-VYIQ-0EKM — `LoadCase` fails loudly (error, no `Case`) on each boundary violation: missing/malformed `gold.json`, empty/missing `document.txt`, a `difficulty` not in `{easy,medium,hard}`, a gold subject with a type outside `{entity,event,concept}` / empty name / zero claims, and a malformed `received_at`.
- R-VZQM-E6BB — `LoadDataset` over a root containing several case subdirectories returns one `Case` per subdirectory (all cases discovered, each loaded through `LoadCase`).
- R-W26F-5PSP — `Run` invokes the production extractor on the case's `Header` and `Text` verbatim and returns its subjects unchanged: a capturing mock provider behind the `Extractor` confirms the request carries extract's production prompt and the case's source text (and the fixed received-on date), proving the harness exercises the real extract path with nothing added.
