# Decision 27 — Merge MCP surface (`merge` + `merges`)

**Decision.** Two new MCP verbs expose the merge feature on the product surface, cloning the established D10/D16 idiom — the surface grows from **fifteen** verbs (D16's thirteen plus these two). `merge` is a write that **fires and returns** a queued handle (the actual fold runs on the worker, off the request path, like `ingest`); `merges` is a paginated **audit** read of completed folds. The merge *operation lifecycle* (queued → working → done/failed) is **not** a third verb here — it is reachable through D16's existing `jobs`/`jobs_count`, which gain an optional `kind` filter (recorded in D16; see "Two surfaces, two tables" below).

| tool | input | → result |
|---|---|---|
| `merge` *(new, WRITE, fire-and-return)* | `{winner (req), loser (req)}` — both `type/slug` paths | `{job_id, status:"queued"}` |
| `merges` *(new, READ, paginated)* | `{limit?, cursor?}` | `{items:[{merged_at, winner, loser_name, alias_norm_name, created_by}], next_cursor}` — newest-first |

Both wire through the standard idiom: a func field on `Handler`, a `WithMergeService`/`WithMergesService` Option, a `handleToolCall` case, a `handleMergeCall`/`handleMergesCall`, a conditional `tools()` append, and a hand-coded `inputSchema` (`objectSchema({winner,loser},[winner,loser])` / `listSchema({})`). The route stays behind `RequireIdentity`; nginx is the trust boundary and wiki does no token logic. Every subject reference crosses as a `type/slug` path — never an internal ULID.

## `merge`

```go
// handleMergeCall: resolve both paths to ids ONCE, here, then enqueue a kind='merge' job (D26).
func (h *Handler) handleMergeCall(ctx, args) (toolResult, error)
```

- **Resolve at request time via `GetByPath`** (D11) — subjects-only, exact. A name that has already been folded away is an *alias*, not a live subject, so `GetByPath` returns `ErrSubjectNotFound`; `merge` does **not** silently follow the alias to the survivor (that would let an owner "merge" a name that no longer names a subject). Both `winner` and `loser` are explicit owner input — there is no auto-pick, so no winner-selection ambiguity logic.
- **Enqueue, don't execute.** Insert a `kind='merge'` job (D26) carrying the resolved `merge_winner_id`/`merge_loser_id`, `status='pending'`, `owner` from `appkit.IdentityFrom(ctx)`; nudge the worker; return `{job_id, status:"queued"}`. The result is **never** `status:"merged"` — completion is asynchronous, observed by the job reaching `done` (via `status`) and by the fold appearing in `merges`.
- **Two-channel errors** (the D10 convention):
  - a `winner` or `loser` path matching **no** subject → clean **not-found** result; no job is enqueued. (Because the path token is the `UNIQUE` `norm_name` (D11), resolution is unambiguous — there is no ambiguous-path channel.)
  - `winner` and `loser` resolving to the **same id** → **tool error**.
- **Identity-gated write.** `owner` (and thus the eventual `aliases.created_by`, D25) comes from `X-Owner-Email`; the verb is reachable only behind `RequireIdentity`.

## `merges`

A keyset-paginated audit over the `aliases` table (D25), newest-first, reusing the D15 cursor contract:

```go
func (a *AliasStore) ListMerges(ctx, p page.Params) (items []Alias, next string, err error) // net-new read
```

- Orders `created_at DESC, norm_name` (`norm_name` is UNIQUE — a stable keyset tiebreak); the cursor encodes that pair. At v1 the merge log is far smaller than the subject set, so the ordering is unindexed for now (an index is a noted later optimization, not a v1 need).
- Each item renders the **winner as a `type/slug` path** by resolving `subject_id → Path` (D11); the D25 FK (`ON DELETE RESTRICT`) guarantees `subject_id` always names a **live** subject, so the path never dangles (a later merge that folded the winner repoints the row to the terminal survivor, which is the correct current answer). The other fields come straight off the alias row: `loser_name` = `aliases.name`, `alias_norm_name` = `aliases.norm_name`, `merged_at` = `aliases.created_at`, `created_by`.
- Returns `pagedResult("merges", items, next_cursor)`. No internal id ever appears.

## Two surfaces, two tables (why `merges` is not a kind-filtered `jobs`)

A merge touches two tables, and the two read verbs answer two different questions:

- **`jobs`/`jobs_count` (D16)** read the `jobs` table — the merge *operation* lifecycle (queued/working/done/failed). They gain an optional `kind` filter (default `ingest`) so a merge job is listable as `kind=['merge']` but excluded from the default ingest view. *(That filter is owned by D16.)*
- **`merges` (here)** reads the `aliases` table — the durable human record of *what folded into what*. It cannot be served from the `jobs` table: a successful merge **deletes the loser subject**, and the merge job row stores only `merge_loser_id` (a now-dead id) and never the loser's *name*. The loser's display name survives **only** in `aliases.name`, which exists regardless (it is the forward-routing record name resolution needs). Reading the audit from `jobs` would mean duplicating the loser name into a second table for nothing.

**Rejected.**
- **Folding `merges` into a kind-filtered `jobs` verb.** Rejected: after the loser subject is deleted, the jobs row cannot render "what folded into what" (no loser name, dead loser id). The audit's data lives in `aliases`; the lifecycle's data lives in `jobs`. One verb cannot serve both without denormalizing the loser name into `jobs`.
- **Resolving `winner`/`loser` via `ResolveByName` (alias-aware).** Rejected for `GetByPath`: an owner naming an already-folded name should get a clean not-found, not a silent redirect to the survivor that could produce a confusing self-merge or a wrong target. Merge inputs are explicit live subjects.
- **`merge` returning `status:"merged"` synchronously.** Rejected: the fold runs on the worker off the request path (D26), exactly like `ingest`. Returning a completed status would either block the request on a multi-second recompile or lie. Fire-and-return a queued handle.
- **Surfacing `loser_type` / `moved_claim_count` in `merges`.** Rejected as out of scope: D25's `aliases` table stores neither, and the product promise — "what folded into what, and when" — is met by loser name, winner path, and timestamp. Adding them would mean extra columns for a non-promised field.
- **An opaque `pair_id` handle (the research §7 shape).** Rejected for v1: that handle belongs to machine-proposed *candidates* (automated detection), which is deferred. With the owner naming the pair directly, the merge's handle is just its `job_id`.

**Verification.**
Driving the JSON-RPC handler in-process with a stubbed identity (D10 strategy), against the D26 service on a real temp SQLite + scripted mock compiler:

- R-DWDM-RVA7 — `tools/list` returns the **fifteen** verbs including `merge` and `merges`, each `inputSchema` well-formed; `merges` (no required field) **omits** `required` entirely — never `"required": null` (the D10 R-N4KO strict-client rule).
- R-DYTF-JERL — `merge` with two valid, distinct `type/slug` paths resolves both via `GetByPath`, enqueues a `kind='merge'` job carrying their ids with `owner` from identity, and returns `{job_id, status:"queued"}` — and it is **fire-and-return**: the loser subject still exists immediately after the call returns (the fold has not run yet), and the result is never `status:"merged"`.
- R-E01B-X6IA — `merge` surfaces its two error channels: a `winner`/`loser` path matching no subject → a clean **not-found** with **no job enqueued**; `winner` and `loser` resolving to the same id → a **tool error**. Neither crashes. (No ambiguous-path channel exists: the path token is the `UNIQUE` `norm_name`, D11.)
- R-E198-AY8Z — `POST /mcp` is behind `RequireIdentity`, and a `merge` stamps the job's `owner` (→ the eventual `aliases.created_by`, D25) from `X-Owner-Email`.
- R-E2H4-OPZO — `merges` returns a `{items, next_cursor}` page, **newest-first**, each item carrying `merged_at`, the winner as a `type/slug` **path** (no internal ULID anywhere in the result), `loser_name`, `alias_norm_name`, and `created_by`; the returned cursor fetches the following page.
- R-E3P1-2HQD — After a merge of `L` into `W` commits its alias (D26), `merges` surfaces that fold with `winner == Path(W)` and `loser_name == L` — proving the audit reads the live `aliases` table and renders the winner by its current path.
