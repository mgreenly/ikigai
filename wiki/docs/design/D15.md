# Decision 15 — Cursor pagination: the contract + the list seams

**Decision.** Every listing that can grow without bound — jobs, subjects, a subject's claims, the LLM-call footprint — returns a **bounded page plus an opaque cursor** for the next, with any filter applied **before** paging (product: "Page large results"). Pagination is **keyset (seek)**, not offset: each list orders by a stable, unique key and the cursor carries the last row's key, so the next page asks for rows strictly *past* it (in the list's sort direction). This is O(log n) per page and immune to skip/duplicate under concurrent inserts — `LIMIT/OFFSET` is O(n) and shifts rows when the set changes underneath the pager.

The **jobs** list additionally answers two filter shapes the others do not: its `status` filter is a **set** (any of the five states, OR semantics — so "everything not done" is one query), and the list is ordered **newest-first**. A companion `CountJobs` seam reports the **filtered total** without returning rows.

## Order keys and filters

Each collection orders by `(sort_column, id)` with the ULID `id` as the unique tiebreak; the filter narrows the set in the same `WHERE` before the keyset predicate. Every collection but jobs orders **ascending**; jobs orders **descending** (newest-first):

| collection | order key | filters |
|---|---|---|
| jobs | `received_at, id` **DESC** (newest-first) | `status` (a **set** — any of the five), time range (`received_at` since/until) |
| subjects | `name, id` (asc) | `type`, name-substring (`norm_name LIKE`) — existing filters, now paged |
| claims (by subject) | `id` (asc) | `subject` (required) |
| llm_calls | `started_at, id` (asc) | `job_id`, `stage`, time range |

## The `internal/page` package (pure; no DB)

Owns the request shape and the cursor codec, table-tested directly:

```go
package page

const (
    DefaultLimit = 50
    MaxLimit     = 200
)

// Params is a cursor-pagination request.
type Params struct {
    Limit  int    // clamped to [1, MaxLimit]; 0 → DefaultLimit
    Cursor string // opaque; "" = first page
}

// EncodeCursor packs ordered key components into one opaque token (base64url over a
// delimited join). DecodeCursor reverses it; ok=false on a malformed token.
func EncodeCursor(parts ...string) string
func DecodeCursor(token string) (parts []string, ok bool)
```

A **malformed cursor is a clean error, never a silent reset to page 1** (fail loudly): a cursor we cannot decode is a client bug to surface, not to paper over by silently restarting from the top (which can loop a client forever re-reading page 1). The decode failure propagates as a typed error the MCP layer renders as a tool error (D16).

## The store seams

Stores return `(items, nextCursor, error)` — a plain tuple the MCP layer wraps as `{items, next_cursor}` (D16); no generic result struct.

```go
type JobFilter     struct { Statuses []string; Since, Until time.Time } // Statuses: nil/empty = all states; else match-any (OR)
type LLMCallFilter struct { JobID, Stage string; Since, Until time.Time }

func (s *JobStore)     ListJobs(ctx, f JobFilter, p page.Params)          (jobs []Job, next string, err error)
func (s *JobStore)     CountJobs(ctx, f JobFilter)                        (n int, err error)                       // filtered total; no paging, no rows
func (s *SubjectStore) List(ctx, typ, nameContains string, p page.Params) (subjects []Subject, next string, err error) // reshaped
func (s *ClaimStore)   ListBySubject(ctx, subjectID string, p page.Params) (claims []Claim, next string, err error)     // reshaped
func (s *LLMCallStore) List(ctx, f LLMCallFilter, p page.Params)          (calls []CallRecord, next string, err error)
```

`JobFilter.Statuses` is the set form of the old scalar `Status`: an empty/`nil` slice matches every state; a non-empty slice builds `status IN (?,?,…)` (match-any). Element validity (each must be one of the five) is enforced at the **MCP boundary** (D16), not re-checked in the store. `CountJobs` shares `JobFilter` verbatim with `ListJobs`, so a count and its corresponding list always describe the same set.

## Mechanics

Each query appends the keyset predicate in the explicit portable form and over-fetches by one to detect a further page. The comparison operator follows the list's sort direction — `>` for the ascending lists, `<` for the descending jobs list:

```sql
-- ascending lists (subjects, claims, llm_calls)
... WHERE <filters>
    AND (sort_col > ?1 OR (sort_col = ?1 AND id > ?2))   -- omitted entirely when cursor is ""
  ORDER BY sort_col, id
  LIMIT <clampedLimit> + 1

-- jobs (newest-first): direction flipped, same shape
... WHERE <filters>
    AND (received_at < ?1 OR (received_at = ?1 AND id < ?2))   -- omitted when cursor is ""
  ORDER BY received_at DESC, id DESC
  LIMIT <clampedLimit> + 1
```

If the `+1` row comes back, drop it and set `nextCursor = EncodeCursor(lastReturned.sortCol, lastReturned.id)`; otherwise `nextCursor = ""` (last page). The cursor codec is direction-agnostic — it carries the boundary row's `(sort_col, id)` either way; only the predicate's operator and `ORDER BY` differ. The single-column claims key encodes just `id`. SQLite serves the jobs newest-first ordering by scanning the existing index in reverse — **no new index, no migration for the direction flip**. Indexes backing the orderings: the existing `jobs_pending_received (status, received_at, id)` (also covers the `status IN (…)` set filter and the `COUNT(*)`), `claims_subject (subject_id)`, and the D13 `llm_calls_*` indexes; this decision's migration adds `subjects (name, id)`.

`CountJobs` runs the **same `WHERE`** as `ListJobs` (the `status IN (…)` + time-range predicates) under `SELECT COUNT(*) FROM jobs WHERE <filters>` — no keyset predicate, no `ORDER BY`, no `LIMIT`. It returns the total matching the filter without materializing any row, so "how many are not done?" stays a single cheap query across an arbitrarily large jobs table.

**How it's tested.** Against a real temp SQLite migrated by the appkit runner:
- Insert N rows and page through at limit `k`: **every row appears exactly once, in key order, and `nextCursor` is empty only on the final page** (no skips, no duplicates across page boundaries). For jobs, "key order" is **newest-first** (descending `received_at, id`).
- A filter narrows before paging: a status **set** (jobs — see below), stage / job_id / time-range (llm_calls), type / name-substring (subjects), subject (claims) — only matching rows are paged.
- **Jobs newest-first:** insert jobs with distinct `received_at`, list them, and assert the page is ordered most-recently-received first; ties broken by descending `id`. The ordering holds whether or not a status/time filter is applied.
- **Jobs multi-state set:** `ListJobs` with `Statuses` of several states returns exactly the jobs in any of those states (OR), and the same rows page cleanly newest-first; an empty/`nil` `Statuses` returns jobs of all states.
- **`CountJobs`:** returns the count of jobs matching a `JobFilter` (state set and/or time range) without returning rows, and that count equals the number `ListJobs` would page through under the same filter; an empty filter counts all jobs.
- `Limit` clamps: `0 → DefaultLimit`, `> MaxLimit → MaxLimit`.
- Codec round-trips as a table test (`Decode(Encode(parts)) == parts`); a corrupted token decodes to `ok=false` and the store list returns a clean error rather than silently restarting.

**Rejected.**
- **`LIMIT/OFFSET` pagination** — O(n) scan to the offset and unstable: an insert/delete before the offset shifts every later page, skipping or repeating rows. Keyset is strictly better for an append-heavy footprint. Rejected.
- **A transparent (plaintext) cursor exposing the raw key** — invites clients to construct or mutate cursors and couples them to the sort columns; opacity keeps the key shape a private implementation detail we can change. Rejected.
- **Silent reset to the first page on a bad cursor** — hides a client bug and risks an infinite re-read loop. Fail loudly. Rejected.
- **A generic `Page[T]` result struct** — adds generic-field friction for no gain; a `(items, next, err)` tuple is simpler and the MCP layer shapes the wire form. Rejected.
- **A separate descending cursor codec for jobs** — the codec is already direction-agnostic (it carries the boundary `(sort_col, id)`); only the SQL predicate operator and `ORDER BY` flip. A second codec would duplicate logic for nothing. Rejected.
- **Returning a running `total` on every `ListJobs` page** — recomputes `COUNT(*)` on each page fetch and couples the list and the count; the product wants a count *without* retrieving rows, so a standalone `CountJobs` is both cheaper and the stated promise. Rejected.

**Verification.**
- R-17C5-VP2I — Paging a collection of N rows at limit `k` yields every row exactly once in key order across pages, with `nextCursor` non-empty until the final page and empty on it (no skipped or duplicated rows at page boundaries).
- R-18K2-9GT7 — A filter is applied before paging: a status filter (jobs), a `stage`/`job_id`/time-range filter (llm_calls), a `type`/name-substring filter (subjects), and the required `subject` (claims) each restrict the paged set to matching rows only.
- R-XYAZ-V0XE — The jobs list is ordered **newest-first** (most-recently-received first; ties broken by descending `id`), and this ordering holds whether or not a status or time-range filter is applied.
- R-XZIW-8SO3 — `ListJobs` with a `JobFilter.Statuses` set returns exactly the jobs whose status is in that set (match-any / OR), and an empty/`nil` `Statuses` returns jobs of every state.
- R-Y1YP-0C5H — `CountJobs` returns the number of jobs matching its `JobFilter` (state set and/or time range) without returning any row, equal to the count `ListJobs` would page through under the same filter; an empty filter counts all jobs.
- R-19RY-N8JW — `Params.Limit` is clamped: `0` yields `DefaultLimit` rows (when available) and a value above `MaxLimit` yields at most `MaxLimit`.
- R-1C7R-ES1A — `EncodeCursor`/`DecodeCursor` round-trip ordered key parts losslessly, and `DecodeCursor` reports `ok=false` for a malformed token.
- R-1DFN-SJRZ — A list call given an undecodable cursor returns a clean error and does not silently restart from the first page.
