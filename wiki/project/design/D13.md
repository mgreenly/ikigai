# Decision 13 — The LLM-call footprint: `llm_calls` table + recorder seam

**Decision.** Every call the wiki makes to an LLM is recorded as one durable row capturing what the model was asked, what it returned, the parameters it ran under, and the tokens it burned, so a failed or aborted ingest leaves an inspectable trail. Recording happens at the **single seam every provider round-trip funnels through** — `sendText` in `internal/llm` — so extract's parse-retries, compile's tighten loop, and ask's calls are all captured by one mechanism, each round-trip its own record.

**The recorder seam (`internal/llm`).** A `Recorder` is injected into `Client` (nil = no-op, so non-recording paths stay simple). `sendText` builds one `CallRecord` per round-trip and hands it to the recorder.

```go
// CallRecord is one provider round-trip's footprint.
type CallRecord struct {
    ID        string    // ULID
    Stage     string    // from CallSite.Stage: "extract"|"compile"|"ask-subject"|"ask-synthesis" (D19); eval adds "judge" (D21)
    JobID     string    // from ctx via WithJobID; "" for ask (no job)
    Attempt   int       // 1-based within a JSON retry / tighten loop
    Provider  string    // e.g. "anthropic"
    Model     string
    Params    string    // JSON: {"temperature":…, "reasoning":…}
    Request   string    // JSON: {"system":…, "user":…} actually sent (incl. corrective re-prompt)
    Response  string    // raw final assistant text ("" on transport error)
    Usage     string    // token/cost JSON when the provider reports it ("" otherwise)
    Err       string    // transport / stream / cancellation error; "" on success
    StartedAt time.Time
    EndedAt   time.Time
}

// Recorder persists one call record. Injected into Client; a nil Recorder is a no-op.
type Recorder interface {
    Record(ctx context.Context, rec CallRecord) error
}
```

**Where the metadata comes from.**
- **Stage** — a new `CallSite.Stage` field. Each stage already owns its `DefaultCallSite` (extract D6, compile D7, ask D9) and labels itself there, so the composition root cannot silently drop it. (D5's faithful-application guarantee now covers `Stage` too.)
- **JobID** — threaded through the Go `context`. `internal/llm` exposes `WithJobID(ctx, id) context.Context` and `JobID(ctx) string`. The **worker** wraps the integrate context once per job (`ctx = llm.WithJobID(ctx, job.ID)`); `sendText` reads it back. `ask` never sets it, so its records carry `JobID == ""`. The association never enters extract/compile/ask signatures.
- **Attempt** — the `JSON[T]` loop stamps its 1-based iteration onto each round-trip's record; compile's tighten loop (D7) does the same. A single-shot call records `Attempt == 1`.

**Eager, detached writes — records are NOT in the job's transaction.** Each record is persisted the moment its round-trip completes, in its own auto-commit write, *outside* the job's final atomic commit (D14), so a job that **fails or is aborted still leaves every call record behind**. Two consequences:
- The recorder writes with a **detached context** (`context.WithoutCancel(ctx)`), so recording an *aborted* call — whose request context was just cancelled — still succeeds, landing a row with `Err: "context canceled"`.
- Under single-writer SQLite the recorder's write is sequential with the rest of the worker; because integrate holds **no** DB transaction across the LLM calls (D14), an eager record write never deadlocks against a held job transaction.

**Faithful capture.** `Request` is the JSON of the messages actually sent (`{system, user}`), so a retry's corrective re-prompt is captured verbatim, not the original prompt. `Params` serializes the `CallSite`'s temperature and reasoning as resolved for the call — the lever that drifted in D6/D7 — so a wiring bug (provider-default temp, reasoning-on) is visible and distinguishable from a prompt bug. `Provider`/`Model` come from the `Client`'s configured provider and `CallSite.Model`. `Usage` is wired from agentkit's reported usage on the final stream event when present; absent that, `""`.

**Persistence (`LLMCallStore` in `internal/wiki`).** A SQLite-backed `Recorder`. New forward migration (`bin/new-migration wiki`), matching the live schema's TEXT-timestamp (`RFC3339Nano`) convention and the suite's no-`FOREIGN KEY` style. Current effective schema:

```sql
CREATE TABLE llm_calls (
  id          TEXT PRIMARY KEY,            -- ULID
  job_id      TEXT NOT NULL DEFAULT '',    -- → jobs.id by convention; '' for ask
  stage       TEXT NOT NULL                -- closed set, DB-enforced (see below)
              CHECK (stage IN ('extract','compile','ask-subject','ask-synthesis','judge')),
  attempt     INTEGER NOT NULL DEFAULT 1,
  provider    TEXT NOT NULL DEFAULT '',
  model       TEXT NOT NULL DEFAULT '',
  params      TEXT NOT NULL DEFAULT '{}',
  request     TEXT NOT NULL DEFAULT '',
  response    TEXT NOT NULL DEFAULT '',
  usage       TEXT NOT NULL DEFAULT '',
  error       TEXT NOT NULL DEFAULT '',
  started_at  TEXT NOT NULL DEFAULT '',
  ended_at    TEXT NOT NULL DEFAULT ''
);
CREATE INDEX llm_calls_job  ON llm_calls (job_id, started_at, id);  -- per-job inspect + D15 paging
CREATE INDEX llm_calls_time ON llm_calls (started_at, id);          -- timeframe paging
```

**`stage` is a closed set, enforced by a DB CHECK.** Allowed values are exactly the stages a `CallSite` emits — `extract`, `compile`, `ask-subject`, `ask-synthesis` (production, D19) and `judge` (eval-only, D21). The pre-D19 label `ask` is **deliberately absent**: D19 split `ask` into `ask-subject` + `ask-synthesis`, no call site emits `ask`, and the live DB is reset so no historical `ask` rows exist. The CHECK catches a code↔schema stage drift at the write boundary (fail loudly) instead of silently storing an unknown label.

**The constraint is corrected by a follow-on rebuild migration.** The Phase-24 migration shipped a CHECK pinned to the *pre-D19* set `('extract','compile','ask')`; D19/D21 widened the stages without updating it, so every `ask-subject`/`ask-synthesis`/`judge` write failed (`CHECK constraint failed: stage IN ('extract','compile','ask')`), breaking `ask` and eval `judge`. Since a committed migration is immutable and SQLite cannot `ALTER` a CHECK, the fix is a **new** forward migration that rebuilds the table (create new → copy rows → drop old → rename) with the corrected CHECK above and re-creates both indexes. Against the freshly reset dev DB the copy carries no rows; in general it preserves all existing rows, none of which can carry the now-removed `ask` label.

Records are **append-only and kept forever** — no update, no delete this release (pruning deferred, product). A re-run (D14) replaces a job's *claims* but its old call records remain, accumulating beside the new attempt's.

**How it's tested.** A **capturing `Recorder` mock** is wired into `llm.Client` beside the scripted provider mock (D5):
- A `JSON[T]` call scripted bad-then-good emits **two** records: `Attempt 1` with the malformed response, `Attempt 2` with the good one — both carrying the `CallSite`'s `Stage`, `Model`, `Params`, and the second carrying the corrective re-prompt in `Request`.
- A scripted transport/stream error emits **one** record with non-empty `Err` and empty `Response`.
- With `WithJobID(ctx, "J")` set, records carry `JobID == "J"`; with no job on the context, `JobID == ""`.
- Detached-write proven by calling `Record` with an **already-cancelled** context against a real temp SQLite and asserting the row still lands.
- `LLMCallStore.Record` round-trips every column against a real temp SQLite migrated by the appkit runner.
- **The stage CHECK is exercised against the real engine, not a mock** — only the migrated SQLite can falsify "the DB accepts this stage." Against a real temp SQLite migrated by the appkit runner, `LLMCallStore.Record` accepts **each** canonical stage (`extract`, `compile`, `ask-subject`, `ask-synthesis`, `judge`) and is **refused** for a non-canonical stage. This is the regression guard the original migration lacked.

**Rejected.**
- **Recording inside the job's atomic commit** — would discard exactly the records you most want (a failed/aborted job's), since the write-set rolls back. Eager out-of-band writes are the whole value. Rejected.
- **An explicit `jobID` parameter on `JSON[T]`/`Converse`** — threads a job id through extract/compile/ask, which have no other use for it, and invites a stage to forget it. Context-threading keeps the association out of stage code. Rejected.
- **Reusing `agentkit.Conversation.Log io.Writer` JSONL sink** — a flat log file is not queryable, filterable, or paginable; the product promises a listable, filterable footprint, which requires a table. The `Log` sink stays as an orthogonal debug convenience. Rejected as primary record.
- **One record per `JSON[T]` call (final attempt only)** — throws away the failed attempts, which are precisely the structured-content failures being diagnosed. One record per round-trip is required. Rejected.

**Verification.**
- R-VNS0-1Z85 — A single successful provider round-trip through `sendText` produces exactly one `CallRecord` carrying the `CallSite`'s `Stage`, `Model`, and `Params`, the `{system,user}` `Request` actually sent, the assistant `Response`, and empty `Err` (captured via the mock recorder).
- R-VOZW-FQYU — A `JSON[T]` call scripted bad-then-good emits two records — `Attempt 1` with the malformed response and `Attempt 2` with the good response — and the `Attempt 2` `Request` carries the corrective re-prompt, not the original prompt.
- R-VRFP-7AG8 — A provider transport/stream error emits one record with non-empty `Err` and empty `Response` (a failure is on the record, not dropped).
- R-VSNL-L26X — A round-trip made under `WithJobID(ctx,"J")` records `JobID=="J"`; a round-trip with no job on the context (ask) records `JobID==""`.
- R-VTVH-YTXM — `Record` invoked with an already-cancelled context still persists the row to a real temp SQLite (detached write), so an aborted call's footprint survives.
- R-VV3E-CLOB — `LLMCallStore.Record` round-trips every `llm_calls` column (stage, attempt, provider, model, params, request, response, usage, error, timestamps) against a real migrated temp SQLite, and a nil `Recorder` on the `Client` is a no-op (no panic, no row).
- R-EMWV-6RK5 — Against a real migrated temp SQLite, `LLMCallStore.Record` accepts a row for **every** canonical stage — `extract`, `compile`, `ask-subject`, `ask-synthesis`, and `judge` — and each row is persisted (so `ask` and eval `judge`, which emit `ask-subject`/`ask-synthesis`/`judge`, can record their calls). A wrong schema pinned to the pre-D19 set fails this.
- R-EO4R-KJAU — Against the same real migrated temp SQLite, `LLMCallStore.Record` for a non-canonical stage (e.g. the retired `ask`, or `bogus`) is **refused by the DB CHECK** and lands no row — proving the closed set is enforced by the engine, not merely by convention.
