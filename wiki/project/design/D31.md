# Decision 31 — The keyword lane: full-text search returns

**Decision.** Bring back the SQLite full-text index over page titles and bodies, and wrap it in a `keywordRetriever` that satisfies the search seam (D08). This is the "match the words" lane of the hybrid.

**The index.** A new forward migration (`bin/new-migration wiki`) recreates `pages_fts` as an **external-content** FTS5 table over `pages(title, body)` — "external-content" means the index doesn't store its own copy of the text, it points at the `pages` rows, so it stays small. (The old drop migration that removed this table stays frozen on disk; this is a new create migration, per the immutable-migrations rule.)

```sql
CREATE VIRTUAL TABLE pages_fts USING fts5(
  title, body,
  content='pages',
  content_rowid='rowid'
);
```

**Keeping it in sync (the one subtle part).** Because the index points at `pages` instead of copying it, an *update* to a page can't just be overwritten in the index — FTS5 needs to be told the **old** values first, then the new ones, or the index quietly rots. So every page write does, inside the same transaction as the page write itself:

- **new page** → insert the page's title/body into `pages_fts`;
- **rewritten page** → issue the FTS5 `'delete'` command with the page's **previous** title/body (read before the row is overwritten), then insert the new title/body.

This is exactly the trap that made this machinery tricky before; the fix is the read-old-then-delete-then-insert dance, done in the page-write path. The reference implementation from the prior build (`ftsPhrase` + the external-content sync) is copied as-is. On first ship, the existing pages are indexed in one shot by the migration itself (`INSERT INTO pages_fts(pages_fts) VALUES('rebuild');`) — pure SQL, no network.

**Cleaning the query.** A user's question can contain quotes, parentheses, and other characters that mean something special to FTS5 — left raw they'd cause a syntax error or worse. So `ftsPhrase` wraps each term as a quoted literal (doubling any internal quote) and OR-joins them, so the search looks for pages containing *any* of the terms and never misinterprets punctuation as a search operator.

**The retriever.**

```go
// keywordRetriever runs the full-text lane. It implements retrieve.Retriever (D08).
type keywordRetriever struct { /* read DB handle (D17) */ }

func (r *keywordRetriever) Search(ctx context.Context, query string, k int) ([]retrieve.Hit, error)
```

It runs an FTS5 `MATCH` with SQLite's built-in `bm25()` relevance ranking, best match first, capped at `k`, and returns each match as a `Hit` whose `PageID` is the page's subject id (the seam's stable key), `Score` the bm25 rank, plus the title and a matched snippet.

**Rejected.**
- **A contentless / standalone FTS table that stores its own copy of every page body.** Doubles the text on disk and needs the same sync on writes anyway. External-content is the leaner, established choice.
- **Feeding the raw question straight into `MATCH`.** A question with a quote or parenthesis throws an FTS5 syntax error, and a crafted question could inject operators. Sanitizing into quoted phrase literals is mandatory, not optional.
- **Rebuilding the whole index on every write instead of the delete-old/insert-new dance.** Correct but absurdly wasteful; the targeted sync is O(one page), the rebuild is O(all pages).

**Verification.**
- R-203P-F1ET — After migrations run against a real temp SQLite, `pages_fts` exists and a `MATCH` for a word that appears in a page's body returns that page (the index is built and queryable; the rebuild backfilled existing rows).
- R-22JI-6KW7 — External-content sync is correct across a rewrite (the regression guard): after a page is rewritten so a word is removed and another added, a `MATCH` for the removed word no longer returns the page and a `MATCH` for the added word does — proving the read-old → delete → insert sync, not a stale index, against a real temp SQLite.
- R-23RE-KCMW — `ftsPhrase` neutralizes FTS5 syntax: a query containing quotes/parentheses/operators runs as literal terms without error, and multiple terms are OR-joined so a page matching **any** term is returned (no injection, no syntax failure).
- R-24ZA-Y4DL — `keywordRetriever.Search` returns `Hit`s whose `PageID` is the matched page's subject id, ordered by bm25 relevance (best first) and capped at the requested `k`.
