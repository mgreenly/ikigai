# Decision 8 — Search returns: hybrid retrieval over pages, behind one seam

**Decision.** `ask` no longer finds pages by matching the question to a subject's exact name. It now **searches the page bodies** and answers from whatever comes back most relevant. Search is a **hybrid** of two independent lanes whose results are merged:

- a **keyword lane** (classic full-text search — good at exact and rare words, names, identifiers), and
- a **meaning lane** (vector embeddings — good at "different words, same idea").

The two lanes run, their result lists are fused into one ranked list, and the top handful of pages go to the answer step.

This decision owns only the **boundary** between the search machinery and the rest of the service — the contract everything else plugs into. The pieces behind it each get their own decision:

- **D30** — where page embeddings are stored.
- **D31** — the keyword lane (the full-text table returns).
- **D32** — the meaning lane (turn the question into a vector, score it against every page).
- **D33** — how the two lanes' results are merged into one ranked list.
- **D34** — configuring and recording the embedding calls.
- **D35** — keeping page embeddings up to date in the background.
- **D36** — preparing the question before it hits the lanes.
- **D9** (rewritten) — `ask` consuming the ranked results, and saying "nothing here" honestly.

**The seam (`internal/retrieve`).** A page that matches a search is a `Hit`. Anything that can search is a `Retriever`. `ask` depends on the `Retriever` interface and nothing more — it never knows whether a hit came from the keyword lane, the meaning lane, or the fused result.

```go
type Hit struct {
    SubjectID string  // the subject this page belongs to
    PageID    string  // stable id used both to merge/dedupe and to cite (== SubjectID)
    Score     float64 // lane-local: keyword score, cosine, or the fused score
    Snippet   string  // matched excerpt, for the answer step's context
    Title     string
}

type Retriever interface {
    Search(ctx context.Context, query string, k int) ([]Hit, error)
}

// SearchLimits clamps a requested result count into a sane range.
type SearchLimits struct{ Default, Cap int }
func (l SearchLimits) Resolve(k int) int   // 0 → Default; otherwise clamp to [1, Cap]
```

Each lane (keyword, meaning) is its own `Retriever`; a `hybridRetriever` calls both and merges them, and is itself a `Retriever`. `Search` returns a **flat list of hits** — the merging is the hybrid retriever's private business, invisible to `ask`. `PageID` equals the subject id, so the same value identifies a page for merging *and* for citing it in the final answer.

**This rewrites the old Decision 8.** It used to record the opposite — that search had been *removed* for this release and the full-text table dropped. That release shipped; search is now being brought back as planned. The two old checks this decision carried (that the full-text table was gone, and that ingest never touched it) describe the world we're now leaving, so they are retired here; the full-text table comes back in D32. The history of the removal-then-return lives in the plan, not here.

**Rejected.**
- **Keyword-only search.** Cheap, but misses "different words, same idea" — the whole reason questions fail today. The meaning lane is the point.
- **Meaning-only search.** Embeddings smear over exact/rare terms — names, IDs, jargon. Keeping the keyword lane catches what embeddings blur, nearly for free.
- **Letting `ask` reach into the search internals directly** (no seam). Couples the answer code to one search implementation and makes each lane impossible to test on its own. The single `Retriever` boundary keeps them swappable and independently testable.

**Verification.** None of its own — this decision is a boundary, not a behavior. Its correctness is proven by the lane and `ask` decisions that sit behind it (D31–D38).
