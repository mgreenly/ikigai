# Decision 51 — Registry adoption: resolve wiki's own port through `registry`

**Problem.** wiki hardcodes its loopback port as a literal — `Port = 3001` in
`internal/wiki/wiki.go` — while `etc/manifest.env` (`PORT=3001`) and the nginx
fragment (`127.0.0.1:3001`) each carry the same number independently. A renumber
would drift silently and only surface at deploy. The suite now has a shared
`registry` package — the single source of truth for every service's port —
already adopted by crm and notify (appkit-side D9/D10 in those services). wiki
should resolve its own port from it. Unlike notify, wiki is **neither a producer
nor a consumer**, so there are no feed URLs to resolve: the adoption is the
own-port resolution only.

**Decision.** wiki takes its port from `registry`, not a literal.

1. **Module wiring.** `wiki/go.mod` gains `require registry v0.0.0` and
   `replace registry => ../registry` (the exact form notify uses).
2. **Own port.** wiki's Spec sets `Port: registry.MustPort("wiki")` in place of
   the `Port = 3001` literal; the `Port` const is deleted from
   `internal/wiki/wiki.go`. `registry.MustPort("wiki")` returns `3001`, so the
   emitted manifest `PORT` stays byte-identical.
3. **Source-scan guard.** A test proves no `127.0.0.1:30xx` and no bare
   service-port literal remains in wiki's Go source. The nginx fragment and the
   committed manifest keep their literal `3001` — they are authored, shipped
   **verbatim config** (D39/D47 fragment, manifest), not Go source, and are out
   of the guard's scope.
4. **Renumber fails a test.** The existing port assertion
   (`internal/wiki/wiki_test.go`, `spec.Port == 3001`) and the manifest
   byte-equality test (`cmd/wiki/main_test.go`) re-point at
   `registry.MustPort("wiki")`, so a registry renumber changes the expectation
   (and the emitted manifest) and fails a wiki test rather than passing against a
   stale literal.

**Rejected.**
- **Keeping the `3001` literal** — the multi-file drift the registry exists to
  remove.
- **Resolving feed URLs too** (notify's fuller adoption) — wiki has no feeds;
  there is nothing to resolve beyond its own port.
- **Templating the nginx fragment from `registry`** — the fragment is authored
  and shipped verbatim (D39/D47), symlinked live by `opsctl setup`; it is not Go
  source and substituting it is out of scope for this conversion.

**Verification.**
- R-JDBC-V0EV — wiki's `Spec.Port` equals `registry.MustPort("wiki")` (`3001`),
  resolved through the registry rather than a literal — a test asserting
  `Spec.Port == registry.MustPort("wiki")`.
- R-JEJ9-8S5K — a scan of wiki's non-test Go source finds no `127.0.0.1:30`
  occurrence and no bare `3001` loopback-port literal — proving the literal is
  gone from source (the guard excludes `_test.go` and `project/`).
- R-JFR5-MJW9 — the manifest byte-equality test (`Spec` → `manifest.Emit`) still
  equals the committed `etc/manifest.env` (`PORT=3001`) with the port sourced
  from `registry.MustPort("wiki")`, so a registry renumber changes the emitted
  manifest and fails the test.
