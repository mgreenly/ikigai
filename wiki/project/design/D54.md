# Decision 54 — Composition-root normalization: one inline `Spec` in `cmd/wiki/main.go`

**Problem.** wiki declares its `appkit.Spec` in **two** places.
`internal/wiki/wiki.go` exports a lightweight `Spec()` (static fields plus a
`Handlers` hook that mounts a **partial** MCP surface over a degenerate
`NewService(conns, nil, nil, …)` and a `Workers` hook running `worker.Run(ctx)`
with **no** service) and a `Main()` that is **never called**. `cmd/wiki/main.go`
uses `wiki.Spec()` for non-serve verbs and, for serve, calls `buildSpec(cfg)`,
which takes `wiki.Spec()` and **replaces** its `.Handlers` and `.Workers` with
the fully-wired serve surface. The reference shape (crm/notify) is **one**
fully-formed `appkit.Spec` declared **inline** in `cmd/<svc>/main.go`, with no
separate internal spec package and no post-construction mutation of
`.Handlers`/`.Workers`. wiki's `wiki.Spec()` `Handlers`/`Workers` are never used
at serve (`buildSpec` overwrites them) and `wiki.Main()` is dead — vestigial.

**Decision.** Dissolve the dual spec into one inline `appkit.Spec` in
`cmd/wiki/main.go`.

- **One inline literal.** `main()` builds a single `appkit.Spec` literal — `App`,
  `Mount`, `Port: registry.MustPort("wiki")` (D51), `MCP: true`, `WWW: true`
  (D52), `ManifestExtras`, `Migrations: db.FS` — with the serve wiring in its
  `Handlers` closure (the current `buildSpec` body: config via
  `wiki.NewConfig(os.Getenv)` fail-loud **inside** the closure, the read handle,
  the LLM client + recorder, embedders, extractor/compiler, `wiki.NewService`,
  retrievers, asker, the web handler over `rt.WWW()` (D52), and the `appkit/mcp`
  handler (D53)) and its `Workers`. `appkit.Main(spec)` is called directly.
- **Dead code leaves `internal/wiki`.** `wiki.Spec()`, `wiki.Main()`, and the
  internal/wiki `serveCommand` are deleted; the `buildSpec` wrapper and the
  `spec := wiki.Spec(); if serveCommand {…}` two-phase `main()` are gone.
  `internal/wiki` keeps every domain export (`NewService`, the stores, `Config`,
  `Conns`, `Path`/`Mount`, …); only the three composition funcs leave it.
- **Config at serve, in the closure.** Config is read at serve time inside the
  `Handlers` closure (like notify), not through a `Spec.Config` hook — config
  genuinely arrives at serve time (the Anthropic/OpenAI keys, the embed model,
  the per-stage call sites), so serve-time wiring is correct; non-serve verbs
  never invoke `Handlers` and so never need config, preserving today's behavior.
- **Tests re-point.** The `main_test.go`/`wiki_test.go` tests that drove
  `wiki.Spec()`/`buildSpec` re-point at the inline spec (or a small serve-spec
  test helper); no behavioral assertion changes.

**The one forced coupling that remains, and why.** The inline literal declares
`var svc *wiki.Service` **before** it, assigned inside `Handlers` and read inside
`Workers` (the ingest worker: `worker.Run(ctx, svc)`). This is **not**
post-construction spec mutation — both closures are set inline — but it is a
genuine ordering dependency (Handlers must run before Workers, which appkit
guarantees: a `Handlers` error aborts serve before any worker starts). It is
**forced** by appkit's frozen `Workers []func(ctx context.Context) error`
signature: a Worker receives only a `ctx`, never a `*Router`, so it cannot
independently reach `rt.DB()` to build its own Service — unlike notify's
consumers, which get `rt` via `Consumer.Handler func(rt *appkit.Router)`. wiki's
ingest worker is a plain background Worker, not an event consumer, so no such
seam exists; capturing the `Handlers`-built `svc` is the only way to hand the
worker its Service. Removing the capture would require an appkit change (a
Router on Workers), which is frozen and out of scope.

**Rejected.**
- **Keeping the dual spec** — `wiki.Main()` is dead and `wiki.Spec()`'s
  `Handlers`/`Workers` are never-used stubs; carrying a second, degenerate
  composition root is exactly the drift this round removes, and it forces every
  other item (D52 web, D53 mcp) to edit two composition roots.
- **A `Spec.Config` hook feeding a `rt.Config()` accessor** — appkit does not
  consume `Spec.Config` today (its doc: "not yet consumed by the dispatcher");
  reading config inside `Handlers` matches the reference (notify) and needs no
  chassis change.
- **Removing the `var svc` capture** — impossible without a frozen-appkit change
  (`Workers` has no `Router`); it is the one forced deviation, recorded here, not
  hidden.

**Verification.** Structural — no `R-XXXX-XXXX` ids (the serve behavior is
already pinned by the D52/D53 ids and the existing domain/MCP/worker suites,
re-pointed at the inline spec). Realized by a plan phase whose done bar is
mechanical:

- `grep -n "func Spec\|func Main" wiki/internal/wiki/wiki.go` returns no matches;
- `grep -n "func buildSpec\|wiki\.Spec()\|wiki\.Main()" wiki/cmd/wiki/main.go`
  returns no matches, and `cmd/wiki/main.go` contains exactly one
  `appkit.Spec{` literal;
- the suite stays green (design *Conventions* commands) with no behavioral
  assertion changes.
