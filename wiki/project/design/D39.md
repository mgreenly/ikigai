# Decision 39 — The landing page: a session-gated human web surface (`GET /{$}`)

**Decision.** wiki gains its **first HTML web surface**: a single **landing
page** served at the bare mount root, `GET /{$}`, rendering the service **name**
and **version**, styled with the Carbon design system. This is a deliberate,
suite-wide introduction — every deployable app gains the same v1 landing page —
not a contradiction of the loopback-only posture. wiki keeps everything it
already is (the MCP tool surface, `/health`, the RFC 9728 PRM doc); the landing
page is **added beside** them, never in place of them. v1 is intentionally the
uniform name+version starting point; the page will diverge later to serve wiki's
own purpose, which is why wiki **owns its own landing code** (handler + template
+ embedded assets) rather than importing a shared handler.

**Two front doors, two audiences (the gate is the spine of this Decision).**
Under `/srv/wiki/` there are now two kinds of caller, each authenticated
differently and the difference is the whole design:

- **Agents / MCP clients → bearer token.** `POST /mcp` stays gated by
  `auth_request /_authn` (opaque bearer). **Unchanged** by this Decision — the
  existing prefix `location /srv/wiki/` fragment (D10's surface) keeps gating it.
- **Humans in a browser → login-session cookie.** The landing page is gated by
  `auth_request /_session-authn` — the dashboard-owned, loopback-only endpoint
  that validates the `dashboard_session` cookie against the web-session store.
  This is the **same** mechanism `sites` already uses for its private static
  tier. It is deliberately **coarse**: any logged-in dashboard user may view any
  app's landing page (no per-resource check). That is correct for a name+version
  page.

A browser cannot satisfy the bearer gate (it has no token), and the bearer gate
is the wrong audience for a human page; so the landing page **must** be
cookie-gated, not bearer-gated — and not fully public, since the suite exposes
nothing un-gated except the PRM bootstrap doc.

**The seam — Go side (`cmd/wiki/main.go` `Spec.Handlers` hook).** The landing
route is mounted **ungated in-process** — nginx is the sole gate, exactly as the
MCP and PRM routes already trust nginx:

```go
rt.HandleFunc("GET /{$}", landingHandler(rt.Service(), rt.Version()))
```

- `{$}` (Go 1.22+ ServeMux) matches **only** the exact root path `/`, so it
  cannot shadow `/mcp`, `/health`, `/feed`, or
  `/.well-known/oauth-protected-resource`. Those routes are untouched.
- It is **not** wrapped in `rt.RequireIdentity(...)` (that wrapper is the bearer
  gate, for MCP). The cookie gate lives entirely in nginx; in-process the route
  is open and trusts the front door — the established suite posture.
- `rt.Service()` returns `"wiki"` and `rt.Version()` returns the build version
  (`VERSION`, e.g. `0.6.0`); together they are the entire v1 page content. The
  handler renders an embedded `html/template`. No DB, no LLM, no identity header
  is read — the page is a pure function of name+version.

**The seam — assets (a new `wiki/internal/web/` package).** wiki embeds its
**own** copy of the Carbon assets and template, mirroring the dashboard
precedent (`dashboard/ui/`, a `ui` package exposing `//go:embed html static` as
an `embed.FS`, with `static/tokens.css` + the IBM Plex / Space Grotesk woff2
fonts):

- `wiki/internal/web/` holds `landing.tmpl` (or `.html`) and `static/tokens.css`
  + the woff2 font files, copied from the repo-root Carbon source
  (`design/tokens.css` + fonts; rules in `design/carbon.md`, reference
  `design/example.html`). They are embedded via `//go:embed` so the binary stays
  one static file — no on-disk asset dependency at runtime.
- The handler serves the template at `GET /{$}` and the embedded static assets
  under a sibling static route (e.g. `GET /static/...`) that the page references
  for `tokens.css` and the fonts. The static route, like the page, is
  ungated-in-process and session-gated by the same nginx exact-match root —
  acceptable because the page and its CSS/fonts are one coherent human surface.
- Visual form (Carbon): a single centered card — service **name** in display
  type (Space Grotesk), **version** as a mono label (IBM Plex Mono), monochrome
  neutrals, blue `#2563EB` the only signal color, 4px spacing grid. Minimal and
  on-system; no JS required.

**The seam — nginx (`wiki/etc/nginx.conf`).** Add an **exact-match** location for
the bare mount root that session-gates and proxies to the app's `GET /{$}`,
beside the existing gated prefix:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (sibling of
# /_authn). Exact-match `=` so it never shadows the gated prefix below or the MCP
# endpoint. A failed session check yields 401.
location = /srv/wiki/ {
    auth_request /_session-authn;
    auth_request_set $wiki_owner $upstream_http_x_owner_email;
    proxy_set_header X-Owner-Email $wiki_owner;
    proxy_pass http://127.0.0.1:__PORT__/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

The existing `location /srv/wiki/ { auth_request /_authn; ... }` **prefix** stays
for everything else (the MCP endpoint, `/health`, `/feed`); the exact-match `=`
root wins for the bare root only (and the existing exact-match PRM location is
unaffected — exact matches over distinct paths coexist).

**Stale-doc consequence.** wiki's `AGENTS.md`/`CLAUDE.md` declare "**no UI** and
no token logic." The "no UI" half is now **false** — wiki serves a human landing
page. Per the suite doctrine (docs state current truth; purge what the change
falsified, rewrite in place), that line must be corrected. It is **not** edited
here — it is a dedicated **plan phase** (Phase 65) so code and docs evolve
together through the build loop. (The "no token logic" half stays true: the
landing page reads no token and runs no token logic — nginx is still the trust
boundary.)

**Rejected.**
- **A shared landing handler in appkit.** Each app's landing page will diverge to
  serve that app's purpose; v1 uniformity is a starting point, not a contract.
  wiki owns its handler + template + assets so divergence is a local edit, not a
  shared-library change. (The Carbon *tokens* are copied, not shared code.)
- **Bearer-gating the landing page (`auth_request /_authn`).** A browser carries
  no opaque bearer token, so the human page would be permanently unreachable.
  The audience is a logged-in human; the cookie session is the right credential.
- **Serving the landing page fully public (no auth).** The suite exposes nothing
  un-gated except the PRM bootstrap doc; a name+version page is innocuous but
  still inside the tenant boundary. Coarse session-gating matches `sites`' private
  tier and keeps one consistent rule.
- **Wrapping `GET /{$}` in `RequireIdentity` in-process.** That is the bearer
  gate; it would reject browsers and duplicate the trust boundary nginx owns.
  The route is ungated in-process by design — nginx is the sole gate, as for
  `/mcp` and PRM.
- **A catch-all `GET /` instead of `GET /{$}`.** A bare `/` prefix would shadow
  `/mcp`, `/health`, `/feed`, and PRM. `{$}` matches the exact root only, so the
  landing route is collision-free by construction.
- **Reusing the dashboard's `ui` package.** wiki is a separate Go module; it
  embeds its own copy of the Carbon assets (the dashboard precedent is the
  *pattern* to copy, not a dependency to import).

**Verification.**
- R-LAND-PG01 — `GET /{$}` (exact root `/`) returns **200** with
  `Content-Type: text/html; charset=utf-8` — driven through the handler the
  composition root mounts (httptest), no DB/LLM/identity required.
- R-LAND-NMVR — the rendered body contains the **service name** (`rt.Service()`,
  `"wiki"`) and the **version** (`rt.Version()`); built with a non-default
  injected version, the page shows that exact version (proving it renders
  `Version()`, not a hard-coded literal).
- R-LAND-CARB — the page is Carbon-styled from **embedded** assets: the
  `wiki/internal/web` `embed.FS` contains `tokens.css` and the woff2 font(s), the
  served page references `tokens.css`, and a request for the embedded
  `tokens.css` asset returns 200 with a CSS content-type — proving the assets
  ship in-binary, not from disk.
- R-LAND-ROOT — `{$}` matches **only** the exact root: a request to a non-root
  path the landing handler does not own (e.g. `/health`, `/mcp`, `/nope`) is
  **not** served the landing page by this route — the landing mount does not
  shadow the existing routes.
- R-LAND-UNGT — the landing route is mounted **ungated in-process** (not wrapped
  in `RequireIdentity`): an httptest request with **no** identity header / no
  bearer still gets the 200 page, proving the in-process route trusts nginx as
  the sole gate (the cookie gate is asserted in nginx, Phase 64).
