# Decision 24 — An overridable extract prompt for the eval (default = the baked-in production prompt)

**Decision.** `eval-extract` gains a way to run extract under an **alternative instruction prompt** supplied at run time, so a developer can A/B a candidate prompt against the gold dataset. The override is **eval-only and opt-in**: with no flag the eval runs the **exact baked-in production prompt** (identity with the running service, D20/D22); with the flag it runs the supplied instructions. Production wiring is unchanged — the service always uses the default. This narrows the prior product exclusion on A/B-testing prompts by one capability, for `extract` only; other call sites' prompts remain non-overridable this release.

## The seam — an optional instruction override on the Extractor

`renderPrompt` builds the extract prompt; its leading **instruction preamble** (the JSON-shape contract + extraction guidance) is the part worth varying, while the **document header and source text** are data the harness must always inject. So the override replaces the *instructions* only; the header + source text are always appended by the harness exactly as production does — an alternative prompt cannot drop the document.

The current preamble is lifted into an exported constant as a copy-start:

```go
// DefaultPromptInstructions is the baked-in production extract instruction preamble
// (the JSON-shape contract + extraction guidance). renderPrompt appends the document
// header and source text after it. Exported so an eval author can copy and adapt it.
const DefaultPromptInstructions = `Extract subjects and claims from the source text.
Return JSON with shape {"subjects":[...]}.
...`

// Option configures an Extractor at construction. Production passes none.
type Option func(*Extractor)

// WithPromptInstructions replaces the instruction preamble for this Extractor.
// Unset → DefaultPromptInstructions (the production prompt). The header + source text
// are still appended by renderPrompt; the supplied instructions own the JSON-shape contract.
func WithPromptInstructions(instructions string) Option

// New builds an Extractor; variadic opts keep every existing call (production + eval) valid.
func New(c *llm.Client, site llm.CallSite, opts ...Option) *Extractor
```

`Extractor` gains an unexported `instructions string`, defaulted to `DefaultPromptInstructions`; `renderPrompt` emits `e.instructions` then the header + source-text blocks. `cmd/wiki` calls `extract.New(client, site)` unchanged → production behavior is byte-identical to today.

## The CLI flag and the run stamp

`eval-extract` gains `-prompt-file <path>`:
- **Absent** → no option; the eval runs `DefaultPromptInstructions` (production identity).
- **Present** → read the file; an unreadable path or **empty/blank** content is a **startup error** (non-zero exit, before any dataset load or LLM call) — same fail-loud rule as a missing API key (D22). The contents become `extract.WithPromptInstructions(...)`.

The `Scorecard` stamp (D22) gains a `Prompt` field so a baseline and a candidate run are distinguishable:

```go
type Scorecard struct {
    Dataset string
    Model   string
    Prompt  string  // "default" or the -prompt-file path — which instruction prompt produced this run
    Config  string
    Judge   string
    Cases   []eval.CaseResult
    Totals  eval.Totals
}
```

`Prompt` is `"default"` with no flag, else the `-prompt-file` path. `WriteHuman` prints it in the header block alongside `model`/`config`/`judge`; `WriteJSON` carries it and round-trips. The exact prompt *text* of every call stays traceable through the D22 `-record` JSONL recorder, so the stamp only *labels* the run.

**Known limitation (deferred).** A candidate prompt that drifts the JSON-shape contract makes a case's extract response fail validation; `RunDataset` surfaces that as a run error (fail-loud) rather than scoring the case 0 and continuing. Graceful per-case capture is left to later work.

**Rejected.**
- **Replacing the *entire* prompt string (header + source text) via placeholder templating.** Lets the author forget the `{{.Text}}` placeholder and silently run extract over no document. Appending header + source text in the harness keeps data injection guaranteed. Rejected.
- **Making the prompt a per-site config field like model/reasoning (D19), resolved in production too.** A multi-line prompt as an env/manifest knob is awkward, and production has no need to vary its prompt this release. Keep production on the default; expose the override only on the eval seam. Rejected (revisitable).
- **Augmenting (appending extra guidance to) the baked-in instructions rather than replacing them.** Can't test a genuinely different prompt — only additions. Replace, with the exported default as the copy-start. Rejected.
- **No run stamp for the prompt.** Two runs differing only by prompt would be indistinguishable in their scorecards. Stamp it. Rejected.

**Verification.**
- R-ODAP-34N6 — With no override (production path / no `-prompt-file`), an `Extractor` built by `extract.New(client, site)` renders `DefaultPromptInstructions`: a capturing mock provider confirms the request carries the baked-in production instruction preamble (and the case's header + source text), proving the default is byte-identical to production.
- R-OEIL-GWDV — `extract.New(client, site, extract.WithPromptInstructions(alt))` makes `Extract` render `alt` **in place of** the default while still appending the document header and source text verbatim: the captured request contains the override text and the case's source text and does **not** contain the default preamble.
- R-OFQH-UO4K — `eval-extract -prompt-file <path>` fails loudly at startup (non-zero exit, before any dataset load or LLM call) when the path is unreadable or its contents are empty/blank, matching the missing-key startup rule; with no `-prompt-file` it runs the default with no error.
- R-OGYE-8FV9 — The `Scorecard` carries `Prompt` stamping which instruction prompt produced the run (`"default"` with no flag, else the `-prompt-file` path); `WriteHuman` emits it in the header block and `WriteJSON` carries it and round-trips into an equal `Scorecard`.
