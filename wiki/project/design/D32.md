# Decision 32 — The meaning lane: in-memory vector search

**Decision.** Turn the question into a vector, compare it against every page's vector held in memory, and return the closest pages. This is the "same idea, different words" lane, wrapped as a `vectorRetriever` behind the search seam (D08).

**An in-memory cache of page vectors.** All page vectors are kept in a small in-memory structure, filled once at startup from `EmbeddingStore.LoadAll` (D30) and kept current afterward as pages are re-embedded (the background updater in D35 calls `Upsert` on it). Searching is then a plain scan over memory — no per-query disk reads.

```go
// vectorCache holds every page's vector in RAM, guarded for concurrent use:
// many searches read it while the background updater occasionally writes one entry.
type vectorCache struct {
    mu      sync.RWMutex
    entries []vectorEntry   // one per page that has a vector
}
type vectorEntry struct {
    SubjectID string
    Title     string
    Vec       []float32     // L2-normalized
}
func (c *vectorCache) Replace(all []vectorEntry)                 // startup hydrate
func (c *vectorCache) Upsert(e vectorEntry)                      // after a (re-)embed
func (c *vectorCache) nearest(q []float32, k int) []retrieve.Hit // top-k by cosine
```

**Scoring is a dot product.** Both the page vectors and the query vector come back L2-normalized from the embedder (D30, D34), so cosine similarity — the standard "how close in meaning" measure — is just the dot product of the two vectors. `nearest` computes that against every cached entry, keeps the top `k`, and returns them as `Hit`s (`PageID` = subject id, `Score` = the cosine value, plus title). A few-millisecond scan at our scale; no approximate-nearest-neighbor index (D30 rejected the extension that would need one).

**The retriever.**

```go
// vectorRetriever embeds the query, then scans the cache. Implements retrieve.Retriever (D08).
type vectorRetriever struct {
    embed func(ctx context.Context, text string) ([]float32, error)  // query-side embed (D34)
    cache *vectorCache
}
func (r *vectorRetriever) Search(ctx context.Context, query string, k int) ([]retrieve.Hit, error)
```

**The question is embedded with the *query* role, not the document role.** The embedding model (`text-embedding-3`) is trained to encode a question slightly differently from a stored document; using the query role on the question side and the document role on the page side is how the two line up. The lane passes the question through the embedder's query input mode, while pages were embedded in document mode (D34).

**Rejected.**
- **Reading vectors from disk per query.** At 100k pages that's 100k blob reads every question — far slower than a single in-RAM scan, for no benefit at this scale. Memory (~200 MB at the ceiling) is cheap; load once.
- **An approximate-nearest-neighbor index (HNSW etc.).** Earns its complexity past ~500k–1M vectors or high query rates — neither applies. A brute-force scan is simpler and fast enough here.
- **Embedding the question with the document role (same mode as pages).** Throws away the query/document asymmetry the model was trained for, costing recall for nothing.

**Verification.**
- R-3WOB-6U4Q — Given a cache of page vectors and a query vector, `Search` returns the pages ordered by descending cosine (nearest in meaning first), limited to `k` — the page whose vector is closest to the query ranks first, a near-orthogonal page ranks last or is excluded.
- R-3XW7-KLVF — `Search` embeds the question with the **query** input role (not the document role), confirmed via a capturing fake embedder — the page side uses the document role (D34), so the two sides are encoded for each other.
- R-3Z43-YDM4 — The cache hydrates from `EmbeddingStore.LoadAll` at startup, and after a re-embed `Upsert`s the new vector so a subsequent `Search` ranks by the updated vector — the in-memory copy stays consistent with the store.
- R-40C0-C5CT — Concurrent `Search` reads while `Upsert` writes the cache run race-free (exercised under the race detector with concurrent goroutines) — the lock discipline holds.
