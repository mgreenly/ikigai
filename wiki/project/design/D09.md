# Decision 9 — `ask` (`internal/ask`): hybrid-retrieval pipeline, grounded/cited/honest-empty

**Decision.** `ask` prepares the question, searches the pages with the hybrid retriever, and synthesizes a cited answer from the pages it finds. Control flow is a **deterministic three-stage pipeline** in Go: analyze → retrieve → synthesize. The LLM is invoked twice as structured generations through `llm.JSON[T]` (D5) — once to analyze the question (D36), once to synthesize — plus the query-embedding call inside the meaning lane (D34). **No exact-name resolution, no tool-loop, no source/claims drill-down**: `ask` answers from the compiled pages only, found by search. Strictly read-only, permanently.

```go
package ask

type Answer struct {
    Found     bool       // false → the wiki holds nothing relevant to the question
    Text      string     // synthesized answer, or the fixed honest-empty sentence
    Citations []Citation // the pages the answer drew on
}
// Citation identifies a cited page by its public type/slug Path (D11), never the internal id.
type Citation struct { Path, Title string }

type Asker struct {
    search     Retriever           // hybrid SearchAnalyzed (D33)
    pages      *wiki.PageStore     // fetch page bodies for the synthesis context
    c          *llm.Client
    analyzeSite llm.CallSite        // stage 1: question analysis ("ask-subject", D36)
    synthSite   llm.CallSite        // stage 3: grounded synthesis ("ask-synthesis")
    floor       float64             // cosine relevance floor (honest-empty gate); config knob, default 0.30
    finalK      int                 // pages retrieved for synthesis (D33 FinalK, default 8)
}
func (a *Asker) Ask(ctx context.Context, owner, question string) (Answer, error)
```

**Flow — analyze → retrieve → synthesize:**

1. **Analyze (one LLM call, D36).** `Analyze` turns the question into a `QueryAnalysis{SubQueries, Keywords, Aliases}`.
2. **Retrieve (deterministic + one query-embed inside the lane).** `search.SearchAnalyzed(ctx, qa, finalK)` returns a `retrieve.Result{Hits, TopDense, Pinned}` — the fused top pages, the best raw meaning-match score, and whether an exact-name page was pinned (D33).
   - **Honest-empty gate:** if **no** exact-name pin fired **and** `TopDense` is below `floor`, return `Answer{Found:false, Text:<fixed sentence>, Citations:nil}` and make **no synthesis call**. Fuzzy search always returns *some* top page, so this relevance bar — judged on the raw cosine, not the fused rank score — is what restores an honest "nothing here," and skipping the synthesis call is the deterministic anti-fabrication lever (the model never runs against irrelevant pages). An exact-name pin always clears the gate (a verbatim-named subject is relevant by definition).
3. **Synthesize (one LLM call).** Fetch the hit pages' bodies and run `llm.JSON[answerResult]` over **those page bodies only**.
   ```go
   type answerResult struct {
       Found     bool       `json:"found"`
       Text      string     `json:"text"`
       Citations []Citation `json:"citations"`
   }
   ```

**Grounding & anti-fabrication (pages-only) — unchanged guarantees:**
- A `Found:true` answer **must carry ≥1 citation**; a found-with-no-citation reply is **downgraded to honest-empty**.
- Each citation is **validated against the retrieved set**; a citation to a page not retrieved this turn is **dropped**. Surviving citations come from the page store, so they are real pages, and each is mapped from subject id to the public `type/slug` **Path** (D11) before returning — never an internal id.
- The synthesis prompt is fed **only the retrieved page bodies** — never raw claims or original source text. No drill-into-source path exists in `ask`.

**`ask` is strictly read-only — permanently.** `Asker` only reads (search + `GetBySubject`); no write path exists on any branch, including honest-empty and parse-failure.

**The relevance floor is a config knob.** Default **0.30** — deliberately conservative; tuned later against real in/out-of-corpus questions. A `_RELEVANCE_FLOOR` manifest knob (fail-loud parse, like the D19 knobs) moves it without a recompile. Raising it above a page's cosine forces honest-empty; lowering it admits the page.

**Prompts.**
- analyze (`ask-subject`) — owned by D36.
- `synthSite.System` — answer **only** from supplied pages; cite each page drawn on by subject id + title (`ask` maps id to public path after validation); return `found:false` if the pages don't answer; never fabricate a fact, page, or citation; surface contradictions with both sides; read-only. JSON output schema + example.

**Call sites** (defaults, configurable per D19): `analyzeSite` = `ask-subject`; `synthSite` = `ask-synthesis`. Both record to `llm_calls` with `JobID==""` (no job). The query-embedding call records as `embed-query` (D34).

**This rewrites the prior D9.** The old pipeline extracted bare subject *names* and resolved them by **exact normalized name**, gathering one page per resolved name and returning honest-empty when no name resolved. That exact-name machinery is replaced by hybrid retrieval: stage 1 is now question analysis (D36) and stage 2 is search (D33), so the four exact-name behaviors retire (their ids and tests go) and are replaced by the retrieval-gather and cosine-floor behaviors below. The integrity guarantees — pages-only, every answer cited, honest-empty, read-only — are unchanged and keep their ids.

**Rejected.**
- **Keeping exact-name resolution as the primary path.** That is the limitation this upgrade removes — a question that doesn't name a subject verbatim found nothing. The exact-name case survives only as the rank-1 pin inside D33.
- **Honest-empty on the fused rank score instead of raw cosine.** The fused score is uncalibrated (rank-based), so a fixed threshold on it is meaningless; the raw cosine is a real similarity the floor can be set against.
- **Agentic tool-loop (model drives retrieve/read).** Makes the honest-empty gate fuzzy and stages hard to pin with checkable ids; the deterministic pipeline keeps the gate in Go.
- **`read_source` / raw-claims drill-down.** Pages-only is the product decision.
