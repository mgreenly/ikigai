# Decision 27 — Merge MCP surface (`merge` + `merges`)

**Decision.** Two new MCP verbs expose merge on the product surface, cloning the D10/D16 idiom — the surface grows from **fifteen** verbs (D16's thirteen plus these two). `merge` is a write that **fires and returns** a queued handle (the fold runs on the worker off the request path, like `ingest`); `merges` is a paginated **audit** read of completed folds. The operation lifecycle (queued → working → done/failed) is **not** a third verb — it is reachable via D16's `jobs`/`jobs_count`, which gain an optional `kind` filter (owned by D16).

| tool | input | → result |
|---|---|---|
| `merge` *(new, WRITE, fire-and-return)* | `{winner (req), loser (req)}` — both `type/slug` paths | `{job_id, status:"queued"}` |
| `merges` *(new, READ, paginated)* | `{limit?, cursor?}` | `{items:[{merged_at, winner, loser_name, alias_norm_name, created_by}], next_cursor}` — newest-first |

Both wire through the standard idiom: a func field on `Handler`, a `WithMergeService`/`WithMergesService` Option, a `handleToolCall` case, a `handleMergeCall`/`handleMergesCall`, a conditional `tools()` append, and a hand-coded `inputSchema` (`objectSchema({winner,loser},[winner,loser])` / `listSchema({})`). The route stays behind `RequireIdentity`. Every subject reference crosses as a `type/slug` path — never an internal ULID.

## `merge`

```go
// handleMergeCall: resolve both paths to ids ONCE, here, then enqueue a kind='merge' job (D26).
func (h *Handler) handleMergeCall(ctx, args) (toolResult, error)
```

- **Resolve at request time via `GetByPath`** (D11) — subjects-only, exact. An already-folded name is an *alias*, not a live subject, so `GetByPath` returns `ErrSubjectNotFound`; `merge` does **not** follow the alias to the survivor. Both `winner` and `loser` are explicit owner input — no auto-pick.
- **Enqueue, don't execute.** Insert a `kind='merge'` job (D26) carrying resolved `merge_winner_id`/`merge_loser_id`, `status='pending'`, `owner` from `appkit.IdentityFrom(ctx)`; nudge the worker; return `{job_id, status:"queued"}`. The result is **never** `status:"merged"` — completion is observed via the job reaching `done` (`status`) and the fold appearing in `merges`.
- **Two-channel errors** (D10 convention):
  - a `winner`/`loser` path matching **no** subject → clean **not-found**; no job enqueued. (The path token is the `UNIQUE` `norm_name` (D11), so resolution is unambiguous — no ambiguous-path channel.)
  - `winner` and `loser` resolving to the **same id** → **tool error**.
- **Identity-gated write.** `owner` (→ `aliases.created_by`, D25) comes from `X-Owner-Email`; reachable only behind `RequireIdentity`.

## `merges`

Keyset-paginated audit over the `aliases` table (D25), newest-first, reusing the D15 cursor contract:

```go
func (a *AliasStore) ListMerges(ctx, p page.Params) (items []Alias, next string, err error) // net-new read
```

- Orders `created_at DESC, norm_name` (`norm_name` UNIQUE — stable keyset tiebreak); the cursor encodes that pair. The merge log is far smaller than the subject set, so the ordering is unindexed at v1 (an index is a noted later optimization).
- Each item renders the **winner as a `type/slug` path** by resolving `subject_id → Path` (D11); the D25 FK (`ON DELETE RESTRICT`) guarantees `subject_id` names a **live** subject, so the path never dangles (a later merge folding the winner repoints the row to the terminal survivor). Other fields come straight off the alias row: `loser_name` = `aliases.name`, `alias_norm_name` = `aliases.norm_name`, `merged_at` = `aliases.created_at`, `created_by`.
- Returns `pagedResult("merges", items, next_cursor)`. No internal id appears.

## Two surfaces, two tables (why `merges` is not a kind-filtered `jobs`)

- **`jobs`/`jobs_count` (D16)** read the `jobs` table — the operation lifecycle. They gain an optional `kind` filter (default `ingest`) so a merge job is listable as `kind=['merge']` but excluded from the default ingest view. *(Filter owned by D16.)*
- **`merges` (here)** reads the `aliases` table — the durable record of *what folded into what*. It cannot be served from `jobs`: a successful merge **deletes the loser subject**, and the merge job row stores only `merge_loser_id` (a now-dead id), never the loser's *name*. The loser's display name survives **only** in `aliases.name` (the forward-routing record). Serving the audit from `jobs` would mean denormalizing the loser name for nothing.

**Rejected.**
- **Folding `merges` into a kind-filtered `jobs` verb.** After the loser subject is deleted, the jobs row cannot render "what folded into what" (no loser name, dead loser id). Audit data lives in `aliases`; lifecycle data in `jobs`.
- **Resolving `winner`/`loser` via `ResolveByName` (alias-aware).** An owner naming an already-folded name should get a clean not-found, not a silent redirect that could produce a confusing self-merge or wrong target. Merge inputs are explicit live subjects.
- **`merge` returning `status:"merged"` synchronously.** The fold runs on the worker off the request path (D26), like `ingest`. A completed status would block on a multi-second recompile or lie.
- **Surfacing `loser_type` / `moved_claim_count` in `merges`.** Out of scope: D25's `aliases` stores neither, and the promise is met by loser name, winner path, and timestamp.
- **An opaque `pair_id` handle (research §7).** Belongs to machine-proposed *candidates* (deferred). With the owner naming the pair directly, the handle is just its `job_id`.

**Verification.**
Driving the JSON-RPC handler in-process with a stubbed identity (D10 strategy), against the D26 service on a real temp SQLite + scripted mock compiler:

- R-DWDM-RVA7 — `tools/list` returns the **fifteen** verbs including `merge` and `merges`, each `inputSchema` well-formed; `merges` (no required field) **omits** `required` entirely — never `"required": null` (D10 R-N4KO strict-client rule).
- R-DYTF-JERL — `merge` with two valid, distinct `type/slug` paths resolves both via `GetByPath`, enqueues a `kind='merge'` job carrying their ids with `owner` from identity, and returns `{job_id, status:"queued"}` — **fire-and-return**: the loser subject still exists immediately after the call returns (fold not yet run), and the result is never `status:"merged"`.
- R-E01B-X6IA — `merge` surfaces both error channels: a `winner`/`loser` path matching no subject → clean **not-found** with **no job enqueued**; `winner` and `loser` resolving to the same id → **tool error**. Neither crashes. (No ambiguous-path channel: the path token is the `UNIQUE` `norm_name`, D11.)
- R-E198-AY8Z — `POST /mcp` is behind `RequireIdentity`, and a `merge` stamps the job's `owner` (→ `aliases.created_by`, D25) from `X-Owner-Email`.
- R-E2H4-OPZO — `merges` returns a `{items, next_cursor}` page, **newest-first**, each item carrying `merged_at`, the winner as a `type/slug` **path** (no internal ULID anywhere), `loser_name`, `alias_norm_name`, and `created_by`; the returned cursor fetches the following page.
- R-E3P1-2HQD — After a merge of `L` into `W` commits its alias (D26), `merges` surfaces that fold with `winner == Path(W)` and `loser_name == L` — proving the audit reads the live `aliases` table and renders the winner by its current path.
