# Decision 14 — Job lifecycle & control: `aborted`, abort, re-run, and atomic integrate

**Decision.** The owner/operator gains direct control over the ingest queue: a job can be **aborted** (pending or working) into a deliberate terminal state, and a terminal job can be **re-run** to reprocess from its original text with its claims **replaced**. Both require `integrate` to be **atomic** — D4 specifies a single all-or-nothing commit (R-MB7F) but the current code writes claims and pages incrementally with no enclosing transaction. This decision rewrites `integrate` into D4's intended shape and makes that commit *idempotent*, which is what makes abort and re-run race-clean. The realigned integrate **supersedes** D4's incremental-write implementation; D4's worker/commit prose now points here.

## (a) `aborted` — a fifth lifecycle state

`internal/wiki` adds `JobAborted = "aborted"` beside `JobPending/JobWorking/JobDone/JobFailed`. It is **terminal** and **distinct from `failed`** (a failure broke; an abort was chosen). The live `jobs.status` carries **no `CHECK` constraint**, so no migration is needed; the closed five-state set is a product constant honored in Go.

## (b) Atomic, idempotent `integrate`

Holds **no DB transaction across the LLM calls** (the D4 intent) and is idempotent so re-run needs no special-case claim surgery:

1. *(reads, no tx)* Read this job's existing claims → `oldSubjects` (subjects this job previously touched). Empty on a first run.
2. *(no tx)* `extract(source_text)` → new subjects + claims; resolve existing / mint new subject ids in memory. `affected = oldSubjects ∪ newSubjects`. For each affected subject, compute its **post-replacement** claim set in memory — `(its persisted claims − this job's old claims) + this job's new claims` — and `compile` its page. (Every LLM call here is recorded eagerly per D13.)
3. *(one commit tx, conditional)*:
   - `DELETE FROM claims WHERE job_id = ?` (this job's prior claims — none on a first run);
   - upsert the affected subjects (first-writer-wins `type`, D3);
   - insert this job's new claims;
   - for each affected subject: upsert its recompiled page, **or delete its page when its final claim set is empty** (the subject row stays — it becomes a `has_page:false` subject; a known minor residue);
   - **`UPDATE jobs SET status='done', finished_at=now WHERE id=? AND status='working'`**.
   If the job-update touches **0 rows** (an abort flipped it to `aborted` concurrently) → **rollback**, so nothing lands. On any step-2 error → `UPDATE … status='failed', error=… WHERE id=? AND status='working'` (same conditional guard), writing no subjects/claims/pages.

Because the terminal-status write lives **inside** the commit tx and is **conditional on `status='working'`**, the abort-vs-finish race resolves cleanly in every interleaving: whoever sets the terminal state first wins; the loser's guarded write is a no-op. A job aborted mid-call commits **zero** subjects/claims/pages. The eager `llm_calls` rows (D13) are written outside this tx and survive a rollback — intended.

## (c) Abort — `Service.Abort(ctx, jobID) (AbortResult, error)`

The worker registers a per-job cancel function in a mutex-guarded `map[string]context.CancelFunc` on the `Service` (shared by the worker goroutine and the MCP handlers): `ProcessNext` does `jobCtx, cancel := context.WithCancel(ctx)`, registers it under `job.ID`, runs `integrate(jobCtx)`, and deregisters in a `defer`.

```go
type AbortResult struct {
    Aborted bool   // true iff this call moved the job to 'aborted'
    Status  string // the job's resulting (or unchanged) status
}
```

- **pending** → `UPDATE … status='aborted', finished_at=now WHERE id=? AND status='pending'`; rows>0 → `{Aborted:true}`, the worker never claims it.
- **working** → `UPDATE … status='aborted', finished_at=now WHERE id=? AND status='working'` (rows>0 = we own the transition), then call the registered `cancel()`. The in-flight LLM call returns `context.Canceled`; integrate's conditional commit is a no-op → no partial writes. The cancelled call still produces a record (D13 detached write, `Err: "context canceled"`).
- **terminal / unknown** → `{Aborted:false, Status:<current>}` — a clean "not abortable" result, status unchanged, never a crash.

## (d) Re-run — `Service.Rerun(ctx, jobID) (RerunResult, error)`

Idempotent integrate makes re-run a pure requeue:

- `UPDATE jobs SET status='pending', started_at='', finished_at='', error='' WHERE id=? AND status IN ('done','failed','aborted')`; rows>0 → requeued; nudge the worker. The worker re-integrates from the untouched `source_text`; step (b) **replaces** this job's claims and recompiles the `old ∪ new` affected pages.
- rows==0 → read the row: `pending`/`working` → clean error "can't re-run a job in progress" (status unchanged); missing → not-found. Never partially mutates.

`source_text` is never rewritten by a re-run; only the job's *derived* claims and the affected pages change.

## (e) Boot sweep

Unchanged in intent (`UPDATE jobs SET status='pending' WHERE status='working'` on worker start), now genuinely safe: atomic integrate guarantees an orphaned job wrote nothing, so its requeued re-run starts clean.

**Rejected.**
- **Claim-deletion logic inside `Rerun` itself** (delete the job's claims, then requeue) — splits the replacement across two writers and two transactions, reopening the atomicity hole. Folding deletion into integrate's single commit keeps re-run a one-line requeue and every run atomic. Rejected.
- **Accepting stale pages for subjects a re-run no longer touches** — leaves a page compiled as if deleted claims still existed; a half-correct footprint erodes trust. The `old ∪ new` recompile fixes it for a bounded cost. Rejected.
- **Abort writing the terminal status from the caller while the worker also writes it unconditionally** — both writers race and can clobber each other (aborted→failed). The conditional `WHERE status='working'` guard inside the commit makes the race well-defined. Rejected.
- **Cancelling the whole worker context to stop a job** — kills every job and the worker loop, not the one job. A per-job child context is required. Rejected.
- **A DB `aborting` intermediate state** — unnecessary; the conditional-commit guard already serializes the transition. Rejected.

**Verification.**
- R-0SCX-95OZ — Aborting a `pending` job moves it to `aborted` and the worker never integrates it (no subjects/claims/pages, no LLM call for that job).
- R-0TKT-MXFO — Aborting a `working` job cancels its in-flight LLM call, the job ends `aborted`, and **zero** subjects/claims/pages are committed (atomic rollback via the conditional commit).
- R-0USQ-0P6D — `aborted` is a terminal state distinct from `failed`: an aborted job reports `aborted` (not `failed`) and carries no integration `error`.
- R-0W0M-EGX2 — Aborting a job that is already terminal (`done`/`failed`/`aborted`) or unknown returns a clean not-abortable result with the status unchanged, never an error or crash.
- R-0X8I-S8NR — Re-running a terminal job (`done`/`failed`/`aborted`) returns it to `pending`, and the worker reprocesses it from the unchanged `source_text`.
- R-0YGF-60EG — After a re-run, the job's claims are **exactly** its new extraction's — the prior attempt's claims are gone and none are duplicated — and the affected subjects' pages are recompiled.
- R-0ZOB-JS55 — A re-run whose new extraction no longer names a previously-touched subject recompiles that subject's page from its reduced claim set, and deletes the page when the subject's final claim set is empty (orphan refresh; subject row retained).
- R-10W7-XJVU — Re-running a job that is still `pending` or `working` returns a clean error and leaves the job's status unchanged.
