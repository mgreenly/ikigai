# Decision 60 — the session-gated web locations opt into the apex `@login_bounce`: a logged-out human navigation goes to sign-in, not a bare 401

**Decision.** wiki's nginx fragment (`wiki/etc/nginx.conf`) opts its **three
browser-facing, session-gated locations** — the exact-match web root
`= /srv/wiki/`, the subject-page tier `/srv/wiki/subject/`, and the asset tier
`/srv/wiki/static/` (the whole human surface D47 session-gated) — into the
dashboard-owned apex `@login_bounce` primitive by adding **one line** —
`error_page 401 = @login_bounce;` — to each. Today a logged-out browser
navigating to any of them gets the bare nginx **401** the `auth_request
/_session-authn` deny produces; with the opt-in, the apex `@login_bounce` (a
dashboard-owned external contract, defined in the apex `server` block exactly
where `/_session-authn` is — D20 in the dashboard's `project/`) turns a
logged-out **navigation** into a `302` to `/login?return_to=…` and returns the
person to the page after they sign in, while a scripted `fetch`/XHR still gets
the clean `401` it can handle. `@login_bounce` makes that navigate-vs-fetch
distinction itself (off `Sec-Fetch-Mode`); wiki only has to reference it. sites
made this exact move first (its D24); this Decision applies the same
one-line-per-location opt-in to wiki's fragment.

**What deliberately does NOT get the line.** The bearer prefix `location
/srv/wiki/` (`auth_request /_authn`, fronting `/srv/wiki/mcp`, `/health`,
`/feed`) must keep returning `401` with its `WWW-Authenticate` challenge so MCP
clients can discover the AS; the PRM bootstrap and the 429 re-emit named
location are not browser session gates. The change is **purely additive** — one
line per session-gated location, every existing directive retained, nothing
rewritten; wiki referenced apex-owned `/_session-authn` before and now also
references apex-owned `@login_bounce`.

**Rejected.**
- **Adding the line to the bearer prefix** — the bearer 401 is a protocol signal
  (the `WWW-Authenticate` challenge); bouncing it to an HTML login page breaks
  MCP client discovery. The opt-in is browser-only.
- **A wiki-local bounce (own named location / `if`)** — the bounce is a
  suite-wide contract owned once by the apex; duplicating it per fragment is the
  copy-paste `@login_bounce` exists to avoid.
- **Bouncing all requests regardless of `Sec-Fetch-Mode`** — a logged-out
  browser-side `fetch` would receive login HTML with a `200` instead of a `401`
  it can handle; the navigate-vs-fetch split is deliberate and inherited free.
- **Minting R-ids and a Go content-assertion test** — wiki's settled convention
  (the doc's test-strategy spine; D39/D47 precedent) is that the nginx fragment
  is config proven by a **named fragment check** in the realizing phase, not an
  `R-id` test; this Decision keeps that convention.

**Verification.** Structural/config decision — **no R-ids** (the D39/D47
convention: nginx config is proven by a named fragment-check phase, not a Go
test). The realizing phase's named check asserts: each of `= /srv/wiki/`,
`/srv/wiki/subject/`, and `/srv/wiki/static/` contains both `auth_request
/_session-authn;` and `error_page 401 = @login_bounce;`; the bearer prefix
`location /srv/wiki/` does **not** contain the line; and every pre-existing
location and directive is still present unchanged.
