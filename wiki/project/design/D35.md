# Decision 35 — Keeping page vectors current, in the background

**Decision.** Page embedding always happens **after** a page is safely committed, never inside the ingest transaction, and the meaning lane keeps using a page's existing vector until a fresh one replaces it. Two cooperating paths do the work, sharing one helper:

1. **Right after a job commits** — the ingest/merge worker (D04/D26), once its atomic commit succeeds, embeds the pages that job just wrote, **attributed to that job**.
2. **A separate catch-up sweep** — a background loop that finds any page whose vector is missing, stale, or made by an old model and embeds it. This covers first-ship backfill, transient-failure retries, and model changes.

Both call the same helper:

```go
// embedAndStore computes the current page fingerprint, embeds the body (document
// role, D34), and overwrites the stored + cached vector in place. Never run inside a tx.
func (s *Service) embedAndStore(ctx context.Context, p wiki.Page) error
```

**Why after commit, never inside the transaction.** Embedding is a network call to OpenAI. The ingest commit holds SQLite's single writer; a network call inside it would block every other write and stall reads. So the commit finishes first (page is correct and durable), and embedding runs afterward. If embedding then fails, the **job still completes** — the page is right, it just temporarily lacks a fresh vector, which the catch-up sweep will supply.

**Attribution.** The after-commit path runs with the job id on the context (D13's `WithJobID`), so its `embed-page` records tie to the ingest (or merge) job — the product's "page-embedding calls attach to their ingest job." The catch-up sweep is maintenance, not an ingest, so its embeds carry no job id.

**What "needs embedding" means (the catch-up query).** A page needs a (re-)embed when:
- it has **no** `page_embeddings` row (brand-new, or first-ship backfill), or
- its stored `content_hash` ≠ the fingerprint of its current title+body (the page was rewritten), or
- its stored `model`/`dims` ≠ the configured embed site (D34) (model changed).

A page whose vector is current on all three is **skipped** — no needless call, no needless cost.

**The old vector stays live until replaced.** When a page is rewritten, its existing `page_embeddings` row and its in-memory cache entry are **left in place** — still searchable by meaning on the slightly-stale vector — until the new vector is computed and **overwrites** it (in storage via `Upsert`, in memory via the cache `Upsert`). There is no window in which a previously-embedded page has no vector. The only vector-less pages are brand-new ones awaiting their first embed; those ride the keyword lane until it lands.

**Worker shape.** The catch-up sweep is its own `Spec.Workers` entry — a loop that drains the "needs embedding" set, then waits on a doorbell (nudged after each commit) with a poll-timeout fallback, exiting only on `ctx.Done()`. It is independent of the ingest worker, so embedding never slows ingest and a backlog drains on its own. On first boot every existing page is "missing," so the sweep backfills the whole corpus with no new command or migration.

**Rejected.**
- **Embedding inside the ingest transaction.** Holds the single writer across a network round-trip; breaks the "reads never block on ingest" promise. The whole point is to keep it out of the tx.
- **Folding the catch-up loop into the ingest worker.** Couples two unrelated cadences; a slow embed backlog would delay ingests. A separate loop keeps them independent.
- **Dropping a page from the meaning lane while its new vector is pending.** Makes a just-edited page briefly invisible to semantic search; keeping the old vector live until replaced is strictly better.
- **An explicit operator "re-embed everything" verb.** Lazy catch-up via the staleness query backfills automatically; a manual trigger is unneeded machinery (can be added later if ever wanted).

**Verification.**
- R-6XNX-FNXO — Embedding runs **outside** the commit: when the after-commit embed fails (embedder errors), the job still ends `done` with its subjects, claims, and page intact (the commit is not rolled back), and the page is left for catch-up — proven with a failing embedder against a real temp SQLite.
- R-6YVT-TFOD — After a job integrates successfully, its page is embedded and stored/cached, and an `embed-page` `CallRecord` carries that job's id — the after-commit embed is attributed to the ingest job (mock embedder + recorder, real SQLite).
- R-703Q-77F2 — The catch-up selection embeds exactly the pages that are missing a vector, whose `content_hash` no longer matches the page text, or whose `model`/`dims` differ from the configured site — and **skips** a page whose vector is current on all three (no redundant embed call).
- R-71BM-KZ5R — When a previously-embedded page is rewritten, its prior vector remains present and searchable (in store and cache) until the new vector overwrites it in place — at no point does that page have zero vectors.
- R-72JI-YQWG — First-boot backfill: given pre-existing pages with no `page_embeddings` rows, the catch-up worker embeds all of them with no new verb or migration, and reads/searches remain answerable throughout (keyword lane covers the not-yet-embedded pages).
- R-73RF-CIN5 — A page whose after-commit embed failed (transient error) is re-selected by a later catch-up sweep and successfully embedded — a failed embed is retried, not lost.
