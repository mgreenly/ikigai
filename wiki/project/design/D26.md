# Decision 26 — The merge work item & execution

**Decision.** A merge runs as a work item on the **same single worker goroutine** that runs `integrate`, executed as one all-or-nothing transaction. Reusing the one queue and one writer *is* the fix for the merge⇄integrate race: both kinds are claimed and run by the same `ProcessNext` loop, so a merge and an ingest never execute concurrently — an integrate's page upsert can't resurrect a subject a merge just deleted, and a merge can't delete a subject mid-integrate. The merge reuses the **existing 3-arg D7 `Compile`** — no new merge prompt. This Decision owns the queue extension, the dispatch, `mergeSubjects`, and the two net-new store methods; the alias table / `AliasStore` / `Resolver` are D25, the MCP `merge`/`merges` verbs D27.

## Queue extension (one timestamped migration: `bin/new-migration wiki merge_jobs`)

The `jobs` table carries merges via a `kind` discriminator plus two FK-free carrier columns, not a second queue — keeping one writer, one `ClaimPending` FIFO, one boot-recovery sweep, one status-guard idiom:

```sql
ALTER TABLE jobs ADD COLUMN kind            TEXT NOT NULL DEFAULT 'ingest'
                 CHECK (kind IN ('ingest','merge'));
ALTER TABLE jobs ADD COLUMN merge_winner_id TEXT NOT NULL DEFAULT '';  -- winner subjects.id; no FK
ALTER TABLE jobs ADD COLUMN merge_loser_id  TEXT NOT NULL DEFAULT '';  -- loser subjects.id; no FK
```

- Existing rows take `kind='ingest'`; the ingest front door (D4) sets it explicitly.
- A merge job reuses the `jobs` row shape with **empty `source_text`/`sha256`** and the two carrier columns populated. `integrate` reads `source_text` only for ingest jobs; `mergeSubjects` ignores it.
- The carriers get **no FK**: the merge *deletes* the loser, so a FK from `merge_loser_id → subjects.id` would block it. (Contrast D25's `aliases.subject_id`, which points at the surviving winner and carries `ON DELETE RESTRICT`.)
- The closed `kind` set is enforced by a column-level `CHECK` in `ADD COLUMN` (SQLite permits it on a brand-new column, no migration friction).

## Dispatch (the race fix, concretely)

`ClaimPending` already claims the oldest `pending` job FIFO, kind-agnostic — so merges and ingests interleave in submission order on the single worker goroutine. `ProcessNext` branches **after** claiming and registering the per-job cancel (D14):

```go
jobCtx, cancel := context.WithCancel(ctx)   // D14 abort plumbing, unchanged
// register cancel under job.ID …
switch job.Kind {
case "merge":
    err = svc.mergeSubjects(jobCtx, job)     // new
default:
    err = svc.integrate(jobCtx, job)         // existing
}
```

The kind-agnostic boot sweep (`UPDATE jobs SET status='pending' WHERE status='working'`, D4/D14) requeues an orphaned merge exactly as an ingest; atomicity makes the re-run clean.

## `mergeSubjects(ctx, job)` — three phases, mirroring `integrate`

**Phase A — resolve (no tx).** Resolve both subjects by **primary key** from the carriers: `SubjectStore.Get(job.MergeWinnerID)` and `Get(job.MergeLoserID)` (indexed PK — *not* `GetByPath`, an O(N) type-partition scan, D11). **Both rows are loaded and kept** — the winner for the recompile (Phase B), the loser because step C-6 needs `loser.Name` for the alias key `Normalize(loser.name)` (the current code existence-checks the loser then discards it, which is why the alias was never minted). Validate both exist and `winner.ID != loser.ID`. If a side no longer resolves (another merge folded it away between enqueue and execution), end the job `failed` with a clear reason and write nothing (stale-pair case).

**Phase B — recompile (no tx, LLM).** Read the winner's and loser's claims, union them in memory, and call `compiler.Compile(ctx, winner, combined)` — the existing 3-arg method (D7), running its own tighten/retry, capping the body at 12,000 runes. No new prompt. Claim ids are stable across a merge (it only repoints `claims.subject`), so any `[job-id]` citation stays valid. Compile outside the tx is the established idiom (D4/D14): an LLM call inside the write tx would serialize all writes behind a multi-second round trip.

**Phase C — one tx on the write handle, strict order.** Build **tx-bound stores** (`NewSubjectStore(tx)`, `NewClaimStore(tx)`, `NewPageStore(tx)`, `NewAliasStore(tx)` — never the read-pool `s.*` stores), then:

1. `claims.RepointSubject(loser → winner)` — relabel the loser's claims onto the winner *(net-new)*
2. `pages.Upsert(winner page)` — the recompiled page from Phase B *(exists)*
3. `pages.DeleteBySubject(loser)` — drop the loser's page *(exists)*
4. `aliases.RepointSubject(loser → winner)` — repoint the loser's **inbound** aliases onto the winner *(D25)*
5. `subjects.Delete(loser)` — remove the folded subject *(net-new)*
6. `aliases.Insert(Normalize(loser.name) → winner)` — the new forward-routing alias *(D25)*. **This step is the one the shipped code omits** — without it the merge leaves no redirect, so `ask`/re-ingest of the folded name (and the D12 alias-aware link projection) all silently fail. Mandatory.
7. `UPDATE jobs SET status='done', finished_at=now WHERE id=? AND status='working'` — the D14 conditional guard
8. `tx.Commit()` — any error in 1–7 → full rollback, nothing lands.

**Step 4 must precede step 5** (the load-bearing ordering): the D25 FK is `ON DELETE RESTRICT`, so deleting the loser while any alias still points at it fails — repointing first clears the block. The step-6 insert names the *winner* (live), so it follows the loser delete for clarity.

**Idempotent re-run.** The terminal-status write is inside the commit and guarded by `status='working'` (D14), so a merge orphaned by a crash rolled back entirely — the loser still exists — and the requeued re-run resolves it by id in Phase A and completes cleanly. A concurrent abort flipping the job to `aborted` makes the step-7 guard touch 0 rows → rollback, so an aborted merge commits nothing.

## Merges and the ingest queue view

A merge job has no subjects and no `source_text`, so it should not clutter the default ingest view. The `jobs`/`jobs_count` verbs (D16) gain an **optional `kind` filter** defaulting to `ingest` — the common call shows only ingests, `kind=['merge']` lists merge jobs, `kind=['ingest','merge']` shows both. The D15 `ListJobs`/`CountJobs` seam and its `JobFilter` gain a `Kind` field. **This filter is owned by D16** (a fresh Verification id there); D26 only introduces the `kind` column it filters on. The merge lifecycle is observable through the existing `jobs` verb (and direct `status` on the returned handle); the human "what folded into what" record is D27's `merges` audit, read from `aliases`.

**Rejected.**
- **A second, parallel merge queue / work-item type.** Rejected: duplicates the claim loop, boot sweep, cancel registry, status-guard idiom, and — fatally — a separate queue drained by a second goroutine *reintroduces* the merge⇄integrate race. One `jobs` table with a `kind` discriminator keeps a single writer and serialization point.
- **Carrying the pair as JSON in `source_text`.** Rejected for explicit `merge_winner_id`/`merge_loser_id` columns: queryable, typed, indexable, and not overloading a permanent ingest field.
- **Resolving winner/loser through `GetByPath` in Phase A (at job-run time).** Rejected: resolution must happen **once, at request time** (D27), not at worker run — by then a concurrent merge may have folded the loser away, so re-resolving its path would fail with `ErrSubjectNotFound`. The carriers capture the ids the owner named; the worker reads them with a PK `Get` (`O(log n)`).
- **A new merge-specific compile prompt.** Rejected: the winner's page is just a full recompile from its combined claim set — exactly D7's 3-arg `Compile`. A second prompt is redundant surface.
- **Holding the tx across the recompile.** Rejected (D4/D14): an LLM call inside the write tx serializes every writer behind the network. Compile runs in Phase B, before `BeginTx`.
- **A FK on `merge_loser_id`.** Rejected: the merge deletes the loser, so a referential FK would block the delete. The carriers are inert handles.

Tested on the worker seam against a **real temp SQLite** (migrated by the appkit runner, `foreign_keys=ON`) with a scripted **mock compiler** — no live LLM:

- R-NEFH-U8IO — The migration adds `kind` defaulting to `'ingest'` (existing rows become `ingest`) plus the `merge_winner_id`/`merge_loser_id` columns, and the `CHECK` rejects a `jobs` row whose `kind` is outside `{ingest,merge}`. *(real SQLite)*
- R-NFNE-809D — The worker claiming a `kind='merge'` job dispatches to `mergeSubjects`; it runs **no** `extract` and no source-text integrate for that job. *(worker seam, mock provider records no extract call)*
- R-NGVA-LS02 — After merging loser `L` into winner `W`: `L`'s claims now have `subject = W`, `W`'s page is the recompile of the combined claim set, `L`'s page row is gone, `L`'s subject row is gone, and an `aliases` row `Normalize(L.name) → W` exists — all present together after one commit. *(real SQLite)*
- R-NI36-ZJQR — When Phase B/C errors (compile fails or a tx step errors), the merge commits **zero** changes — `L` still exists, its claims unmoved, no alias row written, `W`'s page unchanged — and the job ends `failed` with a non-empty reason. *(real SQLite, injected error)*
- R-NJB3-DBHG — A pre-existing alias `X → L` becomes `X → W` after the merge and `L` is deleted, proving the loser's inbound aliases are repointed (step 4) before the loser delete (step 5) so the `ON DELETE RESTRICT` FK permits the delete; reversing the order would trip the FK and roll back. *(real SQLite)*
- R-NKIZ-R385 — The terminal-status write is conditional on `status='working'`: if the merge job is flipped to `aborted` concurrently, the step-7 update touches 0 rows and the whole merge tx rolls back, committing no claim repoint, page, subject delete, or alias. *(real SQLite)*
- R-NLQW-4UYU — A merge job left in `working` (crash orphan) is reset to `pending` by the boot sweep and re-run to completion: because the rolled-back attempt wrote nothing, `L` still resolves by id in Phase A and the re-run merges cleanly with no duplicate alias or lost claim. *(real SQLite)*
- R-NMYS-IMPJ — When `merge_loser_id` (or `merge_winner_id`) no longer resolves at execution time — already folded away by an earlier merge — `mergeSubjects` writes nothing and ends the job `failed` with a clear reason, never a crash or a partial write. *(real SQLite)*
- R-NPEL-A66X — Serialization (the race fix): drained by the single worker, an ingest mentioning `L`'s name submitted *after* the merge resolves (via the D25 alias) to `W` and never re-creates the deleted `L` — proving the merge and the ingest run one-at-a-time on the one goroutine, with the merge's delete fully visible to the later ingest. *(worker seam + real SQLite)*
- R-HUDR-AWS9 — **End-to-end alias round-trip (the regression guard for the shipped bug):** after a real merge of `L` into `W` runs to `done`, `Resolver.ResolveByName(L.name)` returns `W` — proving step C-6 actually inserted a forward-routing alias keyed by `Normalize(L.name)` that the resolver can follow. A merge that skips the insert (today's code) fails this: `ResolveByName(L.name)` would return `ErrSubjectNotFound`. *(worker seam + real temp SQLite, mock compiler — drive the merge, then resolve the folded name)*
