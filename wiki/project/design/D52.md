# Decision 52 — Web surface from `share/www` through the chassis (de-embed the read UI)

**Problem.** wiki's read surface — the home/ask page, subject pages, and the
`static/` assets — is embedded in the binary via `//go:embed` in `internal/web`
(D39 substrate, D42 package, D48–D50 assets), with its own two `html/template`
sets and its own `StaticHandler`. The suite is de-embedding Go-served web assets
onto disk, served by **one** chassis mechanism (appkit D5 www-root resolution,
D6 `appkit/web`, D7 `Spec.WWW`/`Router.WWW()`/chassis static — consumed here as a
fixed external contract). crm and notify converted their single landing pages;
wiki is the **first multi-page surface** to convert — and D7 explicitly
anticipated it ("pages will grow per-service routes … wiki's multi-page UI").
Two wiki-specific facts shape the conversion:

1. **`internal/web` stays.** Unlike notify's `internal/web` (a one-closure
   landing remnant, deleted at conversion), wiki's is a genuine router package: a
   route table (`GET /{$}`, `GET /subject/{type}/{slug}`), injected domain seams
   (`Asker`/`OrphanLister`/`Mentioner`/`PageFinder`), `?q=` ask dispatch,
   markdown rendering, and a styled 404. Only its asset embed and its private
   template parsing move to the chassis.
2. **The single-set parse collides.** `appkit/web.Load` parses every template
   under the root into **one** set (`template.ParseFiles`, D6). wiki's two page
   files **both** define a `{{define "main"}}` content block (`home.tmpl` and
   `subject.tmpl`) — which is exactly why the current code keeps two *separate*
   template sets. A naive single-set parse silently collides (the last-parsed
   `main` wins) and renders one page's body for both. The conversion **must**
   refactor the templates to be collision-free in one set.

**Decision.** wiki serves its read surface from the on-disk www root, rendered
through the chassis-loaded `*web.Site`.

1. **The files move (same bytes).** `internal/web/layout.tmpl`, `home.tmpl`,
   `subject.tmpl` → `wiki/share/www/`; `internal/web/static/tokens.css` and
   `internal/web/static/fonts/*.woff2` (the four fonts) →
   `wiki/share/www/static/`. `bin/ship` bundles `wiki/share/` into the release
   `share/<version>` tier; on the box the chassis resolves `share/current/www`,
   in dev `./share/www` or the `WIKI_WWW_PATH` override.
2. **The templates are refactored collision-free.** The shared
   `{{block "main" .}}` indirection in `layout.tmpl` is replaced by
   unique-named chrome partials so one parsed set holds both pages with no
   duplicate define name: `layout.tmpl` defines the document open
   (`{{define "head"}}…<main>{{end}}`) and close/footer
   (`{{define "foot"}}</main><footer>…</footer>…{{end}}`); `home.tmpl` defines
   `{{define "home"}}{{template "head" .}}…home body…{{template "foot" .}}{{end}}`
   and `subject.tmpl` the same for `{{define "subject"}}`. No two files share a
   define name. The rendered DOM — elements, classes, aria labels, the `.prose`
   container, the `<base href="{{.Mount}}">`, the `static/tokens.css` link, and
   the D50 font preloads — is preserved so the D42–D50 read-surface ids stay
   green. The **exact** block decomposition is at the build agent's discretion
   provided (a) all templates load into one `appkit/web.Load` set with no
   duplicate define name, and (b) rendering `"home"` and `"subject"` from that
   one set each yields its **own** body (R-JI6Y-E3DN).
3. **The Spec opts in.** The composition root (D54) sets `WWW: true`. The chassis
   loads `share/www` at serve (failing startup loudly if the root is missing),
   auto-mounts `GET /static/`, and exposes the loaded site via `rt.WWW()`.
4. **`internal/web` renders through the injected site.** `NewHandler` gains a
   `*appkit/web.Site` parameter, threaded from `rt.WWW()` at the composition
   root; its `home`/`ask`/`subject` handlers call
   `site.Render(w, "home"|"subject", data)` instead of the deleted package-level
   `homeTemplates`/`subjectTemplates`. The `//go:embed`, the
   `template.Must(ParseFS(...))` sets, and the `StaticHandler`/`LandingHandler`
   funcs are **deleted** (the chassis owns `/static/`; the `GET /static/` mount
   is removed from the web mux). The styled-404 path keeps its
   `w.WriteHeader(http.StatusNotFound)` before rendering, so a missing subject
   stays a 404.
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_wiki` exports
   `WIKI_WWW_PATH="$repo/wiki/share/www"` beside its existing `WIKI_DB_PATH`, so
   a repo-root-launched dev binary finds the source tree. This one line crosses
   the `wiki/` boundary and is verified by the live `bin/start` smoke, **not**
   the Go suite; the plan phase flags it.
6. **Tests relocate.** The asset/render tests that drove the embedded FS move to
   load a `Site` from the repo-real `wiki/share/www` tree (`appkit/web.Load` on a
   path relative to the test package), so they exercise the exact files that
   ship. The read-surface behavior tests (D42–D46) keep their stub seams and
   assertions but drive the web handler over a `Site` loaded from `share/www`.

**Rejected.**
- **Deleting `internal/web`** (notify's move) — wiki's package is a real router
  with domain seams and markdown, not a one-closure remnant; inlining the route
  table into the composition root would bury it.
- **Keeping the embed** — the suite-wide reversion this completes; embedded
  assets are invisible on the box and cost a rebuild to edit in dev.
- **Keeping wiki's two private template sets, loading them from disk itself** —
  `rt.WWW()` is opaque (no root accessor), and re-resolving the www path in the
  service would duplicate `appkit/config`'s resolution; the collision-free
  single-set refactor is the honest fix.
- **`state/www` as the root** — that tree is sites' nginx-served user content;
  wiki's UI assets are release artifacts in the shipped `share` tier (appkit D5).

**Verification** (over the repo-real `wiki/share/www` tree; the retained
D42–D50 read-surface ids continue to pin page/asset behaviors through the new
substrate).
- R-JGZ2-0BMY — an `appkit/web` `Site` loaded from `wiki/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `"home"` through
  the web handler with the injected service and version visible in the footer and
  the search `<form>` present — proving the shipped templates, not an embedded
  copy, are the page source.
- R-JI6Y-E3DN — the **same** loaded `Site` renders `"subject"` producing the
  subject-page `<article>` (its title, `.prose` body, and mention footers),
  **distinct** from the home body — proving both pages render their **own**
  content from **one** parsed set (a residual shared-`main` collision would
  render one page's body for both and fail this).
- R-JJEU-RV4C — the chassis static mount over `wiki/share/www` serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  wiki-side `/static/` handler registered — proving the assets ship on disk and
  the chassis serves them.
