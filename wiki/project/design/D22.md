# Decision 22 — The `cmd/eval-extract` binary + the shipped gold case

**Decision.** A second binary `cmd/eval-extract` is the operator-facing front door to the extract evaluation: it builds the **real** extractor and the pinned judge, runs the dataset (D20), scores it (D21), and prints a stamped scorecard. It is a **thin shell** over `internal/eval` — all logic lives in the library (what tests drive, no live LLM in the suite), the binary stays a flag-parser + printer. One blessed gold case ships in-repo so the tool runs against real data out of the box.

## The binary

```
eval-extract \
  -dataset testdata/eval/extract \
  -model claude-sonnet-4-6 -reasoning off -temperature 0 -max-tokens 16384 \   # model under test (extract site)
  -judge-model claude-opus-4-8 -judge-reasoning high \                         # judge site (defaults pinned)
  [-record run.jsonl] [-json]
```

Construction mirrors the service composition root so the eval exercises the production path:
- Reads `ANTHROPIC_API_KEY` via `getenv` and builds the shared agentkit provider + `llm.Client` exactly as `wiki.NewConfig` does; **fails loudly at startup if the key is absent** (same rule as the service).
- Builds the extract site as `extract.DefaultCallSite()` with `-model/-reasoning/-temperature/-max-tokens` layered on through the **same D19 resolver/parsers** (`off|low|medium|high`, numeric temperature or `default`, positive max-tokens), so a malformed flag fails loudly identically to a malformed manifest knob. Constructs `extract.New(client, site)`.
- Builds the judge as `eval.DefaultJudgeCallSite()` (pinned `claude-opus-4-8` at high thinking, D21) with `-judge-*` flags layered on the same way. Constructs `eval.NewJudge(client, judgeSite)`.
- Wires an `llm.Recorder` so every eval LLM round-trip is captured (below).

It then `LoadDataset` → for each case `eval.Run` (real extract) → `eval.Score` (judge) → assembles a `Scorecard` → prints.

## The scorecard — stamped, comparative

```go
type Scorecard struct {
    Dataset string             // dataset path
    Model   string             // model under test (extract site)
    Config  string             // resolved extract CallSite params JSON (temperature/reasoning/max_tokens)
    Judge   string             // judge model + params JSON
    Cases   []eval.CaseResult  // per-case subject + claim partitions (D21)
    Totals  eval.Totals        // overall + per-difficulty precision/recall (D21 Aggregate)
}
func (s Scorecard) WriteHuman(w io.Writer)  // readable table: per-case items + metrics, then aggregate + per-difficulty
func (s Scorecard) WriteJSON(w io.Writer)   // the Scorecard as JSON, for saving and diffing two runs
```

The **stamp** (`Model`, `Config`, `Judge`) makes two runs apples-to-apples: a run under `-model sonnet -reasoning off` and one under `-reasoning high` are distinguishable in their own output. Human output is default; `-json` emits the machine-readable `Scorecard`. **No pass/fail** — the tool prints scores and exits 0 on a completed run regardless of how the model scored.

## Traceability — the D13 recorder seam, to a file (no service DB)

The CLI wires a tiny file-backed `llm.Recorder` (`eval.NewJSONLRecorder(io.Writer)`) into the `llm.Client`, so **every** eval round-trip — each extract attempt **and** each judge call — is appended as one `CallRecord` JSON line (request, response, params, usage, stage, error). This reuses D13's seam rather than standing up the wiki's SQLite, and makes "trace any score to the exact request and response behind it" true off-line. Records go to `-record <path>` when given (else a default run file); the wall clock is fine here (an operator tool, not a deterministic unit test).

## The shipped gold cases

Blessed cases under `testdata/eval/extract/<case-name>/` (`document.txt` + `gold.json`, D20 shape), **authored during the build** by a strong agent and human-reviewed. Each is a factual source with a clear, blessable subject-and-claim set and a `difficulty` from the closed set. The dataset spans the difficulty range: an **easy** case (`tulsa-lab-opening` — one entity) and a **hard** case (`meridian-freshcrate-acquisition` — a multi-entity acquisition with cross-referencing claims, where extract is expected to over-produce subjects/claims the gold does not bless, exactly the discrepancy the itemized scorecard of D23 surfaces). A test asserts each *committed* case loads clean and has ≥1 gold subject with ≥1 claim, so the deliverables are guarded. The dataset continues to grow in later work.

## Live verification — the one check the mock suite cannot make

Every unit test above drives `internal/eval` through a **mock** provider (fast, free, deterministic). But a mock accepts whatever parameters it is handed, so it can never prove the assembled call sites are ones the **real provider** accepts and runs — exactly the failure an illegal generation-parameter combination represents (e.g. a judge `Temperature 0` under extended thinking, which the API rejects). So this Decision additionally requires a **live, operator-run end-to-end check**: with a real `ANTHROPIC_API_KEY`, run `eval-extract` against the shipped gold case and confirm it **completes and emits a populated scorecard**, exercising the real extract *and* judge call sites against the API. It is deliberately **not** part of the mock unit suite (needs a key, spends tokens); it is the substrate-adequate counterpart to the field-shape assertions (D21 R-DWI0-C7E2 checks the judge's configured values; this proves that config actually runs), and the check that would have caught a judge config the provider refuses.

**Rejected.**
- **Recording eval calls into the wiki's SQLite (the service `LLMCallStore`, D13).** Couples an off-line tool to the service DB and its migrations for no gain; a JSONL file over the same `Recorder` interface is sufficient and self-contained. Rejected.
- **A pass/fail exit code keyed to a score threshold.** The product forbids a gate this release; exit reflects whether the *run completed*, not how the model scored. Rejected.
- **Putting flag-parsing/scoring logic in `main`.** An untestable `main` re-implementing the D19 parsers and scorecard assembly. A thin shell over `internal/eval` keeps it all unit-testable. Rejected.
- **Human-only output.** The product is comparative (run A vs run B); `-json` is what lets two runs be saved and diffed mechanically. Kept.

**Verification.**
- R-34NV-WDIP — The CLI layers `-model/-reasoning/-temperature/-max-tokens` onto `extract.DefaultCallSite()` and `-judge-*` onto `eval.DefaultJudgeCallSite()` via the D19 parsers: `-reasoning low` yields `Level("low")` on the extract site, a malformed flag (bad reasoning/temperature/max-tokens) fails loudly, and with no flags the extract site equals the default and the judge stays `claude-opus-4-8`.
- R-35VS-A59E — Running a dataset through a mock-backed client assembles a `Scorecard` stamped with the model-under-test's `Model` and resolved `Config` params and the judge's `Judge` model, carrying one `eval.CaseResult` per case and the aggregated `Totals`.
- R-373O-NX03 — `WriteHuman` and `WriteJSON` over a `Scorecard` each emit the stamp (model/config/judge) and the subject and claim metrics per case, in aggregate, and per difficulty; `WriteJSON` round-trips back into an equal `Scorecard`.
- R-38BL-1OQS — With the JSONL recorder wired, a scripted run records one `CallRecord` per extract round-trip **and** one per judge call, each carrying the round-trip's request, response, and params — so every score traces to its calls (proven via a captured in-memory writer).
- R-39JH-FGHH — Starting `eval-extract` with `ANTHROPIC_API_KEY` unset fails loudly (non-zero exit / startup error) before any dataset load or LLM call, matching the service's startup rule.
- R-3ARD-T886 — The committed **easy** gold case (`tulsa-lab-opening`) under `testdata/eval/extract/` loads through `eval.LoadCase` with no error, carries a `difficulty` in `{easy,medium,hard}`, and has at least one gold subject with at least one claim — guarding the shipped deliverable.
- R-8PSN-NDRQ — The committed **hard** gold case (`meridian-freshcrate-acquisition`) loads through `eval.LoadCase` with no error, carries `difficulty == "hard"`, and has more than one gold subject, each with at least one claim — guarding the demanding case that gives the eval (and the D23 diff output) something non-trivial to score.
- R-ME5L-HXJ3 — A **live** end-to-end run of `eval-extract` against the real provider over the committed gold case (real `ANTHROPIC_API_KEY`, no mock) **completes without error and emits a populated `Scorecard`** — driving the real extract *and* judge call sites against the API, so an illegal provider-parameter combination (such as a judge temperature incompatible with extended thinking) fails here rather than passing against a mock. Operator-run on demand (key required, tokens spent), outside the mock unit suite; the substrate-adequate counterpart to R-DWI0-C7E2's field-shape check (D21).
