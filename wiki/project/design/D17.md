# Decision 17 — DB concurrency: a single-writer handle + a concurrent read pool (reads never blocked)

**Decision.** Reads run in parallel without contention and never wait on ingest or LLM work; writes are serialized — one job's mutations complete before the next begins. On a single SQLite file: **WAL plus a split connection model** — every write goes through one connection so writes serialize *by construction*, and reads go through a separate multi-connection pool so they run concurrently and, under WAL, are never blocked by the in-flight writer. This decision owns only the **pool shape and the read/write routing seam**; the atomic-commit shape is D14, the recorder D13.

Realigns D2: appkit's `rt.DB()` remains the **single writer**, and wiki opens **one additional read-only handle** to the same file. The invariant is "**exactly one writer**," not "exactly one *handle*" — a second *read* handle preserves it. D4/D13/D14's "single-writer SQLite" stays true.

## (a) Two handles, one file

Both handles open the same path with appkit's existing pragma string (`journal_mode(WAL)`, `busy_timeout(5000)`, `foreign_keys(ON)`); they differ only in pool size and intent.

```go
package wiki

// Conns carries the two SQLite handles wiki runs on. The WRITE handle is capped
// at a single connection, so every mutation the service makes is serialized one
// after another with no writer-vs-writer lock contention. The READ handle is a
// small pool (> 1); under WAL its readers run concurrently and never block on the
// open writer — they observe the last committed snapshot.
type Conns struct {
    Read  *sql.DB // SELECT-only; SetMaxOpenConns(readPoolSize), readPoolSize > 1
    Write *sql.DB // every INSERT/UPDATE/DELETE and every tx; SetMaxOpenConns(1)
}
```

Composition root (D2 `Handlers`):

```go
write := rt.DB()                       // appkit's handle: WAL + busy_timeout, MaxOpenConns(1) — the sole writer
read, err := db.OpenRead(cfg.DBPath)   // same file, same pragmas, query_only; SetMaxOpenConns(readPoolSize)
// migrations already ran on `write` before Handlers; `read` runs none and only ever issues SELECTs.
svc = wiki.NewService(wiki.Conns{Read: read, Write: write}, prov, cfg)
```

- `db.OpenRead` opens the **already-migrated** file read-only (runs no migrations) and sets `_pragma=query_only(true)` as a structural guard. `readPoolSize` is a small bounded constant (e.g. 8) — the contract is only that it is **> 1**; the exact value is a tunable.
- The single write handle (`MaxOpenConns(1)`) makes "writes serialized" a **structural** property rather than a `busy_timeout` race outcome: two writers can never hold the file's write lock at once because there is only one write connection.

## (b) Read/write routing seam

Stores receive `Conns` and route per statement: **SELECT-only methods use `Read`; every mutation and every transaction uses `Write`.** A store that does both (e.g. `JobStore`: `Status`/`List` read, `InsertIngest`/`ClaimPending`/`FinishWorking`/`Abort`/`Rerun` write) holds both and picks per method.

```go
func NewJobStore(c Conns) *JobStore // Status, List → c.Read ; InsertIngest, ClaimPending, Finish*, Abort, Rerun, boot-sweep → c.Write

func (s *JobStore) Status(ctx, id) (JobStatus, error)            { /* SELECT … on s.read  */ }
func (s *JobStore) InsertIngest(ctx, job) error                  { /* INSERT  … on s.write */ }
func (s *JobStore) ClaimPending(ctx, t) (Job, bool, error)       { /* tx       on s.write */ }
```

- Pure-read stores reached only from the MCP read path (`SubjectStore.List`, `ClaimStore.List`, `PageStore.Get`/`PageWithLinks`, `LLMCallStore.List`) are constructed with `Read`.
- Integrate write-set (D14): the claim/commit transaction begins on `Write`; the in-memory read of the job's prior claims (D14 step 1) reads committed data, so it uses `Read`. **No transaction is ever begun on `Read`.**
- The recorder (D13) writes on `Write`. Because integrate holds no transaction across the LLM calls (D14), each eager recorder write acquires the single write connection *between* round-trips with nothing held — no self-block. The deployed deadlock (recorder write starving against a transaction held across `compile` on a `MaxOpenConns(1)` pool) cannot occur once integrate follows D14 and the recorder runs on the write handle with no enclosing tx.

## (c) What this guarantees, in product terms

- A read is **never** blocked by ingest or LLM work: during a job's multi-minute LLM phase the write connection is idle (no tx held), and even during the brief final commit a reader on the read pool returns the last committed snapshot under WAL without waiting.
- A **stuck or slow job holds no read resource**, so reads and new submissions stay live and `abort` (a write on the single write connection, not held during the LLM phase) takes effect promptly.
- Writes never interleave: the single write connection serializes the worker's commit, the recorder, `ingest`'s insert, and `abort`/`rerun` one after another.

The connection is held **only** for the prepared statement's execution (or the short commit transaction) and the minimal code around it — never across an LLM call.

**Rejected.**
- **A single pool at `MaxOpenConns(N>1)` (no separate write handle).** Reads scale, but two writers (worker-commit vs. a concurrent `abort`) can both seek the write lock, so write serialization degrades to a `busy_timeout`-mediated race and a collision can surface `SQLITE_BUSY`. The split model keeps "one writer" exact. Rejected.
- **Leaving `rt.DB()` at `MaxOpenConns(1)` for everything (status quo).** The deployed defect: every read serializes behind the single connection, so a read issued while the worker holds it waits. Rejected.
- **Raising `MaxOpenConns` on `rt.DB()` itself.** Turns appkit's single-writer handle into a multi-writer pool, breaking the "exactly one writer" invariant D4/D13/D14 rely on. Rejected.
- **Changing appkit's shared `Open` pragmas/pool.** Out of scope (suite-wide blast radius) and unnecessary — wiki opens its own read handle. Rejected.

**Verification.**
- R-FUCC-IT4M — While an `INSERT`/`UPDATE` transaction is open and uncommitted on the write handle, a concurrent `SELECT` on the read handle returns the last committed snapshot promptly and does not block until that transaction commits (WAL reader non-blocking), against a real temp SQLite.
- R-FVK8-WKVB — Two concurrent write operations issued against the write handle both succeed and apply one-after-another with no `database is locked`/`SQLITE_BUSY` error — writes are serialized by the single write connection.
- R-FWS5-ACM0 — With the write connection idle (simulating a job's LLM phase, no tx held), a `SELECT` on the read handle and a concurrent control write (a status `UPDATE` on the write handle) both complete promptly without waiting on the simulated long phase — a slow job wedges neither reads nor control writes.
- R-FY01-O4CP — After the write handle commits a change, a subsequent `SELECT` on the read handle observes the committed data (read-your-writes visibility across the two handles under WAL).
