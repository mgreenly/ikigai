# Decision 33 — Merging the two lanes: rank fusion + exact-name pin

**Decision.** A `hybridRetriever` runs both lanes, merges their results by rank, optionally pins an exact-name match on top, and returns the top few pages. It is itself a `Retriever` (D08), so `ask` sees one ranked list and never knows two lanes existed.

```go
type hybridRetriever struct {
    keyword  retrieve.Retriever   // D31
    vector   retrieve.Retriever   // D32
    resolver *wiki.Resolver       // exact-name pin (D25 resolver)
    pages    *wiki.PageStore      // fetch the pinned page
    cfg      FusionConfig
}
type FusionConfig struct {
    RRFk          int   // rank-fusion constant; default 60
    PerLane       int   // candidates requested from each lane; default 60
    FinalK        int   // pages returned after fusion; default 8
}
func (r *hybridRetriever) Search(ctx context.Context, query string, k int) ([]retrieve.Hit, error)

// SearchAnalyzed is the entry ask uses (D9): it consumes a prepared, possibly
// decomposed query (D36) and fuses every lane-list in one pass.
func (r *hybridRetriever) SearchAnalyzed(ctx context.Context, qa wiki.QueryAnalysis, k int) (retrieve.Result, error)
```

`SearchAnalyzed` returns a `retrieve.Result`, not a bare `[]Hit`, because the honest-empty gate (D9) must judge relevance on the raw meaning-lane cosine, which the fused rank score hides:

```go
type Result struct {
    Hits     []Hit
    TopDense float64   // highest raw cosine seen across the meaning lane this query
    Pinned   bool      // true if an exact-name page was pinned at rank 1
}
```

**Plain `Search` is the one-string form** — both lanes get the same string; used for the simple/exact path and in tests. **`SearchAnalyzed` is the real entry** `ask` calls with the prepared query (D36).

**Merge by rank, not by score (Reciprocal Rank Fusion).** The two lanes score on incompatible scales — the keyword score and the cosine number can't be added. So each page is scored by its **position** in each lane's list: a page gets `1/(60 + rank)` from every lane it appears in, and those add up. The effect: a page near the top of *either* lane scores well, and a page near the top of *both* scores highest. The constant 60 is the standard, widely-used value (a `RRFk` knob). This needs no score normalization and no tuning data.

**One entry per page.** A page found by both lanes is the same page — it's merged into a single result (keyed on `PageID`, which is the subject id), not listed twice.

**Exact-name pin (the shortcut we're keeping).** Before fusing, the question is checked against the existing subjects by exact normalized name (the shared resolver, honoring merge aliases). If it names a subject exactly, that subject's page is placed at **rank 1** of the result, ahead of the fused list, and removed from the fused tail so it isn't repeated. This preserves today's precise behavior — an exact-name question never does worse than before — as a special case of the new path. If nothing matches exactly, the result is purely the fused list.

**How many.** Each lane is asked for ~60 candidates (`PerLane`), they're fused, and the top `FinalK` (default 8) pages are returned — 8 pages of up to 12,000 chars each fits the answer step's reading budget with room for a multi-part question. All three numbers (`RRFk`, `PerLane`, `FinalK`) are config knobs.

**Decomposed fan-out + per-lane routing (`SearchAnalyzed`).** Given a prepared query (D36) with one or more sub-queries, the hybrid sends each lane the text that suits it: the **meaning lane** gets the full natural-language sub-query sentence, and the **keyword lane** gets the OR-joined keywords + aliases (sanitized via `ftsPhrase`, D31). Every resulting lane-list (two lanes × each sub-query) is fused in a **single** RRF pass into one deduped ranked list, then the exact-name pin (on the original whole question) and `FinalK` cap apply as above. This is what makes a "compare X and Y" question surface **both** X's and Y's pages — each sub-query pulls its own subject, and the union is ranked together.

**Relationship to honest-empty.** The hits this returns carry the fused rank score in `Score` (for ordering). The separate "is anything actually relevant?" gate — which must look at the raw meaning-lane signal (cosine), not the fused score — is owned by `ask` in D9; the hybrid surfaces it as `Result.TopDense` so that gate can apply it.

**Rejected.**
- **Weighted score fusion (add a scaled keyword score to the cosine).** Requires normalizing two incompatible scales and tuning the weights against labeled data we don't have; rank fusion needs neither and performs at least as well here.
- **A second LLM "reranker" pass over the fused list.** Extra cost and latency for marginal gain once same-corpus dual retrieval has produced a clean top-8. If ever needed, the cheap path is to let the answer step itself pick among a slightly larger candidate set — no extra call.
- **Dropping the exact-name pin (hybrid handles everything uniformly).** Simpler, but an exact-named subject can occasionally rank below where the user expects; the pin is cheap insurance that keeps current behavior intact.

**Verification.**
- R-79KD-1622 — Reciprocal Rank Fusion ordering: a page appearing in **both** lanes' lists ranks above a page that is rank 1 in only one lane, per `1/(RRFk+rank)` summed across lanes (k=60) — the fused order follows the formula, not either lane alone.
- R-7AS9-EXSR — A page returned by both lanes appears exactly **once** in the fused output (deduped by `PageID`), with its contributions combined — never two entries for one page.
- R-7C05-SPJG — When the question exactly names an existing subject (normalized-exact via the resolver, alias-honoring), that subject's page is returned at **rank 1** and is not duplicated lower in the list; when the question names no subject exactly, the result is the fused list with no pin.
- R-7D82-6HA5 — `Search` requests `PerLane` candidates from each lane and returns at most `FinalK` (default 8) fused hits; raising/lowering the `FinalK` knob changes the count returned accordingly.
- R-Q8RI-7POG — `SearchAnalyzed` over a two-sub-query analysis ("compare X and Y") fans each sub-query out to both lanes and fuses **all** the resulting lists in one pass into a single deduped ranked list that contains **both** X's page and Y's page — a behavior a single blended query does not produce.
- R-Q9ZE-LHF5 — Per-lane routing: under `SearchAnalyzed` the meaning lane receives the full sub-query **sentence** while the keyword lane receives the OR-joined **keywords + aliases** (not the same string to both) — confirmed with spy lanes capturing the text each was handed.
