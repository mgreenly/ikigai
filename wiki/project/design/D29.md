# Decision 29 — Alias-aware path entry for the read lookups (`page` / `claims`)

**Decision.** Per the alias decree (D25), access *through* an alias must appear as access via the surviving canonical subject. D25 routes the **name**-keyed surfaces through `Resolver.ResolveByName` — ingest (`plannedSubject`) and `ask` (`gatherPages`). But the **path**-keyed read verbs do not: `page` and `claims` resolve their `type/slug` input with `SubjectStore.GetByPath` (D11), which is **subjects-only** — so after a merge folds `L` into `W`, `page L's-old-path` returns not-found and `claims L's-old-path` returns empty despite the forward-routing alias `Normalize(L) → W`. This Decision adds the alias fallback to the path entry, mirroring `ResolveByName`, so a folded-away path forwards to the survivor and serves its canonical page and claims.

This is **new behavior**, not a rewrite of D11. `SubjectStore.GetByPath` stays exactly as D11 specifies — subjects-only, exact, type-checked — because other callers depend on that (see the asymmetry below). The alias fallback is a new seam on the shared `Resolver` (D25); only the two read-path services adopt it.

## The seam — `Resolver.ResolveByPath`

A new method on the existing D25 `Resolver` (it already composes `SubjectStore` and `AliasStore`):

```go
// ResolveByPath maps a public "<type>/<token>" path to its live subject, honoring
// merges. It is the path-keyed analogue of ResolveByName:
//   1. SubjectStore.GetByPath(path) — a live subject at this exact type+token?
//      return it. (D11 semantics, unchanged: exact token, exact type.)
//   2. ErrSubjectNotFound → split the token (the part after "/"), then
//      AliasStore.GetByNormName(token) — an alias? load the survivor via
//      SubjectStore.Get(subject_id) and return it.
//   3. neither → ErrSubjectNotFound.
//
// Single-hop is PROVABLE (same guarantee as ResolveByName): the D25 FK
// (ON DELETE RESTRICT) plus D26's eager RepointSubject mean every
// alias.subject_id names a LIVE, non-aliased survivor — step 2's Get never lands
// on another alias. No chain-following, no depth bound, no cycle logic.
func (r *Resolver) ResolveByPath(ctx context.Context, path string) (Subject, error)
```

**The alias branch is token-only — and that is correct.** The `aliases` table keys on `norm_name` alone (D25); it stores no type. The fallback matches a folded name by its token irrespective of the requested type prefix. This cannot weaken D11's live-subject type discipline, because the fallback runs **only after** `GetByPath` has already missed: a path naming a live subject of the right type returns at step 1, and a right-token/**wrong-type** path with no alias still ends at `ErrSubjectNotFound`. The fallback never resolves a wrong-type path *to a live subject* — it only forwards a not-a-live-subject token that happens to be a registered dead name.

## Adoption — the two read-path services, and only those

In the composition root (`cmd/wiki/main.go`), the page and claims path services swap `SubjectStore.GetByPath` for `Resolver.ResolveByPath`:

- `pathPageService.PageByPath` — on a hit (live or forwarded) it continues into `Service.PageWithLinks(subject.ID)` and `RenderFooter(...)` as today. Because served fields derive from the **resolved** subject (`Path(subject)`, `page.Title`, footer-rendered body), a forwarded page is **byte-identical** to a direct lookup of the survivor's own path.
- `pathClaimService.ListBySubject` — same swap; the forwarded path lists the survivor's claims (the loser's claims were repointed at merge, D26), identical to listing by the survivor's path.

Both keep their existing `ErrSubjectNotFound → sql.ErrNoRows` mapping for a genuine miss, so the MCP layer still emits a clean not-found (D10 R-MYDT-PCRV) on a truly unknown path.

**The forwarded result is identical, not signposted.** The `page` result stays exactly `{subject, title, body}` and `claims` exactly `[{id, text, job}]` — no "redirected" flag, no echo of the requested alias path. D10 R-03GW-PX5K already pins the `page` result to exactly those three fields, which a redirect marker would violate.

## The asymmetry — page/claims forward, merge does not

`GetByPath` stays subjects-only because its **other** callers want the opposite of forwarding:

- `mergePathResolver` (`cmd/wiki/main.go`) and `specMergePathResolver` (`internal/wiki/wiki.go`) resolve `merge`'s `winner`/`loser` via `GetByPath` — and D27 requires a `merge` input naming an already-folded name to get a clean **not-found**, never a silent redirect (which would let an owner "merge" a name that no longer names a subject, or self-merge).

So the fix is placed at the *read* call sites, not pushed into the shared `GetByPath`. Page serving wants transparent forwarding; the merge resolver wants real, live subjects. Keeping `ResolveByPath` separate preserves both. This asymmetry needs **no new id** — it is held by construction (`GetByPath` untouched) and already guarded by the unchanged D27 R-E01B-X6IA (a merge input matching no subject → not-found, no job) and D11 R-DT53-3OJL (subjects-only, type-checked resolution).

## Rejected.

- **Make `GetByPath` itself alias-aware.** It has callers that must *not* forward — the merge resolvers (D27), which need a folded name to read as not-found. Localize forwarding to the read services.
- **Reuse `ResolveByName(token)` for the path entry.** `ResolveByName` is subjects-**first** on the bare name and ignores the type prefix, so for a path whose token matches a live subject of a *different* type it would return that wrong-type subject — silently weakening D11's exact-type guarantee. The path entry must keep `GetByPath`'s type check on the live branch and fall to aliases (token-only) **only** on a genuine miss; hence a distinct `ResolveByPath`.
- **Follow alias chains / add depth bound + cycle detection.** Same as D25: the merge eagerly repoints inbound aliases (D26), so chains never form — resolution is a provable single hop.
- **Signal the redirect in the result (a "forwarded"/"canonical_path" field).** The decree makes the alias invisible, and D10 R-03GW-PX5K fixes the `page` result to exactly `{subject, title, body}`.
- **Type-check the alias path against the loser's original type.** The `aliases` table stores no type (D25), and the survivor renders under its own canonical path/title regardless of the requested prefix — a token match is the right and only available key.

## Verification.

Tested against a **real temp SQLite** opened by the appkit migration runner with `foreign_keys=ON`, composing `SubjectStore`/`AliasStore` on the shared `Resolver`. Alias rows are installed directly via `AliasStore` — no merge needed to exercise resolution.

- R-AF1X-PG7K — `ResolveByPath` for a `type/token` path whose token matches **no live subject** but equals an `aliases.norm_name` returns the **surviving** subject, loaded with a single `SubjectStore.Get` and no second alias lookup on the survivor's own token (single hop — FK + eager repoint guarantee the survivor is never itself an alias). *(real temp SQLite)*
- R-AG2Y-PH8L — `ResolveByPath` for a path matching a live subject by **exact token and type** returns that subject directly, identical to `GetByPath`, without consulting the `aliases` table; and a **right-token/wrong-type** path for which no alias exists returns `ErrSubjectNotFound` — proving the token-only fallback never resolves a wrong-type path to a live subject and D11's type discipline is preserved. *(real temp SQLite)*
- R-AH3Z-PJ9M — `ResolveByPath` for a path whose token is **neither** a live subject **nor** an alias, and for a malformed/empty-token path (e.g. `entity/`), both return `ErrSubjectNotFound` — so `page`/`claims` still report a clean not-found (`found:false` / empty) on a genuinely unknown path. *(real temp SQLite)*
- R-AL5R-PL1P — Forwarding is **invisible and symmetric across the read verbs**: with an alias `Normalize(L) → W` installed and `W` having a page and claims, `ResolveByPath(<L's old path>)` and `ResolveByPath(<W's path>)` return the **same** survivor `W`, so `pathPageService.PageByPath` renders the byte-identical `{subject: Path(W), title, body}` and `pathClaimService.ListBySubject` lists `W`'s claims for the folded path exactly as for `W`'s own path — the page result carrying no redirect marker and no field beyond the D10 R-03GW-PX5K `{subject, title, body}` shape. *(real temp SQLite; the equal-resolved-subject property is the checkable proof of byte-identical rendering)*
