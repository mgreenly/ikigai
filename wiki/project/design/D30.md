# Decision 30 — Where page embeddings are stored

**Decision.** Each page's embedding (the vector that captures what the page is *about*) is kept in its **own table**, one row per page, separate from the `pages` table. This keeps ordinary page reads narrow and makes "which pages have no vector yet?" a one-line lookup.

```sql
-- one embedding per page; page_embeddings.subject_id == pages.subject == subjects.id
CREATE TABLE page_embeddings (
  subject_id   TEXT PRIMARY KEY,            -- the page this vector belongs to
  model        TEXT NOT NULL,               -- which embedding model produced it, e.g. "text-embedding-3-small"
  dims         INTEGER NOT NULL,            -- vector length, e.g. 512
  vec          BLOB NOT NULL,               -- the vector: little-endian float32, already L2-normalized
  content_hash TEXT NOT NULL,               -- fingerprint of the page text the vector was built from
  updated_at   INTEGER NOT NULL             -- epoch ms, matching the pages table
);
```

Added by a **new forward migration** (`bin/new-migration wiki`), in the suite's no-`FOREIGN KEY` style (the comment records the relationship; `subject_id` is the primary key, so there is at most one vector per page).

**Why each field:**
- **`vec`** is stored as raw bytes. A vector is a list of 32-bit floats; we pack them little-endian into a blob and unpack them on read. The agentkit embedder hands us vectors that are already length-normalized, so we store them as-is — no extra math on our side (this is what lets the meaning lane score a page with a plain dot product later, D32).
- **`content_hash`** is a fingerprint (sha256) of the page's title + body at the moment we embedded it, computed with the `hashText` helper that already exists. When a page is rewritten its fingerprint changes; that mismatch is how the background updater (D35) knows the vector is stale and must be recomputed. A matching fingerprint means "still good, skip it."
- **`model` / `dims`** record which model made the vector. If we ever switch embedding models, old rows won't match the configured model and get recomputed — a model swap becomes a re-embed, not a schema change.

**The store (`internal/wiki`).** A small `EmbeddingStore`, sitting beside the existing subject/page/claim stores, over the same read/write handles (D17).

```go
type Embedding struct {
    SubjectID   string
    Model       string
    Dims        int
    Vec         []float32   // L2-normalized
    ContentHash string
    UpdatedAt   int64       // epoch ms
}

type EmbeddingStore struct { /* read + write DB handles (D17) */ }

// Upsert writes (or replaces) the single embedding row for a subject.
func (s *EmbeddingStore) Upsert(ctx context.Context, e Embedding) error

// LoadAll returns every stored embedding — used to fill the in-memory
// search cache at startup (D32).
func (s *EmbeddingStore) LoadAll(ctx context.Context) ([]Embedding, error)

// the float32 <-> blob codec (pure, package-local)
func encodeVec(v []float32) []byte
func decodeVec(b []byte) ([]float32, error)   // errors on a length not a multiple of 4 (fail loud)
```

Finding which pages still **need** a vector (missing, stale, or wrong model) is the background updater's job and lives with it in D35; this decision owns only the table, the codec, and the read/write of a row.

**Backups need no special handling** — the vectors live in the same SQLite file as everything else, so the existing `VACUUM INTO` backup/restore snapshots them atomically along with the rest.

**Rejected.**
- **A `vec` column on the `pages` table.** Pages are read on every list and inspect; a fat blob column on that hot row slows the common path. A side table keeps page reads lean and "missing vector" trivial to find.
- **Storing vectors as JSON text.** Bigger on disk and slower to parse than packed binary floats, for zero benefit — nothing human ever reads a raw vector.
- **A vector index extension (`sqlite-vec`).** It needs CGO; the wiki's SQLite driver is pure-Go with CGO off and cannot load it. At our scale a plain in-memory scan (D32) is fast enough, so the extension buys nothing and breaks the static build.

**Verification.**
- R-9OCK-FJK1 — A float32 vector encoded to a blob and decoded back equals the original element-for-element, same length and order (little-endian round-trip); a blob whose byte length is not a multiple of 4 returns an error rather than silently dropping bytes.
- R-9PKG-TBAQ — A second `Upsert` for a subject that already has a row **replaces** it (new model/dims/vec/content_hash/updated_at) and leaves exactly one row for that subject — never a duplicate (proven against a real migrated temp SQLite).
- R-9QSD-731F — `Upsert` followed by a read round-trips every column against a real migrated temp SQLite: the vector read back is identical to the one written, and model, dims, and content_hash match.
- R-9S09-KUS4 — `LoadAll` returns every stored embedding (all rows, each with its vector intact), so the in-memory search cache can be filled from it.
