# Decision 34 — The embedding call site: configured and recorded like every other AI call

**Decision.** Embedding is its own configured call site (model + vector size), and every embedding call is written to the `llm_calls` log — keeping the "every AI call is on the record" promise honest for search. This extends the per-call-site config (D19) and the call-record log (D13).

**One model, two uses.** A single embedding model serves both sides — embedding a page and embedding a question — because the two must live in the same vector space to be comparable. So there is **one** embed config, used in two places, distinguished in the log by stage:

- **`embed-page`** — embedding a page during ingest or merge (D35); tagged to its ingest job.
- **`embed-query`** — embedding a question at ask time (D32); no job, like the other ask calls.

**Config (knobs `EMBED_MODEL`, `EMBED_DIMS`).** Added to the resolved `Config` alongside the other call sites (D19), with its own two knobs and fail-loud parsing:

```go
type EmbedSite struct {
    Model    string   // default "text-embedding-3-small"
    Dims     int      // default 512
    Provider agentkit.EmbeddingProvider   // OpenAI, built at the composition root from OPENAI_API_KEY
}
```

- `EMBED_MODEL` — model id string; unset keeps `text-embedding-3-small`.
- `EMBED_DIMS` — a positive integer; unset keeps `512`; `0`, negative, or non-numeric is a **startup error** (fail loud, never a silent fallback).

The default is **`text-embedding-3-small` at 512 dimensions** — ~99% of full retrieval quality at a third of the memory and far lower cost; storing `model`/`dims` with every vector (D30) makes a later change a re-embed, not a schema break. The OpenAI key is read once at the composition root via `getenv` (same secret rule as the Anthropic key, D02) and the provider injected; it never enters domain code.

**The page side uses the document role.** The embedder is asked to encode pages in *document* mode and questions in *query* mode (D32) — the asymmetry the model is trained for. This decision owns the page (document) side; D32 owns the query side.

**Recording (a small adapter).** The existing recorder seam (D13) sits on the *chat* path; the embedder is a different interface, so a thin **recording embedder** wraps it: on each `Embed` call it builds one `CallRecord` and hands it to the same `Recorder` / `LLMCallStore.Record` already used by extract/compile/ask.

```go
// recordingEmbedder wraps the agentkit embedder and logs each call to the D13 recorder.
type recordingEmbedder struct {
    inner    *agentkit.Embedder
    recorder llm.Recorder
    stage    string   // "embed-page" or "embed-query"
}
func (e *recordingEmbedder) Embed(ctx context.Context, texts []string, role agentkit.InputRole) (agentkit.EmbedResult, error)
```

Each record carries the stage, the model, the dimensions (in `params`, e.g. `{"dims":512}`), the token usage the provider reports, and any error; the page side reads the job id off the context (D13's `WithJobID`) so an `embed-page` record ties to its ingest job, while `embed-query` carries no job id.

**New stages need a migration.** The `llm_calls.stage` column is a closed set enforced by a DB CHECK (D13). Adding `embed-page` and `embed-query` means a **new table-rebuild migration** (SQLite can't alter a CHECK; the existing `widen_llm_calls_stage` migration is the template — create new table with the widened CHECK, copy rows, drop, rename, recreate indexes). The prior migrations stay frozen.

**Rejected.**
- **Not recording embedding calls.** Breaks the product's "every AI call on the record" promise and hides the cost of search; embeddings are real paid calls.
- **Separate model knobs for the page side and the query side.** They must share one vector space, so two models would silently break search. One config, enforced by construction.
- **Forcing embed through the chat `llm.Client`.** Different interface, different inputs (no system/temperature/reasoning); a thin recording wrapper onto the shared recorder is cleaner than bending the chat client.
