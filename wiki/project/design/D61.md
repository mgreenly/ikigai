# Decision 61 — Structured MCP adoption: `StructuredResult`, typed error codes, and per-tool output schemas

**Problem.** The suite settled a single machine-and-agent verb surface
(`../docs/structured-mcp-design.md`): every domain verb's result carries
`structuredContent` (the machine rendering) beside its mirrored text (the agent
rendering), each tool declares an `outputSchema`, and every tool error carries a
code from the suite's closed vocabulary so a deterministic caller branches on a
code, never on prose. appkit already provides that contract (its phase 12–14
work): `mcp.StructuredResult(v) (map[string]any, error)`, `mcp.ErrorResult(code
ErrorCode, msg string)` over the closed set (`validation`, `not_found`,
`conflict`, `too_large`, `source_unavailable`, `internal`), `Tool.OutputSchema
map[string]any` surfaced in `tools/list`, and chassis `health`/`reflection` that
already declare schemas and return structured results centrally. wiki has not
adopted it. Its `internal/mcp` handlers still call the **deleted**
`appkitmcp.JSONResult` on every success path (20 sites) and route every failure
through a `toolError(text string)` helper that calls the **re-signed**
`appkitmcp.ErrorResult` with no code — so wiki does not compile against the
current appkit, its domain results carry no `structuredContent`, no domain tool
declares an `outputSchema`, and its errors carry no machine code. This Decision
adopts the suite contract for wiki's 13 domain tools.

**Decision.** `internal/mcp`'s domain tools conform to the suite result
contract. Four coordinated changes, all inside `wiki/internal/mcp` (the transport
and the `health`/`reflection` tools are chassis-owned — D53 — and untouched):

- **Success results emit `structuredContent` + mirrored text.** Every domain
  handler's success path swaps `appkitmcp.JSONResult(v)` →
  `appkitmcp.StructuredResult(v)`. Both return `(map[string]any, error)`, so the
  Go error is already threaded at every call site (the handlers return
  `(map[string]any, error)`); the change is that the result now carries
  `structuredContent` (the typed value `v`) **and** a `content[0].text` block
  that is `v` marshalled to JSON — the same value on both channels. The
  reflect-based `public*Result` / `askToolResult` / `pagedResult` shaping that
  builds `v` is unchanged: the machine payload is exactly the JSON wiki emitted
  before, now also surfaced structured. A `StructuredResult` marshal failure is
  the one genuinely-unexpected path that returns a non-nil Go error (JSON-RPC
  `-32603`); every domain outcome still returns a nil Go error with an `isError`
  or success **result**, as D53 established.

- **Typed error codes on every failure.** The `toolError(text string)` helper is
  replaced by code-carrying helpers over `appkitmcp.ErrorResult(code, msg)`, and
  every failure site is classified against the closed vocabulary. wiki uses three
  of the six codes; the mapping is closed and exhaustive over the current sites:

  | code | failure sites |
  |---|---|
  | `validation` | malformed arguments (`decodeArgs`/`decodeJobIDArgs`/`decodeJobsArgs` unmarshal failure); a missing required field (`text`/`from`/`to`/`job_id`/`question`/`subject` empty); a non-RFC3339 `since`/`until`; an invalid `cursor`; `merge` where `from` and `to` resolve to the **same** subject; a missing authenticated identity on `ingest`/`merge`/`ask` |
  | `not_found` | an unknown `job_id` on `status`/`abort`/`rerun`; an unknown subject `type/slug` path on `merge` (either `from` or `to`), `claims`, or `page` — the sites that today return the `{found:false,kind,id}` **success** sentinel via `JSONResult(notFound(...))` |
  | `internal` | an opaque error surfaced from an injected domain service (`h.ingest`/`h.status`/`h.abort`/`h.rerun`/`h.jobs`/`h.jobsCount`/`h.merge`/`h.merges`/`h.ask`/`h.subjects`/`h.claims`/`h.page`/`h.calls` returning a non-`sql.ErrNoRows` error), and the defensive `"<tool> is not configured"` guards |

  `conflict`, `too_large`, and `source_unavailable` are **unused**: wiki's domain
  layer returns opaque `error` values that the MCP layer cannot classify into
  those finer codes without new domain error types, which is out of this
  adoption's scope. Those opaque failures map to `internal` — the honest code for
  "a lower layer failed and the surface cannot say more."

  The `not_found` rows are the one deliberate result-shape change: the former
  `notFound(kind, id)` **success** sentinel (`{found:false,kind,id}`, emitted via
  `JSONResult`) becomes a typed `ErrorResult(ErrNotFound, "<kind> <id> not
  found")` **error** result (`isError: true`). This is the suite contract's
  point — an unknown id is delivered as a code a machine branches on, not as a
  success payload a caller must inspect for `found:false`. It is **not** a reshape
  of any genuine success payload; every success payload's fields are preserved
  verbatim. `ask`'s own `found` field is unaffected: `ask` returning
  `found:false` is a grounded honest-empty **answer** (D9/D60), a legitimate
  success, not a not-found error.

- **Each domain tool declares an `outputSchema`** mirroring the JSON it already
  emits verbatim. Each `*Tool()` descriptor map gains an `"outputSchema"` entry (a
  hand-authored `map[string]any` JSON-schema literal in the appkit house style —
  `type`/`properties`/`required`, `objectSchema`-shaped), and `domainTool` reads
  `desc["outputSchema"]` into `Tool.OutputSchema` when present. The schemas
  formalize the existing `public*Result` shapes; they do not reshape them. The 13
  literals, one per tool:

  | tool | success `structuredContent` shape (from `public*Result`) |
  |---|---|
  | `ingest` | `{job_id: string}` (req `job_id`) |
  | `status` | `{status: string, received_at, started_at, finished_at, error: string, subjects: [string]}` (req `status`; the three timestamps are nullable strings) |
  | `abort` | `{aborted: bool, status: string}` (req both) |
  | `rerun` | `{requeued: bool, status: string}` (req both) |
  | `jobs` | `{jobs: [{id, owner, title, tags:[string], status, received_at, started_at, finished_at, error}], next_cursor: string}` |
  | `jobs_count` | `{count: integer}` (req `count`) |
  | `merge` | `{job_id: string}` (req `job_id`) |
  | `merges` | `{merges: [{norm_name, subject_id, name, created_by, created_at}], next_cursor: string}` |
  | `ask` | `{found: bool, answer: string, citations: [{url, title}]}` (req `found`, `answer`, `citations`) |
  | `subjects` | `{subjects: [{path, type, name, has_page: bool}], next_cursor: string}` |
  | `claims` | `{claims: [{id, text, job}], next_cursor: string}` |
  | `page` | `{subject: string, title: string, body: string}` (req all three) |
  | `llm_calls` | `{llm_calls: [{id, stage, job_id, attempt: integer, provider, model, params, request, response, usage, error, started_at, ended_at}], next_cursor: string}` |

  Unspecified property types are `string`. The paged wrappers declare `next_cursor:
  string` and the named array; array item objects declare their fields. The
  chassis `health`/`reflection` schemas are appkit's, not wiki's.

- **Prose exceptions.** `guide` is the sole prose tool: `handleGuideCall` returns
  `appkitmcp.TextResult(guideDoc)` (a plain text result, no `structuredContent`),
  and `guideTool()` declares **no** `outputSchema` (`Tool.OutputSchema` stays nil,
  so `tools/list` omits it for `guide`). wiki has no `describe` tool. `page` is
  **not** a prose exception: it returns a structured `{subject, title, body}`
  record whose `body` is a Markdown string field — a machine-consumable record,
  not a raw-content read — so it takes `StructuredResult` and an `outputSchema`
  like any other domain tool.

**No loopback-guard change.** wiki hand-copies no `X-Owner-Email OR
X-Forwarded-Proto` loopback predicate: its only loopback-private endpoint is
`/feed`, mounted and wrapped by the chassis (`appkit/server` mounts `GET /feed`
inside `LoopbackOnly`), and its web/`static` routes are session-gated at nginx,
not loopback-guarded in Go. There is nothing to swap to the shared
`Router.HandleLoopback` / `server.LoopbackOnly` helper.

**Relationship to D53/D57.** This Decision owns wiki's current MCP **result
contract** (structured content, typed error codes, output schemas). It supersedes
the result-shape claims D53 and D57 made when they described the transport de-fork
and the self-discovery adoption as result-neutral: D53's "`tools/call` results and
`isError` envelopes byte-for-byte unchanged" and D57's whole-surface byte-for-byte
neutrality (R-YINV-5EPR) are narrowed in place to names/inputs/validation/logical
result **fields**, because the result **envelopes** (structuredContent, mirrored
text, typed error codes) are now this Decision's, and the `not_found` sentinel is
now a typed error. Tool **names**, **input schemas**, decoders, and every success
payload's **fields** are unchanged.

**Rejected.**
- **Keep `notFound` as a `{found:false}` success sentinel.** It leaves the
  `not_found` vocabulary member dead for wiki and forces the six affected tools
  into a dual-shape output (a success object *or* the sentinel) that no single
  honest `outputSchema` describes. The suite contract delivers an unknown id as a
  typed error precisely so a machine branches on the code; the sibling services
  (crm/ledger) already treat not-found as an error code. Converting the sentinel
  is adopting the contract, not reshaping a genuine success payload.
- **A runtime response-vs-schema validator in wiki.** The suite contract makes
  conformance the service test suite's job, not the transport's
  (`../docs/structured-mcp-design.md`, Non-goals); appkit surfaces `outputSchema`
  and helps produce conforming results, it does not validate every response.
- **Mint new domain error types to reach `conflict`/`source_unavailable`.** That
  is a domain-layer change beyond this surface adoption; opaque domain failures
  map honestly to `internal` until a later Decision gives the domain typed errors.
- **A second loopback domain API for machine callers.** The suite rejected the
  fourth surface outright; MCP with `structuredContent` is the single verb surface
  (`../docs/structured-mcp-design.md`, Decisions resolved).

**Verification** (driven through the assembled `appkit/mcp` handler `NewHandler`
builds over the wired domain services — the D53 seam — with a stubbed identity,
per the design's MCP testing strategy).
- R-EMCA-BCU2 — A domain success result carries **both** channels mirrored: for a
  representative tool (`ingest`), `tools/call` returns a result whose
  `structuredContent` equals the emitted value (`{job_id: …}`) **and** whose
  `content[0].text` is that same value marshalled to JSON — proving the
  `JSONResult`→`StructuredResult` swap surfaced the machine channel a bare
  `JSONResult` result never carried (a text-only result, with no
  `structuredContent`, fails).
- R-ENK6-P4KR — Every one of wiki's **13 domain tools** (`ingest`, `status`,
  `abort`, `rerun`, `jobs`, `jobs_count`, `merge`, `merges`, `ask`, `subjects`,
  `claims`, `page`, `llm_calls`) surfaces a **non-nil** `outputSchema` in
  `tools/list`, and the prose tool `guide` surfaces **none** (table-driven over
  `tools/list`; a domain tool missing `outputSchema`, or `guide` carrying one,
  fails).
- R-EPZZ-GO25 — `tools/call guide` returns a **plain text** result:
  `content[0].type == "text"` with a non-empty body and **no** `structuredContent`
  key — the sole prose exception (a `guide` result carrying `structuredContent`
  fails).
- R-ER7V-UFSU — Each domain tool's declared `outputSchema` **mirrors** the payload
  it emits: for every one of the 13 tools, the top-level keys of a representative
  success `structuredContent` are all declared in that tool's
  `outputSchema.properties` (and the schema's `required` fields are all present in
  the emitted payload) — table-driven, so a schema that names a field the tool
  never emits, or omits one it always does, fails. Nested item shapes are checked
  for the paged tools (`jobs`/`subjects`/`claims`/`merges`/`llm_calls`) and
  `ask`'s `citations` (`{url,title}`).
- R-ESFS-87JJ — Input-validation failures each return an `isError` result with
  `structuredContent.code == "validation"`, table-driven over the triggers:
  malformed JSON arguments; a missing required field (`ingest` empty `text`,
  `merge` empty `from`/`to`, `status`/`abort`/`rerun` empty `job_id`, `ask` empty
  `question`, `claims`/`page` empty `subject`); a non-RFC3339 `since`/`until`; an
  invalid `cursor`; and `merge` with `from` and `to` resolving to the same subject
  (any of these returning a code other than `validation`, or no code, fails).
- R-ETNO-LZA8 — An unknown `job_id` (`status`/`abort`/`rerun`) or unknown subject
  `type/slug` path (`page`/`claims`/`merge` `from`) returns an `isError` result
  with `structuredContent.code == "not_found"` — **not** a successful
  `{found:false,kind,id}` payload — table-driven over those sites (a not-found that
  comes back as a non-`isError` result, or with a code other than `not_found`,
  fails).
- R-EW3H-DIRM — An opaque domain-layer failure — an injected domain service (e.g.
  the `ask` or `ingest` backend) returning a non-`sql.ErrNoRows` error — returns an
  `isError` result with `structuredContent.code == "internal"` (a generic domain
  failure surfacing as `validation`/`not_found`, or with no code, fails).
- R-EXBD-RAIB — Every tool error result is a well-formed envelope: `isError ==
  true`, the human message in `content[0].text`, **and** a
  `structuredContent.code` drawn from the closed appkit vocabulary
  (`validation`/`not_found`/`conflict`/`too_large`/`source_unavailable`/`internal`)
  — an error missing `isError`, missing the text message, or carrying a code
  outside the closed set fails.
