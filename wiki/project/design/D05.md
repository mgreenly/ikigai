# Decision 5 — The LLM seam (`internal/llm`): shared Provider, fresh Conversation per call, json-mode helper

**Decision.** agentkit exposes no structured-output / JSON mode and no forced `tool_choice` (research §0.3), so structured JSON is wiki's responsibility, implemented once in `internal/llm` and reused by extract (D6) and compile (D7). The agent path (`ask`, D9) uses the same Client to build a tool-bearing Conversation.

```go
package llm

// Client holds the shared, stateless agentkit Provider (built once at the
// composition root from anthropic.New(cfg.AnthropicKey)). Concurrency-safe.
type Client struct {
    prov agentkit.Provider
    log  io.Writer
}
func New(prov agentkit.Provider, log io.Writer) *Client

// CallSite is the per-stage generation config (extract / compile / ask each have one).
type CallSite struct {
    Model           string                  // e.g. anthropic.ModelSonnet46
    Temperature     *float64                // nil = provider default; extract/compile pin 0
    Reasoning       agentkit.ReasoningValue // extract/compile: DisableReasoning(); ask: Level("low"…)
    System          string                  // the stage's system prompt
    MaxParseRetries int                     // bounded re-prompt on parse/validate failure
    Stage           string                  // call-record label (D13): "extract"|"compile"|"ask-subject"|"ask-synthesis" (D19), plus eval-only "judge" (D21)
    MaxTokens       int                     // output-token ceiling, applied to GenSettings (D18); >0 in prod
}

// JSON runs ONE tool-less structured generation and returns a validated T.
// Builds a FRESH *agentkit.Conversation each call (Provider shared, Conversation
// single-flight): Conversation{Provider: c.prov, Model: site.Model, System: site.System,
// Gen: GenSettings{Temperature: site.Temperature, Reasoning: site.Reasoning, MaxTokens: site.MaxTokens}}.
// Flow: Send(ctx, userText) → drain stream → final message text → ExtractJSON →
// json.Unmarshal into T → validate(&T). On unmarshal/validate error, re-prompt with a
// corrective note up to MaxParseRetries, then return an error (never a silent zero T).
// A truncation (output usage at the MaxTokens ceiling, D18) is returned immediately as a
// distinct ErrTruncated WITHOUT re-prompting — a re-prompt under the same ceiling truncates
// identically. The whole-turn ctx deadline is the request's.
func JSON[T any](ctx context.Context, c *Client, site CallSite, userText string, validate func(*T) error) (T, error)

// Converse builds a fresh tool-bearing Conversation for the agent path (ask, D9).
func (c *Client) Converse(site CallSite, tools []agentkit.Tool) *agentkit.Conversation
```

- **`ExtractJSON`** (exported — the **single** carve, reused by the agent path) carves the **outermost JSON value** out of a possibly-decorated reply rather than pattern-matching a fence: trim, pick whichever of the first `{` or first `[` opens earliest, return the span from there to the **matching last** `}` or `]`. Surrounding decoration — a ```` ```json ```` fence, a non-standard/extra-backtick fence, a prose preamble (`Here is the JSON:`) or trailing commentary, a stray leading `` ` `` — falls outside the carved span and is discarded before `json.Unmarshal`. No-op on already-bare JSON. **Replaces** the ported `stripCodeFence`, whose exact-triple-backtick assumption let an unanticipated fence form leave a stray `` ` `` before the `{`, failing ingest with `invalid character '`' looking for beginning of value`.
- **One carve, everywhere** — `ExtractJSON` is the **only** JSON-from-a-decorated-reply implementation in the service: `JSON[T]` uses it, and the agent path `ask` (D9) parses its final message through the same function. No second fence-stripper anywhere.
- **Fresh Conversation per call** — no history bleeds between extract/compile/ask; the Provider is the only shared, reused object (stateless, concurrency-safe).
- **`validate`** is caller-supplied (extract D6 / compile D7), so the retry loop and the domain rules stay co-located without `internal/llm` knowing the domain.
- Stage **values** (model; temp; reasoning; `Stage` label) belong to D6/D7/D9; this seam only guarantees a `CallSite` is faithfully applied to the Conversation. Every provider round-trip is recorded via the D13 `Recorder` (the `sendText` capture point); D13 owns the record shape and job-id/attempt threading.

**Rejected.**
- **A forced-tool `NewTool[Out]("emit",…)` to coerce structure** — agentkit can't force `tool_choice`; the model may not call the tool. Prompt-and-parse is lighter and deterministic for batch ingest (kept only as a noted fallback).
- **Reusing one Conversation across calls** — single-flight, not concurrency-safe, leaks history between stages.
- **A non-generic `Raw(ctx,site)→string` with unmarshal pushed to callers** — duplicates the retry-on-parse loop in every stage; generic `JSON[T]` keeps it in one place.
- **Keeping the fence-only `stripCodeFence`** — passed any non-`` ``` `` decoration through verbatim to `json.Unmarshal`. Span-carving subsumes every fence case plus the decorated cases the matcher missed.

**Verification.**
- R-J8QP-BETB — `ExtractJSON` yields parseable JSON from a ```` ```json ````…```` ``` ```` fenced reply, from a bare ```` ``` ```` fenced reply, and from already-bare JSON (no-op), by carving the first-`{`/`[`-to-matching-last-`}`/`]` span.
- R-4BCC-0EHJ — `ExtractJSON` recovers the JSON value when the reply carries decoration the fence-only matcher could not: a prose preamble and/or trailing commentary, a non-standard or extra-backtick fence, or a stray leading `` ` `` before the opening `{` — the carved span parses where the unmodified reply fails with a leading-character error. (Regression guard for the fenced-extraction ingest failure.)
- R-J9YL-P6K0 — `JSON` returns the validated `T` for a well-formed (including fenced) response, with the caller's `validate` applied.
- R-JCEE-GQ1E — `JSON` retries on unmarshal/validate failure up to `MaxParseRetries` (bad-then-good succeeds) and otherwise returns an error rather than a silent zero `T` (always-bad errors).
- R-JDMA-UHS3 — Each `JSON` call builds a fresh Conversation carrying no history from a prior call (two sequential calls show the provider no accumulated messages).
- R-JEU7-89IS — The Conversation is constructed with the `CallSite`'s Model, System, Temperature, and Reasoning (a capturing mock provider confirms them).
