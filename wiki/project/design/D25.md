# Decision 25 — Aliases table & name resolution

**Decision.** A manual merge folds one subject into another and deletes the folded subject. Its *name* must not vanish, or the next ingest mentioning it would mint a fresh duplicate — so a merge leaves a durable forward-routing record in a new `aliases` table, and a single shared `Resolver` consults that table on every name lookup. This Decision owns the table, the `AliasStore` that writes it, and the `Resolver` that reads it. The merge execution that *populates* the table is D26; the MCP surface is D27.

An **alias** is not a subject: no page, no claims, no identity — only a `dead name → surviving subject` pointer. Keeping aliases in their own table (not a `merged_into` flag on `subjects`) means a dead name is *physically absent* from the table every listing, page build, and link projection reads, so none can surface a folded subject. The redirect is consulted by exactly one step — name resolution.

## The table

One timestamped migration (`bin/new-migration wiki create_aliases`). This introduces the schema's **first `FOREIGN KEY`** — a deliberate, owner-approved departure from the D3 convention ("no `FOREIGN KEY`; `UNIQUE`/`CHECK` only"), under the standing rule *"FK wherever it aids integrity at low cost."*

```sql
CREATE TABLE aliases (
  norm_name  TEXT NOT NULL UNIQUE,                -- Normalize(folded name) (D3, the one normalizer); the resolution key
  subject_id TEXT NOT NULL
             REFERENCES subjects(id) ON DELETE RESTRICT,  -- the surviving subject
  name       TEXT NOT NULL,                       -- folded display name at merge time
  created_by TEXT NOT NULL,                       -- X-Owner-Email of the approving owner
  created_at TEXT NOT NULL                        -- RFC3339Nano, matching formatTime (data_model.go:709)
);
CREATE INDEX aliases_subject ON aliases (subject_id);  -- supports RepointSubject's sweep (D26)
```

**Why the FK, and why `ON DELETE RESTRICT`.** The invariant the resolver depends on is *every `aliases.subject_id` names a live subject*. With `ON DELETE RESTRICT`, the engine refuses to delete a subject while any alias still points at it — turning an in-code promise into a DB-enforced constraint, and forcing D26's merge to repoint the folded subject's inbound aliases onto the survivor *before* deleting it. FK enforcement is already globally on (`appkit/db/db.go` opens with `_pragma=foreign_keys(ON)`); restore is file-level (`VACUUM INTO` + copyfile), so no row-order import hazard. (D26's `jobs.merge_loser_id` carrier deliberately gets **no** FK — the merge deletes the loser, and a FK there would block it.)

Existing tables (`claims`, `pages`) stay FK-free: retrofitting needs table rebuilds — a separate deferred item, out of scope here.

## AliasStore

A new store in `internal/wiki` over the same `sqlStore` abstraction (`*sql.DB` or `*sql.Tx`) as the existing stores, so D26's merge builds a tx-bound one. Both `Insert` and `GetByNormName` key on **`Normalize(name)`** — the single D3 normalizer, same form `SubjectStore` uses — so the `Resolver` gets one consistent key form across `subjects` and `aliases`:

```go
func NewAliasStore(db sqlStore) *AliasStore

// Insert the forward-routing row (folded norm_name → survivor id). Plain INSERT;
// the norm_name UNIQUE is the guard — a collision is a real invariant breach and
// must fail loudly, not be silently overwritten (v1 has no path that legitimately
// re-inserts an existing alias name — see Rejected).
func (a *AliasStore) Insert(ctx context.Context, al Alias) error

// RepointSubject moves every alias currently pointing at `from` to `to`. D26 runs
// this BEFORE deleting the folded subject, so no alias is ever left pointing at a
// deleted subject and resolution stays single-hop (path compression at write time).
func (a *AliasStore) RepointSubject(ctx context.Context, from, to string) error

// GetByNormName is a point lookup on the norm_name UNIQUE; sql.ErrNoRows on miss.
func (a *AliasStore) GetByNormName(ctx context.Context, normName string) (Alias, error)

// ListAll returns every alias row (NormName + SubjectID suffice for the caller).
// Consumed by D12's PageWithLinks to build the alias-aware SubjectKeys for the
// read-time link projection; aliases are few, so a single unfiltered read is cheap
// and stays within the existing per-read O(N) projection cost.
func (a *AliasStore) ListAll(ctx context.Context) ([]Alias, error)

type Alias struct {
    NormName  string
    SubjectID string
    Name      string
    CreatedBy string
    CreatedAt string // RFC3339Nano
}
```

## Resolver — the single shared name lookup

Name resolution spans two tables (`subjects` for live things, `aliases` for redirects), so it is its own seam — a dedicated `Resolver` composing both stores, not a method on `SubjectStore` (which would force that store into the aliases table). Neither underlying store knows about the other.

```go
func NewResolver(db sqlStore) *Resolver

// ResolveByName maps a display name to its live subject, honoring merges:
//   1. SubjectStore.GetByNormName(name) — a live subject? return it. (A name re-minted
//      as a live subject shadows any stale alias of the same spelling: subjects-first.)
//   2. sql.ErrNoRows → AliasStore.GetByNormName(Normalize(name)) — an alias? load the
//      survivor via SubjectStore.Get(subject_id) and return it.
//   3. neither → ErrSubjectNotFound.
//
// Single-hop is PROVABLE, not depth-bounded. The FK (ON DELETE RESTRICT) plus D26's
// eager RepointSubject guarantee every alias.subject_id names a LIVE, non-aliased
// survivor — so step 2's Get never lands on another alias. No chain-following, no
// depth bound, no cycle logic.
func (r *Resolver) ResolveByName(ctx context.Context, name string) (Subject, error)
```

Both hot-path name-lookup sites adopt `ResolveByName`, so the forward-routing effect reaches both ingest and ask (ask, in a separate package, previously called `GetByNormName` directly and could silently bypass it):

- **Ingest** — `plannedSubject` resolves between the current `GetByNormName` miss (`service.go:379`) and the mint (`service.go:382`): an aliased name updates the survivor instead of minting a duplicate.
- **`ask`** — `gatherPages` (`ask/ask.go:142,157`) resolves the question's named subjects through it.

Both remain O(log n): at most two indexed UNIQUE point lookups, no scan, no LLM.

## Rejected.

- **No FK — keep `aliases` FK-free like every other table (the D3 convention).** Rejected under "FK where it aids integrity at low cost." Without it, "no alias points at a dead subject" is only an in-code promise; with it, the engine refuses the bad delete. Cost is low — enforcement is already on, restore is file-level. (Retrofitting `claims`/`pages` needs table rebuilds; stays deferred.)
- **A `merged_into` flag column on `subjects`.** Rejected: every listing, page build, claim count, and link projection would have to remember to exclude folded rows forever; one forgotten filter leaks a ghost. A separate table makes a dead name *not a subject* by construction.
- **Depth-bounded alias-chain following with cycle detection** (research §6). Rejected: the no-chain invariant erases the problem. Eager repointing at write time means chains never form, so `ResolveByName` is a provable single hop.
- **`ResolveByName` as a `SubjectStore` method.** Rejected: resolution touches two tables; a dedicated `Resolver` keeps each store ignorant of the other.
- **Aliases-first resolution order.** Rejected for subjects-first: a re-minted live subject must shadow a stale alias of the same spelling, and subjects-first is the cheaper common case (one probe, no alias touch).
- **`INSERT OR REPLACE` on alias insert.** Rejected for a plain `INSERT`: with no alias-delete verb, the only way a norm_name re-enters is via `RepointSubject` (an UPDATE), never a colliding INSERT — so the UNIQUE is a real guard that should fail loud.

## Verification.
Tested against a **real temp SQLite** opened by the appkit migration runner with `foreign_keys=ON` — a mock cannot falsify a constraint, and several of these behaviors *are* the constraints. The two pure-resolution cases compose store calls against that same real DB.

- R-BGPF-NVTU — Inserting a second `aliases` row whose `norm_name` equals an existing row's is rejected by the `UNIQUE(norm_name)` constraint (one folded name routes to at most one survivor). *(real SQLite)*
- R-BHXC-1NKJ — With `foreign_keys=ON`, deleting a `subjects` row while an `aliases` row references its id is **rejected by the `ON DELETE RESTRICT` foreign key**, and the subject row still exists afterward; the delete succeeds only once the referencing alias has been repointed away. *(real temp SQLite, `foreign_keys=ON` — the can-not-mock id; the observable proof is the DELETE returning a constraint error and the row surviving, not merely that an FK clause is present.)*
- R-BJ58-FFB8 — After `AliasStore.RepointSubject(loser, winner)`, every alias row that named `loser` as its `subject_id` now names `winner`, and none names `loser` — so a subsequent delete of `loser` passes the FK. *(real SQLite)*
- R-BKD4-T71X — When a norm_name exists as **both** a live subject and an alias row, `ResolveByName` returns the **live subject**, never the alias's survivor (subjects-first shadowing). *(real SQLite)*
- R-BLL1-6YSM — When a norm_name is not a live subject but an alias names survivor `W`, `ResolveByName` returns `W` via a single `Get` and does not perform a second alias lookup on `W`'s own name (single hop). *(real SQLite)*
- R-BMSX-KQJB — A name that is neither a live subject nor an alias returns `ErrSubjectNotFound` — a clean miss, not a wrong subject or a panic. *(real SQLite)*
- R-BO0T-YIA0 — With an alias `Normalize(L) → W` installed, the ingest mint path (`plannedSubject`) processing text mentioning `L` resolves to and updates `W` rather than minting a new subject for `L`. *(worker seam + real DB; alias row installed directly via `AliasStore`, no merge needed)*
- R-BP8Q-CA0P — With the same alias installed, `ask`'s `gatherPages` resolves `L` to `W`'s page rather than treating `L` as unknown — proving `ask` consults the shared `Resolver` and does not bypass the alias effect. *(ask seam + real DB)*
