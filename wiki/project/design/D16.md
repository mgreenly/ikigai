# Decision 16 — MCP surface expansion: control & footprint verbs + paginated lists

**Decision.** The MCP surface gains five verbs — **`jobs`** (filtered, paginated, newest-first job list), **`jobs_count`** (filtered total), **`abort`**, **`rerun`**, **`llm_calls`** (filtered, paginated footprint) — and the existing **`subjects`** and **`claims`** are reshaped to cursor-paginated `{items, next_cursor}` envelopes accepting `limit`/`cursor`. The surface grows from eight to **thirteen**. Wired via the existing `WithXxxService` Option pattern, dispatched by `handleToolCall`, shaped by the existing reflection-based result helpers; nginx remains the trust boundary (`RequireIdentity`), wiki does no token logic. Supersedes D10's eight-verb framing (D10 edited in place: its R-MUQ4 now asserts the thirteen-verb set).

The `jobs` filter exposes the D15 set/newest-first behavior on the wire: `status` is an **array** of states (match-any), the page comes back **newest-first**, and the closed five-state vocabulary is **published in the tool schema**. `jobs_count` reuses the same filter inputs and returns `{count}`.

## The verbs

| tool | input | → result |
|---|---|---|
| `jobs` *(new)* | `{status?: [state…], kind?: [kind…], since?, until?, limit?, cursor?}` | `{items:[{job_id, status, title, received_at, started_at?, finished_at?, error}], next_cursor}` — **newest-first** |
| `jobs_count` *(new)* | `{status?: [state…], kind?: [kind…], since?, until?}` | `{count: <int>}` |
| `abort` *(new)* | `{job_id (req)}` | `{job_id, aborted: bool, status}` |
| `rerun` *(new)* | `{job_id (req)}` | `{job_id, status}` or a clean tool error (in-progress / not-found) |
| `llm_calls` *(new)* | `{job_id?, stage?, since?, until?, limit?, cursor?}` | `{items:[{id, job_id, stage, attempt, provider, model, params, request, response, usage, error, started_at, ended_at}], next_cursor}` |
| `subjects` *(reshaped)* | `{type?, q?, limit?, cursor?}` | `{items:[{path, type, name, has_page}], next_cursor}` |
| `claims` *(reshaped)* | `{subject (req), limit?, cursor?}` | `{items:[{id, text, job}], next_cursor}` |

`jobs` and `jobs_count` share one filter-parse helper mapping `{status, since, until}` → `wiki.JobFilter` (`status` array → `JobFilter.Statuses`, validated; `since`/`until` → time range); `jobs` adds `limit`/`cursor`, `jobs_count` does not. These back onto the D14 service methods (`Abort`, `Rerun`) and D15 seams (`ListJobs`, `CountJobs`, `SubjectStore.List`, `ClaimStore.ListBySubject`, `LLMCallStore.List`). New Options: `WithJobsService`, `WithJobsCountService`, `WithAbortService`, `WithRerunService`, `WithLLMCallsService`. The `tools()` builder appends each when its service is wired.

## Design rules

- **No internal id crosses the boundary.** `job_id`, the `llm_calls` row `id`, and the `claims` `id`/`job` are public handles wiki already mints; every subject reference stays a `type/slug` path. `llm_calls` carries **no subject id** — keyed by job, stage, time only.
- **Time filters are RFC3339 strings** (`since`/`until`) parsed to `time.Time`; an unparseable timestamp returns a clean tool error, never a crash.
- **The `status` filter is an array, each element validated** against the closed five-state set (`pending`/`working`/`done`/`failed`/`aborted`). Empty/omitted means all states; non-empty filters match-any (OR). Any value outside the five yields a clean tool error whose message **names the valid set**, never a silent empty page. Governs both `jobs` and `jobs_count`.
- **The valid states are published in the schema.** The `status` item carries `enum: ["pending","working","done","failed","aborted"]` and the description names the five, so callers discover the vocabulary from `tools/list`. (The closed set is the product contractual constant, honored in Go.)
- **`limit`/`cursor`** map onto `page.Params` (D15); an undecodable cursor (D15 typed error) renders as a clean tool error.
- **`llm_calls` returns request/response inline**; pagination (default 50, D15) bounds the payload, `job_id`/`stage`/time narrow it.
- **Not-found / not-actionable are results, not crashes** (existing convention): `abort`/`rerun`/`status` on an unknown `job_id` return a clean not-found; `abort` on a terminal job returns `{aborted:false, status:<current>}`; `rerun` on a pending/working job returns the D14 "can't re-run a job in progress" error.
- **Schema validity.** The new no-required verbs (`jobs`, `jobs_count`, `llm_calls`) and the still-no-required `subjects` emit `inputSchema` via the shared `objectSchema`, which omits `required` when empty (never `"required": null`) — the D10 R-N4KO strict-client guard, now covering the added verbs.
- **The `kind` filter defaults to `ingest`.** `jobs`/`jobs_count` accept an optional `kind` array over the closed set `{ingest, merge}` (D26). Unlike `status`, an **omitted `kind` means `ingest` only** — a `kind='merge'` work item (D26) does not appear unless asked for. `kind:["merge"]` lists merge jobs; `kind:["ingest","merge"]` both. Every element validated against the two-value set; an unknown `kind` yields a clean tool error naming the valid set. `JobFilter` gains a `Kind` field; the merge *operation* lifecycle is visible here, while the human fold record is D27's `merges` audit (read from `aliases`, not this table).
- **Ungated this release.** All fifteen verbs ship (the thirteen here plus `merge`/`merges`, D27). The product's forward intent — moving part of the surface behind a `--debug` switch — is **not** built now; the control/footprint verbs are its eventual candidates. Recorded as intent, not mechanism.

## How it's tested

Driving the JSON-RPC handler over an in-process server with stubbed identity (D10 strategy), against D14/D15 services on a real temp SQLite + scripted mock provider:
- `tools/list` returns thirteen verbs, each `inputSchema` well-formed — the no-required verbs (`jobs`, `jobs_count`, `subjects`, `llm_calls`) omit `required` (never `null`); the `jobs`/`jobs_count` `status` item carries the five-state `enum`.
- `jobs` with a single-state `status` array returns only matching jobs as a `{items, next_cursor}` page, newest-first; the cursor walks the remainder.
- `jobs` with a multi-state array (e.g. all but `done`) returns the union in one page, newest-first.
- `jobs_count` with the same filter inputs returns `{count}` matching what `jobs` would page through, without items or cursor.
- An unknown `status` value (either verb) returns a clean tool error naming the valid five-state set, not an empty page.
- `abort` returns `{aborted:true}` for pending/working, `{aborted:false, status:<terminal>}` for terminal, not-found for unknown id.
- `rerun` returns `{status:"pending"}` for terminal, a clean error for pending/working, not-found for unknown id.
- `llm_calls` filtered by `job_id` returns that job's records with `request`/`response`/`params`/`model` populated, as a page; `stage` and time-range each narrow it.
- `subjects` and `claims` return `{items, next_cursor}`, accept `limit`/`cursor`, items keep `type/slug` paths with no internal id.
- A malformed `since`/`until`, unknown `status`, or undecodable `cursor` each yield a clean tool error, not a crash.

**Rejected.**
- **Gating the new verbs behind `--debug` now** — the product defers the switch; building it now is speculative. Ship ungated, note the candidates. Rejected (deferred).
- **A combined `job <action>` mega-verb** — muddies schema and result shape, hides three actions behind one name; separate verbs match the one-verb-one-job style (D10). Rejected.
- **A reference/summary-only `llm_calls` (fetch body separately)** — adds a round-trip and a second verb for the exact payload wanted. Rejected.
- **A non-paginated `llm_calls`** — the highest-volume table; the one place pagination is non-negotiable. Rejected.
- **A `count_only: true` flag on `jobs`** — forces one verb to return two shapes; a distinct `jobs_count` keeps one-verb-one-job and reuses the identical filter-parse. Rejected.
- **An untyped/string `status` filter** — a typo silently matches nothing, reading as "no such jobs" rather than "you asked wrong"; an enum-constrained array that fails loudly is the promise. Rejected.

**Verification.**
- R-37NS-BRXR — `jobs` returns a `{items, next_cursor}` page whose items carry `job_id`/`status`/timestamps; a `status` array restricts the page to jobs in any of the given states (match-any), and the returned `next_cursor` fetches the following page.
- R-Y36L-E3W6 — `jobs_count` returns `{count}` equal to the number of jobs matching its filter (`status` set and/or time range), with no items or cursor, using the same filter inputs as `jobs`.
- R-Y4EH-RVMV — The `jobs`/`jobs_count` `status` filter publishes the closed five-state vocabulary in its `inputSchema` (`enum`), and an unrecognized state value returns a clean tool error naming the valid set rather than a silently empty result.
- R-38VO-PJOG — `abort` returns `{aborted:true, status:"aborted"}` for a pending or working job, `{aborted:false, status:<terminal>}` for an already-terminal job, and a clean not-found for an unknown `job_id` (never a crash).
- R-3A3L-3BF5 — `rerun` returns `{status:"pending"}` for a terminal job, a clean tool error for a pending/working job, and a clean not-found for an unknown `job_id`.
- R-3BBH-H35U — `llm_calls` returns a `{items, next_cursor}` page of records carrying `request`/`response`/`params`/`provider`/`model`/`attempt`; a `job_id` filter restricts to that job's calls and a `stage` filter restricts to that stage.
- R-3CJD-UUWJ — `subjects` and `claims` return `{items, next_cursor}` envelopes, accept `limit`/`cursor`, and their items carry `type/slug` paths (subjects) / public handles (claims) with no internal subject id.
- R-3EZ6-MEDX — A malformed `since`/`until` timestamp or an undecodable `cursor` each return a clean tool error rather than a crash or a silent reset. (An invalid `status` value is covered by R-Y4EH.)
- R-3G73-064M — `tools/list` returns the new verbs `jobs`/`jobs_count`/`llm_calls` (and the reshaped `subjects`) with `inputSchema` that omits `required` when empty — never `"required": null` — so a strict client accepts the whole list.
- R-E4WX-G9H2 — `jobs`/`jobs_count` accept an optional `kind` filter over `{ingest, merge}`: an **omitted** `kind` returns ingest jobs only (a `kind='merge'` job does not appear), `kind:["merge"]` returns merge jobs, and an unrecognized `kind` value returns a clean tool error naming the valid set (not a silently empty page).
