# Decision 36 — Preparing the question: one analysis call

**Decision.** Keep the single LLM call `ask` already makes about the question, but change what it produces. Instead of just a list of subject names, it now emits a small structured analysis: the question split into single-subject **sub-queries**, the salient **keywords**, and likely **aliases** (alternate names). This is the same call site as today — `ask-subject` (D19) — so no new call site, config knob, or log stage is needed.

```go
// QueryAnalysis is the prepared form of a question (internal/wiki).
type QueryAnalysis struct {
    SubQueries []string `json:"sub_queries"` // the question split per subject; ≤ 4
    Keywords   []string `json:"keywords"`    // salient terms/entities → keyword lane
    Aliases    []string `json:"aliases"`     // alternate names, to widen the keyword lane
}

// Analyze runs one ask-subject call and returns the parsed, capped analysis.
func Analyze(ctx context.Context, c *llm.Client, site llm.CallSite, question string) (QueryAnalysis, error)
```

It runs through the existing `llm.JSON[QueryAnalysis]` helper (D5) on the `ask-subject` site, so parsing, the carve, and bounded retries are already handled.

**Decomposition is the high-value part.** Each wiki page is about one subject, so a multi-subject question ("compare X and Y") must search for X's page and Y's page **separately** — a single blended query lands between them and matches neither. So the analysis splits such questions into per-subject sub-queries. The downstream fan-out and merge live in D33 (`SearchAnalyzed`), which routes the sentence to the meaning lane and the keywords/aliases to the keyword lane; this decision owns only producing the analysis.

**Caps and fallback (so it can't misbehave):**
- **Sub-queries are capped at 4.** A model returning more is clamped to the first 4 — beyond that, sub-queries add redundancy and cost, not recall.
- **Empty → use the whole question.** If the model returns no sub-queries, `Analyze` falls back to a single sub-query equal to the raw question, so search always has something to run. (Likewise an empty keyword list just means the keyword lane runs on whatever terms it can derive; it is never an error.)

**Prompt.** The `ask-subject` system prompt is rewritten to ask for the JSON above: break the question into the distinct subjects it asks about (one sub-query each, at most 4), list the key terms and any alternate names, and nothing else — no answering, no speculation. JSON schema + a worked example.

**This repurposes part of D9.** D9's stage 1 used to extract bare subject *names* for exact resolution; it now produces this richer analysis, and stage 2 (resolve-by-name) is replaced by hybrid retrieval (D9). The honest-empty and citation guarantees are unchanged and owned by D9.

**Rejected.**
- **No decomposition (embed the whole question once).** Cheaper, but multi-subject questions answer worse — the one case this upgrade most needs to fix.
- **Paraphrase fan-out / multiple rewordings of the same question.** Multiplies cost for recall on broad queries; decomposition already produces the *useful* kind of multiplicity (distinct subjects).
- **HyDE (have the model invent a hypothetical answer to embed).** Backfires on lossy fact pages — it fabricates plausible-but-wrong names and dates. Decomposition + aliases buys the alignment win without the hallucination tax.
- **A second new call site for analysis.** The existing `ask-subject` site already exists, is configured, and is logged; reuse it.

- R-QCF7-D0WJ — Sub-queries are capped at 4: a model response with more than four sub-queries is clamped to the first four.
