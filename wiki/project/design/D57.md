# Decision 57 — Adopt the MCP self-discovery convention: instructions, lean tool descriptions, a `guide` tool

**Problem.** `../docs/mcp-discovery-convention.md` is the normative pattern (piloted on crm) for a service's MCP surface: a connecting agent should discover and correctly drive the service **from the connection alone**, with no external skill. wiki predates it. Today wiki's `initialize` `instructions` (the D53 pinned string) names only its internal nouns, carries no user-vocabulary synonyms and no pointer to a reference; its tool descriptions are terse one-liners; and there is **no guide tool** holding the field catalogs and worked examples an agent needs to call `ingest`/`ask`/`merge` and the path-lookups correctly. Bring the surface in line with the convention's three tiers.

**Decision.** Adopt all five steps of the convention (`../docs/mcp-discovery-convention.md`, "Adopting the pattern in a service"). This is **surface-and-description only** — the domain tools' behavior, schemas' accepted inputs, and validation are untouched. (Result **envelopes** are separately reshaped by later Decisions: `ask`'s citation shape by **D56**, and every domain tool's structured-content/typed-error-code envelope by **D61** — both deliberate, neither a change this Decision makes.)

- **Tier 0 — rewrite `const Instructions`.** Replace the D53 string with one that names the domain in the vocabulary a user actually types (with everyday synonyms), states the verb flow in one clause, and points once at `guide`. Pinned text:

  > wiki is a knowledge base (notes, a second brain) built from source text you ingest. Ingest queues text; a background pipeline distills the subjects it concerns — entities, events, and concepts — and the claims about them, and compiles one cited page per subject. Ask it a question for a grounded, cited answer over your own wiki; read the compiled knowledge by type/slug path with subjects, claims, and page; track ingestion with status, jobs, and jobs_count; merge to fold a duplicate subject into another; and steer or inspect the pipeline with abort, rerun, and llm_calls. Call guide for the full field catalogs, the type/slug path format, and worked examples before your first ingest or merge.

  It stays orientation, not reference: no field catalog, no per-type option lists.

- **Tier 1 — slim/sharpen the tool descriptions.** Each domain tool's `Description` (the `*Tool()` descriptor funcs) says *when to use it*, *how to shape its arguments*, and *what it returns*, plus the cross-cutting semantics an agent needs to call it correctly (e.g. `ingest` is fire-and-return yielding a job handle; `page`/`claims`/`merge` take a `type/slug` **path**, not a bare name; `merge`/`abort`/`rerun` are irreversible/terminal-state constraints; the list verbs are cursor-paginated). Per-type and per-entity reference catalogs move **out** of the descriptions into `guide`. Keyword-forward, because tool search indexes descriptions. No tool description except `guide`'s own references the guide.

- **Tier 2 — add the `guide` tool.** One new tool, `guide`:
  - **Descriptor** `guideTool()` — name `guide`, an input-free schema (`objectSchema(map[string]any{}, nil)` → `{"type":"object"}` with `required` omitted, matching the empty-schema rule of D10 R-N4KO), and a description that says it returns the wiki usage guide and is the one place (besides `instructions`) that points at itself.
  - **Handler** `handleGuideCall` — returns the embedded guide document as an `mcp.TextResult` (a plain text result; the prose exception of D61 — no `structuredContent`, and `guideTool()` declares no `outputSchema`). It is **flat, read-only, input-free**: reads nothing, mutates nothing, ignores/accepts empty args, and does not error.
  - **Enablement** — `guide` is a non-domain tool needing no injected service, so `Tools(...)` appends it **unconditionally** (it is not gated behind a `WithXxxService`). Adding domain entities still adds no tools; the surface's tool count is a function of its verbs plus this one fixed reference tool.
  - **The document is embedded** (`//go:embed guide.md` in `internal/mcp`), versioned and shipped in the binary, so the guide an agent reads always matches the running service. Its content is wiki's own strings: the subject-type set (`entity`/`event`/`concept`), the `type/norm_name` path format and how a name normalizes to a slug, the job-lifecycle statuses (`pending`/`working`/`done`/`failed`/`aborted`) and which are filterable/terminal, the claim shape, the cursor-pagination contract (`limit`/`cursor` → `{items, next_cursor}`), and basic + advanced worked examples (ingest → poll `status` → `ask`; `subjects` → `page`; the `merge` survivor/folded flow; `llm_calls` inspection).

- **Reference the guide in exactly two places.** The Tier 0 `instructions` and the `guide` tool's own description — and nowhere else. Not in any other tool description (repeating it re-inflates Tier 1 for no gain; the always-loaded `instructions` pointer reaches the agent first).

- **`NewHandler` wiring.** `Instructions` is already passed to `appkitmcp.New` (D53); the constant's new text flows through unchanged. `Tools(...)` gains the unconditional `guide` append. Everything else in the composition root is as D53 left it.

**Tool-count supersession (relationship to D53).** D53 R-JKMR-5MV1 pinned "exactly fifteen" tools. Adding the wiki-declared `guide` makes the current membership **sixteen**. D53 is rewritten in place so it no longer asserts a count that is no longer true: R-JKMR-5MV1 now pins the **table/chassis partition** (wiki's 13 domain tools plus the chassis-supplied `health`/`reflection` are all present; a table that itself declares `health`/`reflection` fails `New`'s reserved-name check), and the **exact current membership** (the sixteen tools, including `guide`) is owned by **this** Decision (R-YF06-03HO). D53's frozen realizing phase keeps its history; the tool-count test is updated by this Decision's phase.

**Behavior neutrality.** Per the convention's step 5, this Decision's adoption is provably behavior-neutral for **every existing tool call**: the names, accepted inputs, validation, and logical result **fields** of `ingest`/`status`/`abort`/`rerun`/`jobs`/`jobs_count`/`merge`/`merges`/`subjects`/`claims`/`page`/`llm_calls` are as before — only their *descriptions* change (surface text, not behavior). Result-**envelope** changes are owned by other Decisions and excluded here: `ask`'s citation shape by **D56** (relative `path` → fully-qualified `url`), and every domain tool's structured-content-plus-typed-error-code envelope (and the `not_found` typing of the former success sentinel) by **D61**. This Decision changes no result.

**Rejected.**
- **Also expose the guide as an MCP resource.** The tool is the delivery guarantee (`tools/list` in every client); most clients do not surface resources into an agent's context. A resource is optional, unrequested, and a second surface to keep in sync — skip it; add later if a client ever needs it.
- **Reference the guide in every tool description.** Re-inflates Tier 1 on every connection for no gain; the convention pins the pointer to exactly the `instructions` and the guide's own description.
- **One `guide` per entity, or a parameterized guide.** wiki's guide fits well within the convention's "start flat, read-only, input-free" bar; scope it only if it grows past roughly one screen.
- **Fold this into D56.** The two changes are independent; keeping them separate keeps D56's neutrality-excluded `ask` change and this Decision's whole-surface neutrality claim cleanly separable, and gives each its own build phase.

**Verification** (through the composition-root seam — the `appkit/mcp` handler built by `NewHandler` over the wired domain services).
- R-YDS9-MBQZ — The `guide` tool is flat, read-only, and input-free: its `inputSchema` is `{"type":"object"}` with no `required`; `tools/call guide` (with empty or absent args) returns the embedded, **non-empty** guide document as a successful (non-`isError`) result, mutates no DB state (state identical before and after), and never errors.
- R-YF06-03HO — `tools/list` through the assembled handler returns **exactly sixteen** tools: wiki's 13 domain tools (`ingest`, `status`, `abort`, `rerun`, `jobs`, `jobs_count`, `merge`, `merges`, `ask`, `subjects`, `claims`, `page`, `llm_calls`) plus the wiki-declared `guide`, plus the chassis-supplied `health` and `reflection` — no more, no fewer. (Supersedes the fifteen-count of D53 R-JKMR-5MV1.)
- R-YG82-DV8D — The `initialize` `instructions` string names the domain in user vocabulary including everyday synonyms (contains "notes"/"second brain" and the type words "entities"/"events"/"concepts") and points to the guide (contains "guide") — proving Tier 0 is routing-vocabulary-forward, not internal-nouns-only.
- R-YHFY-RMZ2 — The guide is referenced in **exactly two** places: the `instructions` string and the `guide` tool's own `Description` both mention the guide, and **no other tool's** `Description` contains a guide pointer.
- R-YINV-5EPR — **Behavior neutrality of the self-discovery adoption:** this Decision changes only tool *descriptions* and `instructions`, not behavior — every existing domain tool (`ingest`/`status`/`abort`/`rerun`/`jobs`/`jobs_count`/`merge`/`merges`/`subjects`/`claims`/`page`/`llm_calls`) keeps the same name, accepted inputs, validation, and logical result **fields** across the adoption. Result-**envelope** shape (structured content, typed error codes, `not_found` typing — D61; `ask` citation URLs — D56) is owned by those Decisions and excluded here; the neutrality asserted is that D57 itself reshapes no result.
