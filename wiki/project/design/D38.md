# Decision 38 — Keep the meaning lane consistent across a merge (close the D35 merge gap)

**Decision.** The after-commit vector lifecycle that Decision 35 promised for "the ingest/**merge** worker" must actually run on the merge path. A merge (D26) mutates the meaning lane in two ways its current code ignores: it **rewrites the winner page** (a fresh title/body from the combined claim set, written via `pages.Upsert`) and **deletes the loser subject and its page**. After the merge commit, the worker must do both halves D35 already describes:

1. **Re-embed the winner page** through the existing `embedAndStore` helper — after the commit, never inside the tx, document role — so the winner's stored *and* cached vector reflects its post-merge body, and the resulting `embed-page` `CallRecord` is **attributed to the merge job** (D13 `WithJobID`, already set at `mergeSubjects`).
2. **Purge the loser's vector** from both the `page_embeddings` store and the in-memory `VectorCache`, so no vector survives pointing at a subject that no longer exists.

**The gap this closes.** D35's first path says: "the ingest/merge worker (D04/D26), once its atomic commit succeeds, embeds the pages that job just wrote, **attributed to that job**." But Phase 59 wired `embedAndStore` only into `integrate` (the ingest path) — its post-commit `pagesToEmbed` loop. `mergeSubjects` recompiles and `Upsert`s the winner page and `Delete`s the loser, then commits, and **never touches the vector store or cache**. The `recordingEmbedder` is wired (D37), so the seam is live for ingest — but the merge path never calls through it. This is wiring drift of the same shape D37 closed for D34: a behavior the Decision specifies, built and green for one path, never reached on another.

Live evidence: a merge job (`entity/tsr` → `entity/tactical-studies-rules`) produced exactly one LLM call — a `compile` — and **zero `embed-page` calls**; the winner's stored vector still reflected its pre-merge body, and the loser's vector was left in place.

**Two concrete defects.**

- **Stale winner.** The winner's body is rewritten but not re-embedded after commit, and its cache entry is not refreshed. The D35 catch-up sweep (`content_hash` ≠ fingerprint) will *eventually* re-embed storage, but: (i) not promptly, (ii) not attributed to the merge job as D35 promised, and (iii) the in-memory cache stays stale until a process restart reloads it from storage. Until then, meaning-lane search ranks the winner on a vector for text that no longer exists.
- **Dangling loser (the sharp one).** `page_embeddings` has **no foreign key** to `subjects` (`subject_id TEXT PRIMARY KEY`, no `REFERENCES`, no `ON DELETE CASCADE`), and `EmbeddingStore` has no delete method. Deleting the loser subject therefore **orphans its embedding row**: it persists in storage, and `LoadAll`→`Replace` reloads it into the cache on every boot. The catch-up sweep **cannot** reap it — that sweep iterates *existing* pages to find stale vectors, and the loser no longer has a page, so it is never visited. The orphan is therefore **permanent absent an explicit delete**: `vectorCache.nearest` can return a `Hit` for a deleted `subject_id`, which the resolver cannot render to a live page.

**The seam.**

- **Winner: reuse, don't duplicate.** The winner re-embed uses the existing `embedAndStore` (after commit, never in tx, document role, job-attributed via the `llm.WithJobID(ctx, job.ID)` already set at the top of `mergeSubjects`). `mergeSubjects` captures the winner `Page` it `Upsert`s and, **after** `tx.Commit` succeeds, calls `embedAndStore` on it — exactly the shape `integrate` already uses for its `pagesToEmbed`. `embedAndStore` overwrites the stored vector (`EmbeddingStore.Upsert`) and the live cache entry (the `WithVectorCacheUpdater` hook) in place, so the existing "old vector stays live until replaced" property (D35) holds for the winner too.
- **Loser storage purge: inside the merge tx.** Add `EmbeddingStore.Delete(ctx, subjectID)` (a local `DELETE FROM page_embeddings WHERE subject_id = ?`, no network). `mergeSubjects` runs it on the loser **within the existing merge transaction**, alongside `subjects.Delete`/`pages.DeleteBySubject`, so the embedding row's removal is **atomic with** the structural merge: if the merge commits, the orphan is gone, durably — closing the no-cascade gap without a foreign key or a table-rebuild migration. (Unlike an embed, a `DELETE` is network-free, so keeping it in the writer tx does not violate D35's "no network inside the tx" rule.)
- **Loser cache eviction: after commit, best-effort, self-healing.** Add a remove method to `VectorCache` and wire a composition-root **removal** hook parallel to the existing `WithVectorCacheUpdater` (e.g. `WithVectorCacheRemover(func(subjectID string))`). `mergeSubjects` evicts the loser from the live cache after the commit. A missed eviction self-heals: storage no longer has the row (purged in-tx), so the next `LoadAll`→`Replace` on restart will not reload it. The in-tx storage delete is the durable guarantee; the cache eviction just makes it immediate.

**No new migration.** The fix adds an `EmbeddingStore.Delete` method and a `VectorCache` remove method — code, not schema. `page_embeddings` is unchanged. (Code is mutable; only migrations are immutable.)

**Rejected.**
- **Add a `subjects(id)` foreign key with `ON DELETE CASCADE` to `page_embeddings`.** SQLite cannot add a FK to an existing table without a full table-rebuild migration, and D30 deliberately made `page_embeddings` a standalone side table keyed by `subject_id`. An explicit in-tx `Delete` removes the orphan just as durably with far less machinery and no rebuild. A cascade can be reconsidered later if more subject-delete paths appear.
- **Rely on the D35 catch-up sweep alone.** It re-embeds the *winner* eventually but never reaps the *loser* orphan (no page to iterate), and it drops D35's promised merge-job attribution and leaves the cache stale until restart. The merge path must do its own after-commit lifecycle, exactly as the ingest path does.
- **Re-embed / purge inside the merge transaction (the embed half).** Embedding is a network call to OpenAI; holding the single SQLite writer across it breaks D35's "reads never block on ingest/merge" promise. The winner re-embed stays after commit. Only the network-free storage `DELETE` goes in the tx.
- **Editing D35 / Phase 59 in place.** D35's ids are shipped and frozen; the merge-path behavior is *new* verification, so it gets a new Decision and freshly minted ids. D35 stays unchanged.
- **A merge-path embed recorded by hand.** The recording embedder (D34/D37) is the seam; the winner re-embed flows through `embedAndStore`, which already uses the wired `pageEmbedder`. No second recording surface.

**Verification.**
- R-MRG8-K2WP — After a merge integrates, the winner page is re-embedded through `embedAndStore` **after** the commit, and exactly one `embed-page` `CallRecord` carrying the **merge job's** id lands in the real `LLMCallStore`; the winner's stored + cached vector reflects the post-merge body (its `page_embeddings` `content_hash` equals the post-merge page fingerprint, not the pre-merge one) — mock embedder + recorder against a real temp SQLite.
- R-LP5Q-9XTD — The loser's vector is purged from storage atomically with the merge: after a merge commits, the loser `subject_id` has **no** `page_embeddings` row, `EmbeddingStore.LoadAll` does not return it, and a fresh `LoadAll`→`Replace` (a simulated restart) yields a cache with no entry for the deleted subject — no dangling vector survives a restart. The delete is part of the merge tx: if the merge does not commit, the row is untouched.
- R-EV2H-6RKN — The loser's vector is evicted from the **live** in-memory `VectorCache` immediately after a merge (no restart): a meaning-lane query (`vectorCache.nearest`) cannot return a `Hit` for the merged-away `subject_id`, where before the fix it could.
- R-FM7A-D0SZ — A failed after-commit winner re-embed does not roll back the structural merge: when the embedder errors, the merge job still ends `done` with the alias minted, the loser subject/page/embedding gone, and the winner page and combined claims intact (the commit is not reverted); the stale winner is left for the D35 catch-up sweep — proven with a failing embedder against real SQLite (parallels D35 R-6XNX-FNXO for the merge path).
- R-WS3C-J4QB — Testability seam & wiring guard: `EmbeddingStore` exposes `Delete(ctx, subjectID)` and `VectorCache` a corresponding remove method, and the composition root wires a vector-cache **removal** hook (parallel to `WithVectorCacheUpdater`) that `mergeSubjects` drives — the ids above are covered by tests that exercise the merge worker built the way `buildSpec` constructs it, not a hand-rolled re-wiring; a merge that skips the embed/purge path fails them.
