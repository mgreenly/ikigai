# Decision 15 — Cursor pagination: the contract + the list seams

**Decision.** Every unbounded listing — jobs, subjects, a subject's claims, the LLM-call footprint — returns a **bounded page plus an opaque cursor** for the next, with any filter applied **before** paging. Pagination is **keyset (seek)**: each list orders by a stable unique key and the cursor carries the last row's key, so the next page asks for rows strictly *past* it in the sort direction. O(log n) per page, immune to skip/duplicate under concurrent inserts.

The **jobs** list adds two filter shapes the others lack: its `status` filter is a **set** (any of the five states, OR — so "everything not done" is one query), and the list is ordered **newest-first**. A companion `CountJobs` reports the **filtered total** without returning rows.

## Order keys and filters

Each collection orders by `(sort_column, id)` with the ULID `id` as unique tiebreak; the filter narrows in the same `WHERE` before the keyset predicate. All ascending except jobs (descending, newest-first):

| collection | order key | filters |
|---|---|---|
| jobs | `received_at, id` **DESC** (newest-first) | `status` (a **set** — any of five), time range (`received_at` since/until) |
| subjects | `name, id` (asc) | `type`, name-substring (`norm_name LIKE`) |
| claims (by subject) | `id` (asc) | `subject` (required) |
| llm_calls | `started_at, id` (asc) | `job_id`, `stage`, time range |

## The `internal/page` package (pure; no DB)

Owns the request shape and the cursor codec, table-tested directly:

```go
package page

const (
    DefaultLimit = 50
    MaxLimit     = 200
)

// Params is a cursor-pagination request.
type Params struct {
    Limit  int    // clamped to [1, MaxLimit]; 0 → DefaultLimit
    Cursor string // opaque; "" = first page
}

// EncodeCursor packs ordered key components into one opaque token (base64url over a
// delimited join). DecodeCursor reverses it; ok=false on a malformed token.
func EncodeCursor(parts ...string) string
func DecodeCursor(token string) (parts []string, ok bool)
```

A **malformed cursor is a clean error, never a silent reset to page 1** (silent restart can loop a client forever re-reading page 1). The decode failure propagates as a typed error the MCP layer renders as a tool error (D16).

## The store seams

Stores return `(items, nextCursor, error)` — a plain tuple the MCP layer wraps as `{items, next_cursor}` (D16); no generic result struct.

```go
type JobFilter     struct { Statuses []string; Since, Until time.Time } // Statuses: nil/empty = all states; else match-any (OR)
type LLMCallFilter struct { JobID, Stage string; Since, Until time.Time }

func (s *JobStore)     ListJobs(ctx, f JobFilter, p page.Params)          (jobs []Job, next string, err error)
func (s *JobStore)     CountJobs(ctx, f JobFilter)                        (n int, err error)                       // filtered total; no paging, no rows
func (s *SubjectStore) List(ctx, typ, nameContains string, p page.Params) (subjects []Subject, next string, err error) // reshaped
func (s *ClaimStore)   ListBySubject(ctx, subjectID string, p page.Params) (claims []Claim, next string, err error)     // reshaped
func (s *LLMCallStore) List(ctx, f LLMCallFilter, p page.Params)          (calls []CallRecord, next string, err error)
```

`JobFilter.Statuses`: empty/`nil` matches every state; non-empty builds `status IN (?,?,…)` (match-any). Element validity (each one of five) is enforced at the **MCP boundary** (D16), not re-checked in the store. `CountJobs` shares `JobFilter` verbatim with `ListJobs`, so a count and its list describe the same set.

## Mechanics

Each query appends the keyset predicate in explicit portable form and over-fetches by one to detect a further page. Operator follows sort direction — `>` ascending, `<` descending jobs:

```sql
-- ascending lists (subjects, claims, llm_calls)
... WHERE <filters>
    AND (sort_col > ?1 OR (sort_col = ?1 AND id > ?2))   -- omitted entirely when cursor is ""
  ORDER BY sort_col, id
  LIMIT <clampedLimit> + 1

-- jobs (newest-first): direction flipped, same shape
... WHERE <filters>
    AND (received_at < ?1 OR (received_at = ?1 AND id < ?2))   -- omitted when cursor is ""
  ORDER BY received_at DESC, id DESC
  LIMIT <clampedLimit> + 1
```

If the `+1` row returns, drop it and set `nextCursor = EncodeCursor(lastReturned.sortCol, lastReturned.id)`; else `nextCursor = ""` (last page). The codec is direction-agnostic — it carries `(sort_col, id)` either way; only the operator and `ORDER BY` differ. The single-column claims key encodes just `id`. SQLite serves jobs newest-first by scanning the existing index in reverse — **no new index, no migration for the direction flip**. Backing indexes: existing `jobs_pending_received (status, received_at, id)` (also covers the `status IN (…)` set filter and `COUNT(*)`), `claims_subject (subject_id)`, the D13 `llm_calls_*` indexes; this migration adds `subjects (name, id)`.

`CountJobs` runs the **same `WHERE`** as `ListJobs` under `SELECT COUNT(*) FROM jobs WHERE <filters>` — no keyset predicate, no `ORDER BY`, no `LIMIT` — returning the total without materializing a row.

**How it's tested.** Against a real temp SQLite migrated by the appkit runner:
- Insert N rows, page through at limit `k`: every row appears exactly once, in key order, `nextCursor` empty only on the final page. For jobs, key order is newest-first (descending `received_at, id`).
- A filter narrows before paging: status set (jobs), stage/job_id/time-range (llm_calls), type/name-substring (subjects), subject (claims).
- **Jobs newest-first:** distinct `received_at` jobs list most-recently-received first, ties by descending `id`; holds with or without a status/time filter.
- **Jobs multi-state set:** `ListJobs` with `Statuses` of several states returns exactly the jobs in any (OR), pages cleanly newest-first; empty/`nil` returns all states.
- **`CountJobs`:** returns the count matching a `JobFilter` without rows, equal to what `ListJobs` would page under the same filter; empty filter counts all.
- `Limit` clamps: `0 → DefaultLimit`, `> MaxLimit → MaxLimit`.
- Codec round-trips (`Decode(Encode(parts)) == parts`); a corrupted token decodes `ok=false` and the store list returns a clean error rather than restarting.

**Rejected.**
- **`LIMIT/OFFSET` pagination** — O(n) scan to offset and unstable: an insert/delete before the offset shifts every later page. Keyset is strictly better for an append-heavy footprint. Rejected.
- **A transparent (plaintext) cursor** — invites clients to construct/mutate cursors and couples them to sort columns; opacity keeps the key shape private. Rejected.
- **Silent reset to first page on a bad cursor** — hides a client bug, risks an infinite re-read loop. Fail loudly. Rejected.
- **A generic `Page[T]` result struct** — generic-field friction for no gain; the tuple is simpler and the MCP layer shapes the wire form. Rejected.
- **A separate descending cursor codec for jobs** — codec is already direction-agnostic; only the SQL operator and `ORDER BY` flip. Rejected.
- **A running `total` on every `ListJobs` page** — recomputes `COUNT(*)` per fetch and couples list to count; the product wants a count *without* rows. Rejected.

**Verification.**
- R-17C5-VP2I — Paging a collection of N rows at limit `k` yields every row exactly once in key order across pages, with `nextCursor` non-empty until the final page and empty on it (no skipped or duplicated rows at page boundaries).
- R-18K2-9GT7 — A filter is applied before paging: a status filter (jobs), a `stage`/`job_id`/time-range filter (llm_calls), a `type`/name-substring filter (subjects), and the required `subject` (claims) each restrict the paged set to matching rows only.
- R-XYAZ-V0XE — The jobs list is ordered **newest-first** (most-recently-received first; ties broken by descending `id`), and this ordering holds whether or not a status or time-range filter is applied.
- R-XZIW-8SO3 — `ListJobs` with a `JobFilter.Statuses` set returns exactly the jobs whose status is in that set (match-any / OR), and an empty/`nil` `Statuses` returns jobs of every state.
- R-Y1YP-0C5H — `CountJobs` returns the number of jobs matching its `JobFilter` (state set and/or time range) without returning any row, equal to the count `ListJobs` would page through under the same filter; an empty filter counts all jobs.
- R-19RY-N8JW — `Params.Limit` is clamped: `0` yields `DefaultLimit` rows (when available) and a value above `MaxLimit` yields at most `MaxLimit`.
- R-1C7R-ES1A — `EncodeCursor`/`DecodeCursor` round-trip ordered key parts losslessly, and `DecodeCursor` reports `ok=false` for a malformed token.
- R-1DFN-SJRZ — A list call given an undecodable cursor returns a clean error and does not silently restart from the first page.
