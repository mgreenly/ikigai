# Phase 62 — Keep the meaning lane consistent across a merge

*Realizes design Decision 38 (close the D35 merge gap). Depends on Phase 53 (D30 — `EmbeddingStore`), Phase 56 (D32 — `vectorCache`), Phase 59 (D35 — `embedAndStore` + the after-commit lifecycle), and Phase 61 (D37 — the wired recording embedder). Touches the D26 merge path (`mergeSubjects`, Phases 43/48). Adds no migration.*

D35 named the **merge** worker as an after-commit embed path, but Phase 59 wired `embedAndStore` only into `integrate` (ingest). `mergeSubjects` recompiles and `Upsert`s the winner page and `Delete`s the loser subject/page, then commits, and **never touches the vector store or cache** — so after a merge the winner's vector is stale and the loser's vector is orphaned (a real merge produced a `compile` call and **zero** `embed-page` calls). This phase makes the merge path run the same vector lifecycle the ingest path already does, in two halves.

- **Winner re-embed (reuse `embedAndStore`).** `mergeSubjects` captures the winner `Page` it `Upsert`s and, **after** `tx.Commit` succeeds, calls the existing `Service.embedAndStore` on it — running under the `llm.WithJobID(ctx, job.ID)` already set at the top of `mergeSubjects`, exactly as `integrate`'s post-commit `pagesToEmbed` loop does. `embedAndStore` overwrites the stored vector (`EmbeddingStore.Upsert`) and the live cache entry in place, so the winner's vector reflects its post-merge body and the resulting `embed-page` record carries the merge job id. Never inside the tx (it is a network call).
- **Loser storage purge (in the merge tx).** Add `EmbeddingStore.Delete(ctx, subjectID) error` — a network-free `DELETE FROM page_embeddings WHERE subject_id = ?`. `mergeSubjects` runs it on the loser **inside the existing merge transaction**, alongside `subjects.Delete`/`pages.DeleteBySubject`, so the embedding row is removed atomically with the structural merge — closing the no-FK/no-cascade orphan gap durably, with no foreign key and no table rebuild.
- **Loser cache eviction (after commit).** Add a remove method to `vectorCache` and wire a composition-root removal hook parallel to `WithVectorCacheUpdater` (e.g. `WithVectorCacheRemover(func(subjectID string))`); `mergeSubjects` evicts the loser from the live cache after the commit. A missed eviction self-heals — the row is already gone from storage, so the next `LoadAll`→`Replace` will not reload it.
- **Failure tolerance.** If the after-commit winner re-embed errors, the merge job still ends `done` with the alias minted, the loser fully removed, and the winner page/claims intact (the commit is not rolled back); the stale winner is left for the D35 catch-up sweep.

The composition root wires the new vector-cache removal hook onto the `Service` next to the existing updater hook; the storage delete needs no new wiring (the merge tx already constructs stores over `tx`).

**Done when:** the suite is green (per design *Conventions*) and these ids are covered by clearly-named tests against a **real** temp SQLite with mock embedder/recorder (no live call), driving the merge worker as `buildSpec` constructs it:

- **R-MRG8-K2WP** — after a merge integrates, the winner page is re-embedded through `embedAndStore` after the commit, exactly one `embed-page` `CallRecord` carrying the **merge job's** id lands in the real `LLMCallStore`, and the winner's stored + cached vector reflects the post-merge body (its `page_embeddings` `content_hash` equals the post-merge page fingerprint, not the pre-merge one).
- **R-LP5Q-9XTD** — the loser's vector is purged atomically with the merge: after commit the loser `subject_id` has no `page_embeddings` row, `EmbeddingStore.LoadAll` omits it, and a fresh `LoadAll`→`Replace` (simulated restart) yields a cache with no entry for the deleted subject — no dangling vector survives a restart; if the merge does not commit, the row is untouched.
- **R-EV2H-6RKN** — the loser's vector is evicted from the **live** cache immediately after a merge (no restart): `vectorCache.nearest` cannot return a `Hit` for the merged-away `subject_id`, where before the fix it could.
- **R-FM7A-D0SZ** — a failed after-commit winner re-embed does not roll back the merge: with a failing embedder, the job still ends `done` with the alias minted, the loser subject/page/embedding gone, and the winner page and combined claims intact; the stale winner is left for catch-up.
- **R-WS3C-J4QB** — seam & wiring guard: `EmbeddingStore.Delete` and the `vectorCache` remove method exist, the composition root wires a vector-cache removal hook (parallel to `WithVectorCacheUpdater`) that `mergeSubjects` drives, and the ids above are covered through the merge worker built the way `buildSpec` constructs it — a merge that skips the embed/purge path fails them.
