# Phase 54 — The embedding call site: configured (`EMBED_MODEL`/`EMBED_DIMS`) and recorded like every other AI call

*Realizes design Decision 34 (the embedding call site: configured and recorded like every other AI call). Depends on Decision 19 (per-call-site config — Phase 35, the `resolveCallSite`/`ManifestExtras` machinery and `Config.CallSites`) and Decision 13 (the `llm_calls` recorder seam — Phase 51's widened stage CHECK is the migration template). Consumed by the meaning lane (Phase 56) and the background lifecycle (Phase 59).*

Embedding becomes a first-class, configured, on-the-record call: one model serving both the page side and the query side, with every embed written to `llm_calls`.

- **Config (`internal/wiki/config.go`).** Add an `EmbedSite{Model string, Dims int, Provider agentkit.EmbeddingProvider}` to the resolved `Config` alongside the D19 call sites, with fail-loud parsing of its two knobs: `EMBED_MODEL` (unset → `text-embedding-3-small`), `EMBED_DIMS` (unset → `512`; `0`, negative, or non-numeric → a **startup error**, never a silent fallback), and a `ManifestExtras` entry like the other knobs. The OpenAI key is read once at the composition root via `getenv` (same secret rule as the Anthropic key) and the provider injected; it never enters domain code.
- **A new table-rebuild migration** (generated with `bin/new-migration wiki widen_llm_calls_stage_embed`; **never** hand-name the version) that widens the `llm_calls.stage` CHECK to add `embed-page` and `embed-query` to the existing closed set. SQLite cannot `ALTER` a CHECK, so it is the standard rebuild — create the new table with the widened CHECK → `INSERT … SELECT *` → drop old → rename → re-create both indexes — exactly as Phase 51 did; the prior migrations stay frozen on disk.
- **A recording embedder (`internal/llm`).** `recordingEmbedder` wraps the `agentkit.Embedder` and, on each `Embed(ctx, texts, role)` round-trip, builds one `CallRecord` and hands it to the same `Recorder`/`LLMCallStore.Record` (D13) used by extract/compile/ask. The record carries its `stage` (`embed-page` for the document/ingest side this phase owns, `embed-query` for the query side D32 uses), the model, the dimensions in `params` (e.g. `{"dims":512}`), the provider's token usage, and any error; the page side reads the job id off the context (`WithJobID`) so an `embed-page` record ties to its ingest job, while a query-side call records an empty job id.

The composition root builds the recording embedder from the configured `EmbedSite` and the existing recorder, ready for Phases 56/59 to consume. The page side asks the embedder to encode in **document** mode (the query/document asymmetry; D32 owns the query side).

**Done when:** the suite is green (per design *Conventions*, including `bin/check-migrations wiki`) and these ids are covered by clearly-named tests — config parsing pure, the stage CHECK against a **real** temp SQLite, the recorder behavior via a mock recorder:

- **R-Z932-H2RA** — config resolves the embed knobs: unset → `text-embedding-3-small` @ 512; a set `EMBED_DIMS` overrides; a non-positive or non-numeric `EMBED_DIMS` returns a **startup error** (no silent default).
- **R-ZAAY-UUHZ** — against real migrated SQLite, the rebuild migration lets `Record` accept stages `embed-page` and `embed-query`, still rejects a non-canonical stage via the CHECK, and the prior migrations are unmodified on disk.
- **R-ZBIV-8M8O** — the recording embedder emits exactly one `CallRecord` per `Embed` round-trip carrying stage, model, and `dims` in `params` with the provider's token usage; a page-side call under `WithJobID` records that job id, a query-side call records an empty job id.
- **R-ZCQR-MDZD** — an embedder transport/stream error produces a record with non-empty `Err` (the failed embed is on the record, not dropped).
- **R-ZDYO-05Q2** — **live check (operator-run, real OpenAI embeddings API):** an embed call with the configured model and `Dims=512` returns a length-512, L2-normalized vector (norm ≈ 1.0) — a property no mock can establish; gated so the offline suite stays green with no `OPENAI_API_KEY`.
