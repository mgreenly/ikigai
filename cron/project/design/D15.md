# Decision 15 — Structured MCP adoption

**Problem.** cron's five domain tools build their success results with
`appkitmcp.JSONResult(v)` (a Go value marshalled to JSON and shipped as a single
text block) and their failures with a hand-rolled `errorEnvelope`/`toolErr` pair
that marshals `{error:{code,message,field?}}` to text and hands the string to the
old one-argument `appkitmcp.ErrorResult(string)`. The suite has since settled its
single verb surface (`docs/structured-mcp-design.md`): a domain verb answers
agents and machines in one response — `structuredContent` for machines, mirrored
text for agents — and every tool error carries a code from a **closed
vocabulary** as a structured field so machine callers branch on a code, never on
message text. appkit (phases 12–14, done) realized that contract centrally:
`JSONResult` is **deleted**; `StructuredResult(v) (map[string]any, error)` emits
both renderings from one typed value; `ErrorResult(code ErrorCode, msg string)`
takes a typed code (`ErrValidation`, `ErrNotFound`, `ErrConflict`, `ErrTooLarge`,
`ErrSourceUnavailable`, `ErrInternal`); `Tool.OutputSchema` is surfaced by
`tools/list` when non-nil. cron does not compile against that appkit and does not
yet declare output schemas. This Decision adopts the contract for cron's domain
surface. It cites `docs/structured-mcp-design.md` and does not restate it.

This Decision **reshapes no success result**: the JSON object each tool already
emits (`renderEntry`'s `{name,expr,created_at,updated_at,last_slot}`, `list`'s
`{items:[…]}`, `delete`'s `{ok:true}`) is unchanged; it now travels as
`structuredContent` with a mirrored text block, and a hand-authored
`outputSchema` formalizes exactly that existing shape. The transport-extraction
(D10) and event-routing (D14) decisions stand; this is the result-shape layer on
top of them, and cron's surface remains the same **seven** tools.

**Decision.**

1. **`JSONResult` → `StructuredResult` at every domain success site.** The five
   handlers in `internal/mcp/tools.go` (`toolCreate`, `toolList`, `toolGet`,
   `toolUpdate`, `toolDelete`) return `appkitmcp.StructuredResult(v)` in place of
   `appkitmcp.JSONResult(v)`. Each handler's signature is already
   `(map[string]any, error)`, so the helper's **new error return is propagated
   honestly, never swallowed**: a marshal failure returns `(nil, err)` from the
   handler and surfaces as a JSON-RPC internal error, exactly as any other
   handler error does through `handleToolCall`. Illustratively:

   ```go
   func (h *toolHandlers) toolCreate(ctx context.Context, raw json.RawMessage) (map[string]any, error) {
       // … validate, store.Create …
       return appkitmcp.StructuredResult(renderEntry(*e)) // (map, err) flows straight out
   }
   ```

2. **Typed error codes replace the text envelope.** `errorEnvelope` (the
   `{error:{code,message,field?}}` builder) and its JSON-text marshalling are
   **deleted**; `toolErr(err)` maps the domain sentinel to one closed-vocabulary
   code and calls `appkitmcp.ErrorResult(code, message)`, which emits `isError`
   plus `structuredContent:{code,message}`. The mapping, named exhaustively:

   | domain error | `ErrorCode` | message |
   |---|---|---|
   | `*parseError` (expr fails `cron.Parse`) | `ErrValidation` | the parser message, which names the bad field (e.g. `hour`) |
   | `crontab.ErrExists` (name already exists on create) | `ErrConflict` | `err.Error()` |
   | `crontab.ErrNotFound` (get/update/delete unknown name) | `ErrNotFound` | `err.Error()` |
   | `crontab.ErrInvalid` (name violates the CHECK) | `ErrValidation` | `err.Error()` |
   | anything else | `ErrInternal` | `"internal error"` |

   This is the intended **vocabulary correction**: the old envelope's `duplicate`
   becomes the closed-vocabulary `conflict`, and both the bad-expr and
   bad-name paths land on `validation`. The `field` key does **not** survive into
   the structured error — the closed shape is `{code,message}` only — so the
   discriminating field name is carried by the human `message` (the parser
   already names the field; `ErrInvalid`'s message names the name constraint),
   not a structured key. `ErrInternal` is the untriggerable default (it needs a
   store fault) and is not exercised in isolation.

   ```go
   func toolErr(err error) map[string]any {
       switch {
       case errors.As(err, new(*parseError)):        return appkitmcp.ErrorResult(appkitmcp.ErrValidation, err.Error())
       case errors.Is(err, crontab.ErrExists):       return appkitmcp.ErrorResult(appkitmcp.ErrConflict, err.Error())
       case errors.Is(err, crontab.ErrNotFound):     return appkitmcp.ErrorResult(appkitmcp.ErrNotFound, err.Error())
       case errors.Is(err, crontab.ErrInvalid):      return appkitmcp.ErrorResult(appkitmcp.ErrValidation, err.Error())
       default:                                       return appkitmcp.ErrorResult(appkitmcp.ErrInternal, "internal error")
       }
   }
   ```

3. **Each domain tool declares an `outputSchema`** — a hand-authored
   `map[string]any` JSON-schema literal mirroring the JSON the tool already
   emits verbatim, following the `objectSchema` house style in `appkit/mcp/mcp.go`
   (cron writes its own literals through a small private helper). The schemas:

   - **`create`, `get`, `update`** share the entry schema — properties
     `name`, `expr`, `created_at`, `updated_at` are `{"type":"string"}` and
     `last_slot` is `{"type":["string","null"]}` (RFC3339 UTC or null before the
     schedule first fires); all five are `required` (`last_slot` is always
     present, possibly null). A private `entrySchema()` helper returns it so the
     three sites cannot drift.
   - **`list`** — `{"items": {"type":"array","items": entrySchema()}}`,
     `items` required.
   - **`delete`** — `{"ok": {"type":"boolean"}}`, `ok` required.

   The `desc` constructor (or a variant) sets `Tool.OutputSchema` alongside
   `InputSchema`; `tools/list` then surfaces the schema per tool. The chassis
   `health`/`reflection` schemas are appkit's and are untouched here.

4. **Prose exceptions: cron has none among its own tools.** cron declares **no**
   documentation tool (`guide`/`describe`) and **no** raw-content read; all five
   of its domain tools (`create`, `list`, `get`, `update`, `delete`) are
   machine-shaped and therefore each declare an `outputSchema` and return
   `structuredContent`. The chassis `health`/`reflection` are structured centrally
   by appkit. No cron tool opts out of a schema.

5. **No loopback guard change.** cron copies no `X-Owner-Email OR
   X-Forwarded-Proto` guard predicate — its only loopback-private mount is the
   chassis `/feed`, already wrapped by appkit's shared guard. This Decision
   touches no guard site.

**Rejected.**
- **Swallowing `StructuredResult`'s error** (e.g. `res, _ := StructuredResult(v)`)
  — hides a real marshalling failure behind an empty/partial result; the handler
  signature already carries the error out, so propagation is free and honest.
- **Keeping `field` in the structured error** — the suite's closed error shape is
  `{code,message}`; a bespoke `field` key would fork the vocabulary the machine
  contract exists to standardize. Field detail stays in the human message.
- **Generating output schemas from the Go types at runtime** — appkit deliberately
  does not validate or derive schemas (a stated non-goal); the schema is a
  hand-authored, versioned, testable literal mirroring the emitted JSON.
- **Reshaping any result to "look more structured"** — the refactor formalizes
  the existing JSON verbatim; changing a field would be a separate product/design
  change, not this adoption.

**Verification** (through the composition-root seam — the assembled `appkit/mcp`
handler `NewHandler` builds over a crontab `Store` on a temp SQLite DB, driven by
`tools/list` and `tools/call` via `httptest`; the real handler and real store are
the substrate).

- R-6TVE-C4AC — Every domain tool's **success** result carries a
  `structuredContent` object equal to the JSON the tool emits **and** a mirrored
  `content` text block whose text is that same JSON (table-driven over `create`,
  `list`, `get`, `update`, `delete`): e.g. a `create` of `nightly`/`0 3 * * *`
  yields `structuredContent.name == "nightly"`, `structuredContent.last_slot ==
  null`, and `content[0].text` unmarshals to the identical object; `delete`
  yields `structuredContent.ok == true`; `list` yields `structuredContent.items`
  as an array. A tool still shipping only a text block (no `structuredContent`)
  fails.
- R-6V3A-PW11 — `tools/list` returns a non-nil `outputSchema` for **each** of the
  five domain tools, and each schema matches the emitted shape (the entry schema
  for `create`/`get`/`update` requires `name,expr,created_at,updated_at,last_slot`
  with `last_slot` typed `["string","null"]`; `list`'s requires `items:array`;
  `delete`'s requires `ok:boolean`). A tool with a missing or empty `outputSchema`
  fails.
- R-6WB7-3NRQ — A `create` (or `update`) with a structurally invalid `expr`
  returns a tool result with `isError:true` and `structuredContent.code ==
  "validation"` (the `ErrValidation` code, not a bare/absent code), and the
  message names the offending field. A result carrying no structured code, or a
  code other than `validation`, fails.
- R-6XJ3-HFIF — A `create` whose `name` already exists returns `isError:true`
  with `structuredContent.code == "conflict"` — specifically the closed-vocabulary
  `ErrConflict`, **not** the retired `"duplicate"` string. A result whose code is
  `"duplicate"` (or anything but `conflict`) fails.
- R-6YQZ-V794 — A `get`, `update`, or `delete` naming a schedule that does not
  exist returns `isError:true` with `structuredContent.code == "not_found"` (the
  `ErrNotFound` code). A result missing that code, or carrying `internal`, fails.
- R-6ZYW-8YZT — A `create` with a malformed `name` (rejected by the crontab CHECK,
  surfacing `crontab.ErrInvalid`) returns `isError:true` with
  `structuredContent.code == "validation"` (the `ErrInvalid → ErrValidation`
  branch), distinct from the not_found/conflict codes. A result carrying any
  other code fails.
- R-716S-MQQI — A source-scan guard (in the idiom of cron's existing
  `TestNoHardcodedLoopbackPortLiteralInSource`, walking the module tree and
  excluding `*_test.go`) asserts the token `JSONResult` appears **nowhere** in
  cron's `internal/` and `cmd/` Go source. A lingering `appkitmcp.JSONResult(...)`
  call fails it.
