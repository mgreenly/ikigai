# Decision 9 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** cron's landing page and Carbon assets are embedded in the binary
(`//go:embed` in `cron/internal/web/`: `landing.html` + `static/tokens.css` +
the four woff2 fonts), like every service's originally were. The suite is
reverting that: web assets are release artifacts that belong on disk — visible
and diffable in the versioned release tree, editable in dev without a rebuild,
served by one chassis mechanism instead of twelve drifting copies. crm proved
the conversion; notify followed; cron is converted here to the same shape. This
Decision is the service-side half of appkit's chassis design (its www-root
resolution, `appkit/web`, and `Spec.WWW`/`Router.WWW()` — consumed here as a
fixed external contract).

**Decision.** cron serves its web surface from the on-disk www root.

1. **The files move (same bytes).** `cron/internal/web/landing.html` →
   `cron/share/www/landing.html`; `cron/internal/web/static/` →
   `cron/share/www/static/` (`tokens.css` + the four woff2 fonts). `bin/ship`
   already bundles `cron/share/` into the release `share/<version>` tier; on the
   box the chassis resolves the root to `share/current/www`, in dev to
   `./share/www` or the `CRON_WWW_PATH` override.
2. **The Spec opts in.** `cronSpec()` in `cmd/cron/main.go` (the composition root
   D08 established) sets `WWW: true`. The chassis loads the site at serve (failing
   startup loudly if the root is missing), auto-mounts `GET /static/`, and exposes
   the loaded site as `rt.WWW()`.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` uses a small composition-root handler over
   `rt.WWW().Render(w, "landing.html", …)` — the same
   `landingHandler(site *web.Site, service, version string) http.Handler` shape
   notify uses (D1 carries the exact form). It stays **ungated in-process** (D2 —
   nginx's `/_session-authn` is the gate). The service-side `GET /static/` mount
   is **deleted** from `Handlers` — the chassis owns it.
4. **`cron/internal/web/` is deleted** — handler (`web.go`), embed shim
   (`embed.go`), template, assets, and its tests. The landing/asset/mux tests move
   to the composition-root package (`cmd/cron`), loading the repo-real
   `share/www` tree via `appkit/web.Load` (a relative path from the package dir),
   so they exercise the exact files that ship. The nginx fragment tests
   (`nginx_test.go`) relocate to `cmd/cron` unchanged in their assertions (D4's
   fragment is untouched by this conversion).
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_cron` exports
   `CRON_WWW_PATH="$repo/cron/share/www"` beside its existing `CRON_DB_PATH` /
   `CRON_GENERATION_PATH` exports, so a repo-root-launched dev binary finds the
   source tree. This **one line crosses the `cron/` boundary** — the sole
   sanctioned exception (per the pilot's standing convention) — and is verified by
   the live `bin/start` smoke, **not** the Go suite.
6. **Nothing else moves.** The nginx fragment (D4/D7), the landing markup and
   tokens (D1/D3/D6/D7), the MCP surface (D10), the migrations (`internal/db`), and
   the tick worker are untouched by this Decision. The embedded migrations stay
   embedded — they are not web assets (D12).

**Retained ids re-proven on the new substrate.** The page/asset behaviors keep
their existing ids, now proven by loading the shipped `cron/share/www` tree
through `appkit/web` rather than an embedded `embed.FS`: D1's `R-LAND-*`, D2's
`R-ROUT-*`, D3's `R-ASST-*`, D6's `R-HOME-2K4P`, and D7's `R-21DE-LOX3`/
`R-22LA-ZGNS`/`R-23T7-D8EH`/`R-2513-R056`/`R-2690-4RVV`. Those Decisions are
edited in place to describe the on-disk mechanism; their ids are unchanged and
their assertions carry over verbatim except for swapping the handler/asset source
from the deleted `web.LandingHandler`/`web.StaticHandler` to the chassis-loaded
`appkit/web.Site`.

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work exists to complete;
  embedded assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's nginx-served user
  content, backed up with `state/`; cron's UI assets are release artifacts and
  live in the shipped `share` tier (appkit's www-root Decision records the full
  argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition (the same call
  D08 makes for the Spec).
- **Serving assets from both embed and disk during a transition** — two sources of
  truth; the conversion is atomic within cron.

**Verification** (over the repo-real `cron/share/www` tree, resolved relative to
the `cmd/cron` test package; the retained D1/D3/D6/D7 ids continue to pin the
page/asset content through the new substrate).

- R-LPMQ-FKBR — an `appkit/web` Site loaded from `cron/share/www` (the actual
  shipped tree) renders `landing.html` with an injected service and version
  visible in the output — proving the shipped template, not an embedded copy, is
  the page source (`200`, `text/html; charset=utf-8`, body contains the injected
  service name and version).
- R-LQUM-TC2G — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`, `wOF2` payload)
  with **no** cron-side static handler registered — proving the assets ship on
  disk and the chassis serves them.
