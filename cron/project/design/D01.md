# Decision 1 — The landing handler and its v1 content (service name + version)

**Decision.** cron owns its landing page *content*. The page is a Carbon-styled,
centered card showing the **service name** (display type) and the **running
version** (mono label) and nothing else. The template lives **on disk** at
`cron/share/www/landing.html` (shipped in the release `share` tier, per D9 and
appkit's chassis design); the handler is a small `http.Handler` at the
composition root that renders it through the chassis-loaded site:

```go
// cmd/cron — inside cronSpec()'s Handlers hook; rt.WWW() is the appkit/web Site
// loaded from the www root (Spec.WWW).
func landingHandler(site *web.Site, service, version string) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if r.URL.Path != "/" {
            http.NotFound(w, r)
            return
        }
        _ = site.Render(w, "landing.html",
            struct{ Service, Version string }{service, version})
    })
}
```

There is **no shared landing handler** across apps; the content and the render
helper live under `cron/`. The two content values are passed in as plain strings —
the composition root supplies them from the chassis (`rt.Service()` → `"cron"`,
`rt.Version()` → the build version, D2). The page renders from cron's shipped
`share/www/landing.html`; only the rendering *mechanism* (the on-disk site loaded
by `appkit/web.Load` over the www root) is chassis-owned.

**Rendering.**

- The HTML is the on-disk `cron/share/www/landing.html`, loaded once at serve by
  the chassis (`appkit/web.Load`, exposed as `rt.WWW()`), rendered per request via
  `rt.WWW().Render(w, "landing.html", data)`. The service name and version are the
  only dynamic substitutions; everything else is static markup.
- `appkit/web` renders with `html/template`; the two values are trusted display
  strings.
- The response sets `Content-Type: text/html; charset=utf-8` and returns
  `200 OK` for the page (the chassis site writer sets the content type).
- The markup links the app's **own** `tokens.css` from `share/www/static/` and
  applies the Carbon type scale (Space Grotesk display for the name, IBM Plex Mono
  for the version label) and neutral palette. The Carbon asset wiring is D3; this
  Decision owns that the page is *rendered with name + version* and is a single
  centered card.

**Why a constructor over strings, not over the chassis.** Passing `service` and
`version` as strings (rather than reading them inside the render helper) keeps the
handler pure over its two inputs and trivially table-testable with fixed values.
The composition root is the only place that reaches into `rt` for the two strings
and the loaded site (`rt.WWW()`).

**Rejected.**
- **A shared cross-app landing handler** (e.g. in appkit) — each app's page will
  diverge to serve that app's purpose, so each app owns its page content. v1's
  uniformity is a content coincidence, not a shared component. (The *mechanism* —
  `appkit/web` + `Spec.WWW` — is shared; the content is not.)
- **Keeping the page embedded in the binary (`//go:embed`)** — the suite's
  original approach, deliberately reverted (D9): web assets are release artifacts
  that belong on disk, visible on the box and editable in dev without a rebuild.
- **Rendering anything beyond name + version** in v1 (owner email, crontab counts,
  next-tick time, emitted-event types) — out of product scope; v1 is exactly name
  + version.
- **Pulling the values from env or a new config knob** — the chassis already
  exposes the authoritative service name and build version; a second source could
  drift from what `version`/`/health` report.
- **Reading the identity header (`X-Owner-Email`) to personalize the page** — the
  gate is coarse (any logged-in user); v1 shows no per-user content, so the
  handler ignores identity entirely.

**Verification** (over an `appkit/web` Site loaded from the repo-real
`cron/share/www` tree, resolved relative to the `cmd/cron` test package and driven
with httptest — D9's `R-LPMQ-FKBR` pins the shipped-tree render).
- R-LAND-3C9K — the composition-root landing handler over the loaded site serving
  a `GET /` returns HTTP `200`.
- R-LAND-5E2L — the rendered body contains the service name passed in (the page
  shows `"cron"`).
- R-LAND-7G4M — the rendered body contains the version string passed in (e.g. a
  test version like `"9.9.9-test"` appears verbatim), proving the page reflects
  the running version rather than a hardcoded constant.
- R-LAND-9J6N — the landing response sets `Content-Type: text/html; charset=utf-8`.
