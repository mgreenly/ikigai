# Decision 10 — MCP surface over `appkit/mcp`: `internal/mcp` becomes the tool table

**Problem.** cron's `internal/mcp` carries the copy-pasted JSON-RPC transport
(`ServeHTTP`, the `initialize`/`tools/list`/`tools/call` dispatch, the
`jsonRPCRequest` type, the JSON-RPC error writer, the result-envelope helpers,
the local `Identity` type) plus its own `health` and `reflection` tool
implementations — the same near-identical plumbing every service duplicated. The
chassis now owns all of it (appkit's `appkit/mcp` transport and its
chassis-registered standard `health`/`reflection` tools, consumed here as a fixed
external contract). What is genuinely cron's is the **crontab CRUD** surface —
`create`, `list`, `get`, `update`, `delete` — with its expr validation, its
typed error envelope, and its `renderEntry` shaping.

**Decision.** `internal/mcp` is reshaped from a transport into a **declaration**:

```go
// internal/mcp — what remains.
const Instructions = /* cron's MCP instructions string, the initialize
   "instructions": "Named UTC cron schedules that publish a cron:tick/<name>
   event on a timer. Create a schedule, then wire consumers to its event."
   (event addressing per D14) */

// Tools declares cron's five domain tools over the crontab store: create,
// list, get, update, delete (descriptors + schemas + handlers unchanged in wire
// content — expr validation naming the bad field, the closed error vocabulary,
// renderEntry's {name,expr,created_at,updated_at,last_slot} shape). health and
// reflection are NOT declared — the chassis registers them (declaring them
// locally fails New's reserved-name check).
func Tools(store *crontab.Store) []mcp.Tool

// NewHandler assembles the /mcp http.Handler at the composition root:
// appkit/mcp.New over Instructions + Tools(store) + the Router-threaded
// Service/Version/Health/Events/Publishes/Subscriptions.
func NewHandler(store *crontab.Store, rt *appkit.Router) (http.Handler, error)
```

- **Deleted:** the JSON-RPC plumbing (`jsonRPCRequest`, the `ServeHTTP` dispatch,
  `writeJSONRPCResult`/`writeJSONRPCError`/`idOrNull`, the `handleToolCall`
  dispatcher, the local `Identity` type), the local `toolHealth` and
  `toolReflection` implementations together with their helpers
  (`renderSubscriptions`, `reflectionUnknownTypeError`), and the local
  result-envelope helpers (`toolResultText`/`toolResultErr`/`toolResultJSON`,
  replaced by the chassis `mcp.TextResult`/`mcp.ErrorResult`/`mcp.JSONResult`).
- **Kept:** the descriptor/schema helpers (`desc`/`obj`/`typ`/`descTyp`, now
  producing `mcp.Tool` values), the five domain tool descriptions and schemas
  verbatim, `validateExpr`/`parseError` (the expr boundary check that names the
  bad field), the `errorEnvelope`/`toolErr` typed-error mapping
  (`validation`/`duplicate`/`not_found`/`internal` with `field`), and
  `renderEntry`. The wire bytes agents see are unchanged.
- **The `reflection` surface stays live and dynamic.** cron's reflection
  `publishes` half is the live one-family `tick` registry read from the crontab
  (D14) — its correct chassis surface is `Spec.Publishes` (the LIVE provider),
  which `cronSpec()` already declares (`event.Publishes(store)`). `NewHandler`
  threads it through as `mcp.Options.Publishes: rt.Publishes()`, so the
  chassis-registered reflection tool renders the tick family with the live
  schedule enumeration in its description (preferred over the empty static
  `Events`). `Events` stays empty — cron runs no Append-time family guard on its
  dynamic subjects. This preserves the live-reflection behavior the existing
  `TestCreateThenListGet` pins (create `nightly` ⇒ reflection `publishes`
  names `nightly`), in the family shape D14 specifies.
- **The wire surface is unchanged:** **seven** tools total — `create`, `list`,
  `get`, `update`, `delete` declared by cron **plus** `health` and `reflection`
  chassis-registered — same descriptions and schemas, same instructions, same
  error envelopes. The existing crontab behavioral tests
  (`internal/mcp/tools_test.go`) keep their assertions and swap only the handler
  constructor: the test harness builds the assembled `appkit/mcp` handler over a
  real `server.Router` (the notify `server.New(server.Options{…, Publishes:
  event.Publishes(store), Register: NewHandler})` pattern) so `rt.Publishes()`
  returns the live crontab provider.

**Rejected.**
- **Keeping the local transport** — the status quo; a protocol fix would still be
  a many-service edit.
- **Declaring health/reflection in cron's table** — the chassis reserves the
  names precisely so services cannot fork the standard tools; their inputs
  (Health, Events/Publishes, Subscriptions) are chassis-held already. A table that
  still declares its own `health`/`reflection` fails `New`'s reserved-name check.
- **Passing the live schedule enumeration through `Events` instead of
  `Publishes`.** `Events` is a static compile-time registry that also gates
  Append-time validation; cron's reflected surface is computed at runtime from
  crontab rows, so it must flow through the LIVE `Publishes` provider — exactly
  the field the chassis prefers for reflection. Using `Events` would both be
  wrong (static) and impose an unwanted per-emit guard.
- **Dissolving `internal/mcp` into `cmd/cron`** — the crontab tool schemas, expr
  validation, and typed error mapping with their test harness are a real package,
  unlike the post-D9 web remnant.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a crontab store on a temp DB).

- R-LS2J-73T5 — `tools/list` through the assembled handler returns **exactly
  seven** tools: `create`, `list`, `get`, `update`, `delete` declared by cron
  **plus** `health` and `reflection` supplied by the chassis — proving the
  table/chassis partition (a table that still declares its own health/reflection
  fails `New`'s reserved-name check; a surface missing them fails the count). The
  pre-existing crontab behavioral tests (bad-expr rejected naming `hour`,
  create→list→get round-trip, reflection `publishes` shows the live schedule
  enumeration per D14's tick family)
  continue to pass through the assembled handler with no assertion changes.
