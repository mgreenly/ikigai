# Decision 2 — Route wiring: `GET /{$}` mounted ungated through `Spec.Handlers`

**Decision.** The landing handler is mounted at the service's **exact root path**
`/` through cron's `appkit.Spec.Handlers` hook (declared inline in
`cmd/cron/main.go`'s `cronSpec()`, per D08), beside the `POST /mcp` mount, using
the Go 1.22+ exact-match pattern `GET /{$}`:

```go
Handlers: func(rt *appkit.Router) error {
    conn := rt.DB()
    if conn == nil {
        return fmt.Errorf("cron: no DB handle on router")
    }
    store = crontab.NewStore(conn)

    // Landing page — human web UI, rendered from the on-disk share/www tree
    // through the chassis-loaded site (D1/D9). Ungated IN-PROCESS: nginx is the
    // gate (auth_request /_session-authn, see D4). `{$}` matches ONLY the exact
    // root path, so it never shadows /mcp, /health, /feed, or the PRM well-known.
    // The chassis auto-mounts GET /static/ (Spec.WWW) — no service-side static
    // mount here.
    rt.Handle("GET /{$}", landingHandler(rt.WWW(), rt.Service(), rt.Version()))

    handler, err := mcp.NewHandler(store, rt)
    if err != nil {
        return err
    }
    rt.Handle("POST /mcp", rt.RequireIdentity(handler))
    return nil
},
```

The MCP mount is the assembled `appkit/mcp` handler (D10); the landing mount is
the composition-root `landingHandler` over `rt.WWW()` (D1). Both stay on the same
mux; the ungated/gated partition below is unchanged by the WWW and MCP
conversions.

**Why ungated in-process.** cron runs no token logic; nginx is the sole trust
boundary (Conventions). The landing route is mounted **without** an in-process
identity wrapper — exactly as `POST /mcp` delegates its bearer gate to nginx's
`auth_request /_authn`, the landing route delegates its cookie gate to nginx's
`auth_request /_session-authn` (D4). Wrapping it in `rt.RequireIdentity` would be
wrong: that expects the bearer-injected `X-Owner-Email`/`X-Client-Id` headers,
which a browser session request does not necessarily carry the same way, and the
page is meant to be coarse, not per-resource.

**Why `{$}` and not `/`.** In the Go 1.22+ `http.ServeMux` pattern language, a
trailing `/` is a **subtree** pattern that matches every path under it, while
`{$}` matches **only** the exact path. cron's internal routes are `/`, `/mcp`,
`/health`, `/feed`, and `/.well-known/oauth-protected-resource` (nginx has
already stripped the `/srv/cron/` mount prefix). Mounting the landing page at
`GET /{$}` guarantees it answers the bare root **and nothing else** — it cannot
shadow `/mcp` or any other route. A subtree `/` would swallow unmatched paths and
is explicitly rejected.

**Composition-root growth, not a rewrite.** `cmd/cron/main.go` already carries the
`Handlers` hook (it builds the crontab `Store` and mounts `POST /mcp`); this
Decision adds **one line** to that existing hook. The `POST /mcp` mount, the
`Publishes` provider, and the tick `Producer`/`Workers` wiring are untouched. That
is the sanctioned incremental growth of the one shared wiring file, not a rewrite
of cron's domain.

**Rejected.**
- **A subtree `GET /`** — swallows every unmatched path (would intercept a typo'd
  `/mcpx`, future routes, etc.) and muddies the "exact root only" contract. `{$}`
  is precise.
- **Wrapping the route in `rt.RequireIdentity`** — double-gates with the wrong
  (bearer) mechanism and breaks the coarse cookie-gated human door; the gate is
  nginx's job (D4).
- **A second `appkit.Spec` field or a new hook for web pages** — unnecessary; the
  existing `Handlers` hook mounts arbitrary routes, and the landing page is just
  another route on the same mux.
- **Mounting the page on a sub-path** (e.g. `/landing`) — the product constant is
  that the human reaches it at the mount **root** (`/srv/cron/`), which is the
  service's `/`.

**Verification.**
- R-ROUT-2P8Q — with the landing handler registered on an `http.ServeMux` at
  `GET /{$}` (as the composition root wires it), a `GET /` is dispatched to the
  landing handler and returns the landing page (`200`, name+version body).
- R-ROUT-4R1S — the `{$}` exact-root registration does **not** shadow a sibling
  `POST /mcp` registration on the same mux: a `POST /mcp` still dispatches to the
  mcp handler, not the landing handler.
- R-ROUT-6T3U — a non-root path that is not otherwise registered (e.g. `GET /nope`)
  is **not** captured by the `{$}` landing pattern (it does not return the landing
  page) — proving `{$}` is exact-match, not a subtree.
