# Decision 14 — Event-routing conformance: kind `tick`, subject `/<schedule name>`, live one-family reflection

> **⛔ EXTERNAL ORDERING — operator-sequenced, not built by any cron phase.**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing, `routing.ValidKind`/`ValidSubject`/`Match` —
> specified in `eventplane/project/design/` (its D1–D4) and realized by
> **eventplane plan phases 01–04, which must be BUILT before cron's conformance
> phase runs**. It equally assumes appkit (which holds `Spec.Publishes func()
> outbox.Registry` and the chassis reflection tool over it) compiles against
> that revision — appkit's own conformance is appkit's spec, out of scope here.
> Until both land, cron's conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and the
suite-wide direction for cron is `cron.<name>` → kind `tick`, subject
`/` + schedule name (canonical key `cron:tick/bill-sweep`). cron was the
model the revision generalizes — its old trick of packing the subject *into*
the type (`cron.` + name) is exactly what made cron triggers precise, and
exactly what forced consumers (prompts) to special-case cron as "any `cron.*`
filter is accepted." The revision keeps the precision and deletes the special
case: the schedule name moves from a type suffix to a first-class subject. It
is a hard cutover — no compatibility period — inside the no-live-data
migration window.

**What cron publishes today (ground truth, not vacuous).** cron is a real,
wired producer: `Spec.Feed: "/feed"`, an **empty** `Spec.Events` (no static
registry — cron's types are dynamic), the LIVE `Spec.Publishes` provider
(`event.Publishes(store)` reading the crontab at reflection time, one
`EventType` per row), the `Producer` hook injecting the outbox, and the tick
worker (`internal/tick`) emitting `outbox.Event{Type: "cron."+name, Payload}`
via `event.Build` — one per-schedule transaction that Appends the event AND
advances `crontab.last_slot`, atomically. The type is valid by construction:
the crontab `name` column carries a DB CHECK
(`name <> '' AND name NOT GLOB '*[^a-z0-9-]*'`), so every emitted name is
non-empty `[a-z0-9-]+`.

**Decision.** cron adopts the revised envelope exactly over what it publishes:

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `cron.<name>` | `cron:tick/bill-sweep` | `tick` | `/` + the schedule name |

The payload is **unchanged** — `{name, scheduled_for, fired_at}`, UTC-RFC3339
times, same field names, same staleness semantics (the routing key addresses;
the payload remains the detail record; the `name` field stays in the payload
as the detail-record copy consumers already read).

1. **Constants and payload builder** (`internal/event/event.go`): the type
   namespace dies — `TypePrefix = "cron."` and `Type(name)` are deleted,
   replaced by:

   ```go
   // Kind is cron's one fact class: a schedule fired for a minute slot.
   const Kind = "tick"

   // Subject renders a schedule name as its routing subject: "/" + name.
   func Subject(name string) string { return "/" + name }
   ```

   `Build(name, scheduledFor, firedAt)` emits
   `outbox.Event{Kind: Kind, Subject: Subject(name), Payload: …}` with the
   payload marshaling unchanged. Nothing else in the tick seam changes: the
   same per-schedule tx Appends the event and advances `last_slot`, `Ring()`
   after the scan, the (schedule, slot) at-most-once guard untouched.

2. **Subject safety is by construction — stated, not assumed.** The crontab
   CHECK admits exactly non-empty `[a-z0-9-]+` names. `/` + such a name is a
   single-line `/`-rooted path, so it passes eventplane's
   `routing.ValidSubject` for every storable row — Append's address validation
   can never reject a tick event built from a committed crontab row. The
   charset also contains **no glob metacharacters** (`*?[]{}`) and no `/`, so
   the literal filter `cron:tick/<name>` matches that one schedule's key
   exactly and nothing else — the precision cron's packed types had, preserved
   through the matcher. The DB CHECK remains the validation boundary for the
   routing address, exactly as it was for the type suffix.

3. **The registry becomes ONE family; reflection stays live.** cron declares a
   single family — kind `tick`, subject description `/<schedule name>` — but
   keeps its **dynamic** reflection property: `event.Publishes(store)` remains
   a LIVE provider (wired through `Spec.Publishes`, threaded into the MCP
   handler as `mcp.Options.Publishes: rt.Publishes()`, per D10), rebuilt from
   the crontab on every reflection call:

   ```go
   // Publishes returns a LIVE provider: one family, kind "tick", whose
   // Description enumerates the live schedule names read from the crontab
   // at reflection time (the family-model form of the old per-row listing).
   func Publishes(store *crontab.Store) func() outbox.Registry
   ```

   The returned `outbox.Registry` holds exactly one `outbox.Family`:
   `{Kind: "tick", Subject: "/<schedule name>", Description: …, Sample:
   sample}` — the shared `{name, scheduled_for, fired_at}` sample unchanged.
   The Description carries the live enumeration: it names the current schedule
   names (e.g. `"… Live schedules: bill-sweep, nightly."`, ordered by name; a
   read failure or empty crontab degrades the enumeration to none, as today's
   provider degrades to an empty registry). So cron stops being a special case
   in *consumers* — filter validation is the ordinary
   `Registry.CouldMatch` kind-intersection with the subject left open
   (eventplane D3), no carve-out — while its reflection surface still tells an
   agent which schedules exist right now. The crontab `list` tool remains the
   authoritative instance view. `Spec.Events` stays empty: cron runs no
   family gate on Append (the kind is the compile-time constant `tick`; the
   subject boundary is the DB CHECK).

4. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. cron appends **one new
   timestamped migration** minted with `bin/create-migration cron
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across. The
   frozen `003_outbox.sql` is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest** outbox
   migration, asserting it contains the revised `outbox.SchemaSQL` verbatim.
   (This exactly mirrors gmail D18/phase-16, dropbox D22/phase-23, and ledger
   D15/phase-13.)

5. **Wire-visible prose conforms with the cutover.** The MCP instructions and
   tool descriptions (`internal/mcp`) that name `cron.<name>` as the emitted
   event now name the routing key: instructions become "Named UTC cron
   schedules that publish a `cron:tick/<name>` event on a timer. Create a
   schedule, then wire consumers to its event."; the create/list/update/delete
   descriptions replace "the event type `cron.<name>`" phrasing with "the
   event `cron:tick/<name>`" (schemas, error envelopes, and the seven-tool
   surface unchanged — D10's partition is untouched). No `cron.<name>` /
   `"cron."` reference survives in Go source; the doctrine headers
   (`cmd/cron/main.go`, `internal/event`, `internal/tick`,
   `internal/crontab`) are trued to the kind/subject shape.

**Rejected.**
- **Keeping the packed type as the kind (kind `cron.<name>` or
  `tick.<name>`)** — the revision's core principle is structure in the
  envelope; packing the subject into the kind forces consumers to parse keys
  and resurrects the special case the revision deletes.
- **An empty subject (gmail's choice)** — cron is (with dropbox) the
  motivating case *for* subjects: the schedule name is exactly the routing
  dimension every cron consumer selects on; dropping it would make every
  consumer fire on every tick.
- **Enumerating live schedules as N families (one per crontab row)** — the
  family model describes subjects rather than enumerating them (eventplane
  D3); N same-kind families make `Detail("tick")` ambiguous and re-create the
  closed-enumeration shape. One family + a live Description keeps the model
  and the property.
- **A static `Spec.Events` registry gating Append on kind `tick`** — the gate
  would guard a compile-time constant against nothing, and moving reflection
  off the LIVE `Publishes` provider would silently freeze the schedule
  enumeration (the D10 rejection, still true under the family model).
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are retention data with no live
  consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: the emission claims run the real
tick worker over a temp SQLite database built by the full embedded migration
set through the appkit runner, reading outbox rows back by SQL; the feed claim
runs the real eventplane `FeedHandler` over `httptest`; the reflection claim
drives the assembled `appkit/mcp` handler over the live provider, per D10's
harness).

- R-PQH6-2RYI — after `tick.Worker.Fire` over two schedules (`nightly`,
  `bill-sweep`) whose exprs match the slot, the outbox rows read back by SQL
  from the real SQLite database carry `kind` exactly `tick` on every row (no
  `cron.` prefix) and `subject` exactly `/nightly` and `/bill-sweep` (two rows,
  two different subjects), each appended on the same tx as its `last_slot`
  advance; a second `Fire` for the same slot appends **zero** further rows
  (the (schedule, slot) at-most-once guard preserved across the envelope
  change).
- R-PVCR-LUXA — subject safety by construction: for names at the crontab
  CHECK's boundary (e.g. `a`, `a-1`, `bill-sweep`), `Subject(name)` passes
  `routing.ValidSubject` and `Append` accepts the built event; and the literal
  filter is exact — `routing.Match("cron:tick/bill-sweep",
  "cron:tick/bill-sweep")` is true while the same pattern does **not** match
  `cron:tick/bill-sweep2` or `cron:tick/bill` (the `[a-z0-9-]` charset
  contains no glob metacharacters, so a schedule name never widens a filter).
- R-PRP2-GJP7 — the live provider returns a `Registry` of **exactly one**
  family with kind `tick` and subject description `/<schedule name>`, whose
  schema and example (from the unchanged `{name, scheduled_for, fired_at}`
  Sample) agree — and it is LIVE: through the assembled MCP handler, after
  creating schedules `nightly` and `bill-sweep` the reflected family's
  description names both schedule names, and after deleting `nightly` a fresh
  reflection call no longer names it (still exactly one family throughout).
- R-PSWY-UBFW — applying cron's full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table with
  `kind` and `subject` columns and **no** `type` column; the DDL drift guard
  asserts the newest outbox migration contains the revised `outbox.SchemaSQL`
  verbatim while `003_outbox.sql` remains byte-identical to its committed
  (frozen) body.
- R-PU4V-836L — end-to-end over cron's own substrate: an event appended
  through cron's producer path (`event.Build` → the tick worker's per-schedule
  tx → `Append` on the migration-created table, source `cron` — the `Spec.App`
  value) for schedule `bill-sweep` and served by the real eventplane
  `FeedHandler` over an `httptest` connection frames
  `event: cron:tick/bill-sweep` (the canonical key) with a `data:` JSON
  envelope carrying `kind` and `subject` keys and **no** `type` key.
