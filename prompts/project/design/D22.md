# Decision 22 ‚Äî Content-plane holder: run sandbox files at `GET /run-content`

**Problem.** The content plane (`docs/content-plane-design.md`) makes prompts a
**holder**: "run files are content, too" ‚Äî a run's *product* (an extracted
JSON, a generated report) must be addressable by a loopback content URL so an
agent can hand it onward **by reference** (e.g. dropbox
`put(path, source_url=<run file URL>)`) instead of round-tripping bytes through
`run_fs_read` and model context. Today a run's sandbox files are reachable only
through the line-oriented MCP readers; nothing can name their bytes.

**Decision.** prompts exposes a loopback content endpoint addressed by the
holder's own key, `(run_id, sandbox path)`, and surfaces the reference on
`run_fs_list`.

1. **The endpoint: `GET /run-content?run_id=<id>&path=<sandbox-relative path>`**
   ‚Äî a new `Service.RunContentHandler() http.Handler` in
   `internal/prompt/content.go`, following dropbox's `/content` handler
   pattern verbatim:
   - **Loopback-only guard** (the primary and sufficient guard, same as `/feed`
     and dropbox `/content`): the route is reachable only over loopback. The
     guard is the shared chassis `server.LoopbackOnly` (applied at the mount via
     `Router.HandleLoopback`, D27), which keys on `X-Forwarded-Proto` **only** ‚Äî
     nginx stamps it on every proxied request, so any non-empty value ‚Üí bare
     404, and a loopback caller (which asserts `X-Owner-Email` but no
     `X-Forwarded-Proto`) is served. `RunContentHandler` itself reads no request
     header. The route is otherwise unauthenticated ‚Äî one box, one owner, the
     perimeter is loopback. No owner scoping (an event/consumer caller has no
     identity; same trust model as dropbox's holder route).
   - **Resolution through the sandbox manager.** `run_id` resolves via the
     existing `sandbox.Manager` root (`<sandboxes>/<run_id>/sandbox`); the
     manager gains a byte-level `Open(id, relPath) (*os.File, os.FileInfo,
     error)` applying the **same** `validateID` + `confine` machinery `List`/
     `Read` use. The handler serves the open file with `http.ServeContent`
     (correct `Content-Type` by extension, `Content-Length`, Range/conditional
     for free), passing the file's modtime.
   - **Error mapping never leaks:** missing/invalid `run_id`, an absent path, a
     directory, or a confinement/escape error ‚Üí **404** (never 500, never a
     path echo). There is **no `rev` parameter**: a run's sandbox files are
     unversioned working files, so the plane's optional rev‚Üí409 pin has
     nothing to pin ‚Äî a documented deviation-by-omission, exactly like gmail's
     (D16 there). A consumer needing integrity hashes what it fetched.
   - **Mounted loopback-only in-process** in `registerRoutes`
     (`rt.HandleLoopback("GET /run-content", svc.RunContentHandler())`, D27),
     beside `GET /{$}` ‚Äî the chassis `LoopbackOnly` wrap is the guard, and nginx
     never routes it (defence-in-depth `location = ‚Ä¶ { return 404; }` is
     optional, never the mechanism).

2. **`run_fs_list` entries carry the reference.** Each **non-directory** entry
   in `run_fs_list`'s result gains a `content_url`: the complete, URL-encoded
   loopback URL (`url.Values.Encode` over `run_id` + the listed path joined
   with the entry name) ‚Äî the consumer never constructs or rewrites it.
   Directory entries carry no `content_url` (nothing at the endpoint to name).
   The reference is **`{content_url, size}`** (size is already on the entry):
   no `content_hash` ‚Äî stamping one would read every file on every list, the
   same rejection gmail recorded ‚Äî and no `rev` (above). The base URL is
   resolved by registry name at the composition root and threaded through the
   tool table, gmail D17's exact seam shape:

   ```go
   // cmd/prompts/main.go, in registerRoutes:
   contentBase := registry.BaseURL("prompts") // "http://127.0.0.1:3002" ‚Äî no literal in source (R-RG04-NLIT holds)

   // internal/mcp (revising D17's signatures in place):
   func Tools(svc *prompt.Service, contentBase string) []mcp.Tool
   func NewHandler(svc *prompt.Service, contentBase string, rt *appkit.Router) (http.Handler, error)
   ```

   `run_fs_list`'s description states that entries carry a loopback
   `content_url` for byte fetch **by services** (a run's `Fetch` tool, a
   dropbox `put(source_url)`), not by the agent. No tool is added or removed ‚Äî
   the surface stays eighteen tools (D17's R-DKQP-QZ3Q unchanged).

**Rejected.**
- **Serving bytes through MCP (`run_fs_read` returning base64 / whole files)**
  ‚Äî the by-value crossing the plane retires; `run_fs_read` stays the small
  line-oriented convenience it is.
- **An env override for the content base (`PROMPTS_CONTENT_BASE_URL`)** ‚Äî the
  base names *this service's own* port, fixed by the registry; unlike the
  dropbox/feed peers there is no deploy seam to repoint, and a knob nobody can
  meaningfully set is drift surface. Tests build their expected URLs from
  `registry.MustPort("prompts")`.
- **Addressing by a minted opaque token instead of `(run_id, path)`** ‚Äî the
  plane says holders address by their own key; run_id + sandbox path *is*
  prompts' key, already owner-visible via `run_fs_list`, and a token store
  would add state for no confinement gain (the handler confines anyway).
- **A `rev`/pin dimension** ‚Äî sandbox files have no revision identity; 409
  would be a fiction. Omission recorded above.
- **`content_url` on `run_fs_read` results too** ‚Äî `run_fs_read` returns file
  *text*, not entries; the reference belongs on the enumeration surface the
  agent hands onward. One home, no drift.

**Verification** (real substrates: the handler claims drive the real
`RunContentHandler` over a real temp sandbox tree via `net/http/httptest`; the
`run_fs_list` claim goes through the assembled `appkit/mcp` handler over a real
`prompt.Service` + temp SQLite, expected URLs built from
`registry.MustPort("prompts")`, never a port literal).

- R-6C2D-19HN ‚Äî `GET /run-content?run_id=‚Ä¶&path=‚Ä¶` (path URL-encoded, e.g.
  `reports%2Fout.pdf`) for an existing sandbox file returns 200 with a body
  byte-identical to the file, a `Content-Length` equal to its size, and a
  `Content-Type` derived from the extension (e.g. `application/pdf` for
  `.pdf`).
- The loopback guard on this route is the shared chassis `LoopbackOnly`
  (`Router.HandleLoopback`, D27), keyed on `X-Forwarded-Proto` only; its
  behavior is verified by D27's R-BI5J-4GM6, not here.
- R-6EI5-SSZ1 ‚Äî an unknown `run_id`, an absent `path`, a directory `path`, and
  an escaping `path` (`../../secrets`) each return **404** ‚Äî never 500 and
  never the resolved path in the response body.
- R-6FQ2-6KPQ ‚Äî `run_fs_list` through the assembled MCP handler returns, for
  each non-directory entry, a `content_url` equal to
  `http://127.0.0.1:<registry prompts port>/run-content?<encoded query>` whose
  parsed query round-trips exactly the run's id and the entry's joined
  sandbox-relative path (proving the encoding, including a subdirectory
  listing), while directory entries carry **no** `content_url` key ‚Äî and
  fetching a returned `content_url` against the mounted handler yields that
  file's bytes (the reference is live, not decorative).
