# Decision 23 ‚Äî Box PDF tooling in the framing prompt; model-native PDF is a non-goal

**Problem.** The bill-processing workflow lands PDFs in run sandboxes (a
`Fetch` of a dropbox invoice, D21), but a run's model has no way to know the
box can read them: the framing prompt advertises only the generic sandbox
tools, so a model facing `input.pdf` will either give up or hallucinate
tooling. The box **does** carry the tools: `opsctl init-box` installs
`poppler-utils` ‚Äî `pdftotext`, `pdftoppm`, `pdfinfo` ‚Äî as box-baseline
provisioning (sealed in opsctl's spec, its D10; an **external precondition**
here, not built or re-specified by any prompts phase).

**Decision.** The framing prompt (`framing_prompt.go`) gains one claims
sentence after the sandbox-tools sentence: PDF tooling is available **in
Bash** ‚Äî `pdftotext` (extract text), `pdftoppm` (render pages to images),
`pdfinfo` (metadata) ‚Äî so a run works on PDF files with shell commands, on
disk. The claim names the three binaries and nothing else (no versions, no
other packages, no services ‚Äî D19's no-service-enumeration constraint is
untouched).

**Model-native PDF reading is an explicit NON-GOAL.** Provider document blocks
(sending PDF bytes as model input) are not available to prompts: agentkit's
`Block` union is sealed and text-only ‚Äî verified in
`~/projects/agentkit/block.go`, exactly `TextBlock` / `ToolUseBlock` /
`ToolResultBlock` / `ReasoningBlock`, no document/file block ‚Äî and
`Conversation.Send` takes a single string. Document input therefore waits on a
future agentkit release and a later spec extension here; nothing in prompts
pretends otherwise, and the framing prompt makes no claim about the model
"seeing" PDFs.

**Rejected.**
- **Provisioning poppler from prompts (a setup hook, a vendored binary)** ‚Äî
  box provisioning is opsctl's domain; opsctl D10 already seals it as
  init-box baseline. prompts only *claims* what the box guarantees.
- **A dedicated `pdf_extract` sandbox tool** ‚Äî a wrapper around `pdftotext`
  the Bash tool already runs; a seventh-plus tool for a shell one-liner is
  surface without power.
- **Waiting for agentkit document blocks instead of claiming shell tooling** ‚Äî
  blocks are sealed text-only today; the workflow needs PDFs readable now, and
  shell extraction on disk is the content-plane-honest path (bytes never enter
  context; extracted text enters only as the run chooses).
- **Enumerating poppler versions or more of the toolchain in the prompt** ‚Äî
  claims drift; three stable binary names are the entire load-bearing fact.

**Verification.**

- R-6I5U-Y474 ‚Äî the framing prompt sent as the conversation `System` names
  `pdftotext`, `pdftoppm`, and `pdfinfo` as available in Bash (all three
  present in the assembled `System` string), while still naming no individual
  suite service (composes with R-9OJA-B2KU).
