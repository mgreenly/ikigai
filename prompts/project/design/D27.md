# Decision 27 — Structured MCP adoption: `StructuredResult`, typed error codes, output schemas, shared loopback guard

**Problem.** prompts' domain tools were written for the agent-only era: each
success builds its result with `appkitmcp.JSONResult(v)` (a struct marshalled to
a JSON string in a text block) and each failure with the old single-argument
`appkitmcp.ErrorResult(err.Error())` — a bare human string, no machine-branchable
code. The suite has since settled a single verb surface for agents *and*
deterministic machine callers (`docs/structured-mcp-design.md`): every domain
verb returns `structuredContent` beside its mirrored text, declares an
`outputSchema`, and carries a code from the shared error vocabulary; loopback
endpoints key their guard on `X-Forwarded-Proto` alone. appkit (phases 12–14)
already ships that contract: `StructuredResult(v) (map[string]any, error)`,
`ErrorResult(code ErrorCode, msg string)` over the closed vocabulary
(`validation`, `not_found`, `conflict`, `too_large`, `source_unavailable`,
`internal`), `Tool.OutputSchema`, and `server.LoopbackOnly` /
`Router.HandleLoopback`. `JSONResult` is **deleted** from appkit and
`ErrorResult` is re-signed, so prompts does not compile until it adopts them.
This Decision conforms prompts to that contract. It **formalizes existing
output**; it never reshapes a result body.

**Decision.** Four coordinated changes to prompts' MCP surface
(`internal/mcp/tools.go`), its error taxonomy (`internal/prompt`,
`internal/sandbox`), and its content-holder guard
(`internal/prompt/content.go` + `cmd/prompts/main.go`).

### 1. Structured results on every domain success

Each of the **thirteen structured domain tools** — `create`, `import`, `list`,
`get`, `update`, `delete`, `set_trigger`, `clear_trigger`, `run`, `run_list`,
`run_get`, `run_cancel`, `run_fs_list` — swaps its `JSONResult(v)` for
`StructuredResult(v)`. The value handed in is **byte-for-byte the same object**
these tools marshal today (`{"prompt_id": …}`, `{"runs": […]}`, the
`PromptDetail`, the `entries` array with its `content_url` stamping, …); the
swap adds a `structuredContent` field carrying that object and keeps the
mirrored text block. Because `StructuredResult` returns
`(map[string]any, error)`, each handler propagates that error honestly:

```go
run, err := svc.Run(ctx, id.OwnerEmail, in.PromptID)
if err != nil {
    return fail(err), nil
}
return appkitmcp.StructuredResult(map[string]any{
    "run_id": run.ID, "status": run.Status, "started_at": run.StartedAt,
})
```

A `StructuredResult` marshal error is returned as `(nil, err)` — never
swallowed — surfacing as a JSON-RPC internal error, exactly as any other
`Handler`-returned error does.

### 2. Prose exceptions keep `TextResult` and declare no schema

Three tools are **not** structured and are classified as prose exceptions per
the suite contract:

- **`describe`** — a documentation tool; returns the `describeText` overview for
  a human/agent reader.
- **`run_output`** — a raw-content read: a run's append-only stream-json log,
  one JSON event per line, returned verbatim for tailing. It is a byte stream,
  not a single typed value.
- **`run_fs_read`** — a raw-content read: the text of one sandbox file, returned
  verbatim.

These keep `appkitmcp.TextResult(...)` and declare **no** `OutputSchema` (nil),
so `tools/list` omits `outputSchema` for exactly these three. (`health` and
`reflection` are chassis-owned and already structured; prompts declares
neither.)

### 3. An `outputSchema` per structured tool

Every one of the thirteen structured tools declares an `OutputSchema` — a
hand-authored `map[string]any` JSON-schema literal mirroring the JSON it already
emits, in the house style of `appkit/mcp`'s `objectSchema` helper. prompts keeps
a small private `objSchema(props, required...)` (its existing `obj` helper,
reused for outputs) rather than importing appkit's unexported one. The literals
mirror the emitted shape verbatim and never reshape it; open sub-objects whose
keys are not fixed (the stored `config` blob, a `PromptDetail`'s derived
`last_run`) use `additionalProperties: true`. Representative shapes:

```go
// create → {"prompt_id": "<id>"}
objSchema(map[string]any{"prompt_id": typ("string")}, "prompt_id")

// run → {"run_id","status","started_at"}
objSchema(map[string]any{
    "run_id": typ("string"), "status": typ("string"), "started_at": typ("string"),
}, "run_id", "status", "started_at")

// run_fs_list → {"entries":[{"name","is_dir","size"[,"content_url"]}]}
objSchema(map[string]any{
    "entries": map[string]any{"type": "array", "items": objSchema(map[string]any{
        "name": typ("string"), "is_dir": typ("boolean"),
        "size": typ("integer"), "content_url": typ("string"),
    }, "name", "is_dir", "size")},
}, "entries")
```

`get`/`update` (a `Prompt`/`PromptDetail`), `list`/`run_list` (arrays of those),
and `run_get` (a `Run`) declare object/array literals mirroring their structs,
with `additionalProperties: true` on the open `config`/`last_run` sub-objects.
The schemas are the machine contract, versioned with the service and covered by
its tests — appkit does not validate responses against them at runtime
(`docs/structured-mcp-design.md`, non-goals).

### 4. Typed error codes — the real failure → code mapping

A single classifier in `internal/mcp` turns any domain error into a coded
`ErrorResult`, so every `ErrorResult` call site becomes one shared helper rather
than a bare-string swap:

```go
// fail wraps a domain error as a coded MCP tool error.
func fail(err error) map[string]any { return appkitmcp.ErrorResult(codeFor(err), err.Error()) }

func codeFor(err error) appkitmcp.ErrorCode {
    switch {
    case errors.Is(err, prompt.ErrNotFound):          return appkitmcp.ErrNotFound
    case errors.Is(err, prompt.ErrValidation):        return appkitmcp.ErrValidation
    case errors.Is(err, prompt.ErrTooLarge):          return appkitmcp.ErrTooLarge
    case errors.Is(err, prompt.ErrSourceUnavailable): return appkitmcp.ErrSourceUnavailable
    default:                                          return appkitmcp.ErrInternal
    }
}
```

`parseArgs` failures stay outside this path: a `paramError` is still returned as
`(nil, err)` so the chassis maps it to JSON-RPC `-32602` (invalid params), never
an `isError` content result. The mapping, failure by failure:

| failure condition | domain surface | code |
|---|---|---|
| unknown / foreign-owned prompt or run id | `prompt.ErrNotFound` (store `GetPrompt`/`GetRun`, `runForOwner`) | `not_found` |
| import source path absent from the dropbox mirror | `prompt.ErrNotFound` (`Fetch` 404) | `not_found` |
| sandbox path absent / not a file (`run_fs_read`, `run_fs_list`) | `prompt.ErrNotFound` (translated `sandbox.ErrNotFound`) | `not_found` |
| bad trigger binding (unknown source, malformed filter, filter matches no family) | `prompt.ErrValidation` (`validateTrigger`) | `validation` |
| import `source_path` empty, or body not valid UTF-8 | `prompt.ErrValidation` (`Import`) | `validation` |
| invalid triggering-event arguments (event-triggered run) | `prompt.ErrValidation` | `validation` |
| sandbox path escapes / is malformed (`run_fs_read`, `run_fs_list`) | `prompt.ErrValidation` (translated `sandbox.ErrPathEscape`) | `validation` |
| import body over the 1 MiB limit | `prompt.ErrTooLarge` (`Import`) | `too_large` |
| dropbox mirror unreachable (transport error, or non-200/404 status) | `prompt.ErrSourceUnavailable` (`Fetch`) | `source_unavailable` |
| any other / unexpected error (store or DB failure) | unclassified | `internal` |

Two of these codes have no home in today's domain error surface, so this
Decision **adds the discrimination the mapping needs** rather than forcing every
failure through `ErrValidation`/`internal`:

- **`prompt.ErrTooLarge`** (new sentinel in `internal/prompt/model.go`).
  `Service.Import`'s over-limit branch (`service.go`, the `len(data) >
  maxImportBytes` check) wraps `ErrTooLarge` instead of `ErrValidation` — a
  1 MiB-plus body is a size refusal, and `too_large` is its exact suite code.
- **`prompt.ErrSourceUnavailable`** (new sentinel). `httpFetcher.Fetch`
  (`dropbox.go`) wraps it for a transport error (`hc.Do` failure) and for any
  non-200/404 response status — the mirror being down is a retryable
  source-unavailable condition, distinct from a 404 "no such file" (`not_found`)
  and from a caller mistake (`validation`).
- **Sandbox errors carry sentinels.** `internal/sandbox` exports
  `sandbox.ErrNotFound` (absent path / not a file) and `sandbox.ErrPathEscape`
  (confinement escape / invalid path), wrapped by the existing `%w` sites.
  `Service.RunFsList`/`RunFsRead` translate them into `prompt.ErrNotFound` /
  `prompt.ErrValidation` so the MCP layer classifies uniformly by prompt
  sentinel with no string matching. (The `GET /run-content` holder is
  unaffected — it deliberately collapses every resolution failure to a bare 404
  by D22 R-6EI5-SSZ1, and keeps doing so.)

### 5. The content-holder guard moves to the chassis

`GET /run-content` (D22) is a loopback-only content endpoint. Its guard is
today a hand-copied predicate **inside** `Service.RunContentHandler`
(`content.go`): it 404s any request bearing `X-Owner-Email` *or*
`X-Forwarded-Proto`. That conflates "carries identity" with "came from outside",
which the caller-asserted-identity model breaks. The predicate is deleted from
the handler; the route mounts through the shared chassis guard instead:

```go
// cmd/prompts/main.go — was rt.Handle("GET /run-content", svc.RunContentHandler())
rt.HandleLoopback("GET /run-content", svc.RunContentHandler())
```

`Router.HandleLoopback` wraps the handler in `server.LoopbackOnly`, which keys
on `X-Forwarded-Proto` alone (any non-empty value → bare 404; nginx stamps it on
every proxied request). `RunContentHandler` keeps only its own resolution logic
(`run_id`/`path` required, sandbox `Open`, `http.ServeContent`, 404-never-500);
it reads **no** request header. A loopback caller that asserts `X-Owner-Email`
but no `X-Forwarded-Proto` is now **served**, where the old predicate 404'd it.
The identity-caller assertion in `internal/suite` is unchanged — it is a
conforming loopback caller, not a guard.

**Rejected.**
- **A mechanical two-way swap** (`ErrNotFound`→`not_found`, everything else
  →`validation`/`internal`) — leaves `too_large` and `source_unavailable`
  permanently unused and mislabels a size refusal and a dead-mirror as
  `validation`/`internal`, exactly the machine-branch mistakes the vocabulary
  exists to prevent. The added sentinels are cheap and the mapping is honest.
- **String-matching sandbox error text at the MCP layer** to derive
  `not_found` vs `validation` — fragile against wording changes; the sandbox
  package owns its failure modes, so it exports sentinels and the service
  translates them.
- **Keeping the over-1-MiB import failure as `ErrValidation`** — a size limit
  is what `too_large` names; folding it into `validation` throws away the
  distinction a machine caller wants (retry with a smaller body vs. fix the
  request).
- **Declaring `outputSchema` on the prose tools** — `describe`/`run_output`/
  `run_fs_read` return prose or a raw byte stream, not a typed object; a schema
  would be a fiction the suite contract explicitly exempts.
- **Leaving the guard predicate in `RunContentHandler`** — the suite settled one
  shared loopback guard in appkit; a per-service copy is the drift the extraction
  removes, and this copy encodes the wrong (`X-Owner-Email`) key.

**Verification.** (Real substrate: prompts' own HTTP/MCP layer. Structured-tool
and error-code claims drive the assembled `appkit/mcp` handler built by
`NewHandler` over a real `prompt.Service` + temp SQLite via `net/http/httptest`;
the guard claim drives the real `RunContentHandler` mounted through
`Router.HandleLoopback` over a real temp sandbox tree. No mock stands in for the
transport.)

- R-B4QM-WZGJ — through the assembled handler, `tools/list` returns an
  `outputSchema` (a `type:"object"` schema) for **each** of the thirteen
  structured domain tools and for the chassis `health`/`reflection`, and returns
  **no** `outputSchema` key for exactly the three prose tools `describe`,
  `run_output`, `run_fs_read` (table-driven over the listed tools).
- R-B5YJ-AR78 — for **each** of the thirteen structured tools, a successful
  `tools/call` result carries a `structuredContent` object **and** a text
  content block whose JSON parse is deep-equal to that `structuredContent`
  (proving the `StructuredResult` dual rendering replaced `JSONResult` on every
  one — a tool still building a text-only body fails the mirror check).
- R-B76F-OIXX — a successful `tools/call` of `describe`, `run_output`, and
  `run_fs_read` each returns a text content block and carries **no**
  `structuredContent` key (the prose exception preserved end to end).
- R-B8EC-2AOM — `get` (and `run_get`) for an id that does not exist / is
  foreign-owned returns `isError:true` with `structuredContent.code ==
  "not_found"` (not `internal`, not `validation`).
- R-B9M8-G2FB — `set_trigger` with an unknown source (e.g. `nope:foo/**`)
  returns `isError:true` with `structuredContent.code == "validation"`.
- R-BGXM-QOVH — a tool whose service call returns an **unexpected, unclassified**
  error (a forced store/DB failure, matching no domain sentinel) returns
  `isError:true` with `structuredContent.code == "internal"` — proving the
  classifier's default arm is `internal`, not `validation` (a degenerate
  classifier that defaults everything to `validation` fails this).
- R-BI5J-4GM6 — `GET /run-content?run_id=…&path=…` for an existing sandbox
  file, mounted through `Router.HandleLoopback`: a request bearing
  `X-Forwarded-Proto: https` receives a bare 404 and serves **none** of the
  file's bytes; the **same** request WITHOUT `X-Forwarded-Proto` but WITH
  `X-Owner-Email` set is served 200 with the file's bytes — the discriminating
  flip from the old two-header predicate, which 404'd that `X-Owner-Email` case.
- R-BC21-7LWP — `import` of a mirror file whose body exceeds 1 MiB returns
  `isError:true` with `structuredContent.code == "too_large"` (**not**
  `"validation"`), driven through a fetcher fake serving a >1 MiB UTF-8 body.
- R-BD9X-LDNE — `import` when the dropbox mirror is unreachable (the fetcher
  yields a transport error or a non-200/404 status) returns `isError:true` with
  `structuredContent.code == "source_unavailable"` (distinct from the 404
  `not_found` case).
- R-BEHT-Z5E3 — `run_fs_read` of a sandbox path that is absent returns
  `isError:true` with `structuredContent.code == "not_found"` (the sandbox
  `ErrNotFound` translated, not defaulted to `internal`).
- R-BFPQ-CX4S — `run_fs_read` of an escaping sandbox path (`../../secrets`)
  returns `isError:true` with `structuredContent.code == "validation"` (a bad
  caller path is `validation`, not `not_found` and not `internal`).
