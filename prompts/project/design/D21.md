# Decision 21 — Content-plane acceptor: the `Fetch` sandbox tool

**Problem.** The suite's content plane (`docs/content-plane-design.md`) fixes
the rule *agents pass references; services move bytes*, and names the prompts
sandbox as **the one sanctioned crossing**: the place a content reference
legitimately becomes bytes an agent can operate on — landing on **disk, not in
model context**. Today a run has no way to obtain the bytes behind a
`content_url` (a dropbox event's file, a gmail attachment, another run's
artifact): the sandbox has no network from Bash by design, and `run_fs_read` /
dropbox `get` style base64 crossings are exactly what the plane retires. The
plane's conformance map row for prompts names `fetch(content_url, dest)` as the
acceptor half prompts owes.

**Decision.** A seventh sandbox tool, **`Fetch`** (D5 owns the toolset table;
this Decision owns the tool's contract), executed by the runner **server-side**
— the model sees only the small JSON result, never the bytes.

```go
// prompts/internal/tools — beside the six existing tools.
type fetchInput struct {
    ContentURL string `json:"content_url" jsonschema:"required,description=A suite loopback content URL (e.g. from an event payload or a tool result)"`
    DestPath   string `json:"dest_path" jsonschema:"required,description=Destination file path, relative to the sandbox root"`
}
// fn: closes over sandboxRoot and sourcePortAllowed; returns the JSON object
// {"path": <dest_path>, "size": <bytes written>, "content_hash": <64-hex sha256>}
```

- **URL confinement (the content-plane acceptor rule, validated before any
  network I/O — dropbox D19's exact rule and seam).** The URL must parse, use
  scheme `http`, have host literally `127.0.0.1` or `::1` (no hostnames —
  `localhost` is rejected; the plane is loopback plumbing, not name
  resolution), and carry an explicit port for which `sourcePortAllowed` is
  true. Any violation is a tool error whose text begins `validation:` and
  names the violated rule, and the tool performs **zero** fetch calls. The
  production `sourcePortAllowed` is derived at the composition root from
  `registry.Services` (every registered port true, everything else false);
  tests inject a set containing an `httptest` server's loopback port.
- **Destination confinement.** `dest_path` resolves through the same sandbox
  confinement every other tool uses (D5); an escaping `dest_path` is rejected
  **before** any network I/O — zero fetch calls, no file written anywhere.
- **Server-side fetch, streamed to disk.** A conforming URL is fetched with a
  plain `GET` (a dedicated `http.Client` with a bounded connect/response-header
  timeout, no overall body deadline — the run's TTL context already bounds the
  call; **redirects are not followed** — a redirect could re-target the fetch
  outside the confined origin, so any 3xx is a failure). The body streams into
  a temp file in the sandbox with the SHA-256 computed on the same pass
  (`io.TeeReader` into `crypto/sha256`), then temp+rename to `dest_path` —
  a torn stream leaves no partial file at `dest_path`. There is deliberately
  **no size cap**: bytes land on disk, never in context.
- **`content_hash` is plain SHA-256, lowercase hex (64 chars)** — the plane's
  reference tuple pins only "64-hex content hash"; prompts uses stock
  `crypto/sha256` of the fetched bytes. (Deliberately **not** dropbox's
  block-SHA-256, which is a Dropbox-API-specific holder value; a consumer
  comparing against a dropbox reference's `content_hash` compares dropbox's
  own algorithm on dropbox's side — prompts' hash names what *prompts* wrote.)
- **Pinned failure mapping** (what the model sees as the tool error text; the
  runner returns the error from `Call`, agentkit renders it as the
  tool_result error):
  - malformed / non-`http` / non-loopback / unregistered-port URL, escaping
    `dest_path` → `validation: …` (zero fetch calls);
  - source responds **404** → `not_found: …` ("the reference is stale or
    absent — re-derive it from the holder");
  - source responds **409** (a holder's rev-pin moved on) → `conflict: …`;
  - connection refused, timeout, a redirect, any other non-2xx, or a torn
    body → `source_unavailable: …` (transient, retryable — distinguishable
    from absence).
  In every failure case no file exists at `dest_path`.
- **Framing prompt.** `framing_prompt.go`'s tool sentence names the seven
  tools ("bash, read, write, edit, glob, grep, and fetch") and gains one
  sentence of guidance: fetch takes a suite content URL (from an event payload
  or a tool result) and lands the bytes as a sandbox file — it is the only way
  bytes enter the sandbox from another service. The "NO network access from
  bash" sentence stays, verbatim true: fetch is a runner-executed tool, not
  bash networking, and it reaches only the box's own loopback services. The
  paragraph names no individual service (D19's R-9OJA-B2KU constraint holds).

**Rejected.**
- **Returning fetched bytes (or base64) to the model** — the by-value crossing
  the content plane exists to retire; the sandbox disk is the crossing.
- **Fetching arbitrary URLs / hostnames** — the plane's confinement is
  absolute; internet ingestion is a connector's domain job, never a generic
  fetch. Following redirects would be the same hole via one indirection.
- **A size cap** — the base64-in-JSON caps elsewhere exist because bodies are
  buffered whole in context; this path streams to disk end to end.
- **Dropbox block-SHA-256 for `content_hash`** — a Dropbox-specific chunked
  algorithm adopted suite-wide for no consumer that needs it; plain SHA-256 is
  the stock 64-hex answer and the plane pins only the shape.
- **Implementing fetch as an MCP tool on the foreground surface** — the
  foreground agent passes references; it is the *run* that needs the bytes on
  its own disk. The sandbox tool is the sanctioned crossing.

**Verification** (real substrates: every fetch claim runs against a **real
local HTTP server** — `httptest` on loopback — with its port injected through
the `sourcePortAllowed` seam; confinement rejections assert the server
recorded **zero** requests).

- R-65YV-4ES6 — `Fetch` of an allowed-port `httptest` URL serving known bytes
  streams them **server-side** into the sandbox at `dest_path` (the file's
  bytes equal the served bytes; the server records exactly one `GET`) and
  returns `{path, size, content_hash}` where `size` is the byte count and
  `content_hash` is the lowercase-hex SHA-256 of those bytes.
- R-676R-I6IV — URL confinement discriminates with **zero fetch calls** on
  every rejection: a non-loopback host (`http://10.0.0.5:3202/…`) →
  `validation`; a hostname (`http://localhost:<allowed port>/…`) →
  `validation`; `127.0.0.1` on a port the allowed set does not contain →
  `validation`; the byte-equivalent URL on `127.0.0.1` with an allowed port
  proceeds to fetch.
- R-68EN-VY9K — an escaping `dest_path` (`../outside.bin`, an absolute path
  outside the sandbox) is rejected with a `validation` error, **zero** fetch
  calls occur, and no file is created outside the sandbox.
- R-69MK-9Q09 — the pinned failure mapping against a real local server: a 404
  response → tool error text beginning `not_found`; a 409 → `conflict`; a
  connection-refused target (closed allowed port) → `source_unavailable`; a
  302 redirect response is **not followed** (the redirect target records no
  request) and fails as `source_unavailable` — and after each failure no file
  exists at `dest_path`.
- R-6AUG-NHQY — the framing prompt sent as the conversation `System` names
  `fetch` among the sandbox tools and states that it takes a content URL and
  writes the bytes into the sandbox, while still naming no individual service
  (no `ikigenba_`, no per-service enumeration — composes with R-9OJA-B2KU).
