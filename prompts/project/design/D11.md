# Decision 11 — Conform the landing page to the cron canonical template

**Decision.** prompts's landing page is brought to **visual and structural parity
with the cron landing page**, which is the suite's canonical reference — the one
page that holds up the Carbon style guidelines. cron's page is a centered `main`
with an eyebrow label, a display `<h1>` showing the service name, a one-line lead
paragraph, and a bordered three-cell **Service / Version / API** definition list,
over its own on-disk `tokens.css`. The landing page shipped under D10 has its own
markup and token values; this Decision **replaces that markup and those tokens
wholesale with cron's**, keeping only the data that must reflect **this** service.

**Canonical source of truth.** Two files, as committed on `main`:

- `cron/internal/web/landing.html` — the exact page markup + inline `<style>`.
- `cron/internal/web/static/tokens.css` — the Carbon design tokens.

The conform target is **defined by those two files**. There is no frozen second
copy to keep in sync: if cron's canonical page ever changes, this page re-derives
from it by the same diff.

**What changes (markup + tokens only — no Go logic, no handler signature, no
nginx):**

- **`prompts/share/www/landing.html`** (the on-disk template moved there by D16;
  formerly `internal/web/landing.tmpl`) becomes **identical to cron's canonical
  landing markup except exactly three substitutions.** Every other line — the
  inline `<style>`, the `<h1>{{.Service}}</h1>`, the `Service / Version / API`
  `<dl>` whose API value is `POST /mcp`, and the `static/tokens.css` link — is
  cron's, byte-for-byte:

  | line | cron (canonical) | prompts |
  |---|---|---|
  | `<title>` | `{{.Service}} · cron` | `{{.Service}} · prompts` |
  | `.eyebrow` text | `Scheduled event emitter` | `Agent sessions` |
  | lead `<p>` | `Cron keeps named schedules in SQLite and emits typed event-plane messages at minute boundaries.` | `Prompts runs sandboxed Claude agent sessions on the owner's behalf and exposes them as MCP tools.` |

  The template's data fields are unchanged: the chassis renders the page with a
  `struct{ Service, Version string }`, so `{{.Service}}` / `{{.Version}}` carry
  over verbatim from cron's markup.

- **`prompts/share/www/static/tokens.css`** becomes **byte-identical to cron's
  canonical `tokens.css`** (the shared Carbon token set — also byte-identical to
  the converted `notify/share/www/static/tokens.css`, the stable on-`main`
  anchor). The `static/fonts/` woff2 set is already the shared set and is
  unchanged.

- The `GET /{$}` landing render (D16, through `rt.WWW()`), the chassis static
  mount, and the nginx fragment are **untouched** — this Decision changes only
  rendered markup and the token values.

**Relationship to D10.** D10 created prompts's landing page (its template + assets,
the handler, the route, the nginx fragment; D16 later moved the assets to
`share/www` and the render through the chassis) and owns its behavioral ids
`R-LAND-PG01`, `R-LAND-NMVR`, `R-LAND-CARB`, `R-LAND-ROOT`, `R-LAND-UNGT`. Those
behaviors are **preserved** — the page still renders the service name and version,
returns `200 text/html`, answers the exact root only, and is ungated in-process.
This Decision changes only the *styling and content* of that page, not its
behavior, so it mints **no new ids**; where a D10 test pins the prior markup's
literal text, that assertion is updated to cron's text while the behavioral
assertion stands.

**Rejected.**
- A shared cross-app landing handler — each app still owns its page; this is
  content/style convergence, not code sharing.
- Editing only the eyebrow/description on the existing divergent markup — leaves
  the structural and token differences that are the actual style violation.
- Embedding a frozen copy of cron's HTML in this Decision — would fork the
  canonical page into a drifting copy; pointing at the one cron file keeps a
  single source of truth.

**Verification.** Structural — no `R-XXXX-XXXX` ids (markup/asset content).
Realized by a plan phase whose done bar is a content check:

- `diff prompts/share/www/static/tokens.css notify/share/www/static/tokens.css`
  (run from the repo root) prints nothing — the shared Carbon token set, anchored
  to the converted, on-`main` `notify` tree so the check does not depend on any
  in-flight peer conversion.
- `prompts/share/www/landing.html` differs from cron's canonical landing markup
  on **only** the three lines in the table above — carrying `prompts`,
  `Agent sessions`, and the prompts lead paragraph respectively; all other lines
  are identical.
- The suite stays green: `cd prompts && go build ./...`, `go vet ./...`,
  `gofmt -l .` (prints nothing), and `go test ./...`.
