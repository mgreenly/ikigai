# Decision 19 — Progressive suite-tool discovery (deferred suite tools)

## Decision

The in-run agent no longer carries every suite tool's full definition on every round-trip. The suite's peer tools reach the run as agentkit **deferred tool groups** (research §2): the model sees a compact catalog — one section per service, its blurb, its bare tool names, rendered by agentkit into the built-in `load_tools` tool's description — and loads the specific tools it needs by exact name, on demand. Sandbox tools stay eager; **all** suite tools are deferred, no exceptions (an exception list is policy to maintain; loading is one cheap call). All deferral *semantics* — catalog rendering, `load_tools`, monotonic loading, the corrective error + load-on-error path for unloaded calls, ordering, validation — are agentkit's, specified and proven in agentkit's own project (D23); this Decision owns only the prompts-side wiring.

**Runner wiring** (the D7 conversation assembly, restated here as the owning decision):

```go
// Runner.discover changes type:
discover func(ctx context.Context, owner, promptID string) []agentkit.DeferredToolGroup

// In execute:
conv := &agentkit.Conversation{
    ...
    Tools:         runtools.All(sandboxRoot, sourcePortAllowed),  // eager: the run's core identity (seam per D5/D21)
    DeferredTools: r.discover(ctx, run.OwnerEmail, run.PromptID), // D6 groups
}
```

The default `discover` closure still wraps `suite.Discover(ctx, manifestRoot, owner, promptID)` (D6); the injectable seam is unchanged in shape, only in element type. An empty discovery result (no peers, unreadable inventory) means `DeferredTools` is empty and agentkit synthesizes no `load_tools` — a run on a bare box behaves exactly as today.

**Framing prompt**: `framing_prompt.go`'s `framingPrompt` carries one guidance paragraph after the sandbox-tools sentence, generic on purpose — it names the *mechanism* (`load_tools`, catalog in its description, load before calling, a service's name loads that service's whole group) and never enumerates services, because the catalog is generated per-box per-run from live discovery and hard-coded prose would drift:

> "Beyond the sandbox tools, this account's services are available as deferred tools. The `load_tools` tool's description catalogs every service and tool name; call `load_tools` with tool names — or a service's name to load all of its tools — to make them callable, then call them. Load a tool before calling it."

(Final wording may be polished at build time; the three fixed points are that it names `load_tools`, that it states a service/group name loads that whole group, and that it names no individual service.) The event preamble (`eventPreamble`) is unchanged — its "use the identifiers in `payload` with the suite tools" guidance composes with the paragraph above.

**Dependency pin**: `prompts/go.mod` pins `github.com/ikigenba/agentkit` at `v0.2.1` — the release that ships group-name loading in `load_tools` (agentkit's amended D23: a `load_tools` name failing tool-name resolution loads a matching group's whole tool set; observed live, a model's natural first call passes the group name). **External precondition** (owned outside `prompts/`, like the `registry` module precondition in Conventions): the `v0.2.1` tag is published by agentkit's phase 50. Local dev resolves agentkit through the repo-root `go.work` replace and does not wait on the tag; the production build (`GOWORK=off`) does.

## Rejected

**Stay eager (status quo)** — ~15–18k tokens of tool schemas resident in every round-trip of every run on a full box, paid whether or not a single suite tool is called, plus tool-choice dilution across ~120 tools (research §1).

**A prompts-side meta-dispatch surface (`suite_call(name, arguments)`) with no agentkit change** — the model would drive real tools through a stringly wrapper with no provider-native schema on the actual call; rejected during the cross-repo design split — the mechanism is provider-agnostic and belongs in agentkit (its D23), where it now lives.

**Keep a hand-picked eager subset (e.g. every peer's `health`, `notify` send)** — an exception list is policy that must live somewhere, be justified, and be maintained as services change; the uniform rule is also the only one the framing prompt can state honestly in one sentence.

**Enumerate the services in the framing prompt** — the inventory is per-box and per-run (peers come and go, discovery is best-effort); prose would drift from the live catalog it duplicates.

**Eagerly load the triggering service's tools on event-triggered runs** — special-casing with marginal benefit: the model reads the event, reads the catalog, and loads what it needs in one iteration anyway.
