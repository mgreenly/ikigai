# Decision 16 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** prompts' landing page and static assets are embedded in the binary
(`//go:embed landing.tmpl static` in `prompts/internal/web/efs.go`), the shape
D10 introduced. The suite is reverting that: web assets are release artifacts
that belong on disk — visible and diffable in the versioned release tree,
editable in dev without a rebuild, served by one chassis mechanism instead of
twelve drifting copies. crm proved the conversion and notify followed; prompts
converts its landing surface the same way. This Decision is the service-side
half of appkit's chassis design (its D5 www-root resolution, D6 `appkit/web`,
D7 `Spec.WWW` / `Router.WWW()` — consumed here as a fixed external contract).
The two-audiences gate (D10), the cron-canonical markup (D11), and the
self-served fonts (D13) are unchanged in content; only the *source and serving
mechanism* of the assets moves from the binary to the shipped `share` tier.

**Decision.** prompts serves its web surface from the on-disk www root.

1. **The files move (same bytes).** `prompts/internal/web/landing.tmpl` →
   `prompts/share/www/landing.html` (renamed `.tmpl`→`.html` to match the
   suite reference; the template body — markup, inline `<style>`, `{{.Service}}`
   / `{{.Version}}` fields — is byte-for-byte the same, and `appkit/web.Load`
   parses `*.html` and `*.tmpl` alike). `prompts/internal/web/static/` →
   `prompts/share/www/static/` (same bytes: `tokens.css`, the four woff2 fonts).
   `bin/ship` already bundles `prompts/share/` into the release `share/<version>`
   tier; on the box the chassis resolves the root to `share/current/www`, in dev
   to `./share/www` or the `PROMPTS_WWW_PATH` override.
2. **The Spec opts in.** `cmd/prompts/main.go` sets `WWW: true`. The chassis
   loads the site at serve (failing startup loudly if the root is missing),
   auto-mounts `GET /static/`, and exposes the loaded site as `rt.WWW()`.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `registerRoutes` becomes a few-line handler over
   `rt.WWW().Render(w, "landing.html", struct{ Service, Version string }{
   rt.Service(), rt.Version() })`. The service-side
   `rt.Handle("GET /static/", http.StripPrefix("/static/", web.StaticHandler()))`
   mount is **deleted** — the chassis owns `GET /static/`.
4. **`prompts/internal/web/` is deleted** — `web.go`, `efs.go`, `landing.tmpl`,
   `static/`, and `web_test.go`. The landing / asset / font / nginx-fragment
   tests move to the composition-root package (`cmd/prompts`), loading the
   repo-real `prompts/share/www` tree via `appkit/web` (a relative path from the
   package dir), so they exercise the exact files that ship. The retained ids
   (`R-LAND-PG01`, `R-LAND-NMVR`, `R-LAND-CARB`, `R-LAND-ROOT`, `R-LAND-UNGT` of
   D10; `R-HOME-2T4X` of D12; `R-DFKP-IVZU`, `R-DGSL-WNQJ`, `R-DI0I-AFH8`,
   `R-DJ8E-O77X`, `R-DKGB-1YYM` of D13) keep their assertions, re-proven on the
   new substrate. The cron-conform content check (D11) re-anchors to
   `prompts/share/www` paths.
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_prompts` exports
   `PROMPTS_WWW_PATH="$repo/prompts/share/www"` beside its existing exports, so
   a repo-root-launched dev binary finds the source tree. This one line crosses
   the `prompts/` boundary — the workspace's standing convention for the
   repo-root collaborator — and is verified by the live `bin/start` smoke, not
   the Go suite.
6. **Nothing else moves.** The nginx fragment (D10/D13 — the session-gated
   `= /srv/prompts/` root and `/srv/prompts/static/` locations, still carrying
   their own `127.0.0.1:3002` literal, which registry does not own, D14
   Rejected), the landing markup and tokens (D11/D13), the MCP surface, and the
   migrations are untouched. The embedded migrations (`internal/db`) stay
   embedded — they are not web assets.

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work exists to complete;
  embedded assets are invisible on the box and cost rebuilds in dev. D10's
  "embeds its own copy… so the binary stays one static file" rationale is
  superseded: the assets now ship in the `share` tier, still per-service and
  self-contained, just on disk.
- **`state/www` as the root** — that tree is the sites service's nginx-served
  user content, backed up with `state/`; prompts' UI assets are release
  artifacts and live in the shipped `share` tier (appkit D5 records the full
  argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a render closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition** — two sources
  of truth; the conversion is atomic within prompts.

**Verification** (over the repo-real `prompts/share/www` tree; D10's `R-LAND-*`,
D11's conform check, D12's `R-HOME-2T4X`, and D13's `R-D*-*` ids continue to pin
the page/asset behaviors through the new substrate).
- R-DIAW-ZFMC — an `appkit/web` Site loaded from `prompts/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html`
  with an injected service and version visible in the output — proving the
  shipped template, not an embedded copy, is the page source.
- R-DJIT-D7D1 — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  prompts-side static handler registered — proving the assets ship on disk and
  the chassis serves them.
