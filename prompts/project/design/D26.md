# Decision 26 — File-share sandbox tools: `File*` over the share's loopback filesystem API

> **External ordering (operator-sequenced, not built by any prompts phase).**
> This Decision consumes the **refined loopback mutation error contract** of
> dropbox's filesystem API — mutations return **404** for a missing referenced
> path and 4xx bodies carry the underlying domain error detail — specified in
> `dropbox/project/design/D16.md` (error contract slice) and realized by
> **dropbox plan phase 25, which the operator sequences before this Decision's
> phases**. The tools here are proven hermetically against a local `httptest`
> stand-in either way; the ordering is what makes the shipped `not_found:` /
> `validation:` discrimination true against the real service.

**Problem.** A run's sandbox folder is private to its prompt: nothing another
service or the owner can see, and nothing that triggers workflows. The
account's durable, shared file state lives in the file share (dropbox's local
mirror — the suite's filesystem authority). Today a run can only *read* share
bytes, and only URL-addressed via `Fetch` (D21); it has no way to list the
share, address a file by path, or — above all — **write results back** where
the owner and the event plane see them. The bills workflow needs exactly that:
a run triggered by a statement PDF arriving in a share folder must land its
extraction artifacts durably beside the PDF. The share's MCP surface is the
wrong channel (base64 crossings the content plane retires, 25 MiB caps,
identity-gated); the right channel is the share's **loopback filesystem API**
(`dropbox/docs/filesystem-api.md`), moving bytes disk-to-disk.

**Decision.** Six sandbox tools in `prompts/internal/tools/` — `FileList`,
`FileGet`, `FilePut`, `FileDelete`, `FileMove`, `FileMkdir` (D5 owns the
toolset table and count; this Decision owns their contract) — thin, streaming
wrappers over the share's loopback filesystem API, located by the
`ShareConfig` seam (D5): `BaseURL` is the registry-defaulted
`DROPBOX_BASE_URL` the composition root already resolves, `ClientID` is
`"prompts:" + <prompt id>` (the exact `suite.go` convention). **Everything the
model sees is service-agnostic**: tool names, descriptions, results, and
framing speak of "the account's file share", never of dropbox — the D19/D21
no-individual-service constraint (R-9OJA-B2KU, R-6AUG-NHQY) holds unchanged.

**Input shapes and route mapping** (query params URL-escaped; share paths are
`/`-rooted mirror display paths, validated by the share itself — full share
access, no prompts-side scoping):

```go
// prompts/internal/tools — beside the seven existing tools.
type fileListInput struct {
    Path   string `json:"path,omitempty" jsonschema:"description=Share folder to list (default: the share root)"`
    Cursor string `json:"cursor,omitempty" jsonschema:"description=Continuation cursor from a previous FileList result"`
    Limit  int    `json:"limit,omitempty" jsonschema:"description=Maximum entries to return"`
}
type fileGetInput struct {
    SharePath string `json:"share_path" jsonschema:"required,description=Path of the share file to copy"`
    DestPath  string `json:"dest_path" jsonschema:"required,description=Destination file path, relative to the sandbox root"`
}
type filePutInput struct {
    SourcePath string `json:"source_path" jsonschema:"required,description=Sandbox file to copy, relative to the sandbox root"`
    SharePath  string `json:"share_path" jsonschema:"required,description=Destination path in the share (overwrites)"`
}
type fileDeleteInput struct {
    SharePath string `json:"share_path" jsonschema:"required,description=Share file or folder to delete"`
}
type fileMoveInput struct {
    From string `json:"from" jsonschema:"required,description=Current share path"`
    To   string `json:"to" jsonschema:"required,description=New share path"`
}
type fileMkdirInput struct {
    SharePath string `json:"share_path" jsonschema:"required,description=Share folder to create"`
}
```

| tool | route | result (the small JSON the model sees) |
|---|---|---|
| `FileList` | `GET /list?path=&cursor=&limit=` | the share's entries (`path`, `kind`, and for files `size`/`content_hash`/`rev`/updated time) + continuation cursor, passed through |
| `FileGet` | `GET /content?path=` | `{"path": <dest_path>, "size": <bytes written>, "content_hash": <64-hex sha256>}` |
| `FilePut` | `PUT /content?path=` (body streamed from the sandbox file) | the share's response verbatim: `{path, size, content_hash, rev}` |
| `FileDelete` | `DELETE /content?path=` | `{"deleted": <share_path>}` |
| `FileMove` | `POST /move?from=&to=` | `{"moved": <from>, "to": <to>}` |
| `FileMkdir` | `POST /mkdir?path=` | `{"created": <share_path>}` |

- **Disk-to-disk, never through context.** `FileGet` streams the response body
  into the sandbox exactly as `Fetch` does (temp file + SHA-256 on the same
  pass + rename — a torn stream leaves no partial file at `dest_path`; no size
  cap). `FilePut` streams the sandbox file as the request body (no size cap).
  `FileGet`'s `content_hash` is prompts' own plain SHA-256 of the bytes it
  wrote (the D21 rule); `FilePut`'s result is the share's own metadata, its
  `content_hash` an opaque share-side handle reported as received.
- **Sandbox confinement.** `FileGet.dest_path` and `FilePut.source_path`
  resolve through D5's shared sandbox confinement; an escaping path is
  rejected `validation:` **before** any HTTP call. A `FilePut.source_path`
  that is inside the sandbox but absent is `not_found:` with zero HTTP calls.
- **Caller identity.** Every request carries `X-Client-Id: prompts:<prompt
  id>` and **no** nginx identity headers — in particular no `X-Forwarded-Proto`,
  the header the share's loopback self-guard keys on to 404 front-door requests;
  the share stamps that client id as the emitted event's `origin`.
- **Pinned failure mapping** (the D21 taxonomy, derived mechanically from the
  HTTP status so share-side refinements flow through with no prompts change):
  client-side rejection or HTTP **400** → `validation: <detail>` (the response
  body's text included); **404** → `not_found: …` ("the path is stale or
  absent — re-derive it, e.g. with FileList"); **409** → `conflict: …`;
  connection refused, timeout, any other status, or a torn body →
  `source_unavailable: …` (transient, retryable). In every `FileGet` failure
  no file exists at `dest_path`.
- **Writes are ordinary file events (design fact).** A `FilePut`/`FileDelete`/
  `FileMove` commits the share's local state and emits the share's ordinary
  `create`/`modify`/`delete` events (origin `prompts:<prompt id>`), then syncs
  outward asynchronously — a run-written file triggers watching workflows
  exactly like an owner-written file. Loop safety (a run re-triggering itself
  by writing into its own watched folder) is **deliberately not a mechanism**:
  trigger authors pin their filters (e.g. `dropbox:create/statements/x/*.pdf`
  never matches a written `.extraction.json`), and legitimate run-to-run
  chaining through the share stays possible.
- **Framing prompt.** The tool sentence grows the six file tools, and one
  guidance paragraph states the share model: the account's **file share** is
  its durable, shared file store — files there persist, sync, and are what the
  owner and other workflows see; the run's own folder stays private to this
  prompt; the file tools are the channel between the two. The "NO network
  access from bash" sentence stays verbatim (these are runner-executed tools,
  like `fetch`). The paragraph names **no individual service** (composes with
  R-9OJA-B2KU / R-6AUG-NHQY; final wording polished at build time — the fixed
  points are the ones this paragraph states).

**Rejected.**
- **The share's MCP surface (`load_tools` → `put`/`get`)** — base64 crossings
  through model context with a 25 MiB cap; exactly what the content plane
  retires. The loopback byte API is the sanctioned disk-to-disk channel.
- **Naming the service (`DropboxPut`, "dropbox" in framing)** — the runner
  doesn't need the backing service's name, the framing constraint (R-9OJA-B2KU)
  would need amending, and the surface would break if the share's backing ever
  changed. "File share" is the durable concept.
- **Per-prompt/per-run share scoping** — the box is single-tenant and a run
  already holds the owner's full authority through the suite tools (it could
  reach the share's MCP `put` regardless); a prompts-side fence restricts
  nothing real and costs a policy surface. Same posture as the share's own
  flat-namespace decision.
- **Suppressing self-originated trigger events** — origin is payload, not a
  routing dimension; a suppression mechanism would foreclose legitimate
  A-writes-triggers-B chains. Filter discipline is expressive and free.
- **A `FileStat` tool / rev-pinned `FileGet`** — `FileList` answers metadata
  questions and event payloads already carry current metadata; minimal shapes
  win.
- **Extending `Fetch` with write/list verbs** — `Fetch` is the URL-addressed
  content-plane acceptor with its own pinned contract (D21); overloading it
  with path-addressed share semantics muddles both.

**Verification** (real substrates: every claim runs against a **real local
HTTP server** — `httptest` on loopback standing in for the share's filesystem
API, recording method/path/query/headers/body — with a temp-dir sandbox;
rejection claims assert the server recorded **zero** requests).

- R-F74Y-B8X1 — every request each of the six tools issues carries
  `X-Client-Id` exactly `prompts:<prompt id>` and carries **no**
  `X-Owner-Email` and no `X-Forwarded-Proto` header (recorded across all six
  tools' happy paths).
- R-F8CU-P0NQ — `FileGet` of a share path the server serves with known bytes
  streams them **server-side** into the sandbox at `dest_path` (file bytes
  equal served bytes; the server records exactly one `GET /content` with the
  URL-escaped `path` query) and returns `{path, size, content_hash}` with the
  lowercase-hex SHA-256 of those bytes — while an escaping `dest_path`
  (`../outside.bin`) is rejected `validation:` with zero requests and no file
  created anywhere.
- R-F9KR-2SEF — `FilePut` of an existing sandbox file issues exactly one
  `PUT /content` whose body bytes equal the sandbox file's bytes and whose
  `path` query is the URL-escaped `share_path`, and returns the server's
  `{path, size, content_hash, rev}` fields verbatim — while an escaping
  `source_path` is `validation:` and an in-sandbox but absent `source_path` is
  `not_found:`, each with **zero** requests.
- R-FASN-GK54 — `FileList` issues `GET /list` passing `path`, `cursor`, and
  `limit` through as query params and returns the server's entries and
  continuation cursor in its result (entries' `path`/`kind`/`size` observable
  in the returned JSON).
- R-FC0J-UBVT — `FileDelete` issues exactly one `DELETE /content?path=`,
  `FileMkdir` exactly one `POST /mkdir?path=`, and `FileMove` exactly one
  `POST /move?from=&to=` (params URL-escaped), each returning its pinned small
  success JSON.
- R-FD8G-83MI — the pinned failure mapping against the real local server: a
  **400** response with a detail body → tool error text beginning
  `validation:` and containing that body's detail; a **404** → `not_found:`; a
  **409** → `conflict:`; a connection-refused base URL →
  `source_unavailable:` — and after a failed `FileGet` no file exists at
  `dest_path`.
- R-FEGC-LVD7 — the framing prompt sent as the conversation `System` names the
  six file tools, states the file-share guidance (durable shared store vs the
  prompt-private folder; the file tools as the channel), still contains the
  verbatim "NO network access from bash" sentence, and contains no individual
  service name (no `ikigenba_`, no per-service enumeration — composes with
  R-9OJA-B2KU and R-6AUG-NHQY).
