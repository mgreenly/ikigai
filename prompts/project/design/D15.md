# Decision 15 — Consumer loops through `Spec.Consumers` (chassis-owned)

**Problem.** prompts hand-writes its six event-plane consumer loops at the
composition root. `main()` builds a `workers []func(context.Context) error`
slice — one closure per `sources` entry — each calling `runConsumer(ctx, rt,
src)`, a ~25-line block that resolves `PROMPTS_<SRC>_FEED_URL` /
`PROMPTS_<SRC>_FROM`, assembles `consumer.Config`, logs the start line, and
calls `consumer.Run`, wrapping the error. Those closures reach the finished
Router through a `var rt *appkit.Router` capture that `Handlers` smuggles out
(`rt = r`). The consumer role is declared in three Spec places that must
manually agree: `Consumes: sources` (manifest `CONSUMES=`), `Subscriptions:
func(){ consume.Subscriptions(sources) }` (reflection), and the `Workers` loops
(the actual `consumer.Run` calls). The chassis now owns all of this (appkit
design D10, consumed here as a fixed external contract); prompts is the
multi-upstream conversion (six sources, self-chaining included) beside notify's
two-source pilot.

Unlike notify, prompts' consumer handler depends on the domain `prompt.Service`
built inside `Handlers`/`registerRoutes` (its `RunByEvent` fires runs and its
`PromptsForEvent` fans an event out to matching prompts). That service is
already handed from `Handlers` to the loops through the package-level `svcRef`
capture — the same shape the `storeRef` capture uses to hand the `Store` to the
producer `Producer` hook. `svcRef` therefore stays; only the `Router`-smuggling
`var rt` capture and the hand-rolled loop wiring go.

**Decision.** prompts declares its consumer role as the `Spec.Consumers` table
(six entries in the fixed `sources` order) and deletes its hand-rolled wiring:

```go
// cmd/prompts — the whole consumer declaration, built once from `sources`.
Consumers: func() []appkit.Consumer {
    cs := make([]appkit.Consumer, 0, len(sources))
    for _, src := range sources {
        src := src
        cs = append(cs, appkit.Consumer{
            Source:        src,
            Subscriptions: consume.Subscriptions([]string{src}), // this source's one broad in-edge
            Handler: func(rt *appkit.Router) consumer.Handler {
                logger := rt.Logger()
                fire := func(ctx context.Context, promptID, s, evType, eventID string, payload []byte) error {
                    _, err := svcRef.RunByEvent(ctx, promptID, s, evType, eventID, payload)
                    return err
                }
                return consume.Handler(fire, svcRef.PromptsForEvent, src, logger)
            },
        })
    }
    return cs
}(),
```

- **Deleted:** `runConsumer`, `feedURLEnv`, `fromEnv`, the package-level
  `feedDefaults` map, the `workers` slice construction in `main()`, the `var rt
  *appkit.Router` capture (and the `rt = r` assignment in `Handlers`, now
  `Handlers: func(r *appkit.Router) error { return registerRoutes(r) }`), and
  the legacy `Consumes` and `Subscriptions` Spec fields (both derived from the
  table by the chassis — setting either alongside `Consumers` is a chassis
  startup error). The `strings` import (used only by `feedURLEnv`/`fromEnv`) and
  the `config` import's feed-URL uses go with them; `config` stays for
  `registerRoutes`' other env reads.
- **Kept:** `sources` (still the single ordered list the table is built from and
  the manifest mirrors), the `consume.Subscriptions` / `consume.Handler`
  functions and their `RunByEvent`/`PromptsForEvent` fan-out unchanged, the
  `svcRef` capture (the domain `prompt.Service` is built in `Handlers` and
  consumed by the factories, which the chassis calls strictly afterward — the
  same lifetime `storeRef` already relies on for the producer seam), and the
  **entire producer half** (`Feed: "/feed"`, `Events: prompt.Events`,
  `Producer`, `ManifestExtras`, `storeRef`), which is orthogonal to `Consumers`
  and untouched.
- **Behavior is unchanged:** same six upstreams in the same order
  (`cron`, `crm`, `ledger`, `dropbox`, `scripts`, `prompts`), same broad
  everything-from-this-upstream subscription per source (`"**"` under the
  routing-key dialect, D24), same fire-and-run fan-out, same `feed_offset`
  cursors (`consumer_id` `"prompts"` == `Spec.App`, one row per `Source` — the
  chassis derives the same values `runConsumer` hardcoded via `consumerID`), and
  the manifest still says `CONSUMES=cron,crm,ledger,dropbox,scripts,prompts`
  byte-for-byte (the committed-manifest byte-equality guard `R-8IAN-FB87` keeps
  pinning it).
- **Env names already conform.** prompts already reads
  `PROMPTS_<SRC>_FEED_URL` / `PROMPTS_<SRC>_FROM` (the chassis convention) and
  defaults each feed URL to `registry.BaseURL(src) + "/feed"` and each `From`
  to `"tail"` — exactly what the chassis resolves. So there is **no env
  rename**, **no `.envrc` change** (prompts' `.envrc` exports no feed
  variables), and **no `bin/start` change** for this Decision:
  `launch_prompts` already exports the generic
  `PROMPTS_<SRC>_FEED_URL` form through its `export_feed_urls` helper, and the
  end-to-end boot test (`R-4LKF-FB23`) drives all six of them. Prod sets none of
  these variables, so no box change.

**Interaction with D14 (registry).** The deleted `feedDefaults` map was the
subject of D14's `R-RG02-FEED`. That per-composition-root default resolution now
lives in the chassis (appkit D10 resolves `<APP>_<SRC>_FEED_URL` →
`registry.BaseURL(src) + "/feed"`, pinned by appkit's `R-464U-T3T1`), so
`R-RG02-FEED` **retires with its behavior** and its test
(`TestFeedDefaultsUseRegistryForEverySource` in `registry_test.go`) is deleted
along with the map it asserted. D14 is edited in place to record this; its
other three ids (`R-RG01-PORT` own port, `R-RG03-DBOX` dropbox base, `R-RG04-NLIT`
no-literal guard) are unaffected — the no-literal guard in fact stays green,
since the chassis (a different module, not walked by the guard) now owns the
`registry.BaseURL` call the feed defaults used.

**Rejected.**
- **Rebuilding the domain `prompt.Service` inside each handler factory** (the
  notify shape, where the factory rebuilt its cheap `push.Client` from config) —
  prompts' service owns the store, sandbox tree, runner, and crash-recovery
  sweep built once in `Handlers`; reconstructing it per factory would duplicate
  that expensive setup and fork run state. Closing over the `svcRef` the
  producer seam already uses is the honest shape; the factory takes `rt` only
  for its logger.
- **Keeping the `Workers` slice with an eventplane helper** — folds the Config
  assembly but keeps the `var rt` capture and the three-field agreement across
  `Consumes`/`Subscriptions`/`Workers`; the declared table removes both.
- **One `Consumer` entry with all six subscriptions** — the chassis keys one
  `consumer.Run` cursor per entry `Source`, so the six independent per-upstream
  cursors require six entries; collapsing them would share one cursor across
  all feeds.
- **Dropping the env overrides now that the chassis defaults them** —
  `PROMPTS_<SRC>_FEED_URL`/`_FROM` are the deploy/test seam (idle feed servers
  in `R-4LKF-FB23`); the chassis keeps the override layer, only owning the
  fallback.

**Verification** (the Spec value directly, and the handler-factory behavior over
the real `prompt.Service` + `Store` + temp SQLite substrate the `consume`
smoke test already uses).
- R-DFV4-7W4Y — prompts' Spec declares **exactly six** `Consumers` entries, in
  order `cron`, `crm`, `ledger`, `dropbox`, `scripts`, `prompts`, each entry's
  `Subscriptions` equal to `consume.Subscriptions([]string{Source})` (the one
  broad `Source`/`"**"` in-edge for that upstream, D24), with **no** legacy `Consumes`
  or `Subscriptions` Spec field set — a dropped upstream, a swapped order, a
  diverged filter, or a lingering legacy field fails this.
- R-DH30-LNVN — each entry's `Handler` factory, invoked with a Router whose DB
  and `svcRef` are the post-`Handlers` real service, returns a non-nil
  `consumer.Handler` that, fed a well-formed event of that entry's `Source` whose
  canonical key a prompt trigger filter matches (D24), fires exactly one run
  (observed through the real `prompt.Service` + stubbed runner), and fed a
  well-formed event no trigger matches fires none — the same fan-out the
  pre-conversion `runConsumer` wired.
