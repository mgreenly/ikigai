# Decision 25 — Event-routing conformance: producer kinds `run.succeeded`/`run.failed`, subject = /<prompt name>

> **External ordering (operator-sequenced, not built by any prompts phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before this Decision's phase runs**. It equally assumes appkit (which holds
> `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, this Decision's phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject` and producer registries become **families**. Hard
cutover, no compatibility period, inside the no-live-data migration window.

**What prompts publishes today (ground truth, not vacuous).** prompts is a
real, wired producer: `Spec.Feed: "/feed"`, `Spec.Events: prompt.Events` (an
`outbox.Registry` of two `EventType`s in `internal/prompt/outcome.go`), the
`Producer` hook injecting the outbox onto the `Store`, and one emission path —
`FinishRun` → `outcomeEvent` → `outbox.Append(tx, Event{Type, Payload})` on the
**same transaction** as the run's terminal-state write (at-most-once per run; a
cancelled run emits nothing). Both published types
(`run.succeeded`/`run.failed`) convert.

**Decision.** prompts adopts the revised envelope exactly over what it
publishes, per the routing doc's producer key map — **the prompt's name as the
subject**:

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `run.succeeded` | `prompts:run.succeeded/collect-bills` | `run.succeeded` | `/` + the prompt's name |
| `run.failed` | `prompts:run.failed/collect-bills` | `run.failed` | `/` + the prompt's name |

The kinds keep their dotted names — `run` is prompts' domain *entity*, not a
redundant restatement of the source (unlike `mail.`/`file.`/`transaction.`),
and the routing doc's key map records `prompts:run.succeeded/<prompt name>`
verbatim. The **subject** is `/` + the run's denormalized `prompt_name` — the
routing dimension self-chaining consumers want
(`prompts:run.succeeded/collect-bills` fires exactly one downstream prompt).
Names are free text, so the derivation is pinned: subject is `""` when the
prompt has no name; otherwise `"/" + prompt_name` with any newline replaced by
a space (`routing.ValidSubject` requires a single-line `/`-rooted path; a name
containing `/` simply yields extra routing segments, which is fine — the
subject is a name, and the payload still carries the exact `prompt_name`).

1. **Constants, payload, and event builder** (`internal/prompt/outcome.go`):
   `EventRunSucceeded`/`EventRunFailed` keep their string values, now as
   **kind** constants; `outcomeEvent` derives the subject per the rule above
   and returns `outbox.Event{Kind, Subject, Payload}`. The `outcomePayload`
   follows D24's run-row reshape: `trigger_type` is replaced by
   `trigger_kind` + `trigger_subject` (JSON keys), everything else —
   `prompt_id`, `prompt_name`, `run_id`, `trigger_source`,
   `trigger_event_id`, `error` (failed only) — unchanged. `FinishRun` reads
   the reshaped columns; the same-tx atomicity, the at-most-once rule, and
   the cancelled-emits-nothing rule are untouched.

2. **The registry becomes families** (`internal/prompt/outcome.go`):
   `prompt.Events` becomes an `outbox.Registry` of two `outbox.Family`
   entries — `{Kind: "run.succeeded", Subject: "/<prompt name>",
   Description: …, Sample: sampleOutcomeSuccess}` and likewise `run.failed`
   (Samples updated to the reshaped payload, their `trigger_kind`/
   `trigger_subject` carrying revised-model values, e.g. kind `tick`, subject
   `/nightly`). `Spec.Events` wiring is untouched; Append-time kind gating and
   the chassis reflection surface hang off the same registry, now
   family-shaped.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. prompts appends **one
   new timestamped migration** minted with `bin/create-migration prompts
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across.
   The frozen `005_outbox.sql` is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest**
   outbox migration, asserting it contains the revised `outbox.SchemaSQL`
   verbatim. (This exactly mirrors gmail's D18/phase-16, dropbox's
   D22/phase-23, and ledger's D15/phase-13 handling.)

4. **Reflection/consumer surface.** The chassis reflection tool reflects the
   family registry: prompts' two kinds — and only they — are listed with the
   `/<prompt name>` subject description, beside the D24 `"**"` subscribes
   (D17's R-DLYM-4QUF, conformed in place). The consumer half is D24's;
   nothing here touches it.

**Rejected.**
- **Renaming the kinds to `succeeded`/`failed` (scripts' shape)** — `run` is
  the domain entity, not the source noun; bare `succeeded` invites collision
  with a future non-run outcome and diverges from the routing doc's recorded
  key map for prompts.
- **Subject = `/<prompt_id>`** — nobody routes on a ULID; the name is the
  human routing handle, the id stays in the payload for machines.
- **An empty subject (gmail/ledger's choice)** — prompts has an identified
  routing dimension *today*: self-chaining triggers want "when *that* prompt
  succeeds", which the name gives and emptiness forecloses.
- **Rejecting/renaming prompts whose names make awkward subjects** — the
  create surface is out of scope and a name is a name; the pinned derivation
  (newline → space) keeps `Append` valid without constraining users.
- **Editing `005_outbox.sql` in place / an `ALTER TABLE` preserving rows** —
  committed migrations are immutable, and preserving ≤7-day retention rows
  inside the migration window is pointless ceremony that would fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: emission claims run over a real
SQLite database migrated through prompts' full embedded migration set, outbox
rows read back by SQL; the feed claim runs the real eventplane `FeedHandler`
over `httptest`).

- R-6T4Y-E1VD — after `FinishRun` lands a succeeded run of a prompt named
  `collect-bills`, a failed run of the same prompt, and a cancelled run: the
  real SQLite outbox holds exactly two rows, with `kind` exactly
  `run.succeeded` and `run.failed`, each `subject` exactly `/collect-bills`,
  each appended on the same tx as the terminal-state write; the failed row's
  payload carries `error` and both payloads carry `trigger_kind`/
  `trigger_subject` keys and **no** `trigger_type` key; a run of a
  **nameless** prompt yields `subject = ''`.
- R-6UCU-RTM2 — `prompt.Events` declares exactly two families with kinds
  `run.succeeded`/`run.failed` and subject description `/<prompt name>`;
  through the assembled MCP handler, reflection's index lists exactly those
  two publishes, the detail of `run.succeeded` returns a schema and example
  that agree (every example field in the schema, `outcomePayload`-shaped,
  `trigger_kind`/`trigger_subject` present, no `trigger_type`), and an
  undeclared kind returns the typed unknown-kind error naming the two
  declared kinds.
- R-6VKR-5LCR — applying prompts' full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table
  with `kind` and `subject` columns and **no** `type` column; the DDL drift
  guard asserts the newest outbox migration contains the revised
  `outbox.SchemaSQL` verbatim while `005_outbox.sql` remains byte-identical
  to its committed (frozen) body.
- R-ZS8A-TVOF — end-to-end over prompts' own substrate: an event appended
  through prompts' producer path (`FinishRun` → `outcomeEvent` → `Append` on
  the migration-created table, source `prompts` — the `Spec.App` value) for a
  succeeded run of prompt `collect-bills` and served by the real eventplane
  `FeedHandler` over an `httptest` connection frames
  `event: prompts:run.succeeded/collect-bills` (the canonical key) with a
  `data:` JSON envelope carrying `kind` and `subject` keys and **no** `type`
  key.
