# Decision 24 — Event-routing conformance: triggers become canonical filter strings (trigger surface + consumer)

> **External ordering (operator-sequenced, not built by any prompts phase).**
> This Decision consumes the **revised eventplane API** — the kind/subject
> envelope, `routing.Key`/`routing.Match`/`routing.ValidKind`/
> `routing.ValidSubject`, `outbox.Family`/`Registry.CouldMatch`, and
> `consumer.Event{Kind, Subject}` + `Event.Key()` — specified in
> `eventplane/project/design/` (its D1–D4) and realized by **eventplane plan
> phases 01–04, which must be BUILT before this Decision's phase runs**. It
> equally assumes appkit (the chassis consumer engine, `Spec.Events
> outbox.Registry`, the reflection tool) compiles against that revision —
> appkit's own conformance is appkit's spec, out of scope here. Until both
> land, this Decision's phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): events are addressed by a canonical routing
key `source:kind<subject>` matched with one shared glob matcher. prompts is the
suite's **trigger surface** and its biggest consumer, and everything it does
today is built on the old model (ground truth): a trigger is a
`(prompt_id, source, event_filter)` row (`prompt_triggers`, migration 006);
`validateTrigger` checks the pair against `knownProducers` — a static
per-source list of concrete `type` strings — with a hand-carved **cron special
case** (`cron.*` accepted blind); `PromptsForEvent(source, evType)` matches
filters against the flat type with `path.Match`; `consume.Handler` reads
`ev.Type`; the pinned `input/event.json` envelope is
`{source, type, event_id, payload}` (`eventEnvelope` in `service.go`); the run
row persists `trigger_source`/`trigger_type`/`trigger_event_id`. A prompt that
cares about `/bills/**` must today fire on **every** dropbox file event and
burn a full agent run discovering it wasn't for it — the failure the revision
exists to fix. Hard cutover, no compatibility period, inside the no-live-data
migration window (trigger rows are explicitly wipeable per the routing doc).

**Decision.** A trigger becomes **`(prompt_id, filter)`**, where `filter` is a
single canonical-key glob matched by `eventplane/routing.Match` against an
event's `routing.Key(source, kind, subject)` — e.g.
`dropbox:create/bills/**/*.pdf`, `cron:tick/bill-sweep`,
`prompts:run.succeeded/collect-bills`. The `(source, event_filter)` pair, the
static per-source **type** vocabulary, and the cron special case are all
replaced.

1. **Filter well-formedness: the source segment is literal.** A valid filter
   contains a `:`, and the segment before the first `:` must be a **non-empty
   literal source name** — no glob metacharacters (`*`, `?`, `[`) in the
   source segment. Globs in the source segment are **rejected**: a trigger's
   binding is consumed per-upstream (one `Spec.Consumers` entry per source,
   one feed, one cursor), so a multi-source filter could never be honored by
   the delivery machinery and would silently under-fire. The literal source
   must be one of the six consumed `sources`
   (`cron`, `crm`, `ledger`, `dropbox`, `scripts`, `prompts`); anything else
   is rejected naming the known set (today's unknown-source behavior,
   preserved).

2. **Family validation: a static kinds table + `Registry.CouldMatch`
   semantics.** prompts holds per-source family knowledge the way it holds
   `knownProducers` today — a **static table of kinds**, subjects open:

   ```go
   // internal/prompt/trigger.go — replaces knownProducers. Kinds follow the
   // routing doc's producer key map; each producer's own spec loop owns its
   // final choice, and this table tracks it exactly as the old type table did.
   var knownFamilies = map[string][]string{
       "cron":    {"tick"},
       "crm":     {"contact.created", "contact.updated", "contact.tagged", "contact.untagged"},
       "ledger":  {"recorded"},
       "dropbox": {"create", "modify", "delete"},
       "scripts": {"succeeded", "failed"},
       "prompts": {"run.succeeded", "run.failed"}, // self-chaining: prompts' OWN feed
   }
   ```

   `validateTrigger(filter)` (after well-formedness) builds an
   `outbox.Registry` of `Family{Kind: k}` values from the source's kinds and
   asks `Registry.CouldMatch(source, filter)` — accepted iff the filter could
   match some family's kind with **some** subject (subject left open, the
   eventplane D3 contract). This deletes the cron special case: cron is an
   ordinary producer whose kind is `tick` and whose subjects are dynamic —
   `cron:tick/anything` validates, exactly like `dropbox:create/anything`.
   A malformed glob surfaces `routing.Match`'s error as `ErrValidation`.
   **Live reflection queries are rejected** (the alternative): validating a
   trigger by calling each peer's `reflection` at `set_trigger` time makes a
   write verb depend on every upstream being up, adds an HTTP client prompts'
   domain layer doesn't have, and buys only earlier detection of the same
   drift the static table carries today — the architecture keeps the static
   copy and its known, accepted drift cost.

3. **Storage: one new timestamped migration** minted with
   `bin/create-migration prompts trigger_filters` (never a hand-picked
   number). Inside the migration window it **drops and recreates**
   `prompt_triggers` — columns `(prompt_id, source, filter, created_at)`,
   `PRIMARY KEY (prompt_id, filter)`, index on `source` — carrying no rows
   across (old `(source, event_filter)` pairs are not mechanically
   convertible; trigger rows are wipeable per the routing doc). `source` is
   **derived** from the filter's literal source segment at `SetTrigger` time
   and stored so `PromptsForEvent` keeps its per-source indexed lookup. The
   same migration reshapes the **runs** trigger context: `trigger_type`
   becomes `trigger_kind` + `trigger_subject` (rebuild or ALTER — builder's
   choice; run rows are migration-window state). The frozen
   `006_prompt_redesign.sql` is **not touched**.

4. **Store + service surface** (`trigger.go`, `service.go`, `model.go`):
   `Trigger{PromptID, Source, Filter, CreatedAt}` (Source derived, Filter the
   canonical glob); `SetTrigger`/`ClearTrigger`/`ListTriggers`/
   `DeleteTriggers` keyed by `(prompt_id, filter)`, same upsert/ErrNotFound/
   tombstone semantics as today. `PromptsForEvent(ctx, source, key string)`
   selects the source's rows and matches each stored filter against the
   delivered event's **canonical key** with `routing.Match` (a malformed
   stored filter matches nothing — defense in depth; writes are validated).
   The local `globMatch`/`path.Match` helper is deleted — the shared matcher
   is the only matcher. `Run{TriggerKind, TriggerSubject}` replace
   `TriggerType` (JSON `trigger_kind`/`trigger_subject`); `RunByEvent(ctx,
   id, source, kind, subject, eventID string, payload []byte)` and `startRun`
   thread both; `FinishRun`'s outcome read follows (payload shape is D25's).
   `TriggerSources()` keeps returning the static source set, now the
   `knownFamilies` keys.

5. **Consumer side** (`consume.go`): `consumer.Event` now carries
   `Kind`/`Subject` and `Key()` (eventplane D4). The poison check becomes
   `ev.Kind == "" || ev.ID == ""` → `ErrSkip`; the lookup becomes
   `lookup(ctx, source, ev.Key())` (`LookupFunc` takes the key); the fire
   becomes `fire(ctx, promptID, source, ev.Kind, ev.Subject, ev.ID,
   ev.Payload)`. `Subscriptions` stays one broad in-edge per source, but the
   filter string becomes **`"**"`** — under the doublestar dialect `*` never
   crosses `/`, so the honest "everything from this upstream" glob is `**`
   (the old `"*"` would misdescribe the in-edge for any subject-bearing key).
   Fire-and-forget semantics (never stall, matched-zero is success, unbounded
   goroutine fan-out) are unchanged.

6. **The pinned envelope + MCP surface.** `eventEnvelope` becomes
   `{source, kind, subject, event_id, payload}` — `type` is replaced by
   `kind` + `subject`, all else unchanged; that is what
   `input/event.json` carries and what the run's second user block renders.
   The runner's `eventPreamble` needs **no** change (grounded: it names no
   envelope field except `payload`; its guidance composes unchanged).
   `set_trigger` becomes `(prompt_id, filter)`, `clear_trigger`
   `(prompt_id, filter)`, and `create`'s inline `triggers` array becomes an
   array of filter **strings**; tool descriptions and the `describe` text
   state the canonical-key contract (`source:kind<subject>` examples, the
   doublestar dialect, the new envelope shape) in place of the
   `(source, event_filter)` prose. The error mapping is preserved: an invalid
   filter or unknown source is a domain error → `isError` content envelope
   (D17's mapping), never a JSON-RPC error.

**Rejected.**
- **Keeping `(source, filter)` as two arguments with the filter matching
  `kind<subject>`** — the canonical key already carries the source, and the
  routing doc's principle is one string, one matcher; two fields re-create the
  seam the revision deletes (and every example in the suite's docs is written
  as a whole-key glob).
- **Glob in the source segment (`*:create/**`)** — subscriptions are
  per-upstream; a multi-source trigger would need delivery machinery that
  doesn't exist and would silently miss sources added later. A user who wants
  two sources sets two triggers.
- **Live reflection validation** — rejected in §2 above (availability
  coupling, an HTTP dependency in the domain layer, same drift exposure).
- **No validation beyond well-formedness** — an unsatisfiable trigger
  (`dropbox:nosuchkind/**`) would sit silent forever; today's
  reject-unsatisfiable posture is kept, now via the eventplane-owned
  existential check instead of a local type-list scan.
- **Migrating old trigger rows into filters** — `(source, event_filter)` with
  old type names cannot be mechanically rewritten into kind/subject globs
  (the old filters matched types, not subjects); the migration window
  sanctions the wipe, and hand-set triggers are re-set in seconds.
- **Keeping `path.Match` for stored filters** — the one-matcher principle;
  `path.Match` cannot express `**` and its dialect would diverge from every
  other selector in the suite.

**Verification** (real substrates throughout: migration claims apply prompts'
full embedded migration set through the appkit runner to a fresh real SQLite
database; matching claims run the real `eventplane/routing` matcher; store
claims run real SQLite; the consume claim uses the real `prompt.Service` +
`Store` + stubbed runner substrate D15's smoke test already uses; MCP claims
drive the assembled `appkit/mcp` handler).

- R-6JDR-BVXT — applying the full embedded migration set to a fresh real
  SQLite database yields a `prompt_triggers` table with columns exactly
  `prompt_id`, `source`, `filter`, `created_at` (no `event_filter` column) and
  a `runs` table with `trigger_kind` and `trigger_subject` columns and **no**
  `trigger_type` column, while `006_prompt_redesign.sql` remains
  byte-identical to its committed (frozen) body.
- R-6KLN-PNOI — well-formedness discriminates: `create/bills/**` (no `:`),
  `:create/**` (empty source), `*:create/**` and `drop*:create/**` (glob in
  the source segment), and `github:push/**` (unknown source) are each rejected
  with `ErrValidation` (the unknown-source error naming the six known
  sources); `dropbox:create/bills/**` passes well-formedness.
- R-6LTK-3FF7 — family validation via the real eventplane check:
  `dropbox:create/bills/**/*.pdf` and `dropbox:*` are accepted;
  `dropbox:nosuchkind/**` is rejected; `cron:tick/some-schedule-nobody-declared`
  is accepted (subject open — the cron special case is gone, no `cron.*`
  carve-out remains in source); a malformed glob (`dropbox:create/[`) is
  rejected with `ErrValidation`, never silently accepted.
- R-6N1G-H75W — the reshaped store round-trips: `SetTrigger` of
  `dropbox:create/bills/**` stores derived `source = "dropbox"` (read back by
  SQL); a repeat upserts (no duplicate row); `ListTriggers` returns the
  prompt's filters; `ClearTrigger(promptID, filter)` removes exactly that
  binding and returns `ErrNotFound` for a filter the prompt never held.
- R-6O9C-UYWL — `PromptsForEvent(ctx, "dropbox", key)` over real SQLite with
  the real matcher: with triggers `dropbox:create/bills/**/*.pdf` (prompt A)
  and `dropbox:**` (prompt B), the key `dropbox:create/bills/aws/1.pdf`
  returns exactly {A, B} and `dropbox:create/notes.txt` returns exactly {B};
  a trigger `dropbox:create/bills/*.pdf` does **not** match
  `dropbox:create/bills/aws/1.pdf` (`*` never crosses `/` — the shared
  dialect, not `path.Match`).
- R-6PH9-8QNA — `consume.Handler` fed a `consumer.Event` with
  `Kind`/`Subject` set looks up by the event's canonical `Key()` and fires
  `RunByEvent` with that kind and subject threaded onto the run row (observed
  through the real `prompt.Service` + stubbed runner); an event with empty
  `Kind` returns `ErrSkip`; and `consume.Subscriptions` yields one
  subscription per source with `Filter` exactly `"**"`.
- R-6QP5-MIDZ — an event-triggered run's `input/event.json` is a JSON object
  with exactly the keys `source`, `kind`, `subject`, `event_id`, `payload` —
  and no `type` key — carrying the delivered event's values; a manual run
  still writes no `event.json`.
- R-6RX2-0A4O — through the assembled MCP handler: `set_trigger` accepts
  `(prompt_id, filter)` and returns the binding, `clear_trigger` accepts
  `(prompt_id, filter)`, `create` accepts an inline `triggers` array of
  filter strings (an invalid one rejecting the whole create), and an invalid
  filter or unknown source renders as an `isError` content envelope (not a
  JSON-RPC error) — the schemas advertise `filter` and no `source`/
  `event_filter` argument survives on the wire.
