# Decision 17 — MCP surface over `appkit/mcp`: `internal/mcp` becomes the tool table

**Problem.** prompts' `internal/mcp` carries the copy-pasted JSON-RPC transport
(`ServeHTTP`, the `initialize`/`tools/list`/`tools/call` dispatch, the
`writeJSONRPCError`/`writeJSONRPCResult` writers, `idOrNull`, the local
`Identity` type, the `toolResultText`/`toolResultErr` envelope helpers) plus its
own `health` tool implementation (`toolHealth`) — the same near-identical
plumbing every service duplicated. The chassis now owns all of it (appkit D8
transport, D9 standard tools, consumed here as a fixed external contract); what
is genuinely prompts' is its **sixteen domain tools** — `describe`, `create`,
`import`, `list`, `get`, `update`, `delete`, `set_trigger`, `clear_trigger`,
`run`, `run_list`, `run_get`, `run_output`, `run_cancel`, `run_fs_list`,
`run_fs_read` — their descriptors, schemas, validation, and the `prompt.Service`
hops behind them.

Two prompts-specific facts shape the conversion, both flagged so the wire
consequence is deliberate, not accidental:

- **prompts currently exposes no `reflection` tool.** Its hand-rolled transport
  lists only its own tools plus `health` (17 tools total, no `reflection`).
  Moving to `appkit/mcp` **adds** the chassis `reflection` tool — which is
  meaningful for prompts, the one converting service that is both an event-plane
  **producer** (publishes `run.succeeded`/`run.failed`) and a **consumer** (six
  upstreams). This is the intended suite standardization (every converted
  service exposes `health` + `reflection`), realized here as a deliberate
  one-tool surface addition, not a regression.
- **Chassis-standardized `initialize` and `health` copy.** The chassis
  `serverInfo` becomes `{name: rt.Service()="prompts", version: rt.Version()}`
  (was the hardcoded `{name:"Prompts", version:"1"}`), and the chassis `health`
  descriptor uses the shared wording ("…for this service") in place of prompts'
  "…for the prompts service". The `initialize` **instructions** string is
  preserved byte-for-byte (see below). These are the same chassis-owned changes
  every converted service takes.

**Decision.** `internal/mcp` is reshaped from a transport into a **declaration**:

```go
// internal/mcp — what remains.

// Instructions is byte-identical to the string prompts' old transport returned
// in the initialize `instructions` field ("Prompts runs sandboxed Claude agent
// sessions on your behalf. If you haven't used prompts before, call describe
// first …").
const Instructions = /* the exact current initialize instructions string */

// Tools declares prompts' sixteen domain tools over the prompt.Service (same
// names, descriptions, schemas, and handler bodies as today — describe/create/
// import/list/get/update/delete/set_trigger/clear_trigger/run/run_list/run_get/
// run_output/run_cancel/run_fs_list/run_fs_read). health is NOT declared — the
// chassis registers it (and reflection); declaring `health` locally would fail
// New's reserved-name check.
// contentBase is D22's run-content base URL seam (registry-resolved at the
// composition root), threaded so run_fs_list can stamp content_url entries.
func Tools(svc *prompt.Service, contentBase string) []mcp.Tool

// NewHandler assembles the /mcp http.Handler at the composition root:
// appkit/mcp.New over Instructions + Tools(svc, contentBase) + the
// Router-threaded Service/Version/Health/Events/Publishes/Subscriptions.
func NewHandler(svc *prompt.Service, contentBase string, rt *appkit.Router) (http.Handler, error)
```

- **Deleted:** the JSON-RPC plumbing (`jsonRPCRequest`, `toolCallParams`, the
  `ServeHTTP` dispatch, `handleToolCall`, `writeJSONRPCError`/
  `writeJSONRPCResult`/`idOrNull`, the local `Identity` type), the local
  `toolHealth` implementation and the `health` entry from the descriptor list
  and dispatch switch (chassis-owned), and the local `toolResultText`/
  `toolResultErr` helpers (replaced by `appkit/mcp`'s result helpers —
  `TextResult` for prose and, per D27, `StructuredResult` for domain values and
  the coded `ErrorResult` for failures).
- **Kept:** the descriptor/schema helpers (`desc`/`obj`/`typ`/`configSchema`/
  `triggersSchema`, now producing `mcp.Tool` values), `describeText` and
  `toolDescribe`, every domain tool's description, schema, argument struct, and
  validate-then-call-`svc` body, and the `paramError` type.
- **Two error paths, one split preserved.** A `paramError`
  (genuinely-unparseable tool arguments) is returned as `(nil, err)` so the
  chassis maps it to JSON-RPC `-32602` (invalid params); every **domain** error
  (not-found, validation, sandbox path escape) is returned as an MCP
  `isError:true` content result. This split — parse failure → `-32602`, domain
  failure → `isError` — is what keeps `get` unknown-id, `set_trigger`
  unknown-source, and `run_fs_read` path-escape rendering as `isError` results
  rather than JSON-RPC errors. The **shape** of the domain success and error
  results (the `StructuredResult` machine+text rendering, the coded
  `ErrorResult` vocabulary, and the per-tool `outputSchema`) is D27's contract,
  not this Decision's.
- **The conversion itself changed no domain-tool wire shape:** same sixteen
  tool names, descriptions, schemas, and the `isError`/success **split** above;
  same `initialize` instructions. The surface **gains** the chassis `reflection`
  tool and adopts the chassis `serverInfo`/`health` copy noted above. (Later
  Decisions revise individual tools over this same table: D22 adds
  `content_url` to `run_fs_list` entries; D24 reshapes `set_trigger`/
  `clear_trigger`/`create` to canonical filter strings; D27 renders every
  structured tool's result through `StructuredResult` with a coded error and an
  `outputSchema`. The tool *count* is unchanged by any of them.)
- **Tests re-proven through the new seam.** `mcp_test.go`/`tools_test.go` swap
  their handler construction to go through `appkit/server.New` (a `Register`
  hook calling `NewHandler(svc, rt)`, threading `Events: prompt.Events` and
  `Subscriptions: consume.Subscriptions(sources)` so `reflection` has data) and
  keep their behavioral assertions (create/get/list/run round-trip, trigger
  set/clear, import, error mapping, describe, health envelope). The tool-count
  test moves from 17 to 18 (adds `reflection`); a new reflection test is added.

**Rejected.**
- **Keeping the local transport** — the status quo; a protocol fix would still
  be a many-service edit, and prompts would keep diverging from the suite (no
  `reflection`, a hardcoded `serverInfo`).
- **Declaring `health`/`reflection` in prompts' table** — the chassis reserves
  the names (appkit D8) precisely so services cannot fork the standard tools;
  their inputs (Envelope, Registry, Subscriptions) are Router-held already, and
  prompts' producer `Events` + consumer `Subscriptions` thread straight in.
- **Suppressing the new `reflection` tool to hold the tool count at 17** —
  forking the chassis to hide a standard tool for byte-count parity re-creates
  the drift this conversion removes; the added tool is correct and consistent.
- **Dissolving `internal/mcp` into `cmd/prompts`** — the sixteen tools' schemas,
  validation, and error mapping with their test harness are a real package,
  unlike the post-D16 web remnant.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a real `prompt.Service` + temp DB, driven with
httptest through the `server.New` seam).
- R-DKQP-QZ3Q — `tools/list` through the assembled handler returns **exactly
  eighteen** tools: the sixteen prompts domain tools (`describe`, `create`,
  `import`, `list`, `get`, `update`, `delete`, `set_trigger`, `clear_trigger`,
  `run`, `run_list`, `run_get`, `run_output`, `run_cancel`, `run_fs_list`,
  `run_fs_read`) declared by prompts **plus** `health` and `reflection` supplied
  by the chassis — proving the table/chassis partition (a table that still
  declares its own `health` fails `New`'s reserved-name check; a surface missing
  `reflection` fails the count).
- R-DLYM-4QUF — `reflection` (no arguments) through the assembled handler
  reports `publishes` equal to prompts' two static outcome families
  (kinds `run.succeeded`, `run.failed`, from `prompt.Events` — D25) and
  `subscribes` equal to the six upstream in-edges in `sources` order
  (`cron`, `crm`, `ledger`, `dropbox`, `scripts`, `prompts`, each `"**"` — D24) —
  proving both prompts' producer `Events` and its `Consumers`-derived
  subscriptions thread through the chassis reflection (prompts exposed no
  reflection tool before this conversion).
