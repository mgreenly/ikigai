# Decision 2 — The `routing` package: canonical key and hand-rolled matcher

**Decision.** A new package `eventplane/routing` owns everything key-shaped:
the one rendering rule, the one matcher, and the address-validity rules. Both
`outbox` and `consumer` import it; every selector in the suite (prompts
triggers, service-side consumer filters, any future subscriber) calls this
matcher, so filter semantics cannot diverge — nothing outside this package
renders or parses keys.

```go
package routing

// Key renders the canonical routing key: source + ":" + kind + subject.
// Subject is "" or /-rooted, so no further delimiter is needed.
func Key(source, kind, subject string) string

// Match reports whether pattern matches key. The pattern is anchored: it
// must match the WHOLE key, never a substring. A malformed pattern returns
// a non-nil error, never a silent non-match.
func Match(pattern, key string) (bool, error)

// ValidKind reports whether kind is non-empty lowercase [a-z0-9_.-]+.
func ValidKind(kind string) bool

// ValidSubject reports whether subject is "" or a single-line /-rooted path.
func ValidSubject(subject string) bool
```

*Dialect* (the product constant, pinned here as behavior): `*` matches any
run of characters **within** a segment and never matches `/`; `**` matches
across segments (including zero); `?` matches exactly one non-`/` character;
character classes `[...]` / `[^...]` with ranges as in standard glob; **no
brace expansion** — `{` and `}` are literal characters. The separator `:` is
an ordinary character (only `/` is structural), so `dropbox:*` matches the
subjectless key `dropbox:create`.

The matcher is **hand-rolled** — explicitly not a third-party dependency (see
research: `path.Match` cannot express `**`; `bmatcuk/doublestar` was evaluated
and rejected). Its correctness is pinned by an exhaustive table-driven test in
`routing` — one table row per dialect rule and edge case, so the dialect is
defined by this package's own tests, not by an upstream's semantics.
Implementation shape (iterative or recursive matching) is the builder's
choice; the behaviors below are not.

**Rejected.**

- *`github.com/bmatcuk/doublestar`* — correct dialect, but a dependency for
  one screen of logic and an upstream's semantics as our contract; the
  operator decision is hand-rolled.
- *Translating patterns to `regexp`* — workable but obscures error cases and
  anchoring, and drags regexp semantics (e.g. class syntax differences) into
  the contract; a direct matcher keeps the dialect exactly as specified.
- *Per-consumer matching (each service implements its own filter)* — the
  status quo failure mode: semantics drift between selectors; rejected by the
  one-matcher principle.

**Verification.**

- R-3FIX-KJG7 — `Key` renders exactly:
  `Key("ledger","recorded","")` = `"ledger:recorded"` and
  `Key("dropbox","create","/bills/aws/2026-06.pdf")` =
  `"dropbox:create/bills/aws/2026-06.pdf"`.
- R-3GQT-YB6W — `*` never crosses `/`:
  `dropbox:create/bills/*.pdf` does **not** match
  `dropbox:create/bills/aws/2026-06.pdf`, and does match
  `dropbox:create/bills/x.pdf`.
- R-3HYQ-C2XL — `**` crosses segments:
  `dropbox:create/bills/**/*.pdf` matches
  `dropbox:create/bills/aws/2026-06.pdf` and
  `dropbox:create/bills/a/b/c.pdf`, and does **not** match
  `dropbox:create/bills/aws/readme.txt`.
- R-3J6M-PUOA — matching is anchored to the whole key: `bills/**` does not
  match `dropbox:create/bills/x.pdf` (no prefix elision), and
  `dropbox:create` does not match `dropbox:create/bills/x.pdf` (no suffix
  elision).
- R-3KEJ-3MEZ — `?` matches exactly one non-`/` character (`cron:tick/a?c`
  matches `cron:tick/abc`, not `cron:tick/ac` and not `cron:tick/a/c`);
  character classes work (`dropbox:create/bills/[ab]*.pdf` matches
  `.../a1.pdf`, not `.../c1.pdf`); braces are literal (`dropbox:{a,b}` does
  not match `dropbox:a`).
- R-3LMF-HE5O — a malformed pattern (e.g. unterminated `[`) returns
  `(false, err)` with a non-nil error — never a silent non-match.
- R-3MUB-V5WD — literal precision and subjectless matching:
  `cron:tick/bill-sweep` matches exactly that key and not
  `cron:tick/bill-sweep2`; `dropbox:*` matches the subjectless key
  `dropbox:create` and does not match `dropbox:create/bills/x.pdf`.
- R-41H4-GESP — `ValidKind` accepts `create`, `run.succeeded`, `tick-2`;
  rejects `""`, `Create`, `a b`, `a/b`. `ValidSubject` accepts `""` and
  `/bills/aws`; rejects `bills` and any subject containing a newline.
