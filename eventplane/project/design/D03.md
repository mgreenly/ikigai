# Decision 3 — Producer families: registry, reflection, and filter validation

**Decision.** The producer's closed `outbox.Registry`/`EventType` list becomes
a list of **families** — the kinds a source emits, with the subject described
rather than enumerated, so dynamic subjects (cron's schedule names, dropbox's
mirror paths) are ordinary:

```go
// Family declares one published event family. Sample remains the single
// source for BOTH the reflected JSON Schema and the marshaled worked
// example, so the two cannot drift (unchanged property).
type Family struct {
    Kind        string // literal kind, e.g. "create"
    Subject     string // human description of the subject shape, e.g. "/<mirror path>" ("" = subjectless)
    Description string // what this fact means and when it fires
    Sample      any    // a filled-in instance of the payload struct
}

// Registry is the ordered set of families a service publishes.
type Registry []Family

func (r Registry) Index() []map[string]any                    // {kind, subject, description} per family, in order
func (r Registry) Detail(kind string) (map[string]any, error) // {kind, subject, description, schema, example}
func (r Registry) CouldMatch(source, filter string) (bool, error)

// UnknownKindError replaces UnknownEventTypeError: Detail's typed error for
// an undeclared kind, carrying the declared kinds so callers can build a
// corrective message (the ledger bad_root pattern, unchanged).
type UnknownKindError struct {
    Kind  string
    Valid []string // declared kinds, in registry order
}
```

*Append gating* (revises D1's producer input path): when the registry is
non-empty, `Append` rejects an event whose `Kind` is declared in no family,
with an error naming the declared kinds — reflection still lists everything
emittable. Subjects are **not** gated against the registry: the subject
description is documentation, not a constraint (subjects are dynamic by
design).

*Validation helper.* `CouldMatch(source, filter)` answers the one question a
selector-validating service (e.g. prompts) asks: **could this filter string
match some family of this source?** — true iff, for at least one family,
there exists *some* subject (empty or `/`-rooted) for which the filter
matches `routing.Key(source, family.Kind, subject)`. Kind participates in the
glob intersection; **subject is left open**. A malformed filter returns a
non-nil error (surfacing D2's pattern error). The existential check is
implemented on top of the `routing` matcher (the intersection primitive may
live in `routing`; the exported service-facing seam is this Registry method).
How prompts *uses* this is prompts' spec, out of scope here.

*Reflection schema/example machinery* (`schema`, `exampleOf`, the reflected
kinds and the fail-loud panic on unsupported kinds) is baseline and unchanged
— it re-hangs off `Family.Sample` instead of `EventType.Sample`.

**Rejected.**

- *Keeping a closed enumeration plus a cron carve-out* — the status quo this
  revision exists to delete; every producer with dynamic subjects would need
  its own special case.
- *Enumerating subjects in the registry* — impossible for dropbox/cron
  (unbounded, runtime-known); a service that can enumerate live subjects (as
  cron's reflection does today) may still do so in its own surface.
- *Full glob-vs-glob intersection over the entire key* — general glob
  intersection is needless here; the settled decision fixes the tractable
  contract: intersect on kind, leave subject open.

**Verification.**

- R-3O28-8XN2 — with a non-empty registry declaring only kind `create`,
  `Append` of kind `delete` fails with an error that names `delete` and lists
  the declared kinds; `Append` of kind `create` succeeds. An empty registry
  leaves `Append` ungated (baseline behavior preserved).
- R-3QI1-0H4G — `Index()` on a two-family registry returns exactly one
  `{kind, subject, description}` entry per family, in declared order, with
  the subject description carried through verbatim.
- R-3RPX-E8V5 — `Detail("create")` returns a schema reflected from the
  Sample's *type* and an example marshaled from the Sample's *value* that
  agree (every example field appears in the schema); `Detail("nope")` returns
  a `*UnknownKindError` carrying the declared kinds.
- R-3SXT-S0LU — for a source `dropbox` with the sole family kind `create`:
  `CouldMatch("dropbox", "dropbox:create/bills/**")` is true,
  `CouldMatch("dropbox", "dropbox:delete/**")` is false,
  `CouldMatch("dropbox", "dropbox:*")` is true (matches the kind with an
  empty subject), and a malformed filter (unterminated `[`) returns a
  non-nil error.
- R-3U5Q-5SCJ — subject is left open: `CouldMatch("cron",
  "cron:tick/some-schedule-nobody-declared")` is true for a family with kind
  `tick`, whatever its subject description says — a subject pattern never
  causes rejection.
