# Decision 5 — Feed guard ownership moves to the chassis

**Decision.** `outbox.FeedHandler()` performs **no header checks of its own**.
The loopback-only property of `GET /feed` is owned entirely by the appkit
chassis mount: `appkit/server` wraps the feed handler in `LoopbackOnly`, whose
predicate keys on `X-Forwarded-Proto` **only** (nginx stamps it on every
proxied request; a direct loopback caller never sends it). The chassis is the
sole mount path for `FeedHandler` in the suite — `appkit/feed` constructs the
handler and `server.New` mounts it wrapped — so the guard lives in exactly one
place, outside this module.

Concretely, the inline defence-in-depth block at the top of the handler
returned by `FeedHandler` (the `X-Owner-Email` / `X-Forwarded-Proto` → 404
check) is **deleted**, along with the doc-comment paragraph describing it. The
handler streams for any request that reaches it; keeping requests away from it
is the mounter's job.

This is a correctness fix as well as deduplication: the suite's identity model
now has loopback callers assert `X-Owner-Email` themselves (suite doc
`docs/structured-mcp-design.md`), so an inline guard keyed on `X-Owner-Email`
would 404 legitimate direct callers. The chassis predicate deliberately keys
on `X-Forwarded-Proto` alone; this module must not re-impose the broader,
now-wrong check.

**Rejected.**

- *Keeping the inline guard as defence in depth* — it duplicates a chassis
  invariant in a second place (two copies to keep in sync), and its
  `X-Owner-Email` clause actively contradicts the caller-asserted identity
  model: a legitimate loopback caller carrying its own asserted identity would
  be 404'd. Redundancy that misfires is not defence.
- *Narrowing the inline guard to `X-Forwarded-Proto` only (matching the
  chassis predicate)* — still a second copy of the same predicate; the chassis
  wrap (`LoopbackOnly` on the `/feed` mount) already applies to every mount of
  this handler in the suite, so the narrowed copy guards nothing extra and
  invites drift.

**Verification.**

- R-Z8Y5-5R0C — the bare `FeedHandler`, mounted directly in an
  `httptest.Server` (real HTTP client), serves the SSE stream normally for a
  request carrying `X-Owner-Email` and/or `X-Forwarded-Proto` headers: the
  response is 200 `text/event-stream` and delivers an appended event's frame
  (not a 404). This is the discriminating behavior — the pre-revision handler
  returns 404 for exactly these requests.
