# Decision 1 — Envelope and wire cutover: `kind` + `subject` replace `type`

**Decision.** The event's flat `type` is replaced by the two routing fields
`kind` and `subject`, everywhere at once — struct, table, wire — with no
compatibility period (the suite's no-live-data migration window makes the hard
cutover cheap; see product's scope).

*Producer input.* `outbox.Event` becomes:

```go
type Event struct {
    Kind    string          // fact class, lowercase [a-z0-9_.-]+, e.g. "create"
    Subject string          // "" or a /-rooted routing path, e.g. "/bills/aws/2026-06.pdf"
    Payload json.RawMessage // domain snapshot, marshaled by the caller (unchanged)
}
```

`Append` validates the address before insert using D2's `routing.ValidKind` /
`routing.ValidSubject` (the charset/shape rules are defined once, in
`routing`) and rejects a bad address with an explanatory error — a malformed
address never enters the plane. The existing Append contract (same `*sql.Tx`
as the domain write, id/time minted once at append, `Ring` after commit) is
baseline and unchanged.

*DDL.* `outbox.SchemaSQL`'s `type` column becomes two columns; everything
else (the load-bearing `AUTOINCREMENT`, the retention index) is unchanged:

```sql
CREATE TABLE outbox (
  seq        INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id   TEXT    NOT NULL,
  kind       TEXT    NOT NULL,
  subject    TEXT    NOT NULL DEFAULT '',
  payload    TEXT    NOT NULL,
  created_at TEXT    NOT NULL
);
CREATE INDEX idx_outbox_created_at ON outbox(created_at);
```

As a consequence of eventplane owning only the constants, each consuming
service re-applies the revised DDL through its own migration runner in its own
spec; nothing here reaches into a service's migrations. `consumer.SchemaSQL`
needs **no** change: `feed_offset` carries no event-shape column (the cursor
is opaque), verified in research.

*Wire.* The envelope serialized on the `data:` line becomes
`{id, source, time, kind, subject, payload}` — `kind` and `subject` are
separate JSON fields, and the `type` key is gone. The SSE frame's `event:`
line stops carrying the raw type and carries the **canonical key** rendered by
`routing.Key(source, kind, subject)` (D2) — the one place a key is rendered on
the producer side. The `id:` cursor line and the control frames (`resync`,
`caught-up`, `status`, keepalives) are baseline and untouched.

**Rejected.**

- *Dual-format period (emit both `type` and `kind`/`subject`)* — rejected by
  the settled suite decision: hard cutover inside the migration window;
  carrying both invites drift and consumers that never migrate.
- *Packing the key into a single `type` string (cron's old trick,
  generalized)* — rejected: the routing doc's core principle is structure in
  the envelope, string only for matching; packing forces every consumer to
  parse keys apart.
- *A migration shim in eventplane for existing outbox tables* — rejected:
  services own their migrations; eventplane owns only the DDL constants, and
  the migration window allows state wipes.

**Verification.**

- R-39FF-NOQQ — `Append` rejects an event whose `Kind` is empty or violates
  `[a-z0-9_.-]+` (e.g. `Create`, `file created`) with a non-nil error naming
  the kind; a conforming kind (e.g. `run.succeeded`) is accepted.
- R-3ANC-1GHF — `Append` rejects a `Subject` that is non-empty and not
  `/`-rooted (e.g. `bills/x.pdf`); it accepts `""` and `/bills/x.pdf`.
- R-3BV8-F884 — the revised `outbox.SchemaSQL` applies cleanly to a real
  SQLite database (`modernc.org/sqlite`), the resulting table has `kind` and
  `subject` columns and **no** `type` column, and `Append` inserts an event's
  kind and subject into those columns (read back by SQL).
- R-3D34-SZYT — an event frame served by the real `FeedHandler()` over an
  `httptest` HTTP connection carries a `data:` JSON object with exactly the
  keys `id`, `source`, `time`, `kind`, `subject`, `payload` — and no `type`
  key (hard cutover).
- R-3EB1-6RPI — the same frame's `event:` line is the canonical key, e.g.
  appending kind `create`, subject `/bills/a.pdf` from source `dropbox`
  yields `event: dropbox:create/bills/a.pdf`; a subjectless append from
  source `ledger`, kind `recorded` yields `event: ledger:recorded`.
- R-42P0-U6JE — an event replayed to a second real feed connection is
  byte-identical to its first delivery — same `id:` cursor, same `event:`
  key, same `data:` bytes (id, time, kind, subject stored once at append,
  stable across replays).
