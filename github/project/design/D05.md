# Decision 5 — The loopback `GET /pr` twin for scripts

## Decision

`scripts` is deterministic and does not speak MCP/JSON-RPC; like its dropbox
consumer (`GET /content`), it reaches a peer through a plain loopback HTTP route.
`internal/gh` exposes one such route — `GET /pr` — the loopback twin of the
`pr_get` MCP verb, returning a pull request as JSON.

It is registered **loopback-only, not behind `RequireIdentity`**, via the shared
chassis guard: `rt.HandleLoopback("GET /pr", client.PRHandler())` wraps the
handler in `server.LoopbackOnly`. The guard keys on **`X-Forwarded-Proto` only**
— nginx stamps it on every proxied request, so any request that transited the
public front door is answered with a bare 404 before `PRHandler` runs, while a
loopback machine caller that asserts `X-Owner-Email` (without `X-Forwarded-Proto`)
is served. This replaces the former hand-copied predicate (the
`X-Owner-Email` **or** `X-Forwarded-Proto` check and its `hasIdentityHeader`
helper), which conflated "carries identity" with "came from outside"; the guard
swap and its verification live in **D8**. `PRHandler` no longer inspects headers
— it keeps only its input parsing and PR fetch:

```go
// PRHandler returns the loopback GET /pr handler: ?repo=<name>&number=<n> →
// the pull request as JSON. Loopback-only via the chassis guard (server.
// LoopbackOnly); the handler itself does no header inspection.
func (c *Client) PRHandler() http.Handler
```

Contract:

- `GET /pr?repo=<name>&number=<n>` reaching the handler (loopback; not fronted by
  nginx) → `200 application/json` with the same `PRDetail` shape `pr_get` returns
  (built by the same `Client.PRGet`).
- A request that transited the front door (bears `X-Forwarded-Proto`) → `404` via
  the chassis guard, `PRHandler` never invoked, no PR data (D8 `R-FT0R-PBSD`).
- Missing/malformed `repo` or `number` → `400`; a PR that GitHub reports absent
  (`ErrNotFound`) → `404`. No path leaks a `500` for a client/not-found condition
  and no input crashes the handler.

The nginx fragment (D6) additionally makes the public mount return `404` for
`/srv/github/pr` (defence in depth), like the `/feed` block in other services —
but the chassis guard is the primary defence and stands even if the fragment is
wrong. `github` has no `/feed`, so `/pr` is the only loopback-private route.

## Rejected

- **Behind `RequireIdentity`.** Then `scripts` would have to present a
  dashboard-issued token it does not have; the loopback-only route is exactly how
  dropbox already serves deterministic consumers.
- **A per-handler copied guard predicate.** The former inline
  `X-Owner-Email`/`X-Forwarded-Proto` check is replaced by the shared
  `server.LoopbackOnly` helper (D8): one guard, keyed on the single header that
  answers "did this cross nginx?", not a per-service copy that drifts.
- **Twins for every verb now.** YAGNI: only a proven PR-read path is needed today
  (product). More twins are added per-need, each a thin handler over the same
  `Client`. `GET /pr` establishes the pattern.
- **A separate REST server / port.** The route lives on the same loopback server as
  `/mcp`, distinguished only by being mounted without the identity gate — one
  process, one port (3203).

## Verification

- `R-EPVL-Z2UI` — `GET /pr?repo=<name>&number=<n>` reaching `PRHandler` returns
  `200` and a JSON body matching the `pr_get` PR shape for that repo/number
  (built via `Client.PRGet`). Asserted offline with a stubbed transport, calling
  `PRHandler` directly (its own guard removed; the route-level guard is D8's).
- `R-ETJB-4E2L` — A missing/malformed `repo` or `number` returns `400`; a PR the
  client reports `ErrNotFound` returns `404`; neither a bad input nor a not-found
  yields a `500` or a panic. Asserted offline across the bad-input and
  not-found cases.

(The loopback guard's behavior — front-door requests 404'd, loopback
identity-asserting callers served — is D8 `R-FT0R-PBSD`. The prior verification
for the old two-header predicate is retired to the frozen plan (phase-05): the
behavior it pinned — `X-Owner-Email` alone forcing a 404 — is no longer true
under the chassis guard, so it is not restated as current design.)
