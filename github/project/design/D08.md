# Decision 8 — Structured MCP adoption

Realizes the suite-level contract in `docs/structured-mcp-design.md` for
`github`: a domain verb is registered once and answers every caller, carrying
`structuredContent` for machines and mirrored text for agents in one response,
with a declared `outputSchema` and a typed error code. This Decision is a
**migration**, not a swap: `github` today **hand-rolls its own MCP JSON-RPC
layer** in `internal/mcp/mcp.go` (its own `initialize`/`tools/list`/`tools/call`
dispatch, its own `writeJSONRPCResult`/`writeJSONRPCError`,
`toolResultText`/`toolResultErr`/`toolResultJSON` helpers) and never imports
`appkit/mcp`. That whole layer is replaced by the shared appkit transport. D4
(the tool surface — verbs, bare names, provenance logging) and D5 (the `/pr`
loopback twin) are rewritten in place to cite this Decision for transport, result
shape, error codes, and the loopback guard; this file owns the migration's new
behaviors and their ids.

## Decision

**The hand-rolled transport is deleted; `appkit/mcp` owns the wire.**
`internal/mcp/mcp.go` (the bespoke JSON-RPC `Handler`, its `ServeHTTP` method
switch, the `jsonRPCRequest`/`toolCallParams` types, the
`writeJSONRPCResult`/`writeJSONRPCError`/`idOrNull` writers, and the
`toolResultText`/`toolResultErr` helpers) is removed. `internal/mcp` keeps only
the `GitHubClient` interface (D3's 14-method slice, unchanged) and becomes a
**tool-registration package** over `appkit/mcp`, structurally identical to the
converted `crm/internal/mcp`:

```go
import (
    "appkit"
    appkitmcp "appkit/mcp"
    "appkit/server"
)

// NewHandler builds the POST /mcp handler from the appkit Router seam. The
// shared transport owns JSON-RPC, initialize, tools/list, health, and reflection;
// github declares only its 14 domain tools.
func NewHandler(client GitHubClient, rt *appkit.Router) (http.Handler, error) {
    return appkitmcp.New(appkitmcp.Options{
        Service:       rt.Service(),
        Version:       rt.Version(),
        Instructions:  instructions,
        Tools:         Tools(client, rt.Logger()),
        Health:        rt.Health(),        // the spec.go closure that drives a real ReposList
        Events:        rt.Events(),        // empty — github publishes nothing
        Subscriptions: rt.Subscriptions(), // nil — github consumes nothing
    })
}
```

`internal/githubapp/Spec`'s `Handlers` hook stops constructing the bespoke
handler and instead mounts `appkitmcp.New(...)`:
`rt.Handle("POST /mcp", rt.RequireIdentity(handler))`. `MCP: true`,
`Health: health` (the existing `ReposList`-driving closure), and the empty
event graph stay on the Spec.

**Protocol revision.** The shared transport speaks `2025-06-18` (appkit's
`protocolVersion`); `initialize` returns that string. This replaces github's
hand-rolled `2025-03-26`.

**Tools registered once, with an `outputSchema`.** `Tools(client, logger)`
returns `[]appkitmcp.Tool`, one per D3 client method — the same 14 bare verbs
(`toolPrefix = ""`) with the same input schemas and the same dispatch to the
`GitHubClient` method. Each `Tool` additionally declares a hand-authored
`OutputSchema map[string]any`. For the eleven single-object verbs it **mirrors
the JSON that verb already emits, verbatim** (the refactor formalizes existing
output; it does not reshape those results); the three list verbs are the one
deliberate exception — they wrap their array under an `items` key so
`structuredContent` is a JSON object as the suite contract requires (see below).
The existing `obj`/`descTyp`/`arrayTyp` input-schema helpers are reused; a small
private output-schema literal accompanies each tool. Illustrative shapes,
mirroring the `internal/gh` result structs (list verbs wrapped):

| tool | structuredContent shape (mirrors the gh type) | outputSchema top type |
|---|---|---|
| `repos_list` | `{items: [{name, full_name, private, default_branch}]}` (`[]gh.Repo` under `items`) | `object` |
| `repo_get` | `{name, full_name, private, default_branch}` (`gh.Repo`) | `object` |
| `pr_list` | `{items: [{number, title, state, body, html_url}]}` (`[]gh.PR` under `items`) | `object` |
| `pr_get` | `{number, title, state, body, html_url, Files:[{filename,status,additions,deletions,changes,patch}]}` (`gh.PRDetail`) | `object` |
| `pr_comment` / `issue_comment` | `{id, body, html_url}` (`gh.Comment`) | `object` |
| `pr_review` | `{id, state, body, html_url}` (`gh.Review`) | `object` |
| `pr_merge` | `{sha, merged, message}` (`gh.MergeResult`) | `object` |
| `issue_list` | `{items: [{number, title, state, body, html_url, labels?, user?, assignees?}]}` (`[]gh.Issue` under `items`) | `object` |
| `issue_get` / `issue_create` / `issue_update` | `{number, title, state, body, html_url, labels?, user?, assignees?}` (`gh.Issue`) | `object` |
| `file_get` | `{path, sha, encoding}` (`gh.FileContent`) — **metadata only** | `object` |
| `file_put` | `{content:{path,sha,encoding}, commit:{sha,message}}` (`gh.FileCommit`) | `object` |

Two shapes need explicit treatment the build must preserve:

- **List verbs wrap their array in an object.** `structuredContent` must be a
  JSON **object** (`docs/structured-mcp-design.md`; MCP rev `2025-06-18`), so
  `repos_list`/`pr_list`/`issue_list` return `{"items": [...]}` — the array under
  a single `items` key — with an `outputSchema` of top type `object`,
  `properties.items` an array of the element schema, and `required: ["items"]`.
  This is the one place the migration reshapes a result rather than mirroring the
  bare `[]T` the hand-rolled path emitted: the wire envelope is already changing
  wholesale (protocol bump, structured errors, chassis health/reflection), an
  off-spec top-level array is not a shape worth preserving, and the suite
  precedent is crm's `search → {items, next_cursor}`. The handler builds the
  wrapper (`StructuredResult(map[string]any{"items": repos})` or a small typed
  `{Items []T}` struct); the element shape is unchanged. The other 11 verbs still
  mirror their single-object result verbatim.
- **`file_get` returns metadata, not bytes.** `gh.FileContent.Content` is tagged
  `json:"-"`, so the emitted JSON is `{path, sha, encoding}` with **no** file
  body; `file_put`'s embedded `content` is the same metadata-only object. The
  `outputSchema` mirrors that omission exactly and does not add a `content`
  property. (Bytes never ride MCP — `docs/structured-mcp-design.md`; the byte
  path is D5's `GET /pr` shape only for PRs, and file bodies are simply not part
  of this verb's contract today.)

**Structured success results.** Each tool's success path returns
`appkitmcp.StructuredResult(v)` — one typed value renders both
`structuredContent` (the machine channel) and a mirrored text block (the agent
channel). Its signature is `StructuredResult(v any) (map[string]any, error)`;
the handler returns that pair directly, so a marshal failure surfaces as a
JSON-RPC error rather than being swallowed. This replaces the old
`toolResultJSON`/`clientResult` path, which set only a text block.

**Typed error codes — the failure→code mapping.** Every error result is
`appkitmcp.ErrorResult(code, msg)` from the closed vocabulary. The MCP layer can
only branch on what the client surfaces, so the mapping keys on the D3 sentinels
and the local argument check, and is exhaustive over them:

| failure | source | `ErrorCode` |
|---|---|---|
| missing/malformed required argument, or un-decodable `arguments` | local, before any client call | `ErrValidation` |
| `errors.Is(err, gh.ErrNotFound)` (GitHub 404) | client | `ErrNotFound` |
| `errors.Is(err, gh.ErrInvalid)` (GitHub 422 unprocessable) | client | `ErrValidation` |
| `errors.Is(err, gh.ErrAppAuth)` (app-JWT / installation / token-mint failure) | client | `ErrSourceUnavailable` |
| any other client error — network failure, other non-2xx (403/409/5xx), response-decode failure | client (`default`) | `ErrSourceUnavailable` |

A single private `codeFor(err) appkitmcp.ErrorCode` realizes the client-side
rows; the local-argument rows call `ErrorResult(appkitmcp.ErrValidation, …)`
inline where the old code called `toolResultErr`. `ErrConflict` and `ErrTooLarge`
are **deliberately unused**: the client does not surface 409 distinctly (it folds
into the `default` row → `source_unavailable`) and github has no size-bounded
verb, so minting either code would misdescribe a failure the code cannot
distinguish.

**Prose exceptions: none.** github has no `guide`/`describe` documentation tool
and no raw-content read — all 14 domain verbs emit JSON and therefore all declare
an `outputSchema` and return `StructuredResult`. `file_get` is **not** a prose
exception: it returns JSON metadata, not raw bytes. The only text-only,
schema-less results are the transport-owned error envelopes.

**Chassis health and reflection.** github stops registering `health` and
`reflection` as service tools (they are reserved names in `appkit/mcp`; the
transport appends them). The chassis `health` calls github's existing
`Health` closure (the real `ReposList` auth proof, unchanged) and renders a
structured envelope centrally; the chassis `reflection` renders from the empty
`Events`/nil `Subscriptions`, reporting zero published events and zero
subscriptions. Both keep their prior observable results (D4's `health` and
`reflection` verification); github writes nothing for them.

**The `/pr` loopback guard moves to the chassis.** D5's `GET /pr` self-guard —
the hand-copied `X-Owner-Email` **or** `X-Forwarded-Proto` predicate and its
`hasIdentityHeader` helper in `internal/gh/pr_route.go` — is deleted. `/pr` is
registered with `rt.HandleLoopback("GET /pr", client.PRHandler())` so the shared
`server.LoopbackOnly` guard fronts it. That guard keys on **`X-Forwarded-Proto`
only**: nginx stamps it on every proxied request, so a request bearing it gets a
bare 404 without the handler running, while a loopback machine caller that
asserts `X-Owner-Email` (but no `X-Forwarded-Proto`) is **served**. The old
predicate conflated "carries identity" with "came from outside"; caller-asserted
identity on loopback breaks that conflation, which is exactly the narrowing the
suite contract requires. `PRHandler` keeps its own 400/404 input handling; it no
longer inspects headers.

## Rejected

- **Keeping the hand-rolled transport and only adding fields.** Rejected: the
  suite contract is "one registration in appkit renders both renderings"; a
  second transport that also has to grow `outputSchema`, `structuredContent`, and
  the error vocabulary is a permanent parallel maintenance burden the migration
  exists to remove. appkit already speaks `2025-06-18`.
- **A top-level array in `structuredContent`** for the list verbs (mirroring the
  bare `[]T` the hand-rolled path emitted). Rejected: `structuredContent` is
  defined as a JSON object (`docs/structured-mcp-design.md`; MCP rev
  `2025-06-18`), and a top-level `type:array` `outputSchema` is off-spec. Since
  this is a full-envelope migration, the array is wrapped under `items`
  (`{"items":[…]}`, the crm `search` precedent) rather than preserved off-spec.
- **Adding file bytes to `file_get`'s structured result.** Rejected here: it is a
  result-shape change, not a formalization of existing output, and bytes do not
  ride MCP. Flagged for review as a possible latent gap, but out of scope for a
  refactor that must not reshape results.
- **Minting `conflict`/`too_large` for completeness.** Rejected: the client
  cannot distinguish a 409 or a size overflow, so a code claiming to would be
  untestable against the real failure and would lie to a machine caller.
- **A per-service loopback guard predicate.** Rejected by the suite contract: the
  guard is one shared appkit helper keyed on `X-Forwarded-Proto`; per-service
  copies are what drift.

## Verification

- `R-FI1O-9E44` — `initialize` over the appkit-served `POST /mcp` handler returns
  `protocolVersion` exactly `2025-06-18` (not the old `2025-03-26`). Asserted
  offline against the handler built by `NewHandler` with a fake `GitHubClient`.
- `R-FJ9K-N5UT` — A single-object domain verb (`repo_get`) success result carries
  a `structuredContent` object equal to the `gh.Repo` value **and** a `text`
  content block whose body is that same JSON; a wrong implementation that omits
  `structuredContent` (the old text-only path) fails. Asserted offline with a
  fake client returning a known repo.
- `R-FKHH-0XLI` — Every one of the 14 registered domain tools in `tools/list`
  declares a non-nil `outputSchema` (table-driven over the advertised tools);
  there are no prose exceptions among domain tools. The chassis `health` and
  `reflection` also carry schemas. A tool registered without an `outputSchema`
  fails the table.
- `R-FLPD-EPC7` — The declared `outputSchema` and result shape match D8 for the
  two discriminating cases: `repos_list`/`pr_list`/`issue_list` declare top type
  `object` with `required: ["items"]` and their success `structuredContent` is a
  JSON object whose `items` is the array (never a bare top-level array);
  `file_get`'s `outputSchema` properties are exactly `{path, sha, encoding}` with
  **no** `content` property, and its success `structuredContent` likewise omits
  the file body. Asserted offline against the schemas and a fake-client result.
- `R-FMX9-SH2W` — A `tools/call` whose client returns `gh.ErrNotFound` yields an
  `isError` result whose `structuredContent.code` is exactly `"not_found"` (not a
  bare message, not `internal`). Asserted offline with a fake client returning
  `gh.ErrNotFound`.
- `R-FO56-68TL` — A `tools/call` whose client returns `gh.ErrInvalid` (GitHub 422)
  yields an `isError` result with `structuredContent.code` exactly `"validation"`.
  Asserted offline with a fake client returning `gh.ErrInvalid`.
- `R-FPD2-K0KA` — A `tools/call` whose client returns `gh.ErrAppAuth`, **and** one
  whose client returns a generic transport error (neither sentinel), both yield an
  `isError` result with `structuredContent.code` exactly `"source_unavailable"`.
  Asserted offline with a fake client returning each.
- `R-FQKY-XSAZ` — A `tools/call` for a verb with a missing or malformed required
  argument (e.g. `pr_get` without `number`) yields an `isError` result with
  `structuredContent.code` exactly `"validation"` and does **not** invoke the
  `GitHubClient` (the fake records zero calls). Asserted offline.
- `R-FT0R-PBSD` — `GET /pr` mounted via `rt.HandleLoopback`: a request bearing
  `X-Forwarded-Proto: https` receives a bare 404 and `PRHandler` does **not**
  run (no client call, no PR body); the same `?repo=&number=` request **without**
  `X-Forwarded-Proto` but **with** `X-Owner-Email` set is served `200` with the
  PR JSON. Asserted offline against the route registered on a real
  `appkit`/`server.Router`, exercising both header combinations.
