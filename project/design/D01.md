# Decision 1 — The `/opt/<svc>/` install tree

**Decision.** Every service is a self-contained subtree under `/opt/<svc>/`
holding *all* of its on-box parts. The canonical paths, owned by opsctl's
`Layout` type (rooted at `OPSCTL_ROOT`, default `/opt`):

```
/opt/<svc>/                              0755 <svc>:<svc>
  bin/run            → ../libexec/<svc>-v<semver>   (D02: the atomic swap point)
  libexec/
    <svc>-v<semver>                                  (D02: versioned binaries)
  etc/manifest.env                                   (regenerated each deploy)
  state/                                 0711 <svc>:<svc>   (D05: backed up + restored; TRAVERSE-only)
    <svc>.db (+ -wal/-shm)               0640 <svc>:<svc>   (private — never nginx-readable)
    www/                                 0750 <svc>:web     (served content; nginx reads via the `web` group)
      public/                            0750 <svc>:web     (served directly, no auth)
      private/                           0750 <svc>:web     (served behind the introspection auth gate)
  cache/                                 0750 <svc>:<svc>   (D05: NOT backed up; reset on restore)
    <svc>.db.generation                                    (event-plane epoch sidecar)
  backups/                               0750 <svc>:<svc>   (local pre-migration snapshots; not S3)
    pre-v<semver>.db
```

This **replaces today's `data/` dir** (split into `state/` + `cache/` by D05) and
today's `releases/` + `current` symlinks (replaced by `libexec/` + `bin/run` by
D02). `bin/`, `etc/`, and `backups/` keep their current meaning. The separate
top-level `www/` tree `sites` has today moves **into `state/www/`** — served
content is durable state like everything else, with **no special service-casing
and no "extra backed-up root"**: `state/` is the entire backed-up set (D05). The
apex TLS cert (D07) remains the only artifact handled outside this tree.

**Served content & the `web` group.** Any service may have `state/www/{public,
private}` (the universal version of `sites`' current `served/{public,private}`).
nginx must read it without ever reading the DB; the scheme that guarantees this:

- `state/` is **`0711` traverse-only** — nginx can pass *through* it to reach
  `www/` but cannot list it; the DB at **`0640 <svc>:<svc>`** is unreadable by any
  non-owner. So **the DB's privacy rests on `0711` + `0640`, not on the group.**
- A shared **`web` group** (members: `<svc>` and `www-data`) owns `state/www/**`
  at `0750 <svc>:web`, so nginx reads served content but the rest of the suite/box
  does not. `setup` creates the group and applies ownership.
- **Default-private, then selectively open `www/`:** the safe ordering is
  `chown -R <svc>:<svc> state/` → `chmod 0711 state/` → re-open `state/www/` to
  `<svc>:web`. A skipped re-open yields **403 on `www/`** (loud, fail-safe), never
  a leaked DB. The dangerous `chown -R …:web` direction is never the default.
- **public vs private is an nginx concern, not a filesystem one:** both are
  `0750 <svc>:web` on disk; `setup`'s per-service nginx fragment serves `public/`
  directly and gates `private/` behind the same `/srv/<svc>/` introspection the
  proxy routes use. (Generalizes `sites`' served surface to every service.)

`Layout` grows/loses methods accordingly (illustrative):

```go
func (l Layout) LibexecDir() string            // /opt/<svc>/libexec
func (l Layout) LibexecBinary(v string) string // /opt/<svc>/libexec/<svc>-<v>   (v is the full v-prefixed SemVer)
func (l Layout) RunLink() string               // /opt/<svc>/bin/run             (unchanged path; new target)
func (l Layout) StateDir() string              // /opt/<svc>/state
func (l Layout) DBPath() string                // /opt/<svc>/state/<svc>.db
func (l Layout) WWWDir() string                // /opt/<svc>/state/www
func (l Layout) WWWPublicDir() string          // /opt/<svc>/state/www/public
func (l Layout) WWWPrivateDir() string         // /opt/<svc>/state/www/private
func (l Layout) CacheDir() string              // /opt/<svc>/cache
func (l Layout) GenerationPath() string        // /opt/<svc>/cache/<svc>.db.generation
func (l Layout) BackupsDir() string            // /opt/<svc>/backups   (unchanged)
// MOVED: WWWRoot /opt/<svc>/www → /opt/<svc>/state/www (now under state, backed up)
// REMOVED: ReleasesDir, ReleaseDir, ReleaseBinary, CurrentLink, CurrentBinary, DataDir, WWWServedDir
```

This is primarily a **structural** decision; its behavior is proven by the
Decisions it enables (D02 swap, D05 backup boundary). The behavior it owns is that
`setup` materializes the tree with the right ownership/permissions so a service
can run, the backup boundary holds, and nginx can serve `state/www/` **without**
the DB ever being readable by `www-data`.

**Rejected.**
- *Keep the flat `data/` dir.* It conflates backed-up state with disposable
  runtime data, which is exactly the boundary the whole effort needs to make
  explicit (D05). Rejected.
- *Mirror the full FHS (`bin etc lib libexec var/lib var/cache run`) literally.*
  The `/var` semantics are worth borrowing (D05), but dragging in path depth the
  suite doesn't use is cargo-culting; `state/`/`cache/` carry the meaning without
  it. Rejected.
- *A separate top-level tree for backed-up state (e.g. `/var/lib/<svc>`).* Splits
  one service across two roots and breaks the "`/opt/<svc>/` holds the whole app"
  invariant operators rely on. Rejected.

**Verification.**
- R-3SAU-8T9F — `setup` against a temp `OPSCTL_ROOT` creates `state/`, `cache/`,
  `libexec/`, `bin/`, `etc/`, `backups/` under `/opt/<svc>/`, with `state/` mode
  **`0711`** owned `<svc>:<svc>` (traverse-only — not `0755`, not group/other
  readable), the DB at `0640 <svc>:<svc>`, and `state/www/{public,private}` at
  `0750 <svc>:web`; `cache/` present and writable by `<svc>`. Substrate: real
  filesystem (temp dir).
- R-VB77-BU5O — the access-control property holds: a process whose groups are
  exactly `{web}` (standing in for `www-data`) **can** read a file under
  `state/www/public` and `state/www/private`, and is **denied** reading
  `state/<svc>.db` and **denied** listing `state/` — proving served content is
  reachable while the DB is not. Substrate: real filesystem with real
  uid/gid-dropped reads (not a perms-bit assertion a fake would pass).
- R-VCF3-PLWD — `setup`'s generated per-service nginx fragment serves
  `state/www/public` directly (no `auth_request`) and serves `state/www/private`
  **behind** the `/srv/<svc>/` introspection `auth_request` — the private location
  carries the introspection gate, the public one does not. Substrate: structural
  assertion on the generated fragment (byte/AST), plus an integration check
  through real nginx that an unauthenticated request gets `public/` and is
  refused `private/`.
