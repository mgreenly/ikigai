# Decision 3 — SemVer 2.0 version identity & ordering

**Decision.** A version is **any valid SemVer 2.0 string in canonical
`v`-prefixed form** — `vMAJOR.MINOR.PATCH[-prerelease][+build]`, e.g.
`v0.7.1`, `v0.7.1-rc.2`, `v0.7.1-rc.2+a1b2c3d`. The leading `v` is **required
everywhere** (committed `VERSION` file, `libexec/` filename, `backups/` name, the
binary's `version` verb output, opsctl preflight) — there is no bare form and no
strip/add normalization at internal boundaries.

`golang.org/x/mod/semver` is the **sole** authority; opsctl's hand-rolled
`lessVersion` is deleted. Two distinct uses are kept separate:

- **Identity** — the artifact a path names (`libexec/<svc>-v<full>`,
  rollback/restore target) uses the **full string including `+build`**, so two
  builds of the same core version are distinct, nameable artifacts.
- **Precedence** — ordering for "prior release" (rollback) and "newest N" (prune)
  uses `semver.Compare`, which implements SemVer §11 precedence and **ignores
  build metadata** (`v0.7.1+aaa` and `v0.7.1+bbb` are precedence-equal). When
  precedence ties (same core+prerelease, differing only in `+build`), the
  tiebreak is the `libexec/` file's **mtime** — newest deployed build wins.

Illustrative comparator + validation (opsctl):

```go
// validVersion is the input boundary gate: v-prefixed AND all three numeric
// fields present (x/mod/semver alone accepts "v1"/"v1.2"; we reject partials).
func validVersion(v string) bool // semver.IsValid(v) && exactly MAJOR.MINOR.PATCH present

// compareVersion: precedence first (build metadata ignored), mtime as tiebreak.
func (l Layout) compareVersion(a, b string) int // semver.Compare(a,b), then mtime(LibexecBinary)
```

Every input boundary (`bump` arg/file, `ship`, opsctl deploy preflight) calls the
gate and **fails loudly** on an invalid or partial version; internal code then
trusts the string.

**Rejected.**
- *Keep loose `lessVersion` (split on `.`, numeric compare, string fallback).* It
  silently degrades any `-rel`/`+build` version to lexicographic compare —
  ordering `-rc10 < -rc2` and ordering by SHA (violating "ignore build
  metadata"). The exact cases this design introduces are the ones it gets wrong.
  Rejected.
- *Bare (no-`v`) canonical form.* `x/mod/semver` requires the `v`; a bare form
  means strip/add normalization at every boundary — more code, implicit. Rejected
  for "v everywhere."
- *Use the full string (incl `+build`) for precedence too.* Violates SemVer §10
  and makes ordering depend on an opaque SHA. Rejected; precedence ignores build,
  identity keeps it.

**Verification.**
- R-3X6F-RW87 — `v0.7.10` orders **after** `v0.7.9` (numeric precedence, not
  lexicographic). Substrate: in-process unit.
- R-3YEC-5NYW — pre-release precedence: `v1.0.0-rc.1` < `v1.0.0` (release
  outranks its pre-release) and `v1.0.0-rc.2` < `v1.0.0-rc.10` (numeric
  identifier order, not string). Substrate: in-process unit.
- R-40U4-X7GA — build metadata ignored for precedence: `compareVersion` reports
  `v0.7.1+aaaa` and `v0.7.1+bbbb` as equal-precedence (it does **not** order them
  by the `+` suffix). Substrate: in-process unit.
- R-4221-AZ6Z — two equal-precedence builds are tiebroken by `libexec/` mtime:
  given `v0.7.1+old` (older mtime) and `v0.7.1+new` (newer), "newest" selects
  `+new`. Substrate: real filesystem (temp dir with controlled mtimes).
- R-439X-OQXO — the boundary gate **rejects** each of: a bare version (`0.7.1`),
  a partial version (`v1`, `v1.2`), and a non-SemVer string — opsctl deploy and
  `bump` refuse with a non-zero exit rather than proceeding. Substrate:
  in-process unit (opsctl) + `bin/bump.test.sh` (shell).
