# Decision 9 — Scheduled nightly backup (systemd timer + box sweep)

**Decision.** Backups run **unattended at 03:00 America/Chicago** via a systemd
timer that opsctl writes during `init-box`/`setup` — the same mechanism as the
certbot renewal timer already on the box (`RenewTimerPath`/`RenewServicePath`).
Restore is **never** scheduled (interactive-only, D07): the asymmetry is
deliberate — the non-destructive direction is automated for regularity, the
destructive one is gated on a human every time.

```ini
# ikigenba-backup.timer
[Timer]
OnCalendar=*-*-* 03:00:00 America/Chicago   # 3 AM Central, DST-correct (CST↔CDT)
Persistent=true                              # a missed run (box was down) fires at next boot

# ikigenba-backup.service  → ExecStart=/usr/local/bin/opsctl backup --all
```

`opsctl backup --all` is a **box sweep** that processes services **one at a
time**: for each, `stop · snapshot state/ → S3 · start` (D07). Because the stop is
per-service and sequential, the box is never fully down — each service is offline
only for its own snapshot. The sweep also takes the **apex cert** snapshot
(dashboard's cert stream, D07; no stop — the cert is not app-mutated). The sweep
is **fail-soft**: a failure on one service (or the cert) is logged and the sweep
**continues** to the rest, so one fault never blanks the night. Exit non-zero if
any unit failed (so the timer's status surfaces it), having still backed up the
others.

Stopping a producer/consumer briefly is safe for the event plane: a stop does
**not** re-mint the epoch (only restore does, D06), and peers reconnect from their
cursors via the consumer engine's backoff. Nightly cadence × D07 retention (keep
newest N, default 30) yields N days of recoverable history (≈30 by default).

**Rejected.**
- *Per-service staggered timers.* More units and logs for no gain — the box sweep
  already isolates downtime to one service at a time, and "a single 3 AM backup"
  is the operator's stated mental model. Rejected.
- *cron.* The suite's box scheduling is systemd timers (TZ-aware `OnCalendar`,
  `Persistent`, unit status/journald integration); cron would be a second,
  weaker mechanism. Rejected.
- *Fixed UTC offset (e.g. `08:00 UTC`).* Drifts an hour against Central twice a
  year at DST. `America/Chicago` in `OnCalendar` tracks DST. Rejected.
- *Abort-on-first-failure sweep.* A single wedged service would prevent every
  later service from being backed up that night — the opposite of what a backup
  job should do. Rejected for fail-soft.

**Verification.**
- R-RNKC-HAW8 — the backup timer unit opsctl writes fires
  `OnCalendar=*-*-* 03:00:00 America/Chicago` (the timezone present, **not** a bare
  time or UTC) and sets `Persistent=true`. Substrate: structural assertion on the
  generated unit (written under a temp `SysRoot`, byte/field-checked).
- R-ROS8-V2MX — `opsctl backup --all` is fail-soft: with a stubbed `System` where
  one service's stop/snapshot **fails**, the sweep still backs up every **other**
  service and the cert, and exits non-zero. Substrate: opsctl unit test (stubbed
  `System` + fake `ObjectStore`).
