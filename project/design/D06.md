# Decision 6 — Epoch re-mint by exclusion + the boot-reconstruction invariant

**Decision.** The event-plane generation/epoch sidecar
(`cache/<svc>.db.generation`, event-protocol §9.3) is handled **structurally**,
not by an explicit per-restore delete step:

- The sidecar lives in the **non-state** region (`cache/`, D05), so it is **never
  in a backup archive**.
- `restore` (D07) **replaces `state/` and resets the non-state region to a clean
  slate** (D05's non-state reset). After a restore the sidecar is therefore **absent**.
- `loadOrMintGeneration` already mints a fresh token when the sidecar is absent
  (eventplane `outbox`), so the next boot mints a **new** epoch by construction.

Net: a restore re-mints the epoch because the sidecar *isn't there to restore* and
the non-state reset guarantees no stale one survives — eliminating the
"forgot-to-delete-the-sidecar" class of bug (the current `prompts`/`scripts`
operator scripts have exactly this gap). The explicit `os.Remove(GenerationPath)`
in appkit's `runRestore` becomes redundant insurance for the **local**
`--from`/rollback path, not the load-bearing mechanism for the S3 path.

This makes the model correct **only if** the non-state region is reconstructable —
hence the **boot-reconstruction invariant**: every service must, on startup,
create its own non-state (e.g. `mkdir -p cache/` and any `runs/`-style dirs) and
mint its sidecar, assuming none of it pre-exists. A service that assumes `cache/`
pre-exists would break after a restore wipes it. This invariant is what the
restore drills (D07) continuously prove.

Illustrative (per-service composition root / appkit startup):

```go
// On boot: non-state is assumed absent and is (re)created before use.
ensureDir(cfg.CacheDir)          // mkdir -p; restore may have removed it
// generation sidecar minted lazily by the outbox when absent (existing behavior)
```

**Rejected.**
- *Re-mint via an explicit `rm <svc>.db.generation` in the restore path.* What the
  status quo does; it is a step each restore implementation must remember, and
  `prompts`/`scripts` already forgot it. Making absence the default removes the
  failure mode entirely. Rejected as the primary mechanism (kept only as local-path
  insurance).
- *Store the sidecar inside `state/` and re-mint by editing it on restore.* Puts
  the epoch in the backed-up set, so a restore would faithfully bring back a
  **stale** epoch — the precise bug to avoid. Rejected.

**Verification.**
- R-4BT8-D54J — boot after a restore (sidecar absent) mints a generation token
  that differs from the token present before the restore. Substrate: real
  filesystem + eventplane `outbox` (`loadOrMintGeneration`).
- R-4D14-QWV8 — a consumer cursor minted under the pre-restore epoch is rejected
  with `stale-epoch` at connect after the producer re-mints, forcing a resync
  (not a silent resume onto reused seqs). Substrate: **real eventplane** — the
  actual `outbox.FeedHandler()` wired to an `httptest` server and the real
  `consumer` engine (the eventplane test pattern), not a stub.
- R-4E91-4OLX — starting a service against an **empty** `/opt/<svc>/` non-state
  region (no `cache/`, no sidecar) succeeds: the service recreates the dirs and
  mints the sidecar, and reaches a healthy state. Substrate: real filesystem +
  the service binary (boot smoke).
