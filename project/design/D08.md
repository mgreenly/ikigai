# Decision 8 — Per-service adoption & live-box migration

**Decision.** This redesign touches **every** service; D08 owns the uniform
adoption contract and the one-time migration of the live `int` box. Per service:

1. **Layout move.** DB → `state/<svc>.db`; generation sidecar →
   `cache/<svc>.db.generation`; durable file space (if any) under `state/`;
   everything else stays non-state (D05). Set `<APP>_DB_PATH` /
   `<APP>_GENERATION_PATH` accordingly in the unit's exported env.
2. **Boot reconstruction.** The service recreates its non-state on startup (D06)
   so a restore can wipe it.
3. **Binary placement.** Built and deployed as `libexec/<svc>-v<semver>` with
   `bin/run` selecting it (D02); `VERSION` is `v`-prefixed (D03/D04).
4. **Backup/restore.** Inherits `opsctl backup/restore <svc>` (D07) — no
   per-service script.

Services with extra durable data make a **conscious classification** (the point of
the model): `prompts` (`sandboxes/` durable → `state/`; `runs/` logs → decide),
`scripts` (`runs/` trees → decide), `sites` (its `/opt/sites/www/{served/public,
served/private,working}` → `/opt/sites/state/www/{public,private}` (+ any working
content under `state/`); the `web` group + nginx fragment repointed per D01),
`dashboard` (apex TLS cert via the dedicated cert stream, D07).

**Live-box migration** (sequenced, since `int.ikigenba.com` is the live account
and is touched only on an explicit deploy request): opsctl gains a one-time
conversion that, per service, moves `data/<svc>.db*` → `state/`, moves the sidecar
→ `cache/`, converts `releases/`+`current` → `libexec/`+`bin/run`, and repoints
env — **idempotent** (safe to re-run; a half-converted tree completes) and
**non-destructive** (no data dropped). The existing OLD→new conversion precedent
in opsctl (the legacy `bin/run` wrapper-script handling) is the model.

This is largely an **orchestration/coordination** decision; its per-mechanism
proofs live in D01–D07. The behavior it owns is that the adoption is complete and
the migration is safe.

**Rejected.**
- *Adopt a few services and leave the rest on the old layout.* The whole value is
  uniformity (one backup path, one layout); a partial rollout keeps the
  3-incompatible-ways problem alive. Rejected — every service adopts.
- *Big-bang in-place rename on the live box with no idempotent converter.* A
  failed mid-rename leaves a service unbootable with no safe re-run. Rejected for
  an idempotent, non-destructive converter.

**Verification.**
- R-4LKF-FB23 — for **each** service, a freshly set-up `/opt/<svc>/` (DB under
  `state/`, sidecar under `cache/`, binary under `libexec/`, `bin/run` symlink)
  boots and passes its `health` check. Substrate: the running suite (per-service
  boot smoke; `bin/start` + health, or the service binary against a temp root).
- R-4MSB-T2SS — the one-time converter, run against a fixture of the **old**
  layout (`data/<svc>.db`, `releases/<v>/`, `current`), produces the new layout
  with the DB intact under `state/` and the live binary reachable via `bin/run`,
  and re-running it is a no-op (idempotent). Substrate: opsctl unit test against a
  temp `OPSCTL_ROOT` seeded with the old layout.
