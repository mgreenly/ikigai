# Decision 7 вЂ” opsctl-owned `backup` / `restore` (stopВ·snapshotВ·start + S3)

**Decision.** Backup/restore to S3 becomes a **first-class opsctl verb**, uniform
across **all** services вЂ” `opsctl backup <svc>` / `opsctl restore <svc>`. opsctl
is the only on-box actor that can `systemctl` (the service binary runs as the app
user and cannot), so the stop/start orchestration must live here. This **retires**
the three per-service `bin/backup`/`bin/restore` shell scripts (crm, prompts,
scripts) and the dashboard's in-binary `Spec.Backup`/`Spec.Restore` S3 branch;
the appkit `backup --out`/`restore --from` **local** verb is untouched (it remains
opsctl deploy/rollback's pre-migration snapshot, D-independent).

**Backup** = stop unit В· archive `state/` (D05) в†’ upload В· start unit. S3 layout
keeps the good crm/dashboard conventions (not prompts/scripts' historyless sync):

```
s3://$IKIGENBA_BACKUP_BUCKET/<svc>/
  <svc>.<UTC>.tar.gz          immutable, one per run
  latest                      pointer в†’ newest snapshot key
  pre-restore/<svc>.<UTC>.tar.gz   pre-restore safety copies
```
Count-based retention (keep newest N, operator-configurable; **N defaults to 30**
вЂ” the product contractual constant, used here by value) on the main prefix;
`latest` and `pre-restore/` never pruned by the main pass.

**Restore** = resolve key (default `latest`, or an explicit key) В· **always
interactive, no `--yes` bypass** В· push a pre-restore safety snapshot of current
`state/` В· stop unit В· **replace `state/`** from the snapshot В· **reset the
non-state region** (D05/D06: `cache/`, sidecar gone) В· start unit.

**Apex TLS cert (dashboard / the DEFAULT app).** The one durable artifact outside
`/opt/<svc>/` вЂ” the apex Let's Encrypt cert at `/etc/letsencrypt/` вЂ” is backed up
and restored as a **dedicated snapshot stream** alongside dashboard's state
stream, because **reissue does not scale for mass recovery**: every
`<account>.ikigenba.com` is a subdomain of the single registered domain
`ikigenba.com`, and Let's Encrypt's 50-certs/week limit is **per registered
domain**, so a fleet-wide rebuild that reissued certs would exhaust the shared
quota. Restoring a cert from S3 consumes **zero** issuance quota вЂ” it is the
rate-limit-safe recovery path. `init-box`'s certbot issuance (D-independent)
remains only the cold-start fallback for a box with no backup yet.

```
s3://вЂ¦/dashboard/cert/cert.<UTC>.tar.gz   tars /etc/letsencrypt/{archive,renewal,live}/<domain> (paths relative to /)
s3://вЂ¦/dashboard/cert/latest              pointer, resolved INDEPENDENTLY of the state pointer
```

- Backup produces the cert object **in addition to** the state object; the cert
  stream has its own `latest` and retention.
- A dashboard restore lays the cert back via `tar -C /` and **never invokes
  certbot** (no reissue) вЂ” and resolves `cert/latest` independently of the chosen
  state snapshot, so a DB rollback never drags the cert backward.
- **Secrets:** the cert archive contains the TLS **private key**
  (`archive/<domain>/privkey*.pem`). opsctl handles these as **opaque bytes вЂ”
  never read, printed, or logged**. The backup bucket's privacy is therefore
  load-bearing (it now holds private-key material).

**Testability seam.** S3 sits behind an `ObjectStore` interface so opsctl logic
and the backup/restore round-trip are proven **in-gate, unprivileged, with no
external services**: the retention / `latest` / pre-restore logic and the full
`state/` archive в†’ `Put` в†’ `Get` в†’ restore round-trip run against the in-memory
`fakeStore` with a **real `tar` and real filesystem**, proving the archive format
and the `ObjectStore` contract. That `aws` actually transfers bytes to a real S3
bucket is a separate **on-box/manual** check outside the green gate (never a
`t.Skip`-gated test, which would count as uncovered):

```go
type ObjectStore interface {
    Put(ctx, key string, r io.Reader) error
    Get(ctx, key string, w io.Writer) error
    List(ctx, prefix string) ([]ObjInfo, error) // ObjInfo{Key, Size}
    Delete(ctx, key string) error
}
// awsCLIStore: shells `aws s3 / s3api` using the box instance role; region from
// IKIGENBA_AWS_REGION (default us-east-2), bucket from IKIGENBA_BACKUP_BUCKET.
// fakeStore: in-memory, for opsctl unit tests of retention/latest/round-trip logic.
```

The stop/start path uses the existing `System` seam. The unattended cadence вЂ” the
`opsctl backup --all` box sweep and its 3 AM timer вЂ” is **D09**; restore is never
scheduled.

**Rejected.**
- *Keep S3 in per-service `bin/*` scripts.* Three incompatible implementations
  (immutable-snapshot vs `s3 sync` mirror) and they can't `systemctl` from the
  workstation cleanly; consolidating in opsctl gives one implementation and real
  stop/start. Rejected.
- *Fold S3 into each binary as the `backup` verb (the dashboard path the package
  doc anticipated).* The binary cannot stop its own unit, and the D05 layout
  convention removes the only reason in-binary made sense (knowing its own data
  shape). Rejected вЂ” opsctl owns it.
- *`s3 sync --delete` mirror (prompts/scripts model).* No immutable history, no
  `latest`, no retention, no pre-restore safety, and it bypasses the epoch
  re-mint. Rejected for the immutable-snapshot convention.

**Verification.**
- R-4GOT-W83B вЂ” `opsctl backup` invokes `System.Stop(<svc>)` **before** taking the
  snapshot and `System.Start(<svc>)` **after** (ordering asserted via the seam);
  on a snapshot failure it still restarts the unit. Substrate: opsctl unit test
  with stubbed `System` + fake `ObjectStore`.
- R-4HWQ-9ZU0 вЂ” `opsctl backup` puts an immutable timestamped object **and**
  updates `latest` to that key; with N+1 snapshots present, the retention pass
  deletes the oldest down to N and never deletes `latest` or `pre-restore/`.
  Substrate: opsctl unit test against the fake store (object placement, `latest`
  update, retention) with a **real `tar`/filesystem** producing the archived
  object вЂ” unprivileged, in-gate. (That `aws` actually accepts the object and
  `latest` reads back is the on-box check, outside the green gate.)
- R-4J4M-NRKP вЂ” `opsctl restore` refuses to proceed without interactive
  confirmation (no `--yes` path exists), and on confirmation pushes a
  `pre-restore/` snapshot of the current `state/` **before** replacing it.
  Substrate: opsctl unit test (confirmation gate) + fake store (pre-restore push).
- R-4KCJ-1JBE вЂ” full round-trip: `backup` then `restore` reproduces `state/`
  byte-for-byte (DB + durable files) through the `ObjectStore` seam with a **real
  `tar` and real filesystem**, proving the archive format and the `ObjectStore`
  contract round-trip. Substrate: opsctl test, real `tar`/filesystem +
  `fakeStore`; unprivileged, in-gate. (Real-`aws` wire interop is the on-box
  check.)
- R-TAOX-5LKS вЂ” `opsctl backup dashboard` produces a cert object under
  `dashboard/cert/` **and** its own `cert/latest`, distinct from the state
  object/pointer (the two streams are independently keyed). Substrate: opsctl unit
  test (fake store) + real `tar`/filesystem for the cert archive; unprivileged,
  in-gate. (Real-`aws` interop is the on-box check.)
- R-TBWT-JDBH вЂ” a dashboard restore lays `/etc/letsencrypt/{archive,renewal,live}/
  <domain>` back from `cert/latest` matching the backed-up bytes, and does so
  **without invoking certbot** (no reissue вЂ” `System`'s certbot/issuance seam is
  never called during restore), and resolves the cert pointer independently of the
  chosen state snapshot. Substrate: real filesystem round-trip (cert tree) +
  stubbed `System` asserting no issuance call; cert bytes via the `ObjectStore`
  seam (`fakeStore`). Unprivileged, in-gate.
