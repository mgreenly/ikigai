# Decision 11 — Web surface from `share/www` through the chassis (de-embed)

## Problem.

webhooks's landing page and static assets are embedded in the binary
(`//go:embed landing.html static` in `internal/web/embed.go`) and served by a
service-side `internal/web` package (`LandingHandler` + `StaticHandler`), like
every service's original shape. The suite is reverting that: web assets are
release artifacts that belong on disk — visible and diffable in the versioned
release tree, editable in dev without a rebuild, served by one chassis mechanism
instead of twelve drifting copies. crm proved the conversion; notify followed.
This Decision is webhooks's turn. It is the service-side half of appkit's chassis
web design (its D5 www-root resolution, D6 `appkit/web`, D7 `Spec.WWW` /
`Router.WWW()` — consumed here as a fixed external contract). D9 (the landing
page's content, template, and Carbon assets) is rewritten in place to describe
the on-disk `share/www` shape; this Decision owns the **de-embed conversion
itself**.

## Decision.

webhooks serves its web surface from the on-disk www root.

**1. The files move (same bytes).**
`webhooks/internal/web/landing.html` → `webhooks/share/www/landing.html`;
`webhooks/internal/web/static/` → `webhooks/share/www/static/` (byte-identical:
`tokens.css` and the four woff2 fonts). `bin/ship` already bundles
`webhooks/share/` into the release `share/<version>` tier; on the box the chassis
resolves the root to `share/current/www`, in dev to `./share/www` or the
`WEBHOOKS_WWW_PATH` override.

**2. The Spec opts in.** `cmd/webhooks/main.go`'s `webhooksSpec()` sets
`WWW: true`. The chassis loads the site at serve (failing startup loudly if the
root is missing), auto-mounts `GET /static/` for this path-routed service, and
exposes the loaded site as `rt.WWW()`.

**3. The landing route renders through the chassis.** The `GET /{$}` mount in
`Handlers` uses a few-line closure over `rt.WWW().Render("landing.html", …)`
(the D9 handler shape), passing `rt.Service()` / `rt.Version()`:

```go
func landingHandler(site *web.Site, service, version string) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        _ = site.Render(w, "landing.html",
            struct{ Service, Version string }{service, version})
    })
}
// … in Handlers:
rt.Handle("GET /{$}", landingHandler(rt.WWW(), rt.Service(), rt.Version()))
```

The service-side `GET /static/` mount is **deleted** — the chassis owns it.

**4. `webhooks/internal/web/` is deleted** — `web.go`, `embed.go`,
`landing.html`, `static/`, and its tests (`web_test.go`, `nginx_test.go`). The
landing/asset content tests move to the composition-root package (`cmd/webhooks`),
loading the repo-real `share/www` tree via `appkit/web.Load` (a relative path
from the package dir) so they exercise the exact files that ship. The nginx
fragment tests relocate to `cmd/webhooks` unchanged in their assertions (D10's
registry-derived `proxy_pass` expectations move with them). The still-untagged
mux shadowing test (`TestServeMuxLandingAndStaticDoNotShadowExistingRoutes`)
relocates too, rebuilt against the chassis static mount and the `GET /{$}`
landing route.

**5. Dev wiring (boundary-crossing).** `bin/start`'s `launch_webhooks` exports
`WEBHOOKS_WWW_PATH="$repo/webhooks/share/www"` beside its existing
`WEBHOOKS_DB_PATH` export, so a repo-root-launched dev binary finds the source
tree. **This one line crosses the `webhooks/` boundary** — it is the single
sanctioned exception, verified by the live `bin/start` smoke, **not** the Go
suite.

**6. Nothing else moves.** The nginx fragment (D7), the landing markup and tokens
(D9), the MCP surface, the ingress, and the migrations are untouched. The
embedded migrations (`internal/db`) stay embedded — they are not web assets.

## Rejected.

- **Keeping the embed.** The suite-wide reversion this work completes; embedded
  assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root.** That tree is the sites service's nginx-served user
  content, backed up with `state/`; webhooks's UI assets are release artifacts
  and live in the shipped `share` tier (appkit D5 records the full argument).
- **Keeping a thin `internal/web` package.** With the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition.** Two sources of
  truth; the conversion is atomic within one service.

## Verification.

(Over the repo-real `webhooks/share/www` tree, resolved relative to the test
package. D9's R-TMJH/R-TNRE/R-TOZA/R-TQ77/R-TRF3 continue to pin the
page/asset/structure/FOUT behaviors through the new substrate.)

- R-0GVM-EPJ9 — an `appkit/web` Site loaded from `webhooks/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html`
  with an injected service and version visible in the output — proving the
  shipped template on disk, not an embedded copy, is the page source. (A page
  still sourced from a `//go:embed` in webhooks would not be reachable through
  `appkit/web.Load(share/www)`.)
- R-0I3I-SH9Y — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  webhooks-side static handler registered — proving the assets ship on disk and
  the chassis serves them (a surviving service-side `StaticHandler` mount would
  make this ambiguous; there is none to register).
