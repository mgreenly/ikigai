# Decision 15 — Event-routing conformance: kind `received`, subject = `/` + hook name

> **⛔ EXTERNAL ORDERING (operator-sequenced, not built by any webhooks
> phase).** This Decision consumes the **revised eventplane API** —
> `outbox.Event{Kind, Subject, Payload}`, `outbox.Family`/`Registry`, the
> kind/subject envelope and canonical-key SSE framing — specified in
> `eventplane/project/design/` (its D1–D4) and realized by **eventplane plan
> phases 01–04, which must be BUILT before webhooks' conformance phase runs**.
> It equally assumes appkit (which holds `Spec.Events outbox.Registry` and the
> chassis reflection tool over it) compiles against that revision — appkit's
> own conformance is appkit's spec, out of scope here. Until both land,
> webhooks' conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and the
old type names drop their redundant noun prefix (`source` already says
webhooks deals in webhooks). Crucially for webhooks, the revision gives this
service its missing routing dimension: today a consumer that cares about one
hook must be triggered by **every** `webhook.received` event on the box,
because the discriminating `name` lives only in the payload, where filters
cannot see it. It is a hard cutover — no compatibility period — inside the
no-live-data migration window.

**What webhooks publishes today (ground truth, not vacuous).** webhooks is a
real, wired producer: `Spec.Feed: "/feed"`, `Spec.Events: webhooks.Events`
(an `outbox.Registry` of one type in `internal/webhooks/events.go`), the
`Producer` hook injecting the outbox into the `Service`, and one emission
path — `Service.Record`, called from the D4 ingress handler on every accepted
`POST /in/<name>` — appending
`outbox.Event{Type: "webhook.received", Payload: raw}` on the same
transaction as the `last_triggered_at` touch, `Ring()` after commit. The one
published type converts.

**Decision.** webhooks adopts the revised envelope exactly over what it
publishes, with the **hook name as the subject** — the routing doc's key map
records webhooks' direction as `webhooks:received/<hook name>`:

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `webhook.received` | `webhooks:received/deploy-hook` | `received` | `/` + the hook `name` |

The subject is `"/" + wh.Name` — the same stored name the payload carries and
the public URL routes on. **Subject-safety is structural:** D3's
`validateName` admits only `^[A-Za-z0-9_-]{1,64}$` (default names are
`ids.New()` base32, a subset), so every subject is a single-line, `/`-rooted,
single-segment path — always valid under `routing.ValidSubject` — and
contains no `/` and no glob metacharacters (`*`, `?`, `[`), so a literal
filter `webhooks:received/deploy-hook` selects exactly one hook and
`webhooks:received/*` selects every hook (one segment, `*` never crosses
`/`). Payload fields (`name`, `owner`, `received_at`, `content_type`,
`body`) are **unchanged** — the routing key addresses; the payload remains
the detail record (the payload's `name` is a detail field consumers read, not
a kind discriminator, so nothing is dropped).

1. **Constant and emission site** (`internal/webhooks/events.go`):
   `eventWebhookReceived = "webhook.received"` → `kindReceived = "received"`
   (still declared once, shared by the emit site and the registry).
   `Service.Record` appends
   `outbox.Event{Kind: kindReceived, Subject: "/" + wh.Name, Payload: raw}`.
   Nothing else in the D5 seam changes: same single transaction as
   `TouchLastTriggered`, `Ring()` best-effort after commit,
   durable-before-ack `202` untouched (D4/D5).

2. **The registry becomes one family** (`internal/webhooks/events.go`):
   `webhooks.Events` becomes an `outbox.Registry` of one `outbox.Family`
   entry — `{Kind: "received", Subject: "/<hook name>", Description: …,
   Sample: webhookReceivedPayload{…}}` (the same filled Sample).
   `Spec.Events` wiring is untouched; Append-time kind gating and the chassis
   reflection surface hang off the same registry, now family-shaped.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`;
   each producer re-applies the DDL through its own runner. webhooks appends
   **one new timestamped migration** minted with
   `bin/create-migration webhooks outbox_routing` (never a hand-picked
   number) that drops and recreates the outbox table per the revised
   `outbox.SchemaSQL` verbatim (a `DROP TABLE`/`DROP INDEX` preamble + the
   constant's DDL). Outbox rows are disposable retention data inside the
   migration window; nothing is migrated across. The frozen `003_outbox.sql`
   is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest**
   outbox migration, asserting it contains the revised `outbox.SchemaSQL`
   verbatim — this supersedes D2's old byte-equality claim against `003`
   (its id is retired to plan history). (This exactly mirrors gmail's
   D18/phase-16, dropbox's D22/phase-23, and ledger's D15/phase-13
   handling.)

4. **Reflection/consumer surface.** The chassis reflection tool
   (appkit-owned) reflects the family registry; webhooks' observable is that
   its one kind — and only it — is listed with the `/<hook name>` subject
   description. webhooks consumes nothing (producer only), so there is no
   filter/`consumer.Config` side to conform.

**Rejected.**
- **Keeping the `webhook.` prefix as the kind (`webhook.received`)** — the
  revision's explicit rename rule: `source` carries the noun; a
  `webhooks:webhook.received` key says "webhook" twice.
- **An empty subject (gmail's choice)** — webhooks has an obvious,
  already-validated routing name; leaving the subject empty would preserve
  the fire-on-everything failure the revision exists to fix (D5 rejected
  per-name *types* precisely because the type set had to stay closed — the
  subject is where the per-name precision now lives).
- **Dropping the payload's `name` field (deduplicating against the subject)**
  — the payload is the detail record and its shape is pinned unchanged;
  unlike dropbox's dropped `event` field, `name` duplicates a routing *name*,
  not a kind discriminator that could drift.
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are retention data with no live
  consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: the emission claims run
`Service.Record` over the real `eventplane/outbox` and a real temp-file
SQLite built by the full embedded migration set, reading outbox rows back by
SQL; the feed claim runs the real eventplane `FeedHandler` over `httptest`).

- R-A3FB-J3ZK — after `Record` fires for two webhooks named `deploy-hook`
  and `alpha_1`, the outbox rows read back by SQL from the real SQLite
  database carry `kind` exactly `received` (no `webhook.` prefix) and
  `subject` exactly `/deploy-hook` and `/alpha_1` respectively (`/`-rooted,
  one row each), each appended on the same transaction as that webhook's
  `last_triggered_at` touch (baseline atomicity preserved), with the payload
  JSON keys exactly `name`, `owner`, `received_at`, `content_type`, `body`
  (unchanged).
- R-A4N7-WVQ9 — `webhooks.Events` declares exactly one family with kind
  `received` and subject description `/<hook name>`; through the assembled
  MCP handler, reflection's index lists exactly that one publish, the detail
  of `received` returns a schema and example that agree (every example field
  in the schema, `webhookReceivedPayload`-shaped), and an undeclared kind
  returns the typed unknown-kind error naming the one declared kind.
- R-A5V4-ANGY — applying webhooks' full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table
  with `kind` and `subject` columns and **no** `type` column; the DDL drift
  guard asserts the newest outbox migration contains the revised
  `outbox.SchemaSQL` verbatim while `003_outbox.sql` remains byte-identical
  to its committed (frozen) body.
- R-A730-OF7N — end-to-end over webhooks' own substrate: an event appended
  through the producer path (`Service.Record` → `Append` on the
  migration-created table, source `webhooks` — the `Spec.App` value) for a
  hook named `deploy-hook` and served by the real eventplane `FeedHandler`
  over an `httptest` connection frames `event: webhooks:received/deploy-hook`
  (the canonical key) with a `data:` JSON envelope carrying `kind` and
  `subject` keys and **no** `type` key.
