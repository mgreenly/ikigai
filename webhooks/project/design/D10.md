# Decision 10 тАФ Adopt `registry`: resolve webhooks's own loopback port by name, and guard the deploy artifacts against drift

## Problem.

webhooks writes its own loopback port as a bare literal in three places in its
Go source: the `Port: 3006` integer in the composition-root `Spec`
(`cmd/webhooks/main.go`), and two `127.0.0.1:3006` address strings тАФ the
`proxy_pass` assertions in `internal/web/nginx_test.go` and the `loopback`
constant in `internal/e2e/e2e_test.go`. "Which port webhooks answers on" is
duplicated across the suite this way and drifts silently: a renumber only
surfaces at deploy, uncatchable by the compiler. The shared `registry` library
(`module registry`, the repo-root leaf module) is the single source of truth тАФ
its table already carries `{"webhooks", 3006, Core}` тАФ and webhooks should ask
the table by name instead of repeating the number.

## Decision.

webhooks references itself **by name** through `registry` and proves, with
tests, that no port literal survives in its Go source.

**1. Own port by name at the composition root.** `Spec.Port` becomes a name
lookup in the strict, panic-on-unknown form (an unknown own-name is a build-time
programming error, not a recoverable condition):

```go
Port: registry.MustPort("webhooks"),   // was: Port: 3006
```

Resolution happens **once, at boot** тАФ appkit reads `Spec.Port` once; no handler
or request path calls `registry`. This is **behavior-preserving**: the table
carries `webhooks=3006`, so the resolved value is byte-identical to the literal
it replaces.

**2. Module wiring (inside `webhooks/` only).** `webhooks/go.mod` gains
`require registry v0.0.0` and the committed `replace registry => ../registry`,
mirroring the existing `appkit` / `eventplane` in-repo replace-siblings.

**3. A source-scan guard** proves no bare `127.0.0.1:30xx` loopback-address
string literal remains anywhere in webhooks's Go source. A test walks every
`*.go` file under the `webhooks/` module root and fails on any file whose source
contains the loopback-port address form (`127.0.0.1:30` followed by two digits).
Two constraints make it robust: it **assembles the forbidden needle at runtime**
(e.g. `"127.0.0.1:" + "30"`) rather than embedding the full literal, and **skips
its own filename**, so the guard can never trip itself. It scans real Go source,
not prose тАФ a doc comment that merely names a port in words (no address literal)
does not trip it. After this Decision's rewrites there are zero such literals, so
the scan is green; reintroducing a hardcoded loopback URL turns it red. The
guard deliberately does **not** flag the bare integer `3006` (an int literal is
ambiguous and noisy to grep); the own-port integer is eliminated by change (1)
and proven gone by change (4).

**4. A deploy-artifact drift guard** re-points webhooks's own tests at
`registry` so their expected port is *derived*, not hardcoded, for the two static
artifacts that legitimately still carry the literal port (nginx cannot call Go;
the manifest is generated deploy metadata):

- **`cmd/webhooks/main_test.go`** тАФ the manifest byte-equality test
  (`TestManifestLibraryByteEqualsCommittedFile`) emits from `webhooksSpec()`,
  whose `Port` is now `registry.MustPort("webhooks")`. The emitted `PORT=3006`
  still byte-matches the committed `etc/manifest.env` today; a `registry`
  renumber makes `manifest.Emit` produce a different `PORT` line and the
  byte-equality assertion fails, surfacing the stale artifact.
- **the nginx fragment tests** (`nginx_test.go`, in `internal/web` until D11
  relocates it to `cmd/webhooks`) тАФ the exact-match and static-location tests
  assert the `proxy_pass` upstream. Build the expected origin from
  `registry.BaseURL("webhooks")` (== `"http://127.0.0.1:3006"`) and assert the
  fragment contains `"proxy_pass " + registry.BaseURL("webhooks") + "/;"` and the
  `"тАж/static/;"` variant. The assertion holds today; a renumber makes the
  still-literal `etc/nginx.conf` disagree with `registry.BaseURL("webhooks")` and
  the test goes red.

Both re-pointed tests keep their **existing** behavioral assertions (the session
gate, the exact-match vs prefix distinction, the manifest's non-port fields);
only the **port value** they compare against stops being a hardcoded number and
becomes a `registry` call. The `loopback` constant in `internal/e2e/e2e_test.go`
is likewise rebuilt from `registry.BaseURL("webhooks")` so the last
`127.0.0.1:30xx` string literal leaves the source.

**Scope.** Tests + the `Spec.Port` line + `go.mod` only. No production behavior
change beyond the resolved-identical port, no new package required, **no schema
change / no migration**, and no edit to `etc/nginx.conf` or `etc/manifest.env`
themselves тАФ their literals stay; the tests now police them. Nothing outside
`webhooks/` is touched (the repo-root `go.work use ./registry` and the
`registry` module are preconditions owned at the root, already satisfied).

## Rejected.

- **Per-request `registry` resolution / a lookup in a handler.** The table is
  compile-time constant; resolve at the composition root, store the value, reuse
  it. Per-request lookup buys nothing.
- **`registry.Port("webhooks")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error; `MustPort`'s panic is the
  correct fail-loud form.
- **Regenerating `etc/nginx.conf` / `etc/manifest.env` from `registry`.** Out of
  scope: their generation is owned elsewhere. A drift-catching test delivers the
  safety without a codegen step.
- **An AST/int-literal scan for the bare `3006`.** Too noisy to be falsifiable тАФ
  `3006` carries no context marking it a port. Deriving the manifest and nginx
  expectations from `registry` proves the own-port literal is gone far more
  precisely.
- **Leaving the nginx/e2e/manifest tests as literal `3006` assertions.** They
  would keep bare `30xx` literals in the source (violating this Decision's
  end-state) and, worse, keep *passing* after a renumber тАФ the exact silent drift
  this adoption removes.
