# Decision 6 — The issue protocol: labels, the check gate, and runner-side GitHub I/O

**Decision.** Everything GitHub-facing is **runner-side, deterministic Go**
(`internal/repos/protocol.go`); the agent only does the work. The protocol
drives the github service — never GitHub directly — through a thin loopback
client:

```go
type GitHubPeer struct {                    // internal/repos/ghpeer.go
    baseURL string                          // registry.BaseURL("github")
    http    *http.Client
}
// one method per consumed verb, each a JSON-RPC tools/call POST to /mcp with
// asserted identity headers X-Owner-Email: <session owner>, X-Client-Id: "repos:<session id>"
func (p *GitHubPeer) IssueGet(ctx, id Identity, repo string, n int) (Issue, error)
func (p *GitHubPeer) IssueComments(ctx, id Identity, repo string, n int) ([]Comment, error)
func (p *GitHubPeer) IssueComment(ctx, id Identity, repo string, n int, body string) error
func (p *GitHubPeer) LabelAdd(ctx, id Identity, repo string, n int, labels []string) error
func (p *GitHubPeer) LabelRemove(ctx, id Identity, repo string, n int, label string) error
func (p *GitHubPeer) PRCreate(ctx, id Identity, repo, title, head, base, body string) (PR, error)
```

(Consumed contract: github D9 verbs + D2 loopback identity assertion. On this
box a loopback caller asserts identity headers; nginx owns them at the front
door.)

**Label lifecycle** (issue-driven sessions; label names are the product
constants):

1. **Ack, at admission:** swap `execute` → `executing` (`LabelRemove` +
   `LabelAdd`) and comment the session id — the ack doubles as the visible
   double-trigger guard. A session that only queued (D5 gates) comments
   `queued` once at enqueue instead, and acks when admitted.
2. **Success:** push the branch (D4), `PRCreate` with head
   `ikibot/issue-<N>[.k]`, base = default branch, body carrying `Fixes #<N>`,
   the session id, and the check summary; comment the PR link; remove
   `executing` (and `failed` if present).
3. **Failure** (engine error, TTL, check failure): push the branch anyway
   when it has commits (the crime scene stays inspectable), swap `executing`
   → `failed`, comment the reason (for a check failure: the check's tail
   output).
4. **Retry** is a fresh `execute` label: intake (D3) starts a new session
   (attempt+1); its ack removes `failed`.
5. Manual (MCP-started) sessions skip all label traffic; they still push and
   open a PR (no `Fixes` line — no issue), with the same check gate.

**The check gate — the in-repo definition of done.** After the agent's
conversation ends cleanly, the runner (never the transcript) decides:

- `worktree/.ikibot/check` exists and is executable → the runner runs it with
  cwd = the worktree, remaining session TTL as its bound, capturing combined
  output to `state/sessions/<id>/check.log`. Exit 0 → success path (PR). Non-
  zero → failure path: branch pushed, **no PR**, `failed` + check output
  comment.
- No check declared → success path, and the PR body states **"no check
  declared"** — the agent's self-assessment (its final message) is the body's
  summary either way.
- A session whose agent produced **no commits** is a failure ("no commits
  produced"), nothing pushed, no PR.

**Rejected.**
- **Agent-performed push/PR with scoped credentials** — the settled
  credential-less division of labor: issue text is untrusted input to the
  agent; the runner holding all GitHub I/O makes exfiltration a non-path.
- **Trusting the transcript's claim that checks passed** — the runner
  executes `.ikibot/check` itself; a transcript is an account, not evidence.
- **Draft PRs on check failure** — a failed check means no PR at all; a
  half-artifact invites merging unverified work. The pushed branch is the
  inspection surface.
- **`issue_update` whole-set label replace** — race-prone for a label state
  machine (concurrent human labeling would be clobbered); atomic
  `label_add`/`label_remove` verbs were added to github (its D9) precisely
  for this.
- **Talking REST to GitHub directly from repos** — the github service is the
  suite's single GitHub API actor and token custodian; two actors means two
  credential custodies and two rate-limit owners.

**Verification** (substrate: a recording `httptest` stub implementing the
github `/mcp` wire shape — asserting exact verb names and arguments — plus
real git fixture remotes and a scripted fake engine; the real-GitHub half of
the contract is proven on github's side, and live end-to-end runs against
`bin/start`).

- R-FDAF-MVIC — at admission of an issue-session the peer receives, in
  order: `label_remove` (`execute`), `label_add` (`executing`), and an
  `issue_comment` containing the session id — each call carrying
  `X-Owner-Email` = the session's owner and `X-Client-Id` =
  `repos:<session id>`.
- R-FEIC-0N91 — a session whose scripted agent commits and whose fixture repo
  declares a passing `.ikibot/check`: the branch is on the remote, exactly
  one `pr_create` was received with head `ikibot/issue-<N>`, base the default
  branch, and a body containing `Fixes #<N>` and the session id; then a
  PR-link comment and `label_remove` (`executing`) follow; the session row
  ends `succeeded` with `pr_url` set.
- R-FFQ8-EEZQ — same setup with a failing check: the branch **is** pushed, no
  `pr_create` occurs, the labels end swapped to `failed`, the failure comment
  contains the check output's tail, `check.log` holds the full combined
  output, and the session row ends `failed`.
- R-FGY4-S6QF — a fixture repo with no `.ikibot/check`: the PR is created and
  its body contains "no check declared"; the check gate never executes
  anything.
- R-FI61-5YH4 — a scripted agent that produces no commits: nothing is pushed,
  no PR, `failed` with reason "no commits produced".
- R-FKLT-XHYI — a retry after a recorded failure: the new session's ack
  removes the `failed` label, and its work lands on `ikibot/issue-<N>.2`.
- R-FLTQ-B9P7 — a manual session (no issue): zero label or comment calls
  reach the peer; the PR is created without a `Fixes` line; the check gate
  behaves per the fixture exactly as in the issue-driven cases.
