# Decision 8 — Events: the session-outcome families

**Decision.** repos is an event-plane producer (`Spec.Feed: "/feed"`,
`Spec.Events: repos.Events`, outbox injected via the `Producer` hook). v1
publishes exactly two families — session outcomes — declared in
`internal/repos/events.go`:

```go
var Events = outbox.Registry{
    {Kind: "session.succeeded", Subject: "/<repo name>", Description: …, Sample: sessionOutcomePayload{…}},
    {Kind: "session.failed",    Subject: "/<repo name>", Description: …, Sample: sessionOutcomePayload{…}},
}

type sessionOutcomePayload struct {
    Repo        string `json:"repo"`
    SessionID   string `json:"session_id"`
    IssueNumber *int   `json:"issue_number,omitempty"`
    Branch      string `json:"branch"`
    PRURL       string `json:"pr_url,omitempty"`   // succeeded only
    Error       string `json:"error,omitempty"`    // failed only
    EndedAt     string `json:"ended_at"`           // RFC3339Nano UTC
}
```

Canonical keys: `repos:session.succeeded/<repo>` /
`repos:session.failed/<repo>` — subject = `/` + repo name (a GitHub repo
name: `[A-Za-z0-9_.-]+`, no `/`, no glob metacharacters, so a literal filter
selects one repo and `*` selects all). A `cancelled` session emits
`session.failed` with error `cancelled` — consumers care that work did not
land, not why, and a third family would push taxonomy into every filter.

Emission is atomic with the terminal write: `Store.FinishSession` appends the
event on the same `*sql.Tx` as the status update (D2's atomicity id proves
the store seam), `Ring()` after commit. Events carry references, never bytes:
no transcript content, no diff, no instructions.

**Rejected.**
- **No producer role in v1** — session outcomes are the one fact the rest of
  the suite plainly wants (notify push on failure is the worked consumer
  pattern); the outbox is chassis-cheap and this family set is stable.
- **`repos:release/*` in v1** — release machinery is explicitly out of v1
  scope.
- **A distinct `session.cancelled` family** — collapsed into `failed` above.
- **Payload carrying the check output or PR diff** — event-plane rule:
  references, never bytes.

**Verification** (real substrates: `Service`/`Store` over real migrated
SQLite with the real eventplane outbox; the real `FeedHandler` over
`httptest`).

- R-FT54-LW5D — `repos.Events` declares exactly the two families; through
  the assembled MCP handler, reflection lists both with the `/<repo name>`
  subject description, and appending an undeclared kind is rejected by the
  outbox.
- R-FUD0-ZNW2 — finishing a succeeded session writes one outbox row with
  kind `session.succeeded`, subject `/<repo>`, and a payload whose JSON keys
  are exactly the populated fields above (`pr_url` present, `error` absent);
  a failed and a cancelled session each write `session.failed` (`error`
  populated, `cancelled` for the latter) — each row on the same transaction
  as its terminal status write.
- R-FVKX-DFMR — end-to-end: an outcome appended through the producer path
  and served by the real `FeedHandler` frames
  `event: repos:session.succeeded/<repo>` with a `data:` envelope carrying
  `kind` and `subject` keys.
