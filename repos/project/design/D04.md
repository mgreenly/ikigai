# Decision 4 вАФ Repo lifecycle & git custody

**Decision.** `internal/repos/git.go` owns every git invocation in the
service; nothing else shells out to git. It wraps the real `git` binary
(missing tooling fails loudly):

```go
type Git struct {
    stateRoot string            // <state>/repos
    tokens    TokenSource       // github loopback GET /token (below)
    runner    func(ctx context.Context, dir string, env []string, args ...string) ([]byte, error)
}
func (g *Git) Clone(ctx, cloneURL, name string) error            // вЖТ state/repos/<name>
func (g *Git) Freshen(ctx, name, defaultBranch string) error     // fetch + reset --hard origin/<default>
func (g *Git) WorktreeAdd(ctx, name, branch, path, defaultBranch string) error
func (g *Git) WorktreeRemove(ctx, name, path string) error
func (g *Git) Push(ctx, worktreePath, branch string) error
func (g *Git) BranchExists(ctx, name, branch string) (bool, error) // local + ls-remote
```

**Credential custody.** The github service is the single custodian of the App
key; repos never sees it. `TokenSource` fetches a short-lived installation
token from github's loopback `GET /token` (github D10) at
`registry.BaseURL("github")`, caching it until near expiry. Authenticated
operations (`Clone`, `Freshen`, `Push`) inject it **per invocation** as
`-c http.extraHeader=Authorization: Basic <b64(x-access-token:TOKEN)>` вАФ
the token is never written to `.git/config`, any file under the clone or
worktree, or a persistent environment variable, so the session agent's
workspace holds no credential material. (On this single-owner box the
loopback twin is reachable by any local process вАФ the same accepted posture
as every loopback surface; custody here is defense in depth, not a second
trust boundary.)

**Lifecycle.**
- **Lazy provisioning:** intake (D3) calls `Service.EnsureRepo(ctx,
  RepoFacts{Name, CloneURL, DefaultBranch, Owner})` вАФ inserts the row and
  clones on first contact; idempotent when the row exists.
- **Explicit onboarding:** the MCP `clone` verb (D7) takes a repo `name`,
  derives `https://github.com/<REPOS_GITHUB_ORG>/<name>.git`, and calls the
  same `EnsureRepo`; a repo that already exists is `conflict`.
- **v1 is clone-only:** every repo has a GitHub remote; there is no `init`.
- **Before each session:** `Freshen` pins the canonical clone to
  `origin/<default_branch>`; sessions branch from that tip (D5). The
  canonical tree is never an agent's cwd and never has a session branch
  checked out.
- **Removal:** the MCP `delete` verb removes the repo row and
  `state/repos/<name>` entirely; sessions and transcripts survive (D9).

**Rejected.**
- **Credential helper baked into the clone's git config** (the pre-v1
  sketch) вАФ a worktree shares the canonical repo's config, so the agent
  could read or invoke the helper from inside its session; per-invocation
  header injection keeps credential material out of everything the agent
  can reach.
- **Tokens in remote URLs** (`https://x-access-token:<tok>@githubвА¶`) вАФ the
  token lands in `.git/config` and in error output; rejected for the same
  reason.
- **go-git instead of the git binary** вАФ worktrees, credential-header
  injection, and exact CLI parity are what the design leans on; the binary
  is guaranteed on the box and is what sessions themselves use.
- **Deleting a repo tombstones the row but keeps the clone** вАФ the clone is
  recoverable from GitHub at any time (clone-only v1); keeping dead trees
  contradicts "delete means the disk is reclaimed".

**Verification** (real substrate: the real `git` binary against local bare
fixture remotes in `t.TempDir()`; the token source as a recording `httptest`
stub).

- R-EXFQ-NUVB вАФ `EnsureRepo` for an unseen repo creates the row and a
  canonical clone at `state/repos/<name>` whose `origin` is the given URL; a
  second call with the same facts changes nothing and errors nothing.
- R-EYNN-1MM0 вАФ the MCP-facing path: `Service.CloneRepo(owner, "acme")`
  under `REPOS_GITHUB_ORG=ikigenba` records `clone_url`
  `https://github.com/ikigenba/acme.git`; cloning an already-tracked name
  returns the domain conflict sentinel.
- R-EZVJ-FECP вАФ after the fixture remote's default branch advances,
  `Freshen` moves the canonical clone's `origin/<default>` and local default
  ref to the remote tip (asserted by SHA equality), even when the canonical
  clone had drifted.
- R-F13F-T63E вАФ an authenticated `Push` against a stub that requires the
  `Authorization` header succeeds and, afterward, no file under the canonical
  clone or the worktree (including `.git/config` and
  `.git/worktrees/**/config`) contains the token string; the `TokenSource`
  was consulted (recorded call) rather than any ambient credential.
- R-F3J8-KPKS вАФ end-to-end over the real plumbing: clone a fixture remote,
  freshen, add a worktree on a new branch, commit in the worktree, push, and
  observe the commit on the bare remote's branch; remove the worktree and
  the canonical clone remains intact.
