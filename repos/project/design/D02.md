# Decision 2 — Data model & migrations

**Decision.** SQLite, owned by this service alone, created by embedded
migrations (`internal/db/migrations/`, `//go:embed`, applied by the appkit
runner). Three domain tables plus the eventplane outbox and the eventplane
consumer offset store:

```sql
CREATE TABLE repos (
    name           TEXT PRIMARY KEY,   -- GitHub repo name within the org; also the state dir name
    owner_email    TEXT NOT NULL,      -- provisioning owner; stamped on sessions and events
    clone_url      TEXT NOT NULL,      -- https clone URL
    default_branch TEXT NOT NULL,      -- e.g. main
    created_at     TEXT NOT NULL       -- RFC3339Nano UTC
);

CREATE TABLE sessions (
    id           TEXT PRIMARY KEY,     -- ULID
    repo_name    TEXT NOT NULL,        -- no FK: sessions survive repo deletion (D9)
    owner_email  TEXT NOT NULL,        -- denormalized at start
    issue_number INTEGER,              -- NULL for manual (MCP-started) sessions
    attempt      INTEGER NOT NULL,     -- 1, 2, 3… per issue; drives branch suffix (D5)
    branch       TEXT NOT NULL,
    instructions TEXT NOT NULL,        -- the pinned instructions.md content source
    status       TEXT NOT NULL,        -- queued | running | succeeded | failed | cancelled
    error        TEXT,                 -- terminal failure reason
    pr_url       TEXT,                 -- set on success
    created_at   TEXT NOT NULL,        -- enqueue time; FIFO order key
    started_at   TEXT,
    ended_at     TEXT,
    log_path     TEXT NOT NULL         -- state/sessions/<id>/output.jsonl
);
CREATE INDEX idx_sessions_repo   ON sessions(repo_name, created_at);
CREATE INDEX idx_sessions_status ON sessions(status);
CREATE INDEX idx_sessions_issue  ON sessions(repo_name, issue_number);
```

plus the outbox table created from the current eventplane `outbox.SchemaSQL`
(kind/subject envelope) in its own migration, pinned by a DDL drift guard the
same way webhooks/gmail pin theirs.

repos is also an event-plane **consumer** (D1 declares the webhooks consumer;
D3 is its handler), and the eventplane consumer engine reads and writes its
resume position from a `feed_offset` table (`eventplane/consumer/store.go`).
That DDL is **owned by the eventplane library** as `consumer.SchemaSQL`; repos
carries it as its own migration — created via `bin/create-migration repos
feed_offset`, body byte-identical to the library constant — so there is a
single migration authority per DB file while the schema's source of truth
stays in the library. A DDL drift guard in `internal/db` asserts the
byte-identity, mirroring notify's `migrations_feed_offset_test.go` and the
outbox guard above. There is deliberately **no dedup table**: repos' intake is
idempotent at the domain layer (D3's double-trigger guard via
`ActiveSessionForIssue`), so the cursor plus the first-subscription marker is
its only durable consumer state — the same posture as notify and scripts.

`internal/repos/store.go` — `type Store struct{ db *sql.DB }` over the shared
single-writer handle, value-object rows (`repos.Repo`, `repos.Session`), with
the queries the engine needs: `InsertRepo`, `GetRepo`, `ListRepos(owner)`,
`DeleteRepo`, `InsertSession`, `GetSession`, `ListSessions(repo?, owner)`,
`ActiveSessionForIssue(repo, issue)` (status IN queued,running),
`CountRunning()`, `NextQueued()` (oldest `queued`, FIFO),
`MarkRunning/FinishSession` (terminal write; D8 appends the outcome event on
the same `*sql.Tx`), `MaxAttempt(repo, issue)`, `SweepRunning` (boot
recovery, D5). Repos are **not** owner-scoped for intake (any collaborator
can dispatch via GitHub); `owner_email` is provenance and the identity the
runner asserts to the github service.

**Rejected.**
- **A queue table separate from sessions** — `status='queued'` plus the
  `created_at` index is the whole FIFO; a second table adds a two-phase
  insert for nothing.
- **FK sessions→repos with cascade** — transcripts must survive repo removal
  (product promise); denormalize instead, exactly as prompts' runs table
  does.
- **Storing worktree paths in the DB** — derivable from the session id
  (`state/sessions/<id>/worktree`); a stored path can only drift.
- **Letting the consumer engine create `feed_offset` at boot** — the appkit
  migration runner is the single schema authority per DB file (suite
  convention); engine-owned DDL would bypass `schema_migrations` tracking and
  the downgrade guard.
- **A dedup table alongside the offset store** — intake is already idempotent
  at the domain layer; a dedup table is state without a consumer.

**Verification.**
- R-EMGN-7X72 — applying the full embedded migration set through the appkit
  runner to a fresh real SQLite database yields the `repos`, `sessions`, and
  `outbox` tables with exactly the columns above (including `outbox.kind` and
  `outbox.subject`, no `type`), and the DDL drift guard asserts the newest
  outbox migration contains the current eventplane `outbox.SchemaSQL`
  verbatim.
- R-ENOJ-LOXR — over a real migrated temp SQLite: a `Repo` and a `Session`
  round-trip through the Store with every field intact;
  `ActiveSessionForIssue` finds queued and running sessions but not terminal
  ones; `NextQueued` returns the oldest queued session; `MaxAttempt` returns
  0 for an unseen issue and the highest attempt otherwise.
- R-EOWF-ZGOG — `FinishSession` writes the terminal status, `error`/`pr_url`,
  and `ended_at` in one transaction such that a mid-transaction failure
  (forced rollback in the test) leaves the session row unchanged and no
  outbox row behind (atomicity of the D8 pairing, proven at the store seam).
- R-TY2R-GFRU — the embedded migration set contains a `feed_offset` migration
  whose body is byte-identical to the current eventplane `consumer.SchemaSQL`
  (the drift guard fails when the migration is absent or diverges), and
  applying the full set through the appkit runner to a fresh real SQLite
  yields the `feed_offset` table.
- R-TZAN-U7IJ — the real eventplane consumer engine, configured for source
  `webhooks` against a fully-migrated real temp SQLite and an `httptest` SSE
  feed, starts, completes its subscription handshake, and persists a
  `feed_offset` row for `webhooks` — proving the consumer the D1 Spec declares
  can actually run against this service's migrated schema (the substrate that
  falsifies a missing or wrong offset store, which handler-level `httptest`
  assertions cannot).
