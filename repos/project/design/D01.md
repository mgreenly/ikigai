# Decision 1 â€” Composition root & chassis boot

**Decision.** repos is a core-block service on the appkit chassis, composed
exactly like the newest services (gmail is the model): `cmd/repos/main.go` is
`func main() { appkit.Main(reposSpec()) }`, and `reposSpec()` returns one
inline `appkit.Spec`:

```go
appkit.Spec{
    App:        "repos",
    Mount:      "/srv/repos/",
    Port:       registry.MustPort("repos"),   // 3007, Core
    MCP:        true,
    WWW:        true,
    Feed:       "/feed",
    Migrations: db.FS,
    Events:     repos.Events,                 // D8 registry
    Consumers:  consumers,                    // D3: source "webhooks"
    Handlers:   handlers,                     // below
    Producer:   producer,                     // injects outbox into the Service (D8)
    Workers:    workers,                      // session dispatcher loop (D5)
    Health:     health,                       // running/queued session counts
}
```

`Handlers(rt)` reads all `REPOS_*` env at this root (the config set in the
spine's Conventions), constructs the one domain `*repos.Service` over
`rt.DB()`, the state root, the real git runner, the github peer client
(base URL `registry.BaseURL("github")`), and the session engine, then mounts:
`POST /mcp` behind `rt.RequireIdentity` (D7 handler), `GET /{$}` landing via
`rt.WWW()` (D10). Model validation (D5) runs here: an unknown
`REPOS_PROVIDER`/`REPOS_MODEL` pair fails boot loudly. Hook order is the
chassis's `Handlers â†’ Producer â†’ Workers`; a boot-time recovery sweep (D5)
runs at the start of the dispatcher Worker.

Internal layout: `internal/db` (migration embed + DDL guards only),
`internal/repos` (domain: store, git, sessions, intake, events, protocol),
`internal/runner` + `internal/tools` (the copied session engine, D5),
`internal/mcp` (`mcp.go` + `tools.go`, D7), `share/www/` (D10),
`etc/manifest.env`, `etc/nginx.conf`, `.envrc`, `AGENTS.md` (+ `CLAUDE.md`
symlink), `VERSION` (`v0.1.0`).

Suite-level preconditions this module consumes but does not own: the appended
registry row `{"repos", 3007, Core}`, the `./repos` entry in `go.work`, and
the `repos` entry in `bin/start`'s `SERVICES`. No Decision or phase here edits
those files.

**Rejected.**
- **A worker pool service outside appkit** â€” every deployable app rides the
  chassis; the fixed verb set, manifest, and deploy tooling assume it.
- **Registering repos under Connectors (32xx)** â€” repos owns a first-party
  domain (git trees + sessions), not a third-party bridge; it is a Core
  service and 3007 is the next Core slot.
- **Reading env below the composition root** â€” suite convention: config is
  resolved once at the root and passed down as values.

**Verification.**
- R-EISY-2LYZ â€” the built binary's `manifest` verb emits exactly
  `APP=repos`, `MOUNT=/srv/repos/`, `PORT=3007` (from the registry, no
  literal in source â€” a source-scan test proves no `127.0.0.1:30` literal
  outside tests), `MCP=true`, `FEED=/feed`, and a `CONSUMES` naming
  `webhooks`.
- R-EL8Q-U5GD â€” through the assembled handler over `httptest`: `POST /mcp`
  without identity headers is refused, with asserted identity reaches the D7
  tool table; `GET /{$}` renders the D10 landing; `GET /feed` serves the
  eventplane SSE handler.
