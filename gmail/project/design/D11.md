# Decision 11 — Adopt `registry`: resolve gmail's own loopback port by name

**Decision.** gmail writes no loopback port literal in its Go source: it
references itself **by name** through the shared `registry` library
(`module registry`, the repo-root leaf module). Resolution happens **once, at the
composition root** — never per request — because the table is compile-time data:
if a port ever changes we rebuild the binaries anyway. This is
**behavior-preserving**: `registry`'s table carries gmail's current value
(`gmail=3202`), so the resolved value is byte-identical to the literal it
replaced.

**Why.** "Which port a service answers on" used to be duplicated across the suite
and drift silently — a renumber only surfaced at deploy, uncatchable by the
compiler. Making `registry` the single source of truth means gmail asks the table
for its address and never repeats the number.

**The call site (in the gmail `appkit.Spec`).** The `Spec.Port` value becomes a
name lookup, the strict composition-root form that panics on an unknown name (a
programming error):

```go
appkit.Spec{
    App:   "gmail",
    Mount: "/srv/gmail/",
    Port:  registry.MustPort("gmail"),
    ...
}
```

The `3202` integer literal is the only value that changes; every other Spec field
is untouched.

**gmail is a producer, not a consumer — no peer feed resolution.** Unlike the
consumer services, gmail resolves **no** upstream feed URLs: it publishes to its
own `/feed` and consumes nothing, so there are no `*_FEED_URL` defaults to route
through `registry`. The adoption is exactly one call: gmail's own port.

**Module wiring (inside `gmail/` only).** `gmail/go.mod` gains
`require registry v0.0.0` and the committed `replace registry => ../registry`,
mirroring the existing `appkit` and `eventplane` in-repo replace-siblings.

> **External precondition (OUT OF SCOPE — not built by any gmail phase).** The
> repo-root `go.work` must carry `use ./registry`, and the `registry` module
> itself must exist and be green. Both are owned at the repo root / by the
> `registry` project and are assumed satisfied. No gmail Decision or phase edits
> `../go.work`, `../registry/`, `../bin/`, nginx, or any sibling module — the
> executor runs from `gmail/` and cannot reach outside it (save the single
> sanctioned `bin/start` line in D9).

**Resolve-at-startup, not per-request.** `Spec.Port` is read once by appkit at
boot. No handler or request path calls `registry`.

**Rejected.**
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; looking it up per request buys nothing.
- **`registry.Port("gmail")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form here.
- **Leaving `Port: 3202` and adopting `registry` only for guards.** Keeps the
  drift-prone literal that the adoption exists to remove.

**Verification.**
- R-9QEG-KF05 — **gmail's own listen port is `registry.MustPort("gmail")`**
  (== `3202`), not a `3202` integer literal, at the composition root.
  Falsifiable: the emitted port must equal `registry.MustPort("gmail")`; a
  hardcoded `Port: 3202` in the Spec fails the intent. Proven concretely by the
  manifest drift guard (D12's `R-9SU9-BYHJ`), which emits with
  `registry.MustPort("gmail")` and byte-matches the committed `PORT=3202` in
  `etc/manifest.env`.
