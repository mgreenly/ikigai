# Decision 10 — MCP surface over `appkit/mcp`: `internal/mcp` becomes the tool table

**Problem.** gmail's `internal/mcp` carries the copy-pasted JSON-RPC transport
(`ServeHTTP`, the `initialize`/`notifications/initialized`/`tools/list`/
`tools/call` dispatch, `writeJSONRPCError`/`writeJSONRPCResult`, the local
`Identity` type, the `toolResultText`/`toolResultErr` envelope helpers) plus its
own `health` and `reflection` implementations (`toolHealth`, `toolReflection`,
`renderSubscriptions`, `reflectionUnknownTypeError`) — the same near-identical
plumbing every service duplicated. The chassis now owns all of it (appkit D8
transport, D9 standard tools, consumed here as a fixed external contract). What
is genuinely gmail's is the **ten normal-mailbox tools** — `list`, `read`,
`thread`, `labels`, `send`, `draft`, `label`, `unlabel`, `trash`, `delete` —
their descriptors, schemas, validation, and the P2 Gmail-client hops.

**Decision.** `internal/mcp` is reshaped from a transport into a **declaration**:

```go
// internal/mcp — what remains.

// Client is the slice of the P2 Gmail REST client the tool handlers drive
// (unchanged: an interface so handlers unit-test against a fake, no network).
type Client interface { /* MessagesList … MessageDelete, as today */ }

// Instructions is gmail's MCP instructions string, verbatim as today:
//   "Read and send Gmail and manage labels. Start with list to find messages,
//    then read or thread."
const Instructions = /* gmail's instructions string */

// Tools declares gmail's ten domain tools over the Gmail client, each an
// appkitmcp.Tool (name + description + inputSchema + handler). health and
// reflection are NOT declared — the chassis registers them (their inputs all
// come from Router accessors); declaring either locally fails New's
// reserved-name check. contentBase is the loopback base URL stamped into
// read/thread attachment content_url references (D17).
func Tools(client Client, contentBase string) []mcp.Tool

// NewHandler assembles the /mcp http.Handler at the composition root:
// appkitmcp.New over Instructions + Tools(client, contentBase) + the
// Router-threaded Service/Version/Health/Events/Publishes/Subscriptions. A nil
// client or an empty contentBase is a wiring error and panics at this seam.
func NewHandler(client Client, contentBase string, rt *appkit.Router) (http.Handler, error)
```

- **Deleted:** the JSON-RPC plumbing (`jsonRPCRequest`, the `ServeHTTP`
  dispatch, `writeJSONRPCError`/`writeJSONRPCResult`, `idOrNull`, the local
  `Identity` type), the local `toolHealth`/`toolReflection`/
  `renderSubscriptions`/`reflectionUnknownTypeError` implementations, the
  `toolDescriptors()` builder, and the local result-envelope helpers
  (`toolResultText`/`toolResultErr`/`toolResultJSON`) — replaced by the chassis
  `mcp.TextResult`/`mcp.ErrorResult`/`mcp.JSONResult`.
- **Kept:** the `Client` interface; the descriptor/schema helpers
  (`desc`/`obj`/`descTyp`, now producing `mcp.Tool` values); the ten tool
  descriptions and schemas; every tool handler body (`toolList`…`toolDelete`,
  `modifyLabel`, `composeArgs`, `buildRawMessage`, `renderMessage`,
  `collectHeaders`/`collectAttachments`, `decodeArgs`, `orEmpty`) with its
  validate-and-call logic unchanged (`renderMessage`/`collectAttachments` also
  stamp the D17 attachment references); the `Events` family registry and the
  mail payload structs — kinds `received`/`sent`/`deleted` per D18
  (`Spec.Events` still points at `mcp.Events`). Each handler's
  signature adapts to `func(ctx, args json.RawMessage, id server.Identity)
  (map[string]any, error)` — the identity argument replaces the request-header
  read, and the mailbox handlers ignore it exactly as they ignore identity
  today.
- **The wire surface:** the ten domain tool names and schemas are unchanged;
  the `read`/`thread` descriptions state the attachment-reference truth (D17 —
  entries carry `attachment_id` + a loopback `content_url`; the old "blob
  download not supported" sentence is gone) and every other description is
  unchanged. The chassis-registered `health` and `reflection` carry the
  canonical chassis descriptions (health returns the same envelope **plus**
  `owner_email`/`client_id`, reflection returns the `publishes`/`subscribes`
  index over the `Events` family registry, and `subscribes` is empty because
  gmail is a producer). The full listed surface stays **exactly twelve** tools.

**Rejected.**
- **Keeping the local transport** — the status quo; a protocol fix would still
  be a many-service edit.
- **Declaring health/reflection in gmail's table** — the chassis reserves the
  names (appkit D8/D9) precisely so services cannot fork the standard tools;
  their inputs (Envelope, Registry, Subscriptions, Identity) are chassis-held
  already, and the chassis `health` already emits gmail's `owner_email`/
  `client_id` with the identical description, so nothing is lost.
- **Dissolving `internal/mcp` into `cmd/gmail`** — the ten mailbox tools with
  their schemas, validation, RFC-2822 composition, and their fake-client test
  harness are a real package, unlike the post-D9 web remnant.

**Verification** (through the composition-root seam — the appkit/mcp handler
built by `NewHandler` over a `fakeClient`; the existing mailbox behavioral tests
in `tools_test.go` keep their assertions and swap only the handler constructor).
- R-9NYN-SVIR — `tools/list` through the assembled handler returns **exactly
  twelve** tools: the ten domain tools declared by gmail (`list`, `read`,
  `thread`, `labels`, `send`, `draft`, `label`, `unlabel`, `trash`, `delete`)
  **plus** `health` and `reflection` supplied by the chassis — proving the
  table/chassis partition (a table that still declares its own
  health/reflection fails `New`'s reserved-name check; a surface missing them
  fails the count).
