# Decision 13 ‚Äî Composition-root normalization: the `appkit.Spec` inline in `cmd/gmail/main.go`

**Problem.** gmail is the only converted-shape service that declares its
`appkit.Spec` in a **separate package** ‚Äî `gmail/internal/gmailapp/spec.go`
exposes `func Spec() appkit.Spec`, and `cmd/gmail/main.go` is a two-line shim
that calls `appkit.Main(gmailapp.Spec())`. The reference shape the suite is
converging on (crm's inline `appkit.Spec{‚Ä¶}` in `cmd/crm/main.go`, notify's
`func notifySpec() appkit.Spec` in `cmd/notify/main.go`) puts the Spec **at the
composition root**, in `cmd/<svc>/main.go`, with no separate `internal/<svc>app`
package. The `gmailapp` indirection buys nothing: the Spec's `Handlers`/
`Producer`/`Workers` closures already capture a single local `var engine`, which
works identically whether the enclosing function lives in `internal/gmailapp` or
`package main`.

**Decision.** Move the Spec to the composition root and delete the indirection.

1. **`internal/gmailapp/spec.go` moves into `cmd/gmail/main.go`** as a
   package-level `func gmailSpec() appkit.Spec` (mirroring notify's
   `notifySpec()`). The body is unchanged: the shared `var engine *gm.Engine`
   stays declared inside `gmailSpec` and is captured by the `Handlers`,
   `Producer`, and `Workers` closures exactly as today (appkit calls
   `Handlers ‚Üí Producer ‚Üí Workers` in that order ‚Äî the ordering contract is
   preserved verbatim). `main()` becomes `appkit.Main(gmailSpec())`.

2. **`internal/gmailapp/` is deleted** ‚Äî the package has no other members.

3. **`cmd/gmail/main_test.go`** stops importing `gmail/internal/gmailapp` and
   calls the local `gmailSpec()` where it currently calls `gmailapp.Spec()`
   (the manifest byte-equality tests read `spec.App`, `spec.Mount`,
   `spec.Port`, etc. from the returned Spec). This is a harness import change
   only ‚Äî **no test assertion changes**.

4. **Nothing else changes.** This Decision is a pure relocation: the Spec's
   fields, the connector/producer/poll-daemon wiring, the secret reads at the
   `Handlers` boundary (the three `GMAIL_*` values + `GMAIL_POLL_INTERVAL`,
   still read via `os.Getenv` at gmail's own composition root and never logged),
   and the manifest output are byte-identical before and after. It does not
   touch the WWW conversion (D9), the MCP conversion (D10), or the registry
   adoption (D11) ‚Äî those edit the Spec's fields in their own phases; this
   Decision only settles **where** the Spec lives so those edits land at the
   composition root like the reference.

**Ordering note.** This normalization is sequenced **after** the registry
adoption (D11/D12) and **before** the WWW (D9) and MCP (D10) conversions, so that
`main.go` is the single settled home the WWW and MCP phases then edit. The
registry phases edit the `Port` field in place wherever the Spec lives at that
point; this Decision moves the whole Spec afterward without disturbing that
value.

**Rejected.**
- **Keeping `internal/gmailapp`** ‚Äî it is a one-function package whose only role
  is to hold a Spec that every peer service declares at the composition root; it
  is exactly the ceremony the normalization removes, and it makes gmail the odd
  service out.
- **Inlining the Spec literal directly in `main()` (crm's form) rather than a
  `gmailSpec()` helper** ‚Äî either is the reference; the named-helper form
  (notify's) keeps the manifest tests able to obtain the Spec by calling
  `gmailSpec()` without booting the process, which crm's inline form would force
  a different test seam for. The helper is the lower-friction match for gmail's
  existing manifest tests.
- **Post-construction mutation of `.Handlers`/`.Workers`** ‚Äî not present today
  and not introduced; the Spec is declared fully formed and returned once.

**Verification.** Structural ‚Äî no `R-XXXX-XXXX` ids (a pure relocation; the
behaviors are already pinned by the manifest, boot-health, MCP, and producer
suites). Realized by a plan phase whose done bar is:

- `cmd/gmail/main.go` contains `func gmailSpec() appkit.Spec` and
  `appkit.Main(gmailSpec())`; `ls gmail/internal/gmailapp 2>/dev/null` reports no
  such directory; `grep -rn "internal/gmailapp" gmail --include=*.go` (outside
  `project/`) returns no matches.
- The suite stays green (design Conventions commands) with no test assertion
  changes ‚Äî only the `main_test.go` import/constructor swap.
