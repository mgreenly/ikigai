# Decision 5 — Docs state current truth: purge the stale "no UI" line

**Decision.** gmail's service doctrine — `gmail/AGENTS.md` and its symlinked
`gmail/CLAUDE.md` (one file; edit `AGENTS.md`) — frames gmail as a loopback-only
connector that serves **only machine surfaces** (the bearer-gated `/mcp`, the PRM
well-known, `/health`, the loopback `/feed`) and declares **no human web page** —
the same "**no UI**" doctrine the rest of the suite carried. That framing is now
**false**: gmail serves a human web landing page (D1–D4). Per the suite doctrine —
*docs state current truth; purge what the change falsified, rewrite in place, no
archaeology, no footnotes* — this Decision **rewrites that framing in place** to
state the landing-page truth.

> **gmail note.** gmail does not yet carry a `gmail/AGENTS.md`/`CLAUDE.md`; its
> identity doctrine currently lives in the `cmd/gmail/main.go` package comment,
> which presents gmail as wiring **only** its machine domain surface (MCP +
> producer feed) and no human page. This Decision is the build-loop's signal that
> wherever gmail states its doctrine — the service `AGENTS.md`/`CLAUDE.md` (the
> file every sibling service carries) — that doctrine must state the
> landing-page truth and carry **no** "no UI" / machine-surfaces-only claim. The
> `cmd/gmail/main.go` package comment is **not** edited by this Decision (it is Go,
> owned by the connector design); the doctrine doc is.

**What changes (doc edits only — no code):**

- Remove the "**no UI**" / "serves only machine surfaces" / "no human web page"
  claim wherever it appears in gmail's service doctrine (`gmail/AGENTS.md`).
- Replace it with the current truth, in gmail's own voice: gmail is a loopback-only
  Gmail connector that serves an **MCP surface for agents** (bearer-gated) **and a
  human web landing page** (session-cookie-gated) under `/srv/gmail/`, with its
  History-API poll daemon and `mail.*` event producer unchanged; it still runs
  **no token logic** — nginx remains the sole trust boundary for both doors. Keep
  the "no token logic" half (still true); drop only the "no UI" half (now false).
- Do **not** add a changelog, a "previously…", or a footnote. State the present
  as if it were always so.

**Why this is its own phase, performed by the build loop — not by the doc
author.** The doc author (this design+plan workspace) deliberately does **not**
edit the doctrine doc now: code and docs must evolve **together**, so the purge is
sequenced as a plan phase that the build loop performs **after** the landing page
exists (D1–D4 shipped). Doing it earlier would make the docs claim a UI the code
does not yet serve; doing it never would leave a false statement on `main`.

**Scope guard.** Only the "no UI" falsehood is purged. gmail's other doctrine — the
normal-mailbox MCP tool surface, the History-API poll daemon, the `mail.*`
event-plane producer, the header-trust model, "no token logic," the loopback bind,
the deploy/manifest sections — is **unchanged and still true**; this Decision does
not touch it.

**Verification.** This is a **documentation** Decision with no Go behavior, so it
mints no `R-XXXX-XXXX` ids; it is realized by a **structural** plan phase whose
done bar is a content check rather than an id-tagged test:

- gmail's service doctrine (`gmail/AGENTS.md`, hence `gmail/CLAUDE.md`) no longer
  contains the string `no UI` (case-insensitive) — `grep -i "no UI" gmail/AGENTS.md`
  finds nothing.
- `gmail/AGENTS.md` states the landing-page truth: it mentions that gmail serves a
  human web **landing page** under `/srv/gmail/` alongside the MCP surface.
- gmail's `AGENTS.md`/`CLAUDE.md` symlink invariant is preserved (one file; the
  edit goes through `AGENTS.md`), and the suite stays green
  (`bin/check-migrations gmail` and `cd gmail && go build ./...` unaffected — this
  is a docs-only change).
