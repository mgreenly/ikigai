# Decision 16 — Attachment content endpoint: loopback `GET /attachment` (content plane holder)

**Problem.** gmail can enumerate a message's attachments (filename, size, mime
type — D10's `read`/`thread` render) but exposes no way to obtain their bytes:
an emailed PDF is unreachable by any consumer. The suite's content plane
(`docs/content-plane-design.md`) makes gmail a **holder**: a service that owns
bytes exposes a loopback-private `GET` content endpoint, and everything
agent-facing carries only references. This Decision is gmail's holder half; the
references stamped into `read`/`thread` results are D17.

**Decision.** gmail serves attachment bytes from a loopback-private
`GET /attachment` endpoint, addressed by the Gmail API's own addressing —
`message_id` + `attachment_id` (`users.messages.attachments.get` returns the
part body as URL-safe base64). The service is a connector: it fetches from
Google **per request** and keeps **no local blob cache** — stateless, exactly
like every other gmail read path (nothing in the codebase argues otherwise; the
service holds no blob storage today and gains none).

1. **The client grows one method** (in `internal/gmail/client.go`, the only
   HTTP-to-Gmail site, same offline-stub test seam as the rest of the client):

   ```go
   // AttachmentGet fetches one attachment body via
   // GET /messages/{messageID}/attachments/{attachmentID} and returns the
   // DECODED bytes (Gmail returns {attachmentId, size, data} with data in
   // URL-safe base64; decode tolerates absent padding — try padded
   // base64.URLEncoding, fall back to RawURLEncoding). Empty ids →
   // ErrValidation; a Google 404 → ErrNotFound.
   func (c *Client) AttachmentGet(ctx context.Context, messageID, attachmentID string) ([]byte, error)
   ```

2. **The handler lives in the domain package** (`internal/gmail/attachment.go`,
   mirroring dropbox's `internal/dropbox/content.go`), behind a narrow seam so
   handler tests inject a fake and never touch the network:

   ```go
   // AttachmentSource is the slice of the client the handler drives.
   type AttachmentSource interface {
       MessageGet(ctx context.Context, id, format string) (Message, error)
       AttachmentGet(ctx context.Context, messageID, attachmentID string) ([]byte, error)
   }

   // AttachmentHandler returns the loopback GET /attachment handler.
   func AttachmentHandler(src AttachmentSource) http.Handler
   ```

   Request flow:
   - **Primary guard first** (copied from dropbox's `/content` and `/feed`):
     if the request carries an nginx-injected identity header —
     `X-Owner-Email` **or** `X-Forwarded-Proto` — respond **404**. The
     endpoint is loopback-private by handler guard, never by nginx.
   - Read `message_id` and `attachment_id` from the query (net/http
     percent-decodes them). Either missing/blank → **404**.
   - `MessageGet(message_id, "full")`, then walk the MIME part tree
     (`MessagePart.Parts`, recursive) for the part whose `Body.AttachmentID`
     equals `attachment_id`. `ErrNotFound` from the client, or no matching
     part, → **404**. The walk yields the part's `MimeType` (the
     Content-Type source) — this second Google call is the price of a
     self-contained, correct Content-Type and a clean existence check.
   - `AttachmentGet(message_id, attachment_id)` → the decoded bytes; write
     them with `Content-Type` = the matched part's mime type
     (`application/octet-stream` when the part carries none) and
     `Content-Length` = the decoded byte count. The body is buffered (Gmail
     returns the whole base64 body in one response; there is no upstream
     stream to relay), which is fine at Gmail's ~25 MB message ceiling.
   - Any **other** upstream error (Google 5xx, network fault, token refresh
     failure) → **502** — a connector's transient upstream fault is a gateway
     condition, distinguishable from not-found so a consumer can retry.
     `ErrInvalidGrant` also surfaces as 502 (the poll daemon already fails
     loudly on it; the endpoint must not mask it as absence).

3. **Mounting** (composition root, `cmd/gmail/main.go` `Handlers`): registered
   verbatim, **not** behind `rt.RequireIdentity` — the handler is the guard,
   exactly like dropbox's `/content` routes:

   ```go
   rt.Handle("GET /attachment", gm.AttachmentHandler(client))
   ```

4. **nginx defence-in-depth** (`gmail/etc/nginx.conf`): an exact-match
   `location = /srv/gmail/attachment { return 404; }` beside the existing
   `= /srv/gmail/feed` 404 — belt to the handler's braces, optional per the
   content-plane doc and never the mechanism (without it the bearer-gated
   prefix location would proxy the request *with* identity headers, which the
   handler guard 404s anyway).

**Addressing honesty — stability of `attachment_id`.** Gmail attachment ids
are the API's own handles but are not guaranteed stable across message
fetches. A reference minted by a fresh `read` resolves; a stale
`attachment_id` that no longer matches any part in the message simply returns
404, and the consumer re-runs `read` for a fresh reference. There is no
rev/pin dimension (see D17's deviation note): Gmail attachments are immutable
per message, so "the bytes moved on" cannot happen — only "the handle went
stale."

**Rejected.**
- **A local blob cache / mirror of fetched attachments** — gmail is a
  connector; Google is the source of truth and every other gmail read is
  per-request. A cache adds state, retention, and invalidation for no
  identified consumer need.
- **Content-Type from a `mime_type` query parameter stamped into the URL** —
  saves the `MessageGet` call but makes the served type caller-asserted and
  lets a hand-built URL lie; the part walk also provides the existence check
  that maps cleanly to 404.
- **Uniform 404 for every failure (dropbox's mapping)** — dropbox's failures
  are local-disk/index misses where "not found" is true; gmail's upstream can
  fail transiently, and collapsing that to 404 would tell consumers a real
  attachment does not exist. 502 keeps absence and fault distinguishable.
- **Gating the route behind `RequireIdentity`** — backwards: the route must be
  reachable *only* by bare loopback callers; identity headers are precisely
  the reject signal.
- **Streaming Google's base64 through a decoder without buffering** — possible
  but pointless: Content-Length would be unknowable (chunked response) and the
  payload ceiling is small; buffering keeps the handler simple and the
  Content-Length exact.

**Verification** (client claims on the client's existing offline substrate — a
stub `http.Client` RoundTripper / redirected host vars, no network; handler
claims against the **real** `AttachmentHandler` over `httptest` with a fake
`AttachmentSource`).

- R-WVZH-M0IY — `Client.AttachmentGet` issues
  `GET /messages/{id}/attachments/{aid}` (both path segments URL-escaped) with
  the bearer, decodes the `{size, data}` response's URL-safe base64 `data`
  (padded and unpadded both decode) and returns exactly the original bytes; a
  Google 404 maps to `ErrNotFound`; an empty `messageID` or `attachmentID`
  returns `ErrValidation` without any HTTP call.
- R-WX7D-ZS9N — the identity-header guard discriminates: a request **with**
  `X-Owner-Email: a@b.c` gets 404, a request **with** `X-Forwarded-Proto:
  https` gets 404, and the byte-identical request **without** either header
  gets 200 with the attachment bytes — proving the endpoint is loopback-private
  by handler guard, not by accident.
- R-WYFA-DK0C — the happy path: for a message whose MIME tree contains a part
  with `Body.AttachmentID` equal to the requested `attachment_id`, the response
  is 200, the body is byte-equal to the decoded attachment bytes, `Content-Type`
  equals that part's `mimeType` (e.g. `application/pdf`), and `Content-Length`
  equals the decoded byte count.
- R-WZN6-RBR1 — a request missing `message_id`, missing `attachment_id`, or
  with either blank gets 404 without any call on the `AttachmentSource` (the
  fake records zero calls).
- R-X0V3-53HQ — unknown-resource mapping: `MessageGet` returning `ErrNotFound`
  → 404; a message fetched successfully but containing **no** part with the
  requested `attachment_id` → 404; a non-not-found upstream error (e.g. a
  wrapped 503) → **502**, never 200 and never 404 — absence and upstream fault
  stay distinguishable.
- R-X22Z-IV8F — `gmail/etc/nginx.conf` contains the exact-match
  `location = /srv/gmail/attachment { return 404; }` while the pre-existing
  locations remain (the PRM well-known, the `= /srv/gmail/feed` 404, the
  session-gated `= /srv/gmail/` root, the session-gated `/srv/gmail/static/`,
  and the bearer-gated `/srv/gmail/` prefix) — proven by the existing
  content-assertion pattern over the shipped fragment (D4's substrate).
