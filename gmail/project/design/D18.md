# Decision 18 — Event-routing conformance: kinds `received`/`sent`/`deleted`, empty subject

> **External ordering (operator-sequenced, not built by any gmail phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before gmail's conformance phase runs**. It equally assumes appkit (which
> holds `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, gmail's conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and the
old type names drop their redundant noun prefix (`source` already says gmail
deals in mail). It is a hard cutover — no compatibility period — inside the
no-live-data migration window.

**What gmail publishes today (ground truth, not vacuous).** gmail is a real,
wired producer: `Spec.Feed: "/feed"`, `Spec.Events: mcp.Events` (an
`outbox.Registry` of three `EventType`s), the `Producer` hook attaching the
outbox as the engine's `EventSink`, and the poll engine
(`internal/gmail/sync.go`) deriving three event types per the History API —
`mail.received`, `mail.sent`, `mail.deleted` — appended via
`outbox.Append(tx, Event{Type, Payload})` atomically with the cursor advance
(`internal/gmail/events.go` `buildPayload`). The `mail.*` names and payload
structs are duplicated deliberately in `internal/mcp/tools.go` (the registry)
and `internal/gmail/events.go` (the producer). Conformance is therefore
**not** vacuous: all three published types convert.

**Decision.** gmail adopts the revised envelope exactly over what it publishes:

| today (`type`) | revised key | kind | subject |
|---|---|---|---|
| `mail.received` | `gmail:received` | `received` | `""` |
| `mail.sent` | `gmail:sent` | `sent` | `""` |
| `mail.deleted` | `gmail:deleted` | `deleted` | `""` |

The subject is **empty for all three** — the routing doc's producer key map
records `gmail:received` with "subject open until a routing need appears," and
no routing need has been identified; a future subject (e.g. a from-domain) is
a fresh design change, not scoped here. Payload shapes are **unchanged** — the
routing key addresses; the payload remains the detail record.

1. **Constants and payload builder** (`internal/gmail/events.go`, mirrored in
   `internal/mcp/tools.go` as today): `EventMailReceived = "mail.received"` →
   `KindReceived = "received"` (likewise `sent`, `deleted`; the `MailEvent.Type`
   field and its switch follow). `buildPayload` emits
   `outbox.Event{Kind: …, Subject: "", Payload: …}`. `sync.go`'s three
   derivation sites set the new kind constants; nothing else in the
   engine/EventSink seam changes (same tx, same `Ring()` after commit).

2. **The registry becomes families** (`internal/mcp/tools.go`): `mcp.Events`
   becomes an `outbox.Registry` of three `outbox.Family` entries —
   `{Kind: "received", Subject: "", Description: …, Sample:
   mailReceivedPayload{…}}` etc. Subjects are declared empty (subjectless
   kinds); descriptions keep their current meaning with the type names
   updated. `Spec.Events` wiring is untouched; Append-time kind gating and the
   reflection surface hang off the same registry as today, now family-shaped.

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. gmail appends **one new
   timestamped migration** minted with `bin/create-migration gmail
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across.
   The frozen `003_outbox.sql` is **not touched**; the DDL drift guard
   (`internal/db/migrations_outbox_test.go`) re-points at the **newest** outbox
   migration, asserting it contains the revised `outbox.SchemaSQL` verbatim.

4. **Reflection/consumer surface.** The chassis reflection tool (appkit-owned)
   reflects the family registry; gmail's observable is that its three kinds —
   and only they — are listed, with schema/example still derived from the same
   Sample values. gmail consumes nothing (producer only), so there is no
   filter/`consumer.Config` side to conform.

**Rejected.**
- **Keeping `mail.` prefixes as kinds (`mail.received`)** — the revision's
  explicit rename rule: `source` carries the noun; a `gmail:mail.received` key
  says "mail" twice.
- **A subject now (e.g. `/<message-id>` or `/<from-domain>`)** — subjects are
  routing names, and no consumer routes on any mail dimension yet; minting one
  speculatively bakes in a choice the first real consumer should make.
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are ≤7-day retention data with no
  live consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: gmail's engine/store tests run
against real SQLite via the existing sync-test harness; the feed claim runs the
real eventplane `FeedHandler` over `httptest`).

- R-X6YL-1Y77 — after the poll engine applies history records producing an
  inbound message, a sent message, and a trashed message, the outbox rows read
  back by SQL from the real SQLite database carry `kind` values exactly
  `received`, `sent`, and `deleted` (no `mail.` prefix) with `subject = ''` on
  every row, each appended on the same tx as the cursor advance (baseline
  atomicity preserved).
- R-X86H-FPXW — `mcp.Events` declares exactly three families with kinds
  `received`/`sent`/`deleted` and empty subjects; through the assembled MCP
  handler, reflection's index lists exactly those three publishes, the detail
  of `received` returns a schema and example that agree (every example field in
  the schema, `mailReceivedPayload`-shaped), and an undeclared kind returns the
  typed unknown-kind error naming the three declared kinds.
- R-X9ED-THOL — applying gmail's full embedded migration set (`db.FS`, through
  the appkit runner) to a fresh real SQLite database yields an `outbox` table
  with `kind` and `subject` columns and **no** `type` column; the DDL drift
  guard asserts the newest outbox migration contains the revised
  `outbox.SchemaSQL` verbatim while `003_outbox.sql` remains byte-identical to
  its committed (frozen) body.
- R-XAMA-79FA — end-to-end over gmail's own substrate: an event appended
  through gmail's producer path (`buildPayload` → `Append` on the
  migration-created table, source `gmail` — the `Spec.App` value) and served by
  the real eventplane `FeedHandler` over an `httptest` connection frames
  `event: gmail:received` (the canonical subjectless key) with a `data:` JSON
  envelope carrying `kind`/`subject` keys and **no** `type` key.
