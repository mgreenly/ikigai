# Decision 9 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** gmail's landing page and static assets are embedded in the binary
(`//go:embed` in `gmail/internal/web/`), like every service's original shape.
The suite is reverting that: web assets are release artifacts that belong on
disk — visible and diffable in the versioned release tree, editable in dev
without a rebuild, served by one chassis mechanism instead of twelve drifting
copies. crm and notify proved the conversion; gmail follows the same shape. This
Decision is the service-side half of appkit's chassis design (its D5 www-root
resolution, D6 `appkit/web`, D7 `Spec.WWW` / `Router.WWW()` — consumed here as a
fixed external contract, never changed).

**Decision.** gmail serves its web surface from the on-disk www root.

1. **The files move (same bytes).** `gmail/internal/web/landing.html` →
   `gmail/share/www/landing.html`; `gmail/internal/web/static/` →
   `gmail/share/www/static/` (`tokens.css` + the four woff2 fonts:
   `space-grotesk.woff2`, `ibm-plex-sans.woff2`, `ibm-plex-mono-400.woff2`,
   `ibm-plex-mono-500.woff2`). `bin/ship` already bundles `gmail/share/` into
   the release `share/<version>` tier; on the box the chassis resolves the root
   to `share/current/www`, in dev to `./share/www` or the `GMAIL_WWW_PATH`
   override (appkit D5).

2. **The Spec opts in.** The gmail `appkit.Spec` sets `WWW: true`. The chassis
   loads the site at serve (failing startup loudly if the root is missing),
   auto-mounts `GET /static/` over the site's static tree, and exposes the
   loaded site as `rt.WWW()` (appkit D7).

3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` renders `rt.WWW().Render(w, "landing.html", …)` (D1/D2 carry the
   exact shape). The service-side `GET /static/` mount is **deleted** — the
   chassis owns it.

4. **`gmail/internal/web/` is deleted** — `LandingHandler`, `StaticHandler`, the
   embed shim (`embed.go`), the template, the assets, and its tests
   (`web_test.go`, `nginx_test.go`). The landing/asset tests move to the
   composition-root package (`cmd/gmail`), loading the repo-real `share/www`
   tree via `appkit/web` (a relative path from the package dir), so they
   exercise the exact files that ship. The nginx fragment tests
   (`nginx_test.go`) relocate to `cmd/gmail` with their assertions intact (the
   D12 registry drift guard, already applied in an earlier phase, keeps their
   port expectations derived from `registry`).

5. **Dev wiring (boundary-crossing).** `bin/start`'s gmail launch function
   exports `GMAIL_WWW_PATH="$repo/gmail/share/www"` beside its existing
   `GMAIL_*` exports, so a repo-root-launched dev binary finds the source tree.
   This one line crosses the `gmail/` boundary and is the **only** sanctioned
   out-of-tree change; it is verified by the live `bin/start` smoke, not the Go
   suite.

6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6/D7/D8), the MCP surface (D10), the migrations (`internal/db`), and
   the connector/producer/poll-daemon wiring (`Producer`/`Workers`) are
   untouched. The embedded migrations stay embedded — they are not web assets.

**Retained ids on the new substrate.** D1's `R-LAND-*`, D2's `R-ROUT-*`, D3's
`R-ASST-*`, D7's `R-HOME-7Q9U`, and D8's `R-3X4A-*`/`R-3YC7-*`/`R-3ZK3-*`/
`R-40S0-*`/`R-41ZW-*` continue to pin the page/asset/gate behaviors — now proven
against the real `gmail/share/www` tree loaded via `appkit/web` and the
chassis static mount, at `cmd/gmail`. D4's `R-NGNX-*` (nginx fragment) relocate
unchanged. Their Decision files are rewritten in place to state the on-disk
substrate.

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work completes; embedded
  assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's nginx-served
  user content, backed up with `state/`; gmail's UI assets are release
  artifacts and live in the shipped `share` tier (appkit D5 records the full
  argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a render closure; the composition root is the right
  home for composition.
- **Serving assets from both embed and disk during a transition** — two sources
  of truth; the conversion is atomic within gmail.

**Verification** (over the repo-real `gmail/share/www` tree, resolved relative to
the `cmd/gmail` test package via `appkit/web`).
- R-9LIV-1C1D — an `appkit/web` Site loaded from `gmail/share/www` renders
  `landing.html` with an injected service and version visible in the output and
  `Content-Type: text/html; charset=utf-8` — proving the shipped template, not
  an embedded copy, is the page source.
- R-9MQR-F3S2 — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  gmail-side static handler registered — proving the assets ship on disk and the
  chassis serves them.
