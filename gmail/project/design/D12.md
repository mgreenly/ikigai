# Decision 12 — Prove no loopback-port literal survives, and guard the deploy artifacts against registry drift

**Decision.** The adoption in D11 is only complete if it is **enforced by a
test**, not by good intentions (the suite's "enforce with systems" principle).
Two guards, both living entirely under `gmail/` and running in the ordinary
`cd gmail && go test ./...`:

1. **A source-scan guard** proving no bare loopback-address string literal of the
   form `127.0.0.1:3xxx` remains anywhere in gmail's Go source.
2. **A drift guard** proving the two static deploy artifacts that *must* carry the
   literal port — `etc/manifest.env` and `etc/nginx.conf` — agree with the
   authoritative `registry` value, so a future renumber fails a gmail test
   instead of silently reaching deploy.

**Why two guards, and why this split.** After D11 there are two distinct kinds of
port literal left in the `gmail/` tree:

- **Go-source loopback URLs** — after D11 there are none in production code; the
  live examples are the `proxy_pass http://127.0.0.1:3202/…` string literals in
  the nginx fragment *test* (`nginx_test.go`), which guard (2) converts to a
  `registry`-derived expectation so guard (1) is clean.
- **Non-Go deploy artifacts** — `etc/nginx.conf` is an nginx fragment (nginx
  cannot call Go) and `etc/manifest.env` is generated deploy metadata; both
  legitimately contain the literal `3202`. `registry` cannot *own* their contents
  without generating them, which is out of scope. What it *can* do is make
  gmail's own tests assert those artifacts match `registry`, converting a silent
  drift into a red test.

**Guard (1) — source-scan (`R-9RMC-Y6QU`).** A test in `cmd/gmail` (a small guard
file) walks every `*.go` file under the `gmail/` module and fails if any file's
source contains a bare loopback-port address literal (`127.0.0.1:3` followed by
three digits — the whole `3000–3999` suite range, which covers gmail's own
connector-range `3202`). Two constraints make it robust:

- **It must not match itself.** The test assembles the forbidden needle from
  fragments at runtime (e.g. `"127.0.0.1:" + "3"`) rather than embedding the full
  literal, and skips its own filename, so the guard's own source can never trip
  it.
- **It scans real Go source.** After D11 + this phase there are zero such
  literals (D11 removed the `Port` int literal — not a `127.0.0.1:` string
  anyway; guard (2) below re-points the two nginx-test string literals to
  `registry`), so the scan is green; reintroducing a hardcoded loopback URL like
  `"http://127.0.0.1:3202/feed"` turns it red.

This guard covers the **string-literal** loopback URLs. It deliberately does
**not** try to flag the bare integer `3202` (an int literal is ambiguous and
noisy to grep). The own-port integer literal is instead eliminated by D11's
`registry.MustPort("gmail")` change and proven gone by guard (2).

**Guard (2) — deploy-artifact drift (`R-9SU9-BYHJ`).** Two existing tests are
re-pointed at `registry` so their expected port is *derived*, not hardcoded:

- **`cmd/gmail/main_test.go`** — the manifest byte-equality test currently emits
  `manifest.Emit(manifest.Fields{… Port: spec.Port …})` and compares to the
  committed `etc/manifest.env`. Because D11 makes `spec.Port ==
  registry.MustPort("gmail")`, the emitted field is already registry-derived; the
  guard's intent is that this test asserts the committed `PORT=3202` still
  byte-matches what `registry` yields. Keep every other assertion and every other
  `Fields` value unchanged; if `registry` renumbers gmail, `manifest.Emit`
  produces a different `PORT` line and the byte-equality assertion fails —
  surfacing that the committed `manifest.env` is stale.
- **the nginx fragment tests** (housed in `cmd/gmail` after the D9 web conversion
  dissolved `internal/web`, but re-pointed here *before* D9 in build order, so at
  this phase they still live in `internal/web/nginx_test.go`) — the exact-match
  landing location test (`R-NGNX-7F1G`) and the static location test
  (`R-41ZW-HBBA`) assert the `proxy_pass` upstream with a literal
  `http://127.0.0.1:3202/…`. Build the expected origin from
  `registry.BaseURL("gmail")` (== `"http://127.0.0.1:3202"`) and assert the
  fragment contains `"proxy_pass " + registry.BaseURL("gmail") + "/"` and
  `"proxy_pass " + registry.BaseURL("gmail") + "/static/"`. Keep the exact-match
  vs prefix distinction, the `auth_request /_session-authn` gate assertion, and
  the PRM/bearer/feed survival checks exactly as they are — only the port value
  they compare against becomes a `registry` call. (`internal/web` will need
  `registry` in scope; the module already requires it from D11.)

Both re-pointed tests keep their **existing** behavioral assertions (the session
gate, the exact-match vs prefix distinction, the manifest's non-port fields);
only the **port value** they compare against stops being a hardcoded number and
becomes a `registry` call. This removes the last `127.0.0.1:3202` string literals
from gmail's Go source while turning the static artifacts into drift-caught, not
drift-prone.

**Scope guard.** These guards add / re-point tests only — no production code
beyond D11, no new package required (a single guard test file is enough), no
schema change (**no migration**), and no edit to `etc/nginx.conf` or
`etc/manifest.env` themselves (their literals stay; the tests now police them).
Nothing outside `gmail/` is touched.

**Rejected.**
- **Regenerating `etc/nginx.conf` / `etc/manifest.env` from `registry`.** Out of
  scope and unnecessary; a drift-catching test delivers the safety without a
  codegen step or a cross-module dependency.
- **An AST/int-literal scan for the bare `3202`.** Too noisy to be falsifiable —
  `3202` carries no context that marks it as a port. Deriving the manifest and
  nginx expectations from `registry` (guard 2) proves the own-port literal is
  gone far more precisely.
- **Leaving the nginx/manifest tests as literal `3202` assertions.** They would
  keep two bare `3202` string literals in gmail's Go source (violating D11's
  end-state) and, worse, would keep *passing* after a `registry` renumber — the
  exact silent drift this adoption removes.

**Verification.**
- R-9RMC-Y6QU — **no bare `127.0.0.1:3xxx` loopback-address string literal exists
  in gmail's Go source.** A guard test walks the module's `*.go` files (skipping
  itself, needle assembled at runtime) and fails on any such literal. Falsifiable:
  reintroducing a hardcoded loopback URL like `"http://127.0.0.1:3202/feed"`
  makes it red; with D11 + guard (2) applied it is green.
- R-9SU9-BYHJ — **the committed `etc/manifest.env` and `etc/nginx.conf` ports
  agree with `registry`.** The manifest test emits with the D11-resolved
  `spec.Port` (== `registry.MustPort("gmail")`) and byte-matches the committed
  `manifest.env`; the nginx tests assert the fragment contains
  `registry.BaseURL("gmail") + "/"` and `registry.BaseURL("gmail") + "/static/"`
  proxy targets. Falsifiable: a `registry` value that differs from the committed
  artifacts' `3202` fails these tests (the intended drift alarm); matching values
  pass.
