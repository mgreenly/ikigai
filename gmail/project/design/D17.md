# Decision 17 — Attachment references in `read`/`thread` results (content plane references)

**Problem.** With D16's holder endpoint in place, the agent-facing surface must
carry the reference: the content plane's rule is *agents pass references;
services move bytes*. Today the `read`/`thread` attachment entries carry
metadata only (`filename`, `size`, `mime_type`) — `collectAttachments` drops
the Gmail attachment id on the floor, so nothing downstream can name the bytes.

**Decision.** The `read` and `thread` tools' attachment entries gain
`attachment_id` and a complete, URL-encoded `content_url`. Metadata fields are
unchanged.

1. **The reference shape.** For a MIME part with a filename **and** a non-empty
   `Body.AttachmentID`, the rendered entry becomes:

   ```json
   {
     "filename":      "2026-06-inv.pdf",
     "size":          184230,
     "mime_type":     "application/pdf",
     "attachment_id": "ANGjdJ8…",
     "content_url":   "http://127.0.0.1:3202/attachment?attachment_id=ANGjdJ8%E2%80%A6&message_id=18f2a1b3c4d5e6f7"
   }
   ```

   `content_url` is complete and URL-encoded (built with `url.Values.Encode`
   over `message_id` + `attachment_id`); the consumer never constructs or
   rewrites it. A filename-bearing part **without** an attachment id (an
   inline part whose body data rides in the message itself) stays a
   metadata-only entry — no `attachment_id`, no `content_url`: there is
   nothing at the endpoint for it to name.

2. **The base URL is resolved by registry name at the composition root**
   (dropbox's exact pattern; gmail's own loopback guard test,
   `cmd/gmail/loopback_guard_test.go`, forbids any `127.0.0.1:3xxx` literal in
   Go source — **including test expectations**, which must derive the port from
   `registry.MustPort("gmail")`):

   ```go
   // cmd/gmail/main.go, in Handlers:
   port, err := config.EnvOrInt(os.Getenv, "GMAIL_PORT", registry.MustPort("gmail"))
   ip   := config.EnvOr(os.Getenv, "GMAIL_IP", "127.0.0.1")
   contentBase := "http://" + ip + ":" + strconv.Itoa(port)
   ```

3. **The seam threads through the tool table** (revising D10's signatures —
   D10 is updated in place to match):

   ```go
   func Tools(client Client, contentBase string) []appkitmcp.Tool
   func NewHandler(client Client, contentBase string, rt *appkit.Router) (http.Handler, error)
   ```

   `renderMessage`/`collectAttachments` take the base and stamp the reference
   per attachment part. An empty `contentBase` is a wiring error and panics at
   the `NewHandler` seam, like a nil client.

4. **Tool descriptions state the current truth.** `read`'s description drops
   "Attachment blob download is not supported." and both `read` and `thread`
   say attachment entries carry `attachment_id` + a loopback `content_url` for
   byte fetch by services (not by the agent). No other descriptor or schema
   changes; the surface stays exactly twelve tools (D10's R-9NYN-SVIR).

**Documented deviation-by-omission from the full content-plane reference
tuple.** The plane's reference is `{content_url, content_hash, size, rev?}`.
gmail's reference is **`{content_url, size, mime_type}`** (plus `filename` and
`attachment_id` as metadata/addressing):

- **No `content_hash`** — Google exposes no content hash for an attachment
  without fetching the bytes; stamping one would force a full blob download
  per `read`. Consumers that need integrity verify what they fetched by size
  or hash it themselves post-fetch.
- **No `rev`, no pin dimension** — Gmail attachments are immutable per
  message; the bytes a reference names can never "move on," so the `rev`→409
  pin contract has nothing to pin. The only staleness is D16's handle
  staleness (a rotated `attachment_id`), which surfaces as 404 → re-`read`.

This is an honest narrowing, recorded here as gmail's conformance statement to
`docs/content-plane-design.md` (whose conformance map row for gmail this
Decision and D16 realize).

**Rejected.**
- **Stamping `content_hash` by prefetching each attachment during `read`** —
  turns every message read into N blob downloads; the tool is a metadata read.
- **Returning attachment bytes inline (`content_base64`) as dropbox's `get`
  does** — exactly the by-value crossing the content plane exists to retire;
  bytes never enter model context.
- **A relative or template `content_url` the consumer completes** — the plane
  requires the URL appear complete and encoded; consumers never construct.
- **Emitting `attachment_id`/`content_url` with empty values for inline
  parts** — noise that invites consumers to fetch a URL that must 404; absence
  states the truth.

**Verification** (through the assembled `appkit/mcp` handler over the existing
`fakeClient` — the D10 substrate; expected URLs built from
`registry.MustPort("gmail")`, never a port literal).

- R-X3AV-WMZ4 — a `read` of a message whose MIME tree carries an attachment
  part (filename, size, mime type, `Body.AttachmentID` containing at least one
  character requiring percent-encoding) renders that entry with `filename`/
  `size`/`mime_type` unchanged **plus** `attachment_id` equal to the raw id
  and `content_url` equal to
  `http://127.0.0.1:<registry gmail port>/attachment?<encoded query>` — and
  parsing the URL's query round-trips exactly the original `message_id` and
  `attachment_id` (proving the encoding).
- R-X4IS-AEPT — `thread` renders the same reference shape: each message's
  attachment entries carry `attachment_id` + `content_url` addressed by **that
  message's** id (two messages in one thread yield two distinct
  `message_id` values in their URLs).
- R-X5QO-O6GI — a filename-bearing part whose `Body.AttachmentID` is empty
  (inline data) renders metadata only: the entry has `filename`/`size`/
  `mime_type` and **no** `attachment_id` or `content_url` key.
