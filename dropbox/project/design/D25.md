# Decision 25 — `content_url` on loopback `/stat` and `/list` file entries

**Problem.** The event payloads carry a `content_url` (D18/D22: the
loopback `GET /content` reference other services fetch bytes through), but the
loopback filesystem API's `/stat` and `/list` responses do not. A consumer
that arrives at a file by *lookup* rather than by *event* — the scripts
runtime's `suite.files.stat`/`list` feeding `suite.fetch` is the concrete
case (observed in the 2026-07-15 live e2e, where the fetch step had nothing
to fetch) — has to hand-assemble the URL from parts it should not need to
know. The builder already exists (`contentURL(contentBase, path)` in
`internal/dropbox/events.go`); this Decision surfaces it on the lookup
responses.

**Decision.**

**1. File entries gain `content_url`; directory entries never do.** The
loopback `GET /stat` response for a file, and every `kind: "file"` entry in
the loopback `GET /list` response, carry one additional field:

```json
"content_url": "http://127.0.0.1:3200/content?path=%2Fe2e%2Fartifact.txt"
```

built by the existing `contentURL(contentBase, path)` with the entry's
canonical (D24-normalized) display path. Directory stats and `kind: "dir"`
list entries carry no `content_url` — there are no directory bytes to fetch.

**2. One content base, injected once.** The `Service` gains a `ContentBase
string` field, set at the composition root (`cmd/dropbox/main.go`) from the
same registry-resolved loopback base already wired to `NewOutboxProducer` —
one value, two consumers, zero drift. The rendering happens where the
responses are shaped: the `Entry`-to-JSON encoding used by the `/stat` and
`/list` handlers. When `ContentBase` is empty (a unit fixture that never set
it), the field is omitted rather than rendered as a broken relative URL.

**3. The MCP surface is deliberately excluded.** MCP `list` and `get` gain
**no** `content_url`: MCP callers sit outside nginx, where a loopback URL is
unreachable noise, and MCP `get` already delivers bytes as `content_base64`.
The loopback filesystem API is the only surface whose callers can dereference
the URL.

**4. Docs.** `docs/filesystem-api.md`'s `/stat` and `/list` sections document
the field: present on file entries, absent on directories, dereferenced via
loopback `GET /content`.

**Rejected.**

- **Also adding `content_url` to the MCP tools.** An unreachable URL for
  every MCP caller; it would invite agents to try fetching it and teach a
  false capability. The exclusion is pinned by id so it cannot drift in
  accidentally.
- **Letting clients assemble the URL** (documenting the `/content?path=`
  recipe instead). Spreads the service's route shape into every consumer;
  the event payloads already set the precedent that references travel as
  complete URLs (D18), and `run_fs_list` in scripts follows the same
  convention.
- **A separate resolver endpoint** (`GET /content-url?path=`). A second
  round-trip for a string the lookup response can carry for free.

**Verification.**

- R-59OM-EIY8 — loopback `GET /stat` on a file answers with `content_url`
  equal to `<contentBase>/content?path=<url-encoded canonical path>`; `GET
  /stat` on a directory answers with no `content_url` key.
- R-5AWI-SAOX — loopback `GET /list` renders `content_url` on every `kind:
  "file"` entry (URL-encoded canonical path) and on no `kind: "dir"` entry.
- R-5C4F-62FM — the MCP exclusion holds: the MCP `list` and `get` tool
  results contain no `content_url` key anywhere in `structuredContent`.
- R-5DCB-JU6B — the reference actually dereferences, proven against a real
  HTTP server (real `Service` + tempdir mirror + real index behind an
  `httptest.Server` mounting the real routes): write a file, `GET /stat` it,
  then `GET` the returned `content_url` verbatim and receive exactly the
  written bytes. A wrong implementation (mis-encoded path, wrong base, wrong
  route) fails the second fetch, not just a string comparison.
