# Decision 2 â€” Route wiring: `GET /{$}` mounted ungated through `Spec.Handlers`

**Decision.** The landing handler is mounted at the service's **exact root path**
`/` through dropbox's existing `appkit.Spec.Handlers` hook, beside the
`POST /mcp` mount and the loopback `GET /content` / `GET /list` mounts, using the
Go 1.22+ exact-match pattern `GET /{$}`:

```go
Handlers: func(r *appkit.Router) error {
    rt = r
    conn := rt.DB()
    if conn == nil {
        return fmt.Errorf("dropbox: no DB handle on router")
    }
    // â€¦ the existing mirror/client/engine wiring â€¦
    svc = dropbox.NewService(conn)

    // Landing page â€” human web UI. Ungated IN-PROCESS: nginx is the gate
    // (auth_request /_session-authn, see D4). `{$}` matches ONLY the exact root
    // path, so it never shadows /mcp, /health, /feed, /content, /list, or the
    // PRM well-known.
    rt.HandleFunc("GET /{$}", web.LandingHandler(rt.Service(), rt.Version()))

    rt.Handle("POST /mcp", rt.RequireIdentity(
        mcp.NewHandler(svc, rt.Version(), rt.Service(), rt.Health(),
            rt.Events(), rt.Subscriptions())))
    rt.Handle("GET /content", svc.ContentHandler())
    rt.Handle("GET /list", svc.ListHandler())
    return nil
},
```

**Why ungated in-process.** dropbox runs no token logic; nginx is the sole trust
boundary (Conventions). The landing route is mounted **without** an in-process
identity wrapper â€” exactly as `POST /mcp` delegates its bearer gate to nginx's
`auth_request /_authn`, the landing route delegates its cookie gate to nginx's
`auth_request /_session-authn` (D4). Wrapping it in `rt.RequireIdentity` would be
wrong: that expects the bearer-injected `X-Owner-Email`/`X-Client-Id` headers,
which a browser session request does not necessarily carry the same way, and the
page is meant to be coarse, not per-resource. (The loopback `/content`/`/list`
routes are likewise *not* wrapped â€” they self-guard in-handler â€” and are a
separate concern from the landing page.)

**Why `{$}` and not `/`.** In the Go 1.22+ `http.ServeMux` pattern language, a
trailing `/` is a **subtree** pattern that matches every path under it, while
`{$}` matches **only** the exact path. dropbox's internal routes are `/`, `/mcp`,
`/health`, `/feed`, `/content`, `/list`, and
`/.well-known/oauth-protected-resource` (nginx has already stripped the
`/srv/dropbox/` mount prefix). Mounting the landing page at `GET /{$}` guarantees
it answers the bare root **and nothing else** â€” it cannot shadow `/mcp`,
`/content`, or any other route. A subtree `/` would swallow unmatched paths and
is explicitly rejected.

**Composition-root growth, not a rewrite.** `cmd/dropbox/main.go` already carries
the `Handlers` hook (it builds the mirror, client, engine, and Service and mounts
`POST /mcp`, `GET /content`, `GET /list`); this Decision adds **one line** to that
existing hook. That is the sanctioned incremental growth of the one shared wiring
file, not a rewrite of dropbox's domain â€” the Service/Producer/Workers wiring and
the byte routes stay intact.

**Rejected.**
- **A subtree `GET /`** â€” swallows every unmatched path (would intercept a typo'd
  `/contentx`, future routes, etc.) and muddies the "exact root only" contract.
  `{$}` is precise.
- **Wrapping the route in `rt.RequireIdentity`** â€” double-gates with the wrong
  (bearer) mechanism and breaks the coarse cookie-gated human door; the gate is
  nginx's job (D4).
- **A second `appkit.Spec` field or a new hook for web pages** â€” unnecessary; the
  existing `Handlers` hook mounts arbitrary routes, and the landing page is just
  another route on the same mux.
- **Mounting the page on a sub-path** (e.g. `/landing`) â€” the product constant is
  that the human reaches it at the mount **root** (`/srv/dropbox/`), which is the
  service's `/`.

**Verification.**
- R-ROUT-2B5C â€” with the landing handler registered on an `http.ServeMux` at
  `GET /{$}` (as the composition root wires it), a `GET /` is dispatched to the
  landing handler and returns the landing page (`200`, name+version body).
- R-ROUT-4D7E â€” the `{$}` exact-root registration does **not** shadow a sibling
  `POST /mcp` registration on the same mux: a `POST /mcp` still dispatches to the
  mcp handler, not the landing handler.
- R-ROUT-6F9G â€” a non-root path that is not otherwise registered (e.g.
  `GET /nope`) is **not** captured by the `{$}` landing pattern (it does not
  return the landing page) â€” proving `{$}` is exact-match, not a subtree.
