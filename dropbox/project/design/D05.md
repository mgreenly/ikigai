# Decision 5 — Docs state current truth: purge the stale "no UI" line

**Decision.** dropbox's service doctrine doc — `dropbox/CLAUDE.md` — currently
asserts in its opening one-liner that dropbox is "A loopback-only **daemon +
event-plane producer** (not an API wrapper) with **no UI** and **no token logic
of its own**…". That "no UI" claim is now **false**: dropbox serves a human web
landing page (D1–D4). Per the suite doctrine — *docs state current truth; purge
what the change falsified, rewrite in place, no archaeology, no footnotes* — this
Decision **rewrites that line in place** to state the landing-page truth.

> **Doc layout note (delta from the crm template).** Unlike crm, dropbox has **no
> `AGENTS.md` symlink** — `dropbox/CLAUDE.md` is a single regular file and is the
> doctrine doc to edit directly. There is no symlink invariant to preserve here;
> the edit goes straight into `dropbox/CLAUDE.md`. (The brief's blanket "do not
> touch AGENTS.md now" still holds — this is plan work, not done by the doc
> author — and is moot for dropbox, which has no `AGENTS.md`.)

**What changes (doc edits only — no code):**

- Remove the "**no UI**" claim from `dropbox/CLAUDE.md`'s opening one-liner
  ("…with **no UI** and **no token logic of its own**…"). The "no UI" string
  appears only there in the file.
- Replace it with the current truth, in dropbox's own voice: dropbox is a
  loopback-only mirror-sync daemon + event-plane producer that serves an **MCP
  surface for agents** (bearer-gated) **and a human web landing page**
  (session-cookie-gated) under `/srv/dropbox/`; it still runs **no token logic of
  its own** — nginx remains the sole trust boundary for both doors. Keep the "no
  token logic of its own" half (still true); drop only the "no UI" half (now
  false).
- Do **not** add a changelog, a "previously…", or a footnote. State the present
  as if it were always so.

**Why this is its own phase, performed by the build loop — not by the doc
author.** The doc author (this design+plan workspace) deliberately does **not**
edit `dropbox/CLAUDE.md` now: code and docs must evolve **together**, so the
purge is sequenced as a plan phase that the build loop performs **after** the
landing page exists (D1–D4 shipped). Doing it earlier would make the docs claim a
UI the code does not yet serve; doing it never would leave a false statement on
`main`.

**Scope guard.** Only the "no UI" falsehood is purged. dropbox's other doctrine —
the mirror-sync daemon model, the four-tool MCP surface, the event-plane producer
edges, the loopback `/content`/`/list` byte routes, the header-trust model, "no
token logic of its own," the loopback bind, the no-backup decision, the
deploy/manifest sections — is **unchanged and still true**; this Decision does
not touch it.

**Verification.** This is a **documentation** Decision with no Go behavior, so it
mints no `R-XXXX-XXXX` ids; it is realized by a **structural** plan phase whose
done bar is a content check rather than an id-tagged test:

- `dropbox/CLAUDE.md` no longer contains the string `no UI` (case-insensitive) —
  `grep -i "no UI" dropbox/CLAUDE.md` finds nothing.
- `dropbox/CLAUDE.md` states the landing-page truth: it mentions that dropbox
  serves a human web **landing page** under `/srv/dropbox/` alongside the MCP
  surface.
- The suite stays green (`bin/check-migrations dropbox` and
  `cd dropbox && go build ./...` unaffected — this is a docs-only change).
