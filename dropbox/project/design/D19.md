# Decision 19 ‚Äî MCP write tools: reference-based `put` + capped inline convenience

**Problem.** The loopback streaming routes (D16) serve on-box services, but an
**off-box agent** (no local mount, no loopback reach ‚Äî a prompts sandbox has no
network) can only mutate the mirror through MCP. The write tools exist
(`put`/`mkdir`/`delete`/`move` over the D16 `Service` methods), but `put` accepts
bytes **only as base64 in the tool call** ‚Äî every write is bounded by the model's
context and the 25 MiB base64 cap, and a file held by another suite service
(a gmail attachment, a prompts run artifact) cannot reach the mirror without its
bytes round-tripping through an agent. The suite's content plane
(`docs/content-plane-design.md`) fixes the rule: *agents pass references;
services move bytes* ‚Äî every agent-facing ingestion verb takes a **`source_url`**
and the service fetches server-side. Its conformance map names dropbox's MCP
`put(path, source_url)` as the acceptor half dropbox owes.

**Decision.** dropbox declares four domain write tools over the same `Service`
write methods (D16), added to `internal/mcp.Tools` beside `list`/`get`. `put` is
**reference-based first**: its primary form takes a `source_url` the service
fetches itself; the capped base64 form remains as the plane's sanctioned inline
convenience (small content an agent authored has no holder to reference).

```go
// internal/mcp ‚Äî declared alongside the existing list/get.
//   put(path, source_url | content_base64)  ‚Üí Service.Write
//     exactly ONE of source_url / content_base64 (both or neither ‚Üí validation)
//     source_url:     fetched SERVER-SIDE and streamed into Service.Write ‚Äî
//                     no size cap (the 25 MiB cap is a base64-in-JSON
//                     constraint; the mirror write streams, D14, and the
//                     uploader chunks >150 MiB via upload_session, D17)
//     content_base64: decoded, capped at 25 MiB (too_large above it, like get)
//   mkdir(path)                ‚Üí Service.Mkdir
//   delete(path)               ‚Üí Service.Delete   (file or dir, recursive)
//   move(from, to)             ‚Üí Service.Move
//
// The allowed-port seam: the confinement check takes the permitted loopback
// ports as DATA, derived from the service registry at the composition root, so
// tests can inject a set containing an httptest server's port while production
// wiring admits exactly the registered ports.
func Tools(svc *dropbox.Service, sourcePortAllowed func(port int) bool) []mcp.Tool
func NewHandler(svc *dropbox.Service, sourcePortAllowed func(int) bool, rt *appkit.Router) (http.Handler, error)

// cmd/dropbox/main.go ‚Äî the production allowed-port set is the registry:
//   allowed := map[int]bool{}; for _, s := range registry.Services { allowed[s.Port] = true }
//   ‚Ä¶ NewHandler(svc, func(p int) bool { return allowed[p] }, rt)
```

- **`source_url` confinement (the content-plane acceptor rule, validated before
  any network I/O).** The URL must parse, use scheme `http`, have host literally
  `127.0.0.1` or `::1` (no hostnames ‚Äî `localhost` is rejected; the plane is
  loopback plumbing, not name resolution), and carry an explicit port for which
  `sourcePortAllowed` is true. Any violation is a **`validation`** tool error and
  the service performs **zero** fetch calls. dropbox never fetches an arbitrary
  URL on an agent's say-so.
- **Server-side fetch, streamed.** A conforming `source_url` is fetched with a
  plain `GET` (a dedicated `http.Client` with a bounded connect/response-header
  timeout and **no overall body deadline** ‚Äî a large stream must be allowed to
  finish); the response body is streamed straight into `Service.Write`, which is
  the existing mirror-write + index/event tx + `upload_queue` enqueue seam (D16)
  ‚Äî temp+rename atomicity means a torn stream aborts with **no** index, event, or
  queue change.
- **Pinned failure mapping** (what the MCP caller sees, via the existing
  `{error:{code,message}}` envelope; the message names the `source_url` detail):
  - malformed / non-`http` / non-loopback / unregistered-port `source_url`,
    both-or-neither of `source_url`/`content_base64` ‚Üí **`validation`**;
  - source responds **404** ‚Üí **`not_found`** ("the reference is stale/absent ‚Äî
    re-derive it from the holder");
  - source responds **409** (a holder's rev-pin moved on) ‚Üí **`conflict`**;
  - connection refused, timeout, any other non-2xx, or a torn body ‚Üí
    **`source_unavailable`** (a transient source fault, retryable ‚Äî
    distinguishable from absence, the gmail 404-vs-502 lesson).
  In every failure case the mirror, index, and `upload_queue` are unchanged.
- **Result shape** ‚Äî both `put` forms return what `Service.Write` returns:
  `{path, size, content_hash, rev}`.
- **Same seam, same semantics.** Each tool calls the identical `Service` method
  the loopback route calls, so behavior (confinement, created-vs-modified,
  recursive delete idempotency, one-op move, upload enqueue, origin) is shared ‚Äî
  the MCP tools add only argument decoding, the source fetch, and the
  sentinel‚Üíerror-code envelope
  (`not_found`/`conflict`/`validation`/`too_large`/`source_unavailable`/`internal`),
  reusing the existing `toolErr` mapping.
- **Origin.** An MCP write's `origin` is the authenticated caller's `client_id`
  (the chassis threads it as `server.Identity` ‚Äî nginx-injected `X-Client-Id`, or
  set by a loopback caller as prompts runs do), passed to the `Service` methods'
  `clientID` parameter exactly as today (D18).
- **Inline cap.** `content_base64` rejects a decoded body over 25 MiB with
  `too_large`, mirroring `get` ‚Äî the inline path is deliberately the small-file
  convenience; anything large travels by reference or over `PUT /content`.
- **Surface count.** The declared domain table stays **6** (`list`, `get`, `put`,
  `mkdir`, `delete`, `move`); with the chassis `health`+`reflection` the
  advertised surface is **8** tools. `list`/`get` and the loopback filesystem
  HTTP API are unchanged.

**Rejected.**
- **Keep `put` base64-only.** Leaves dropbox non-conformant with the content
  plane's acceptor contract, caps every agent write at 25 MiB of context, and
  forces service-held bytes through a model to reach the mirror.
- **Drop the base64 form entirely.** The plane explicitly sanctions a capped
  inline convenience, and without it an agent authoring a *new* small file has no
  holder to mint a reference from; the behavior is built and keeps its id.
- **A separate `put_url` tool.** Two tools for one write muddy the surface and
  the docs; one `put` with mutually exclusive body forms keeps "write a file"
  a single verb.
- **Fetching arbitrary URLs (or hostnames).** The plane's confinement is
  absolute: internal plumbing only, `127.0.0.1`/`::1` + registered ports;
  internet ingestion is a connector's domain job, never a generic fetch.
- **A size cap on the `source_url` form.** The 25 MiB cap exists because
  base64-in-JSON is buffered whole; the server-side path streams end to end
  (D14) and the uploader already handles >150 MiB via `upload_session` (D17).
  Capping it would reintroduce the limit the reference form exists to remove.
- **Stream large writes through MCP.** base64-in-JSON can't stream and would
  reintroduce whole-file residency ‚Äî the exact thing D14 removes.
- **A single polymorphic `write` tool.** Overloading create/mkdir/delete/move
  into one tool muddies schemas and error mapping; four verbs match the four
  Service methods one-to-one.

**Verification** (through the composition-root seam ‚Äî the `appkit/mcp` handler
built by `NewHandler` over a domain `Service`, driven through the real `ServeHTTP`
seam, exactly as the existing `list`/`get` tests do; a fake upload sink stands in
for Dropbox. The `source_url` fetch claims run against a **real local HTTP
server** ‚Äî `httptest` on loopback ‚Äî with its port injected via the allowed-port
seam; the registry derivation is proven separately against the real
`registry.Services`).
- R-KRXK-IQE2 ‚Äî `tools/list` returns **8** tools: `list`, `get`, `put`, `mkdir`,
  `delete`, `move` (dropbox-declared) plus chassis `health`, `reflection` ‚Äî the
  expanded partition (a table declaring `health`/`reflection` would fail the
  reserved-name check; a missing write tool fails the count) ‚Äî and `put`'s input
  schema advertises both `source_url` and `content_base64`.
- R-KT5G-WI4R ‚Äî MCP `put` of a small base64 body writes the file (a following
  `get` returns the bytes) and enqueues an upload; a decoded body **over 25 MiB**
  is rejected with `too_large`.
- R-KUDD-A9VG ‚Äî MCP `mkdir`/`delete`/`move` invoke the same `Service` methods as
  the loopback routes (behavior parity: empty dir listable, recursive delete,
  one-op move), and a failure surfaces through the sentinel‚Üícode error envelope.
- R-Q52B-JQLP ‚Äî MCP `put(path, source_url)` against a **real local HTTP server**
  serving known bytes fetches them **server-side** (the server records exactly one
  `GET`), streams them through `Service.Write` ‚Äî a following `get` returns the
  identical bytes, an upload is enqueued for the path, the emitted event's
  `origin` is the caller's client id ‚Äî and the result is
  `{path, size, content_hash, rev}` matching the written file.
- R-Q6A7-XICE ‚Äî `source_url` confinement discriminates, with **zero fetch
  calls** on every rejection: a non-loopback host (e.g.
  `http://10.0.0.5:3202/‚Ä¶`) ‚Üí `validation`; a hostname (`http://localhost:‚Ä¶`) ‚Üí
  `validation`; a **loopback host on a port the allowed set does not contain** ‚Üí
  `validation`; the byte-equivalent URL on a loopback host with an **allowed**
  port proceeds to fetch. Supplying both `source_url` and `content_base64`, or
  neither, ‚Üí `validation`.
- R-Q8Q0-P1TS ‚Äî the pinned failure mapping: the source responding **404** ‚Üí
  `not_found`; the source responding **409** ‚Üí `conflict`; a connection-refused
  target (a closed allowed port) ‚Üí `source_unavailable` ‚Äî and after each failure
  the mirror holds no file at `path`, the index has no row, and `upload_queue`
  is empty (nothing was written, emitted, or enqueued).
- R-Q9XX-2TKH ‚Äî the production allowed-port set is **registry-derived**: the set
  built the composition root's way from `registry.Services` returns true for
  **every** registered service port (3000, 3200, 3202, ‚Ä¶) and false for
  unregistered ports **8080** and **39999** ‚Äî a hardcoded or partial set fails.
