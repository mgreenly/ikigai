# Decision 4 — nginx fragment: the exact-match session-gated `= /srv/dropbox/` location

**Decision.** dropbox's nginx fragment (`dropbox/etc/nginx.conf`) gains **one new
location** — an **exact-match** location for the bare mount root that
session-gates the human landing page and proxies it to the service's root. It
sits alongside (does not replace) the existing locations:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated /srv/dropbox/ prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/dropbox/ {
    auth_request /_session-authn;
    auth_request_set $dropbox_session_owner $upstream_http_x_owner_email;
    proxy_set_header  X-Owner-Email $dropbox_session_owner;
    proxy_pass http://127.0.0.1:__PORT__/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**How it coexists with the existing fragment.** dropbox's fragment already has,
in source order:

1. `location = /srv/dropbox/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match).
2. `location = /srv/dropbox/content { return 404; }` — defence-in-depth: the
   loopback byte endpoint `GET /content` is never reachable through the public
   edge (exact-match; belt to the handler's own identity-header guard).
3. `location /srv/dropbox/ { auth_request /_authn; … }` — the bearer-gated
   **prefix** for the MCP endpoint and everything else under the mount, carrying
   the `$authn_status`/`error_page 500 = @dropbox_authn_500` rate-limit re-emit
   machinery.
4. `location @dropbox_authn_500` — the named location the prefix block's
   `error_page` jumps to, which re-emits a faithful `429` (with `Retry-After`)
   when introspection was actually rate-limited, else a real `500`.

nginx longest-prefix/exact-match rules make an **exact-match `=` location win over
a prefix location** for that one path regardless of source order. So the new
`= /srv/dropbox/` exact match wins for the **bare root only**, while the
`/srv/dropbox/` prefix (item 3) still serves `/srv/dropbox/mcp` and every other
sub-path under the mount. The two PRM/content exact matches (items 1–2) are
unaffected (different exact paths), and the `@dropbox_authn_500` named location
(item 4) is reached only from the prefix block and is untouched. Nothing in the
existing fragment is removed or rewritten — the change is **purely additive**.

> **Delta from the crm template.** dropbox's fragment has **no**
> `= /srv/dropbox/feed { return 404; }` location — its `/feed` is safe purely on
> the handler's identity-header guard (per `dropbox/CLAUDE.md`), so unlike crm
> there is no feed 404 block to assert. The exact-match dropbox carries instead is
> the `= /srv/dropbox/content` defence-in-depth block, plus the
> `@dropbox_authn_500` named location the bearer prefix needs. The retained-locations
> assertion (R-NGNX-8V1W) enumerates *these* — not a feed block.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + `X-Owner-Email` on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier (`sites/etc/nginx.conf` `location /srv/sites/private/`). The gate is
deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @dropbox_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location does not need the 429 re-emit dance — it
maps the session subrequest's allow/deny directly (`200`→serve, `401`→401),
matching the `sites` private-tier location, which likewise omits it. (It also
takes only the owner from the session subrequest — no `X-Client-Id`, since a
browser session is not a registered MCP client.)

**Templating.** `__PORT__` stays templated (substituted from `etc/manifest.env`
`PORT=3005` at install time by `opsctl setup dropbox` on the box / `nginx/run` in
dev — which regenerates `nginx/locations/dropbox.conf` from `etc/nginx.conf` and
**must include `dropbox`** in its service loop), exactly like the existing
locations. The fragment is **not** a vhost — no `server{}`, no `listen`, no cert;
it is dropped at `/etc/nginx/conf.d/locations/dropbox.conf` and included by the
dashboard's apex server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get `401`. The page is gated, just coarsely.
- **A prefix `location /srv/dropbox/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@dropbox_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Editing the existing `/srv/dropbox/` prefix block or the `= /srv/dropbox/content`
  defence-in-depth block** — unnecessary and risky; the change is purely an added
  exact-match location.

**Verification.** (Proven by a test that reads `dropbox/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-2P4Q — `dropbox/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/dropbox/ {` block (distinct from the
  prefix `location /srv/dropbox/ {`).
- R-NGNX-4R6S — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-6T8U — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:__PORT__/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-8V1W — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/dropbox/ {` prefix with `auth_request /_authn`, the
  `location = /srv/dropbox/content { return 404; }` defence-in-depth block, the
  `location = /srv/dropbox/.well-known/oauth-protected-resource` PRM bootstrap, and
  the `location @dropbox_authn_500` named location all still appear in the file.
