# Decision 22 — Event-routing conformance: kinds `create`/`modify`/`delete`, subject = mirror path

> **External ordering (operator-sequenced, not built by any dropbox phase).**
> This Decision consumes the **revised eventplane API** — `outbox.Event{Kind,
> Subject, Payload}`, `outbox.Family`/`Registry`, the kind/subject envelope and
> canonical-key SSE framing — specified in `eventplane/project/design/` (its
> D1–D4) and realized by **eventplane plan phases 01–04, which must be BUILT
> before dropbox's conformance phase runs**. It equally assumes appkit (which
> holds `Spec.Events outbox.Registry` and the reflection tool over it) compiles
> against that revision — appkit's own conformance is appkit's spec, out of
> scope here. Until both land, dropbox's conformance phase cannot build.

**Problem.** The suite's event addressing is revised
(`docs/event-routing-design.md`): the envelope's flat `type` becomes routing
fields `kind` + `subject`, producer registries become **families**, and the old
type names drop their redundant noun prefix (`source` already says dropbox deals
in files). Crucially for dropbox, the revision exists *for* dropbox's shape: a
consumer that cares about `/bills/**` must today be triggered by **every** file
event on the box because the path lives only in the payload, where filters
cannot see it. It is a hard cutover — no compatibility period — inside the
no-live-data migration window.

**What dropbox publishes today (ground truth, not vacuous).** dropbox is a
real, wired producer: `Spec.Feed: "/feed"`, `Spec.Events: dropbox.Events` (an
`outbox.Registry` of three `EventType`s in `internal/dropbox/events.go`), the
`Producer` hook attaching the outbox as the Service's `EventSink`, and two
emission paths — the download engine's apply path (origin `dropbox`) and the
D16 write path (origin = caller client id) — each appending
`outbox.Event{Type, Payload}` via `buildFilePayload` on the same tx as the
index change. All three published types (`file.created`/`file.modified`/
`file.deleted`) convert.

**Decision.** dropbox adopts the revised envelope exactly over what it
publishes, with the **mirror display path as the subject**:

| today (`type`) | revised key (example) | kind | subject |
|---|---|---|---|
| `file.created` | `dropbox:create/bills/aws/2026-06.pdf` | `create` | the `/`-rooted mirror display path |
| `file.modified` | `dropbox:modify/bills/aws/2026-06.pdf` | `modify` | the `/`-rooted mirror display path |
| `file.deleted` | `dropbox:delete/bills/aws/2026-06.pdf` | `delete` | the `/`-rooted mirror display path |

The subject is the event's **display path** (`FileEvent.Path` — already
`/`-rooted, e.g. `/notes/meeting.md`), the same literal path the payload
carries; a folder delete still fans out one event per removed file, each with
**that file's** path as its subject. Payload fields (`path`, `rev`,
`content_hash`, `size`, `content_url`, `origin`, `occurred_at`) are
**unchanged** — the routing key addresses; the payload remains the detail
record. The payload's `event` discriminator field (`"event":"file.created"`)
is **dropped**: the envelope's `kind` is the discriminator now, and keeping a
second, stale-named copy of it in the payload invites drift.

1. **Constants and payload builder** (`internal/dropbox/events.go`):
   `EventFileCreated = "file.created"` → `KindCreate = "create"` (likewise
   `KindModify = "modify"`, `KindDelete = "delete"`; `FileEvent.Type` becomes
   `FileEvent.Kind` and its construction sites in the sync apply path and the
   D16 `Service` write methods follow). `buildFilePayload` emits
   `outbox.Event{Kind: ev.Kind, Subject: ev.Path, Payload: …}` and the
   `filePayload` struct loses its `Event` field. Nothing else in the
   Service/EventSink seam changes (same tx as the index change, `Ring()` after
   commit; the `OriginDropbox` sentinel and D18 origin threading untouched).

2. **The registry becomes families** (`internal/dropbox/events.go`):
   `dropbox.Events` becomes an `outbox.Registry` of three `outbox.Family`
   entries — `{Kind: "create", Subject: "/<mirror path>", Description: …,
   Sample: sampleFilePayload(…)}` etc. Descriptions keep their current meaning
   with the addressing updated (a reference, never bytes; origin's two forms).
   `Spec.Events` wiring is untouched; Append-time kind gating and the chassis
   reflection surface hang off the same registry, now family-shaped, with
   schema/example still derived from the one Sample (which no longer carries an
   `event` key).

3. **The outbox schema converts by a NEW migration.** eventplane's revised
   `outbox.SchemaSQL` replaces the `type` column with `kind` + `subject`; each
   producer re-applies the DDL through its own runner. dropbox appends **one
   new timestamped migration** minted with `bin/create-migration dropbox
   outbox_routing` (never a hand-picked number) that drops and recreates the
   outbox table per the revised `outbox.SchemaSQL` verbatim (a `DROP TABLE`/
   `DROP INDEX` preamble + the constant's DDL). Outbox rows are disposable
   retention data inside the migration window; nothing is migrated across.
   The frozen `003_outbox.sql` is **not touched**; the byte-equality drift
   guard (`internal/db/migrations_outbox_test.go`) re-points at the **newest**
   outbox migration, asserting it contains the revised `outbox.SchemaSQL`
   verbatim. (This exactly mirrors gmail's D18/phase-16 handling.)

4. **Reflection/consumer surface.** The chassis reflection tool (appkit-owned)
   reflects the family registry; dropbox's observable is that its three kinds —
   and only they — are listed with the `/<mirror path>` subject description.
   dropbox consumes nothing (producer only), so there is no
   filter/`consumer.Config` side to conform.

**Rejected.**
- **Keeping `file.` prefixes as kinds (`file.created`)** — the revision's
  explicit rename rule: `source` carries the noun; a `dropbox:file.created` key
  says "file" twice.
- **An empty subject (gmail's choice)** — dropbox is the motivating case for
  subjects: the mirror path is exactly the routing dimension consumers need
  (`dropbox:create/bills/**/*.pdf`); leaving it empty would preserve the
  fire-on-everything failure the revision exists to fix.
- **Keeping the payload's `event` discriminator (renamed to the kind)** — a
  duplicated copy of an envelope field that can only drift; consumers route on
  the key and read details from the payload.
- **Editing `003_outbox.sql` in place to the new DDL** — committed migrations
  are immutable; the runner would silently skip the edited body on existing
  databases. A new migration is the only sanctioned schema change.
- **An in-place `ALTER TABLE` migration preserving outbox rows** — pointless
  ceremony inside the migration window (rows are retention data with no live
  consumers to preserve continuity for) and it would have to fabricate
  kind/subject for old rows.

**Verification** (real substrates throughout: the emission claims run over the
existing hermetic harness — temp SQLite through the full embedded migration
set, temp mirror, the real `Service` paths — reading outbox rows back by SQL;
the feed claim runs the real eventplane `FeedHandler` over `httptest`).

- R-QB5T-GLB6 — after a download-engine apply creates a file, a D16 service
  write overwrites it, and a folder delete removes two indexed files, the
  outbox rows read back by SQL from the real SQLite database carry `kind`
  values exactly `create`, `modify`, and `delete` (no `file.` prefix), each
  row's `subject` equal to **that file's** `/`-rooted display path (the two
  delete rows carry two different subjects), each appended on the same tx as
  the index change — and each row's payload JSON has exactly the keys `path`,
  `rev`, `content_hash`, `size`, `content_url`, `origin`, `occurred_at` and
  **no** `event` key.
- R-QCDP-UD1V — `dropbox.Events` declares exactly three families with kinds
  `create`/`modify`/`delete` and subject description `/<mirror path>`; through
  the assembled MCP handler, reflection's index lists exactly those three
  publishes, the detail of `create` returns a schema and example that agree
  (every example field in the schema, `filePayload`-shaped, `origin` present,
  no `event` field), and an undeclared kind returns the typed unknown-kind
  error naming the three declared kinds.
- R-QDLM-84SK — applying dropbox's full embedded migration set (through the
  appkit runner) to a fresh real SQLite database yields an `outbox` table with
  `kind` and `subject` columns and **no** `type` column; the DDL drift guard
  asserts the newest outbox migration contains the revised `outbox.SchemaSQL`
  verbatim while `003_outbox.sql` remains byte-identical to its committed
  (frozen) body.
- R-QETI-LWJ9 — end-to-end over dropbox's own substrate: an event appended
  through dropbox's producer path (`buildFilePayload` → `Append` on the
  migration-created table, source `dropbox` — the `Spec.App` value) for a
  created file at `/notes/meeting.md` and served by the real eventplane
  `FeedHandler` over an `httptest` connection frames
  `event: dropbox:create/notes/meeting.md` (the canonical key) with a `data:`
  JSON envelope carrying `kind` and `subject` keys and **no** `type` key.
