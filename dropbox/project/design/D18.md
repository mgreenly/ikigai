# Decision 18 — Origin-tagged file events

**Problem.** dropbox is now both an event **producer** and (across the suite) part
of a producer/consumer web where a service can be both. Every mirror change emits
a `create`/`modify`/`delete` file event (D22 addressing), but the event does not say **who** caused it.
If a service writes a file and also consumes file events, an actor-less event
means it reacts to its own write — a self-trigger loop that is a real footgun on
an event plane. And an operator debugging the feed cannot tell a human/Dropbox
change from a suite-service write.

**Decision.** Add an `origin` field to the file event payload, carried on the same
`/feed` for every event, filled from where the change came from.

```go
// internal/dropbox/events.go
type FileEvent struct {
    // …existing fields…
    Origin string // writing service's X-Client-Id, or the sentinel "dropbox"
}

type filePayload struct {
    // …existing fields…
    Origin string `json:"origin"` // "dropbox" for a pulled change; the client id for a service write
}
```

- **Where it is set.** The write path (D16) threads the caller's `X-Client-Id`
  into `FileEvent.Origin`. The **download** apply path (existing `applyUpsert`/
  `applyRename`/`applyDelete`) sets `Origin` to the sentinel **`dropbox`** — a
  pulled change is authored by Dropbox/an external device, not by a suite service.
- **Additive and reflected.** `origin` is a purely additive JSON field; existing
  consumers ignore unknown fields. The `Events` registry's `Sample` filePayload
  gains `origin`, so the `reflection` tool's emitted JSON Schema **and** worked
  example both describe it (schema/example/wire shape can't diverge — they share
  the one Sample). The event descriptions note the field and its two forms.
- **Consumer use.** A consumer that is also a writer filters events where
  `origin == <its own client id>` to avoid reacting to its own writes; a consumer
  that only wants human/Dropbox-authored changes filters `origin == "dropbox"`.
  dropbox does not enforce any of this — it only records provenance ("explicit
  over implicit").

**Rejected.**
- **No actor (events identical to today).** Simplest, but leaves the self-trigger
  footgun and gives operators no provenance.
- **A separate boolean `internal`.** Coarser than the client id — it can't tell
  *which* service wrote, so a self-writing consumer still can't distinguish its
  own writes from a sibling's.
- **A new schema column / new event types.** The provenance is payload data, not
  a table concern; no migration is needed, and forking per-origin event kinds (a `create.byservice`)
  event types would double the surface for one field.

**Verification** (producer tests with a recording sink capturing the marshaled
payload; the reflection assertion through the assembled MCP handler).
- R-KO9V-DF5Z — a service-originated write (D16) emits an event whose `origin`
  equals the writing caller's `X-Client-Id`.
- R-KPHR-R6WO — a Dropbox-pull change (a `list_folder` delta applied by the
  download engine) emits an event whose `origin` is the sentinel **`dropbox`**.
- R-KQPO-4YND — the `reflection` tool's file-event detail (per published kind, D22 addressing) includes `origin` in
  both the JSON Schema and the worked example (driven by the shared `Sample`).
