# Decision 15 — First-class directories in the index

**Problem.** The index tracks **files only** (`files` table; "the index does not
track directories"). A directory is implied by the files under it, so an **empty**
directory cannot be represented: `mkdir /a/b` with nothing inside it would vanish
from `list`/`stat`, and could not be pushed to Dropbox as a folder. The
service-facing filesystem promises real-filesystem semantics ("how any Dropbox
user sees it"), and real filesystems have empty directories and lay down directory
skeletons before writing into them. Directories become first-class.

**Decision.** Add a `directories` table (a new additive migration) parallel to
`files`, and teach the store/service/mirror to treat directories as real entries.

```sql
-- internal/db/migrations/004_directories.sql  (additive; created via
-- bin/create-migration dropbox directories — timestamped, not hand-numbered)
CREATE TABLE directories (
    path        TEXT PRIMARY KEY,   -- current DISPLAY path, no trailing slash
    path_lower  TEXT NOT NULL,      -- case-folded, matched like files.path_lower
    updated_at  TEXT NOT NULL
);
CREATE INDEX idx_directories_path_lower ON directories (path_lower);
```

```go
// internal/dropbox — store + service directory surface.
func (s *Store) UpsertDir(tx *sql.Tx, path string) error
func (s *Store) DeleteDirSubtree(tx *sql.Tx, path string) (files, dirs []string, err error) // recursive
func (s *Store) RenameDirSubtree(tx *sql.Tx, old, new string) error

// Service.Stat resolves a path to a file OR a directory entry; unknown → not_found.
func (s *Service) Stat(path string) (Entry, error) // Entry.Kind ∈ {file, dir}
// Service.List (and the recursive walk it already backs) reports directories
// interleaved with files, ordered by path.
```

- **Directories are structural — no file-lifecycle events.** Consistent with the
  existing "folders are structural" rule, `mkdir`/`rmdir` themselves emit **no**
  file-lifecycle event (a directory is not a file). A **recursive delete** of a
  directory still emits one `delete`-kind file event (D22 addressing) per file removed beneath it (the
  existing `applyDelete` fan-out), and removes the directory rows in the same tx.
- **Case-folding is identical to files.** `path_lower` is matched for Dropbox's
  case-insensitive semantics, exactly as `files.path_lower`, so a directory and a
  file cannot collide on a case-only difference.
- **Push mapping (realized in D17).** `mkdir` enqueues a `files/create_folder_v2`;
  a recursive `delete` enqueues a `files/delete_v2` of the folder; a directory
  `move` enqueues one `files/move_v2` (see D16/D17).
- **Implicit parents still work.** Writing `/a/b/x.md` still auto-creates the
  on-disk parent dirs (`MkdirAll`, unchanged); those implicit parents are indexed
  as directory rows too so `list`/`stat` see a coherent tree.

**Rejected.**
- **Directories implicit (a dir exists iff a file is under it).** Simplest and
  no schema change, but an empty `mkdir` then `list` shows nothing — it silently
  breaks the real-filesystem mental model the moment someone creates an empty dir.
- **A `kind` discriminator column on `files`.** Overloads a table whose every
  other column (rev, content_hash, size) is meaningless for a directory, and
  every existing `files` query would need a `WHERE kind='file'` retrofit. A
  separate table keeps file queries untouched.
- **Reconstruct the tree from paths at query time.** Cannot represent an empty
  directory at all, and pushes O(n) path-prefix work into every `list`.

**Verification** (store/service tests over a temp SQLite DB + temp mirror; the
migration applied by the appkit runner is the real substrate for the schema
claim).
- R-JZVV-Q0C3 — `mkdir` of an empty path makes it appear in `Stat` (as `Kind:dir`)
  and in `List`, with **no** file under it; the implicit-dir model (which would
  return not_found) is excluded.
- R-K13S-3S2S — a recursive `Delete` of a directory removes the directory row(s)
  and **every** file beneath it, emitting exactly one `delete`-kind file event per removed
  file (count matches the subtree file count), in a single tx.
- R-K2BO-HJTH — a directory `move`/rename reparents the directory and its whole
  subtree: after moving `/a` to `/b`, `/a` and `/a/x.md` are absent and `/b` and
  `/b/x.md` are present, with folded paths updated.
- R-K3JK-VBK6 — migration `004_directories` is recorded in `schema_migrations`
  and the `directories` table exists with a `path_lower` fold that prevents a
  case-only collision between a directory and an existing file.
