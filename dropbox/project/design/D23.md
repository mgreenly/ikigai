# Decision 23 — Structured MCP adoption

**Problem.** dropbox's MCP surface predates the suite's **structured-MCP**
contract (`docs/structured-mcp-design.md`). Its domain tools build results with
`appkit/mcp.JSONResult(v)` — a Go value marshalled to a single text block — and
its errors ride a hand-built `{error:{code,message}}` body passed to the old
string-taking `ErrorResult`. Neither declares an `outputSchema`. Meanwhile the
three loopback filesystem routes each copy the same two-header rejection
(`X-Owner-Email` **or** `X-Forwarded-Proto`) inline. appkit has since landed the
suite contract (protocol `2025-06-18`; `Tool.OutputSchema`; a re-signed
`StructuredResult(v) (map[string]any, error)`; a typed `ErrorResult(code
ErrorCode, msg)`; the deletion of `JSONResult`; and a shared loopback guard
`server.LoopbackOnly` / `Router.HandleLoopback` keyed on `X-Forwarded-Proto`
only). dropbox no longer compiles against it. This Decision conforms dropbox to
that contract; it cites `docs/structured-mcp-design.md` and reshapes **no**
result content — it formalizes the JSON dropbox already emits.

**Decision.**

**1. `StructuredResult` on every domain tool.** Each of dropbox's six domain
tools (`list`, `get`, `put`, `mkdir`, `delete`, `move`) returns its success
result through `appkitmcp.StructuredResult(v)` instead of `JSONResult(v)`. The
new signature returns `(map[string]any, error)`; the marshal error is propagated
honestly (returned as the handler's `error`, surfaced by the transport as a
JSON-RPC internal error), **never** swallowed into a nil result. The emitted
value `v` is byte-for-byte the map each handler builds today — `list`'s
`{files, next_cursor?}`, `get`'s six-key envelope, `put`'s
`{path,size,content_hash,rev}`, `mkdir`'s `{path}`, `delete`'s `{removed}`,
`move`'s `{from,to}` — so `structuredContent` carries the machine rendering and
the mirrored text block carries the identical JSON for agents.

**2. Typed error codes.** The hand-built `{error:{code,message}}` body and the
string-taking `ErrorResult` are gone. `toolErr(err)` maps dropbox's domain
sentinels onto the closed `appkitmcp.ErrorCode` vocabulary and returns
`appkitmcp.ErrorResult(code, err.Error())`:

| condition | `ErrorCode` |
|---|---|
| `dropbox.ErrNotFound` | `ErrNotFound` (`not_found`) |
| `dropbox.ErrRevMismatch` | `ErrConflict` (`conflict`) |
| `dropbox.ErrValidation`, `dropbox.ErrPathEscape` | `ErrValidation` (`validation`) |
| any other / unrecognized | `ErrInternal` (`internal`) |

The two size caps (`get` on the indexed size, `put` on the decoded inline body)
return `ErrorResult(ErrTooLarge, …)`; the `source_url` unavailability path
(`sourceUnavailable`) returns `ErrorResult(ErrSourceUnavailable, …)`. The
`too_large` and the `source_url` code **values** are already pinned behaviors
(D19 R-KT5G-WI4R, R-Q8Q0-P1TS); this Decision only re-homes them from the retired
nested body into the chassis envelope. The retired `toolErrorJSON` helper is
deleted. The resulting error result is the chassis shape: `isError: true`, a text
block equal to the human message, and `structuredContent` = the flat
`{code, message}` object — **not** the old `{error:{…}}` nesting.

**3. Declared `outputSchema` per domain tool.** Each domain tool declares an
`OutputSchema map[string]any` JSON-schema literal that mirrors the exact JSON it
emits, following the `objectSchema` house style in `appkit/mcp/mcp.go`
(dropbox writes its own literals via a small private helper paralleling its
existing input-schema `obj`/`descTyp`). The schemas, verbatim to the emitted
keys and their Go types (`Size int64` → `integer`; `Rev`/`ContentHash`/
`UpdatedAt`/`Path` → `string`; `removed` → `integer`):

- `list` → `object` `{files: array of object{path,kind,size,hash,rev,updated_at}
  (all required), next_cursor: string (optional)}`; required `[files]`.
- `get` → `object` `{path,size,content_hash,rev,updated_at,content_base64}`; all
  six required.
- `put` → `object` `{path,size,content_hash,rev}`; all four required.
- `mkdir` → `object` `{path}`; required `[path]`.
- `delete` → `object` `{removed}`; required `[removed]`.
- `move` → `object` `{from,to}`; both required.

`tools/list` surfaces each schema (appkit emits `outputSchema` when
`Tool.OutputSchema` is non-nil). The schemas formalize existing output; they do
not reshape it.

**4. No prose exceptions.** dropbox declares **no** documentation tool (`guide`,
`describe`) and no raw-content read: all six domain tools emit structured JSON
objects, so all six declare an `outputSchema` and none opts out. In particular
`get` is **not** a raw-content read — it returns a structured metadata envelope
whose `content_base64` field carries the bytes inside JSON — so it declares a
schema like the rest. The chassis `health` and `reflection` tools already declare
schemas and return structured results centrally (appkit); dropbox does nothing
for them.

**5. Shared loopback guard.** The three hand-copied inline predicates —
`internal/dropbox/list.go` (`ListHandler`), `internal/dropbox/content.go`
(`ContentHandler`), and the `loopbackRejected` helper in
`internal/dropbox/write.go` (used by `WriteHandler`, `MkdirHandler`,
`MoveHandler`, `StatHandler`) — are removed. The seven loopback filesystem routes
are mounted at the composition root through `rt.HandleLoopback(pattern, h)`
instead of `rt.Handle`, so the shared `server.LoopbackOnly` wrapper is the single
guard. It keys on `X-Forwarded-Proto` **only**: identity headers are deliberately
ignored, because legitimate loopback machine callers (scripts, prompts) assert
`X-Owner-Email`/`X-Client-Id` themselves. This narrows the guard to its real
question — "did this cross nginx?" — matching `docs/structured-mcp-design.md`'s
identity contract. The handler bodies keep their existing per-route logic
(resolution, error mapping, response shaping) minus the deleted first-line
rejection.

**Rejected.**
- **A per-service `StructuredResult` shim** keeping the `JSONResult` call sites
  intact — pointless indirection around a mechanical rename; the chassis helper
  is the contract.
- **Keeping the `{error:{code,message}}` body** as a nested field inside
  `structuredContent` — machine callers branch on a top-level `code`
  (`docs/structured-mcp-design.md`); a bespoke nesting would be a second,
  drift-prone error shape.
- **Leaving the inline guard predicates and only adding the chassis wrapper** —
  two guards with different keys (identity-or-proto vs proto-only) would keep the
  bug the contract fixes: a caller-asserted `X-Owner-Email` on loopback would
  still 404.
- **Runtime schema validation of every result** — out of scope per the suite
  contract; `outputSchema` is the declared machine contract, and conformance is
  this service's own test suite's job, not a transport check.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `internal/mcp.NewHandler` over a domain `dropbox.Service`, driven
through the real `ServeHTTP` seam via `server.New`, exactly as the existing
`list`/`get`/`put` tests do; loopback-guard claims run against the real mounted
routes via `net/http/httptest`).
- R-7PKS-A5KE — a successful `tools/call` of **each** domain tool (`list`, `get`,
  `put`, `mkdir`, `delete`, `move`) returns a result that carries **both** a
  `structuredContent` object **and** a `content[0]` text block whose text is the
  identical JSON marshalling of that same object (the `StructuredResult` dual
  rendering) — never a text-only result and never a bare marshalled string.
- R-7QSO-NXB3 — `tools/list` advertises a non-nil `outputSchema` for **every** one
  of the six domain tools (no prose-exception tool omits one), and each schema is
  an `object` whose `required` keys match exactly the keys that tool emits: `list`
  requires `files` (its `files` items require `path,kind,size,hash,rev,
  updated_at`); `get` requires `path,size,content_hash,rev,updated_at,
  content_base64`; `put` requires `path,size,content_hash,rev`; `mkdir` requires
  `path`; `delete` requires `removed`; `move` requires `from,to`. A tool missing a
  schema, or a schema whose required set diverges from the emitted keys, fails.
- R-7S0L-1P1S — a domain-tool error result has the chassis shape: `isError` true,
  a `content[0]` text block equal to the human message, and `structuredContent`
  equal to the **flat** `{code, message}` object where `code` is one of the closed
  `ErrorCode` vocabulary strings — and specifically **not** the retired nested
  `{error:{code,message}}` body (a result carrying an `error` key under
  `structuredContent`, or a top-level non-vocabulary `code`, fails).
- R-7T8H-FGSH — `toolErr` maps dropbox's domain sentinels to the typed codes:
  `ErrNotFound`→`not_found`, `ErrRevMismatch`→`conflict`,
  `ErrValidation`→`validation`, `ErrPathEscape`→`validation`, and an
  unrecognized/other error → `internal` (a table over the sentinels; a wrong code
  for any row, or a leaked bare transport error for the unrecognized case, fails).
- R-7UGD-T8J6 — the loopback filesystem routes are guarded by the shared chassis
  guard keyed on `X-Forwarded-Proto` only: a request to a loopback route
  (`GET /content`, `PUT /content`, `DELETE /content`, `POST /mkdir`, `POST /move`,
  `GET /list`, `GET /stat`) bearing `X-Forwarded-Proto: https` receives a bare
  `404` and its handler does **not** run (no mirror/index side effect); the
  byte-equivalent request **without** that header but **with** `X-Owner-Email`
  set is served normally (the retired two-header predicate would have 404'd it).
