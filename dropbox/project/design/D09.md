# Decision 9 — Adopt `registry`: resolve dropbox's own loopback address by name

**Decision.** dropbox writes no loopback port literals in its Go source: it
references itself **by name** through the shared `registry` library
(`module registry`, the repo-root leaf module). Resolution happens **once, at
the composition root** (`cmd/dropbox/main.go`) — never per request — because the
table is compile-time data: if a port ever changes we rebuild the binaries
anyway. This is **behavior-preserving**: `registry`'s table carries dropbox's
current value (`dropbox=3200`, in the connectors block `3200–3299`), so every
resolved value is byte-identical to the literal it replaced.

**Why.** "Which port a service answers on" used to be duplicated across the
suite and drift silently — a renumber only surfaced at deploy, uncatchable by
the compiler. Making `registry` the single source of truth means dropbox asks
the table for the address and never repeats the number. notify and crm are the
prior conversions; dropbox follows the same shape.

**The call sites (all inside `cmd/dropbox/main.go` and `internal/dropbox`).**
dropbox has **three** own-address literals today, and all three become a
`registry` lookup byte-identically:

1. **The listen port.** The appkit `Spec.Port` value `3200` →
   `registry.MustPort("dropbox")` — the strict composition-root form that panics
   on an unknown name (a programming error, per notify D9):

   ```go
   appkit.Main(appkit.Spec{
       App:   "dropbox",
       Mount: "/srv/dropbox/",
       Port:  registry.MustPort("dropbox"),
       ...
   })
   ```

2. **The content-base default port.** In the `Handlers` hook, the loopback
   `/content` base URL is composed from `DROPBOX_IP` + `DROPBOX_PORT`; the
   `DROPBOX_PORT` **default** literal `3200` →
   `config.EnvOrInt(os.Getenv, "DROPBOX_PORT", registry.MustPort("dropbox"))`.
   The env override (set by `bin/start` and the box) is unchanged; only the
   fallback default stops being a bare literal. `DROPBOX_IP` (default
   `127.0.0.1`, no port) is untouched — it carries no `:30xx` address form.

3. **The reflection example origin.** `internal/dropbox/events.go` builds the
   file-event reflection **Sample** (the `create`/`modify`/`delete` payload sample, D22 addressing) with a hardcoded
   `contentURL("http://127.0.0.1:3200", "/notes/meeting.md")`. The origin
   literal `"http://127.0.0.1:3200"` → `registry.BaseURL("dropbox")` (==
   `"http://127.0.0.1:3200"`, byte-identical). The adjacent doc comment on
   `NewOutboxProducer` that spells the same `"http://127.0.0.1:3200"` address in
   prose is reworded to describe the loopback `/content` origin **without** the
   literal address, so the D10 source-scan guard has no comment-embedded needle
   to trip on. (This is the one call site notify did not have — dropbox is a
   producer and carries a worked content_url example; see Rejected.)

**Peer addresses are not dropbox's concern.** dropbox is a producer, not a
consumer: it reads no peer feed URLs, so there is no `<APP>_<SRC>_FEED_URL`
resolution here. The loopback `/content` and `/list` byte routes and the
`content_url` stamped into live events use the runtime-composed content base
(call site 2), which is dropbox's **own** address, resolved from `registry`.

**Module wiring (inside `dropbox/` only).** `dropbox/go.mod` gains
`require registry v0.0.0` and a committed `replace registry => ../registry`,
mirroring the existing `appkit` and `eventplane` in-repo replace-siblings.

> **External precondition (OUT OF SCOPE — not built by any dropbox phase).** The
> repo-root `go.work` must carry `use ./registry`, and the `registry` module
> itself must exist and be green. Both are owned at the repo root / by the
> `registry` project and are assumed satisfied. No dropbox Decision or phase
> edits `../go.work`, `../registry/`, `../bin/`, nginx, or any sibling module —
> the executor runs from `dropbox/` and cannot reach outside it.

**Resolve-at-startup, not per-request.** `Spec.Port`, the content-base port, and
the static reflection Sample are all read/built once at boot. No handler or
request path calls `registry`. This adds no runtime cost.

**Rejected.**
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; resolve at the composition root, store the value, reuse
  it.
- **`registry.Port("dropbox")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form.
- **Leaving the `events.go` example literal in place and exempting it from the
  source-scan guard.** A per-file exemption is drift by construction: the next
  hardcoded address in that file would slip through. The example is dropbox's
  own loopback origin, so `registry.BaseURL("dropbox")` is the truthful source
  and removes the literal without changing the reflected bytes.
- **Leaving the `DROPBOX_PORT` default as `3200`.** A second own-port literal
  that would silently disagree with `registry` after a renumber; the default is
  the same value `registry` already owns.

**Verification.**
- R-QJ8F-AXWP — **dropbox's own listen port is `registry.MustPort("dropbox")`**,
  not a `3200` integer literal, at the composition root. Falsifiable: the
  served/emitted port must equal `registry.MustPort("dropbox")`; a hardcoded
  `Port: 3200` in the Spec fails the intent (proven concretely by the manifest
  drift guard R-QMW4-G94S, which emits with `registry.MustPort("dropbox")` and
  matches the committed `PORT=3200`).
- R-QKGB-OPNE — **the file-event reflection Sample's `content_url` origin is
  `registry.BaseURL("dropbox")`.** A test over the `internal/dropbox` `Events`
  registry asserts each sampled payload's `content_url` begins with
  `registry.BaseURL("dropbox") + "/content?path="` (== the old
  `http://127.0.0.1:3200/content?path=…` byte-for-byte); a reintroduced literal
  origin fails it.
